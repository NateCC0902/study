# USV 的 PID 控制 — 從案例中學習

如何使用這份筆記：每個案例都是一場實驗。先讀過設定，預測會發生什麼，再看圖，
然後在 `usv_heading_sim_zh.py` 裡靠改變增益（gain）自己重現一遍。這裡每一張圖
都是用該模擬器（simulator）裡那個一模一樣的模型產生的（`_make_figures_SPOILERS_zh.py` —
裡面有寫好的完整 PID，在你自己動手寫之前別打開它）。

---

## 主角：一艘船，貫穿所有案例

| 項目 | 數值 | 意義 |
|---|---|---|
| 受控體模型（plant model） | Nomoto 模型（Nomoto model）：`T·ṙ + r = K·δ`、`ψ̇ = r` | 舵（rudder）δ → 偏航角速度（yaw rate）r → 航向（heading）ψ |
| K = 0.6 | (rad/s)/rad | 每 rad 舵角（rudder angle）對應的穩態偏航角速度：10° 舵 → 6 °/s 轉向 |
| T = 2.5 s | 時間常數（time constant） | 船體（hull）要花多久才「醞釀」到那個角速度 |
| 舵止（rudder stop） | ±35° | 致動器（actuator）飽和（saturation） |
| 伺服機變化率（servo slew） | 25 °/s | 舵在物理上不可能瞬間跳動 — 這很重要（案例 9） |
| 水流（current）／風（wind） | ≡ 4° 卡住的舵 | 持續推動船首（bow）的固定擾動（disturbance） |
| 控制迴圈（control loop） | 20 Hz（dt = 0.05 s） | 典型業餘 USV 的速率 |

## 案例 0 — 沒有控制器：船自己會怎麼動

**設定：** 把舵固定在 10°，手放開，看著它。

![開迴路](figures/fig0_openloop.png)

**該注意的地方：**
- 偏航角速度不會瞬間跳到 6 °/s — 它是「爬」上去的，在 t = T = 2.5 s 時達到約 63%。
  船體有轉動慣性（inertia）與水動力延遲（lag）。
- 航向永遠不會穩定下來。它會一直斜坡上升，因為航向是偏航角速度的**積分（integral）**。

**底下這一切都源於這兩個事實：**
1. *舵控制的是角速度，不是航向。* 你沒辦法直接命令一個航向；你命令的是轉向，
   而航向是累積出來的。
2. *響應（response）有延遲。* 不管舵現在做了什麼，船還是會把舊的動作再持續個幾秒。

延遲＋累積＝帶著動量衝進目標＝超調（overshoot）。把這個案例記在腦子裡；後面每個案例
都是在跟這兩個事實纏鬥。

---

## 你將要實作的控制律

每個時間步（timestep）（dt = 0.05 s）：

```python
error      = wrap(setpoint - heading)        # rad，永遠先纏繞（案例 8）
integral  += error * dt                      # rad·s，跨呼叫持續保留
derivative = (error - prev_error) / dt       # rad/s

u = kp*error + ki*integral + kd*derivative   # 舵的 rad 數
u = clamp(u, -RUDDER_MAX, +RUDDER_MAX)
```

各增益的單位（值得內化 — 它能讓整定少一點神祕感）：

| 增益 | 單位 | 讀作 |
|---|---|---|
| kp | rad 舵 / rad 誤差 | 「我偏差每一度，要打多少舵」 |
| ki | rad 舵 / (rad·s) | 「我對持續存在的誤差施壓的速度有多快」 |
| kd | rad 舵 / (rad/s) | 「擺動每秒一度，要打多少舵」 |

### 實作範例 — 手算一個時間步

操船途中的一個快照，增益 kp=2、ki=0.2、kd=3：
航向 60°、設定值（setpoint）90°、前一次誤差（error）31°、目前為止的積分量 0.80 rad·s。

| 步驟 | 計算 | 結果 |
|---|---|---|
| error | wrap(90° − 60°) = 30° | 0.5236 rad |
| P | 2 × 0.5236 | **+1.047 rad**（= 60°） |
| integral | 0.80 + 0.5236 × 0.05 | 0.8262 rad·s |
| I | 0.2 × 0.8262 | **+0.165 rad**（= 9.5°） |
| derivative | (0.5236 − 0.5411) / 0.05 | −0.349 rad/s（船首正以 20 °/s 收斂） |
| D | 3 × (−0.349) | **−1.047 rad**（= −60°） |
| 總和 | 1.047 + 0.165 − 1.047 | 0.165 rad = **9.5°**，在 ±35° 內 → 輸出 9.5° |

再讀一次這個結果：船離設定值還有 30°，可是 P 項與 D 項*恰好互相抵消*，
控制器幾乎是在滑行。船首已經以 20 °/s 收進來 — 再用力推只會超調得更兇。
那個抵消**就是**阻尼（damping）。（單靠 P 會命令出 60° → 被箝位（clamp）到 35° →
全速衝進超調。）

## 案例 1 — 只有 P：彈簧

**設定：** `u = kp·e`，從 0° 轉到 90°。掃描（sweep）kp。

![P 掃描](figures/fig1_p_sweep.png)

**該注意的地方：**
- **kp=0.5** — 遲鈍、超調又大又邋遢，最後穩定在約 98°（有偏差（offset）！→ 案例 2）。
- **kp=2** — 俐落，振鈴（ringing）一次，穩定在約 92°。可用。
- **kp=5** — 永遠不穩定。±16° 振盪（oscillation），永無止境。這就是這艘船的
  *臨界增益（ultimate gain, Ku）*（週期（period）Tu ≈ 9 s）。
- **kp=8** — 振盪更大、更快，±25°。

**為什麼會振盪：** P 是一個彈簧（spring）— 力與拉伸量成正比。彈簧上的質量（mass）
若阻尼很弱就會振盪。這艘船唯一的「摩擦」就是船體的偏航阻尼，過了 kp≈5 它就不夠用了
（舵伺服機的延遲會把剩下的工夫補完 — 案例 9）。

**要記住的規律：** kp 越大 = 響應越快、殘餘偏差越小、振鈴越多 — 一直到它再也停不下來為止。

## 案例 2 — P 贏不了穩定的水流

**設定：** 同一艘只有 P 的船，但仔細看它最後穩定在哪裡。水流推船首的力道
就像 4° 卡住的舵。

![偏差](figures/fig2_offset.png)

**數學（算一次就好，只有 3 行）：** 要維持固定航向，淨偏航力矩必須為零，
所以舵必須維持在 −4° 來抵消水流。一個 P 控制器只能輸出 `kp·e`，所以：

```
kp·e_ss = −4°   →   e_ss = −4°/kp
```

| kp | 偏差 | 穩定於 |
|---|---|---|
| 1 | 4° | 94° |
| 2 | 2° | 92° |
| 4 | 1° | 91° |

**陷阱：**「那我把 kp 拉大來縮小偏差就好啦」— 但案例 1 說 kp ≥ 5 會振盪。
用可用的增益時，這個偏差是**結構性的**：誤差為零時，P 控制器輸出零舵，
而零舵贏不了水流。必須有別的東西在誤差為零時把舵撐住 → 案例 4。

## 案例 3 — 加上 D：阻尼器

**設定：** kp=2 固定，加上 `kd·(de/dt)`。掃描 kd。

![PD](figures/fig3_pd.png)

**該注意的地方：**
- kd=0：超調到約 118°，振鈴。
- kd=1：一個小超調。
- kd=3：像泊車小弟一樣滑進去。零超調。

**為什麼：** 誤差快速縮小 ⇒ 微分（derivative）為負 ⇒ 在收斂過程中 D *對抗* P
（再讀一次實作範例 — 那就是這個機制）。D 只對*變化*起作用：一旦上了航向且穩定下來，
D = 0。它無法修正偏差，無法完成轉向；它只馴服運動。

**滑行的經驗法則：** 當 `kp·e = kd·ė` 時控制器歸於中性。在 kp=2、kd=3 時，
一旦收斂速率（°/s）超過剩餘誤差（°）的 ⅔，推力就停止 — 例如離目標 30°，
收斂快過 20 °/s → 開始鬆手。

## 案例 4 — I：配平片（積分到底是「為了什麼」）

**設定：** 船已經穩定在 90°。在 t=20 s 時水流開啟。
比較 PD（kp=2、kd=3）與 PID（同上再加 ki=0.2）。

![積分](figures/fig4_integral.png)

**該注意的地方（下方子圖（panel）才是重點）：**
- PD 那艘船被推歪到 92° 後就停在那裡。它的 P 項最後撐住了 −4° 的舵 —
  但 P 要輸出 −4° 的唯一辦法就是*讓誤差維持在 −2°*。**PD 是靠維持錯誤來保持航向的。**
- PID 那艘船下探到約 91.7°，然後恰好回復到 90.0。它的 I 項爬升到 −4° 並停在那裡 —
  積分已經*把水流學會了*。由 I 扛起配平（trim），誤差回到零，P 也放鬆到零。
- 這要花約 30 s。那份遲緩是個特色：積分應該對*持續存在*的東西作出反應，
  而不是對每一道浪。

**為什麼 ki 必須小：** 積分是一份記憶。ki 大 = 從短記憶裡就跳結論 = 超調與緩慢蛇行
（在模擬器裡試試 ki=2）。積分應該是迴圈裡最慢的東西。

## 案例 5 — 積分飽和：I 的黑暗面

**設定：** 大幅度操船 — 0° 轉到 180° — 配上一個過熱的積分（ki=1、kp=2、kd=3）。
轉向過程中，舵被釘在 +35° 約 9 秒，而此時誤差非常大。積分並不知道舵已經到頂；
它持續累積：約 1.5 rad 平均誤差 × 9 s × ki=1 ≈ 13 rad ≈ 存進 I 項裡的
**750° 命令舵角**。

![積分飽和](figures/fig5_windup.png)

**該注意的地方：** 積分有箝位時（藍色），漂亮地完成 180° 轉向。沒箝位時（橘色），
船衝過 180°，而等到誤差變號時，積分已經臃腫到讓舵一直釘在頂端 —
這艘船**永遠在繞圈圈**（上方子圖：航向衝過 2000°；積分每一圈部分鬆開又重新捲起，
永遠無法恢復）。

這不是什麼罕見的故障。命令一個 USV 去你身後的航點（waypoint），舵飽和，積分捲起來 —
每個業餘自動駕駛儀（autopilot）作者都會遇到這個 bug。

**修法（實作 `update()` 時挑一個）：**
1. **箝位（Clamp）** 積分，讓 `ki·integral` 永遠不會超過舵的極限
   （藍色軌跡做的就是這個 — 最簡單、有效）。
2. **條件積分（conditional integration）** — 在輸出已飽和的取樣點（sample）上不要積分。
3. **反算法（back-calculation）** — 每一步從積分中減去 `(u_unclamped − u_clamped)·k`；
   業界標準，最平滑。

## 案例 6 — 微分衝擊

**設定：** 在 90° 巡航，新航點 → 設定值階躍到 100°。*誤差*在一個 0.05 s 的取樣點內
跳了 10°，所以：

```
derivative = 10° / 0.05 s = 200 °/s   →   D = kd·3.49 rad/s ≈ 600° 的舵（!）
```

![衝擊](figures/fig6_dkick.png)

**該注意的地方：** 用對誤差微分（derivative-on-error）（藍色）時，一個取樣點就把命令
猛甩到 +35° 的舵止 — 一個什麼都沒達成、只會把伺服機（servo）操壞的尖峰。
船並沒有突然開始擺動；*是你的設定值移動了*。

**修法：** 改成對**量測值（measurement）**微分。當設定值固定時，`de/dt = −dψ/dt`，
所以是同一個訊號 — 但它會忽略設定值的跳動。在你的程式裡翻一個符號：

```python
derivative = -(measurement - prev_measurement) / dt    # 取代對 d(error) 微分
```

## 案例 7 — 雜訊：D 的另一個黑暗面

**設定：** 真實的羅盤，每次讀數有 σ = 0.5° 的雜訊（noise）。雜訊的微分非常大：
相鄰兩次讀數平均差約 0.7°，所以

```
derivative noise ≈ 0.5°·√2 / 0.05 s ≈ 14 °/s   →   ×kd=3  ≈ 42° 的垃圾舵角
```

![雜訊](figures/fig7_noise.png)

**該注意的地方：** 上方 — 命令（細線）是白雜訊在猛撞舵止；伺服機（粗線）不停地
在它 25 °/s 的變化率限制（slew rate）上磨。船倒是維持航向維持得好好的！但你的伺服機
一週就掛了，電池一小時就乾了。

**修法：** 對微分做低通濾波（low-pass filter）：

```python
d_filtered = alpha * d_filtered + (1 - alpha) * d_raw    # alpha ≈ 0.85
```

下方子圖：同樣的雜訊，濾波後 — 抖動被砍到只剩一小部分。取捨：濾波器會替 D 加一點延遲，
使阻尼略為削弱。（更嚴重的航向雜訊問題要往上游解，靠融合陀螺儀（gyro）＋羅盤 —
那就是飛控板（flight controller）上的 EKF／互補濾波器（complementary filter）在做的事。）

## 案例 8 — 角度纏繞：經典的載具 bug

**設定：** 航向 350°，設定值 10°。船其實只差 20°。天真的相減卻不這麼認為：

```
e = 10 − 350 = −340°        # 「順時針轉 340°」— 災難性地錯誤
wrap(e) = +20°              # 正確
```

![纏繞](figures/fig8_wrap.png)

**該注意的地方：** 天真的控制器繞了 340° 走遠路。在開闊水域那是喜劇；靠近碼頭時就是撞船。
而且這個 bug 在測試裡是看不見的，除非你的測試剛好跨過正北。

**修法，用在每一個進入 PID 之前的誤差上：**

```python
def wrap(angle):                                  # → (-pi, pi]
    return (angle + math.pi) % (2 * math.pi) - math.pi

error = wrap(setpoint - heading)
```

## 案例 9 — 為什麼「把 kp 拉到它開始振盪」真的有用

關於這個模型的老實告白：一艘純一階（first-order）Nomoto 船，配上一個理想的舵，
**永遠不會**因為 kp 而變得不穩定 — 在早期測試裡，連 kp=100 都穩定得好好的。
那為什麼每一艘真實的船在高增益下都會振盪？

因為真實的致動器有延遲。我們的伺服機以 25 °/s 變化 — 從一頭打到另一頭要 2.8 s。
在高 kp 時，命令在兩端之間翻動的速度比伺服機跟得上的還快：

![速率限制](figures/fig9_ratelimit.png)

**該注意的地方：** 命令（細線）是方波；實際的舵（粗線）是落在後面的鋸齒波。
舵總是在做船一秒鐘前需要的事。那個延遲就是*相位延遲（phase lag）*，
而高增益下的相位延遲就是引發振盪的萬用配方。真實的船有更多相位延遲：
感測器濾波延遲、通訊延遲時間（latency）、推進器（thruster）起轉、船體動態。
延遲越多 → 可用的 kp 越低。

這也是為什麼模擬器內建了變化率限制 — 對著一個不切實際的理想受控體（plant）去整定，
教給你的增益一下水就會崩潰。

## 整定演練 — 這艘船，照規矩來

逐步配方（在模擬器裡現場做）：

1. **ki=0、kd=0。把 kp 拉到出現持續振盪（sustained oscillation）。** 在這艘船上大約是
   kp ≈ 5（= Ku），以週期 Tu ≈ 9 s 振盪。退回一半：**kp = 2 — 2.5**。
2. **把 kd 拉到超調消失。** 這裡 kd ≈ 3 就能消掉它。kd 太多 → 神經兮兮、
   對雜訊敏感（案例 7）。
3. **加上能在合理時間內消除偏差的最小 ki。**
   ki = 0.2 大約在 30 s 內把水流配平掉。如果你看到緩慢蛇行，把它砍半。

結果：**kp=2、ki=0.2、kd=3** — 案例 3–7 裡用的增益。

Ziegler–Nichols 作為對照，從同樣的 Ku=5、Tu=9 算出：

| | kp = 0.6·Ku | ki = 1.2·Ku/Tu | kd = 0.075·Ku·Tu |
|---|---|---|---|
| ZN 經典版 | 3.0 | 0.67 | 3.4 |

kd 跟我們的一致；ZN 的 kp、尤其是 ki 都更熱（ZN 以快速的擾動抑制（disturbance rejection）
為目標，並容忍超調）。把 ZN 當成一個起點，然後往平滑的方向退調（de-tune）。

## 從模擬器到你真正的 USV

| 在模擬器裡 | 在船上（ROS2） |
|---|---|
| `psi` | 來自 IMU／`/odometry/filtered` 的偏航（已融合） |
| `wrap(sp − psi)` | 一樣 — 在你的節點（node）裡做，每一週期都做 |
| `DT = 0.05` | 計時器回呼（timer callback）約 20–50 Hz，但要用*量測到的* dt |
| 舵 δ | 舵伺服機，或差動推力（differential thrust）：`left = base+u, right = base−u` |
| `RUDDER_MAX` | 你的致動器極限 — 在軟體裡也要箝位 |
| 水流擾動 | 真實的水流／風 — 那是你 I 項的工作 |

實地整定的順序是一樣的：P → D → I，從低開始。兩個安全提醒：
在任何自主航行（autonomy）測試之前要有一個硬體緊急斷電開關（kill switch），
而且每次控制器重新接管時都要 `reset()` 一下 PID（5 分鐘前留下的陳舊積分，
就是一個你還沒遇過的積分飽和 bug）。
