CAN & SocketCAN: the wire that runs the actuators

Serial and Ethernet move data. CAN is the bus you reach for when a thruster, a rudder servo, and a battery monitor all have to share two wires, never collide, and keep working when one node browns out. And the best part on Linux: it shows up as a network interface, so you drive it with sockets and ip, not some bespoke driver. This chapter takes you from the differential pair on the wire up to candump can0.

1Why a robot needs CAN at all

① Intuition build the picture first

You already have UART (Ch 06) and Ethernet (Ch 09). Why add a third bus? Because actuators have a different problem than sensors. A GPS just streams; if a byte is late, who cares. But a motor controller, a steering servo, and a power monitor are all talking on the same two wires, all the time, and the message that says "EMERGENCY STOP" must win the wire over the message that says "battery is at 84%." CAN — Controller Area Network, born in 1980s automotive — is built for exactly that: many nodes, one shared pair, deterministic priority, and graceful behaviour when something on the bus dies.

💡 The one-sentence model

CAN is a shared two-wire broadcast bus where the lowest message ID always wins the wire — and it wins without corrupting the loser. That single property gives you collision-free, priority-ordered, multi-master messaging on a pair of cheap twisted wires.

Notice what CAN is not: it is not addressed to a node. A CAN frame has no destination field. Every node hears every frame and decides, by the 11-bit (or 29-bit) identifier, whether it cares. The ID is simultaneously the message's name and its priority. Hold that thought — it explains arbitration, hardware filtering, and bus-load all at once.

🏭 Real case — the USV actuator bus

On our Jetson AGX Orin USV, the heavy sensors (LiDAR, cameras) ride Ethernet and USB. The actuators — twin thruster ESCs, the rudder/azimuth servo, and the battery management system — sit on a single CAN bus at 500 kbit/s. Separately, the marine instruments (GPS compass, depth, wind) speak NMEA 2000, which is just CAN at 250 kbit/s on a beefier connector. Two physical CAN segments, one mental model.

2The physical layer: two wires, one dominant bit

① Intuition what's actually on the wire

CAN is a differential bus: two signals, CAN High (CANH) and CAN Low (CANL). A receiver looks at the difference $V_{\text{diff}} = V_{\text{CANH}} - V_{\text{CANL}}$, not the absolute voltage. That is why CAN survives in an engine bay or a boat hull full of motor noise: a spike hits both wires roughly equally and cancels in the subtraction. This is the same common-mode-rejection trick that makes Ethernet's twisted pairs robust.

There are exactly two bus states, and their names matter:

StateLogical bit$V_{\text{diff}}$ (high-speed CAN)Who drives it
Dominant0~2 V (CANH≈3.5 V, CANL≈1.5 V)a transceiver actively pulls the pair apart
Recessive1~0 V (both ≈2.5 V)nobody drives; resistors relax the pair together
🔑 Dominant beats recessive — physically

It's a wired-AND. If any node drives dominant (0) while others sit recessive (1), the whole bus reads 0. A recessive bit can never overwrite a dominant one. This is not a software rule — it is what the transceivers physically do. Every clever thing CAN does flows from this one fact.

② Math termination and the 60-ohm test

The pair is a transmission line, so it must be terminated at both ends with $120\,\Omega$ resistors to kill reflections. The two terminators sit in parallel across the bus, so an ohmmeter on a powered-off, healthy bus reads:

$$ R_{\text{bus}} = \frac{R_1 \, R_2}{R_1 + R_2} = \frac{120 \cdot 120}{120 + 120} = 60\;\Omega $$

This is the single most useful field test you own. Power down the bus, measure CANH-to-CANL with a multimeter:

ReadingDiagnosis
≈ 60 ΩHealthy — two terminators present.
≈ 120 ΩOne terminator missing — may work short & slow, fails at speed/length.
≈ 40 ΩThree terminators — someone added one in the middle. Remove it.
open / very highBoth terminators missing or a broken wire.
⚠️ Pitfall — terminators go at the ends, not on every node

Termination belongs to the bus, not to nodes. Exactly two, one at each physical extreme. A common USV failure: every dev board ships with a jumper-enabled 120 Ω terminator, you wire up four nodes, and now the bus reads 30 Ω and won't run at 500k. Pull the jumpers on the inner nodes.

3Arbitration: the lowest ID wins, with no collision

① Intuition bit-by-bit listening

Here is the elegant part. Multiple nodes may start transmitting in the same bit slot. As each node sends a bit of its identifier, it also listens to the bus. If a node sends recessive (1) but reads back dominant (0), it knows someone with a higher-priority message is also talking — so it immediately, silently, backs off and becomes a receiver. The node sending dominant never even noticed a contest; its frame continues uninterrupted.

Because dominant = 0 wins, the message whose ID has the smallest numeric value wins arbitration. Lower ID = more leading dominant bits = higher priority. No collision, no retransmission of the winner, no wasted bus time. The loser simply retries when the bus is idle again.

sequenceDiagram participant A as Node A id 0x100 participant Bus participant B as Node B id 0x101 Note over A,B: identifier bits 10..1 identical so far A->>Bus: bit 0 = 0 (dominant) B->>Bus: bit 0 = 1 (recessive) Note over B: reads back 0, not 1 → loses, backs off A->>Bus: sends rest of frame unaware Note over B: waits for idle, retries
⚠️ Pitfall — never reuse an ID

Two nodes transmitting the same ID with different data will not arbitrate cleanly — they win arbitration together and then collide in the data field, triggering a bit error and a retransmit storm. Every transmitter on the bus must own a unique set of IDs. NMEA 2000 solves this with a dynamic address-claim procedure that is, at its core, this same lowest-ID-wins game.

🏭 Real case — priority on the USV

We assign the thruster command frames the lowest IDs on the bus, the BMS over-current/fault frames lower still, and slow telemetry (temperatures, firmware version) the highest IDs. When the bus gets busy during an aggressive maneuver, the safety-critical traffic literally pushes the chatter aside on the wire. Priority is a wiring decision you make once, in the ID map.

4Frame anatomy and bit-stuffing

② Math count the bits

A classic CAN data frame (Standard format, 11-bit ID) is more than its payload. The fields, in order on the wire:

FieldBitsPurpose
SOF1Start-of-frame (dominant) — wakes everyone, syncs clocks.
Identifier11Name + priority (arbitration field).
RTR + IDE + r03Remote/data flag, extended-ID flag, reserved.
DLC4Data length code: 0–8 payload bytes.
Data$8 \times p$$p$ payload bytes, $0 \le p \le 8$.
CRC + delimiter1615-bit CRC + 1 recessive delimiter.
ACK + delimiter2Receivers pull this dominant to acknowledge.
EOF7End-of-frame (recessive).
IFS3Inter-frame space before the next frame.

Sum the fixed overhead and you get a clean formula for the nominal frame length of a standard 11-bit frame carrying $p$ payload bytes:

$$ N_{\text{nominal}}(p) = 47 + 8p \quad\text{bits} \qquad\Rightarrow\qquad N_{\text{nominal}}(8) = 47 + 64 = 111\ \text{bits} $$

But the wire carries more bits than that, because of bit-stuffing. CAN has no separate clock wire; receivers recover timing from edges in the data. After 5 consecutive identical bits the transmitter inserts one opposite "stuff" bit to guarantee an edge, and the receiver silently strips it. Stuffing applies to everything from SOF through the CRC — that is $34 + 8p$ stuffable bits — so the worst case adds:

$$ N_{\text{stuff,max}}(p) = \left\lfloor \frac{34 + 8p - 1}{4} \right\rfloor, \qquad N_{\text{worst}}(p) = N_{\text{nominal}}(p) + N_{\text{stuff,max}}(p) $$

For $p = 8$: $\lfloor (34+64-1)/4 \rfloor = \lfloor 97/4 \rfloor = 24$ extra bits, so worst case $= 111 + 24 = 135$ bits on the wire. This variability is exactly why CAN gives you bounded latency for high-priority frames, not constant latency — the worst-case bound assumes the worst-case stuffing on everything ahead of you. (Those constants — $47+8p$ and the stuffing floor — are precisely what the sim below computes, so its numbers will line up with these.)

5SocketCAN: CAN as a network interface

① Intuition the Linux design choice

Here is where Linux earns its keep. Other operating systems hand you a vendor SDK and a character device per CAN chip. Linux instead models a CAN controller as a network interface — can0, right next to eth0 and wlan0 in ip link. You open a socket in the new address family AF_CAN, bind it to can0, and then read()/write() fixed-size struct can_frame records. All the tooling you already know — ip, sockets, tcpdump-style sniffers — applies. That is the whole idea: CAN is just another network.

③ Code read every line

A raw CAN socket is almost boringly familiar if you've written UDP. The one twist is binding by interface index via SIOCGIFINDEX:

can_rx.c C
#include <linux/can.h>
#include <linux/can/raw.h>
#include <net/if.h>
#include <sys/ioctl.h>
#include <sys/socket.h>
#include <string.h>
#include <unistd.h>
#include <stdio.h>

int main(void) {
    int s = socket(PF_CAN, SOCK_RAW, CAN_RAW);   // AF_CAN raw socket
    struct ifreq ifr;
    strcpy(ifr.ifr_name, "can0");
    ioctl(s, SIOCGIFINDEX, &ifr);                 // resolve can0 -> index

    struct sockaddr_can addr = {0};
    addr.can_family  = AF_CAN;
    addr.can_ifindex = ifr.ifr_ifindex;
    bind(s, (struct sockaddr *)&addr, sizeof(addr));

    struct can_frame f;
    while (read(s, &f, sizeof(f)) == (ssize_t)sizeof(f)) {
        printf("id=%03X dlc=%u  ", f.can_id & CAN_SFF_MASK, f.can_dlc);
        for (int i = 0; i < f.can_dlc; i++) printf("%02X ", f.data[i]);
        printf("\n");
    }
    close(s);
    return 0;
}

Reads are whole frames, not byte streams — no partial-read dance like the serial chapter, because read() returns exactly one struct can_frame or nothing. You can also push hardware filters into the kernel with setsockopt(s, SOL_CAN_RAW, CAN_RAW_FILTER, ...) so the controller only wakes you for the IDs you care about — essential when you sit on a busy NMEA 2000 backbone and want only the heading PGN.

6Bring-up, bit timing, and the sample point

③ Code configure the interface

A SocketCAN interface comes up down and with no bitrate. You configure it with the same ip link you use for Ethernet, then bring it up:

can-up.sh bash
# configure can0: 500 kbit/s, auto-recover 100 ms after bus-off
sudo ip link set can0 type can bitrate 500000 restart-ms 100
sudo ip link set up can0

# verify state, bitrate, sample point, and error counters
ip -details -statistics link show can0

# live sniff and decode every frame on the bus
candump can0

# send one frame: id 0x123, payload DE AD BE EF
cansend can0 123#DEADBEEF

# offered-load / frame-rate statistics for the running bus
canbusload can0@500000
⚠️ Pitfall — you cannot reconfigure a live interface

ip link set ... type can bitrate ... fails with RTNETLINK answers: Device or resource busy if the link is up. Bring it down first (sudo ip link set down can0), change the bitrate, bring it back up. And every node on the bus must use the same bitrate — a single 250k node on a 500k bus throws errors for everyone.

② Math bit time and the sample point

One bit isn't sampled at its centre. CAN divides each bit into time quanta grouped into segments, and the controller samples the bus at the sample point — the fraction of the bit elapsed before the decision is made:

$$ \text{sample point} = \frac{T_{\text{sync}} + T_{\text{seg1}}}{T_{\text{sync}} + T_{\text{seg1}} + T_{\text{seg2}}} $$

For high-speed CAN the recommended sample point is ~87.5% of the bit. Why so late? Because arbitration needs the signal to propagate to the far terminator and back within one bit time so a node can read its own transmitted bit before sampling. That round-trip is what caps bus length at a given bitrate: 1 Mbit/s tops out near 40 m, while 250k (NMEA 2000) comfortably spans ~250 m. The kernel computes the segment split for you from the bitrate and the controller's clock; you usually only override it (sample-point 0.875) when mixing pernickety third-party nodes.

7Error states, bus-off, and CAN-FD

① Intuition a bus that polices itself

CAN nodes are not passive. Every controller keeps two counters — a Transmit Error Counter (TEC) and a Receive Error Counter (REC) — and bumps them on each detected error, decrements them on success. These counters drive a state machine that is the heart of CAN's fault containment:

stateDiagram-v2 [*] --> ErrorActive ErrorActive --> ErrorPassive: TEC or REC > 127 ErrorPassive --> ErrorActive: counters fall < 128 ErrorPassive --> BusOff: TEC > 255 BusOff --> ErrorActive: restart-ms timer

Error-active is normal: the node can signal errors loudly (active error flags). At TEC>127 it drops to error-passive — still talking, but politely, so a babbling node can't dominate. At TEC>255 it goes bus-off: it disconnects from the bus entirely. This is the safety valve — one short-circuited or insane node removes itself rather than jamming the whole bus.

🔑 restart-ms is your auto-recovery

By default a bus-off interface stays dead until you manually cycle it. The restart-ms 100 we set at bring-up tells the kernel to automatically re-enable the controller 100 ms after bus-off. On a USV that means a momentary wiring glitch self-heals instead of killing thruster control until a human power-cycles. Watch the counters with ip -details -statistics link show can0 — a climbing bus-off count is your early-warning of a flaky connector.

④ Real case CAN-FD and NMEA 2000

CAN-FD (Flexible Data-rate) keeps the same arbitration but switches to a faster bitrate after arbitration and allows up to 64-byte payloads. You bring it up with both a nominal and a data bitrate: sudo ip link set can0 type can bitrate 500000 dbitrate 2000000 fd on. Same sockets, same tools — candump just shows longer frames. NMEA 2000, the marine standard, is plain classic CAN at 250 kbit/s with a 29-bit extended ID, a defined connector/power scheme, and a fixed PGN message catalogue (heading, position, depth, wind). To Linux it is just another can0 at 250k — you decode the PGNs in userspace.

8Sizing the bus: load and the live gauge

② Math bus load percent

Bus load is the fraction of wire-time consumed by frames. With $f$ frames per second of $N$ bits each at bitrate $B$:

$$ \text{load} = \frac{f \cdot N}{B} \times 100\% $$

Keep nominal load under ~50–70%. Above that, worst-case bit-stuffing plus retransmissions pushes you toward saturation, and low-priority frames start starving. Play with the gauge — drag bitrate, frame rate, and payload and watch the green/amber/red zones. The dashed marker is the worst-case-stuffed load.

🚦 CAN bus-load gauge (bitrate × frame-rate × payload)
🔑 What to notice

Load scales linearly with frame rate and (through $47+8p$) roughly with payload, and inversely with bitrate — so doubling the bitrate halves the load. Watch the dashed worst-case marker pull away from the solid nominal bar as payload grows: that gap is bit-stuffing, and it is the headroom you must leave. A bus that reads 65% nominal can hit 80%+ worst-case. Size for the worst case, not the average — that is the whole reason you chose CAN for the actuators.

9Your turn

Q1 Conceptual check
Three nodes start transmitting in the same slot with standard IDs 0x2A0, 0x180, and 0x181. Which node wins arbitration and keeps transmitting?
  • The node with ID 0x2A0 — highest number, highest priority.
  • The node with ID 0x180 — the lowest numeric ID wins.
  • The node with ID 0x181 — it sent first.
  • All three collide and every node retransmits after a backoff.
Dominant (0) overrides recessive (1) bit-by-bit, so the smallest ID has the most leading dominant bits and wins. 0x180 and 0x181 agree until the last bit, where 0x180 sends 0 and 0x181 sends 1; 0x181 reads back 0, loses, and backs off. 0x2A0 diverges even earlier. The winner never notices a contest; the losers retry at the next idle.
Q2 Compute it
A 500 kbit/s bus carries 1000 frames/s, each a standard 11-bit frame with an 8-byte payload. Using the nominal length $N = 47 + 8p$ and $\text{load} = fN/B \times 100\%$, what is the nominal bus load in percent?
%
$N = 47 + 8\times 8 = 111$ bits. $\text{load} = (1000 \times 111)/500000 \times 100 = 111000/500000 \times 100 = 22.2\%$. Comfortable headroom — even worst-case stuffing ($135$ bits → $27\%$) stays well under 50%. That matches the green zone on the gauge above for these settings.
Q3 Hands-on — bring up and sniff a CAN bus
You have a fresh can0 on the USV. Write the commands to: configure it for 500 kbit/s with 100 ms bus-off auto-recovery, bring it up, watch live traffic, and send a single test frame with ID 0x123 and payload bytes DE AD BE EF. Then say how you'd confirm it actually went out.
Hint
Configuration is an ip link set ... type can call; the interface starts down so you need a second set up. The sniffer and the sender are candump / cansend from can-utils.
Show one solution
bring up + test bash

The injected frame appears in the candump window as can0 123 [4] DE AD BE EF. Because SocketCAN echoes locally-sent frames to other sockets by default, you see your own cansend even with no other node present. If nothing shows up and the TX error counter in ip -details climbs, you have no ACK — meaning no second node is acknowledging, or termination/bitrate is wrong. That distinction (frame echoed but no real ACK) is exactly why the 60 Ω test and matched bitrates matter.

🔑 Chapter recap

CAN is a shared, differential, two-wire broadcast bus terminated $120\,\Omega$ at each end (the 60 Ω test). Dominant bits beat recessive ones physically, so the lowest ID wins arbitration with no collision. A standard frame is $47 + 8p$ bits plus worst-case bit-stuffing, and bus load is $fN/B$ — keep it under ~50–70%. Linux exposes CAN as a network interface: open an AF_CAN socket, bring it up with ip link set can0 type can bitrate 500000 restart-ms 100, and sniff with candump. Error counters drive error-active → passive → bus-off, and restart-ms auto-recovers. CAN-FD adds speed and 64-byte payloads; NMEA 2000 is just CAN at 250k for the marine instruments.
