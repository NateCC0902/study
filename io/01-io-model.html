The Linux I/O Model: how bytes actually cross the boundary

Every sensor reading on your robot — every IMU sample, every GPS fix, every LiDAR packet — is a sequence of bytes that had to cross from a device, through the kernel, into your process. Before you can debug a dropped sample or a stuttering control loop, you need a correct mental model of that crossing: what a file descriptor is, what read() actually promises, and why a single innocent-looking call can quietly cost you a microsecond you didn't budget for.

1Why I/O is the hard part of robotics

① Intuition build the picture first

The interesting code in a robot — the EKF, the path planner, the PID loop — is the part everyone talks about, but it is rarely the part that breaks at 2 a.m. on the water. What breaks is I/O: a serial port that returns half a message, a sensor that vanishes when you unplug a different one, a logging write that blocks the control thread for 40 ms while an SD card does garbage collection. The algorithm is deterministic math. I/O is where your clean code meets a physical, asynchronous, fallible world, and that boundary is where the bodies are buried.

Here is the uncomfortable truth this whole course exists to fix: I/O is not a library call, it is a negotiation with the kernel. When you call read(), you are not "getting data" — you are asking a different, privileged piece of software to do something on your behalf and tell you how it went. Treat it like a function that returns a value and you'll be fine until the day it returns less data than you asked for, and then your code corrupts a packet and you spend a week blaming the sensor.

🔑 The thesis of this chapter

read() and write() return a count, not "success." A short return is normal, not an error. Almost every robust-I/O bug is a failure to take that one sentence seriously.

2User space, kernel space & the syscall boundary

① Intuition two worlds, one door

Your process runs in user space: a sandbox where it can do arithmetic and touch its own memory but cannot talk to hardware, cannot read another process's memory, and cannot do anything that would let a buggy program take down the machine. The kernel runs in a privileged CPU mode where it can do all of that. The two are separated by a hardware wall the CPU enforces. The only sanctioned door through that wall is the system call (syscall).

A syscall is not a normal function call. A normal call jumps to an address in your own code. A syscall executes a special instruction (svc on the AArch64 of a Jetson, syscall on x86-64) that traps into the kernel: the CPU switches to privileged mode, saves your registers, and jumps to a fixed kernel entry point selected by a number you placed in a register. The kernel does the work, puts a return value back in a register, and switches the CPU back to user mode. That round trip is a mode switch, and it is not free — typically a few hundred nanoseconds on modern hardware even when the kernel does almost nothing.

② Code the call you don't see

You almost never invoke a syscall directly. You call read() in libc (glibc, or bionic in parts of the Jetson L4T userland), and libc is the thin shim that loads the syscall number, executes the trap instruction, and translates the kernel's error convention (a negative return like $-\texttt{EINTR}$) into the familiar "return $-1$ and set errno" you see in C. So the real path for every byte is four layers deep:

You can watch this happen. strace intercepts every syscall a process makes; perf trace times them. Run them against any program touching a sensor and you see the true cost of your I/O, not the cost of your algorithm.

trace the syscalls a program really makes bash
# Show every read/write a program does, with byte counts and timing
strace -T -e trace=read,write,openat,close ./read_imu

# Aggregate: which syscalls dominate, total time, call count
strace -c ./read_imu

# Per-CPU view of mode switches under load (needs perf installed)
perf trace -s ./read_imu
⚠️ The cost is per-call, not per-byte

A mode switch costs roughly the same whether read() returns 1 byte or 64 KB. That single fact drives the throughput math in §7 and is the reason naive byte-at-a-time loops are catastrophic on a busy robot.

3File descriptors & the fd table

① Intuition a handle, not the thing

A file descriptor (fd) is just a small non-negative integer. It is your process's handle to an open file description the kernel maintains on your behalf — a structure holding the current offset, the access mode (read/write/nonblocking flags), and a pointer to the actual object (a tty, a socket, a regular file). You never see that structure; you only hold the integer index into your process's fd table, and you pass that integer to every I/O syscall.

Three fds are open before your main() runs, by convention inherited from the shell: 0 = stdin, 1 = stdout, 2 = stderr. That is why printf goes to fd 1 and a crash log goes to fd 2 — and why 2>&1 in the shell means "make fd 2 point at whatever fd 1 points at."

🔑 The lowest-free-fd rule

open() always returns the lowest-numbered fd not currently in use. This is a guarantee, not a hint. It is exactly why the classic redirection trick works: close(1) then open("/var/log/mission.log") reuses fd 1, so everything that writes to stdout now lands in the file. Surprising the first time, dependable forever after.

You can inspect any process's live fd table from /proc — no debugger needed. Each entry is a symlink to the object it refers to.

inspect a running process's open fds bash
# Every fd of process <PID>, shown as symlinks to the underlying object
ls -l /proc/<PID>/fd
# lrwx------ 1 root root 0 -> /dev/pts/0
# lrwx------ 1 root root 1 -> /dev/pts/0
# lrwx------ 1 root root 2 -> /dev/pts/0
# lrwx------ 1 root root 3 -> /dev/ttyTHS1      # the IMU's serial port

# How many fds can this process open at once? (default soft limit)
ulimit -n            # e.g. 1024 — matters once you run epoll on many sensors
⚠️ Leaked fds are silent until they aren't

Forget to close() in a reconnection loop and you leak one fd per retry. Nothing fails — until you hit the ulimit -n ceiling and the next open() returns -1 with errno = EMFILE, usually mid-mission. Watch the count grow with ls /proc/<PID>/fd | wc -l.

4The core syscalls & the partial-read truth

② Math the contracts

The whole everything-is-a-file API is six syscalls. Learn their return contracts, not just their names — the contract is where the bugs live.

SyscallSignature (simplified)Returns
openopen(path, flags, mode)new fd (lowest free), or −1 + errno
readread(fd, buf, count)bytes read (0…count); 0 = EOF; −1 + errno
writewrite(fd, buf, count)bytes written (0…count); −1 + errno
closeclose(fd)0, or −1 + errno (and the fd is gone either way)
lseeklseek(fd, off, whence)new absolute offset; −1 + errno (fails on pipes/sockets)
ioctlioctl(fd, request, arg)device-specific; the "everything else" escape hatch

The two return values that ruin careers are on read and write: they return a count, and the count can be less than you asked for. A read(fd, buf, 64) may legally return 17. This is not corruption and not an error — it is the kernel saying "here is what was available right now." On a serial fd it happens constantly, because bytes arrive at the baud rate (one byte every $\sim$87 µs at 115200 baud) and the kernel hands you whatever is in the tty buffer the moment you ask. A 40-byte IMU frame is very often delivered as a 31-byte read followed by a 9-byte read.

⚠️ The short-read bug, in one line

read(fd, frame, 40); // assumes 40 — this is the single most common serial bug in robotics. The fix is never "read again until I feel better"; it is a loop that accumulates until it has a full frame, handling 0 (EOF) and -1 (error/EINTR) explicitly. We build exactly that loop in the §8 exercise.

Two error codes you must treat as retry, not failure: EINTR means a signal interrupted the call before any data moved — just call again. EAGAIN/EWOULDBLOCK means a non-blocking fd had nothing ready — come back later (the foundation of epoll in Ch 08). Everything else is a real error.

5"Everything is a file" & the VFS

① Intuition one API, many objects

The reason the same six syscalls work on a text file, a serial port, a network connection, and a hardware sensor is the Virtual File System (VFS). The VFS is a kernel-internal abstraction layer: every open object exposes a table of function pointers (read, write, llseek, poll, …), and when you call read(fd, …), the VFS looks up the object behind that fd and dispatches to its read. Your code is identical; the kernel routes it to the tty driver, the socket layer, or the filesystem.

"Everything is a file" really means "everything presents the file interface." But the objects behind fds are genuinely different, and the differences leak through in which syscalls succeed and how data behaves. You will meet each of these in later chapters; know the cast now:

🏭 Real case — the IMU is a character device

On the AGX Orin USV the IMU hangs off a UART exposed as /dev/ttyTHS1 (the Tegra high-speed serial driver). The c in its ls -l flag tells you everything: it is a byte stream, not a file. There is no offset, lseek() fails with ESPIPE, and bytes you don't read are buffered by the tty layer until they overflow and get silently dropped. That stream-vs-file distinction is why partial reads are the norm here, not the exception.

6Buffered stdio vs raw fds — and why mixing bites

① Intuition two layers, one device

There are two ways to read the same fd, and confusing them is a real source of pain. The raw way is the syscall layer: an int fd and direct read/write. The convenient way is buffered stdio: a FILE* from fopen, with fread/fgets/fscanf. The FILE* wraps an fd plus a user-space buffer (typically 4 KB). When you ask fgets for one line, stdio may do a single big read() of 4 KB behind your back, then hand you lines out of that buffer with no further syscalls. Fewer mode switches — great for log files, exactly what §7 is about.

④ Real case where it bites

The trap is that stdio's buffer is invisible. Open /dev/ttyTHS1 with fopen, do one fgetc, and stdio may have already pulled 4 KB out of the kernel into its private buffer. Now you decide you want low latency and call raw read() on the same underlying fd: the bytes stdio buffered are gone from the kernel and invisible to your read. You read stale or missing data and conclude the sensor is flaky. The rule below is absolute.

⚠️ Never mix FILE* and raw read()/write() on the same fd

Pick one layer per fd. For a real-time sensor stream, use raw fds — you need control over exactly when and how much you read, and you cannot afford a hidden 4 KB buffer adding latency or eating bytes. Reserve FILE* for line-oriented config and log files. And remember stdio's write buffer too: printf to a pipe is line-buffered, but to a file it is fully buffered, so output can sit unflushed until fflush or exit — your last log line before a crash may never hit disk.

7Throughput vs syscall overhead: the amortization math

② Math derive the effective rate

Now make the cost concrete. Say each read() costs a fixed overhead $t_o$ (the mode switch plus kernel bookkeeping) regardless of size, plus a per-byte transfer cost $t_b$. To move a total of $N$ bytes in chunks of size $B$, you make $N/B$ calls, so total time is:

$$ t_\text{total} = \frac{N}{B}\,t_o + N\,t_b $$

The effective throughput — bytes moved per second of real work — is $N / t_\text{total}$, which simplifies to a clean expression in the buffer size $B$:

$$ R_\text{eff}(B) = \frac{N}{t_\text{total}} = \frac{1}{\dfrac{t_o}{B} + t_b} $$

Read it the way it wants to be read. As $B$ grows, the $t_o/B$ term shrinks toward zero and $R_\text{eff}$ rises toward the hardware ceiling $1/t_b$. The overhead is amortized over more bytes per call. With a tiny buffer the $t_o/B$ term dominates and you spend almost all your time crossing the boundary, not moving data. Concretely, take $t_o = 0.5\ \mu\text{s}$ per call:

Buffer $B$Calls for 1 MBOverhead timeOverhead-limited rate $\approx B/t_o$
1 byte1,048,576524 ms2 MB/s
64 bytes16,3848.2 ms128 MB/s
4 KB256128 µs8 GB/s
64 KB168 µs128 GB/s
∑ Why the curve flattens

The gain from 1 byte → 4 KB is enormous; from 4 KB → 64 KB it is nearly nothing, because by 4 KB the per-byte cost $t_b$ already dominates and there is no more overhead left to amortize. This is the law of diminishing returns made arithmetic: pick a buffer big enough that $t_o/B \ll t_b$, then stop. Bigger buffers past that point just waste memory and add latency — which on a real-time loop is a cost of its own.

8The robust read loop, in C and Python

③ Code read every line

Here is the function that should be reflexive by the end of this course: read exactly n bytes, looping over partial reads, retrying on EINTR, and reporting EOF honestly. This is the correct way to pull a fixed-size frame off a serial fd.

read_full.c C
#include <unistd.h>
#include <errno.h>
#include <stddef.h>

// Read exactly n bytes into buf. Returns n on success, a smaller
// count on EOF before n, or -1 on a real error (errno set).
ssize_t read_full(int fd, void *buf, size_t n) {
    size_t got = 0;
    char  *p   = buf;
    while (got < n) {
        ssize_t r = read(fd, p + got, n - got);
        if (r < 0) {
            if (errno == EINTR) continue;   // signal: just retry
            return -1;                       // genuine error
        }
        if (r == 0) break;                   // EOF: peer closed / stream ended
        got += (size_t)r;                    // accumulate the partial read
    }
    return (ssize_t)got;
}

The three branches are the whole point. r < 0 with EINTR is not an error — a signal landed, retry. r == 0 is EOF — the other end is gone, stop. r > 0 but less than requested is the partial read — accumulate and loop. Miss any one and you have a bug that only shows up under load, which is to say, on the water.

Python gives you the same raw layer through os.read, which is a thin wrapper over the syscall and obeys the identical count contract. Avoid the buffered open()/file.read() on a live sensor for the §6 reasons.

read_full.py Python
import os, errno

def read_full(fd: int, n: int) -> bytes:
    """Read exactly n bytes from a raw fd, handling partial reads + EINTR."""
    chunks, got = [], 0
    while got < n:
        try:
            data = os.read(fd, n - got)      # raw syscall, returns a COUNT
        except InterruptedError:             # == EINTR; retry
            continue
        if not data:                         # b'' means EOF
            break
        chunks.append(data)
        got += len(data)
    return b"".join(chunks)

# Open the IMU UART as a raw fd — no FILE*, no hidden buffer
fd = os.open("/dev/ttyTHS1", os.O_RDONLY | os.O_NOCTTY)
frame = read_full(fd, 40)                     # exactly one 40-byte IMU frame
🛠️ One more layer to come

Since Python 3.5, os.read already retries EINTR internally (PEP 475), so the except branch above is belt-and-suspenders — but writing it by hand makes the contract explicit, and in C you must do it yourself. The frame boundary problem (where does one IMU message end and the next begin?) is a layer up: that is parsing, which sits on top of this loop. Get the bytes right first.

9Real case: an IMU at 200 Hz over serial

④ Real case this runs on the USV

The AGX Orin USV needs attitude at 200 Hz for its control loop. The IMU streams 40-byte frames over /dev/ttyTHS1 at 921600 baud. Let's check the numbers, because they decide your whole I/O strategy.

Payload rate: $200\ \text{Hz} \times 40\ \text{bytes} = 8000$ bytes/s. At 8N1 framing each byte is 10 bits on the wire, so the line carries $80\,000$ bits/s — a rounding error against the 921600 baud link. Bandwidth is never the problem here. The problem is cadence and overhead. A new frame is fully on the wire every 5 ms. If you naively read() one byte at a time you make $8000$ syscalls/s; at $t_o = 0.5\ \mu\text{s}$ that is 4 ms/s of pure mode-switch tax — small, but pure waste, and it scales badly once you add GPS, LiDAR, and CAN to the same loop.

🏭 Real case — the read budget for one frame

Read in frame-sized chunks with read_full(fd, 40), and expect it to loop two or three times per frame because bytes dribble in at the baud rate (≈11 µs/byte at 921600). The control loop calls read_full once per cycle; if a full frame isn't there yet it gets a short return and either waits on poll() (Ch 08) or uses the previous estimate. The tty driver buffers incoming bytes, so a 5 ms control jitter never loses data — but only because you sized the loop to the device, not the other way around. The right way to wait for bytes (instead of byte-by-byte polling) is the termios VMIN/VTIME settings in Ch 06; for now, the takeaway is: chunk to the frame, never to the byte.

10Your turn

Q1 Conceptual check
You call read(fd, buf, 64) on a serial fd and it returns 17. What does that mean?
  • An error occurred; you should check errno and abort.
  • The device is broken — it should always return 64.
  • 17 bytes were available and copied to your buffer right now; this is a normal partial read, so loop to get the rest.
  • End of file was reached after 17 bytes.
read returns a count of bytes actually moved, anywhere from 0 to the requested size. A positive value less than requested is a partial read — completely normal on streams like serial ports and sockets, where bytes arrive at the line rate. Only 0 means EOF and only -1 means error. The correct response is to accumulate and call read again (handling EINTR), exactly as read_full does.
Q2 Compute it
Each read() costs a fixed overhead $t_o = 1\ \mu\text{s}$ and moves data with negligible per-byte cost. You read 1,000,000 bytes 500 bytes per call (buffer size $B = 500$ bytes). In the overhead-limited regime $R_\text{eff} \approx B/t_o$, what is the effective throughput in MB/s (use 1 MB $= 10^6$ bytes)?
MB/s
With $B = 500$ bytes per call and $t_o = 1\ \mu\text{s}$: $R_\text{eff} = B/t_o = 500\ \text{bytes} / 10^{-6}\ \text{s} = 5\times10^8$ bytes/s $= 500$ MB/s. For contrast, reading one byte per call gives $R_\text{eff} = 1 / 10^{-6} = 10^6$ bytes/s $= 1$ MB/s — five hundred times slower. The lesson is the slope: every factor you grow the buffer multiplies the overhead-limited rate by the same factor, until $t_b$ takes over. (Answer: 500 MB/s at $B=500$.)
Q3 Coding challenge — a robust read-all loop
Write a C function read_full(int fd, void *buf, size_t n) that reads exactly n bytes, correctly handling (a) partial reads, (b) EINTR, and (c) EOF before n bytes arrive. State what each of the three read return cases means and how your code reacts.
Hint
read can return three kinds of value: r > 0 (some bytes — maybe fewer than asked), r == 0 (EOF), r < 0 (error; check errno == EINTR for "retry"). Keep a running got counter, advance the buffer pointer by got, and loop while got < n.
Show one solution
read_full.c C
#include <unistd.h>
#include <errno.h>
#include <stddef.h>

ssize_t read_full(int fd, void *buf, size_t n) {
    size_t got = 0;
    char  *p   = buf;
    while (got < n) {
        ssize_t r = read(fd, p + got, n - got);
        if (r < 0) {
            if (errno == EINTR) continue;   // (b) signal hit us; retry, no data lost
            return -1;                       //     real error: propagate, errno set
        }
        if (r == 0) break;                   // (c) EOF before n bytes; return short
        got += (size_t)r;                    // (a) partial read; accumulate and loop
    }
    return (ssize_t)got;                      // == n on full success, < n on EOF
}

The three cases: r > 0 is the partial read — copy what you got, advance, and keep going until you have all n. r == 0 is EOF — the stream ended (peer closed, USB unplugged), so return the short count and let the caller decide. r < 0 with errno == EINTR is a benign interruption (a signal delivered before any byte moved) — just retry; any other errno is a genuine failure. The caller checks the return: == n means a complete frame, < n means EOF, -1 means error. For a non-blocking fd you would additionally treat EAGAIN as "nothing ready right now" and return up to the event loop — the path into Ch 08.

🔑 Chapter recap

I/O is a negotiation with the kernel across the syscall boundary: app → libc → trap (mode switch) → VFS → driver → device. A file descriptor is a small integer handle into your process's fd table; 0/1/2 are stdin/out/err and open() hands back the lowest free fd. The core six are open/read/write/close/lseek/ioctl, and the load-bearing fact is that read/write return a count — partial reads are normal, EOF is 0, errors are −1, and EINTR means retry. The VFS makes one API work on files, ttys, sockets, and devices, but never mix buffered FILE* with raw fds on the same stream. Each syscall carries fixed overhead $t_o$, so effective rate $R_\text{eff} = 1/(t_o/B + t_b)$ rewards bigger buffers until $t_b$ dominates — chunk to the frame, never to the byte.
