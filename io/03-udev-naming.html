udev & Persistent Naming: making /dev/gps mean GPS, forever

You wired a GPS and an IMU into the USV's USB hub, opened /dev/ttyUSB0 for the GPS, shipped it, and a week later the boat steered toward a buoy because a cold-boot race handed ttyUSB0 to the IMU instead. This chapter is how you make a device's name follow the device, not the order the kernel happened to probe it.

1The renumbering problem

① Intuition why the number is a lie

Kernel device nodes like ttyUSB0, ttyUSB1, video0, sda are numbered in the order the kernel enumerates them, which is the order their drivers finish probing. That order is not stable. It depends on USB hub topology, on which device powers up and trains its link first, on driver load order, and on whether you replugged something. Two identical FTDI adapters can swap numbers between two boots of the same machine. The index is an arrival ticket, not an identity.

⚠️ Pitfall — the kernel name is a race result

Opening /dev/ttyUSB0 by name is opening "whichever serial device probed first this boot." On a robot with two or more serial dongles, that is a coin flip you re-run on every power cycle. Never hard-code a kernel-assigned index in production.

② Math how bad does it get

Quantify the gamble. If you have $n$ otherwise-indistinguishable serial devices on the bus, the kernel can assign the indices ttyUSB0..ttyUSB(n-1) in any one-to-one mapping. The number of distinct orderings is the number of permutations of $n$ objects:

$$ P(n) = n! = n \times (n-1) \times \cdots \times 2 \times 1 $$

Only one of those orderings is the one your code assumes, so the probability that a random boot hands you the layout you hard-coded is $1/n!$. The factorial bites fast: with two devices there are $2! = 2$ orderings (a 50% chance of being wrong), with three there are $3! = 6$, with four $4! = 24$, with five $5! = 120$. A USV carrying GPS, IMU, a USB-to-CAN adapter and a USB modem already lives in a 24-way lottery every time it boots.

🔑 The whole chapter in one line

Stop naming devices by arrival index. Name them by identity — vendor, product, serial number, or physical port — and let a userspace daemon, udev, create a stable alias the moment the device appears.

2From kernel uevent to a node in /dev

① Intuition who actually makes the node

When a device appears, the kernel does not decide what to call it in /dev. The kernel creates the device internally, registers it in the device model (the /sys tree from Chapter 02), and then fires a small message called a uevent — a few lines of KEY=value text — up to userspace over a netlink socket (NETLINK_KOBJECT_UEVENT). A daemon, systemd-udevd, listens on that socket, matches the event against a pile of rules, and is the thing that actually applies ownership, mode, and your symlinks.

💡 Mechanism vs. policy

The kernel supplies the mechanism (the device exists, here are its attributes). udev supplies the policy (call it this, own it by that group, symlink it there). That split is why naming is configurable at all: it lives in userspace text files you control, not in the kernel.

Note that devtmpfs (Chapter 02) already gives you a bare node like /dev/ttyUSB0 very early — the kernel pre-populates it so the system isn't node-less before udev runs. udev then comes along to relabel / chmod / symlink it according to policy. Both are involved: devtmpfs makes the raw node, udev applies the rules.

3Rule anatomy: match keys vs. assignments

① Intuition a rule is "if all this, then do that"

A udev rule is a single line: a comma-separated list of match keys (conditions) followed by assignment keys (actions). udev reads every rule for every event; if all the match keys on a line are true, it runs that line's assignments. The single most common beginner mistake is confusing the two kinds of =, so burn this in:

OperatorKindMeaning
==match (test)"this key equals this value?" — a condition
!=match (test)"this key is not this value?"
=assignset the value (replaces any prior list)
+=assignappend to a list-valued key (e.g. add a symlink)
:=assignset and lock — later rules can't change it
⚠️ Pitfall — = where you meant ==

Writing SUBSYSTEM="tty" (single =) doesn't test the subsystem — it tries to assign to a read-only match key, and your rule silently does the wrong thing or nothing. Conditions are always == / !=; actions are = / += / :=. Get this one backwards and you will lose an afternoon.

② The vocabulary the keys you'll actually use

Match keys you'll lean on: SUBSYSTEM (e.g. tty, net, block), KERNEL (the kernel name, supports globs like ttyUSB*), ATTRS{idVendor} (a sysfs attribute on the device or any parent — this is how you reach up to the USB device for idVendor), and ENV{ID_SERIAL} (a property set by an earlier rule, like ID_SERIAL). Assignment keys: NAME (network interfaces — renames the device itself), SYMLINK+= (add a stable alias in /dev), MODE, OWNER, GROUP, and RUN+=.

🛠️ ATTR vs. ATTRS (one letter, big difference)

ATTR{idVendor} matches a sysfs attribute on the current device only. ATTRS{idVendor} (with the S, for "search") walks up the parent chain until it finds a device with that attribute. A ttyUSB node has no idVendor of its own — that lives on the USB device several levels up — so you almost always want ATTRS{idVendor}, not ATTR{idVendor}.

4Finding the match keys with udevadm info

③ Code interrogate the device first

Never guess attributes — ask. udevadm info --attribute-walk (short -a) starts at the device node and walks up the parent chain, printing every ATTRS{idVendor}-style key you can match at each level. This is the single command that turns "I have a dongle" into "I have a rule."

walk the device tree for the GPS bash
udevadm info --attribute-walk --name=/dev/ttyUSB0
# or the short form:
udevadm info -a -n /dev/ttyUSB0

The output is a stack of blocks, the device itself first, then each parent. Read it from the top down and grab the keys from the parent block that has the USB identifiers:

trimmed udevadm output bash
  looking at device '/devices/3610000.usb/usb1/1-2/1-2.1/1-2.1:1.0/ttyUSB0/tty/ttyUSB0':
    KERNEL=="ttyUSB0"
    SUBSYSTEM=="tty"

  looking at parent device '/devices/3610000.usb/usb1/1-2/1-2.1/1-2.1:1.0/ttyUSB0':
    KERNELS=="ttyUSB0"
    SUBSYSTEMS=="usb-serial"

  looking at parent device '/devices/3610000.usb/usb1/1-2/1-2.1':   # <-- the USB device
    KERNELS=="1-2.1"
    SUBSYSTEMS=="usb"
    ATTRS{idVendor}=="067b"
    ATTRS{idProduct}=="2303"
    ATTRS{serial}=="0001"
    ATTRS{product}=="USB-Serial Controller"
🔑 What to harvest

You want idVendor + idProduct to pick the model, and serial to pick the individual unit. Two different-model dongles? Vendor+product is enough. Two identical dongles (same vendor/product)? You must use serial to tell them apart — or fall back to physical port (§7).

⚠️ Don't mix attributes from different parent blocks

All the ATTRS{idVendor}-style keys you put in one rule must come from a single parent block. udev matches them against one device on the chain — it won't stitch idVendor from one parent to serial from another. Take them all from the SUBSYSTEMS=="usb" block.

5Writing the stable-symlink rule

③ Code the rule itself

Rules live in /etc/udev/rules.d/ (your overrides) and /lib/udev/rules.d/ (the distro's). Files are read in lexical order by filename — that's why everyone prefixes with a number, e.g. 99-usv.rules, to run late and win. Here is the rule that pins the GPS:

/etc/udev/rules.d/99-usv.rules bash
# GPS: PL2303 dongle, serial 0001 -> /dev/gps
SUBSYSTEM=="tty", ATTRS{idVendor}=="067b", ATTRS{idProduct}=="2303", \
  ATTRS{serial}=="0001", SYMLINK+="gps", GROUP="dialout", MODE="0660"

# IMU: identical-model dongle, different serial 0002 -> /dev/imu
SUBSYSTEM=="tty", ATTRS{idVendor}=="067b", ATTRS{idProduct}=="2303", \
  ATTRS{serial}=="0002", SYMLINK+="imu", GROUP="dialout", MODE="0660"

Read it as a sentence: "if the subsystem is tty and the parent USB vendor is 067b and product is 2303 and serial is 0001, then add a symlink /dev/gps pointing at this node, set the group to dialout, and the mode to 0660." Every == is a condition; every +=/= is an action. The backslash just continues the line for readability — a real rule is one logical line.

💡 Why a symlink, not a rename

SYMLINK+= adds /dev/gps as an alias while /dev/ttyUSB0 still exists — nothing else that relied on the kernel name breaks. Renaming the node outright (changing NAME) is reserved for network interfaces (§7); for char/block devices you symlink. And += (not =) means you can add several symlinks to the same device without clobbering ones other rules added.

② The math of permissions what 0660 means

The MODE is an octal permission triple — owner, group, other — each digit a sum of read (4), write (2), execute (1). So 0660 is read+write for owner ($4+2=6$), read+write for group ($6$), nothing for other ($0$): $\,rw\text{-}\,rw\text{-}\,---$. Paired with GROUP="dialout", any process whose user is in the dialout group can open the serial port without root. That's exactly what you want for a robot service that should not run as root.

6Testing: udevadm test and trigger

③ Code don't reboot to find out

After editing rules, reload them, then either dry-run a single device or re-fire events for a whole subsystem. Two distinct commands, two distinct jobs:

CommandWhat it does
udevadm control --reloadreload rules from disk (do this after every edit)
udevadm test /sys/class/tty/ttyUSB0dry-run: simulate the event, print which rules matched and what they'd do — makes no real changes
udevadm trigger --action=addre-emit add uevents for existing devices, so rules apply now without replugging
the edit → test → apply loop bash
sudo nano /etc/udev/rules.d/99-usv.rules     # edit
sudo udevadm control --reload                # reload rules

# dry-run against the real device path under /sys (no changes made):
sudo udevadm test /sys/class/tty/ttyUSB0 2>&1 | grep -E 'SYMLINK|GROUP|MODE'

# apply for real to currently-attached tty devices:
sudo udevadm trigger --subsystem-match=tty --action=add
ls -l /dev/gps /dev/imu                       # verify the symlinks exist
🛠️ test simulates, trigger commits

udevadm test tells you what would happen and is safe to run all day. udevadm trigger actually replays the events and applies the rules. The classic workflow is: reload, test until the output shows your SYMLINK 'gps' line, then trigger to make it real — no reboot, no unplugging.

7by-id, by-path & predictable net names

① Intuition you may not even need a rule

The default udev rules already build two stable directories for serial devices, so for a quick fix you can skip writing rules entirely:

  • /dev/serial/by-id/ — keyed on vendor+product+serial. Stable across reboots and across ports. Use this when each device has a unique serial.
  • /dev/serial/by-path/ — keyed on the physical USB topology (which hub, which port). Stable as long as you don't move the cable. Use this for identical devices with no usable serial — the port becomes the identity.
the symlinks udev already made for you bash
ls -l /dev/serial/by-id/
# usb-Prolific_USB-Serial_Controller_0001-if00-port0 -> ../../ttyUSB0

ls -l /dev/serial/by-path/
# platform-3610000.usb-usb-0:2.1:1.0-port0 -> ../../ttyUSB0
🛠️ by-id for unique serials, by-path for "same model in known ports"

Two GPS units with distinct serials → by-id (or a serial-keyed rule). Two identical sensors that you always plug into the same two ports → by-path (or an ATTRS{devpath} rule), because the port is the only thing that differs.

② Predictable network interfaces same problem, the .link answer

NICs have the identical race: the kernel's old scheme named them eth0, eth1 in probe order, so a USB-Ethernet adapter could steal eth0 from the onboard port. systemd's fix is Predictable Network Interface Names — names derived from the hardware location, like enp1s0 (PCI bus 1, slot 0) or enx + MAC. They're ugly but deterministic. To pin a chosen name, write a systemd .link file that matches on the MAC and renames the interface:

/etc/systemd/network/10-sensor-net.link bash
[Match]
MACAddress=00:04:4b:aa:bb:cc

[Link]
Name=eth_lidar

A .link file is processed by systemd-udevd very early (during the net device's add event), before the interface is brought up, which is why renaming is safe there — unlike a late udev NAME= rule, which the kernel may reject once the interface is in use. Stick to the .link mechanism for renaming NICs.

🏭 Real case — the Jetson AGX Orin USV bring-up

On the USV's AGX Orin we ship 99-usv.rules pinning the u-blox GPS to /dev/gps (serial-keyed) and the VectorNav IMU to /dev/imu, both GROUP="dialout", MODE="0660" so the ROS 2 driver runs as the unprivileged usv user. The onboard Ethernet port that carries the LiDAR is pinned to eth_lidar via a MAC-matched .link file, so a plugged-in USB-Ethernet dongle (for the field laptop) can never steal it. Result: every launch file opens /dev/gps, /dev/imu, and binds the LiDAR socket to eth_lidar — names that survive reboots, replugs, and added hardware.

8RUN, MODE, GROUP: ownership and side effects

④ Real case making the device usable, and reactive

A pinned name is half the battle; the process also has to be allowed to open it, and sometimes something should happen when it appears. That's what the remaining assignment keys do:

KeyEffect
OWNER="usv"set the node's owning user
GROUP="dialout"set the owning group (members get group access)
MODE="0660"permission bits (rw for owner+group, none for other)
RUN+="program"run a short program after the device is set up
TAG+="systemd"let a systemd unit be BindsTo= this device
⚠️ Pitfall — RUN is not a place to start daemons

RUN+= runs synchronously inside the udev event and is killed if it doesn't return fast. It is for quick setup — set FTDI latency_timer, bring up a CAN link, drop a flag file — never for launching a long-running service. To start a service on plug-in, tag the device with systemd and use a BindsTo=/WantedBy= unit. Blocking in RUN stalls all of udev.

RUN for a quick, bounded setup task bash
# When the FTDI GPS appears, drop its read latency from 16ms to 1ms.
SUBSYSTEM=="tty", ATTRS{idVendor}=="0403", ATTRS{idProduct}=="6001", \
  ATTRS{serial}=="A50285BI", SYMLINK+="gps", \
  RUN+="/bin/sh -c 'echo 1 > /sys%p/device/latency_timer'"

Here %p expands to the kernel device path under /sys, so the rule writes the FTDI's latency_timer attribute (from Chapter 02) the instant the device is recognized — a one-shot, bounded write, exactly what RUN is for.

9Your turn

Q1 Conceptual check
In a udev rule, which operator tests a condition (a match key) rather than performing an action?
  • = — it assigns the value to compare against.
  • == — match keys test equality; =/+=/:= assign.
  • += — it appends the condition to a list.
  • := — it locks the condition so it can't change.
Match keys use == (or !=) and are conditions — "is this attribute equal to that value?". Assignment keys use = (set), += (append, as in SYMLINK+=), or := (set and lock). Writing SUBSYSTEM="tty" with a single = is the classic bug: it tries to assign to a read-only match key instead of testing it.
Q2 Compute it
The USV has 5 indistinguishable USB-serial devices on one hub. In how many distinct orderings can the kernel assign ttyUSB0..ttyUSB4? (Hint: $n!$.)
orderings
$5! = 5 \times 4 \times 3 \times 2 \times 1 = 120$. Only one of those 120 is the layout your code assumes, so a blind boot has a $1/120 \approx 0.83\%$ chance of being right. Serial-keyed udev symlinks make the chance 100%.

A u-blox GPS reports ATTRS{idVendor}=="1546", ATTRS{idProduct}=="01a8", ATTRS{serial}=="DBE1F2C0" on the tty subsystem. Write a single udev rule that creates /dev/gps, owned by group dialout with mode 0660. Then give the two commands to apply it without rebooting.
Hint
Conditions use ==; the symlink uses SYMLINK+=. Reload the rules, then re-fire the add event for the tty subsystem.
Show one solution

Every == is a condition on the parent USB device (so it must be ATTRS, not ATTR); SYMLINK+= adds the alias without disturbing the kernel name; GROUP+MODE let your non-root service open the port. --reload picks up the file, trigger replays the event so the link appears immediately.

🔑 Chapter recap

Kernel indices like ttyUSB0 are arrival tickets — with $n$ devices there are $n!$ possible orderings and only one is yours. The kernel emits a uevent; systemd-udevd matches it against rules and applies policy. A rule is match keys (==/!=) then assignments (=/+=/:=); find the keys with udevadm info -a -n /dev/ttyUSB0, key on ATTRS{idVendor/idProduct/serial}, add SYMLINK+="gps" with GROUP/MODE, then udevadm test (dry-run) and trigger (apply) — no reboot. For free, use /dev/serial/by-id and by-path; for NICs, pin names with a systemd .link file. Next: drivers, modules, and firmware — what actually binds to the device behind that name.
