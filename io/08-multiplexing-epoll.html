Multiplexing I/O: how one thread watches every sensor

A robot's brain reads GPS, IMU, LiDAR and CAN at the same time — four file descriptors, four data rates, all live at once. The naive answer is "one thread per device." It works on a laptop and quietly falls apart at scale. This chapter is the kernel machinery that lets a single thread wait on hundreds of fds and wake only for the ones with data: select, poll, epoll, and io_uring.

1Blocking, non-blocking, and the wait problem

① Intuition build the picture first

Every read() you've written so far blocks: if the GPS hasn't sent its next NMEA line yet, the calling thread goes to sleep inside the kernel until bytes arrive. That's perfect when you have exactly one thing to watch. The instant you have two, blocking traps you — while you sleep on the GPS, the LiDAR's socket buffer overflows and you drop a scan. You can only block on one fd at a time, but a robot has many.

The first escape is non-blocking mode. Open the fd with the O_NONBLOCK flag (or set it later with fcntl), and a read() that has no data returns immediately with -1 and errno == EAGAIN (identical to EWOULDBLOCK on Linux) instead of sleeping. Now you can visit every fd in a loop without getting stuck on one — but watch what that costs.

⚠️ Busy-polling burns a whole core

A bare while(1) that read()s every non-blocking fd and ignores EAGAIN never sleeps. It spins at 100% CPU asking "anything yet? anything yet?" millions of times a second. EAGAIN is not an error — it means "nothing right now, ask later." The whole game of multiplexing is: who tells us when to ask, so we can sleep until then?

② Math why a busy loop is hopeless

Say your control loop runs at 20 Hz, so you only need fresh data every $50\ \text{ms}$. A busy poll loop doing nothing but read()+EAGAIN at, say, $2\times10^{6}$ iterations per second does almost nothing useful:

$$ \eta = \frac{f_{\text{useful}}}{f_{\text{poll}}} \approx \frac{80\ \text{reads with data/s}}{2\times10^{6}\ \text{reads/s}} \approx 0.004\% $$

Every one of those checks is a syscall (user→kernel→user). We are paying for two million crossings to catch eighty payloads. We need a primitive that blocks until at least one fd is ready, then hands us the ready set. That primitive is I/O multiplexing.

nonblock.c C
#include <fcntl.h>
#include <errno.h>
#include <unistd.h>

int set_nonblocking(int fd) {
    int fl = fcntl(fd, F_GETFL, 0);             /* read current flags */
    if (fl == -1) return -1;
    return fcntl(fd, F_SETFL, fl | O_NONBLOCK); /* add O_NONBLOCK     */
}

/* a read that tolerates "no data yet" */
ssize_t try_read(int fd, void *buf, size_t n) {
    ssize_t r = read(fd, buf, n);
    if (r < 0 && (errno == EAGAIN || errno == EWOULDBLOCK))
        return 0;            /* not an error: ask again later */
    return r;                /* >0 bytes, 0 = EOF, <0 real error */
}

2Thread-per-fd and why it doesn't scale

① Intuition the obvious answer

"Blocking is fine if each fd gets its own thread." Spin up one thread per device, let each block on its own read(), done. This is genuinely the right call for a handful of fds — for a robot with four sensors it's a perfectly reasonable architecture. The trouble is it doesn't generalize, and understanding exactly where it breaks is what motivates everything else.

② Math two costs that grow with N

Memory. Each pthread reserves a stack — the Linux default is $8\ \text{MiB}$ of virtual address space (ulimit -s shows it in KiB). It's lazily faulted in, so it doesn't all become RAM, but the virtual reservation plus the per-thread kernel task_struct (a few KiB of real RAM each) are unavoidable. For $N$ threads:

$$ M_{\text{vm}} = N \cdot S_{\text{stack}} = N \cdot 8\ \text{MiB} \qquad\Rightarrow\qquad N=10{,}000 \;\Rightarrow\; \approx 78\ \text{GiB of VM} $$

Scheduler cost. Worse than memory is the context-switch tax. When one thread blocks and another wakes, the kernel saves and restores registers, swaps the page-table root, and — the expensive part — pollutes the L1/L2 cache and TLB so the new thread runs cold. A context switch is roughly $1$–$5\ \mu s$ of direct cost plus cache-miss aftershock. With $N$ threads each waking on its own traffic, switch rate scales with $N$:

$$ C_{\text{cpu}} \approx N \cdot f_{\text{events}} \cdot t_{\text{switch}} $$

This is the famous C10k problem: a server that wanted to handle 10,000 concurrent connections couldn't, because thread-per-connection drowned the scheduler long before it ran out of RAM. The fix wasn't more threads — it was one thread that the kernel could tell which fds are ready.

🏭 Real case — the AGX Orin USV has a dozen fds, so why care?

Our USV has maybe a dozen I/O sources, so thread-per-fd would run. But the same scaling logic shows up in miniature: each blocking-read thread that wakes 200×/s for the IMU is a context switch that evicts the cache lines your 20 Hz PID loop just warmed. Twelve such threads turn a deterministic control loop into a jittery one. The event-loop pattern below lets the control thread own its cache and wake only when real data lands.

3select & poll: the O(n) ancestors

① Intuition hand the kernel the whole list

The first multiplexers solved the wait problem with one idea: hand the kernel your entire set of fds and say "sleep until any of these is readable/writable, or a timeout fires." When it returns, you scan the set to see which ones woke. That's select(2) (1983, fixed-size bitmap capped at FD_SETSIZE = 1024) and its cleaner successor poll(2) (an array of struct pollfd, no hard limit).

② Math where the cost hides

Both share a fatal property: the interest set lives in userspace and is rebuilt every call. So each invocation does $O(n)$ work twice over — the kernel copies and scans all $n$ descriptors on entry, and you scan all $n$ again on return to find the ready ones, even if only one fd actually fired:

$$ T_{\text{select}} = \underbrace{O(n)}_{\text{copy in}} + \underbrace{O(n)}_{\text{kernel scan}} + \underbrace{O(n)}_{\text{user scan}} = O(n)\ \text{per call} $$

With $n=10{,}000$ fds where only $1$ is ready, you do $\sim10{,}000$ units of work to deliver $1$ event — a $0.01\%$ payoff that repeats on every loop iteration. The cost is tied to the size of the watched set, not the amount of actual I/O. That is precisely backwards from what you want.

poll_loop.c C
#include <poll.h>

struct pollfd pfds[4];
pfds[0] = (struct pollfd){ .fd = gps_fd,   .events = POLLIN };
pfds[1] = (struct pollfd){ .fd = imu_fd,   .events = POLLIN };
pfds[2] = (struct pollfd){ .fd = lidar_fd, .events = POLLIN };
pfds[3] = (struct pollfd){ .fd = can_fd,   .events = POLLIN };

for (;;) {
    int ready = poll(pfds, 4, 50 /* ms timeout */);
    if (ready < 0) { if (errno == EINTR) continue; break; }
    if (ready == 0) { /* 50 ms passed: run the control tick */ continue; }
    for (int i = 0; i < 4; i++)                 /* O(n) rescan EVERY call */
        if (pfds[i].revents & POLLIN)
            drain(pfds[i].fd);                  /* read until EAGAIN */
}

For four fds this is fine — at $n=4$ the $O(n)$ rescan is free, and poll is the right, portable tool. The lesson is about the asymptote, not this loop.

4epoll: the kernel keeps the set, O(ready)

① Intuition stop re-handing the same list

The waste in poll is that you re-declare your interest set on every single call. Linux's epoll (2002) fixes that by making the interest set a persistent kernel object. You build it once with epoll_ctl(... EPOLL_CTL_ADD ...); thereafter epoll_wait() returns only the fds that are actually ready. No userspace set to copy in, no full rescan. The kernel hooks each fd's wake-up path, so when data arrives it appends that fd to a ready-list internally.

② Math the cost moves to the work

Registration is amortized $O(1)$ per fd (the interest set is a red-black tree keyed by fd). The hot path — epoll_wait — costs only the number of ready fds, $r$, not the total $n$:

$$ T_{\text{epoll\_wait}} = O(r),\qquad r = \#\{\text{ready fds}\} \;\le\; n $$

When $1$ of $10{,}000$ fds is ready, epoll_wait does $O(1)$ work and returns one event — versus poll's $O(10{,}000)$. The cost finally scales with the I/O you actually have, which is the entire point. Pull the slider in the sim below to watch the red (thread-per-fd) and orange (epoll) curves diverge as $N$ grows.

🔑 What to notice

Push the fd count right and the red curve climbs — thread-per-fd memory (8 MiB stack each) and O(N) scheduler pressure grow linearly with every fd. The orange epoll curve tracks the active fraction, not the total: drop "% active" to 5% and epoll's work stays near the floor even at 50,000 fds, on a single thread with ~0.3 KB of kernel state per fd. The gap between the lines is the C10k problem, and epoll closing it is why every modern event loop is built on it.

③ Code build the interest set once
epoll_setup.c C
#include <sys/epoll.h>

int ep = epoll_create1(EPOLL_CLOEXEC);     /* the kernel-side interest set */

struct epoll_event ev = { .events = EPOLLIN, .data.fd = gps_fd };
epoll_ctl(ep, EPOLL_CTL_ADD, gps_fd, &ev); /* register ONCE, not per loop */

/* later, the wait returns only ready fds */
struct epoll_event events[64];
int n = epoll_wait(ep, events, 64, 50 /* ms */);
for (int i = 0; i < n; i++)                 /* i runs over READY fds only   */
    drain(events[i].data.fd);

5Level vs edge trigger: the bug everyone hits

① Intuition a state vs a transition

epoll has two notification modes, and choosing wrong is the single most common epoll bug. Level-triggered (LT) — the default — reports an fd ready as long as data is buffered, like a level sensor: every epoll_wait keeps telling you "still readable" until you drain it. Edge-triggered (ET), set with the EPOLLET flag, fires only on the transition from not-ready to ready — once per arrival. If you don't drain the whole buffer on an ET notification, the leftover bytes sit there and epoll_wait never mentions that fd again until new data arrives. You hang holding unread data.

🛠️ The ET rule: always drain to EAGAIN

With EPOLLET, the fd must be non-blocking, and on every wake you must read() in a loop until it returns EAGAIN. That EAGAIN is the signal "buffer empty, you've consumed the whole edge." Skip it and you'll silently drop data. Edge mode trades one syscall-per-event efficiency for that strict drain discipline.

Why bother with ET? Fewer wakeups. Under LT, a fast LiDAR that delivers a 4 KB scan in chunks can wake epoll_wait several times for one logical message; ET wakes you once and you drain it all. For most robotics nodes LT is simpler and plenty fast — reach for ET only when wakeup overhead is measurably hurting you.

edge_drain.c C
/* EPOLLET requires O_NONBLOCK fds + drain-to-EAGAIN */
struct epoll_event ev = { .events = EPOLLIN | EPOLLET, .data.fd = lidar_fd };
epoll_ctl(ep, EPOLL_CTL_ADD, lidar_fd, &ev);

/* on a ready event, loop until the kernel says "empty" */
for (;;) {
    ssize_t r = read(lidar_fd, buf, sizeof buf);
    if (r > 0)            { handle(buf, r); continue; }   /* keep draining */
    if (r == 0)           { close_peer(lidar_fd); break; } /* EOF          */
    if (errno == EAGAIN)  { break; }                       /* edge consumed */
    if (errno == EINTR)   { continue; }                    /* retry        */
    perror("read"); break;
}

6io_uring: from readiness to true async

① Intuition tell me when it's done, not when it's ready

epoll is a readiness interface: it tells you "this fd can be read now," and you still issue the read() yourself (another syscall). io_uring (Linux 5.1+) flips the model to completion: you submit the actual operation — "read 4 KB from this fd into this buffer" — and the kernel tells you when the data is already there. The work happens asynchronously; no separate read syscall per event.

The mechanism is two shared ring buffers in memory mapped between your process and the kernel — a submission queue (SQ) you push requests into and a completion queue (CQ) the kernel pushes results into. Because both rings are shared memory, you can batch many operations and submit them with one io_uring_enter() syscall — or, with SQ-polling enabled, zero syscalls in steady state.

② Math amortizing the syscall

If a syscall costs $t_{\text{sc}}$ and you batch $b$ operations per io_uring_enter, the per-operation syscall overhead collapses:

$$ t_{\text{per-op}} = \frac{t_{\text{sc}}}{b} \;\xrightarrow[\;b\to\infty\;]{}\; 0 \qquad\text{vs. epoll's } 2\,t_{\text{sc}}\text{ per event (wait + read)} $$

For our control loop the wins are modest — we're not syscall-bound at a dozen fds — but io_uring is the direction the kernel is moving, and it shines for disk-heavy mission logging where epoll has historically had no good answer (regular files are always "ready," so epoll can't async them; io_uring can).

🔑 Three generations, one trend

poll: you own the set, kernel rescans it — $O(n)$. epoll: kernel owns the set, returns ready fds — $O(r)$, but still readiness (you do the read). io_uring: kernel owns the set and does the I/O, returns completions in shared-memory rings — batched, true async. Each step moves work from your loop into the kernel and shrinks the syscall count.

7The event-loop pattern in a ROS2-style node

④ Real case one thread, four sensors, on the AGX Orin

Here is the shape every reactor — libev, libuv, the ROS 2 executor, an nginx worker — boils down to: register all your fds, then loop on epoll_wait dispatching ready ones to handlers, using the timeout as your periodic tick. On the USV this one thread owns the GPS (/dev/serial/by-id/...), the IMU (another tty), the LiDAR (a UDP socket), and the CAN bus (a SocketCAN fd via AF_CAN) — all multiplexed, all cache-warm, no thread thrash.

sensor_loop.c C
int ep = epoll_create1(EPOLL_CLOEXEC);

/* every fd is non-blocking; register each with its handler tag */
add_fd(ep, gps_fd,   EPOLLIN);   /* NMEA over /dev/serial/by-id/... */
add_fd(ep, imu_fd,   EPOLLIN);   /* 200 Hz binary over ttyTHS       */
add_fd(ep, lidar_fd, EPOLLIN);   /* UDP scans                       */
add_fd(ep, can_fd,   EPOLLIN);   /* SocketCAN AF_CAN                */

struct epoll_event evs[16];
const int TICK_MS = 50;                 /* 20 Hz control deadline */

for (;;) {
    int n = epoll_wait(ep, evs, 16, TICK_MS);
    if (n < 0 && errno == EINTR) continue;   /* signal: just retry */

    for (int i = 0; i < n; i++) {            /* O(ready) dispatch  */
        int fd = evs[i].data.fd;
        if (evs[i].events & (EPOLLERR | EPOLLHUP)) { reopen(fd); continue; }
        if (fd == gps_fd)        drain_nmea(fd);
        else if (fd == imu_fd)   drain_imu(fd);
        else if (fd == lidar_fd) drain_lidar(fd);
        else if (fd == can_fd)   drain_can(fd);
    }

    run_control_tick();   /* fires on every wake OR the 50 ms timeout */
}
⚠️ One slow handler stalls everyone

A single-thread event loop is cooperative: if drain_lidar blocks for 30 ms parsing a giant scan, the GPS, IMU and CAN all wait, and your 50 ms control tick slips. Handlers must be quick and non-blocking. Heavy CPU work (point-cloud processing, SLAM) belongs on a separate thread fed by a queue — keep the I/O loop lean so the control deadline is sacred.

8Your turn

Q1 Conceptual check
You register an fd with EPOLLIN | EPOLLET. A 4 KB message arrives; your handler reads only the first 1 KB, then returns to epoll_wait. No new data ever arrives on that fd. What happens to the remaining 3 KB?
  • The next epoll_wait reports the fd readable again, so you get the rest.
  • It sits unread — edge mode already fired for the arrival, so epoll_wait stays silent until new data comes in. You should have drained to EAGAIN.
  • The kernel discards it after the timeout.
  • epoll automatically retries the read for you.
Edge-triggered notification fires on the transition to ready, once. The 3 KB are still in the socket buffer, but because there's no new transition, epoll_wait won't mention the fd again. The ET contract is: read in a loop until EAGAIN on every wake. Level-triggered mode would have re-reported it, which is why LT is the safer default.
Q2 Compute it
You watch $n = 10{,}000$ fds, of which exactly $r = 1$ is ready on a given wake. A poll call does $O(n)$ work; an epoll_wait call does $O(r)$ work. Counting one "unit" per fd examined, how many more units does poll do than epoll_wait for this single wake? (Compute $n - r$.)
units
$n - r = 10{,}000 - 1 = 9{,}999$. poll scans all 10,000 descriptors to find the one that's ready; epoll_wait returns just that one and examines 1. The $\sim$10,000× gap repeats on every loop iteration — that's the asymptotic difference between $O(n)$ and $O(r)$, and the whole reason epoll exists.
Q3 Coding challenge — an epoll event-loop skeleton
Write a minimal level-triggered epoll loop in C: create the instance, register a single non-blocking fd for read events, then wait with a 100 ms timeout and drain the fd when it's ready. Handle EINTR and a zero-return timeout. (Assume set_nonblocking and handle exist.)
Hint
Three syscalls do all the work: epoll_create1 makes the instance, epoll_ctl(... EPOLL_CTL_ADD ...) registers the fd once, and epoll_wait blocks for ready events. A return of 0 means the timeout elapsed; a return of -1 with errno == EINTR means a signal interrupted it — just loop again.
Show one solution
loop.c C
#include <sys/epoll.h>
#include <errno.h>
#include <unistd.h>

int run_loop(int fd) {
    set_nonblocking(fd);

    int ep = epoll_create1(EPOLL_CLOEXEC);
    if (ep < 0) return -1;

    struct epoll_event ev = { .events = EPOLLIN, .data.fd = fd };
    if (epoll_ctl(ep, EPOLL_CTL_ADD, fd, &ev) < 0) { close(ep); return -1; }

    struct epoll_event out[8];
    for (;;) {
        int n = epoll_wait(ep, out, 8, 100 /* ms */);
        if (n < 0) {
            if (errno == EINTR) continue;   /* signal: retry */
            break;                          /* real error    */
        }
        if (n == 0) { periodic_tick(); continue; } /* timeout */

        for (int i = 0; i < n; i++) {
            if (out[i].events & EPOLLIN) {
                char buf[1024]; ssize_t r;
                /* LT: a single read is safe, but draining is fine too */
                while ((r = read(out[i].data.fd, buf, sizeof buf)) > 0)
                    handle(buf, r);
                /* r < 0 with EAGAIN => buffer empty, normal */
            }
        }
    }
    close(ep);
    return 0;
}

Under level-triggered mode the inner drain loop is optional — even if you read just once, the next epoll_wait will re-report the fd while data remains. Under EPOLLET that drain-to-EAGAIN loop becomes mandatory.

🔑 Chapter recap

Blocking traps one thread on one fd; O_NONBLOCK returns EAGAIN instead of sleeping but busy-polling burns a core. Thread-per-fd is fine for a few fds but doesn't scale — 8 MiB stacks and $O(N)$ context switches sink it (the C10k problem). select/poll re-scan the whole set every call ($O(n)$); epoll keeps the interest set in the kernel and returns only ready fds ($O(r)$), with level- vs edge-trigger (EPOLLET drains to EAGAIN) as the key choice. io_uring goes further — completion not readiness, batched syscalls over shared SQ/CQ rings. The payoff is the event-loop pattern: one cache-warm thread multiplexing GPS+IMU+LiDAR+CAN while the 20 Hz control deadline stays sacred. Next: the packet path from NIC to socket.
