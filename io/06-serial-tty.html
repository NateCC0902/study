Serial & the TTY Layer: getting bytes off a GPS without lag

A serial port looks like the simplest device in the box — two wires and a baud rate. But between your read() and the silicon sits the TTY layer, a 40-year-old subsystem built for teletype terminals that, by default, will mangle your binary sensor stream and add tens of milliseconds of latency. This chapter is how to make a UART behave like the dumb byte pipe a robot actually needs.

1The UART frame: how one byte goes down a wire

① Intuition build the picture first

A UART (Universal Asynchronous Receiver/Transmitter) sends bytes with no clock wire. That's the "asynchronous" part — and it's the whole trick. Both ends agree on a speed (the baud rate) ahead of time, and the receiver re-syncs on every single byte using the shape of the frame. There's no shared clock to drift, so the framing is the synchronization.

The idle line sits high (logic 1). To send a byte the transmitter pulls the line low for exactly one bit period — that falling edge is the start bit, the receiver's "wake up, a byte is coming" signal. Then 8 data bits go out LSB first, an optional parity bit, and finally the line is driven high again for one or two stop bits, returning to idle. Having locked onto the start edge, the receiver samples each following bit in the middle of its window.

💡 Why idle-high and start-low

A broken or unplugged wire floats — and a pulled-up line reads as idle (1), not an endless stream of garbage start bits. The start bit is a transition (1→0), detectable even when the two clocks disagree by a few percent; that's why a UART tolerates ~±3% baud mismatch and a dead cable just looks like silence.

Parity is the cheapest possible error check: one bit chosen so the total number of 1-bits (data + parity) is even (even parity) or odd (odd parity). It catches any single-bit flip and is blind to two flips — useless against bursty noise, which is why almost all robotics links run no parity and push integrity up to a real checksum (NMEA's *hh, a CRC in a binary protocol). The shorthand 8N1 means 8 data bits, No parity, 1 stop bit — the near-universal default.

④ Real case this ships on the USV
🏭 Real case — the GPS that "drops characters"

On the Jetson AGX Orin USV a u-blox GPS streams NMEA at 8N1, 115200 baud. A field report of "garbled fixes" turned out to be the receiver configured for even parity: the GPS's parity bit was being interpreted as a 9th data bit, shifting every byte. The wire was fine; the frame format was wrong on one end. Frame mismatch is the #1 serial bring-up bug, ahead of baud mismatch.

2Baud, bit-time, and the 80% you don't get

② Math derive it

Baud is symbols per second; on a plain UART one symbol is one bit, so baud = bits/s on the wire. The time to send one bit is just its reciprocal:

$$ t_\text{bit} = \frac{1}{\text{baud}}, \qquad t_\text{bit}(115200) = \frac{1}{115200}\,\text{s} \approx 8.68\ \mu\text{s} $$

But you don't get to send 8 useful bits per byte — every frame carries overhead. With $N_\text{frame}$ bits per frame (start + data + parity + stop), the achievable byte rate and the wire efficiency are:

$$ N_\text{frame} = 1 + 8 + p + s,\qquad \dot B = \frac{\text{baud}}{N_\text{frame}},\qquad \eta = \frac{8}{N_\text{frame}} $$

For 8N1, $N_\text{frame} = 1 + 8 + 0 + 1 = 10$, so $\eta = 8/10 = 80\%$ and $\dot B = 115200/10 = 11{,}520$ bytes/s. That clean factor of 10 is exactly why people pick 8N1: divide baud by ten to get bytes/s in your head. Go to 8E2 (even parity, 2 stop) and $N_\text{frame}=12$, efficiency drops to $8/12 \approx 67\%$ — you just gave away a sixth of your bandwidth for one weak error bit.

∑ Sizing a sensor budget

Flip it around: if a sensor emits a 64-byte binary record at 200 Hz, that's $64 \times 200 = 12{,}800$ bytes/s of payload. At 8N1 you need $12{,}800 \times 10 = 128{,}000$ bits/s on the wire — 115200 baud won't fit (it tops out at 11,520 B/s). You'd jump to 230400 or 460800 baud. Always size with the frame overhead included, never the raw payload.

🔑 What to notice

At 9600 baud one bit is ~104 µs and a whole 8N1 frame is ~1.04 ms — slow enough that a few bytes of latency is milliseconds. Crank to 1,000,000 baud and the same frame is 10 µs; now the bottleneck stops being the wire and becomes how often the kernel hands you the bytes. Add a parity bit or a second stop bit and watch efficiency fall below 80% — every non-data bit is bandwidth you paid for and can't use.

4The TTY stack: why your bytes get edited

① Intuition the layer you didn't ask for

When you open("/dev/ttyUSB0") you are not talking to the UART. You're talking to the TTY layer, three stacked pieces: the tty core (the /dev/tty* character device and its buffers), the line discipline (the byte-processing policy), and the driver (the chip-specific code that pushes bytes to hardware). The line discipline is the part that bites robots.

flowchart TD A["your app
read/write"] --> C["tty core
buffers + termios"] C --> LD["line discipline
N_TTY: canonical / raw"] LD --> DRV["uart driver
ttyUSB* or ttyTHS*"] DRV --> HW["UART / USB-serial chip"]

The default line discipline, N_TTY, is in canonical mode — it pretends your serial port is a terminal where a human is typing. It buffers input line-by-line until it sees a newline, lets the user backspace, echoes characters back, expands tabs, and treats certain bytes as control: 0x03 (Ctrl-C) raises a signal, 0x11/0x13 (XON/XOFF) pause the stream, 0x04 (Ctrl-D) signals EOF. For a login shell that's exactly right. For a binary GPS stream it's a disaster: a 0x03 in your payload kills your process, a 0x13 silently stops the flow, and you never see a byte until a 0x0A happens to come by.

⚠️ Pitfall — canonical mode eats binary data

The classic symptom: your reader blocks forever even though the device is clearly transmitting (you can see traffic in dmesg or on a scope). Canonical mode is holding every byte hostage until a newline. The fix isn't a bigger buffer — it's switching to raw mode, where the line discipline becomes a transparent pass-through and a byte is a byte.

5termios: configuring the port in C

③ Code read every line

Every TTY setting lives in a struct termios: four flag words (c_iflag input, c_oflag output, c_cflag control, c_lflag local/line) plus a c_cc[] array of control characters and the baud rate. You tcgetattr() the current settings, edit the struct, then tcsetattr() it back. The whole "make this a raw byte pipe" dance is captured by one helper, cfmakeraw(), which clears the canonical/echo/signal flags and the input/output translations in one shot.

serial_open.c C
#include <termios.h>
#include <fcntl.h>
#include <unistd.h>

int serial_open(const char *path, speed_t baud) {
    /* O_NOCTTY: don't let this tty become our controlling terminal
       (a stray Ctrl-C on the line must not signal us). */
    int fd = open(path, O_RDWR | O_NOCTTY);
    if (fd < 0) return -1;

    struct termios tio;
    if (tcgetattr(fd, &tio) < 0) { close(fd); return -1; }

    cfmakeraw(&tio);              /* raw mode: no canonical, no echo, no signals */
    cfsetispeed(&tio, baud);     /* input baud  */
    cfsetospeed(&tio, baud);     /* output baud */

    tio.c_cflag |= (CLOCAL | CREAD);   /* ignore modem ctrl lines; enable receiver */
    tio.c_cflag &= ~CSTOPB;            /* 1 stop bit (clear = 1, set = 2)          */
    tio.c_cflag &= ~PARENB;            /* no parity  ->  this is the N in 8N1       */
    tio.c_cflag &= ~CRTSCTS;           /* no hardware flow control (set to enable)  */

    /* Blocking read policy: return as soon as >=1 byte is available,
       but wait at most 100 ms (VTIME is in deciseconds). */
    tio.c_cc[VMIN]  = 0;
    tio.c_cc[VTIME] = 1;

    tcflush(fd, TCIFLUSH);                       /* drop stale bytes in the kernel buffer */
    if (tcsetattr(fd, TCSANOW, &tio) < 0) {      /* apply immediately */
        close(fd); return -1;
    }
    return fd;
}
🛠️ Set the speed with the speed_t constants

Baud is not a free integer: cfsetispeed() takes a speed_t token like B115200 or B460800, which the driver maps to a divisor. Standard rates have constants; for an off-list rate (say 250000) you need the termios2 / BOTHER ioctl path. cfmakeraw() is POSIX shorthand — it's equivalent to clearing ICANON|ECHO|ISIG in c_lflag, IXON|ICRNL in c_iflag, and OPOST in c_oflag.

The two fields that decide your latency are VMIN and VTIME. They form a small state machine for how a blocking read() returns:

6Flow control: telling the sender to wait

If the receiver can't keep up, bytes overrun the FIFO and are lost — and a UART overrun is silent (you get a bumped error counter, not an error on read()). Flow control is backpressure on the wire. Two flavors:

  • Hardware (RTS/CTS): two extra wires. The receiver de-asserts RTS ("I'm not ready"); the sender watches CTS and pauses. It's out-of-band — it never touches your data — and it reacts within a byte time. Enable with CRTSCTS in c_cflag. This is the right choice for high rates, if the wires are connected.
  • Software (XON/XOFF): in-band control bytes 0x11 (XON, resume) and 0x13 (XOFF, pause), enabled with IXON/IXOFF. No extra wires — but those two byte values are now reserved and can never appear in your payload. For binary data that's fatal, so robotics links keep XON/XOFF off and use RTS/CTS or simply run a rate the receiver can always drain.
⚠️ Pitfall — XON/XOFF on a binary stream

Leave IXON set (it often is by default) and the moment your binary payload contains a 0x13 byte, the kernel quietly stops giving you data until a 0x11 shows up. The link "hangs" with no error. cfmakeraw() clears IXON for you — one more reason to always start from raw.

7Latency: the 16 ms you didn't budget for

④ Real case 200 Hz with no lag

Here's the trap that ruins control loops. The wire is fast — an 8N1 byte at 115200 is 86.8 µs — but the path from "byte in the chip's FIFO" to "byte in your buffer" has two latency sources that dwarf it:

  1. The FTDI / USB-serial latency timer. USB-serial adapters (FTDI, CP210x) don't interrupt the host per byte — they batch bytes into USB packets and flush on a timer. The FTDI default is 16 ms. So a 1-byte GPS sentence can sit in the adapter for up to 16 ms before it ever crosses USB. For a 200 Hz loop (5 ms period) that's three loop iterations of pure dead time.
  2. VMIN/VTIME on the kernel side. Ask for VMIN=64 and the kernel won't wake you until 64 bytes pile up — great for throughput, terrible if your record is 20 bytes at 200 Hz.
🏭 Real case — fixing IMU lag on the AGX Orin

An IMU on a USB-serial bridge into the Orin showed a stubborn ~16 ms lag in the EKF no matter how the code was tuned. The fix was one sysfs write — drop the FTDI latency timer from 16 ms to 1 ms:

terminal bash
# find the device, then read/write its latency_timer (milliseconds)
cat /sys/bus/usb-serial/devices/ttyUSB0/latency_timer    # -> 16
echo 1 | sudo tee /sys/bus/usb-serial/devices/ttyUSB0/latency_timer

# make it persistent with a udev rule (see Ch 03):
# SUBSYSTEM=="usb-serial", ATTR{latency_timer}="1"

Combined with VMIN=0, VTIME=1 (wake on the first byte, 100 ms safety timeout), worst-case latency dropped from ~16 ms to ~1 ms — under one quarter of the 5 ms loop period. The on-chip UART (next section) doesn't have this timer at all.

The general latency model for a USB-serial port adds the adapter's flush window to the wire time:

$$ t_\text{latency} \approx t_\text{flush} + n\cdot t_\text{frame}, \qquad t_\text{flush} = \text{latency\_timer} $$

At 16 ms vs 1 ms the latency_timer term is the budget; the frame time is noise next to it. Lower the timer, and you trade a little USB overhead (more, smaller packets) for responsiveness — exactly the right trade for a control loop.

On the Jetson you'll see two families of serial device, and they behave differently enough to matter:

The on-chip ttyTHS ports have no USB hop and no batching timer, so they're the low-latency choice for a tightly-timed sensor — but they're 3.3 V logic-level pins, so a level shifter or transceiver is mandatory for anything off-board. USB adapters are convenient and hot-pluggable but inherit USB's latency floor, and their ttyUSBn number depends on plug order — which is exactly why Chapter 03's udev by-id symlinks exist. One more rule of the road: the kernel boot console may be parked on a UART; if a port is in /proc/cmdline as console=ttyTHSx, the kernel is writing log spam to it and it's not yours to use.

9Your turn

Q1 Conceptual check
Why is 8N1 said to be only 80% efficient, even with a perfect, noise-free wire?
  • The line discipline drops one byte in five.
  • Each 8-bit byte rides in a 10-bit frame (1 start + 8 data + 1 stop), so 8/10 of the wire bits are payload.
  • USB-serial adapters lose 20% of bytes to the latency timer.
  • Parity consumes 20% of the bandwidth.
Efficiency is $\eta = 8/N_\text{frame}$. For 8N1, $N_\text{frame} = 1+8+0+1 = 10$, so $\eta = 8/10 = 80\%$ — the start and stop bits are framing overhead, not data, and (note) 8N1 has no parity at all. Adding even parity (8E1) makes it 11 frame bits, $\approx 73\%$.
Q2 Compute it
A port runs 8N1 at 115200 baud. What is the maximum sustained throughput in bytes per second? (Hint: $\dot B = \text{baud}/N_\text{frame}$, and for 8N1 $N_\text{frame}=10$.)
B/s
$\dot B = 115200 / 10 = 11{,}520$ bytes/s. Each frame is $10 \times t_\text{bit} = 10 \times 8.68\,\mu\text{s} = 86.8\,\mu\text{s}$, and $1/86.8\,\mu\text{s} \approx 11{,}520$ frames/s — one byte each. The handy mental shortcut for 8N1 is simply "baud ÷ 10."
Q3 Coding challenge — open a port in raw mode
Write the C that opens /dev/ttyUSB0 at 115200 8N1, in raw mode, with a read that returns on the first byte but times out after ~100 ms. Name the three calls that set raw mode and the baud, and the two fields that set the read timing.
Hint
cfmakeraw() does the raw flag work; cfsetispeed()/cfsetospeed() set baud; VMIN=0, VTIME=1 means "return on ≥1 byte, or after 1 decisecond." Open with O_NOCTTY so a stray Ctrl-C on the line can't signal you.
Show one solution
raw_open.c C
#include <termios.h>
#include <fcntl.h>
#include <unistd.h>

int raw_open(const char *path) {
    int fd = open(path, O_RDWR | O_NOCTTY);
    if (fd < 0) return -1;

    struct termios tio;
    tcgetattr(fd, &tio);
    cfmakeraw(&tio);                 /* raw: no canon/echo/signals/translation */
    cfsetispeed(&tio, B115200);     /* baud in */
    cfsetospeed(&tio, B115200);     /* baud out */
    tio.c_cflag |= (CLOCAL | CREAD);
    tio.c_cflag &= ~CSTOPB;         /* 1 stop  */
    tio.c_cflag &= ~PARENB;         /* no parity -> the N in 8N1 */
    tio.c_cc[VMIN]  = 0;            /* return on >=1 byte... */
    tio.c_cc[VTIME] = 1;           /* ...or after 100 ms */

    tcflush(fd, TCIFLUSH);
    tcsetattr(fd, TCSANOW, &tio);
    return fd;
}

The three "shape the port" calls are cfmakeraw, cfsetispeed, cfsetospeed; the two read-timing fields are c_cc[VMIN] and c_cc[VTIME]. Clearing PARENB + CSTOPB gives you the N1 of 8N1 (8 data bits is cfmakeraw's default via CS8).

🔑 Chapter recap

A UART frame is idle-high, a start bit, 8 data bits LSB-first, optional parity, 1–2 stop bits; 8N1 is 10 wire bits per byte, so efficiency is 80% and byte rate is baud ÷ 10. Linux wraps the port in the TTY stack (core → line discipline → driver); the default canonical mode edits and buffers your bytes, so always switch to raw via cfmakeraw() + cfset{i,o}speed(), and use VMIN/VTIME to trade latency against batching. Keep XON/XOFF off for binary data; prefer RTS/CTS. The real latency killer on USB-serial is the latency_timer (16 ms default — drop it to 1 ms); the on-chip ttyTHS ports skip USB entirely for the lowest latency.
