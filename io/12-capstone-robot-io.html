1The whole-system view: every I/O on one map

① Intuition see the whole vessel at once

A robot is not a CPU with sensors bolted on; it is a concurrency problem with a hull. Heading sensors, position sensors, the LiDAR, the thrusters, and the shore link all move bytes at the same time, on different buses, with wildly different rates and failure tolerances. Each previous chapter taught one slice. The capstone skill is holding the entire data-flow graph in your head: from the physical pin, up through the bus controller, into a kernel driver, out to a file descriptor your code calls read() on.

Here is the USV's complete I/O map. Follow any leaf left-to-right and you walk the exact path of one byte — physical port → bus → kernel driver → the fd userspace touches:

② Math why one map, not four

The reason you draw one graph and not four separate diagrams is that the kernel collapses all of them into a single namespace: integer file descriptors. WiFi sockets, the LiDAR UDP socket, /dev/gps, and the CAN socket are all just non-negative ints in one per-process table. That uniformity is the whole point of "everything is a file" from Chapter 01 — it's what makes the unified epoll loop in §4 possible. If there are $N$ I/O sources and each gets one fd, the entire vessel's input is a set of $N$ integers you can watch with a single syscall.

🏭 Real case — the AGX Orin's split personality

Notice the buses split into two camps. PCIe-class buses (the MGBE Ethernet, USB host) auto-enumerate: plug a device in and the kernel walks descriptors and binds a driver (Ch 05). The on-module WiFi sits on SDIO, which has no enumeration protocol — the kernel only knows it exists because the device tree declares it (Ch 05). CAN (mttcan) is the same: declared, not discovered. This single fact dictates half your bring-up work in §3.

2Interface-role assignment: who carries what

① Intuition match the medium to the message

Every interface has a personality you learned the hard way in earlier chapters, and architecture is mostly about not fighting it. WiFi is a shared, half-duplex, jittery medium (Ch 09) — wonderful for "where are you, are you alive, here's a new waypoint," catastrophic for a 76 Mbit/s point cloud. Ethernet is full-duplex and switched — born to carry heavy, latency-sensitive sensor streams. Serial over USB is low-rate and human-paced but needs stable names (Ch 03/06). CAN is slow but deterministic with priority arbitration (Ch 10) — exactly what actuators want.

So the assignment writes itself once you respect each medium:

InterfaceRoleWhy this mediumWhat must NOT ride it
WiFi (wlan0)Control & telemetryLow data rate, tolerates jitter; only link to shoreLiDAR, video, anything > a few Mbit/s
Ethernet (eth0)Heavy sensors (LiDAR, camera)Full-duplex, switched, ~118 MB/s usableThe default route (Ch 09 route theft!)
USB/serial (/dev/gps, /dev/imu)Low-rate point sensorsSimple, ubiquitous; udev gives stable namesAnything time-critical without flow control
CAN (can0)Actuators & field instrumentsPriority arbitration, multi-drop, noise-immuneBulk data — 500 kbit/s is the ceiling
⚠️ Pitfall — the default route lives on exactly one NIC

The most common two-NIC bug (Ch 09): the LiDAR's Ethernet link comes up with DHCP, grabs a default route, and now your SSH-over-WiFi packets try to exit through the LiDAR's dead-end network. The control link must own the default route; the sensor NIC gets a host-scoped address and no gateway. We enforce that in netplan in §3.

3The bring-up chain as persistent config

① Intuition nothing typed by hand survives a reboot

Everything you proved interactively across the course — declaring the bus, loading the firmware, naming the device, addressing the NIC, ordering the daemons — has a persistent home on disk. A robot you have to babysit through boot is a robot that strands you at sea. The bring-up chain is a layered stack; each layer can only succeed if the one below it already did.

③ Code the four config files that boot the vessel

Layer 1–3 you established earlier; here is what they crystallize into on disk. First, the udev rule pinning the serial devices to stable symlinks regardless of plug order (Ch 03):

/etc/udev/rules.d/90-usv.rules bash
# Match the GPS by its FTDI serial number, never by ttyUSB enumeration order
SUBSYSTEM=="tty", ATTRS{idVendor}=="0403", ATTRS{serial}=="A50285BI", \
    SYMLINK+="gps", MODE="0660", GROUP="dialout"

# The IMU is a CDC-ACM device; match by USB serial string
SUBSYSTEM=="tty", ATTRS{idVendor}=="1546", ATTRS{serial}=="3779A1C2", \
    SYMLINK+="imu", MODE="0660", GROUP="dialout"

Next, netplan — the control link owns the default route; the LiDAR NIC is host-scoped with no gateway, killing the route-theft bug:

/etc/netplan/01-usv.yaml bash
network:
  version: 2
  ethernets:
    eth0:                      # LiDAR — heavy sensor, NO gateway
      addresses: [192.168.10.2/24]
      dhcp4: false
      routes: []              # host-scoped: never steals the default route
  wifis:
    wlan0:                     # control + telemetry — owns the default route
      dhcp4: true
      access-points:
        "usv-shore": {password: "redacted-key"}

Finally, the systemd unit that brings CAN up. SocketCAN's can0 is declared by the device tree but starts down; you must set the bitrate and bring the link up (Ch 10). It depends on nothing networked, so order it before the app — but the app's WiFi sockets need the network, so the app waits on network-online.target:

/etc/systemd/system/can0.service bash
[Unit]
Description=Bring up SocketCAN can0
# The mttcan driver must have bound the netdev first
After=sys-subsystem-net-devices-can0.device
BindsTo=sys-subsystem-net-devices-can0.device

[Service]
Type=oneshot
RemainAfterExit=yes
# 500 kbit/s; auto-recover from bus-off after 100 ms (Ch 10)
ExecStart=/sbin/ip link set can0 type can bitrate 500000 restart-ms 100
ExecStart=/sbin/ip link set can0 up
ExecStop=/sbin/ip link set can0 down

[Install]
WantedBy=multi-user.target
🛠️ network-online.target is not network.target

network.target only means "the network stack is configured" — it can be reached before a single interface has an IP. The sensor-manager needs an actually-routable WiFi link, so its unit must After=network-online.target and Wants=network-online.target (the Wants is what pulls in systemd-networkd-wait-online; After alone never waits). Getting this wrong is why a service "works when I start it by hand but not at boot."

4One epoll loop to read them all

① Intuition four buses, one thread, zero polling

You now have four (or more) file descriptors: a UDP socket for the LiDAR, two raw TTYs for GPS and IMU, and a CAN socket. The naive design — a thread per device, each blocking on its own read() — burns context switches and makes shutdown a synchronization nightmare (Ch 08). The right design is one thread, all fds registered with epoll, blocking in a single epoll_wait() until any source is ready. This is the multiplexing payoff: cost scales with the number of ready fds, not the total.

② Math why one loop wins

With thread-per-fd, $N$ devices cost roughly $N$ kernel threads and $O(N)$ context switches per scheduling round. With poll()/select() the kernel rescans all $N$ fds every call, so a wakeup costs $O(N)$. epoll returns only the ready set $k$, so a wakeup costs:

$$ C_\text{epoll} = O(k), \qquad k \le N, \quad k \approx 1\text{–}2 \text{ in steady state} $$

For our 4–8 sources $k$ is tiny, but the structural win is the same one that solved C10k: the loop doesn't get more expensive as you add quiet sensors. One epoll_wait() call, woken only when bytes actually arrive.

③ Code the sensor-manager skeleton

Every fd is non-blocking and edge-triggered (drain until EAGAIN), and a timerfd drives the 20 Hz control tick from inside the same loop — so the controller and the sensors share one scheduler with no extra thread:

sensor_manager.c C
int ep = epoll_create1(EPOLL_CLOEXEC);

// Each open_* returns a non-blocking fd; tag it so we know who fired.
int fds[] = { lidar_udp, gps_tty, imu_tty, can_sock, tick_timerfd };
for (int i = 0; i < 5; i++) {
    struct epoll_event ev = {
        .events = EPOLLIN | EPOLLET,        // edge-triggered: drain fully
        .data.fd = fds[i],
    };
    epoll_ctl(ep, EPOLL_CTL_ADD, fds[i], &ev);
}

struct epoll_event events[16];
for (;;) {
    int n = epoll_wait(ep, events, 16, -1);
    if (n < 0) { if (errno == EINTR) continue; break; }   // Ch 11: handle EINTR

    for (int i = 0; i < n; i++) {
        int fd = events[i].data.fd;
        if (events[i].events & (EPOLLHUP | EPOLLERR)) {
            mark_source_down(fd);            // §6 failure matrix takes over
            continue;
        }
        if (fd == tick_timerfd)      run_control_step();   // 20 Hz PID tick
        else                         drain_source(fd);     // read until EAGAIN
    }
}
🔑 What ties the course together here

This one loop uses Chapter 01 (fds + partial reads), 06 (raw termios TTYs), 08 (epoll, edge-triggered), 09 (the UDP socket), 10 (the CAN socket), and 11 (EINTR + a timerfd for deterministic timing). Five chapters collapse into ~30 lines. EPOLLHUP is your free disconnect detector — the kernel tells you when a device vanishes, so you never have to poll for liveness.

5The bandwidth budget across buses

② Math does it physically fit?

Before you trust the architecture, prove the data fits on the wires. Each source has a payload rate; each bus has a usable ceiling (always below the line rate — protocol overhead and, for WiFi, contention). The rule is simple:

$$ \text{utilization}_\text{bus} = \frac{\sum_i R_i^{\,\text{payload}} \cdot (1+h_i)}{B_\text{usable}} \;<\; 1 $$

where $R_i$ is a stream's payload rate, $h_i$ is its framing overhead fraction, and $B_\text{usable}$ is the realistic throughput of the bus. Take the LiDAR: a spinning unit emitting $6.0\times10^{5}$ points/s at 16 bytes/point is

$$ R_\text{lidar} = 6.0\times10^{5}\times16 = 9.6\times10^{6}\ \text{B/s} = 76.8\ \text{Mbit/s} $$

On a 1000BASE-T link with ~118 MB/s usable, that's $9.6/118 \approx 8\%$ utilization — comfortable. Now run the whole vessel:

SourcePayload rateBusBus usableUtilization
LiDAR @ 600 k pts/s76.8 Mbit/sEthernet (1 Gbit/s)~944 Mbit/s~8%
GPS @ 10 Hz, 80 B~6.4 kbit/sUSB serial (115200)~92 kbit/s~7%
IMU @ 200 Hz, 40 B~64 kbit/sUSB serial (921600)~737 kbit/s~9%
Thrusters + N2K~80 kbit/sCAN (500 kbit/s)~250 kbit/s~32%
Telemetry to shore~200 kbit/sWiFi (shared)~20 Mbit/s real~1%
⚠️ Pitfall — the WiFi ceiling is a lie if you average it

WiFi's "20 Mbit/s real" is an aggregate, contended number with fat jitter (Ch 09). At 1% average you'll still see 200 ms stalls when the medium is busy. That is exactly why telemetry rides WiFi and the LiDAR does not — and why the failure matrix in §6 assumes the WiFi link can vanish for seconds at a time.

6Health monitoring & the failure matrix

① Intuition plan the failure before it happens

At sea, every link will fail. Robust I/O (Ch 11) isn't a try/catch; it's a per-source policy decided in advance. The kernel hands you the failure signal cheaply — EPOLLHUP on a vanished USB device, recv() timeouts on a dead socket, the CAN error counters crossing into bus-off — and your job is to map each signal to a degraded mode that keeps the boat safe rather than crashing the whole stack.

③ Code a pre-flight check that refuses to launch broken

Before you ever arm thrusters, run a pre-flight that asserts every link the matrix depends on. Fail loud, fail on the dock — not 2 km out:

preflight.sh bash
#!/usr/bin/env bash
set -euo pipefail
fail=0
check() { if eval "$2"; then echo "  OK  $1"; else echo "FAIL  $1"; fail=1; fi; }

echo "== USV pre-flight =="
check "GPS symlink present"   '[ -e /dev/gps ]'
check "IMU symlink present"   '[ -e /dev/imu ]'
check "can0 is UP"            '[ "$(cat /sys/class/net/can0/operstate)" = up ]'
check "eth0 carrier (LiDAR)"  '[ "$(cat /sys/class/net/eth0/carrier)" = 1 ]'
check "wlan0 has a route"     'ip route show default | grep -q wlan0'
check "LiDAR answering"       'timeout 2 nc -uz 192.168.10.10 2368'
# CAN traffic seen in 1 s? (Ch 10 candump)
check "thruster CAN traffic"  'timeout 1 candump -n 1 can0 >/dev/null 2>&1'

[ "$fail" -eq 0 ] && { echo "PRE-FLIGHT PASS"; exit 0; } \
                  || { echo "PRE-FLIGHT FAIL — do not arm"; exit 1; }
🏭 Real case — health from /sys and /proc, not guesses

Live health monitoring reads the same kernel surfaces you toured in Ch 02: /sys/class/net/<if>/operstate and carrier for link state, /sys/class/net/can0/can/state for CAN error state, /proc/interrupts to confirm the LiDAR NIC's IRQ is actually firing (Ch 07). No heuristics — the kernel already knows the truth and exposes it as files. Your monitor is a loop of cheap read()s on virtual files.

8Your turn

Q1 Interface-role assignment
You're adding a 4K bow camera streaming ~30 Mbit/s of H.264 to the operator. Which interface should carry it, given the roles we assigned?
  • WiFi (wlan0) — it's the link to the operator, so put the video there.
  • Ethernet (eth0) onto the vessel, transcoded/decimated before any WiFi hop — heavy media belongs on the full-duplex switched link, never raw on the shared WiFi medium.
  • CAN (can0) — it's already the field bus for devices.
  • USB serial (/dev/cam) — give it a stable udev name and stream it.
30 Mbit/s exceeds WiFi's realistic contended ceiling (~20 Mbit/s) and would starve the control link. Heavy media rides Ethernet on-vessel; only a heavily compressed, rate-limited preview should ever touch WiFi. CAN tops out at 500 kbit/s and USB serial at <1 Mbit/s — both are off by orders of magnitude.
Q2 Compute the LiDAR's bus load
The LiDAR emits $6.0\times10^{5}$ points per second at 16 bytes per point (payload only). What is its payload throughput in Mbit/s? (Hint: bytes/s × 8, then ÷ $10^{6}$.)
Mbit/s
$6.0\times10^{5}\times16 = 9.6\times10^{6}$ B/s. $\times 8 = 7.68\times10^{7}$ bit/s $= 76.8$ Mbit/s. On a 1 Gbit/s link (~944 Mbit/s usable after framing) that's ~8% utilization — which is precisely why this stream rides Ethernet and would be impossible on WiFi.
Q3 Build it — a systemd unit + pre-flight gate
Write a systemd unit for the sensor-manager that (a) starts only after the WiFi link is actually routable, (b) requires can0 to be up first, and (c) refuses to start unless preflight.sh exits 0. Then say why Wants=network-online.target matters.
Hint
Use After= for ordering, Requires=/Wants= for dependency, and ExecStartPre= for the gate. Remember After alone never waits for the network — you must also Wants the wait-online target.
Show one solution
/etc/systemd/system/sensor-manager.service bash
[Unit]
Description=USV unified sensor-manager (epoll loop)
# Ordering: only after a routable network AND can0 is up
After=network-online.target can0.service
Wants=network-online.target        # pulls in systemd-networkd-wait-online
Requires=can0.service              # if CAN fails to come up, do not start

[Service]
Type=simple
# Pre-flight gate: nonzero exit here blocks the service from starting
ExecStartPre=/usr/local/bin/preflight.sh
ExecStart=/usr/local/bin/sensor-manager
Restart=on-failure
RestartSec=2

[Install]
WantedBy=multi-user.target

Why Wants=network-online.target: After=network-online.target only orders the unit after that target is reached, but it does not cause the target to be reached. The target is only pulled into the boot transaction by something that Wants/Requires it (which in turn activates systemd-networkd-wait-online.service, the thing that actually blocks until an interface is up). Without the Wants, network-online.target may be reached instantly while WiFi is still associating, the sensor-manager starts, its connect() on wlan0 fails, and you get the classic "works by hand, broken at boot." The ExecStartPre gate is the belt to that suspenders: even if every dependency is satisfied, a failed physical link (no GPS symlink, dead LiDAR carrier, no CAN traffic) exits nonzero and systemd refuses to launch, keeping the thrusters disarmed.

🔑 Chapter recap

A robot's I/O is one data-flow graph: every source maps physical port → bus → driver → fd. Assign interfaces by their nature — WiFi=control/telemetry, Ethernet=heavy sensors, USB/serial=low-rate with udev names, CAN=actuators — and never let a sensor NIC steal the default route. The bring-up chain lives on disk as persistent config: device tree + firmware + udev rules + netplan + ordered systemd units gated by network-online.target and a can0 unit. One epoll loop reads every fd and drives the 20 Hz tick from a timerfd, synthesizing Chapters 01, 06, 08, 09, 10, and 11. Prove the bandwidth budget fits each bus, map every failure (WiFi drop, USB unplug, CAN bus-off, link-down) to a pre-planned degraded mode, and gate launch behind a pre-flight that reads the truth straight from /sys. Draw the §1 map from memory and you own the whole course.
