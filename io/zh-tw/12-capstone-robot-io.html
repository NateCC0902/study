規劃機器人的 I/O 架構：整艘 USV，一個系統

十一章的系統呼叫（syscall）、sysfs、udev、驅動程式（driver）、匯流排（bus）、序列（serial）、DMA、epoll、網路、CAN 與即時（real-time）。現在我們把它們組裝起來。我們要把這艘 Jetson AGX Orin USV 從一塊裸板，帶到一艘每一個感測器與致動器（actuator）都對應到一條匯流排、一支驅動程式、一個檔案描述子（file descriptor）的載具——它能確定性地開機、被單一事件迴圈（event loop）讀取，並挺過大海丟給它的種種故障。

1全系統視角：所有 I/O 攤在同一張圖上

① 直覺 一眼看完整艘船

機器人不是一顆栓上感測器的 CPU；它是一個帶著船殼的並行（concurrency）問題。航向感測器、定位感測器、光達（LiDAR）、推進器（thruster）以及岸基連線全都同時在搬移位元組，跑在不同的匯流排上，速率與容錯能力天差地遠。先前每一章都教了其中一個切片。總整理的功力，在於把整張資料流圖（data-flow graph）整個裝進腦袋裡：從實體接腳（physical pin），往上經過匯流排控制器，進入核心驅動程式（kernel driver），再出來成為你的程式碼會對它呼叫 read() 的那個檔案描述子。

這就是這艘 USV 完整的 I/O 地圖。沿著任何一片葉子由左讀到右，你就走過了一個位元組的確切路徑——實體埠 → 匯流排 → 核心驅動程式 → 使用者空間（user space）所碰觸的那個 fd：

② 數學 為什麼是一張圖，不是四張

你之所以畫一張圖而不是四張各自獨立的示意圖，是因為核心把它們全部塌縮成單一命名空間：整數型的檔案描述子。WiFi socket、光達 UDP socket、/dev/gps、CAN socket，全都只是同一張行程內表格裡的非負整數。這份一致性正是第 01 章「一切皆檔案」的全部重點——也正是它讓 §4 那個統一的 epoll 迴圈成為可能。若有 $N$ 個 I/O 來源、每個拿到一個 fd，那整艘船的輸入就是 $N$ 個整數的集合，你可以用單一系統呼叫去監看它們。

🏭 真實案例——AGX Orin 的雙重性格

注意匯流排分成兩派。PCIe 等級的匯流排（MGBE 乙太網、USB host）會自動列舉（enumeration）：插上一個裝置，核心就走訪描述子並綁定一支驅動程式（第 05 章）。模組上的 WiFi 坐在 SDIO 上，它沒有列舉協定——核心之所以知道它存在，純粹是因為裝置樹（device tree）宣告了它（第 05 章）。CAN（mttcan）也是一樣：是被宣告的，不是被發現的。光這一個事實，就決定了 §3 裡你一半的系統初始化工作。

2介面角色指派：誰負責載送什麼

① 直覺 讓媒介配上對的訊息

每個介面都有一個你在前面幾章吃過苦頭才認識的個性，而架構設計大半就是在不跟它對著幹。WiFi 是一種共享、半雙工、抖動很大的媒介（第 09 章）——拿來做「你在哪、還活著嗎、這是新的航點」棒極了，拿去載 76 Mbit/s 的點雲（point cloud）則是災難。乙太網是全雙工、有交換機的——天生就是要載送沉重、對延遲敏感的感測器串流。USB 上的序列傳輸速率低、節奏接近人類操作，但需要穩定的名稱（第 03／06 章）。CAN 慢，但有優先權仲裁（arbitration）、是確定性的（第 10 章）——正是致動器所要的。

所以只要你尊重每一種媒介，這份指派自己就寫好了：

介面角色為什麼用這個媒介絕對不能放上去的東西
WiFi（wlan0）控制與遙測（telemetry）資料速率低、容忍抖動；唯一通往岸基的連線光達、影像、任何 > 數 Mbit/s 的東西

最常見的雙網卡 bug（第 09 章）：光達的乙太網連線用 DHCP 起來、搶下一條預設路由，於是你那些走 WiFi 的 SSH 封包就想從光達那條死路網路出去。控制連線必須擁有預設路由；感測器網卡只拿一個主機範圍（host-scoped）的位址、不設閘道（gateway）。我們在 §3 用 netplan 把這件事強制下來。

最後是把 CAN 拉起來的 systemd unit。SocketCAN 的 can0 由裝置樹宣告，但起來時是down 的；你必須設定位元率（bitrate）並把連結拉起來（第 10 章）。它不依賴任何網路相關的東西，所以把它排在 app 之前——但 app 的 WiFi socket 需要網路，所以 app 要等 network-online.target：

[Install]
WantedBy=multi-user.target
🛠️ network-online.target 不等於 network.target

network.target 只代表「網路堆疊已設定完成」——在任何一張介面拿到 IP 之前它就可能達成了。sensor-manager 需要一條真正可路由的 WiFi 連線，所以它的 unit 必須 After=network-online.target 而且 Wants=network-online.target（那個 Wants 才是把 systemd-networkd-wait-online 拉進來的東西；單靠 After 永遠不會等）。把這件事弄錯，就是為什麼一個服務會「我手動啟動就好好的，開機卻不行」。

4一個 epoll 迴圈讀盡所有來源

① 直覺 四條匯流排、一個執行緒、零輪詢

你現在有了四個（或更多）檔案描述子：光達的一個 UDP socket、GPS 與 IMU 的兩個原始（raw）TTY，還有一個 CAN socket。最天真的設計——每個裝置開一個執行緒、各自阻塞在自己的 read() 上——會燒掉一堆內容切換（context switch），還把關機搞成同步惡夢（第 08 章）。對的設計是一個執行緒、所有 fd 都註冊到 epoll、阻塞在單一一次 epoll_wait()，直到任何一個來源就緒。這就是多工（multiplexing）的回報：成本只隨就緒的 fd 數量增長，而不是隨總數。

② 數學 為什麼單一迴圈會贏

用「每個 fd 一條執行緒」的做法，$N$ 個裝置大約要 $N$ 條核心執行緒，外加每一輪排程 $O(N)$ 次內容切換。用 poll()／select()，核心每次呼叫都要重掃全部 $N$ 個 fd，所以一次喚醒要 $O(N)$。epoll 只回傳就緒集合 $k$，所以一次喚醒只要：

$$ C_\text{epoll} = O(k), \qquad k \le N, \quad k \approx 1\text{–}2 \text{ in steady state} $$

對我們這 4–8 個來源來說 $k$ 很小，但結構上的勝利和當年解決 C10k 的那個一模一樣：你加進更多安靜的感測器，迴圈也不會變貴。一次 epoll_wait() 呼叫，只在真的有位元組到達時才被喚醒。

③ 程式碼 sensor-manager 骨架

每個 fd 都是非阻塞（non-blocking）且邊緣觸發（edge-triggered，要排乾到 EAGAIN 為止），還有一個 timerfd 從同一個迴圈裡面驅動 20 Hz 的控制節拍（tick）——這樣控制器和感測器就共用一個排程器，不必額外開執行緒：

sensor_manager.c C
int ep = epoll_create1(EPOLL_CLOEXEC);

// Each open_* returns a non-blocking fd; tag it so we know who fired.
int fds[] = { lidar_udp, gps_tty, imu_tty, can_sock, tick_timerfd };
for (int i = 0; i < 5; i++) {
    struct epoll_event ev = {
        .events = EPOLLIN | EPOLLET,        // edge-triggered: drain fully
        .data.fd = fds[i],
    };
    epoll_ctl(ep, EPOLL_CTL_ADD, fds[i], &ev);
}

struct epoll_event events[16];
for (;;) {
    int n = epoll_wait(ep, events, 16, -1);
    if (n < 0) { if (errno == EINTR) continue; break; }   // Ch 11: handle EINTR

    for (int i = 0; i < n; i++) {
        int fd = events[i].data.fd;
        if (events[i].events & (EPOLLHUP | EPOLLERR)) {
            mark_source_down(fd);            // §6 failure matrix takes over
            continue;
        }
        if (fd == tick_timerfd)      run_control_step();   // 20 Hz PID tick
        else                         drain_source(fd);     // read until EAGAIN
    }
}
🔑 整門課在這裡被串起來

這一個迴圈用上了第 01 章（fd + 部分讀取）、06 章（原始 termios 的 TTY）、08 章（epoll、邊緣觸發）、09 章（那個 UDP socket）、10 章（那個 CAN socket），以及 11 章（EINTR + 一個用來做確定性計時的 timerfd）。五章塌縮成約 30 行。EPOLLHUP 是你免費的斷線偵測器——裝置消失時核心會告訴你，所以你永遠不必輪詢（polling）去確認它還活著。

5跨匯流排的頻寬預算

② 數學 它在物理上塞得下嗎？

在你信任這套架構之前，先證明資料塞得下這些線路。每個來源有一個負載（payload）速率；每條匯流排有一個可用上限（永遠低於線路速率——協定有額外開銷，WiFi 還要加上爭用（contention））。規則很簡單：

$$ \text{utilization}_\text{bus} = \frac{\sum_i R_i^{\,\text{payload}} \cdot (1+h_i)}{B_\text{usable}} \;<\; 1 $$

其中 $R_i$ 是某條串流的負載速率，$h_i$ 是它的封裝（framing）開銷比例，$B_\text{usable}$ 是這條匯流排現實可達的吞吐量。拿光達來說：一台旋轉式單元以 $6.0\times10^{5}$ 點／秒、每點 16 位元組發出，於是

$$ R_\text{lidar} = 6.0\times10^{5}\times16 = 9.6\times10^{6}\ \text{B/s} = 76.8\ \text{Mbit/s} $$

在一條 1000BASE-T 連線上、約 118 MB/s 可用，那是 $9.6/118 \approx 8\%$ 使用率——綽綽有餘。現在把整艘船跑一遍：

來源負載速率匯流排匯流排可用使用率
光達 @ 600 k 點/s76.8 Mbit/s乙太網（1 Gbit/s）約 944 Mbit/s約 8%
GPS @ 10 Hz, 80 B約 6.4 kbit/sUSB 序列（115200）約 92 kbit/s約 7%
IMU @ 200 Hz, 40 B約 64 kbit/sUSB 序列（921600）約 737 kbit/s約 9%
推進器 + N2K約 80 kbit/sCAN（500 kbit/s）約 250 kbit/s約 32%
遙測回傳岸基約 200 kbit/sWiFi（共享）約 20 Mbit/s 實測約 1%
⚠️ 陷阱——WiFi 的天花板若你拿來平均，那就是個謊

WiFi 的「20 Mbit/s 實測」是一個整體、被爭用的數字，帶著肥厚的抖動（第 09 章）。在 1% 的平均值下，當媒介忙起來時你照樣會看到 200 ms 的停頓。這正是為什麼遙測走 WiFi、光達不走——也正是為什麼 §6 的故障矩陣假設 WiFi 連線可能一次消失好幾秒。

6健康監測與故障矩陣

① 直覺 在故障發生前就規劃好它

在海上，每一條連線都會故障。穩健的 I/O（第 11 章）不是一個 try／catch；它是事先決定好的、每個來源各自的策略。核心很便宜地把故障訊號交到你手上——消失的 USB 裝置會給 EPOLLHUP、死掉的 socket 會讓 recv() 逾時、CAN 的錯誤計數器會越界進入 bus-off——而你的工作，是把每一個訊號對應到一個降級模式（degraded mode），讓船保持安全，而不是把整個堆疊搞當。

故障你如何偵測它核心／驅動的行為架構上的應對
WiFi 中斷（控制連線）持續 T 沒有心跳；send() 的 EAGAIN 越堆越多Socket 保持開啟；封包被默默丟棄失效安全（failsafe）：定點停留／盤旋，長時間斷線就切掉推力，繼續記錄
USB GPS／IMU 拔除fd 上出現 EPOLLHUP；read() 回傳 0 / ENODEVchardev 消失，重新插上後 udev 重新建立 /dev/gps標記為陳舊，退回航位推算（dead-reckoning，IMU）或最後定位；udev 重新加入時重新開啟
CAN bus-offNetlink 狀態 = BUS_OFF；TX 錯誤計數器 > 255控制器停止 TX/RX 直到重置restart-ms 100 自動恢復；若反覆 bus-off，命令推力歸中
乙太網連結斷開（光達）SIOCGIFFLAGS 失去 IFF_RUNNING；持續 T 無 UDP載波（carrier）遺失；socket 開著卻安靜無聲把感知切換到最後一次良好掃描；發出警報；導航是降級，不是當機
③ 程式碼 一個會拒絕帶傷起飛的飛行前檢查

在你武裝推進器之前，跑一個飛行前檢查（pre-flight），斷言矩陣所依賴的每一條連線都在。要吵著失敗、要在碼頭上失敗——不要等到 2 公里外才失敗：

preflight.sh bash
#!/usr/bin/env bash
set -euo pipefail
fail=0
check() { if eval "$2"; then echo "  OK  $1"; else echo "FAIL  $1"; fail=1; fi; }

echo "== USV pre-flight =="
check "GPS symlink present"   '[ -e /dev/gps ]'
check "IMU symlink present"   '[ -e /dev/imu ]'
check "can0 is UP"            '[ "$(cat /sys/class/net/can0/operstate)" = up ]'
check "eth0 carrier (LiDAR)"  '[ "$(cat /sys/class/net/eth0/carrier)" = 1 ]'
check "wlan0 has a route"     'ip route show default | grep -q wlan0'
check "LiDAR answering"       'timeout 2 nc -uz 192.168.10.10 2368'
# CAN traffic seen in 1 s? (Ch 10 candump)
check "thruster CAN traffic"  'timeout 1 candump -n 1 can0 >/dev/null 2>&1'

[ "$fail" -eq 0 ] && { echo "PRE-FLIGHT PASS"; exit 0; } \
                  || { echo "PRE-FLIGHT FAIL — do not arm"; exit 1; }
🏭 真實案例——健康狀態取自 /sys 與 /proc，不是用猜的

即時健康監測讀的就是你在第 02 章逛過的那幾個核心介面：/sys/class/net/<if>/operstate 與 carrier 看連結狀態、/sys/class/net/can0/can/state 看 CAN 錯誤狀態、/proc/interrupts 確認光達網卡的 IRQ 真的有在觸發（第 07 章）。沒有啟發式臆測——核心早就知道真相，而且把它攤成檔案。你的監測器，就是一個對虛擬檔案（virtual file）做廉價 read() 的迴圈。

7整門課，凝在一個畫面裡

退一步看。總整理其實就是把這門課照著依賴順序重讀一遍。每一章都是位元組從水面走到你的控制迴圈、再走回推進器這條路上所經過的一層：

章路徑中的這一層它替 USV 貢獻了什麼

8換你了

Q1 介面角色指派
你要加一台 4K 船首相機，把約 30 Mbit/s 的 H.264 串流給操作員。依照我們指派的角色，該由哪個介面來載送它？
  • WiFi（wlan0）——它是通往操作員的連線，所以把影像放那裡。
  • 在船上走乙太網（eth0），任何 WiFi 跳接之前先轉碼／抽稀——沉重媒體屬於全雙工的交換式連線，絕不能以原始形式放上共享的 WiFi 媒介。
  • CAN（can0）——它本來就是裝置的場域匯流排。
  • USB 序列（/dev/cam）——給它一個穩定的 udev 名稱然後串流出去。
30 Mbit/s 超過 WiFi 現實的爭用上限（約 20 Mbit/s），會把控制連線餓死。沉重媒體在船上要走乙太網；只有經過大幅壓縮、限速的預覽才應該碰 WiFi。CAN 頂多到 500 kbit/s、USB 序列 <1 Mbit/s——兩者都差了好幾個數量級。
Q2 算出光達的匯流排負載
光達每秒發出 $6.0\times10^{5}$ 個點，每點 16 位元組（僅負載）。它的負載吞吐量是多少 Mbit/s？（提示：bytes/s × 8，再 ÷ $10^{6}$。）
Mbit/s
$6.0\times10^{5}\times16 = 9.6\times10^{6}$ B/s。$\times 8 = 7.68\times10^{7}$ bit/s $= 76.8$ Mbit/s。在一條 1 Gbit/s 連線上（扣掉封裝後約 944 Mbit/s 可用）那是約 8% 使用率——這正是為什麼這條串流走乙太網、而在 WiFi 上根本不可能。
Q3 動手做——一個 systemd unit + 飛行前閘門
替 sensor-manager 寫一個 systemd unit，要做到：（a）只在 WiFi 連線真的可路由之後才啟動，（b）要求 can0 先被拉起來，（c）除非 preflight.sh 以 0 結束否則拒絕啟動。然後說明為什麼 Wants=network-online.target 很重要。
提示
用 After= 做排序、Requires=／Wants= 做依賴、ExecStartPre= 做那道閘門。記得單靠 After 永遠不會等網路——你還必須 Wants 那個 wait-online target。
看一個解答
/etc/systemd/system/sensor-manager.service bash
[Unit]
Description=USV unified sensor-manager (epoll loop)
# Ordering: only after a routable network AND can0 is up
After=network-online.target can0.service
Wants=network-online.target        # pulls in systemd-networkd-wait-online
Requires=can0.service              # if CAN fails to come up, do not start

[Service]
Type=simple
# Pre-flight gate: nonzero exit here blocks the service from starting
ExecStartPre=/usr/local/bin/preflight.sh
ExecStart=/usr/local/bin/sensor-manager
Restart=on-failure
RestartSec=2

[Install]
WantedBy=multi-user.target

為什麼要 Wants=network-online.target：After=network-online.target 只是把這個 unit 排在那個 target 達成之後，但它並不會讓那個 target 被達成。那個 target 只有在某個 Wants／Requires 它的東西把它拉進開機交易（boot transaction）時才會被拉進來（這又會啟動 systemd-networkd-wait-online.service，也就是真正會阻塞到某張介面起來為止的那個東西）。少了 Wants，network-online.target 可能在 WiFi 還在連線（associating）時就瞬間達成，sensor-manager 就啟動了，它在 wlan0 上的 connect() 失敗，於是你就拿到了那個經典的「手動就好好的、開機就壞掉」。那道 ExecStartPre 閘門則是同一條保險帶外的另一道：就算每一個依賴都滿足了，一條故障的實體連結（沒有 GPS 符號連結、光達載波掛了、沒有 CAN 流量）會以非零結束，systemd 便拒絕啟動，讓推進器保持解除武裝。

🔑 本章回顧

機器人的 I/O 就是一張資料流圖：每個來源都對應到 實體埠 → 匯流排 → 驅動 → fd。依介面的本性去指派——WiFi=控制/遙測、乙太網=沉重感測器、USB/序列=低速率配 udev 名稱、CAN=致動器——而且永遠別讓感測器網卡偷走預設路由。系統初始化鏈以持久設定的形式活在磁碟上：裝置樹 + 韌體 + udev 規則 + netplan + 受 network-online.target 與一個 can0 unit 把關的有序 systemd unit。一個 epoll 迴圈讀盡每一個 fd，並從一個 timerfd 驅動 20 Hz 的節拍，把第 01、06、08、09、10、11 章綜合在一起。證明頻寬預算塞得下每一條匯流排，把每一種故障（WiFi 中斷、USB 拔除、CAN bus-off、連結斷開）對應到一個事先規劃好的降級模式，並把起飛擋在一個直接從 /sys 讀取真相的飛行前檢查後面。憑記憶畫出 §1 的地圖，你就擁有了整門課。
