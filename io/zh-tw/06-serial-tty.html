序列埠與 TTY 層：怎麼把 GPS 的位元組無延遲地讀出來

序列埠（serial port）看起來像箱子裡最單純的裝置——兩根線、一個鮑率（baud rate）。但在你的 read() 與矽晶片之間，坐著一整層 TTY，一個為電傳打字機終端機（teletype terminal）打造、已有 40 年歷史的子系統；它預設就會把你的二進位感測器串流搞爛，再添上數十毫秒的延遲。本章要教你怎麼讓一個 UART 表現得像機器人真正需要的那種笨笨的位元組管線。

⏱ 約 40 分鐘🧠 UART + termios⚡ 即時訊框模擬∑ 鮑率數學

1UART 訊框：一個位元組如何沿著線送出去

① 直覺 先把畫面建起來

UART（通用非同步收發器，Universal Asynchronous Receiver/Transmitter）送位元組時沒有時脈線。這正是「非同步」的意思——也是整套把戲的關鍵。兩端事先約好一個速度（鮑率），接收端再靠每一個位元組的訊框形狀，逐位元組重新同步。沒有共享時脈會漂移，所以訊框本身就是同步機制。

閒置時線路維持高電位（邏輯 1）。要送一個位元組，發送端把線路拉低剛好一個位元週期——那道下降邊緣就是起始位元（start bit），是給接收端的「醒醒，有位元組要來了」訊號。接著 8 個資料位元由 LSB 先送，一個可選的同位位元（parity bit），最後線路被拉回高電位維持一到兩個停止位元（stop bit），回到閒置。鎖定起始邊緣之後，接收端會在每個後續位元的視窗中央取樣。

💡 為什麼閒置高、起始低

一條斷掉或沒插的線會浮接——而上拉（pull-up）的線會讀成閒置（1），而不是源源不絕的垃圾起始位元。起始位元是一個跳變（1→0），即使兩端時脈差個幾個百分點也偵測得到；這正是為什麼 UART 能容忍約 ±3% 的鮑率不匹配，而一條死掉的線看起來就只是沉默。

同位是最廉價的錯誤檢查：挑一個位元，讓 1 位元的總數（資料＋同位）為偶數（偶同位）或奇數（奇同位）。它能抓出任何單一位元翻轉，但對兩個翻轉視而不見——對抗突發性雜訊毫無用處，這正是為什麼幾乎所有機器人鏈路都跑無同位，再把完整性檢查推到真正的校驗碼上（NMEA 的 *hh、二進位協定裡的 CRC）。簡寫 8N1 意思是 8 個資料位元、No（無）同位、1 個停止位元——近乎通用的預設值。

④ 實際案例 這就跑在 USV 上
🏭 實際案例——那個「掉字元」的 GPS

在 Jetson AGX Orin 的 USV 上，一顆 u-blox GPS 以 8N1、115200 鮑率串流 NMEA。一份「定位資料亂碼」的現場回報，原因竟是接收端被設成了偶同位：GPS 的同位位元被當成第 9 個資料位元解讀，使每個位元組都移了位。線路沒問題；錯的是其中一端的訊框格式。訊框不匹配是序列埠開機調試的頭號 bug，排在鮑率不匹配之前。

2鮑率、位元時間，與你拿不到的那 80%

② 數學 推導它

鮑率是每秒符號數；在普通 UART 上一個符號就是一個位元，所以線上的鮑率＝位元/秒。送一個位元的時間，就是它的倒數：

$$ t_\text{bit} = \frac{1}{\text{baud}}, \qquad t_\text{bit}(115200) = \frac{1}{115200}\,\text{s} \approx 8.68\ \mu\text{s} $$

但你沒辦法每個位元組送滿 8 個有用位元——每個訊框都帶著額外負擔（overhead）。設每個訊框有 $N_\text{frame}$ 個位元（起始＋資料＋同位＋停止），可達到的位元組速率與線路效率為：

$$ N_\text{frame} = 1 + 8 + p + s,\qquad \dot B = \frac{\text{baud}}{N_\text{frame}},\qquad \eta = \frac{8}{N_\text{frame}} $$

對 8N1，$N_\text{frame} = 1 + 8 + 0 + 1 = 10$，所以 $\eta = 8/10 = 80\%$、$\dot B = 115200/10 = 11{,}520$ 位元組/秒。這個漂亮的 10 倍係數，正是大家挑 8N1 的原因：鮑率除以十，心算就得出位元組/秒。改成 8E2（偶同位、2 停止位元），$N_\text{frame}=12$，效率掉到 $8/12 \approx 67\%$——你才剛為了一個孱弱的錯誤位元，白送掉六分之一的頻寬。

反過來算：若某感測器以 200 Hz 吐出 64 位元組的二進位紀錄，那就是 $64 \times 200 = 12{,}800$ 位元組/秒的有效酬載（payload）。在 8N1 下，線上需要 $12{,}800 \times 10 = 128{,}000$ 位元/秒——115200 鮑率塞不下（它頂多到 11,520 B/s）。你得跳到 230400 或 460800 鮑率。永遠要把訊框額外負擔算進去，絕不能只看原始酬載。

3看見訊框——也看數字跟著動

拖動位元組數值，看資料位元翻轉（記住：LSB 先送，所以位元 0 是緊接在起始之後的那一格）。切換同位與停止位元，讀即時讀數：位元時間、訊框總位元數、訊框時間、最大位元組速率、效率。這就是上面的數學，化成像素。

📡 單一 UART 訊框——鮑率、位元組、同位、停止
🔑 該注意的地方

在 9600 鮑率下，一個位元約 104 µs，整個 8N1 訊框約 1.04 ms——慢到幾個位元組的延遲就是毫秒等級。飆到 1,000,000 鮑率，同一個訊框只剩 10 µs；現在瓶頸不再是線路，而是核心多久才把位元組交給你。加一個同位位元或多一個停止位元，看著效率跌破 80%——每一個非資料位元都是你付了錢卻用不到的頻寬。

4TTY 堆疊：為什麼你的位元組會被改寫

① 直覺 你沒要求的那一層

當你 open("/dev/ttyUSB0") 時，你並不是在跟 UART 對話。你是在跟 TTY 層對話，它由三塊疊起來的東西組成：tty 核心（/dev/tty* 字元裝置與它的緩衝區）、行規（line discipline）（位元組處理策略），以及驅動程式（driver）（把位元組推給硬體的晶片專屬程式碼）。會咬機器人一口的，正是行規。

flowchart TD A["你的應用
read/write"] --> C["tty 核心

N_TTY：標準 / 原始"] LD --> DRV["uart 驅動

預設的行規 N_TTY 處於標準模式（canonical mode）——它假裝你的序列埠是個有人在打字的終端機。它逐行緩衝輸入直到看見換行符，讓使用者退格，把字元回顯（echo）回去，展開定位字元（tab），並把某些位元組當成控制用：0x03（Ctrl-C）會發出訊號，0x11/0x13（XON/XOFF）會暫停串流，0x04（Ctrl-D）代表 EOF。對登入 shell 來說這恰到好處。但對二進位的 GPS 串流就是場災難：酬載裡的一個 0x03 會殺掉你的行程，一個 0x13 會悄悄停掉資料流，而你在某個 0x0A 剛好飄過來之前，連一個位元組都看不到。

⚠️ 陷阱——標準模式會吃掉二進位資料

典型症狀：明明裝置顯然在發送（你在 dmesg 或示波器上看得到流量），你的讀取端卻永遠卡死。是標準模式把每個位元組都扣著當人質，直到出現換行符。修法不是加大緩衝區——而是切到原始模式（raw mode），讓行規變成透明的直通管線，位元組就是位元組。

5termios：用 C 設定序列埠

③ 程式碼 每一行都讀

每一項 TTY 設定都住在一個 struct termios 裡：四個旗標字（c_iflag 輸入、c_oflag 輸出、c_cflag 控制、c_lflag 本地/行）外加一個控制字元陣列 c_cc[] 與鮑率。你先 tcgetattr() 讀出目前設定，編輯這個結構，再 tcsetattr() 寫回去。整套「把它變成原始位元組管線」的動作，被一個輔助函式 cfmakeraw() 一手包辦：它一口氣清掉標準/回顯/訊號旗標與輸入/輸出轉換。

serial_open.c C
#include <termios.h>
#include <fcntl.h>
#include <unistd.h>

int serial_open(const char *path, speed_t baud) {
    /* O_NOCTTY: don't let this tty become our controlling terminal
       (a stray Ctrl-C on the line must not signal us). */
    int fd = open(path, O_RDWR | O_NOCTTY);
    if (fd < 0) return -1;

    struct termios tio;
    if (tcgetattr(fd, &tio) < 0) { close(fd); return -1; }

    cfmakeraw(&tio);              /* raw mode: no canonical, no echo, no signals */
    cfsetispeed(&tio, baud);     /* input baud  */
    cfsetospeed(&tio, baud);     /* output baud */

    tio.c_cflag |= (CLOCAL | CREAD);   /* ignore modem ctrl lines; enable receiver */
    tio.c_cflag &= ~CSTOPB;            /* 1 stop bit (clear = 1, set = 2)          */
    tio.c_cflag &= ~PARENB;            /* no parity  ->  this is the N in 8N1       */
    tio.c_cflag &= ~CRTSCTS;           /* no hardware flow control (set to enable)  */

    /* Blocking read policy: return as soon as >=1 byte is available,
       but wait at most 100 ms (VTIME is in deciseconds). */
    tio.c_cc[VMIN]  = 0;
    tio.c_cc[VTIME] = 1;

    tcflush(fd, TCIFLUSH);                       /* drop stale bytes in the kernel buffer */
    if (tcsetattr(fd, TCSANOW, &tio) < 0) {      /* apply immediately */
        close(fd); return -1;
    }
    return fd;
}
🛠️ 用 speed_t 常數來設速度

鮑率不是隨便填的整數：cfsetispeed() 吃的是 speed_t 代號，例如 B115200 或 B460800，由驅動程式對映成除頻值。標準速率有對應常數；至於不在清單上的速率（例如 250000），你得走 termios2 / BOTHER 的 ioctl 路徑。cfmakeraw() 是 POSIX 的簡寫——它等同於清掉 c_lflag 裡的 ICANON|ECHO|ISIG、c_iflag 裡的 IXON|ICRNL，以及 c_oflag 裡的 OPOST。

決定你延遲的兩個欄位是 VMIN 與 VTIME。它們組成一個小型狀態機，決定一個阻塞式 read() 何時返回：

VMINVTIMEread() 在何時返回…適用於

6流量控制：叫發送端等一等

如果接收端跟不上，位元組就會溢出 FIFO 而遺失——而且 UART 溢位是無聲的（你拿到的是一個被加一的錯誤計數器，而不是 read() 上的錯誤）。流量控制（flow control）就是線路上的反壓（backpressure）。兩種口味：

  • 硬體（RTS/CTS）：兩根額外的線。接收端取消宣告 RTS（「我還沒準備好」）；發送端盯著 CTS 並暫停。它是頻外（out-of-band）的——它從不碰你的資料——而且能在一個位元組時間內反應。在 c_cflag 裡用 CRTSCTS 啟用。對高速率而言這是正確選擇，前提是那兩根線有接好。
  • 軟體（XON/XOFF）：頻內（in-band）的控制位元組 0x11（XON，恢復）與 0x13（XOFF，暫停），用 IXON/IXOFF 啟用。不用額外的線——但那兩個位元組值現在被保留了，永遠不能出現在你的酬載裡。對二進位資料來說這是致命的，所以機器人鏈路會把 XON/XOFF 關掉，改用 RTS/CTS，或乾脆跑一個接收端永遠排得完的速率。
⚠️ 陷阱——對二進位串流開 XON/XOFF

把 IXON 留著開（它常常預設就是開的），那麼一旦你的二進位酬載含有一個 0x13 位元組，核心就會悄悄停止給你資料，直到出現一個 0x11。鏈路「掛住」，沒有任何錯誤。cfmakeraw() 會幫你清掉 IXON——這又是一個永遠從原始模式起步的理由。

7延遲：你沒編進預算的那 16 ms

④ 實際案例 200 Hz 無延遲

這就是毀掉控制迴圈的那個陷阱。線路很快——115200 下的一個 8N1 位元組是 86.8 µs——但從「位元組在晶片 FIFO 裡」到「位元組在你緩衝區裡」的這段路，有兩個延遲來源大到讓它相形見絀：

  1. FTDI / USB 序列的延遲計時器（latency timer）。USB 序列轉接器（FTDI、CP210x）不會每個位元組就中斷主機一次——它們把位元組批次打包成 USB 封包，再依計時器一次沖出。FTDI 的預設是 16 ms。所以一句 1 位元組的 GPS 語句，可能在轉接器裡待上 16 ms 才跨過 USB。對一個 200 Hz 迴圈（週期 5 ms）來說，那是三個迴圈迭代的純死時間。
  2. 核心端的 VMIN/VTIME。要求 VMIN=64，核心就不會叫醒你，直到累積了 64 個位元組——對吞吐量很棒，但若你的紀錄是 200 Hz 下的 20 位元組就糟透了。
🏭 實際案例——修掉 AGX Orin 上的 IMU 延遲

一顆透過 USB 序列橋接進 Orin 的 IMU，在 EKF 裡顯出頑固的約 16 ms 延遲，不管程式怎麼調都消不掉。修法是一次 sysfs 寫入——把 FTDI 延遲計時器從 16 ms 降到 1 ms：

terminal bash
# find the device, then read/write its latency_timer (milliseconds)
cat /sys/bus/usb-serial/devices/ttyUSB0/latency_timer    # -> 16
echo 1 | sudo tee /sys/bus/usb-serial/devices/ttyUSB0/latency_timer

# make it persistent with a udev rule (see Ch 03):
# SUBSYSTEM=="usb-serial", ATTR{latency_timer}="1"

搭配 VMIN=0, VTIME=1（第一個位元組就醒、100 ms 安全逾時），最壞情況的延遲從約 16 ms 降到約 1 ms——不到 5 ms 迴圈週期的四分之一。晶片內建的 UART（下一節）根本沒有這個計時器。

USB 序列埠的通用延遲模型，把轉接器的沖出視窗加到線路時間上：

$$ t_\text{latency} \approx t_\text{flush} + n\cdot t_\text{frame}, \qquad t_\text{flush} = \text{latency\_timer} $$

在 16 ms 對上 1 ms 時，latency_timer 這一項就是整個預算；訊框時間在它旁邊只是雜訊。把計時器調低，你是拿一點點 USB 額外負擔（更多、更小的封包）去換取反應靈敏度——對控制迴圈而言，這正是該做的取捨。

8ttyUSB* vs ttyTHS*：兩種大不相同的序列埠

在 Jetson 上你會看到兩個家族的序列裝置，它們的行為差異大到值得在意：

/dev/ttyUSB*/dev/ttyTHS*
它是什麼USB 序列橋接（FTDI、CP210x、CH340）Orin 晶片內建 UART（Tegra 高速）

晶片內建的 ttyTHS 埠沒有 USB 這一跳、也沒有批次計時器，所以對時序緊湊的感測器而言，它們是低延遲的選擇——但它們是 3.3 V 邏輯電位的腳位，凡是要接到板外的東西，電位轉換器（level shifter）或收發器（transceiver）就是必備的。USB 轉接器方便又可熱插拔，卻繼承了 USB 的延遲下限，而它們的 ttyUSBn 編號取決於插入順序——這正是第 03 章的 udev by-id 符號連結存在的原因。還有一條路上的規矩：核心的開機主控台（console）可能停駐在某個 UART 上；若某個埠在 /proc/cmdline 裡是 console=ttyTHSx，核心就在往那裡寫日誌垃圾，它就不是你能用的。

9換你了

Q1 觀念檢查
為什麼說 8N1 只有 80% 效率，就算用一條完美、零雜訊的線也一樣？
  • 行規每五個位元組就丟掉一個。
  • 每個 8 位元的位元組搭在一個 10 位元的訊框裡（1 起始 + 8 資料 + 1 停止），所以線上只有 8/10 的位元是酬載。
  • USB 序列轉接器因為延遲計時器而丟掉了 20% 的位元組。
  • 同位消耗了 20% 的頻寬。
效率是 $\eta = 8/N_\text{frame}$。對 8N1，$N_\text{frame} = 1+8+0+1 = 10$，所以 $\eta = 8/10 = 80\%$——起始與停止位元是訊框額外負擔，不是資料，而且（注意）8N1 根本沒有同位。加上偶同位（8E1）會變成 11 個訊框位元，$\approx 73\%$。
Q2 算一算
某個埠跑 8N1、115200 鮑率。最大可持續吞吐量是多少位元組每秒？（提示：$\dot B = \text{baud}/N_\text{frame}$，且 8N1 的 $N_\text{frame}=10$。）
B/s
$\dot B = 115200 / 10 = 11{,}520$ 位元組/秒。每個訊框是 $10 \times t_\text{bit} = 10 \times 8.68\,\mu\text{s} = 86.8\,\mu\text{s}$，而 $1/86.8\,\mu\text{s} \approx 11{,}520$ 訊框/秒——每個訊框一個位元組。8N1 好用的心算捷徑就是「鮑率 ÷ 10」。

寫出能以 115200 8N1、原始模式開啟 /dev/ttyUSB0 的 C，且讀取要在第一個位元組就返回、但約 100 ms 後逾時。說出設定原始模式與鮑率的那三個呼叫，以及設定讀取時序的那兩個欄位。
提示
cfmakeraw() 處理原始旗標的工作；cfsetispeed()/cfsetospeed() 設鮑率；VMIN=0, VTIME=1 意思是「≥1 個位元組就返回，或過 1 個十分之一秒後返回」。開檔時帶 O_NOCTTY，這樣線上一個亂飄的 Ctrl-C 就不能對你發訊號。
看一個解答
raw_open.c C
#include <termios.h>
#include <fcntl.h>
#include <unistd.h>

int raw_open(const char *path) {
    int fd = open(path, O_RDWR | O_NOCTTY);
    if (fd < 0) return -1;

    struct termios tio;
    tcgetattr(fd, &tio);
    cfmakeraw(&tio);                 /* raw: no canon/echo/signals/translation */
    cfsetispeed(&tio, B115200);     /* baud in */
    cfsetospeed(&tio, B115200);     /* baud out */
    tio.c_cflag |= (CLOCAL | CREAD);
    tio.c_cflag &= ~CSTOPB;         /* 1 stop  */
    tio.c_cflag &= ~PARENB;         /* no parity -> the N in 8N1 */
    tio.c_cc[VMIN]  = 0;            /* return on >=1 byte... */
    tio.c_cc[VTIME] = 1;           /* ...or after 100 ms */

    tcflush(fd, TCIFLUSH);
    tcsetattr(fd, TCSANOW, &tio);
    return fd;
}

那三個「塑形序列埠」的呼叫是 cfmakeraw、cfsetispeed、cfsetospeed；那兩個讀取時序欄位是 c_cc[VMIN] 與 c_cc[VTIME]。清掉 PARENB + CSTOPB 給你 8N1 的 N1（8 個資料位元是 cfmakeraw 透過 CS8 的預設值）。

🔑 本章回顧

一個 UART 訊框是閒置高、一個起始位元、8 個 LSB 先送的資料位元、可選的同位、1–2 個停止位元；8N1 是每個位元組 10 個線上位元，所以效率是 80%、位元組速率是鮑率 ÷ 10。Linux 把序列埠包進 TTY 堆疊（核心 → 行規 → 驅動）；預設的標準模式會編輯並緩衝你的位元組，所以永遠透過 cfmakeraw() + cfset{i,o}speed() 切到原始模式，並用 VMIN/VTIME 在延遲與批次之間取捨。對二進位資料把 XON/XOFF 關掉；偏好 RTS/CTS。USB 序列上真正的延遲殺手是 latency_timer（預設 16 ms——把它降到 1 ms）；晶片內建的 ttyTHS 埠完全跳過 USB，延遲最低。下一章：資料究竟如何移動——中斷、DMA 與 MMIO。
