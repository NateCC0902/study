驅動程式、模組與韌體：擁有硬體的那段程式碼

在第 02 章，一個裝置以檔案的樣子出現；在第 03 章，udev 給了它一個穩定的名字。但是誰把對暫存器的戳弄變成一顆能用的感測器？是驅動程式（driver）。本章談的是驅動程式碼住在哪裡、它怎麼被載入、為什麼它有時載入了卻苦苦哀求一個它找不到的韌體（firmware）二進位檔，以及為什麼昨天還能用的廠商模組在核心（kernel）升級後就拒絕載入。讀完之後，你會像讀堆疊追蹤（stack trace）一樣讀懂 dmesg 裡的一次綁定（bind）。

1驅動程式能住的兩個地方

① 直覺 先把畫面建起來

驅動程式不過就是懂某一個硬體私有語言的那段程式碼——哪一個暫存器代表「開始一次轉換」、哪一個位元代表「資料就緒」、一個中斷（interrupt）如何對應到位元組。真正唯一的問題是：這段程式碼跑在系統呼叫（syscall）邊界的哪一側。

核心驅動程式（kernel driver）跑在核心裡，在 ring 0。它能碰實體記憶體、接收中斷、直接設定 DMA。當它出錯時，可以把整台機器拖垮。使用者空間驅動程式（userspace driver）則是一支普通程式，透過一層薄薄的通用核心墊片（spidev、i2c-dev、libusb、UIO/VFIO）去戳硬體。它弄不掛核心，你可以用 gdb 接上去除錯，而且不必重新編譯任何東西就能出貨——但每一次暫存器存取都是一個系統呼叫，所以它比較慢，也無法在下半部（bottom half）服務硬中斷。

💡 唯一的那個決定

需要原始速度、真正的中斷、或 DMA？用核心驅動程式。想快速迭代、在使用者空間（user space）除錯、而且絕不為了一顆跑 100 Hz 的感測器去冒核心恐慌（kernel panic）的風險？用走 i2c-dev/spidev/libusb 的使用者空間驅動程式。大多數機器人感測器在使用者空間都過得很愉快；網路卡和 IMU 的 DMA 引擎可不是。

2模組的一生：init、exit，與兩者之間的縫隙

② 數學 把生命週期看成狀態機

核心模組（一個 .ko 檔——「kernel object」）是你可以在執行期載入、卸載的驅動程式碼，而不必把它烤進核心映像裡。每個模組都剛好有兩個儀式函式：一個在載入時跑，一個在卸載時跑。其餘一切都掛在它們身上。

usv_baro.c C
#include <linux/module.h>
#include <linux/init.h>

static int __init baro_init(void)
{
    pr_info("usv_baro: loaded, registering driver\n");  /* lands in dmesg */
    return 0;            /* return < 0 to ABORT the load (e.g. -ENODEV) */
}

static void __exit baro_exit(void)
{
    pr_info("usv_baro: unloaded\n");
}

module_init(baro_init);      /* called by insmod / modprobe       */
module_exit(baro_exit);      /* called by rmmod  / modprobe -r    */

MODULE_LICENSE("GPL");                 /* non-GPL taints the kernel */
MODULE_AUTHOR("USV team");
MODULE_DESCRIPTION("Example barometer driver");
MODULE_VERSION("1.0");

細微之處在於：module_init 不代表「現在就跟我的裝置講話」。對一個匯流排（bus）驅動程式來說，init 只是把驅動程式向某條匯流排（PCI、USB、I²C…）註冊進去。之後核心會再呼叫你的 .probe()——對該匯流排列舉（enumeration）出的每一個相符裝置各呼叫一次。Init 可以成功而 probe 卻從不觸發——模組載入了，卻沒綁定到任何東西。那個區別正是 §6 的全部重點。

🛠️ 四個動詞

lsmod 列出已載入的模組及其使用計數（use count）（你不能對 refcount > 0 的模組做 rmmod）。modinfo foo.ko 傾印它的中介資料（metadata）——授權、參數、相依性，以及 vermagic 字串。insmod 載入一個確切的檔案。modprobe foo 則依名稱、從系統模組樹中載入 foo 以及它所相依的一切。

3insmod 對 modprobe：相依性解析

③ 程式碼 看相依圖被解開

模組之間會呼叫彼此匯出的符號（symbol），於是它們構成一張相依圖。insmod /path/to/foo.ko 只依絕對路徑載入那一個檔案——如果 foo 需要 bar 的符號而 bar 還沒載入，insmod 就會以 Unknown symbol in module 失敗。它不做搜尋、也不做解析。它是那把鈍器。

modprobe foo 才是文明的工具。它會在 /lib/modules/$(uname -r)/modules.dep 裡查 foo、走過整條相依鏈，並以正確順序——由底層往上——把一切載入。那張相依映射表由 depmod 產生，它會掃描目前核心模組目錄底下的每一個 .ko，記錄誰匯出、誰需要每一個符號。

解析一個廠商 WiFi 模組 bash
# After dropping new .ko files into /lib/modules/$(uname -r)/extra/,
# rebuild the dependency + alias maps for THIS kernel:
sudo depmod -a

# modprobe now knows moal depends on mlan and loads both, in order:
sudo modprobe moal mod_para=nxp/wifi_mod_para.conf

# What insmod would have forced you to do by hand, in order:
sudo insmod /lib/modules/$(uname -r)/extra/mlan.ko
sudo insmod /lib/modules/$(uname -r)/extra/moal.ko mod_para=nxp/wifi_mod_para.conf

# Inspect before loading — never trust a blob you didn't build:
modinfo moal | grep -E '^(filename|version|vermagic|depends|parm):'
⚠️ 忘了 depmod，弄壞 modprobe

如果你把一個新鮮的 .ko 複製進模組樹卻略過 depmod -a，modprobe 會說 FATAL: Module foo not found——就算那個檔案就擺在那兒。modprobe 讀的是 modules.dep，不是目錄。沒有 depmod，就沒有索引，就找不到。

兩個工具最終都會呼叫同一個系統呼叫——finit_module(2)（或較舊的 init_module(2)）——把 .ko 的一個檔案描述子（file descriptor）交給核心並觸發 module_init。modprobe 不過就是 insmod 再加上一次圖的走訪與一次別名（alias）查找。

4模組參數與 /etc/modprobe.d

③ 程式碼 不重新編譯就能設定

模組以參數（parameter）的形式對外露出可調項，在 C 裡用 module_param() 宣告。你可以在載入時於命令列設定它們、在執行期從 /sys/module/<name>/parameters/ 底下讀它們，而對於那些可寫的參數——透過寫入那個 sysfs 檔案來即時改動它們。

C 裡的模組參數 C
static int sample_rate_hz = 100;
module_param(sample_rate_hz, int, 0644);   /* 0644 => appears in sysfs, root-writable */
MODULE_PARM_DESC(sample_rate_hz, "barometer sample rate in Hz");
/* modprobe usv_baro sample_rate_hz=200
 * cat /sys/module/usv_baro/parameters/sample_rate_hz   -> 200            */

每次開機都手動打參數，正是你會忘掉它們的原因。把它們持久化在 /etc/modprobe.d/*.conf。四個指示詞最要緊：

指示詞作用
options moal mod_para=…模組每次載入時的預設參數
blacklist nouveau別自動載入這個模組（不會被 udev 挑中）
alias usb:v1286p… moal把一個硬體 ID／別名對應到一個模組名稱
install foo /bin/false執行一個命令來取代載入（一個硬性封鎖）
/etc/modprobe.d/usv-wifi.conf bash
# Vendor params persist across reboots, and udev picks them up on hotplug.
options moal mod_para=nxp/wifi_mod_para.conf
# Keep the in-tree mwifiex out of the way so our out-of-tree moal binds instead:
blacklist mwifiex
blacklist mwifiex_pcie
⚠️ blacklist 不是一個總開關

blacklist 只擋下透過別名解析的自動載入。一個直接的 modprobe mwifiex，或另一個模組的相依性，仍然會把它拉進來。要真正禁止載入，用 install mwifiex /bin/true。

5韌體載入器與錯誤 -2

① 直覺 驅動程式是大腦，韌體是性格

不少裝置出廠時就是一顆裡頭沒裝任何程式的笨微控制器。Linux 驅動程式知道怎麼跟這顆晶片講話，但晶片本身需要一段二進位資料（blob）——它的韌體——在 probe 時被推進去，它才會做任何有用的事。WiFi 無線電、某些相機、GPU，以及許多 SoC 周邊都正是這樣運作。驅動程式請求那個二進位檔；核心的韌體載入器（firmware loader）在磁碟上找到那個檔案並把它遞過去。

② 數學 載入器去哪裡找

驅動程式呼叫 request_firmware(&fw, "nxp/sd8997_uapsta.bin", dev)。核心會依固定順序搜尋一組路徑，去找那個相對名稱——第一個命中者勝出，而 updates/ 那幾棵樹排在裸的發行版樹之前，正是為了讓它們能蓋過後者：

$$ \underbrace{\texttt{firmware\_class.path}}_{\text{optional, custom}} \;\to\; \texttt{/lib/firmware/updates/}\langle\texttt{uname -r}\rangle\texttt{/} \;\to\; \texttt{/lib/firmware/updates/} \;\to\; \texttt{/lib/firmware/}\langle\texttt{uname -r}\rangle\texttt{/} \;\to\; \underbrace{\texttt{/lib/firmware/}}_{\text{distro default, last}} $$

如果檔案被找到，呼叫回傳 0，而 fw->data / fw->size 就指向那些位元組。如果它不見了，呼叫回傳一個負的 errno，而驅動程式通常會記下那條如今惡名昭彰的訊息。那個 -2 不是隨機的：它是 -ENOENT，「No such file or directory」——跟 open() 對一個不存在的路徑回傳的，是完全相同的 errno。

dmesg——那個經典畫面 bash
$ dmesg | grep -i firmware
moal: Direct firmware load for nxp/sd8997_uapsta.bin failed with error -2
moal: firmware: failed to load nxp/sd8997_uapsta.bin (-2)
moal: WLAN FW download failed

# Fix: the file just isn't in the search path. Put it there:
$ ls /lib/firmware/nxp/sd8997_uapsta.bin        # -> No such file
$ sudo install -D -m644 sd8997_uapsta.bin /lib/firmware/nxp/sd8997_uapsta.bin
$ sudo rmmod moal mlan && sudo modprobe moal   # re-probe, re-request firmware
🏭 真實案例——AGX Orin 上的 USV WiFi

在 Jetson AGX Orin 上，WiFi M.2 卡跑的是 NXP 的 mlan/moal 樹外（out-of-tree）堆疊。模組乾淨地載入了（init 成功），SDIO/PCIe 裝置被列舉，probe 開始——接著 moal 為 nxp/sd8997_uapsta.bin 呼叫 request_firmware()。如果你在燒錄映像時忘了把那個二進位檔備妥到 /lib/firmware/nxp/，你就會得到一個出現了（ip link 會顯示 mlan0）卻永遠不會關聯（associate）的網路介面——日誌裡是錯誤 -2。驅動程式匹配上了；裝置只是沒裝進大腦。這是每一顆嗜韌體周邊的通用模式，而不是一個一次性的 bug。

6樹內、樹外、kABI 與 DKMS

② 數學 為什麼一次核心升級就弄壞你的驅動程式

一個樹內（in-tree）驅動程式住在核心原始碼裡，是跟著核心一起被建置與版本化的——升級核心，驅動程式就一起來，永遠匹配。一個樹外（out-of-tree）驅動程式（像 moal 這種廠商二進位檔）則是對著某一個特定核心建置分開編譯的，而那正是所有痛苦的源頭。

核心並不對模組承諾一個穩定的二進位介面。每個 .ko 都帶著一個 vermagic 字串，以及它所用核心符號的 CRC 校驗和——兩者合起來就是 kABI（核心應用程式二進位介面，kernel Application Binary Interface）。在載入時核心會檢查它們。如果執行中核心的符號 CRC 跟模組當初對著建置的那一份不符，載入就會被拒絕：

$$ \texttt{load ok} \iff \texttt{vermagic}_{module} = \texttt{vermagic}_{kernel} \;\;\wedge\;\; \forall s \in \text{symbols}:\; \mathrm{CRC}_{module}(s) = \mathrm{CRC}_{kernel}(s) $$

vermagic 字串把核心版本、SMP/preempt 旗標，以及編譯器細節綑在一起，例如 5.15.148-tegra SMP preempt mod_unload aarch64。改動其中任何一項——哪怕只是從 5.15.148 到 5.15.149 的修補版號跳動——字串就不同了，於是模組會被拒絕，並附上：

核心升級造成的失敗 bash
$ sudo modprobe moal
modprobe: ERROR: could not insert 'moal': Exec format error
$ dmesg | tail -2
moal: version magic '5.15.148-tegra SMP preempt mod_unload aarch64'
      should be '5.15.149-tegra SMP preempt mod_unload aarch64'
# The module was built for the OLD kernel. It must be REBUILT for the new one.

能擴展的修法是 DKMS（動態核心模組支援，Dynamic Kernel Module Support）。你用一個 dkms.conf 把驅動程式的原始碼註冊一次；此後，每當有新核心被安裝，DKMS 就會自動對著新的標頭檔重新編譯模組並安裝它。沒有 DKMS 意味著：每一次核心升級都會悄悄地讓你落得一台沒有 WiFi 的機器人，直到有人手動把模組重建起來。

🛠️ 一口氣講完 DKMS

dkms add -m moal -v 1.0 註冊原始碼樹，dkms build 對著 /lib/modules/$(uname -r)/build 編譯，dkms install 放下 .ko 並執行 depmod。之後，核心套件升級就會替你觸發重建。用 dkms status 確認。

7使用者空間驅動程式：spidev、i2c-dev、libusb、UIO/VFIO

③ 程式碼 從一支普通行程驅動硬體

你不見得每次都要寫核心 C。核心隨附了通用的墊片驅動程式，把一條匯流排以字元裝置（char device）的形式露給使用者空間，而你的程式就透過 ioctl/read/write 去跑協定：

墊片露出你使用適合
i2c-dev/dev/i2c-Nioctl(I2C_RDWR)I²C 氣壓計/IMU/RTC
spidev/dev/spidevB.Cioctl(SPI_IOC_MESSAGE)SPI ADC、快速感測器
libusb直接操作 USB 裝置函式庫呼叫USB GPS、RealSense
UIO/dev/uioN + sysfsmmap + 透過 read 收 IRQ簡單的 MMIO 周邊
VFIO/dev/vfio/*IOMMU 安全的 MMIO + DMA高速率、DMA、直通
透過 i2c-dev 讀一個暫存器 C
int fd = open("/dev/i2c-1", O_RDWR);
ioctl(fd, I2C_SLAVE, 0x77);                 /* 7-bit address of the baro */
unsigned char reg = 0x00, val;              /* WHO_AM_I register         */
write(fd, &reg, 1);                         /* set the register pointer  */
read(fd, &val, 1);                          /* read it back              */
/* No kernel module to compile. Just a process holding an fd.            */

UIO 與 VFIO 走得更遠：它們讓使用者空間能 mmap 裝置的 MMIO 區域，甚至接收中斷（UIO 把一個 IRQ 變成一個可讀的 fd；VFIO 再加上 IOMMU 保護，讓使用者空間的 DMA 沒辦法亂塗核心記憶體）。DPDK 和許多相機/FPGA 堆疊就是這樣，以近乎核心的速度在使用者空間裡跑一個完整的驅動程式。§1 那個取捨依然成立——除非你 mmap，否則每次存取一個系統呼叫——但對一顆 100 Hz 的氣壓計來說，那個下限是看不見的。

8在 dmesg 裡讀一次綁定

④ 實際案例 跟著這場握手走

當你懷疑有驅動程式問題時，dmesg 就是真相的來源——它是核心對列舉、匹配、probe 與韌體載入的即時旁白。由上往下讀，這次綁定自己會說出自己的故事：

一次成功的綁定，附註解 bash
$ sudo dmesg -w        # -w = follow, like tail -f, while you replug
mmc1: new high speed SDIO card at address 0001          # 1. bus enumerated device
moal: MOAL driver version 1.3.x, mlan version 1.3.x     # 2. module init / registered
moal: SDIO: max_segs=128 max_seg_size=65535            #    driver bound to the device
moal: wlan: Driver loading firmware nxp/sd8997_uapsta.bin  # 3. request_firmware()
moal: WLAN FW is active                                 # 4. firmware accepted
moal: version = SD8997-...                              # 5. device reports its identity
mlan0: associated                                       # 6. usable netdev is up

如果這條鏈在步驟 1 之後就停了，那是你的驅動程式沒匹配——檢查 id 表／別名（modinfo、/etc/modprobe.d）。如果它停在步驟 3 並帶著 error -2，那是韌體檔案不見了（§5）。如果是 modprobe 自己以 Exec format error 失敗，你根本沒走到步驟 2——那是 §6 的 kABI 不符。每一種失敗模式都住在一個特定的步驟上，而 dmesg 會告訴你是哪一個。

🔑 三個命令對應三種失敗

lsmod——它到底有沒有載入？modinfo——它想要什麼（參數、韌體、vermagic）？dmesg——綁定走到哪一步了？把這三個練熟，大多數「感測器沒出現」的工單都能在兩分鐘內解掉。

9換你了

Q1 觀念檢查
你那個樹外的 moal WiFi 模組本來運作得完美無瑕。在 apt upgrade 拉進一個新核心、你也重開機之後，modprobe moal 以 Exec format error 失敗。為什麼？
  • 韌體二進位檔 sd8997_uapsta.bin 在升級時被刪掉了。
  • 模組是對著舊核心建置的；它的 vermagic／符號 CRC（kABI）不再與執行中的核心相符，所以載入被拒絕。
  • 新核心為了安全，一律把樹外模組列入黑名單。
  • 新核心隨附了一個樹內驅動程式，它把 moal.ko 實際移除了。
核心不對模組提供穩定的 kABI。.ko 內嵌一個 vermagic 字串，以及它所匯入符號的 CRC；載入時核心會把它們跟自己的比對。一次版本變動——哪怕只是修補版號跳動——就會改變 vermagic，於是載入以 Exec format error 被拒。修法是對著新核心重建；DKMS 正是在每次核心安裝時把這件事自動化。韌體（錯誤 -2）是另一種、發生得更晚的失敗。
Q2 算一算
你必須在起飛前把韌體二進位檔備妥到 USV 的 eMMC 上。有 7 顆嗜韌體的周邊，每顆各需要一個平均 $2\ \text{MiB}$ 的二進位檔。你的 install 步驟以有效 $1\ \text{MiB/s}$ 寫入 eMMC。把所有二進位檔都備妥要花多少秒？（提示：總位元組數 ÷ 速率。）
s
總量 $= 7 \times 2 = 14\ \text{MiB}$。在 $1\ \text{MiB/s}$ 下，時間 $= 14 / 1 = 14\ \text{s}$。重點不在算術——而在韌體備妥是你映像建置裡一個真實、可量測的步驟，忘了它就會產生 §5 那個無聲的「介面出現了卻永遠不關聯」的錯誤 -2。
Q3 動手做——帶一個參數載入並驗證
以 sample_rate_hz=200 載入 usv_baro 模組，然後證明 (a) 它載入了、且 (b) 那個參數真的生效了。假設 .ko 已經在模組樹裡，但你才剛把它複製進去。
提示
一個剛複製進去的模組，需要先重建相依索引，modprobe 才找得到它。然後在 lsmod 裡查名稱，並從 /sys/module/<name>/parameters/ 讀出即時值。dmesg 確認綁定。
看一個解答
載入 + 驗證 bash
# Rebuild modules.dep so modprobe can resolve the freshly-copied .ko:
sudo depmod -a

# Load it (and any deps) with the parameter:
sudo modprobe usv_baro sample_rate_hz=200

# (a) Confirm it's loaded — name + nonzero size, used-by count:
lsmod | grep usv_baro
#   usv_baro    16384  0

# (b) Confirm the parameter took — read it back from sysfs:
cat /sys/module/usv_baro/parameters/sample_rate_hz
#   200

# Follow the bind in the log:
dmesg | grep usv_baro
#   usv_baro: loaded, registering driver

這三項檢查對應到 §8 的三個命令：lsmod 證明它載入了，讀 /sys/module/.../parameters/ 證明選項抵達了驅動程式，而 dmesg 證明生命週期跑過了。如果你略過了 depmod -a，第二步就會以 FATAL: Module usv_baro not found 失敗——即使檔案確實存在。

🔑 本章回顧

驅動程式要嘛跑在核心裡（快、危險、直接 MMIO/IRQ/DMA），要嘛跑在使用者空間（安全、可除錯，透過 i2c-dev/spidev/libusb/UIO/VFIO）。核心模組有一個 init/exit 生命週期，但 module_init 只註冊——綁定發生在稍後的 .probe()。modprobe 勝過 insmod，是因為它從 modules.dep（由 depmod 重建）解析相依性；參數持久化在 /etc/modprobe.d，並出現在 /sys/module 底下。嗜韌體的晶片會呼叫 request_firmware()；少了二進位檔就是那個經典的 error -2（-ENOENT）。樹外驅動程式會在核心升級時壞掉，因為 kABI（vermagic + 符號 CRC）必須完全相符——DKMS 會自動把它們重建。要診斷以上任何一項，就在 dmesg 裡讀那次綁定：列舉 → 匹配 → probe → 韌體 → 就緒。下一章：這些驅動程式所綁定的裝置，匯流排是怎麼把它們列舉出來的。
