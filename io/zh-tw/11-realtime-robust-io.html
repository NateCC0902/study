即時與穩健 I/O：絕不錯過期限，絕不信任任何一條線

你那個 20 Hz 的航向控制器，每個週期恰好只有 50 ms 來讀感測器、做計算、命令舵——每一個週期，永遠如此，包括 WiFi 掉線、SD 卡卡住、又剛好碰上一次分頁錯誤（page fault）的那一個週期。本章的主題就是讓那條撐著整艘船的迴圈無法被掀翻：守住最壞情況的期限，並在每一次 I/O 出錯時都能存活下來。

1「即時」到底是什麼意思

① 直覺 先把畫面建起來

即時（real-time）不等於快。它的意思是準時。一個系統若能保證在已知的期限（deadline）之前完成每一項工作，它才是即時的——而這份保證談的是最壞情況，不是平均值。一個平均每週期 3 ms、但每分鐘一次飆到 80 ms 的控制器，對一個 50 ms 的迴圈來說，是個壞掉的控制器。船不在乎其中 999 個週期都很棒；在第 1000 個週期上舵令過期了，於是它失控橫滾（broach）。

💡 要記住的那一句話

平均延遲（latency）是一個跑分數字。最壞情況延遲是一條安全性質。即時工程這門功夫，是去縮小並界定那條尾巴（tail），而不是去管平均值。

它分三個等級。硬即時（hard real-time）：錯過一次期限就是系統失效（安全氣囊、馬達換相的 ISR）。韌即時（firm real-time）：遲到的結果毫無用處，你寧可丟掉也不要用它（一份過期的感測器取樣）。軟即時（soft real-time）：遲到會降低品質但還能忍受（影片播放）。一個 USV 的航向迴圈介於韌到硬之間：一個遲到的舵令比沒有命令更糟，而連續累積夠多次就會讓任務翻船。

② 數學 期限、週期、抖動

讓一個週期性任務以頻率 $f$ 執行。它的週期（period）是 $T = 1/f$，而你隱含的期限就是「在下一個週期開始之前完成這個週期」，也就是一個週期長。對那個 20 Hz 的迴圈：

$$ T = \frac{1}{f} = \frac{1}{20\ \text{Hz}} = 50\ \text{ms} \quad=\quad \text{deadline per cycle} $$

有兩個數字描述你打得有多準。延遲（latency） $L$ 是從預定喚醒到你的程式碼真正開始執行之間的耽擱。抖動（jitter） $J$ 是這份延遲跨越各週期的散布——也就是你量到的最大值減最小值（或某個高百分位數減中位數）：

$$ J = L_{\max} - L_{\min}, \qquad L_{\max} \le T - C_{\text{worst}} $$

其中 $C_{\text{worst}}$ 是你最壞情況的計算時間。第二條不等式才是整盤棋的關鍵：排程延遲加上最壞情況計算時間，必須塞得進一個週期裡。如果你的喚醒最多可能晚 $L_{\max}$、你的計算最多可能花 $C_{\text{worst}}$，那麼只有在 $L_{\max} + C_{\text{worst}} \le T$ 時你才守得住期限。

🏭 真實案例——量 AGX Orin 的尾巴

在把控制迴圈交給 Jetson AGX Orin 之前，先量它。cyclictest（來自 rt-tests）每個週期喚醒一條執行緒，並回報實際喚醒與預期喚醒的落差。在原廠核心上，你可能會看到 Avg: 8us Max: 4200us——一條從 8 µs 平均值拖出來的 4.2 ms 尾巴。吃掉你預算的就是這 4.2 ms，而它對任何只報平均值的跑分工具都是隱形的。

2延遲的尾巴從哪裡來

① 直覺 誰偷走了你的 CPU

你的迴圈會遲到，是因為在計時器觸發到你的程式碼開始執行之間，核心拿那顆 CPU 去做了別的事。有四個大盜，而你修每一個的方法都不一樣。

⚠️ 陷阱——第一個週期會騙你

分頁錯誤這個小偷最愛第一次迭代。你 malloc() 來的記憶體，在你碰到它之前並沒有實際的實體分頁撐著（需求分頁，demand paging）。你的暖機週期把每一個分頁都錯入了，看起來一切正常；可是在第 30 分鐘錯入一個新記錄緩衝區的那個週期，就會炸掉期限。這正是 mlockall() 存在要消滅的東西。

② 數學 IRQ 風暴吃掉預算

中斷之所以陰險，是因為它的成本是一個速率乘上一個每事件成本。如果一張負載中的 NIC 以速率 $\lambda$ 觸發中斷、每個處理常式偷走 $c$ 的 CPU 時間，那麼一顆核心被它吃掉的比例是：

$$ \text{IRQ load} = \lambda \cdot c $$

一張沒有做合併（coalescing）的 gigabit NIC 可以衝到 $\lambda = 80{,}000$ IRQ/s；在 $c = 3\ \mu\text{s}$ 時那就是 $0.24$——一顆核心去掉四分之一，而且這每一個處理常式都是一次搶占你控制迴圈的機會。合併（第 07 章）以及把那個 IRQ 釘到另一顆核心（下一節），兩者都能縮小你的迴圈所看到的傷害。

3PREEMPT_RT、優先級與鎖定記憶體

① 直覺 讓核心讓位

在原廠的 Linux 核心上，一條在核心裡執行的執行緒（一次系統呼叫（syscall）、一個驅動程式）通常要到抵達一個排程點才能被搶占。多毫秒級的卡頓就藏在那裡。PREEMPT_RT 修補集（逐步合入主線，約在 6.12 核心完全併入）讓幾乎整個核心都可被搶占：自旋鎖（spinlock）變成可睡眠的互斥鎖（mutex），中斷處理常式跑在你可以排優先級的執行緒裡，而一條高優先級的 RT 任務幾乎可以立刻奪下一顆 CPU。

💡 RT 是一紙合約，不是加速

PREEMPT_RT 通常會稍微降低吞吐量（那麼多可搶占性是有開銷的）。它替你換來的，是一條小得多、可預測得多的尾巴。你用尖峰速度去換一個有界的最壞情況——這正是控制迴圈想要的交易。

② 數學 優先級與 RT 節流

有兩個排程類別要緊。SCHED_FIFO 和 SCHED_RR 是即時類別，優先級為 1–99；任何可執行的 RT 執行緒都會搶占所有普通（SCHED_OTHER）任務。數字越大優先級越高。一條 SCHED_FIFO 執行緒會一直跑到它阻塞、或有更高優先級的 RT 執行緒醒來為止——它不會與同優先級者做時間切片。

最後這個事實是個會打到自己腳的陷阱：一條因 bug 而空轉的 SCHED_FIFO 執行緒可以永遠鎖死一顆核心。Linux 的安全閥是 RT 節流（throttle），它預設把 RT 類別封頂在每 $1000000\ \mu\text{s}$ 視窗內 $950000\ \mu\text{s}$ 的 CPU 時間：

$$ \text{RT budget} = \frac{950000}{1000000} = 95\% \ \text{of each core, per 1 s window} $$

對一顆真正隔離出來的控制核心，你通常會把迴圈規劃得遠低於這個值，並把節流當成最後的防線。

③ 程式碼 每一行都讀

一條控制執行緒的配方永遠是同樣三步：鎖定記憶體、設定 RT 優先級，然後進迴圈。鎖定要先做，因為在你變成 RT 之後才錯入分頁，正是你想要避免的那種卡頓。

rt_setup.c C
#include <sched.h>
#include <sys/mman.h>
#include <string.h>
#include <stdio.h>

int make_realtime(int prio) {
    /* 1. Pin ALL current + future pages to RAM: no demand paging,
          no swap, no fault stalls inside the loop. */
    if (mlockall(MCL_CURRENT | MCL_FUTURE) != 0) {
        perror("mlockall");        /* needs CAP_IPC_LOCK / root */
        return -1;
    }

    /* 2. Become a SCHED_FIFO thread at the given RT priority. */
    struct sched_param sp;
    memset(&sp, 0, sizeof(sp));
    sp.sched_priority = prio;      /* 1..99; control loop ~ 80 */
    if (sched_setscheduler(0, SCHED_FIFO, &sp) != 0) {
        perror("sched_setscheduler");
        return -1;
    }

    /* 3. Pre-fault the stack so the FIRST cycle isn't special either. */
    volatile char touch[512 * 1024];
    memset((void *)touch, 0, sizeof(touch));
    return 0;
}
🛠️ 別讓你的整個行程跑 RT

只把控制執行緒設成 SCHED_FIFO。記錄、遙測、雜務都留在 SCHED_OTHER，這樣它們永遠不可能餓死迴圈，還能吸收掉空檔。一條 RT 執行緒、許多普通執行緒——這就是標準的機器人架構。

4隔離 CPU 與導引中斷

① 直覺 給迴圈一個自己的房間

AGX Orin 有 12 顆 Arm 核心。最乾淨的即時配置，是把所有東西都趕出一顆核心、只在那裡跑控制迴圈。兩根槓桿：isolcpus 阻止普通排程器把任務擺到某顆核心上，而IRQ 親和性（affinity）阻止裝置中斷落到那裡。兩者合起來，把一顆核心變成一間安靜的房間，讓迴圈幾乎永遠不被打斷。

② 程式碼 核心命令列 + /proc

隔離從開機時的核心命令列開始（在 Jetson 上，於 /boot/extlinux/extlinux.conf 編輯）。把核心 11 保留給迴圈：

kernel cmdline (extlinux.conf APPEND line) bash
# isolate core 11 from the scheduler, RCU callbacks, and the tick
isolcpus=11 nohz_full=11 rcu_nocbs=11
# then pin your control thread to it from code or the shell:
taskset -c 11 ./control_loop          # CPU affinity = core 11 only

接著把中斷導離核心 11。每個 IRQ 在 /proc/irq/<N>/smp_affinity_list 裡有一個 CPU 親和性遮罩。把吵雜的那些（NIC、USB）指向雜務核心 0–10：

irq_affinity.sh bash
# find the NIC's IRQ number(s)
grep eth0 /proc/interrupts        # column 1 is the IRQ number, e.g. 142

# keep that IRQ off the isolated core (allow cores 0-10)
echo 0-10 | sudo tee /proc/irq/142/smp_affinity_list

# verify nothing is targeting core 11
cat /proc/irq/142/smp_affinity_list
⚠️ 陷阱——irqbalance 跟你作對

irqbalance 守護行程會樂呵呵地把中斷重新攤回你那顆隔離核心上。在一台 RT 機器人上，把它停掉（sudo systemctl disable --now irqbalance），親和性自己釘。也要注意有些中斷（計時器、每核心 IPI）是由核心管的、不會移動——這沒關係；它們便宜又可預測。

對網路繁重的船還有一個旋鈕：RPS（接收封包導引，Receive Packet Steering）用軟體把 softirq 的封包處理攤到各核心上，這樣單佇列的 NIC 就不會把它所有的接收工作全倒到它 IRQ 落到的那一顆核心上。在 /sys/class/net/<iface>/queues/rx-0/rps_cpus 設它的遮罩——而且，一樣，把隔離核心排除在外。

5架構：熱迴圈與其他一切

① 直覺 兩個世界，一艘船

決定一台即時機器人成敗的關鍵，是什麼東西跑在熱路徑（hot path）上。控制執行緒只做絕對的最低限度：讀取新鮮的感測器值、計算、命令致動器。任何可能阻塞的事——寫記錄到快閃儲存、透過 WiFi 遙測、錄一段 rosbag——都跑在別的執行緒上，並以無鎖（lock-free）或有界佇列餵食。熱迴圈永遠不等磁碟、永遠不等通訊端（socket）。

🔑 該注意的地方

熱迴圈與工作執行緒之間的那個環形（ring），就是那道防火牆。控制執行緒只會推入（一次有界、非阻塞的寫入）；如果某個工作執行緒落後了，最壞情況是丟掉一筆記錄取樣——永遠不會是錯過一個舵令。遙測上的背壓（backpressure）絕不可以傳回到迴圈。

④ 實際案例 為什麼重要

在 USV 上，rosbag 錄製器和 WiFi 遙測都想搬好幾 MB 的資料。把它們任何一個直接放進迴圈裡跑，那麼一次快閃刷新（flush）或一次 TCP 重傳，遲早會炸掉 50 ms 的預算。有了兩個世界的架構，SD 卡可以卡住 200 ms，船依然能操舵——記錄執行緒事後再從環形把進度追回來就好。

6穩健的 I/O 迴圈：處理每一種失敗

① 直覺 read() 會騙你

就算計時完美無瑕，I/O 本身還是會跟你作對。一次序列埠 read() 可能回傳比你要求更少的位元組、可能因為信號到來而回傳 -1 且 errno = EINTR、或在一個沒東西可讀的非阻塞 fd 上回傳 EAGAIN。只要你把這其中任何一個當成致命錯誤，你的迴圈就會在第一個信號或第一個不完整訊框上死掉。一個穩健的迴圈會預期它們全部都會發生。

② 數學 逾時預算

讀取必須是有界的：在一個死掉的感測器上做阻塞讀取，會永遠卡死迴圈。給讀取一個塞得進週期、還留有餘裕的逾時值。如果迴圈週期是 $T$、最壞情況計算是 $C_{\text{worst}}$，那麼留給等待 I/O 的空檔是：

$$ t_{\text{timeout}} \le T - C_{\text{worst}} - L_{\max} $$

對 $T = 50$ ms、$C_{\text{worst}} = 5$ ms、$L_{\max} = 4$ ms，你有 $\le 41$ ms 可以等——但你會把逾時設得遠低於這個值（比方說 20 ms），這樣一筆遲到或缺失的取樣才會被偵測到並跳過，還有時間用最後一個良好估計去命令致動器。

③ 程式碼 能存活的讀取迴圈

這是一個穩健驅動程式裡最重要的單一函式：恰好讀 count 個位元組，或在逾時時乾淨地失敗——沿途處理 EINTR 與不完整讀取。一行一行讀；它就是下面那道習題的答案。

robust_read.c C
#include <unistd.h>
#include <errno.h>
#include <poll.h>
#include <stdint.h>
#include <time.h>

/* Milliseconds remaining until `end`, clamped to >= 0. */
static int ms_left(const struct timespec *end) {
    struct timespec now;
    clock_gettime(CLOCK_MONOTONIC, &now);
    long ms = (end->tv_sec  - now.tv_sec)  * 1000
            + (end->tv_nsec - now.tv_nsec) / 1000000;
    return ms > 0 ? (int)ms : 0;
}

/* Read exactly n bytes into buf, or stop after deadline_ms total.
   Returns bytes read (== n on success, < n on timeout, -1 on hard error). */
ssize_t read_n(int fd, void *buf, size_t n, int deadline_ms) {
    uint8_t *p = (uint8_t *)buf;
    size_t got = 0;
    struct timespec end;                        /* absolute deadline */
    clock_gettime(CLOCK_MONOTONIC, &end);
    end.tv_sec  += deadline_ms / 1000;
    end.tv_nsec += (deadline_ms % 1000) * 1000000L;
    if (end.tv_nsec >= 1000000000L) { end.tv_sec++; end.tv_nsec -= 1000000000L; }
    while (got < n) {
        struct pollfd pfd = { .fd = fd, .events = POLLIN };
        int pr = poll(&pfd, 1, ms_left(&end)); /* wait only the time LEFT */
        if (pr < 0) {
            if (errno == EINTR) continue;       /* signal: retry (clock keeps ticking) */
            return -1;                          /* real poll error */
        }
        if (pr == 0) return (ssize_t)got;       /* TIMEOUT: partial/none */

        ssize_t r = read(fd, p + got, n - got);
        if (r < 0) {
            if (errno == EINTR) continue;        /* retry */
            if (errno == EAGAIN) continue;       /* spurious: poll again */
            return -1;                           /* EIO/ENXIO: device gone */
        }
        if (r == 0) return (ssize_t)got;         /* EOF: peer closed */
        got += (size_t)r;                        /* PARTIAL: keep going */
    }
    return (ssize_t)got;                          /* got == n */
}
🛠️ 寫入也需要同樣的講究

一次 write() 同樣可能不完整、同樣可能回傳 EINTR。對應的鏡像迴圈會推進一個偏移量，直到所有位元組都送出去；而在一個串流通訊端上，你必須把 EPIPE（對端消失）當成觸發重連的訊號，而不是當機。把兩者都包進輔助函式裡；在正式的機器人程式碼裡，永遠不要裸呼叫 read()／write()。

④ 實際案例 冪等重開

當 GPS 的 ttyUSB0 電壓驟降（brown out）時，核心可能把它以 ttyUSB1 交回來——這正是第 03 章要你釘一個 /dev/gps udev 符號連結的原因。重連常式必須是冪等的（idempotent）：若舊的 fd 還開著就關掉它、重開 /dev/gps、重新套用 termios、重建剖析器狀態——每次失敗時都能安全呼叫，不會洩漏 fd。在重開成功之前，迴圈跑的是航位推算（dead reckoning），而不是過期的 GPS。

7不阻塞迴圈的持久化記錄

① 直覺 write() 很快，持久化很慢

對檔案做一次普通的 write() 很快，因為它只是把資料拷進核心的分頁快取（page cache）；真正的快閃寫入是稍後才非同步發生的。這對熱路徑很完美——但這也意味著一次當機或斷電會丟掉最後幾秒的資料。fsync() 強迫資料一路寫到穩定儲存，並阻塞直到裝置確認為止。在一張 SD 卡上那可能是數十到數百毫秒。永遠不要在熱路徑上呼叫它。

② 數學 SD 卡的寫入放大與磨損

SD／eMMC 快閃以較大的區塊（block）為單位抹除，卻以較小的分頁（page）為單位寫入。更新一個位元組可能逼得控制器去讀-抹-重寫一整個抹除區塊——寫入放大（write amplification）。定義它為：

$$ \text{WA} = \frac{\text{bytes physically written to flash}}{\text{bytes your app wrote}} $$

細小、fsync 過的記錄會給出糟糕的 WA：一筆 64 B 的遙測記錄若觸發一次 512 KB 抹除區塊重寫，就是 $\text{WA} = 8192$。快閃單元只能承受有限次數的寫入/抹除（P/E）循環，所以以位元組計的耐久度大致是：

$$ \text{Bytes endurance} \approx \frac{\text{capacity} \times P\!/\!E\ \text{cycles}}{\text{WA}} $$

修法是讓每一次寫入都大、對齊、且循序：在 RAM 緩衝區裡把記錄批次累積起來，再由記錄執行緒以抹除區塊大小的區塊（例如 512 KB 或 1 MB）刷新。大的循序寫入會把 WA 推向 1，並把卡的壽命拉長好幾個數量級。

③ 程式碼 離開熱路徑的批次刷新
log_writer.c C
#define FLUSH_BYTES (1u << 20)   /* 1 MiB: big, aligned, sequential */
static uint8_t batch[FLUSH_BYTES];
static size_t  fill = 0;

/* runs on the SCHED_OTHER log thread, draining the ring buffer */
void log_record(int fd, const void *rec, size_t len) {
    if (fill + len > FLUSH_BYTES) {        /* batch full -> one big write */
        size_t off = 0;
        while (off < fill) {                /* handle partial writes */
            ssize_t w = write(fd, batch + off, fill - off);
            if (w < 0) { if (errno == EINTR) continue; break; }
            off += (size_t)w;
        }
        fill = 0;
        /* fsync HERE, on the worker, never on the control thread.
           Do it on a coarse schedule (e.g. every few MiB), not per record. */
        fsync(fd);
    }
    memcpy(batch + fill, rec, len);
    fill += len;
}
🏭 真實案例——USV 任務記錄

Orin 以 20 Hz 記錄 IMU + GPS + 控制狀態。對 SD 卡逐筆 fsync 既會卡住迴圈、又會把卡磨爛。改作法是：取樣經由無鎖環形送到一條記錄執行緒，它批次累積約 1 MiB 後循序刷新，並只每隔幾秒呼叫一次 fsync。結果：熱迴圈永遠不碰快閃、卡撐得過整趟任務，而一旦斷電你最多丟掉幾秒的遙測——這是可接受、有界的損失。

8看門狗與失效安全狀態

① 直覺 為自己的死亡預作打算

穩健的 I/O 假設軟體終究會當掉。硬體看門狗（watchdog）是 SoC 裡的一個倒數計時器，如果沒有及時被「拍一下（pet）」就會重置板子；你那條健康的控制執行緒每個週期拍它一下。如果迴圈死鎖了，看門狗會把 Orin 重開機到一個已知狀態，而不是任由船繞圈亂開。Linux 把它暴露成一個字元裝置：開啟 /dev/watchdog，定期對它寫入（或用 WDIOC_KEEPALIVE 做 ioctl），核心就會去設定那個計時器。

⚠️ 陷阱——從迴圈拍，不要從計時器拍

要從控制執行緒自己的工作裡、在一個成功的週期之後拍看門狗——絕不要從一條獨立的計時器執行緒拍。一條在控制迴圈卡死時還持續拍的計時器，會把整件事的意義全毀掉：板子保持「活著」，而船卻已失控。

第二層是失效安全狀態（failsafe state）：當某個輸入消失時，該做的安全動作。對 USV 來說，標準的觸發是WiFi 掉線（第 09 章那條共享、抖動的連線）。如果在一個期限內沒有操作員的心跳（heartbeat）抵達，船就必須自主退守——把舵置中、收油門、定位停留或盤旋——而不是把上一個命令一路執行進一塊礁石。

#define HEARTBEAT_TIMEOUT_NS  1000000000LL   /* 1 s */

/* called once per 50 ms control cycle, on the RT thread */
void control_step(struct ctx *c, int64_t now_ns) {
    int64_t age = now_ns - c->last_heartbeat_ns;

    if (age > HEARTBEAT_TIMEOUT_NS) {
        /* WiFi/operator lost: known-safe outputs, NOT the stale command */
        set_rudder(0.0);            /* center */
        set_throttle(0.0);          /* cut */
        c->state = FAILSAFE;
    } else {
        c->state = ACTIVE;
        run_pid(c);                 /* normal heading control */
    }
    pet_watchdog(c->wd_fd);         /* only reached if the loop is alive */
}

9換你了

Q1 觀念檢查
你那個 20 Hz 的迴圈平均每週期 2 ms，但因為在熱路徑上對 SD 卡做了 fsync，每分鐘一次飆到 70 ms。對一個即時控制器來說，這可以接受嗎？
  • 可以——平均值遠低於 50 ms 的週期，所以快得很。
  • 不行——即時談的是最壞情況；那 70 ms 的尖峰錯過了 50 ms 的期限，而一個過期的舵令可能讓船翻覆。
  • 可以——每分鐘一次錯過週期，在統計上微不足道。
  • 不行——但只是因為 2 ms 太慢了；它應該要在微秒等級。
即時是一份關於尾巴的保證，不是平均值。在一個 50 ms 的迴圈裡出現 70 ms 的週期，就是一次錯過的期限——是安全失效，不是跑分上的小瑕疵。修法正是本章：把 fsync 從熱路徑移到一條由環形緩衝餵食的 SCHED_OTHER 記錄執行緒上。

Q3 程式挑戰——一個能存活的讀取迴圈
寫一個函式，從一個序列埠 fd 恰好讀 n 個位元組，但在一個總逾時之後放棄。它必須正確處理 EINTR（重試）、不完整讀取（繼續讀）、以及逾時（回傳你已拿到的）。裸呼叫一次 read(fd, buf, n) 的 bug 是什麼？
提示
裸 read() 可能（a）在信號上回傳 -1/EINTR，（b）即使資料還在來、也回傳少於 n 個位元組，以及（c）在一個死掉的裝置上永遠阻塞。你需要一個有界的等待（poll），加上一個會把資料累積進偏移量的迴圈。
看一個解答

單一次 read() 是錯的，因為它把不完整的回傳當成整個訊息、在第一個 EINTR 上就死掉、而且沒有逾時。穩健的版本帶著一個期限去 poll、在 EINTR/EAGAIN 上重試、把不完整的部分累積進偏移量、並在逾時或 EOF 時提早回傳：

robust_read.c C
/* ms remaining until the absolute deadline `end`, clamped to >= 0 */
static int ms_left(const struct timespec *end) {
    struct timespec now; clock_gettime(CLOCK_MONOTONIC, &now);
    long ms = (end->tv_sec - now.tv_sec) * 1000
            + (end->tv_nsec - now.tv_nsec) / 1000000;
    return ms > 0 ? (int)ms : 0;
}

ssize_t read_n(int fd, void *buf, size_t n, int deadline_ms) {
    uint8_t *p = (uint8_t *)buf;
    size_t got = 0;
    struct timespec end; clock_gettime(CLOCK_MONOTONIC, &end);
    end.tv_sec  += deadline_ms / 1000;
    end.tv_nsec += (deadline_ms % 1000) * 1000000L;
    if (end.tv_nsec >= 1000000000L) { end.tv_sec++; end.tv_nsec -= 1000000000L; }
    while (got < n) {
        struct pollfd pfd = { .fd = fd, .events = POLLIN };
        int pr = poll(&pfd, 1, ms_left(&end));   /* only the time LEFT, not the full budget */
        if (pr < 0) { if (errno == EINTR) continue; return -1; }
        if (pr == 0) return (ssize_t)got;          /* timeout */
        ssize_t r = read(fd, p + got, n - got);
        if (r < 0) {
            if (errno == EINTR || errno == EAGAIN) continue;
            return -1;                              /* device gone */
        }
        if (r == 0) return (ssize_t)got;            /* EOF */
        got += (size_t)r;                           /* partial: keep going */
    }
    return (ssize_t)got;                            /* got == n */
}

同樣的形狀也適用於寫入（推進一個偏移量直到所有位元組都送出去，在 EINTR 上重試）。在機器人程式碼裡，永遠不要裸呼叫 read()／write()。

🔑 本章回顧

即時的意思是守住最壞情況期限，而不是一個漂亮的平均值——要去工程那條尾巴。延遲來自排程器、IRQ、分頁錯誤與省電狀態；用 PREEMPT_RT、SCHED_FIFO 優先級、mlockall()、isolcpus 與 IRQ 親和性去馴服它們。讓熱迴圈維持最精簡：非阻塞讀取、計算、致動，並透過一個環形把取樣推給負責記錄與遙測的 SCHED_OTHER 工作執行緒。讓 I/O 穩健的作法是處理 EINTR、不完整的讀取/寫入、EAGAIN、逾時、以及冪等重開。離開熱路徑做持久化記錄，用大、對齊、循序的寫入，並罕用 fsync 以省下 SD 卡。最後，一個只由健康迴圈拍動的硬體看門狗，加上一個以心跳把關的失效安全，就能把任何卡死或 WiFi 掉線都轉成一個安全狀態，而不是一艘失控的船。
