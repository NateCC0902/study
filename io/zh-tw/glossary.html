1核心 I/O Core I/O

最底層的基石：你的程式與核心（kernel）之間的邊界，以及那個讓其餘一切都掛在上面的抽象——檔案描述子（file descriptor）。這些若是模糊不清，上面的東西就都說不通了。

術語中文一句話定義
System call (syscall)系統呼叫從使用者空間（user space）進入核心空間（kernel space）的受控門戶；CPU 觸發陷阱（trap）、切換特權等級、執行核心程式碼、再返回。read、write、ioctl 都是系統呼叫。
User space / kernel space使用者空間 / 核心空間兩個特權環（ring）。你的行程（process）以無特權身分執行；只有核心能碰硬體。系統呼叫是唯一合法的跨越方式。
File descriptor (fd)檔案描述子一個小的非負整數，當作你行程的開啟檔案表（open-file table）索引。0/1/2 是 stdin/stdout/stderr；你每開啟一個東西就拿到下一個可用的編號。
VFS (Virtual File System)虛擬檔案系統（VFS）讓檔案、socket、裝置與管線（pipe）全都回應同一套 read/write/close API 的核心層——也就是「一切皆檔案」背後的機制。
inode索引節點（inode）一個檔案系統物件中繼資料（metadata）的磁碟（或記憶體）紀錄——型別、權限，以及若是裝置節點，還有主／次編號（major/minor）這一對。它不是名字；名字住在目錄裡。
Blocking I/O阻塞 I/O會讓你的執行緒（thread）睡著、直到資料到來才醒的 read()。簡單，但一個慢裝置就能讓整條執行緒卡死。
Non-blocking I/O非阻塞 I/O用 O_NONBLOCK 開啟；沒資料時 read() 不睡覺，而是立刻回傳 -1 並帶有 errno == EAGAIN。這是事件迴圈（event loop）的前提。
Partial read / short read部分讀取read() 可能回傳比你要的還少的位元組——這不是錯誤。穩健的程式碼會迴圈讀取，直到湊齊一則完整訊息或碰到 EOF。
Buffering (stdio)緩衝（buffering）C 函式庫把位元組囤在使用者空間的緩衝區裡，再大塊大塊地刷出（flush），所以一次 fwrite ≠ 一次系統呼叫。對吞吐量（throughput）很好，但若你忘了 flush，對延遲（latency）就是致命傷。
Raw / unbuffered I/O無緩衝 I/O直接呼叫 read/write——一次呼叫一次系統呼叫，沒有函式庫囤積。當你要讀一個 200 Hz 感測器、而資料過時就是 bug 時，這正是你要的。
Page cache頁快取（page cache）核心對檔案／區塊資料的 RAM 快取。對檔案的 write() 會先落在這裡，稍後才刷到媒體上——很快，但在 fsync() 之前這些位元組並不持久（durable）。
errno錯誤碼（errno）失敗的系統呼叫會設定的一個執行緒區域整數，用來說明為什麼失敗。EINTR、EAGAIN、ENODEV 是你在機器人上會處理到的幾個。
EINTR系統呼叫被中斷「Interrupted system call」——系統呼叫進行到一半，有個訊號（signal）來了。不是真正的失敗；重試（retry）這次呼叫即可。
🔑 你會反覆回來查的攤提公式

每次系統呼叫都有一筆固定開銷（陷阱＋特權切換＋返回），記作 $C_s \approx 1\text{–}3\,\mu\text{s}$。如果你每次呼叫讀 $B$ 個位元組、每位元組成本為 $c_b$，那麼有效速率是

$$ R_\text{eff} = \frac{B}{C_s + B\,c_b} \xrightarrow{B \to \infty} \frac{1}{c_b} $$

小量讀取把時間全花在 $C_s$ 上；大量讀取則把它攤提掉。光這一個事實，就是 stdio 要緩衝、你要批次讀感測器、以及 io_uring 之所以存在的原因。

2裝置與驅動 Devices & drivers

一塊矽晶片如何變成一個你能 open() 的檔案。驅動（driver）是翻譯官；/dev 是電話簿；/sys 是即時呈現有什麼被插上的活地圖。

術語中文一句話定義
/dev裝置目錄裝置節點（device node）的目錄。每個節點是一個名字，攜帶著一組（型別、主編號、次編號）三元組，把 open() 路由到某個驅動。開機時由 devtmpfs 填入。
sysfs (/sys)系統檔案系統一個常駐 RAM 的檔案系統，把核心的裝置模型攤成可瀏覽的檔案。cat 一個屬性可讀出一個近似暫存器的值；echo 進去則可戳硬體。
procfs (/proc)行程檔案系統sysfs 的姊妹，呈現行程與核心狀態——/proc/interrupts、/proc/cpuinfo、/proc/PID/fd。
Character device字元裝置位元組串流裝置——UART、I²C 匯流排、GPIO、相機。你和驅動之間沒有頁快取。在 ls -l 裡顯示為 c。
Block device區塊裝置固定區塊、隨機存取、有快取的裝置——eMMC、NVMe、SD 卡。走區塊層（block layer）與頁快取。顯示為 b。
Major / minor number主設備號 / 次設備號裝置節點裡的兩個整數。主編號選驅動；次編號選哪一個實例。一起打包進一個 dev_t。
dev_t裝置號（dev_t）那個 32 位元（現代：12 位元主 + 20 位元次）打包好的裝置編號。用 MKDEV(maj, min) 編碼；用 MAJOR()/MINOR() 解碼。
devtmpfs裝置臨時檔案系統核心開機時自動掛載在 /dev 上的 tmpfs，並在驅動註冊時預先填入節點，這樣 /dev 在 udev 跑起來之前就已存在。
udev / udevd裝置管理器監聽核心 uevent 並套用規則的使用者空間常駐程式——重新命名、建立符號連結、設定權限。你那個穩定的 /dev/gps 就是這樣來的。
uevent裝置事件核心在裝置 新增/移除/變更 時發出的一則 netlink 訊息。它是 udev 的輸入。
Kernel module (.ko)核心模組用 modprobe/insmod 在執行期載入運行中核心的驅動程式碼——免重開機。用 modinfo 檢視、用 lsmod 列出。
In-tree / out-of-tree主線 / 樹外驅動主線（in-tree）驅動隨核心原始碼一起出貨、一起重新編譯；樹外（out-of-tree）驅動（廠商 blob、DKMS）是單獨編譯的，核心升級時就壞了。
Firmware韌體（firmware）驅動在探測（probe）時透過 request_firmware() 從 /lib/firmware 載入進裝置（Wi-Fi、GPU）的一塊二進位 blob。檔案不見了在 dmesg 裡會顯示為錯誤 -2（ENOENT）。
Device tree (DT)裝置樹（device tree）一個資料結構（.dts → .dtb），向核心描述不可自我探索的硬體——位址、IRQ、compatible 字串——好讓正確的驅動綁定（bind）。在像 Orin 這種 ARM SoC 上不可或缺。
ioctl裝置控制（ioctl）專收那些塞不進 read/write 的裝置命令的萬用系統呼叫——設定序列埠鮑率、查詢 CAN 位元率、發出自訂命令。ioctl(fd, REQUEST, &arg)。
🏭 真實案例——USV 的 Wi-Fi 就是這整個第二欄

在 AGX Orin 上，板載 Wi-Fi 模組掛在 SDIO（一個不可自我探索的匯流排）上，所以它必須在裝置樹裡被宣告；它的樹外廠商核心模組會從 /lib/firmware 載入一塊韌體 blob；接著 udev 給最終的網路介面一個可預期的名字。這句話裡的每個術語都是上面表格的一列——而只要其中任一步驟失敗，dmesg 都會告訴你是哪一步。

3匯流排與列舉 Buses & enumeration

硬體實際上拿來對話的那些線。最有用的一個區別：會自己找到自己裝置的匯流排（PCIe、USB），對上你必須透過裝置樹告訴核心那裡有什麼的匯流排（I²C、SPI、SDIO）。

術語中文一句話定義
Enumeration列舉（enumeration）發現匯流排上有什麼、並指派位址的過程。PCIe 與 USB 自動列舉；I²C/SPI/SDIO 裝置則必須在裝置樹裡被宣告。
PCIePCIe 高速匯流排給 NVMe、網卡、GPU 用的高速、交換式、點對點序列匯流排。會自我列舉；用 lspci -t 檢視拓樸。
BDF (Bus:Device.Function)PCIe 位址一個 PCIe 裝置的位址，例如 0000:05:00.0 = domain:bus:device.function。也就是 lspci 印出來的那個代號。
BAR (Base Address Register)基址暫存器一個 PCIe 設定空間（config-space）暫存器，告訴作業系統要把裝置的 MMIO 視窗映射到實體記憶體的哪裡。
USB通用序列匯流排可熱插拔、樹狀拓樸、會自我列舉的匯流排。裝置透過描述符（descriptor）自我描述；用 lsusb -t 看那棵樹。
USB descriptorUSB 描述符USB 裝置插上時回報的自我描述資料（idVendor、idProduct、序號、端點 endpoint）。udev 規則就比對這些。
I²CI²C 匯流排給低速率感測器用的兩線（SDA/SCL）多點匯流排。7 位元定址 → 128 個位址（少數保留）。必須在 DT 裡宣告。
SPISPI 匯流排四線（MOSI/MISO/SCLK/CS）全雙工匯流排；比 I²C 快，以晶片選擇（chip-select）線、而非位址來定址。在 DT 裡宣告。
SDIOSDIO 匯流排把 SD 卡的電氣匯流排挪去接 I/O 周邊（最著名的是 Wi-Fi）。不可自我探索——必須在裝置樹裡宣告。
MMIO記憶體映射 I/OMemory-Mapped I/O：裝置暫存器以實體記憶體位址的樣子出現，所以一次 CPU load/store 就讀／寫一個暫存器。從使用者空間透過 mmap 與 volatile 存取。
compatible string相容性標識那個 DT 屬性（例如 "ti,tmp102"），用來標明一個裝置的型別，好讓核心挑出相符的驅動。
Device tree overlay裝置樹覆蓋疊加在基底 DT 之上的一片補丁（.dtbo），用來啟用一支排針或掛上一個感測器，而不必重新編譯整棵樹——例如透過 jetson-io。
∑ 為什麼「這匯流排夠快嗎？」只是一個比較

挑匯流排的方法，就是拿你所需的吞吐量去跟它的上限比。I²C 標準模式頂在 $100\,\text{kbit/s}$ 附近（快速模式 $400\,\text{k}$）；單一條 PCIe Gen4 通道（lane）在 128b/130b 編碼後約可承載 $1.97\,\text{GB/s}$ 可用頻寬。一顆 16 線束的光達（lidar），以 $1\,\text{Mpoint/s} \times 16\,\text{B/point}$ 計，需要

$$ R_\text{req} = 1{\times}10^6 \cdot 16 = 16\,\text{MB/s} $$

——在 I²C 上荒謬地不可能，在 PCIe 或 Gigabit 乙太網上則輕而易舉。挑一條匯流排，就是逐裝置算這一條不等式 $R_\text{req} \le R_\text{bus}$，就這樣。

4資料搬運與多路複用 Data movement & multiplexing

兩個問題：位元組如何實際從裝置進到 RAM（中斷對上 DMA），以及一條執行緒如何不靠忙等（busy-wait）就同時盯著許多裝置（epoll、io_uring）。

術語中文一句話定義
PIO (Programmed I/O)程式控制 I/OCPU 親自一個暫存器一個暫存器地把每個位元組搬過去。簡單，但傳輸期間會把一顆核心燒滿。
Interrupt中斷（interrupt）一個硬體訊號，把 CPU 從手邊工作硬扯開去執行一段中斷服務程式（ISR）——「資料好了，快來處理」，而不是用輪詢（polling）去問。
IRQ (Interrupt Request line)中斷請求裝置用來拉起一個中斷的那條編號線路。計數住在 /proc/interrupts 裡。
ISR / top half / bottom half中斷服務程式 / 上半部 / 下半部處理器（handler）。上半部在關中斷的情況下快速跑完；繁重工作則延後給一個可以睡覺的下半部（softirq/tasklet/執行緒化 IRQ）。
DMA直接記憶體存取Direct Memory Access：一個專屬引擎不經 CPU 就把位元組在裝置與 RAM 之間搬移，CPU 只在整塊搞定時收到一個中斷。這是在重 I/O 下仍讓控制迴圈（control loop）保持反應靈敏的關鍵。
Coherent vs. streaming DMA一致性 / 串流式 DMA一致性（coherent）緩衝區會自動維持快取一致（小、長壽）；串流式（streaming）緩衝區則逐次傳輸映射、並做明確的快取同步（大、一次性）。
IRQ affinity中斷親和性透過 /proc/irq/N/smp_affinity 把一個 IRQ 釘在特定 CPU 上，這樣中斷風暴就不會落在跑你即時迴圈的那顆核心上。
Coalescing中斷合併把許多事件併成一個中斷（例如網卡 rx-coalesce），在高速率下拿一點點延遲去換取大幅降低的 CPU 開銷。
select / poll多路複用（select/poll）一次等待許多 fd 的舊系統呼叫。成本隨被監看的 fd 數量呈 $O(n)$ 成長——它們每次呼叫都重掃整個集合。
epollepoll 事件機制Linux 可擴展的就緒（readiness）API：fd 只註冊一次，之後每次等待只花 $O(\text{ready})$ 而非 $O(n)$。支援水平觸發（level-triggered）與邊沿觸發（edge-triggered）模式。它是感測器管理器的骨幹。
Level vs. edge triggered水平 / 邊沿觸發水平觸發只要還有資料就持續回報該 fd；邊沿觸發只在轉變為可讀的那一刻回報，所以你必須把它徹底排空（迴圈讀到 EAGAIN 為止）。
io_uringio_uring 非同步 I/O現代的共享記憶體環（提交＋完成兩個佇列），用於真正的非同步 I/O，每次操作幾乎零系統呼叫——多路複用的高階選擇。
C10k problemC10k 問題同時服務一萬條連線這個經典挑戰；也是「每 fd 一執行緒」輸給事件迴圈的原因。
⚠️ 邊沿觸發的自爆陷阱

用邊沿觸發模式的 epoll，如果你每個事件只讀一次，就會把位元組留在緩衝區裡、再也不會被喚醒——那個 fd 看起來像是死了。約定是：每次被喚醒時，read() 要迴圈讀到拿到 EAGAIN 為止。一半的「我的感測器隨機卡死」bug 都是這個。

5網路 Networking

從網卡到 socket fd。最常咬機器人的兩個概念：路由表（routing table）（以及誰偷走了預設路由），還有頻寬時延積（BDP）（多少資料正「在途」）。

術語中文一句話定義
Socket套接字（socket）一個會說網路的檔案描述子。socket() 回傳一個 fd，你接著就像對其他任何 fd 一樣對它 read/write——把網路當成檔案。
TCP vs. UDPTCP / UDPTCP 是可靠、有序的串流——代價是重傳與隊頭阻塞（head-of-line blocking）。UDP 是發了就不管的資料包（datagram）——對感測器資料完美，因為一個新鮮封包勝過一個重送的陳舊封包。
NIC網卡Network Interface Controller——硬體本身。核心驅動透過中斷／DMA 與 NAPI 輪詢，把收到的訊框（frame）往上交給協定堆疊。
Routing table路由表決定一個封包該從哪個介面送出去的規則，以最長前綴優先（longest-prefix-first）比對。用 ip route 檢視。
Default route預設路由那條萬用條目（0.0.0.0/0），用於任何沒有特定路由相符的目的地。你應該只有確切一條值得信任的。
Default-route theft預設路由搶占當第二個介面（剛接上的 Wi-Fi）裝上自己的預設路由、悄悄偷走所有出口（egress）流量時——也就是經典的「我一插上乙太網路 SSH 就斷了」bug。用路由度量值（metric）來修。
Metric路由度量值一條路由上的優先級數字；越低越優先。你就是靠它讓 Wi-Fi 成為偏好的預設路由、乙太網路成為只走感測器的路徑。
MTU最大傳輸單元Maximum Transmission Unit——一條鏈路在一個訊框裡承載的最大酬載（乙太網路預設 $1500\,\text{B}$）。超過它，封包就會被分片（fragment）或丟棄。
BDP (Bandwidth-Delay Product)頻寬時延積（BDP）頻寬 × 來回時間（RTT）＝ 一條鏈路上「在途」的位元組數；也是要把管道灌滿所需的最小 socket 緩衝區。
SO_RCVBUF / SO_SNDBUF套接字緩衝區調整核心收／送緩衝區大小的 socket 選項。太小會在高 BDP 鏈路上限制吞吐量；太大則招來緩衝膨脹（bufferbloat）。
Bufferbloat緩衝膨脹過大的緩衝區囤積封包，在負載下增加延遲——即使吞吐量看起來沒問題——對控制鏈路是致命的。
Backpressure背壓（backpressure）那個告訴快生產者放慢、好讓慢消費者跟得上的訊號（緩衝區滿、EAGAIN、TCP 視窗關閉）。
∑ BDP，以及緩衝區為什麼要緊

一條鏈路在飛行途中能裝下的位元組數——也就是要灌滿它所需的 socket 緩衝區——是

$$ \text{BDP} = \text{Bandwidth} \times \text{RTT} $$

一條 RTT 為 $20\,\text{ms}$ 的 $1\,\text{Gbit/s}$ 岸對 USV 鏈路，在途量為 $1{\times}10^9/8 \cdot 0.020 = 2.5\,\text{MB}$。如果 SO_RCVBUF 還是預設的 $\sim 200\,\text{kB}$，發送端就會卡著等 ACK，你永遠看不到線速（line rate）。

6CAN 與即時 CAN & real-time

致動器匯流排，以及要讓一個控制迴圈每一次都趕上它的截止期限（deadline）需要付出什麼。即時（real-time）講的是最壞情況，從來不是平均值。

術語中文一句話定義
CANCAN 匯流排Controller Area Network——給致動器與儀器用的穩健、差動、多點匯流排。訊框攜帶一個 ID 與最多 8 個資料位元組（CAN-FD 為 64 個）。
Arbitration仲裁（arbitration）CAN 如何化解同時發送：數值上最低的 ID 那則訊息勝出（顯性位元 dominant 蓋過隱性位元 recessive）；輸的那方退讓並重試。ID 最低 = 優先級最高。
Bit stuffing位元填充在 5 個相同位元之後插入一個相反位元，好讓接收端保持時脈同步——這會讓最壞情況的訊框長度比標稱值長個幾位元。
SocketCANSocketCANLinux 把一個 CAN 介面當成網路 socket（AF_CAN）來看待的框架，於是你可以像對任何 fd 那樣 read/write CAN 訊框，並使用 candump/cansend。
Sample point採樣點控制器採樣電位所在的那個位元時間比例（通常 75–87.5%）。所有節點必須一致，否則就會出錯。
Termination (60 Ω test)終端電阻匯流排兩端各一顆 120 Ω 電阻；兩顆都裝上時，在斷電的匯流排上量 CAN-H/CAN-L 之間約讀到 ~60 Ω。終端電阻不對 = 反射 = 錯誤。
Error counters / bus-off錯誤計數 / 總線關閉每個節點都計數 TX/RX 錯誤；超過 255 它就進入bus-off，悄悄從匯流排上掉線，直到被重置。也就是「致動器不回應了」的症狀。
restart-ms自動重啟間隔那個 SocketCAN 選項（ip link set can0 type can restart-ms 100），在指定延遲後自動把一個 bus-off 介面救回來。
CAN-FDCAN-FDFlexible Data-rate CAN：最多 64 位元組酬載與更快的資料相位，在保留 CAN 仲裁機制的同時提供更高吞吐量。
NMEA 2000NMEA 2000架在 CAN 上、跑 $250\,\text{kbit/s}$ 的船用儀器協定——USV 上的 GPS、深度、風。
Real-time即時一個對最壞情況反應的保證，不是速度。一個 20 Hz 的迴圈，只要它從不錯過它的 $50\,\text{ms}$ 截止期限，它就是即時的——平均延遲無關緊要。
Deadline截止期限一個任務為了正確，最晚可以完成的時間。在硬即時（hard-real-time）系統裡錯過它就是失敗，不只是慢。
Jitter抖動（jitter）一個週期性任務實際耗時逐週期之間的變動。低平均＋高抖動照樣會炸掉截止期限。
PREEMPT_RT即時核心補丁讓幾乎所有核心程式碼都可被搶占（preemptible）的核心補丁集，把最壞情況的排程延遲從毫秒砍到微秒。
SCHED_FIFO / rtprio即時排程 / 即時優先級一種即時排程策略，讓一條執行緒在所有一般任務之前執行、直到它主動讓出（yield）——控制迴圈就是這樣準時拿到 CPU 的。
mlockall鎖定記憶體把一個行程的頁面鎖在 RAM 裡，這樣一個分頁錯誤（page fault）就絕不會在週期中途卡住即時迴圈。
Watchdog看門狗（watchdog）一個計時器，如果軟體沒有及時「踢」它就重置（或進入失效安全）系統——當迴圈卡死時的最後一道防線。
🏭 真實案例——USV 的最壞情況預算

Orin 以 20 Hz 跑轉向 PID，所以截止期限是 $50\,\text{ms}$。平均迴圈時間是 $4\,\text{ms}$——沒有意義。重要的是最壞情況：一場 Wi-Fi IRQ 風暴落在迴圈的核心上、一個分頁錯誤、一個日誌 fsync 在阻塞。用 IRQ 親和性（把網卡移離迴圈的 CPU）、SCHED_FIFO、mlockall，以及一個會在迴圈真的錯過時把舵失效到中立位的看門狗，來修這份預算。

Q1 CAN 仲裁
兩個 CAN 節點在同一瞬間開始發送——一個 ID 是 0x0A0，另一個是 0x320。哪則訊息贏得匯流排，另一則又會怎樣？
  • 0x0A0 勝出（ID 較低 = 優先級較高）；0x320 的發送端偵測到自己輸了，退讓並自動重試。
  • 0x320 勝出，因為數字較大者有優先權；0x0A0 被丟棄。
  • 兩者相撞並損毀；每個節點必須像乙太網路那樣等一段隨機退避（backoff）。
  • 不論 ID 如何，先開始發送的那個節點勝出。
CAN 仲裁是非破壞性且基於優先級的：一個顯性位元（0）蓋過隱性位元（1），所以數值上最低的 ID 會逐位元勝出而毫無資料損失。輸的那方看到不符，立即停止，並在下一個匯流排閒置時重新仲裁。見 §6 — 仲裁。
Q2 匯流排與裝置樹
下列哪一種匯流排，因為無法自己發現裝置，所以需要你在裝置樹裡宣告所接的裝置？
  • PCIe——它會自我列舉，你從 lspci 讀出位址。
  • USB——裝置插上時回報描述符，核心自動比對驅動。
  • SDIO（以及 I²C/SPI）——不可自我探索，所以硬體必須在 DT 裡被描述，驅動才能綁定。
  • 沒有——核心永遠會探測每條匯流排，自己把一切都找出來。
PCIe 與 USB 是可列舉的：匯流排協定讓主機能走遍它、並讀出每個裝置的身分。I²C、SPI 與 SDIO 沒有這種機制——沒有辦法可靠地問「0x48 位址上的是誰？」——所以核心要倚賴裝置樹才知道什麼接在哪裡、在哪個位置。這正是 Orin 的 SDIO Wi-Fi 必須在 DT 裡的原因。見 §3 — 列舉。
Q3 epoll 觸發
你用邊沿觸發模式的 epoll，而且每次喚醒只呼叫 read() 恰好一次。你的感測器明明還在發送，卻「隨機停止」送資料。為什麼？
  • 邊沿觸發模式有 bug，永遠不該拿來用在感測器上。
  • 邊沿觸發只在轉變為可讀時觸發；殘留的位元組不會再次觸發，所以你必須迴圈 read() 到 EAGAIN 為止，把 fd 排空。
  • 這個 fd 從來沒被註冊到 epoll；你得在每次讀取後重新加入它。
  • 邊沿觸發模式下的 read() 回傳 0 個位元組，所以資料根本不會到。
邊沿觸發的 epoll 回報的是邊沿——一個 fd 變成可讀的那一刻——而不是持續的水平狀態。如果你把位元組留著沒讀，就沒有新的邊沿，所以沒有新的喚醒，於是 fd 看起來像死了、而資料卻越堆越多。規則是：每個事件，迴圈 read() 到它回傳 EAGAIN 為止。見 §4 — 水平 / 邊沿觸發。
Q4 「即時」是什麼意思
你的 20 Hz（$50\,\text{ms}$ 截止期限）控制迴圈每週期平均 $4\,\text{ms}$，但偶爾會尖峰到 $60\,\text{ms}$。這個迴圈是即時的嗎？
  • 是——平均值遠低於截止期限，所以它從容地算即時。
  • 不是——即時是一個對最壞情況的保證；單一個 $60\,\text{ms}$ 尖峰就錯過了 $50\,\text{ms}$ 截止期限，這就是失敗。
  • 是，只要尖峰發生的時間少於 1%。
  • 不知道 CPU 時脈頻率就無法判斷。
即時的正確性是由最壞情況延遲、而非平均值來定義的。一個迴圈趕上截止期限 999 次、錯過一次，就不是硬即時。那個尖峰才是要追的 bug——通常是一個 IRQ、分頁錯誤，或阻塞的 fsync——而拿來幹掉它的工具是 PREEMPT_RT、SCHED_FIFO、mlockall 與 IRQ 親和性。見 §6 — 即時。
