udev 與持久命名：讓 /dev/gps 永遠代表 GPS

你把一個 GPS 與一個 IMU 接進 USV 的 USB 集線器，開啟 /dev/ttyUSB0 給 GPS 用，出貨了，一週後這艘船卻朝著浮標轉去——因為一次冷開機（cold-boot）競態（race）把 ttyUSB0 交給了 IMU。本章要教你的，就是怎麼讓裝置的名字跟著裝置本身走，而不是跟著核心碰巧探測（probe）它的順序走。

1重新編號問題

① 直覺 為什麼編號是個謊言

像 ttyUSB0、ttyUSB1、video0、sda 這些核心裝置節點（node），是按照核心列舉（enumeration）它們的順序編號的，而那個順序就是它們的驅動程式完成探測的順序。這個順序並不穩定。它取決於 USB 集線器的拓樸（topology）、哪個裝置先上電並完成連結建立、驅動程式載入順序，以及你有沒有重新插拔過什麼。兩個一模一樣的 FTDI 轉接器，在同一台機器的兩次開機之間就可能互換編號。那個索引是一張到場號碼牌，不是身分。

⚠️ 陷阱——核心名稱是競態的結果

用名字開啟 /dev/ttyUSB0，等於是在開啟「本次開機最先探測到的那個序列裝置」。在一台帶有兩個以上序列轉接頭的機器人上，那是一場你每次重新上電都要重擲一次的擲硬幣賭局。在正式環境裡，永遠不要把核心指派的索引寫死。

② 數學 到底能爛到什麼地步

把這場賭注量化。如果匯流排（bus）上有 $n$ 個彼此無法區分的序列裝置，核心可以用任何一對一映射把索引 ttyUSB0..ttyUSB(n-1) 指派出去。相異排列的數目，就是 $n$ 個物件的排列數：

$$ P(n) = n! = n \times (n-1) \times \cdots \times 2 \times 1 $$

這些排列裡，只有一個是你的程式碼所假設的那個，所以一次隨機開機剛好給你寫死的那種佈局的機率是 $1/n!$。階乘咬得很快：兩個裝置有 $2! = 2$ 種排列（弄錯的機率 50%），三個有 $3! = 6$ 種，四個有 $4! = 24$ 種，五個有 $5! = 120$ 種。一艘載著 GPS、IMU、一個 USB 轉 CAN 轉接器和一個 USB 數據機的 USV，每次開機就已經活在一場 24 選一的樂透裡。

🔑 整章濃縮成一句話

別再用到場索引來命名裝置了。用身分來命名它們——廠商、產品、序號，或實體連接埠——並讓一個使用者空間（user space）守護程式（daemon），也就是 udev，在裝置一出現的那一刻就建立一個穩定的別名。

2從核心 uevent 到 /dev 裡的一個節點

① 直覺 到底是誰造出那個節點的

當一個裝置出現時，核心並不決定在 /dev 裡要叫它什麼。核心在內部建立這個裝置，把它註冊進裝置模型（也就是第 02 章那棵 /sys 樹），接著發出一則叫做 uevent 的小訊息——幾行 KEY=value 文字——透過一個 netlink 通訊端（socket）（NETLINK_KOBJECT_UEVENT）往上送到使用者空間。一個守護程式 systemd-udevd 在那個通訊端上監聽，把事件拿去跟一大堆規則（rule）比對，它才是真正套用擁有權、權限模式與你那些符號連結（symlink）的東西。

💡 機制 vs. 政策

核心提供機制（mechanism）（裝置存在了，這是它的屬性）。udev 提供政策（policy）（叫它這個名字、讓那個群組擁有它、把符號連結放這裡）。正是這個切分，命名才得以可設定：它住在你掌控的使用者空間文字檔裡，而不是在核心裡。

注意，devtmpfs（第 02 章）其實已經很早就給了你一個像 /dev/ttyUSB0 這樣的裸節點——核心會預先填好它，免得系統在 udev 跑起來之前處於沒有節點的狀態。udev 接著才過來，依照政策替它重新標籤／改權限（chmod）／建立符號連結。兩者都有份：devtmpfs 造出原始節點，udev 套用規則。

3規則剖析：match 鍵 vs. 賦值

① 直覺 一條規則就是「若全部成立，則做某事」

一條 udev 規則就是單獨一行：一串以逗號分隔的 match 鍵（match keys）（條件）後面接著 賦值鍵（assignment keys）（動作）。udev 為每一個事件讀過每一條規則；若一行上的所有 match 鍵都為真，它就執行那一行的賦值。初學者最常犯的單一錯誤，就是把兩種 = 搞混，所以把這點刻進腦子裡：

運算子種類意義
==match（測試）「這個鍵等於這個值嗎?」——一個條件
!=match（測試）「這個鍵不等於這個值嗎?」
=賦值設定其值（取代先前的任何清單）
+=賦值附加到一個清單值的鍵上（例如新增一個符號連結）
:=賦值設定並且鎖定——後面的規則無法再更動它
⚠️ 陷阱——你本想寫 == 卻寫成 =

寫 SUBSYSTEM="tty"（單一 =）並不會測試子系統——它試圖去賦值給一個唯讀的 match 鍵，於是你的規則會默默地做錯事或什麼都不做。條件永遠是 == / !=；動作是 = / += / :=。把這個搞反，你會白白損失一個下午。

② 詞彙 你實際會用到的那些鍵

你會倚重的 match 鍵：SUBSYSTEM（例如 tty、net、block）、KERNEL（核心名稱，支援像 ttyUSB* 這樣的萬用字元）、ATTRS{idVendor}（在這個裝置或它任一個上層裝置上的一個 sysfs 屬性——這就是你往上探到 USB 裝置去拿 idVendor 的方法），以及 ENV{ID_SERIAL}（由先前某條規則設定的屬性，例如 ID_SERIAL）。賦值鍵：NAME（網路介面——重新命名裝置本身）、SYMLINK+=（在 /dev 裡加一個穩定別名）、MODE、OWNER、GROUP，以及 RUN+=。

🛠️ ATTR vs. ATTRS（差一個字母，差很多）

ATTR{idVendor} 只比對當前裝置上的一個 sysfs 屬性。ATTRS{idVendor}（帶 S，代表「search／搜尋」）會沿著上層鏈往上走，直到找到一個帶有那個屬性的裝置。一個 ttyUSB 節點本身沒有 idVendor——那住在好幾層之上的 USB 裝置上——所以你幾乎總是要用 ATTRS{idVendor}，而不是 ATTR{idVendor}。

4用 udevadm info 找出 match 鍵

③ 程式碼 先審問這個裝置

永遠別猜屬性——去問。udevadm info --attribute-walk（簡寫 -a）從裝置節點出發，沿著上層鏈往上走，在每一層把你能比對的每一個 ATTRS{idVendor} 之類的鍵都印出來。就是這一道指令，把「我手上有個轉接頭」變成「我有一條規則」。

替 GPS 遍歷裝置樹 bash
udevadm info --attribute-walk --name=/dev/ttyUSB0
# or the short form:
udevadm info -a -n /dev/ttyUSB0

輸出是一疊區塊，裝置本身在最前，接著是每一個上層裝置。從上往下讀，從帶有 USB 識別碼的那個上層區塊裡抓出那些鍵：

精簡過的 udevadm 輸出 bash

    KERNEL=="ttyUSB0"

🔑 該採收什麼

你要的是 idVendor + idProduct 來挑出型號，以及 serial 來挑出個別的那一台。兩個不同型號的轉接頭？廠商＋產品就夠了。兩個一模一樣的轉接頭（相同廠商／產品）？你必須用 serial 才能把它們分開——或者退而用實體連接埠（§7）。

⚠️ 別把不同上層區塊的屬性混在一起

你放進同一條規則裡的所有 ATTRS{idVendor} 之類的鍵，都必須來自同一個上層區塊。udev 是拿它們去比對鏈上的同一個裝置——它不會把某個上層的 idVendor 跟另一個上層的 serial 拼湊在一起。把它們全部從 SUBSYSTEMS=="usb" 那個區塊裡拿。

5撰寫穩定符號連結的規則

③ 程式碼 規則本體

規則住在 /etc/udev/rules.d/（你的覆寫）與 /lib/udev/rules.d/（發行版的）。檔案是依檔名按字典序讀取的——這就是為什麼大家都用一個數字當前綴，例如 99-usv.rules，好讓它晚點才跑、勝出。下面這條規則就把 GPS 釘死：

/etc/udev/rules.d/99-usv.rules bash
# GPS: PL2303 dongle, serial 0001 -> /dev/gps
SUBSYSTEM=="tty", ATTRS{idVendor}=="067b", ATTRS{idProduct}=="2303", \
  ATTRS{serial}=="0001", SYMLINK+="gps", GROUP="dialout", MODE="0660"

# IMU: identical-model dongle, different serial 0002 -> /dev/imu
SUBSYSTEM=="tty", ATTRS{idVendor}=="067b", ATTRS{idProduct}=="2303", \
  ATTRS{serial}=="0002", SYMLINK+="imu", GROUP="dialout", MODE="0660"

把它當成一句話來讀：「若子系統是 tty 且上層 USB 廠商是 067b 且產品是 2303 且序號是 0001，則加一個指向這個節點的符號連結 /dev/gps，把群組設成 dialout，把模式設成 0660。」每一個 == 都是條件；每一個 +=／= 都是動作。那個反斜線只是為了好讀而把這一行接續下去——一條真正的規則是邏輯上的一行。

💡 為什麼用符號連結，而不是改名

SYMLINK+= 是在 /dev/ttyUSB0 仍然存在的同時加上 /dev/gps 當別名——其他任何倚賴核心名稱的東西都不會壞掉。把節點整個改名（更動 NAME）是保留給網路介面用的（§7）；對於字元／區塊裝置，你用符號連結。而用 +=（而非 =）的意思是，你可以對同一個裝置加上好幾個符號連結，而不會把其他規則加上去的那些蓋掉。

② 權限的數學 0660 是什麼意思

MODE 是一組八進位權限三元組——擁有者、群組、其他人——每一位數都是讀（4）、寫（2）、執行（1）之和。所以 0660 是擁有者可讀＋寫（$4+2=6$）、群組可讀＋寫（$6$）、其他人什麼都不行（$0$）：$\,rw\text{-}\,rw\text{-}\,---$。配上 GROUP="dialout"，任何使用者屬於 dialout 群組的行程，都能不靠 root 就開啟那個序列埠。對一個不該以 root 執行的機器人服務來說，這正是你要的。

6測試：udevadm test 與 trigger

③ 程式碼 別靠重開機才知道結果

編輯完規則後，先重新載入它們，然後要嘛對單一裝置做乾跑（dry-run），要嘛對整個子系統重新發送事件。兩道不同的指令，兩件不同的工作：

指令它做什麼
udevadm control --reload從磁碟重新載入規則（每次編輯後都做這件事）
udevadm test /sys/class/tty/ttyUSB0乾跑：模擬該事件，印出哪些規則命中、它們會做什麼——不做任何真實更動
udevadm trigger --action=add為既有裝置重新發出 add uevent，讓規則當下就套用而不必重新插拔
編輯 → 測試 → 套用 的循環 bash
sudo nano /etc/udev/rules.d/99-usv.rules     # edit
sudo udevadm control --reload                # reload rules

# dry-run against the real device path under /sys (no changes made):
sudo udevadm test /sys/class/tty/ttyUSB0 2>&1 | grep -E 'SYMLINK|GROUP|MODE'

# apply for real to currently-attached tty devices:
sudo udevadm trigger --subsystem-match=tty --action=add
ls -l /dev/gps /dev/imu                       # verify the symlinks exist
🛠️ test 模擬，trigger 落實

udevadm test 告訴你會發生什麼事，整天跑都安全。udevadm trigger 則真的把事件重播、把規則套用上去。經典的工作流程是：reload，test 跑到輸出裡出現你那行 SYMLINK 'gps' 為止，然後 trigger 讓它成真——不重開機、不拔插。

7by-id、by-path 與可預測的網路名稱

① 直覺 你說不定根本不需要寫規則

預設的 udev 規則已經替序列裝置建好了兩個穩定的目錄，所以想快速搞定的話，你大可完全跳過寫規則這件事：

  • /dev/serial/by-id/ ——以廠商＋產品＋序號為鍵。跨重開機、跨連接埠都穩定。每個裝置都有唯一序號時用這個。
  • /dev/serial/by-path/ ——以實體 USB 拓樸（哪個集線器、哪個埠）為鍵。只要你不挪動線材就穩定。沒有可用序號的一模一樣裝置用這個——連接埠就成了它的身分。
udev 早就替你建好的符號連結 bash
ls -l /dev/serial/by-id/
# usb-Prolific_USB-Serial_Controller_0001-if00-port0 -> ../../ttyUSB0

ls -l /dev/serial/by-path/
# platform-3610000.usb-usb-0:2.1:1.0-port0 -> ../../ttyUSB0
🛠️ 唯一序號用 by-id，「相同型號插在已知埠」用 by-path

兩台序號相異的 GPS → by-id（或一條以序號為鍵的規則）。兩個一模一樣、而你總是插進固定那兩個埠的感測器 → by-path（或一條 ATTRS{devpath} 規則），因為連接埠是唯一不同的東西。

② 可預測的網路介面 同樣的問題，.link 是答案

網卡（NIC）有一模一樣的競態：核心的舊方案按探測順序把它們命名成 eth0、eth1，所以一個 USB 轉乙太網路的轉接器，可能從內建埠那兒把 eth0 偷走。systemd 的解法是可預測網路介面名稱（Predictable Network Interface Names）——由硬體位置衍生出來的名字，像 enp1s0（PCI 匯流排 1、插槽 0）或 enx + MAC。它們很醜，但具決定性。要釘上一個你選定的名字，就寫一個 systemd .link 檔，以 MAC 來比對並重新命名介面：

/etc/systemd/network/10-sensor-net.link bash
[Match]
MACAddress=00:04:4b:aa:bb:cc

[Link]
Name=eth_lidar

一個 .link 檔由 systemd-udevd 在非常早的時候處理（在網路裝置的 add 事件期間），趕在介面被啟用之前，這就是為什麼在那裡改名是安全的——不像一條晚到的 udev NAME= 規則，一旦介面正在使用中，核心可能就拒絕它。重新命名網卡，請固守 .link 這套機制。

🏭 真實案例——Jetson AGX Orin USV 的開機調校

在這艘 USV 的 AGX Orin 上，我們出貨一份 99-usv.rules，把 u-blox GPS 釘到 /dev/gps（以序號為鍵）、把 VectorNav IMU 釘到 /dev/imu，兩者都 GROUP="dialout", MODE="0660"，好讓 ROS 2 驅動程式以非特權的 usv 使用者身分執行。承載 LiDAR 的那個內建乙太網路埠，透過一個以 MAC 比對的 .link 檔被釘到 eth_lidar，所以一個插進來的 USB 轉乙太網路轉接頭（給現場筆電用的）永遠偷不走它。結果：每一個 launch 檔都開啟 /dev/gps、/dev/imu，並把 LiDAR 的通訊端綁定到 eth_lidar——這些名字熬得過重開機、重新插拔，以及新增的硬體。

8RUN、MODE、GROUP：擁有權與副作用

④ 真實案例 讓裝置能用、而且會回應

釘好一個名字只是戰鬥的一半；行程還得被允許去開啟它，而且有時候裝置一出現就該發生某件事。這正是其餘那些賦值鍵在做的事：

鍵效果
OWNER="usv"設定節點的擁有使用者
GROUP="dialout"設定擁有群組（成員可取得群組存取權）
MODE="0660"權限位元（擁有者＋群組可讀寫，其他人不行）
RUN+="program"在裝置設定完成之後跑一支短程式
TAG+="systemd"讓一個 systemd 單元可以 BindsTo= 這個裝置
⚠️ 陷阱——RUN 不是用來啟動守護程式的地方

RUN+= 是同步地在 udev 事件內部執行的，若它不快點返回就會被殺掉。它是給快速設定用的——設定 FTDI 的 latency_timer、把一條 CAN 連結拉起來、丟下一個旗標檔——絕不用來啟動長時間執行的服務。要在插入時啟動一個服務，請用 systemd 替裝置打標籤，並用一個 BindsTo=／WantedBy= 的單元。在 RUN 裡阻塞會卡死整個 udev。

RUN 用於一次快速、有界的設定任務 bash
# When the FTDI GPS appears, drop its read latency from 16ms to 1ms.
SUBSYSTEM=="tty", ATTRS{idVendor}=="0403", ATTRS{idProduct}=="6001", \
  ATTRS{serial}=="A50285BI", SYMLINK+="gps", \
  RUN+="/bin/sh -c 'echo 1 > /sys%p/device/latency_timer'"

這裡 %p 展開成 /sys 底下的核心裝置路徑，所以這條規則在裝置一被辨識出來的那一刻，就寫入 FTDI 的 latency_timer 屬性（來自第 02 章）——一次性、有界的寫入，正是 RUN 該做的事。

9換你了

Q1 觀念檢查
在一條 udev 規則裡，哪一個運算子是測試一個條件（一個 match 鍵），而非執行一個動作?
  • = ——它把值賦上去以供比較。
  • == ——match 鍵測試相等；=/+=/:= 是賦值。
  • += ——它把這個條件附加到一個清單裡。
  • := ——它把這個條件鎖住讓它不能改。
match 鍵用 ==（或 !=），它們是條件——「這個屬性等於那個值嗎?」。賦值鍵用 =（設定）、+=（附加，如 SYMLINK+=）或 :=（設定並鎖定）。用單一 = 寫成 SUBSYSTEM="tty" 是經典的 bug：它試圖賦值給一個唯讀的 match 鍵，而不是去測試它。
Q2 算一算
這艘 USV 在一個集線器上有 5 個無法區分的 USB 序列裝置。核心可以用多少種相異的排列來指派 ttyUSB0..ttyUSB4?（提示：$n!$。）
種排列
$5! = 5 \times 4 \times 3 \times 2 \times 1 = 120$。這 120 種裡只有一種是你程式碼所假設的佈局，所以一次盲目的開機只有 $1/120 \approx 0.83\%$ 的機率對。以序號為鍵的 udev 符號連結讓這機率變成 100%。
Q3 動手做——寫一條能用的規則
一個 u-blox GPS 在 tty 子系統上回報 ATTRS{idVendor}=="1546"、ATTRS{idProduct}=="01a8"、ATTRS{serial}=="DBE1F2C0"。寫一條 udev 規則，建立 /dev/gps，由群組 dialout 擁有、模式為 0660。然後給出不重開機就能套用它的那兩道指令。
提示
條件用 ==；符號連結用 SYMLINK+=。重新載入規則，然後對 tty 子系統重新發送 add 事件。
看一個解答
/etc/udev/rules.d/99-usv.rules bash
SUBSYSTEM=="tty", ATTRS{idVendor}=="1546", ATTRS{idProduct}=="01a8", \
  ATTRS{serial}=="DBE1F2C0", SYMLINK+="gps", GROUP="dialout", MODE="0660"
套用它 bash
sudo udevadm control --reload
sudo udevadm trigger --subsystem-match=tty --action=add
ls -l /dev/gps        # verify the symlink resolves to the right ttyUSB*

每一個 == 都是對上層 USB 裝置的一個條件（所以必須是 ATTRS，不是 ATTR）；SYMLINK+= 在不干擾核心名稱的前提下加上別名；GROUP＋MODE 讓你的非 root 服務能開啟那個埠。--reload 讀進這個檔案，trigger 重播事件，於是連結立刻出現。

🔑 本章回顧

像 ttyUSB0 這類核心索引就是到場號碼牌——有 $n$ 個裝置時就有 $n!$ 種可能排列，而只有一種是你的。核心發出一則 uevent；systemd-udevd 把它跟規則比對並套用政策。一條規則是 match 鍵（==／!=）後接賦值（=／+=／:=）；用 udevadm info -a -n /dev/ttyUSB0 找出那些鍵，以 ATTRS{idVendor/idProduct/serial} 為鍵，加上 SYMLINK+="gps" 並帶 GROUP／MODE，然後 udevadm test（乾跑）與 trigger（套用）——不重開機。免費的部分，就用 /dev/serial/by-id 與 by-path；對網卡，用一個 systemd .link 檔把名字釘住。下一章：驅動程式、模組與韌體——在那個名字背後，到底是什麼東西綁定到裝置上。
