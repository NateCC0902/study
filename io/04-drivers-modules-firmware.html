Drivers, Modules & Firmware: the code that owns the hardware

In Chapter 02 a device showed up as a file; in Chapter 03 udev gave it a stable name. But who turns register pokes into a working sensor? A driver. This chapter is about where driver code lives, how it gets loaded, why it sometimes loads but then begs for a firmware blob it can't find, and why the vendor module that worked yesterday refuses to load after a kernel upgrade. By the end you'll read a bind in dmesg the way you read a stack trace.

⏱ ~40 min🧠 kernel modules⚡ modprobe∑ kABI coupling

1Two places a driver can live

① Intuition build the picture first

A driver is just the code that knows the private language of one piece of hardware — which register means "start a conversion," which bit means "data ready," how an interrupt maps to bytes. The only real question is which side of the syscall boundary that code runs on.

A kernel driver runs inside the kernel, in ring 0. It can touch physical memory, take interrupts, and program DMA directly. When it fails, it can take the whole machine down. A userspace driver is an ordinary program that pokes the hardware through a thin generic kernel shim (spidev, i2c-dev, libusb, UIO/VFIO). It can't crash the kernel, you can attach gdb to it, and you can ship it without recompiling anything — but every register access is a syscall, so it's slower and can't service hard interrupts in the bottom half.

💡 The one decision

Need raw speed, real interrupts, or DMA? Kernel driver. Want to iterate fast, debug in userspace, and never risk a kernel panic for a sensor that runs at 100 Hz? Userspace driver over i2c-dev/spidev/libusb. Most robot sensors are happy in userspace; the network card and the IMU's DMA engine are not.

2A module's life: init, exit, and the gap between

② Math the lifecycle as a state machine

A kernel module (a .ko file — "kernel object") is driver code you can load and unload at runtime instead of baking it into the kernel image. Every module has exactly two ceremony functions: one that runs at load, one at unload. Everything else hangs off them.

usv_baro.c C
#include <linux/module.h>
#include <linux/init.h>

static int __init baro_init(void)
{
    pr_info("usv_baro: loaded, registering driver\n");  /* lands in dmesg */
    return 0;            /* return < 0 to ABORT the load (e.g. -ENODEV) */
}

static void __exit baro_exit(void)
{
    pr_info("usv_baro: unloaded\n");
}

module_init(baro_init);      /* called by insmod / modprobe       */
module_exit(baro_exit);      /* called by rmmod  / modprobe -r    */

MODULE_LICENSE("GPL");                 /* non-GPL taints the kernel */
MODULE_AUTHOR("USV team");
MODULE_DESCRIPTION("Example barometer driver");
MODULE_VERSION("1.0");

The subtle part: module_init does not mean "talk to my device now." For a bus driver, init just registers the driver with a bus (PCI, USB, I²C…). The kernel then calls your .probe() later, once for every matching device the bus enumerates. Init can succeed while probe never fires — the module is loaded but bound to nothing. That distinction is the whole game in §6.

🛠️ The four verbs

lsmod lists loaded modules and their use count (you can't rmmod a module with refcount > 0). modinfo foo.ko dumps its metadata — license, parameters, dependencies, and the vermagic string. insmod loads one exact file. modprobe foo loads foo and everything it depends on, by name, from the system module tree.

3insmod vs modprobe: dependency resolution

③ Code watch the dependency graph resolve

Modules call each other's exported symbols, so they form a dependency graph. insmod /path/to/foo.ko loads only that file by absolute path — if foo needs symbols from bar and bar isn't loaded yet, insmod fails with Unknown symbol in module. It does no searching and no resolution. It is the blunt instrument.

modprobe foo is the civilized tool. It looks up foo in /lib/modules/$(uname -r)/modules.dep, walks the dependency chain, and loads everything in the right order — bottom-up. That dependency map is generated by depmod, which scans every .ko under the current kernel's module directory and records who exports and who needs each symbol.

resolving a vendor WiFi module bash
# After dropping new .ko files into /lib/modules/$(uname -r)/extra/,
# rebuild the dependency + alias maps for THIS kernel:
sudo depmod -a

# modprobe now knows moal depends on mlan and loads both, in order:
sudo modprobe moal mod_para=nxp/wifi_mod_para.conf

# What insmod would have forced you to do by hand, in order:
sudo insmod /lib/modules/$(uname -r)/extra/mlan.ko
sudo insmod /lib/modules/$(uname -r)/extra/moal.ko mod_para=nxp/wifi_mod_para.conf

# Inspect before loading — never trust a blob you didn't build:
modinfo moal | grep -E '^(filename|version|vermagic|depends|parm):'
⚠️ Forgot depmod, broke modprobe

If you copy a fresh .ko into the module tree and skip depmod -a, modprobe says FATAL: Module foo not found even though the file is sitting right there. modprobe reads modules.dep, not the directory. No depmod, no index, no find.

Both tools ultimately call the same syscall — finit_module(2) (or the older init_module(2)) — to hand the kernel a file descriptor for the .ko and trigger module_init. modprobe is just insmod plus a graph walk and an alias lookup.

4Module parameters & /etc/modprobe.d

③ Code configure without recompiling

Modules expose tunables as parameters, declared in C with module_param(). You can set them on the command line at load time, read them at runtime under /sys/module/<name>/parameters/, and — for the ones that are writable — change them live by writing that sysfs file.

module parameter in C C
static int sample_rate_hz = 100;
module_param(sample_rate_hz, int, 0644);   /* 0644 => appears in sysfs, root-writable */
MODULE_PARM_DESC(sample_rate_hz, "barometer sample rate in Hz");
/* modprobe usv_baro sample_rate_hz=200
 * cat /sys/module/usv_baro/parameters/sample_rate_hz   -> 200            */

Typing parameters by hand every boot is how you forget them. Persist them in /etc/modprobe.d/*.conf. Four directives matter:

DirectiveDoes
options moal mod_para=…default parameters whenever the module loads
blacklist nouveaudon't auto-load this module (won't be picked by udev)
alias usb:v1286p… moalmap a hardware ID / alias to a module name
install foo /bin/falserun a command instead of loading (a hard block)
/etc/modprobe.d/usv-wifi.conf bash
# Vendor params persist across reboots, and udev picks them up on hotplug.
options moal mod_para=nxp/wifi_mod_para.conf
# Keep the in-tree mwifiex out of the way so our out-of-tree moal binds instead:
blacklist mwifiex
blacklist mwifiex_pcie
⚠️ blacklist is not a kill switch

blacklist only stops automatic loading via alias resolution. A direct modprobe mwifiex or another module's dependency will still pull it in. To truly forbid loading, use install mwifiex /bin/true.

5The firmware loader and error -2

① Intuition the driver is the brain, firmware is the personality

Plenty of devices ship as a dumb microcontroller with no program in it. The Linux driver knows how to talk to the chip, but the chip itself needs a binary blob — its firmware — pushed into it at probe time before it does anything useful. WiFi radios, some cameras, GPUs, and many SoC peripherals work exactly this way. The driver requests the blob; the kernel's firmware loader finds the file on disk and hands it over.

② Math where the loader looks

A driver calls request_firmware(&fw, "nxp/sd8997_uapsta.bin", dev). The kernel searches a fixed set of paths, in order, for that relative name — first match wins, and the updates/ trees come before the bare distro tree precisely so they can override it:

$$ \underbrace{\texttt{firmware\_class.path}}_{\text{optional, custom}} \;\to\; \texttt{/lib/firmware/updates/}\langle\texttt{uname -r}\rangle\texttt{/} \;\to\; \texttt{/lib/firmware/updates/} \;\to\; \texttt{/lib/firmware/}\langle\texttt{uname -r}\rangle\texttt{/} \;\to\; \underbrace{\texttt{/lib/firmware/}}_{\text{distro default, last}} $$

If the file is found, the call returns 0 and fw->data / fw->size point at the bytes. If it is missing, the call returns a negative errno, and the driver typically logs the now-infamous line. That -2 is not random: it's -ENOENT, "No such file or directory" — the exact same errno open() returns for a missing path.

dmesg — the classic bash
$ dmesg | grep -i firmware
moal: Direct firmware load for nxp/sd8997_uapsta.bin failed with error -2
moal: firmware: failed to load nxp/sd8997_uapsta.bin (-2)
moal: WLAN FW download failed

# Fix: the file just isn't in the search path. Put it there:
$ ls /lib/firmware/nxp/sd8997_uapsta.bin        # -> No such file
$ sudo install -D -m644 sd8997_uapsta.bin /lib/firmware/nxp/sd8997_uapsta.bin
$ sudo rmmod moal mlan && sudo modprobe moal   # re-probe, re-request firmware
🏭 Real case — USV WiFi on the AGX Orin

On the Jetson AGX Orin, the WiFi M.2 card runs the NXP mlan/moal out-of-tree stack. The module loads cleanly (init succeeds), the SDIO/PCIe device is enumerated, probe starts — and then moal calls request_firmware() for nxp/sd8997_uapsta.bin. Forget to stage that blob into /lib/firmware/nxp/ when you flash the image and you get a network interface that appears (ip link shows mlan0) but never associates — error -2 in the log. The driver matched; the device just has no brain loaded. This is the general pattern for every firmware-hungry peripheral, not a one-off bug.

6In-tree, out-of-tree, kABI & DKMS

② Math why a kernel bump breaks your driver

An in-tree driver lives in the kernel source and is built and versioned with the kernel — upgrade the kernel and the driver comes along, always matched. An out-of-tree driver (a vendor blob like moal) is compiled separately against one specific kernel build, and that is the source of all the pain.

The kernel does not promise a stable binary interface to modules. Every .ko carries a vermagic string and CRC checksums of the kernel symbols it uses — together the kABI (kernel Application Binary Interface). At load time the kernel checks them. If the running kernel's symbol CRCs don't match what the module was built against, the load is rejected:

$$ \texttt{load ok} \iff \texttt{vermagic}_{module} = \texttt{vermagic}_{kernel} \;\;\wedge\;\; \forall s \in \text{symbols}:\; \mathrm{CRC}_{module}(s) = \mathrm{CRC}_{kernel}(s) $$

The vermagic string bundles the kernel version, SMP/preempt flags, and compiler details, e.g. 5.15.148-tegra SMP preempt mod_unload aarch64. Change any of those — even a patch bump from 5.15.148 to 5.15.149 — and the string differs, so the module is refused with:

the kernel-bump failure bash
$ sudo modprobe moal
modprobe: ERROR: could not insert 'moal': Exec format error
$ dmesg | tail -2
moal: version magic '5.15.148-tegra SMP preempt mod_unload aarch64'
      should be '5.15.149-tegra SMP preempt mod_unload aarch64'
# The module was built for the OLD kernel. It must be REBUILT for the new one.

The fix that scales is DKMS (Dynamic Kernel Module Support). You register the driver's source once with a dkms.conf; thereafter, whenever a new kernel is installed, DKMS automatically recompiles the module against the new headers and installs it. No DKMS means: every kernel upgrade silently leaves you with a WiFi-less robot until someone rebuilds the module by hand.

🛠️ DKMS in one breath

dkms add -m moal -v 1.0 registers the tree, dkms build compiles against /lib/modules/$(uname -r)/build, dkms install drops the .ko and runs depmod. After that, kernel package upgrades trigger the rebuild for you. Confirm with dkms status.

7Userspace drivers: spidev, i2c-dev, libusb, UIO/VFIO

③ Code drive hardware from a normal process

You don't always need to write kernel C. The kernel ships generic shim drivers that expose a bus to userspace as a char device, and your program does the protocol over ioctl/read/write:

ShimExposesYou useGood for
i2c-dev/dev/i2c-Nioctl(I2C_RDWR)I²C baro/IMU/RTC
spidev/dev/spidevB.Cioctl(SPI_IOC_MESSAGE)SPI ADC, fast sensors
libusbUSB devices directlylibrary callsUSB GPS, RealSense
UIO/dev/uioN + sysfsmmap + IRQ via readsimple MMIO peripherals
VFIO/dev/vfio/*IOMMU-safe MMIO + DMAhigh-rate, DMA, passthrough
read a register over i2c-dev C
int fd = open("/dev/i2c-1", O_RDWR);
ioctl(fd, I2C_SLAVE, 0x77);                 /* 7-bit address of the baro */
unsigned char reg = 0x00, val;              /* WHO_AM_I register         */
write(fd, &reg, 1);                         /* set the register pointer  */
read(fd, &val, 1);                          /* read it back              */
/* No kernel module to compile. Just a process holding an fd.            */

UIO and VFIO go further: they let userspace mmap the device's MMIO region and even receive interrupts (UIO turns an IRQ into a readable fd; VFIO adds IOMMU protection so userspace DMA can't scribble over kernel memory). That's how DPDK and many camera/FPGA stacks run a full driver in userspace at near-kernel speed. The trade-off from §1 still holds — a syscall per access unless you mmap — but for a 100 Hz baro that floor is invisible.

8Reading a bind in dmesg

④ Real case follow the handshake

When you suspect a driver problem, dmesg is the source of truth — it's the kernel's running narration of enumeration, matching, probing, and firmware loads. Read it top to bottom and the bind tells its own story:

a successful bind, annotated bash
$ sudo dmesg -w        # -w = follow, like tail -f, while you replug
mmc1: new high speed SDIO card at address 0001          # 1. bus enumerated device
moal: MOAL driver version 1.3.x, mlan version 1.3.x     # 2. module init / registered
moal: SDIO: max_segs=128 max_seg_size=65535            #    driver bound to the device
moal: wlan: Driver loading firmware nxp/sd8997_uapsta.bin  # 3. request_firmware()
moal: WLAN FW is active                                 # 4. firmware accepted
moal: version = SD8997-...                              # 5. device reports its identity
mlan0: associated                                       # 6. usable netdev is up

If the chain stops after step 1, your driver didn't match — check the id table / aliases (modinfo, /etc/modprobe.d). If it stops at step 3 with error -2, the firmware file is missing (§5). If modprobe itself failed with Exec format error, you never got to step 2 — that's the kABI mismatch from §6. Each failure mode lives at a specific step, and dmesg tells you which one.

🔑 Three commands for three failures

lsmod — is it even loaded? modinfo — what does it want (params, firmware, vermagic)? dmesg — how far did the bind get? Master those three and most "the sensor isn't showing up" tickets resolve in two minutes.

9Your turn

Q1 Conceptual check
Your out-of-tree moal WiFi module worked perfectly. After apt upgrade pulled a new kernel and you rebooted, modprobe moal fails with Exec format error. Why?
  • The firmware blob sd8997_uapsta.bin was deleted by the upgrade.
  • The module was built against the old kernel; its vermagic/symbol-CRC (kABI) no longer matches the running kernel, so the load is rejected.
  • A new kernel always blacklists out-of-tree modules for security.
  • The new kernel ships an in-tree driver that physically removed moal.ko.
The kernel offers no stable kABI to modules. The .ko embeds a vermagic string and CRCs of the symbols it imports; at load the kernel compares them to its own. A version change — even a patch bump — changes vermagic, so the load is refused with Exec format error. The fix is to rebuild for the new kernel; DKMS automates exactly this on every kernel install. Firmware (error -2) is a different, later failure.
Q2 Compute it
You must stage firmware blobs onto the USV's eMMC before flight. There are 7 firmware-hungry peripherals, each needing one blob averaging $2\ \text{MiB}$. Your install step writes to eMMC at an effective $1\ \text{MiB/s}$. How many seconds does staging all the blobs take? (Hint: total bytes ÷ rate.)
s
Total $= 7 \times 2 = 14\ \text{MiB}$. At $1\ \text{MiB/s}$, time $= 14 / 1 = 14\ \text{s}$. The point isn't the arithmetic — it's that firmware staging is a real, measurable step in your image build, and forgetting it produces the silent "interface appears but never associates" error -2 from §5.
Q3 Hands-on — load with a parameter and verify
Load the usv_baro module with sample_rate_hz=200, then prove (a) it's loaded and (b) the parameter actually took effect. Assume the .ko is already in the module tree but you just copied it in.
Hint
A freshly copied module needs the dependency index rebuilt before modprobe can find it. Then check lsmod for the name and read the live value from /sys/module/<name>/parameters/. dmesg confirms the bind.
Show one solution
load + verify bash
# Rebuild modules.dep so modprobe can resolve the freshly-copied .ko:
sudo depmod -a

# Load it (and any deps) with the parameter:
sudo modprobe usv_baro sample_rate_hz=200

# (a) Confirm it's loaded — name + nonzero size, used-by count:
lsmod | grep usv_baro
#   usv_baro    16384  0

# (b) Confirm the parameter took — read it back from sysfs:
cat /sys/module/usv_baro/parameters/sample_rate_hz
#   200

# Follow the bind in the log:
dmesg | grep usv_baro
#   usv_baro: loaded, registering driver

The three checks map to the three commands in §8: lsmod proves it loaded, the /sys/module/.../parameters/ read proves the option reached the driver, and dmesg proves the lifecycle ran. If you'd skipped depmod -a, step two would have failed with FATAL: Module usv_baro not found even though the file exists.

🔑 Chapter recap

A driver runs either in the kernel (fast, dangerous, direct MMIO/IRQ/DMA) or in userspace (safe, debuggable, via i2c-dev/spidev/libusb/UIO/VFIO). Kernel modules have an init/exit lifecycle, but module_init only registers — binding happens later in .probe(). modprobe beats insmod because it resolves dependencies from modules.dep (rebuilt by depmod); parameters persist in /etc/modprobe.d and appear under /sys/module. Firmware-hungry chips call request_firmware(); a missing blob is the classic error -2 (-ENOENT). Out-of-tree drivers break on kernel bumps because the kABI (vermagic + symbol CRCs) must match exactly — DKMS rebuilds them automatically. To diagnose any of this, read the bind in dmesg: enumerate → match → probe → firmware → ready. Next: how buses enumerate the devices these drivers bind to.
