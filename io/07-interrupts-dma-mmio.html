How Data Moves: Interrupts, DMA & MMIO

A device has bytes. Your program wants them. Between those two facts lives the single most important performance decision in robotics I/O: who actually copies the data. Get it wrong and a chatty sensor starves your control loop. Get it right and a 100 Mb/s LiDAR barely registers on top.

1Three ways to move a byte

① Intuition build the picture first

There are exactly three mechanisms a CPU can use to get data out of a device's registers and into memory. Everything else — coalescing, NAPI, ring buffers — is an optimization layered on top of these three.

  1. Programmed I/O (PIO): the CPU itself reads each word from a device register and writes it to memory, in a loop. The CPU is the data path. Simple, and a catastrophe at high rates.
  2. Interrupt-driven I/O: the CPU goes off and does other work; when the device has data ready it raises an IRQ (interrupt request) on a physical line. The CPU stops what it's doing, runs a handler, copies the data, and resumes. No busy-waiting — but still the CPU copying.
  3. DMA (Direct Memory Access): a dedicated engine on the bus moves the bytes from the device straight into RAM with the CPU uninvolved. The CPU gets only one interrupt at the end: "your buffer is full." The CPU never touches the payload.
💡 The mailbox analogy

PIO is standing at the mailbox refreshing it every second. Interrupts are a doorbell that rings when mail arrives — you do other things until it rings. DMA is hiring a courier who silently fills your inbox and taps you once it's full. As the mail rate climbs, the doorbell rings nonstop and you get nothing else done; the courier scales effortlessly.

② Math why PIO dies

Say a device delivers $R$ bytes per second and the CPU moves $w$ bytes per access. Under PIO the CPU performs

$$ \text{accesses/s} = \frac{R}{w} $$

A 100 Mb/s sensor is $R = 12.5$ MB/s. With 4-byte word accesses that is $3.1\times10^{6}$ register reads every second, each one a slow uncached MMIO access that can stall the pipeline for hundreds of nanoseconds. The CPU does nothing else. This is why PIO is fine for a one-shot config register and fatal for a stream.

2MMIO: the device looks like memory

① Intuition registers with addresses

How does the CPU "read a device register" at all? On every modern SoC the answer is memory-mapped I/O. The device's control and status registers are wired onto the same physical address bus as RAM, just at a different range. Reading address 0x0c260000 doesn't hit DRAM — the memory controller routes it to a peripheral. To the CPU it's an ordinary load/store; to the hardware it's a command.

The kernel maps that physical range into a driver's virtual address space with ioremap(). Userspace can do the same trick on a permitted device by mmap-ing /dev/mem or a UIO node (we met UIO drivers in Ch 04). You write a C struct pointer over the register block and access fields by name.

③ Code and why volatile is mandatory

Here is the subtle, career-defining part. The compiler assumes memory is normal: if you read the same address twice with nothing writing in between, it caches the value and skips the second load. For a status register the device updates behind the compiler's back, that optimization turns a poll loop into an infinite loop. The fix is volatile: it forbids the compiler from caching, reordering, or eliding the access.

mmio.c C
#include <fcntl.h>
#include <sys/mman.h>
#include <stdint.h>

#define STATUS_REG  0x18      /* offset within the register block */
#define DATA_VALID  (1u << 0)

int fd = open("/dev/mem", O_RDWR | O_SYNC);          /* O_SYNC => uncached */
volatile uint32_t *regs =
    mmap(NULL, 0x1000, PROT_READ | PROT_WRITE,
         MAP_SHARED, fd, 0x0c260000);                /* phys base, page-aligned */

/* WRONG without volatile: compiler reads STATUS_REG once, loops forever. */
while (!(regs[STATUS_REG / 4] & DATA_VALID))
    ;                                                /* spin until device sets bit */

uint32_t sample = regs[0x20 / 4];                    /* read the data register */
⚠️ Pitfall — volatile is not a memory barrier

volatile stops the compiler from reordering. It does not stop the CPU or the bus from reordering, and it is not atomic. Real drivers use readl()/writel() accessors, which add the architecture's MMIO barriers so a "configure, then start" sequence actually reaches the device in order. On ARM64 (the Orin) that ordering is not free — skipping the barrier is the classic "works on x86, hangs on ARM" bug.

3Interrupts: top half, bottom half, threaded IRQ

① Intuition the doorbell and the unwritten rule

An interrupt is brutal: the CPU drops whatever userspace or kernel code was running, masks further interrupts, and jumps to the driver's interrupt service routine (ISR). While the ISR runs, that core is doing nothing else — not your control loop, not the scheduler. So there is one unwritten rule: an ISR must be short. Microseconds, not milliseconds. Never sleep, never wait on a lock that might block.

But real work — parsing a packet, waking a reader, refilling a DMA ring — isn't always short. Linux solves this with a split:

StageContextDoesConstraint
Top half (hard IRQ)interrupt context, IRQs offack the device, grab data into a buffer, schedule the restfast, atomic, cannot sleep
Bottom half (softirq / tasklet / workqueue)softirq or kernel threadthe heavy lifting — protocol parsing, waking readerscan be deferred; workqueue may sleep
Threaded IRQdedicated kernel threadthe whole handler, but schedulable & preemptiblethe default on PREEMPT_RT
③ Code a threaded handler

The modern in-tree pattern is request_threaded_irq(): a tiny top half that quiets the device and returns IRQ_WAKE_THREAD, and a thread function that does everything else in schedulable context. This is gold for real-time robotics — you can give the IRQ thread an RT priority and it competes fairly with your control loop instead of preempting it blindly.

sensor_drv.c C
/* TOP HALF: runs in hard-IRQ context. Keep it tiny. */
static irqreturn_t sensor_hardirq(int irq, void *dev_id)
{
    struct sensor *s = dev_id;
    u32 st = readl(s->base + STATUS_REG);     /* MMIO read w/ barrier */
    if (!(st & IRQ_PENDING))
        return IRQ_NONE;                       /* not ours (shared line) */
    writel(st, s->base + STATUS_REG);          /* ack: clear the bit */
    return IRQ_WAKE_THREAD;                    /* defer the real work */
}

/* BOTTOM HALF: runs in a kernel thread; may sleep, take mutexes. */
static irqreturn_t sensor_thread(int irq, void *dev_id)
{
    struct sensor *s = dev_id;
    parse_and_queue(s);                        /* heavy work, safely */
    wake_up_interruptible(&s->readq);          /* unblock read() */
    return IRQ_HANDLED;
}

request_threaded_irq(s->irq, sensor_hardirq, sensor_thread,
                     IRQF_SHARED, "sensor", s);

4/proc/interrupts and IRQ affinity

① Intuition where interrupts land

Every interrupt is delivered to a CPU core. On a multi-core SoC like the Orin (12 Cortex-A78AE cores) that choice matters enormously: if a high-rate NIC and your control loop both land on core 0, the NIC's interrupt storm steals cycles from the loop. The cure is SMP affinity — pinning an IRQ to a core (or set of cores) you've reserved for it.

Start by reading the live tally. /proc/interrupts has one row per IRQ, one column per CPU, plus the controller and device name:

shell bash
# one row per IRQ; columns are per-CPU counts
cat /proc/interrupts
#            CPU0     CPU1     CPU2     CPU3
#  56:    1048576        0        0        0   GICv3  72 Edge    eth0
#  58:        412        0        0        0   GICv3  98 Level   nvme0q0
#  61:       2048        0       17        0   GICv3 104 Edge    can0

# watch the rate live (delta per second)
watch -n1 'grep -E "eth0|nvme" /proc/interrupts'

# which core is IRQ 56 allowed on? (bitmask, hex)
cat /proc/irq/56/smp_affinity        # e.g. fff = any of 12 cores
cat /proc/irq/56/smp_affinity_list   # human form, e.g. 0-11
③ Code pin the IRQ

To move IRQ 56 (eth0) onto core 3 alone, write the bitmask. Core 3 is bit 3, so the mask is 8. Use smp_affinity_list if you'd rather name cores. Note: on most systems irqbalance runs as a daemon and will happily undo your pin — stop or mask it first for a deterministic setup.

shell bash
sudo systemctl stop irqbalance            # stop it fighting you

echo 8     | sudo tee /proc/irq/56/smp_affinity        # core 3 (bit 3)
echo 3     | sudo tee /proc/irq/56/smp_affinity_list   # same, by name

# pair it with isolating the control-loop core from the scheduler:
#   kernel cmdline: isolcpus=2 nohz_full=2 rcu_nocbs=2
# then run the 20 Hz loop pinned to core 2:
taskset -c 2 ./control_loop
🛠️ The pattern that works

Sensor IRQs on a couple of "housekeeping" cores; the control loop alone on an isolcpus core with affinity locked to it. Now a LiDAR burst can hammer its core all it wants — the loop's core never sees the interrupt.

5DMA: descriptors, coherent vs streaming, and the cache trap

① Intuition the courier needs an address

DMA hands the data path to an engine, but that engine speaks physical (or IOMMU-mapped bus) addresses, not the virtual addresses your driver uses. So the driver must (1) allocate a buffer, (2) get its DMA address, (3) program that address plus a length into the engine via MMIO, (4) start it, and (5) handle one completion IRQ. For streams the buffer is usually a ring of descriptors so the engine keeps filling slots while you drain earlier ones.

② Math the whole point

Recall §1's PIO cost of $R/w$ accesses per second. DMA collapses the CPU's per-stream interrupt cost to roughly one IRQ per buffer, not per word. If the engine fills a buffer of $B$ bytes before interrupting, the interrupt rate drops to

$$ f_{\text{IRQ}}^{\text{DMA}} = \frac{R}{B} \quad\ll\quad f_{\text{IRQ}}^{\text{PIO/word}} = \frac{R}{w} $$

For $R=12.5$ MB/s and a $B=64$ KiB buffer that's $\approx 190$ interrupts/s instead of millions of CPU copies. The sim below makes this concrete.

① Intuition coherent vs streaming

There are two ways to set up the buffer, and the difference is all about the cache:

MappingAPICacheUse for
Coherent (consistent)dma_alloc_coherent()kept coherent by HW/uncached; no manual synclong-lived rings, descriptor tables
Streamingdma_map_single() / dma_map_sg()cached; you sync before/afterone-shot transfers of existing buffers
⚠️ Pitfall — the cache coherency trap (very real on ARM64)

With a streaming map, the CPU's cache and the DMA engine see memory differently. Before the device reads your buffer you must dma_sync_single_for_device() (flush dirty cache lines out); after the device writes, you must dma_sync_single_for_cpu() (invalidate stale lines) before reading. Skip the sync and you read stale cached data while fresh bytes sit in RAM — an intermittent, alignment-dependent corruption that x86's snooping hardware often hides but the Orin will not. This bug eats weeks. An IOMMU (SMMU on Tegra) sits in the path and remaps/bounds the engine's addresses, but it does not relieve you of cache sync.

6Coalescing, NAPI, and the latency–throughput knob

① Intuition even one IRQ per packet is too many

DMA gives you one IRQ per buffer — but a busy NIC receiving 100k packets/s with one DMA-filled descriptor each would still fire 100k interrupts/s. Each interrupt costs context save/restore, cache pollution, and a trip through the handler. Two mechanisms tame this:

  • Interrupt coalescing: the NIC waits until it has accumulated N packets or a timeout T elapses, then fires one IRQ for the batch. Fewer interrupts, lower CPU — at the cost of up to $T$ extra latency. This is a direct throughput-vs-latency dial, set via ethtool -C.
  • NAPI: when interrupts start storming, the driver disables the RX interrupt and switches to polling the DMA ring in a softirq, draining many packets per poll. When the ring drains, it re-enables the interrupt. Best of both: interrupt-driven when idle (low latency), poll-driven under load (no storm).
③ Code tune coalescing
shell bash
# see current coalescing settings for the LiDAR NIC
ethtool -c eth0
#   rx-usecs: 50        (wait up to 50 µs to batch RX IRQs)
#   rx-frames: 64       (or until 64 frames arrive)

# favor THROUGHPUT (batch harder; CPU drops, latency rises):
sudo ethtool -C eth0 rx-usecs 100 rx-frames 128

# favor LATENCY (fire sooner; more IRQs, more CPU):
sudo ethtool -C eth0 rx-usecs 10  rx-frames 8

# count how hard NAPI is working / drops under load:
ethtool -S eth0 | grep -E "rx_packets|rx_dropped|napi"
∑ The CPU budget of an interrupt

If each interrupt (entry + handler + exit) costs $c$ seconds of CPU and you take $f_{\text{IRQ}}$ of them per second, the steady-state CPU fraction is $\;\text{CPU} = f_{\text{IRQ}}\cdot c$. With $c \approx 3\,\mu\text{s}$ and 100k IRQ/s that's $0.30 = 30\%$ of a core gone to overhead alone. Coalesce to 5k IRQ/s and it's $1.5\%$. That recovered core is your control loop's headroom.

8Real case: high-rate LiDAR on the AGX Orin USV

④ Real case this is the whole point

Our USV's spinning LiDAR streams point clouds over Gigabit Ethernet — roughly $R \approx 100$ Mb/s of UDP, tens of thousands of packets per second at full spin. The Jetson AGX Orin also runs the 20 Hz heading-and-thrust control loop that absolutely cannot stutter (every chapter of the PID course depends on that $dt=50$ ms being honest).

If the NIC interrupted per packet on the control-loop core, the loop would miss deadlines under LiDAR load. The fix stacks every idea in this chapter:

  • DMA: the Orin's Ethernet controller DMAs frames straight into a ring in RAM. The CPU never copies a packet body.
  • Coalescing + NAPI: under the LiDAR storm the driver polls the ring in softirq, collapsing ~30k+ IRQ/s into a few thousand batched wakeups.
  • Affinity + isolation: eth0's IRQ and its NAPI softirq are pinned to a housekeeping core; the control loop owns an isolcpus core by itself.
🏭 Real case — the numbers that let it ship

Per-packet IRQs at 30k/s with $c\approx3\,\mu$s would burn $\sim$9% of a core just on interrupt overhead, spiking the loop core's latency. After DMA + coalescing to $\sim$3k batched IRQ/s on a separate core, the control loop's core shows near-zero interrupt activity in /proc/interrupts and holds its 50 ms period with microsecond jitter. Same LiDAR, same data — the difference is entirely who moves the bytes and where the interrupts land.

9Your turn

Q1 Conceptual check
A sensor streams 12.5 MB/s continuously. Why is DMA the right choice over interrupt-per-word PIO for the data path?
  • DMA is faster per byte than the CPU's memory copy.
  • DMA moves the payload with a dedicated engine, so the CPU spends ~1 interrupt per buffer instead of millions of register reads per second — freeing it for the control loop.
  • DMA avoids using RAM, so there is no memory pressure.
  • DMA does not require the volatile keyword.
The win isn't raw copy speed — it's offload. PIO makes the CPU the data path ($R/w \approx 3.1$M accesses/s here); DMA reduces the CPU's involvement to programming descriptors and fielding one completion IRQ per buffer. The recovered CPU cycles are what keep the 20 Hz loop on time.
Q2 Compute it
A NIC receives 30,000 packets/s and fires one interrupt per packet. Each interrupt costs $c = 3\,\mu\text{s}$ of CPU (entry + handler + exit). What percentage of a single core does interrupt handling consume? (Hint: $\text{CPU} = f_{\text{IRQ}}\cdot c$, then ×100.)
% of a core
$\text{CPU} = 30000 \times 3\times10^{-6}\,\text{s} = 0.09 = 9.0\%$ of one core — just on interrupt overhead, before any useful work. Coalescing to 3,000 batched IRQ/s drops it to $0.9\%$. That recovered headroom is exactly why we coalesce and pin affinity off the control-loop core.
Q3 Hands-on — find an IRQ and pin it
On the Orin, find the IRQ number for the Ethernet interface eth0 from /proc/interrupts, then pin it to core 3 alone and confirm. Account for the daemon that would otherwise undo your change.
Hint
The IRQ number is the leftmost column of the row whose name (rightmost field) is eth0. SMP affinity lives in /proc/irq/<N>/smp_affinity as a CPU bitmask in hex — core 3 is bit 3. And remember irqbalance.
Show one solution
shell bash
# 1) find the IRQ number for eth0 (e.g. it prints "56")
grep eth0 /proc/interrupts | awk -F: '{print $1}' | tr -d ' '

# 2) stop the balancer so the pin sticks
sudo systemctl stop irqbalance

# 3) pin IRQ 56 to core 3 only (bit 3 => mask 0x8)
echo 8 | sudo tee /proc/irq/56/smp_affinity
#   or the readable form:
echo 3 | sudo tee /proc/irq/56/smp_affinity_list

# 4) confirm, then verify counts now climb only under CPU3
cat /proc/irq/56/smp_affinity_list      # => 3
watch -n1 'grep eth0 /proc/interrupts'

The interrupt count should now increment only in the CPU3 column. Keep this core distinct from the one running your isolcpus-pinned control loop, and the LiDAR storm can never steal the loop's cycles.

🔑 Chapter recap

Three ways to move bytes: PIO (CPU copies every word, $R/w$ accesses/s — fatal for streams), interrupts (a doorbell; ISR top half stays tiny, real work goes to a bottom half or threaded IRQ), and DMA (an engine moves the payload, the CPU gets ~one IRQ per buffer). Devices appear as memory via MMIO, which is why register access needs volatile plus real barriers. /proc/interrupts shows where IRQs land; /proc/irq/N/smp_affinity pins them off your control-loop core. DMA streaming maps demand cache sync on ARM64 or you read stale data. Coalescing and NAPI trade a little latency for a lot of CPU headroom — exactly what keeps the Orin's 20 Hz loop on time under a LiDAR flood.
