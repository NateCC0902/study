Networking I/O: from the NIC to your socket

A network card is just another device that DMAs bytes into RAM and raises interrupts — but on top of it sits a whole stack: drivers, softirqs, the protocol layers, socket buffers. And the moment you bolt a second interface onto your robot, a quiet DHCP server can steal your default route and silently kill the link you depend on. We trace one packet end to end, then make two NICs coexist on a USV without stepping on each other.

1A packet's journey: NIC to recv()

① Intuition follow one frame

Everything you learned in Chapter 07 about interrupts and DMA applies directly here — a NIC is exactly that kind of device. When a frame arrives on the wire, the card doesn't hand it to the CPU byte by byte. It DMAs the whole frame into a ring of pre-allocated kernel buffers in RAM, then raises an interrupt to say "there's new mail." From that point the kernel walks the frame up the protocol layers until the payload lands in a per-socket buffer, and your blocked recv() finally returns.

💡 The whole stack in one sentence

The NIC DMAs the frame into RAM and interrupts; a softirq drains the receive ring (NAPI), the IP/UDP/TCP layers strip headers and demultiplex by port, the payload is queued on the matching socket's receive buffer, and the kernel wakes whoever is sleeping in recv(). Hardware to userspace, with no copies you can avoid except the last one.

The single most important design choice in that path is that the hardware interrupt does almost nothing. A 10 Gb/s link can deliver well over a million frames per second; one interrupt per frame would melt the CPU. So the driver uses NAPI: the hard IRQ just schedules a softirq (NET_RX_SOFTIRQ) and masks further RX interrupts, then the softirq polls the ring in a batch until it drains. This is the interrupt-vs-polling trade-off from Chapter 07, applied where it matters most. You can watch the softirq budget being spent in /proc/net/softnet_stat — the third column is the count of times the budget ran out before the ring emptied, which is your early warning of an overloaded receive path.

2Sockets ARE file descriptors

① Intuition the same VFS trick

Chapter 01 told you everything is a file. A socket is the cleanest proof: socket() returns an int file descriptor that lives in the exact same fd table as your open files and serial ports. You can read()/write() it, close() it, set it non-blocking with fcntl(fd, F_SETFL, O_NONBLOCK), and — crucially — hand it to epoll alongside your GPS serial fd and your CAN socket (Chapter 08). One event loop, every I/O source, because they're all just descriptors.

The socket-specific calls are a small, learnable set. socket(domain, type, proto) creates the endpoint; bind() nails it to a local address/port; connect() sets the peer (mandatory for TCP, optional-but-useful for UDP); send()/recv() (or sendto()/recvfrom() for connectionless) move bytes. That's the whole API surface you need for sensor I/O.

② Math two type flavors

The type argument picks the delivery semantics: $\texttt{SOCK\_STREAM}$ (TCP) gives you a reliable, ordered, byte-stream with no message boundaries — one send() of 100 bytes may arrive as two recv()s, so you must frame messages yourself. $\texttt{SOCK\_DGRAM}$ (UDP) gives you whole datagrams: each recvfrom() returns exactly one packet, or nothing. For sensors that publish self-contained samples, datagram boundaries are a feature, not a bug.

③ Code a minimal UDP receiver
udp_recv.c C
#include <arpa/inet.h>
#include <string.h>
#include <stdio.h>
#include <unistd.h>

int main(void) {
    int fd = socket(AF_INET, SOCK_DGRAM, 0);   /* fd just like any file */
    if (fd < 0) { perror("socket"); return 1; }

    struct sockaddr_in addr = {0};
    addr.sin_family = AF_INET;
    addr.sin_port   = htons(2368);             /* listen port, network byte order */
    addr.sin_addr.s_addr = htonl(INADDR_ANY);  /* any local interface */

    if (bind(fd, (struct sockaddr *)&addr, sizeof addr) < 0) {
        perror("bind"); return 1;
    }

    char buf[2048];
    struct sockaddr_in src; socklen_t slen = sizeof src;
    for (;;) {
        ssize_t n = recvfrom(fd, buf, sizeof buf, 0,
                             (struct sockaddr *)&src, &slen);
        if (n < 0) { perror("recvfrom"); break; }
        printf("got %zd bytes from %s\n", n, inet_ntoa(src.sin_addr));
    }
    close(fd);
    return 0;
}
⚠️ Pitfall — byte order & partial TCP reads

Ports and addresses go on the wire big-endian; forget htons() and you'll bind to a port nobody is sending to. And never assume a TCP recv() returns a whole message — it returns "whatever's buffered now." Loop until you've read your declared length, exactly like the robust read loop from Chapter 01.

3Interfaces, addresses & the routing table

① Intuition which door does a packet leave by

When you send() to an address, the kernel has to decide which interface the packet leaves through. That decision is the routing table: an ordered set of rules matching destination prefixes to an outgoing interface and next hop. The ip tool (from iproute2 — forget ifconfig) is how you read and write it.

terminal bash
# interfaces and their addresses
ip -brief addr        # eth0  UP  192.168.1.50/24
                      # wlan0 UP  10.42.0.7/24

# the routing table, longest-prefix wins within a metric
ip route
# default via 192.168.1.1 dev eth0  proto dhcp  metric 100
# default via 10.42.0.1   dev wlan0 proto dhcp  metric 600
# 192.168.1.0/24 dev eth0  proto kernel scope link src 192.168.1.50
# 10.42.0.0/24   dev wlan0 proto kernel scope link src 10.42.0.7

# ask the kernel exactly how it would route a specific destination:
ip route get 8.8.8.8
# 8.8.8.8 via 192.168.1.1 dev eth0 src 192.168.1.50

Two rules govern every lookup. First, longest-prefix match: a packet to 192.168.1.99 matches the 192.168.1.0/24 route (24 bits) over the default route (0 bits), so it goes out eth0 directly. Second, the default route (0.0.0.0/0, prefix length 0) is the catch-all for everything not on a directly-connected subnet — it's where "the internet" and any host you don't have a specific route for goes.

② Math metrics break ties

When two routes have the same prefix length, the metric (lower = preferred) breaks the tie. With two default routes, the kernel uses the one with the smaller metric for all non-local traffic. Formally, the chosen route for destination $d$ is

$$ \text{route}(d) = \operatorname*{arg\,min}_{r\,:\,d\in\text{prefix}(r)}\;\bigl(-\text{plen}(r),\; \text{metric}(r)\bigr) $$

i.e. maximize prefix length first, then minimize metric. The negative sign just says "longer prefix wins"; among equal-length matches, the smaller metric wins. This formula is the entire reason the next section is a hazard.

4Default-route theft

① Intuition the silent killer

Here is the failure that costs robotics teams hours. Your USV runs fine on Ethernet with default via 192.168.1.1 dev eth0 metric 100. You plug in WiFi — or it auto-reconnects — and its DHCP lease arrives carrying a default gateway and a metric. If that metric is lower than your Ethernet's, the WiFi default route wins, and suddenly all non-local traffic leaves by wlan0. Your SSH session over Ethernet may survive (it's an established connection to a directly-connected subnet), but any new connection to a host reached via the default route now goes out the wrong door — or nowhere, if WiFi can't actually reach it.

⚠️ Pitfall — DHCP installs whatever it's told

A DHCP client doesn't know your intent. If the WiFi AP's DHCP server hands out a default gateway and your client installs it at a low metric (NetworkManager's default for WiFi is often 600, but a misconfigured profile or a wired-style metric can be lower), it can outrank your Ethernet's 100 — or, worse, both land at the same metric and routing becomes nondeterministic between boots.

The fix is to make the route order deterministic and yours. Pin metrics so the link you trust always wins, or strip the default route from the link that shouldn't carry general traffic. On the Orin we want Ethernet (carrying the LiDAR and the route to the rest of the boat's network) to own a low metric, and WiFi (just SSH from the shore laptop) to either have a high metric or no default route at all.

netplan: deterministic metrics bash
# /etc/netplan/01-usv.yaml  — Ethernet wins the default route
network:
  version: 2
  ethernets:
    eth0:
      dhcp4: true
      dhcp4-overrides:
        route-metric: 100          # low metric → preferred default
  wifis:
    wlan0:
      dhcp4: true
      dhcp4-overrides:
        route-metric: 700          # high metric → never the default if eth0 is up
        use-routes: false          # or: ignore WiFi-supplied routes entirely
      access-points:
        "usv-shore": {}

# apply and verify the order is what you intended:
sudo netplan apply
ip route | grep default
# default via 192.168.1.1 dev eth0  metric 100   <- wins
# default via 10.42.0.1   dev wlan0 metric 700
🛠️ Debug it like a senior

Don't guess — ask the kernel. ip route get <dest> tells you the exact interface and source address a packet to that destination will use, right now. If it says dev wlan0 when you expected eth0, you've found your theft. ip monitor route will even print routing-table changes live, so you can catch the moment DHCP rewrites it.

5Two-NIC coexistence on a USV

④ Real case roles, not just links

The clean architecture assigns each interface a role and keeps their traffic on separate physical controllers. On the boat: Ethernet is the heavy, deterministic sensor link (LiDAR, plus the rest of the wired network); WiFi is the human control/telemetry link (SSH, a low-rate status feed). They sit on different subnets, so each subnet route is directly-connected and unambiguous — only the default route is contested, which is exactly the problem §4 solved.

🔑 The design rule

Separate subnets remove ambiguity on direct traffic; deterministic metrics remove ambiguity on the default route; and putting heavy data on the wire keeps the lossy, jittery medium (WiFi) carrying only what can tolerate it. Three independent decisions, each closing a class of failure.

One subtlety: with two interfaces, a packet sourced from eth0's address but routed out wlan0 (or vice versa) can be silently dropped by reverse-path filtering (rp_filter in /proc/sys/net/ipv4/conf/*/rp_filter). If you deliberately need asymmetric routing, you must loosen it (mode 2) or use policy routing (ip rule) to keep each interface's replies on its own link. For our role-separated design we don't need asymmetry, which is one more reason to prefer it.

6WiFi: a shared half-duplex medium

① Intuition why WiFi jitters and Ethernet doesn't

Switched Ethernet is full-duplex: each port has a dedicated pair, TX and RX run simultaneously, no one else is on your wire. WiFi is the opposite — every station on a channel shares one half-duplex medium and must take turns. The arbitration is CSMA/CA: listen, and if the air is busy, back off a random interval and try again. Add a second talker and the air gets contended; latency stops being a constant and becomes a distribution with a long tail. That tail is jitter, and a control loop hates jitter far more than it hates a slightly higher mean.

② Math where the latency goes

On a contended channel, expected access delay rises sharply as offered load $\rho$ approaches the medium's effective capacity. A simple queueing intuition (treat the channel as one server) gives mean latency that blows up as utilization nears 1:

$$ \mathbb{E}[T] \;\approx\; T_{\text{tx}} + \frac{T_{\text{tx}}\,\rho}{1-\rho}, \qquad \rho = \frac{\lambda}{\mu} $$

where $\lambda$ is arrival rate, $\mu$ the service rate, and $T_{\text{tx}}$ the bare transmit time. Two effects compound on WiFi: $\mu$ itself shrinks as more stations contend (airtime lost to backoff and collisions), and the variance — not just the mean — grows. Watch it live in the sim below.

📡 Latency vs load — Ethernet (full-duplex) vs WiFi (shared)
🔑 What to notice

The green Ethernet curve stays flat and low almost all the way to line rate — a dedicated full-duplex link barely cares about offered load until it's nearly saturated. The orange WiFi curve sits higher even when idle and climbs steeply well before "100%," and adding stations drags its goodput cap (the knee) leftward. That's the whole argument for keeping the 50 Mb/s LiDAR on Ethernet: on WiFi it would push you onto the steep part of the curve, where latency and jitter explode. WiFi gets SSH; the wire gets the data.

7Socket buffers, backpressure & the BDP

① Intuition buffers are a trade, not a free win

Between the wire and your recv() sits a per-socket receive buffer (kernel limit sk_rcvbuf, tunable via SO_RCVBUF). It absorbs bursts so a momentary scheduling hiccup in your reader doesn't drop packets. But a buffer that's too big is its own disease: bufferbloat. Oversized queues hold stale packets, so by the time a datagram reaches you it may describe a world that's seconds out of date — useless for control. The right size is "big enough for your worst burst, no bigger."

② Math bandwidth-delay product

The buffer a throughput-bound (TCP) flow needs to keep the pipe full is the bandwidth-delay product — how many bytes are "in flight" on the link at once:

$$ \text{BDP} = \text{bandwidth} \times \text{RTT} $$

For a 1 Gb/s link with a 1 ms round trip: $\text{BDP} = 10^{9}\,\text{bit/s} \times 10^{-3}\,\text{s} = 10^{6}$ bits $= 125\,000$ bytes $\approx 122$ KiB. A TCP window smaller than the BDP throttles throughput; much larger just adds latency. For a latency-bound sensor stream the logic flips — you want the buffer just large enough to ride out your reader's worst stall, by Little's law $L = \lambda W$: at $\lambda = 5\,000$ packets/s, tolerating a $W = 2$ ms reader stall means queuing up to $L = 10$ packets, so size the buffer for that, not for a fat pipe.

tune SO_RCVBUF C
int rcvbuf = 4 * 1024 * 1024;   /* 4 MiB request */
setsockopt(fd, SOL_SOCKET, SO_RCVBUF, &rcvbuf, sizeof rcvbuf);
/* the kernel doubles your request (bookkeeping) and caps it at
   net.core.rmem_max — raise that sysctl if you need more:
   sysctl -w net.core.rmem_max=8388608  */

socklen_t len = sizeof rcvbuf;
getsockopt(fd, SOL_SOCKET, SO_RCVBUF, &rcvbuf, &len);
/* read it back: this is what you actually got, not what you asked for */
⚠️ Pitfall — UDP has no backpressure

TCP applies backpressure: if your reader is slow, the receive window shrinks and the sender is told to slow down. UDP does not. If the socket buffer fills, the kernel just drops packets and bumps the RcvbufErrors counter in /proc/net/snmp. For a high-rate LiDAR that means silently missing scans. Size SO_RCVBUF for your burst, read fast (epoll, Ch 08), and monitor that counter.

8UDP vs TCP for sensor streams

① Intuition pick the right failure mode

The choice isn't "reliable vs unreliable" — it's "which failure do you prefer." TCP guarantees delivery and order by retransmitting lost segments, which means a single dropped packet stalls everything behind it (head-of-line blocking) until the retransmit arrives. For a sensor publishing fresh state at high rate, a retransmitted old sample is worthless; you'd rather skip it and use the next fresh one. That's UDP's model: lose a packet, move on, no stall.

🏭 Real case — the Orin's split

On the USV: the LiDAR streams UDP — a missed scan is irrelevant next cycle, and we never want a retransmit to stall the freshest frame. Mission waypoints, parameter updates, and the SSH session ride TCP — those must arrive, exactly and in order, and the data volume is tiny so retransmit latency is a non-issue. Same socket API, opposite guarantees, each matched to its job.

9Your turn

Q1 Conceptual check
Your Orin has default via 192.168.1.1 dev eth0 metric 100 working fine. WiFi reconnects and DHCP installs default via 10.42.0.1 dev wlan0 metric 50. New connections to hosts reached via the default route suddenly fail. What happened?
  • The Ethernet route was deleted by the kernel because two defaults can't coexist.
  • Both are 0.0.0.0/0, so the lower metric (50, wlan0) wins the tie and now carries all non-local traffic — out an interface that can't reach those hosts.
  • WiFi has a longer prefix than Ethernet, so longest-prefix match sends traffic to wlan0.
  • The receive buffer overflowed and the kernel rerouted traffic to recover.
Both default routes have prefix length 0, so longest-prefix match is a tie; the metric breaks it and lower wins. WiFi's 50 beats Ethernet's 100, stealing the default route. Fix it by pinning Ethernet to a lower metric than WiFi (or stripping WiFi's default with use-routes: false). Diagnose with ip route get <dest>.

Q3 Coding challenge — read the route, then a non-blocking UDP socket
First, give the one command that asks the kernel exactly which interface a packet to 8.8.8.8 will use. Then write the C to create a UDP socket, set it O_NONBLOCK, bind() to a port, and drain it in a loop that returns cleanly on EAGAIN (so it can live inside an epoll loop, Ch 08).
Hint
ip route get takes a destination. For non-blocking, fetch flags with fcntl(fd, F_GETFL), OR in O_NONBLOCK, set them back. A non-blocking recvfrom() with nothing to read returns -1 with errno == EAGAIN (a.k.a. EWOULDBLOCK) — that's "drained," not an error.

Read the route the kernel would actually pick:

terminal bash
ip route get 8.8.8.8
# 8.8.8.8 via 192.168.1.1 dev eth0 src 192.168.1.50
nb_udp.c C
#include <arpa/inet.h>
#include <fcntl.h>
#include <errno.h>
#include <string.h>
#include <stdio.h>
#include <unistd.h>

int main(void) {
    int fd = socket(AF_INET, SOCK_DGRAM, 0);
    if (fd < 0) { perror("socket"); return 1; }

    int fl = fcntl(fd, F_GETFL, 0);            /* set non-blocking */
    fcntl(fd, F_SETFL, fl | O_NONBLOCK);

    struct sockaddr_in a = {0};
    a.sin_family = AF_INET;
    a.sin_port   = htons(2368);
    a.sin_addr.s_addr = htonl(INADDR_ANY);
    if (bind(fd, (struct sockaddr *)&a, sizeof a) < 0) {
        perror("bind"); return 1;
    }

    /* in a real app this runs when epoll says fd is readable */
    char buf[2048];
    for (;;) {
        ssize_t n = recvfrom(fd, buf, sizeof buf, 0, NULL, NULL);
        if (n >= 0) {
            /* process the datagram of n bytes here */
            continue;
        }
        if (errno == EAGAIN || errno == EWOULDBLOCK) break;  /* drained */
        if (errno == EINTR) continue;                        /* retry */
        perror("recvfrom"); break;                           /* real error */
    }
    close(fd);
    return 0;
}

The loop reads every queued datagram, then exits on EAGAIN when the socket buffer is empty — exactly the edge-triggered drain pattern epoll wants. Handle EINTR (interrupted syscall) by retrying, just like the robust read loops from Chapter 01.

🔑 Chapter recap

A packet's path is NIC DMA → hard IRQ → NAPI softirq → IP/UDP/TCP demux → socket buffer → recv(). Sockets are file descriptors, so they live in the same fd table and epoll loop as everything else. The routing table picks the exit interface by longest-prefix match, then metric — which is why a second NIC's low-metric DHCP default route can steal your traffic; pin metrics deterministically and put heavy data on full-duplex Ethernet, leaving jittery shared-medium WiFi for SSH. Size SO_RCVBUF for your worst burst (not bigger — bufferbloat), use the BDP for TCP throughput, and pick UDP for fresh-sample sensor streams where a retransmit would only deliver stale data.
