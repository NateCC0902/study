Real-Time & Robust I/O: never miss a deadline, never trust a wire

Your 20 Hz heading controller has exactly 50 ms to read sensors, compute, and command the rudder — every cycle, forever, including the cycle where WiFi drops, the SD card stalls, and a page fault lands. This chapter is about making the loop that holds the boat impossible to derail: meet the worst-case deadline, and survive every I/O that goes wrong.

1What "real-time" actually means

① Intuition build the picture first

Real-time does not mean fast. It means on time. A system is real-time if it is guaranteed to finish each job before a known deadline — and the guarantee is about the worst case, not the average. A controller that averages 3 ms per cycle but spikes to 80 ms once a minute is, for a 50 ms loop, a broken controller. The boat doesn't care that 999 cycles were great; on the 1000th the rudder went stale and it broached.

💡 The one sentence to keep

Average latency is a benchmark number. Worst-case latency is a safety property. Real-time engineering is the discipline of shrinking and bounding the tail, not the mean.

There are three tiers. Hard real-time: a missed deadline is a system failure (an airbag, a motor commutation ISR). Firm real-time: a late result is useless and you'd rather drop it than use it (a stale sensor sample). Soft real-time: late degrades quality but is tolerable (video playback). A USV heading loop is firm-to-hard: a late rudder command is worse than no command, and enough of them in a row capsizes the mission.

② Math deadline, period, jitter

Run a periodic task at frequency $f$. Its period is $T = 1/f$ and your implicit deadline is "finish this cycle before the next one starts," i.e. one period. For the 20 Hz loop:

$$ T = \frac{1}{f} = \frac{1}{20\ \text{Hz}} = 50\ \text{ms} \quad=\quad \text{deadline per cycle} $$

Two numbers describe how well you hit it. Latency $L$ is the delay from the scheduled wake-up to when your code actually runs. Jitter $J$ is the spread of that latency across cycles — what you measure as max minus min (or a high percentile minus the median):

$$ J = L_{\max} - L_{\min}, \qquad L_{\max} \le T - C_{\text{worst}} $$

where $C_{\text{worst}}$ is your worst-case compute time. That second inequality is the whole game: scheduling latency plus worst-case compute must fit inside one period. If your wake-up can be late by up to $L_{\max}$ and your compute can take up to $C_{\text{worst}}$, you only stay inside the deadline when $L_{\max} + C_{\text{worst}} \le T$.

🏭 Real case — measuring the AGX Orin's tail

Before trusting the Jetson AGX Orin with a control loop, measure it. cyclictest (from rt-tests) wakes a thread every period and reports actual vs. expected wake-up. On a stock kernel you might see Avg: 8us Max: 4200us — a 4.2 ms tail from a 8 µs average. That 4.2 ms is what eats your budget, and it's invisible to any benchmark that reports means.

2Where the latency tail comes from

① Intuition who steals your CPU

Your loop is late because, between the timer firing and your code running, the kernel did something else with the CPU. There are four big thieves, and you fix each one differently.

SourceWhat happensTypical hitFix (later sections)
SchedulerA lower-priority task is running and won't yield until a preemption point1–100+ ms on non-RTPREEMPT_RT + SCHED_FIFO
IRQ / softirqA device interrupt (NIC, USB) runs its handler on your CPU10 µs – several msIRQ affinity, threaded IRQs
Page faultsYour code/stack/heap page isn't resident; kernel must fault it in10 µs – ms (worse if swapped)mlockall()
CPU power statesCore was in a deep C-state or low P-state; must wake/ramp up10s of µs of exit latencydisable deep idle, pin frequency
⚠️ Pitfall — the first cycle is a lie

The page-fault thief loves the first iteration. Memory you malloc() isn't physically backed until you touch it (demand paging). Your warm-up cycle faults in every page and looks fine; the cycle that faults a new log buffer at minute 30 blows the deadline. This is exactly what mlockall() exists to kill.

② Math IRQ storms eat the budget

Interrupts are sneaky because their cost is a rate times a per-event cost. If a NIC under load fires interrupts at rate $\lambda$ and each handler steals $c$ of CPU time, the fraction of a core lost to it is:

$$ \text{IRQ load} = \lambda \cdot c $$

A gigabit NIC with no coalescing can hit $\lambda = 80{,}000$ IRQ/s; at $c = 3\ \mu\text{s}$ that's $0.24$ — a quarter of a core gone, and every one of those handlers is a chance to preempt your control loop. Coalescing (Ch 07) and pinning that IRQ to a different core (next section) both shrink the damage your loop sees.

3PREEMPT_RT, priorities & locking memory

① Intuition make the kernel yield

On a stock Linux kernel, a thread running in the kernel (a syscall, a driver) generally can't be preempted until it reaches a scheduling point. That's where multi-millisecond stalls hide. The PREEMPT_RT patch set (mainlined progressively, fully merged around the 6.12 kernel) makes nearly the entire kernel preemptible: spinlocks become sleeping mutexes, interrupt handlers run in threads you can prioritize, and a high-priority RT task can seize a CPU almost immediately.

💡 RT is a contract, not a speedup

PREEMPT_RT usually lowers throughput a little (all that preemptibility has overhead). What it buys you is a far smaller, more predictable tail. You trade peak speed for a bounded worst case — exactly the trade a control loop wants.

② Math priority & the RT throttle

Two scheduling classes matter. SCHED_FIFO and SCHED_RR are real-time classes with priorities 1–99; any runnable RT thread preempts all normal (SCHED_OTHER) tasks. Higher number = higher priority. A SCHED_FIFO thread runs until it blocks or a higher-priority RT thread wakes — it is not time-sliced against equal priorities.

That last fact is a foot-gun: a SCHED_FIFO thread spinning in a bug can lock a core forever. Linux's safety valve is the RT throttle, which by default caps RT classes at $950000\ \mu\text{s}$ of CPU per $1000000\ \mu\text{s}$ window:

$$ \text{RT budget} = \frac{950000}{1000000} = 95\% \ \text{of each core, per 1 s window} $$

For a real isolated control core you typically size your loop well under that and keep the throttle as a backstop.

③ Code read every line

The recipe for a control thread is always the same three moves: lock memory, set RT priority, then loop. Locking comes first because faulting after you're RT is exactly the stall you're trying to prevent.

rt_setup.c C
#include <sched.h>
#include <sys/mman.h>
#include <string.h>
#include <stdio.h>

int make_realtime(int prio) {
    /* 1. Pin ALL current + future pages to RAM: no demand paging,
          no swap, no fault stalls inside the loop. */
    if (mlockall(MCL_CURRENT | MCL_FUTURE) != 0) {
        perror("mlockall");        /* needs CAP_IPC_LOCK / root */
        return -1;
    }

    /* 2. Become a SCHED_FIFO thread at the given RT priority. */
    struct sched_param sp;
    memset(&sp, 0, sizeof(sp));
    sp.sched_priority = prio;      /* 1..99; control loop ~ 80 */
    if (sched_setscheduler(0, SCHED_FIFO, &sp) != 0) {
        perror("sched_setscheduler");
        return -1;
    }

    /* 3. Pre-fault the stack so the FIRST cycle isn't special either. */
    volatile char touch[512 * 1024];
    memset((void *)touch, 0, sizeof(touch));
    return 0;
}
🛠️ Don't run your whole process RT

Make only the control thread SCHED_FIFO. Logging, telemetry, and housekeeping stay SCHED_OTHER so they can never starve the loop and can soak up the slack. One RT thread, many normal threads — that is the standard robot architecture.

4Isolating CPUs & steering interrupts

① Intuition give the loop its own room

The AGX Orin has 12 Arm cores. The cleanest real-time setup is to evict everything off one core and run only the control loop there. Two levers: isolcpus keeps the normal scheduler from placing tasks on a core, and IRQ affinity keeps device interrupts from landing there. Together they turn one core into a quiet room where the loop is almost never interrupted.

② Code kernel cmdline + /proc

Isolation starts at boot via the kernel command line (on Jetson, edited in /boot/extlinux/extlinux.conf). Reserve core 11 for the loop:

kernel cmdline (extlinux.conf APPEND line) bash
# isolate core 11 from the scheduler, RCU callbacks, and the tick
isolcpus=11 nohz_full=11 rcu_nocbs=11
# then pin your control thread to it from code or the shell:
taskset -c 11 ./control_loop          # CPU affinity = core 11 only

Now steer interrupts away from core 11. Each IRQ has a CPU affinity mask in /proc/irq/<N>/smp_affinity_list. Point the noisy ones (NIC, USB) at the housekeeping cores 0–10:

irq_affinity.sh bash
# find the NIC's IRQ number(s)
grep eth0 /proc/interrupts        # column 1 is the IRQ number, e.g. 142

# keep that IRQ off the isolated core (allow cores 0-10)
echo 0-10 | sudo tee /proc/irq/142/smp_affinity_list

# verify nothing is targeting core 11
cat /proc/irq/142/smp_affinity_list
⚠️ Pitfall — irqbalance fights you

The irqbalance daemon will happily re-spread interrupts back onto your isolated core. On an RT robot, disable it (sudo systemctl disable --now irqbalance) and pin affinities yourself. Also note some interrupts (timer, per-CPU IPIs) are managed by the kernel and won't move — that's fine; they're cheap and predictable.

One more knob for network-heavy boats: RPS (Receive Packet Steering) spreads softirq packet processing across cores in software, so a single-queue NIC doesn't dump all its receive work onto the one core its IRQ lands on. Set its mask in /sys/class/net/<iface>/queues/rx-0/rps_cpus — and, again, exclude the isolated core.

5The architecture: hot loop vs. everything else

① Intuition two worlds, one boat

The decision that makes or breaks a real-time robot is what runs on the hot path. The control thread does the absolute minimum: read fresh sensor values, compute, command the actuator. Everything that can block — logging to flash, telemetry over WiFi, recording a rosbag — runs on other threads and is fed by lock-free or bounded queues. The hot loop never waits on disk and never waits on a socket.

🔑 What to notice

The ring between the hot loop and the workers is the firewall. The control thread only ever pushes (a bounded, non-blocking write); if a worker falls behind, the worst case is a dropped log sample — never a missed rudder command. Backpressure on telemetry must never reach the loop.

④ Real case why this matters

On the USV, the rosbag recorder and the WiFi telemetry both want to ship megabytes. Run either of them inline and a flash flush or a TCP retransmit will, sooner or later, blow the 50 ms budget. With the two-world architecture, the SD card can stall for 200 ms and the boat still steers — the log thread just catches up afterward from the ring.

6The robust I/O loop: handling every failure

① Intuition read() lies to you

Even with perfect timing, the I/O itself fights you. A serial read() can return fewer bytes than you asked for, return -1 with errno = EINTR because a signal arrived, or return EAGAIN on a non-blocking fd that has nothing ready. Treat any of these as a hard error and your loop dies on the first signal or partial frame. A robust loop expects all of them.

Symptomerrno / returnMeaningCorrect response
Interrupted-1, EINTRA signal hit mid-syscallRetry the call
Would block-1, EAGAIN/EWOULDBLOCKNon-blocking fd, nothing readyGo back to epoll/timeout
Partial0 < n < countOnly part of the message arrivedLoop, read the remainder
Peer gone0 (read) / EPIPE (write)EOF / connection closedReconnect / reopen
Device unplugged-1, EIO/ENXIOThe ttyUSB vanishedFailsafe + idempotent reopen
② Math timeout budget

Reads must be bounded: a blocking read on a dead sensor will hang the loop forever. Give the read a timeout that fits the period with margin. If the loop period is $T$ and worst-case compute is $C_{\text{worst}}$, the slack available for waiting on I/O is:

$$ t_{\text{timeout}} \le T - C_{\text{worst}} - L_{\max} $$

For $T = 50$ ms, $C_{\text{worst}} = 5$ ms, $L_{\max} = 4$ ms, you have $\le 41$ ms to wait — but you'd set the timeout well below that (say 20 ms) so a late or missing sample is detected and skipped with time to still command the actuator from the last good estimate.

③ Code the read loop that survives

This is the single most important function in a robust driver: read exactly count bytes, or fail cleanly on timeout — handling EINTR and partials along the way. Read it line by line; it is the answer to the exercise below.

robust_read.c C
#include <unistd.h>
#include <errno.h>
#include <poll.h>
#include <stdint.h>
#include <time.h>

/* Milliseconds remaining until `end`, clamped to >= 0. */
static int ms_left(const struct timespec *end) {
    struct timespec now;
    clock_gettime(CLOCK_MONOTONIC, &now);
    long ms = (end->tv_sec  - now.tv_sec)  * 1000
            + (end->tv_nsec - now.tv_nsec) / 1000000;
    return ms > 0 ? (int)ms : 0;
}

/* Read exactly n bytes into buf, or stop after deadline_ms total.
   Returns bytes read (== n on success, < n on timeout, -1 on hard error). */
ssize_t read_n(int fd, void *buf, size_t n, int deadline_ms) {
    uint8_t *p = (uint8_t *)buf;
    size_t got = 0;
    struct timespec end;                        /* absolute deadline */
    clock_gettime(CLOCK_MONOTONIC, &end);
    end.tv_sec  += deadline_ms / 1000;
    end.tv_nsec += (deadline_ms % 1000) * 1000000L;
    if (end.tv_nsec >= 1000000000L) { end.tv_sec++; end.tv_nsec -= 1000000000L; }
    while (got < n) {
        struct pollfd pfd = { .fd = fd, .events = POLLIN };
        int pr = poll(&pfd, 1, ms_left(&end)); /* wait only the time LEFT */
        if (pr < 0) {
            if (errno == EINTR) continue;       /* signal: retry (clock keeps ticking) */
            return -1;                          /* real poll error */
        }
        if (pr == 0) return (ssize_t)got;       /* TIMEOUT: partial/none */

        ssize_t r = read(fd, p + got, n - got);
        if (r < 0) {
            if (errno == EINTR) continue;        /* retry */
            if (errno == EAGAIN) continue;       /* spurious: poll again */
            return -1;                           /* EIO/ENXIO: device gone */
        }
        if (r == 0) return (ssize_t)got;         /* EOF: peer closed */
        got += (size_t)r;                        /* PARTIAL: keep going */
    }
    return (ssize_t)got;                          /* got == n */
}
🛠️ Writes need the same care

A write() can also be partial and can also return EINTR. The mirror loop advances an offset until all bytes are out, and on a stream socket you must handle EPIPE (peer gone) as a reconnect trigger, not a crash. Wrap both in helpers; never call bare read()/write() in production robot code.

④ Real case idempotent reopen

When the GPS ttyUSB0 browns out, the kernel may hand it back as ttyUSB1 — which is exactly why Ch 03 had you pin a /dev/gps udev symlink. The reconnect routine must be idempotent: close the old fd if open, re-open /dev/gps, re-apply termios, and rebuild the parser state — safe to call on every failure without leaking fds. Until reopen succeeds, the loop runs on dead reckoning, not on stale GPS.

7Durable logging without blocking the loop

① Intuition write() is fast, durable is slow

A normal write() to a file is fast because it only copies into the kernel's page cache; the actual flash write happens later, asynchronously. That's perfect for the hot path — but it means a crash or power loss can lose the last few seconds. fsync() forces the data all the way to stable storage and blocks until the device acknowledges. On an SD card that can be tens to hundreds of milliseconds. Never call it on the hot path.

PathDurabilityLatencyUse on hot loop?
write() → page cacheLost on power-cut~µsYes (via worker)
write() + fsync()Durable10–200 ms (SD)Never
O_DIRECT + O_SYNCDurable, bypass cacheHigh, alignment-boundNever; for big sequential logs
② Math SD-card write amplification & wear

SD/eMMC flash erases in large blocks but writes in smaller pages. Updating one byte can force the controller to read-erase-rewrite a whole erase block — write amplification. Define it as:

$$ \text{WA} = \frac{\text{bytes physically written to flash}}{\text{bytes your app wrote}} $$

Tiny fsync'd records give terrible WA: a 64 B telemetry record that triggers a 512 KB erase block rewrite is $\text{WA} = 8192$. Flash cells endure a finite number of program/erase cycles, so endurance in bytes is roughly:

$$ \text{Bytes endurance} \approx \frac{\text{capacity} \times P\!/\!E\ \text{cycles}}{\text{WA}} $$

The fix is to make every write large, aligned, and sequential: batch records in a RAM buffer and flush in erase-block-sized chunks (e.g. 512 KB or 1 MB) from the log thread. Big sequential writes drive WA toward 1 and stretch card life by orders of magnitude.

③ Code batched flush off the hot path
log_writer.c C
#define FLUSH_BYTES (1u << 20)   /* 1 MiB: big, aligned, sequential */
static uint8_t batch[FLUSH_BYTES];
static size_t  fill = 0;

/* runs on the SCHED_OTHER log thread, draining the ring buffer */
void log_record(int fd, const void *rec, size_t len) {
    if (fill + len > FLUSH_BYTES) {        /* batch full -> one big write */
        size_t off = 0;
        while (off < fill) {                /* handle partial writes */
            ssize_t w = write(fd, batch + off, fill - off);
            if (w < 0) { if (errno == EINTR) continue; break; }
            off += (size_t)w;
        }
        fill = 0;
        /* fsync HERE, on the worker, never on the control thread.
           Do it on a coarse schedule (e.g. every few MiB), not per record. */
        fsync(fd);
    }
    memcpy(batch + fill, rec, len);
    fill += len;
}
🏭 Real case — the USV mission log

The Orin logs IMU + GPS + control state at 20 Hz. Per-record fsync to the SD card would both stall the loop and shred the card. Instead, samples go through the lock-free ring to a log thread that batches ~1 MiB and flushes sequentially, calling fsync only every few seconds. Result: the hot loop never touches flash, the card survives the mission, and on a power-cut you lose at most a couple of seconds of telemetry — an acceptable, bounded loss.

8Watchdogs & the failsafe state

① Intuition plan for your own death

Robust I/O assumes the software will hang. A hardware watchdog is a countdown timer in the SoC that resets the board if it isn't "petted" in time; your healthy control thread pets it every cycle. If the loop deadlocks, the watchdog reboots the Orin into a known state instead of letting the boat drive in circles. Linux exposes it as a character device: open /dev/watchdog, write to it (or ioctl with WDIOC_KEEPALIVE) periodically, and the kernel programs the timer.

⚠️ Pitfall — pet from the loop, not a timer

Pet the watchdog from inside the control thread's work, after a successful cycle — never from an independent timer thread. A timer that keeps petting while the control loop is wedged defeats the entire point: the board stays "alive" while the boat goes uncontrolled.

The second layer is the failsafe state: the safe thing to do when an input goes missing. For the USV, the canonical trigger is a WiFi drop (Ch 09's shared, jittery link). If no operator heartbeat arrives within a deadline, the boat must autonomously fall back — center the rudder, cut throttle, hold station or loiter — rather than continue the last command into a rock.

③ Code heartbeat-gated command
failsafe.c C
#define HEARTBEAT_TIMEOUT_NS  1000000000LL   /* 1 s */

/* called once per 50 ms control cycle, on the RT thread */
void control_step(struct ctx *c, int64_t now_ns) {
    int64_t age = now_ns - c->last_heartbeat_ns;

    if (age > HEARTBEAT_TIMEOUT_NS) {
        /* WiFi/operator lost: known-safe outputs, NOT the stale command */
        set_rudder(0.0);            /* center */
        set_throttle(0.0);          /* cut */
        c->state = FAILSAFE;
    } else {
        c->state = ACTIVE;
        run_pid(c);                 /* normal heading control */
    }
    pet_watchdog(c->wd_fd);         /* only reached if the loop is alive */
}

Q3 Coding challenge — a read loop that survives
Write a function that reads exactly n bytes from a serial fd, but gives up after a total timeout. It must correctly handle EINTR (retry), partial reads (keep going), and the timeout (return what you have). What is the bug in calling bare read(fd, buf, n) once?
Hint
Bare read() can (a) return -1/EINTR on a signal, (b) return fewer than n bytes even with data coming, and (c) block forever on a dead device. You need a bounded wait (poll) plus a loop that accumulates into an offset.
Show one solution

A single read() is wrong because it treats a partial return as the whole message, dies on the first EINTR, and has no timeout. The robust version polls with a deadline, retries on EINTR/EAGAIN, accumulates partials into an offset, and returns early on timeout or EOF:

robust_read.c C
/* ms remaining until the absolute deadline `end`, clamped to >= 0 */
static int ms_left(const struct timespec *end) {
    struct timespec now; clock_gettime(CLOCK_MONOTONIC, &now);
    long ms = (end->tv_sec - now.tv_sec) * 1000
            + (end->tv_nsec - now.tv_nsec) / 1000000;
    return ms > 0 ? (int)ms : 0;
}

ssize_t read_n(int fd, void *buf, size_t n, int deadline_ms) {
    uint8_t *p = (uint8_t *)buf;
    size_t got = 0;
    struct timespec end; clock_gettime(CLOCK_MONOTONIC, &end);
    end.tv_sec  += deadline_ms / 1000;
    end.tv_nsec += (deadline_ms % 1000) * 1000000L;
    if (end.tv_nsec >= 1000000000L) { end.tv_sec++; end.tv_nsec -= 1000000000L; }
    while (got < n) {
        struct pollfd pfd = { .fd = fd, .events = POLLIN };
        int pr = poll(&pfd, 1, ms_left(&end));   /* only the time LEFT, not the full budget */
        if (pr < 0) { if (errno == EINTR) continue; return -1; }
        if (pr == 0) return (ssize_t)got;          /* timeout */
        ssize_t r = read(fd, p + got, n - got);
        if (r < 0) {
            if (errno == EINTR || errno == EAGAIN) continue;
            return -1;                              /* device gone */
        }
        if (r == 0) return (ssize_t)got;            /* EOF */
        got += (size_t)r;                           /* partial: keep going */
    }
    return (ssize_t)got;                            /* got == n */
}

The same shape works for writes (advance an offset until all bytes are out, retry on EINTR). Never call bare read()/write() in robot code.

🔑 Chapter recap

Real-time means meeting the worst-case deadline, not a good average — engineer the tail. Latency comes from the scheduler, IRQs, page faults, and power states; tame them with PREEMPT_RT, SCHED_FIFO priority, mlockall(), isolcpus, and IRQ affinity. Keep the hot loop minimal: read non-blocking, compute, actuate, and push samples through a ring to SCHED_OTHER workers that log and telemeter. Make I/O robust by handling EINTR, partial reads/writes, EAGAIN, timeouts, and idempotent reopen. Log durably off the hot path with big, aligned, sequential writes and rare fsync to spare the SD card. Finally, a hardware watchdog petted only by a healthy loop, plus a heartbeat-gated failsafe, turn any hang or WiFi drop into a safe state instead of a runaway boat.
