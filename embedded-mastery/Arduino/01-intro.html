Foundations: Microcontrollers, the AVR & Your First Sketch

Before you write a single line, you need a correct mental model of what you are programming. We build that model from the silicon up — then make an LED blink and understand every nanosecond of why it works.

⏱ ~35 min🧠 Mental model⚡ First hardware🔧 Toolchain

1What is a microcontroller?

① Intuition build the picture first

Your laptop is a computer: a CPU plus separate chips for memory, storage, graphics, and I/O, all wired together on a big board. A microcontroller (MCU) shrinks that entire idea onto one chip — CPU, memory, and peripherals fused into a single sliver of silicon the size of a fingernail. It costs a couple of dollars, sips milliwatts, and boots in microseconds with no operating system in sight.

💡 The one-sentence model

A microcontroller is a tiny, self-contained computer that runs exactly one program forever, talking directly to the physical world through its pins.

The "Arduino" on your desk is a convenient board built around an MCU. On the classic Arduino Uno, that MCU is Microchip's ATmega328P, an 8-bit AVR chip. The board adds a USB connector, a voltage regulator, a clock crystal, and header pins — conveniences. The brain is the 328P.

🔑 Key distinction

A microprocessor (like in a PC) needs external chips to do anything. A microcontroller already contains memory and peripherals — it is "ready to run the physical world" the instant it powers on.

2Anatomy of the ATmega328P

① Intuition what's inside

Three things dominate how you write embedded code: how fast the CPU runs, how much memory you get, and which memory holds what. The 328P is deliberately tiny — and that scarcity shapes every decision you make later.

🔑 Three memories, three jobs

Flash (32 KB) — non-volatile; holds your compiled program. Survives power loss.
SRAM (2 KB) — volatile; holds variables while running. Lost on power-off.
EEPROM (1 KB) — non-volatile; holds settings you choose to persist (e.g. a calibration value).

The AVR uses a Harvard architecture: program (Flash) and data (SRAM) live on separate buses and address spaces. That is why a string constant "hello" secretly lives in Flash but gets copied to your precious SRAM at startup — a classic source of "why did I run out of RAM?" surprises we'll defuse in Chapter 8.

② Math budgeting scarce memory

Memory math is constant in embedded work. Suppose you log temperature samples as 16-bit integers (int16_t = 2 bytes each). How many fit in SRAM, ignoring overhead?

$$ N_{\max} = \left\lfloor \frac{\text{SRAM}}{\text{bytes per sample}} \right\rfloor = \left\lfloor \frac{2048 \text{ B}}{2 \text{ B}} \right\rfloor = 1024 \text{ samples} $$

In reality the stack, globals, and the Arduino core libraries already claim a few hundred bytes — so plan for headroom. This back-of-envelope habit is the difference between firmware that runs for years and firmware that mysteriously corrupts itself at 3 a.m.

3How your code becomes a blinking light

① Intuition the pipeline

When you press Upload, a surprising amount happens. Your human-readable C++ is transformed, in stages, into raw machine code, then squirted byte-by-byte into the chip's Flash. Knowing this pipeline turns "it won't compile" from panic into diagnosis.

🛠️ What the IDE hides

Your .ino file is not quite C++. The IDE concatenates your tabs, auto-generates function prototypes, wraps everything with the Arduino core, and inserts a hidden main(). The "magic" is just preprocessing — you can replicate it all on a command line with arduino-cli compile.

🏭 Real case — the bootloader

How does a fresh chip accept code over plain USB with no programmer? A small program called the bootloader lives in a protected corner of Flash. On reset it listens for ~1 second; if the IDE is talking, it writes the incoming bytes into Flash, then jumps to your program. That 1-second listen is exactly why an Uno takes a beat to start after power-up.

4The anatomy of a sketch: setup() & loop()

① Intuition run once, then forever

Every Arduino sketch has exactly two required functions. setup() runs once at boot — configure your pins here. loop() then runs over and over, forever, as fast as the CPU can go. There is no return, no "end of program" — embedded software never stops.

flowchart TB P([Power / Reset]) --> S["setup() — runs ONCE"] S --> L["loop() body"] L --> L2{reached end?} L2 -- yes --> L L2 -. never exits .-> X[(halt)] style X stroke-dasharray: 5 5,opacity:0.3
🛠️ The hidden main()

Behind the scenes the Arduino core does have a standard C++ main(). It is roughly: int main(){ init(); setup(); for(;;){ loop(); } }. So loop() is literally called inside an infinite for(;;). You are always writing inside an eternal loop.

5Your first program: Blink, fully annotated

③ Code read every line
blink.ino C++ / Arduino
// The Uno has a built-in LED wired to digital pin 13.
// LED_BUILTIN is a named constant that equals 13 on the Uno.

void setup() {
  // Run ONCE. Configure pin 13 as an OUTPUT so the MCU
  // can DRIVE a voltage onto it (instead of listening).
  pinMode(LED_BUILTIN, OUTPUT);
}

void loop() {
  // Run FOREVER.
  digitalWrite(LED_BUILTIN, HIGH);  // 5 V on the pin → LED on
  delay(500);                       // do nothing for 500 ms
  digitalWrite(LED_BUILTIN, LOW);   // 0 V on the pin → LED off
  delay(500);                       // wait another 500 ms
}                                   // loop() returns → core calls it again
💡 What each call really does

pinMode(pin, OUTPUT) flips a bit in a hardware register so the pin becomes a tiny controllable voltage source. digitalWrite(pin, HIGH) sets the pin to ~5 V; LOW sets it to 0 V. delay(ms) spins the CPU in a counting loop doing nothing useful for that many milliseconds. (That "doing nothing" is a sin we'll repent for in Chapter 5.)

⚠️ Pitfall — forgetting pinMode

If you skip pinMode(.., OUTPUT), the pin stays an input and digitalWrite(.., HIGH) quietly enables an internal pull-up resistor instead of driving the pin. Your LED stays dim or dark and nothing errors. Always configure direction first.

6Timing math: clocks, cycles & the meaning of 16 MHz

② Math where time comes from

The Uno's crystal oscillates at $f_{\text{clk}} = 16\,\text{MHz} = 16{\times}10^6$ cycles per second. Each instruction takes one or more clock cycles. The duration of a single cycle is the reciprocal:

$$ T_{\text{cycle}} = \frac{1}{f_{\text{clk}}} = \frac{1}{16\times10^6\,\text{Hz}} = 62.5\,\text{ns} $$

So in the half-second your LED is on, the CPU executes about

$$ N = f_{\text{clk}} \cdot t = 16\times10^6 \,\tfrac{\text{cyc}}{\text{s}} \times 0.5\,\text{s} = 8\times10^6 \text{ cycles} $$

— eight million cycles, almost all spent inside delay() doing absolutely nothing. Now flip it around: the blink frequency and duty cycle of our square wave are

$$ T = t_{\text{on}} + t_{\text{off}}, \qquad f = \frac{1}{T}, \qquad D = \frac{t_{\text{on}}}{T}\times100\% $$

For $t_{\text{on}} = t_{\text{off}} = 500\,\text{ms}$: period $T = 1\,\text{s}$, frequency $f = 1\,\text{Hz}$, duty $D = 50\%$. Hold this formula — it is the seed of PWM in Chapter 4, where we blink so fast the eye sees brightness instead of flashing.

7Real case: the humble status LED

④ Real case this scales to products

"Blink" feels like a toy, but the blinking status LED is everywhere in shipped hardware — routers, 3-D printers, medical devices, satellites. It is the cheapest possible human-readable diagnostic: a device with no screen can still tell you "booting" (fast blink), "running" (slow pulse), or "fault" (SOS pattern) using one LED and zero extra cost.

🏭 Real case — heartbeat as a watchdog

Mature firmware often toggles a "heartbeat" LED from inside its main loop. If the LED stops blinking, an engineer instantly knows the loop has hung — no debugger required. The blink pattern is the liveness signal. You just learned a real production technique.

Q1 Conceptual check
You define int sensorValue = 0; inside loop(). While the program runs, where does that variable physically live?
  • Flash memory
  • SRAM
  • EEPROM
  • The bootloader
Running variables live in SRAM (volatile, 2 KB on the Uno). Flash holds the program; EEPROM holds settings you explicitly persist. This is the Harvard-architecture split in action.
Q2 Compute it
A different AVR board runs at 16 MHz. How long is one clock cycle, in nanoseconds? (Recall $T_{\text{cycle}} = 1/f_{\text{clk}}$, and $1\,\text{s}=10^9\,\text{ns}$.)
ns
$T = \dfrac{1}{16\times10^6} = 62.5\times10^{-9}\,\text{s} = 62.5\,\text{ns}$. At 8 MHz it would be 125 ns; at 1 MHz, 1000 ns. Cycle time is just the reciprocal of clock frequency.
Q3 Coding challenge — Morse SOS
Modify Blink so the LED signals SOS: three short flashes (·) , three long (–), three short (·), then a long pause. Short = 200 ms on, long = 600 ms on, gap between flashes = 200 ms off. Think about how to avoid copy-pasting digitalWrite nine times.
Hint
Write one helper, flash(int onMs), that turns the LED on for onMs, off for 200 ms. Then call it three times with 200, three with 600, three with 200, and add a long delay(700) at the end of loop().
Show one solution
sos.ino C++ / Arduino
const int LED = LED_BUILTIN;

void setup() { pinMode(LED, OUTPUT); }

// one flash: on for onMs, then a 200 ms gap
void flash(int onMs) {
  digitalWrite(LED, HIGH); delay(onMs);
  digitalWrite(LED, LOW);  delay(200);
}

void loop() {
  for (int i = 0; i < 3; i++) flash(200);   // S : · · ·
  for (int i = 0; i < 3; i++) flash(600);   // O : – – –
  for (int i = 0; i < 3; i++) flash(200);   // S : · · ·
  delay(700);                                // word gap
}

The flash() helper is the lesson: factor repetition into a function. This is also a preview of why delay() is limiting — a real product needs to read buttons while blinking, which blocking delay() forbids. Chapter 5 fixes exactly this.
