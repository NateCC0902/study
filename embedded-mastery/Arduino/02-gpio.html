Digital I/O: GPIO, Ohm's Law, Buttons & Debouncing

A pin is the chip's handshake with the physical world. In Chapter 1 we blinked one. Now we make pins do work — drive LEDs without burning them, listen to buttons without lying to ourselves, and tame the messy electrical reality of a metal contact slapping shut. This is where firmware meets a soldering iron.

⏱ ~45 min🧠 GPIO model⚡ Ohm's law🔘 Buttons🪲 Debounce

1A GPIO pin is a controllable switch

① Intuition build the picture first

GPIO = General-Purpose Input/Output. Each of those numbered pins on the Uno's header is a wire that runs straight into the silicon, and a tiny bit of logic decides what that wire does. Critically, a GPIO pin has a direction, and it is one of exactly two things at any instant:

  • OUTPUT — the pin drives. The chip clamps the wire to either ~5 V (HIGH) or 0 V (LOW). You are talking.
  • INPUT — the pin listens. The chip leaves the wire alone and measures the voltage some other circuit puts on it. You are reading.
💡 The one-sentence model

A GPIO pin is a software-controlled switch: as an OUTPUT it connects the wire to +5 V or to ground; as an INPUT it disconnects and quietly senses whatever voltage arrives.

When an output is HIGH, current flows out of the pin into your circuit — the pin sources current. When it is LOW, current flows into the pin from your circuit down to ground — the pin sinks current. Same pin, opposite directions of flow. Which one you use changes how you wire the LED, as we'll see.

🔑 Two registers behind every pin

On the AVR, a pin's direction is one bit in the DDRx register (Data Direction Register) and its output level is one bit in PORTx. Reading an input pin reads the PINx register. pinMode() and digitalWrite() are just friendly wrappers that flip those bits for you.

2The electrical model: Ohm's law & current limits

① Intuition voltage pushes, resistance resists

Before you connect anything to a pin you must think in volts, amps, and ohms. Voltage ($V$, volts) is electrical "pressure". Current ($I$, amps) is the rate of charge flow. Resistance ($R$, ohms) opposes that flow. The relationship between them is the single most important equation in all of electronics:

② Math derive the resistor
$$ V = I \cdot R \qquad\Longleftrightarrow\qquad I = \frac{V}{R} \qquad\Longleftrightarrow\qquad R = \frac{V}{I} $$

Now the classic beginner mistake: wiring an LED straight from a pin to ground. An LED is a diode — past its forward voltage $V_f$ it behaves almost like a short circuit, so current rockets up until something melts. We tame it with a series current-limiting resistor. Walk the loop with Kirchhoff's voltage law: the pin supplies $V_{cc}$, the LED drops $V_f$, and the resistor must drop whatever's left, $V_{cc}-V_f$. Apply Ohm's law to the resistor:

$$ R = \frac{V_{cc} - V_f}{I} $$
∑ Worked example — sizing an LED resistor

Drive a red LED ($V_f \approx 2\,\text{V}$) from a 5 V pin and target $I = 10\,\text{mA} = 0.010\,\text{A}$:

$$ R = \frac{5\,\text{V} - 2\,\text{V}}{0.010\,\text{A}} = \frac{3\,\text{V}}{0.010\,\text{A}} = 300\ \Omega $$

No 300 Ω in your kit? Round up to the nearest stocked value (330 Ω) — more resistance means less current, which is the safe direction. A 220 Ω would push $I = 3/220 \approx 13.6\,\text{mA}$: still fine, just brighter.

How much heat does that resistor dissipate? Power is voltage times current, and combining with Ohm's law gives the form you'll quote most:

$$ P = V \cdot I = I^2 \cdot R = \frac{V^2}{R} $$

For our resistor: $P = I^2 R = (0.010)^2 \times 300 = 0.03\,\text{W} = 30\,\text{mW}$. A standard ¼-watt (250 mW) resistor laughs at that — nearly 10× margin. Power math only gets interesting when you drive motors or many LEDs, but the habit of checking it starts now.

⚠️ Pitfall — the Uno pin current limit is real

An ATmega328P pin can physically source or sink about 40 mA absolute maximum — exceed it and you risk permanently damaging the pin's driver. The safe design target is ≤ 20 mA per pin, and the whole chip has a combined limit (~200 mA across all pins, with per-port caps). Our 10 mA LED sits comfortably below the line. Need to drive something hungry? Use a transistor or MOSFET — never the bare pin.

3Outputs: sourcing, sinking & LED polarity

③ Code read every line

Configuring an output is two calls: set the direction once in setup(), then drive the level with digitalWrite(). Here is a single external LED on pin 8, wired active-high (pin → resistor → LED → ground), blinking:

led_output.ino C++ / Arduino
const uint8_t LED_PIN = 8;          // our LED's anode side, via a 330R resistor

void setup() {
  pinMode(LED_PIN, OUTPUT);         // DDRB bit for D8 := 1  → pin now DRIVES
}

void loop() {
  digitalWrite(LED_PIN, HIGH);      // pin = 5V → current SOURCES out → LED on
  delay(300);
  digitalWrite(LED_PIN, LOW);       // pin = 0V → no drop across LED → LED off
  delay(300);
}

That LED is active-high: HIGH turns it on because current flows out of the pin (sourcing), through the LED, to ground. But you can flip the wiring. Connect +5V → resistor → LED → pin instead, and now the pin must go LOW to light it — current flows into the pin (sinking) toward ground. That is an active-low output.

🛠️ Why sinking is sometimes preferred

Many MCUs (and logic families generally) can sink a touch more current than they source, and sinking keeps the LED at a fixed +5 V rail rather than depending on the pin's slightly-below-5 V high level. For one LED it rarely matters; for arrays and 7-segment displays it does. Just stay consistent and know which one you wired — flipped logic in software is the usual culprit when "my LED is on when it should be off".

4Inputs & the floating-pin trap

① Intuition an unconnected input is an antenna

Reading a pin sounds trivial: call digitalRead(), get HIGH or LOW. The trap is what happens when nothing actively drives the pin. A bare INPUT pin is extremely high-impedance — it barely draws any current — so if you leave it unconnected it does not read a clean 0. It floats: it picks up stray electromagnetic noise from your hand, the mains wiring, a nearby motor, and flickers randomly between HIGH and LOW.

⚠️ Pitfall — floating inputs

A button typically connects a pin to ground when pressed and connects it to nothing when released. With a plain INPUT, "nothing" means floating, so released reads are garbage. The fix is to gently tie the pin to a known voltage with a pull resistor.

② Math why "pull" needs a resistor

A pull-up resistor connects the pin to +5 V; a pull-down connects it to ground. It "pulls" the idle pin to a defined level. Why not a plain wire? Because when the button does connect the pin to the opposite rail, the pull resistor and the button form a path from 5 V to ground — the resistor limits that short-circuit current to something harmless:

$$ I_{\text{pull}} = \frac{V_{cc}}{R_{\text{pull}}} $$

The AVR's internal pull-up is roughly $R_{\text{pull}} \approx 20\text{–}50\,\text{k}\Omega$. Take a mid value of 30 kΩ: when the button is pressed it ties the pin to ground, and the current through the pull-up is $I = 5/30000 \approx 0.17\,\text{mA}$ — utterly negligible, but enough to hold a solid logic level when the button is open. That high resistance is exactly why it's "weak": strong enough to define the idle state, weak enough that a button easily overrides it.

🔑 INPUT_PULLUP — free, built-in, the right default

Rather than soldering an external resistor, the AVR has a pull-up built into every GPIO. Enable it with pinMode(pin, INPUT_PULLUP). The pin idles HIGH; a button to ground reads LOW when pressed. This is the standard way to read a button on Arduino — no extra parts.

5Reading a button: active-low logic

① Intuition pressed = LOW, and that's not a bug

Wire a momentary push-button between pin 2 and ground, enable the internal pull-up, and you get the canonical Arduino button. Here is the circuit as a graph:

flowchart LR VCC["+5V (internal)"] -- "≈30kΩ pull-up" --> NODE(("pin D2")) NODE --> MCU["digitalRead(2)"] NODE -- "button (momentary)" --> GND["GND"]

Trace the two states. Released: the button is open, so the pull-up alone wins and holds D2 at +5 V → digitalRead returns HIGH. Pressed: the button shorts D2 to ground, overpowering the weak pull-up, so D2 sits at 0 V → digitalRead returns LOW. The pressed state reads LOW: this is active-low logic, and it is the normal, correct outcome of using a pull-up.

③ Code read every line
button_read.ino C++ / Arduino
const uint8_t BTN_PIN = 2;          // button between D2 and GND
const uint8_t LED_PIN = LED_BUILTIN;

void setup() {
  pinMode(BTN_PIN, INPUT_PULLUP);   // idle HIGH; pressed pulls to LOW
  pinMode(LED_PIN, OUTPUT);
}

void loop() {
  // Active-low: LOW means pressed. Invert so 'pressed' reads as true.
  bool pressed = (digitalRead(BTN_PIN) == LOW);
  digitalWrite(LED_PIN, pressed ? HIGH : LOW);   // LED mirrors the button
}
🛠️ Make intent obvious

Comparing against LOW and naming the result pressed hides the active-low inversion behind a readable word. Future-you (and the next engineer) should never have to mentally re-derive "wait, is LOW pressed or released?" — the variable name answers it.

6The problem: contact bounce

① Intuition metal slapping metal rings

A push-button is two springy metal contacts. When you press, they don't mate cleanly — they slam together, bounce apart, slam again, and chatter for a few milliseconds before settling. Electrically the pin doesn't see one crisp HIGH→LOW edge; it sees a burst of edges. Typical mechanical switches bounce for ~1–10 ms. Here's the scope trace, described in time:

HIGH"] --> B["press!
↯ bounce

LOW (1–10ms later)"] C --> D["release!
↯ bounce again"] D --> E["Released
HIGH"]

The CPU runs loop() millions of times a second, far faster than those bounces. So a single physical press looks to naïve code like several presses. If you're toggling an LED or — worse — counting presses, one push registers as three, five, a random number. This is the most common "my code is haunted" bug in beginner embedded.

🏭 Real case — the counter that lied

An industrial parts counter watches a limit switch trip once per item on a conveyor. Un-debounced, contact bounce made it tally 12,000 parts when 9,800 had passed — a 22% phantom overcount that wrecked inventory reconciliation and shipped short. Input conditioning (debounce in firmware, sometimes an RC filter in hardware) isn't polish; it's the difference between a count you can trust and one you can't. Every PLC and every product-grade button input debounces.

7Debouncing: from blocking to non-blocking

① Intuition wait for the ringing to stop

Debouncing means: ignore the chatter and report one clean transition. There are three escalating approaches.

1. The blocking delay() hack. See an edge, then delay(20) to sleep through the bounce window, then re-read. Simple, and it works — but it freezes the entire program for 20 ms. Nothing else can run: no other buttons, no sensors, no blinking. Fine for a one-button toy, fatal in a real product.

debounce_blocking.ino C++ / Arduino
if (digitalRead(BTN_PIN) == LOW) {  // possible press
  delay(20);                        // ⛔ FREEZES everything for 20 ms
  if (digitalRead(BTN_PIN) == LOW)  // still pressed? it was real
    handlePress();
}
② Math picking the window

How long to wait? Long enough to outlast the bounce, short enough to feel instant. With bounce up to $t_b \approx 10\,\text{ms}$, a debounce window of $t_d$ must satisfy:

$$ t_b < t_d \ll t_{\text{human}} \qquad\Rightarrow\qquad 10\,\text{ms} < t_d \ll 100\,\text{ms} $$

A human can't press faster than ~10 times a second (≈100 ms apart), so anything from 20–50 ms sits safely between "longer than the bounce" and "shorter than a real second press". We'll use 25 ms.

③ Code the non-blocking millis() debounce

2. The non-blocking millis() debounce. Instead of sleeping, we remember the time of the last bounce and only accept the input once it has been stable for the whole window. millis() returns milliseconds since boot without ever blocking. We track the last raw reading, the last stable reading, and a timestamp — classic state tracking:

debounce_millis.ino C++ / Arduino
const uint8_t BTN_PIN = 2;
const unsigned long DEBOUNCE_MS = 25;     // > max bounce, < human reaction

int  lastRaw     = HIGH;                   // previous *raw* reading
int  stableState = HIGH;                   // last *accepted* (debounced) level
unsigned long lastChange = 0;              // when raw last flipped
long pressCount = 0;

void setup() {
  pinMode(BTN_PIN, INPUT_PULLUP);
  Serial.begin(9600);
}

void loop() {
  int raw = digitalRead(BTN_PIN);          // active-low: LOW == pressed

  if (raw != lastRaw) {                     // raw flipped → (re)start the timer
    lastChange = millis();                  // note WHEN it bounced
    lastRaw = raw;
  }

  // Has the raw level held steady longer than the debounce window?
  if ((millis() - lastChange) >= DEBOUNCE_MS) {
    if (raw != stableState) {               // a NEW, settled level
      stableState = raw;
      if (stableState == LOW) {             // settled LOW = a real press
        pressCount++;
        Serial.println(pressCount);
      }
    }
  }
  // loop() never blocks — other tasks can run right here, every pass.
}
💡 What the timer really measures

Every time the raw pin changes, we reset lastChange to "now". So millis() - lastChange is "how long the pin has been quiet". We only trust the reading once that quiet stretch exceeds 25 ms — i.e., once the bouncing has demonstrably stopped. Counting on the HIGH→LOW settled edge (not every edge) is what guarantees one count per press.

This is the same pattern as the non-blocking blink you'll meet in Chapter 5: keep a timestamp, compare against millis(), never delay(). It's the backbone of doing many things at once on a single-threaded MCU.

🛠️ Don't reinvent it forever — but understand it once

Production code often reaches for a vetted library like Bounce2, which wraps exactly this state machine. That's fine — after you can write it from scratch. A library you don't understand is a bug you can't fix.

Q1 Conceptual check
You wire a button between pin D2 and GND and use pinMode(2, INPUT_PULLUP). When the button is pressed, what does digitalRead(2) return, and why?
  • HIGH — the pull-up forces it high whenever pressed
  • LOW — the button shorts the pin to ground, overriding the weak pull-up
  • HIGH — pressing connects the pin to +5 V
  • It floats randomly because nothing drives the pin
With a pull-up the pin idles HIGH. Pressing connects it to ground, and since the internal pull-up is weak (~20–50 kΩ) the button easily wins, dragging the pin to 0 V → LOW. This is active-low logic: pressed = LOW. It's not a bug; it's the defined behaviour of a pull-up button.
Q2 Compute it
Size the current-limiting resistor for an LED with forward voltage $V_f = 2.1\,\text{V}$ driven from $V_{cc} = 5\,\text{V}$ at a target current of $I = 15\,\text{mA}$. Use $R = (V_{cc}-V_f)/I$. Give $R$ in ohms.
Ω
$R = \dfrac{5 - 2.1}{0.015} = \dfrac{2.9}{0.015} \approx 193\,\Omega$. In practice you'd round up to a stocked 220 Ω (slightly less current, the safe direction). The power burned is $P = I^2R = (0.015)^2 \times 193 \approx 43\,\text{mW}$ — easy for a ¼-watt part.
Q3 Coding challenge — non-blocking toggle counter
Write a sketch that uses a non-blocking millis() debounce on a button (D2, INPUT_PULLUP) to toggle the built-in LED on each clean press, and print a running press count over Serial. No delay() anywhere. The key constraint: exactly one toggle per physical press, no matter how badly the contact bounces.
Hint
Reuse the state-tracking pattern: store lastRaw, stableState, and lastChange. Reset lastChange = millis() whenever the raw reading flips. Only act once millis() - lastChange >= 25 and the settled level is a new LOW. On that settled-LOW edge, flip an ledOn boolean and digitalWrite it.
Show one solution
toggle_counter.ino C++ / Arduino
const uint8_t BTN = 2, LED = LED_BUILTIN;
const unsigned long DEBOUNCE_MS = 25;

int  lastRaw     = HIGH;
int  stableState = HIGH;
unsigned long lastChange = 0;
bool ledOn = false;
long presses = 0;

void setup() {
  pinMode(BTN, INPUT_PULLUP);
  pinMode(LED, OUTPUT);
  Serial.begin(9600);
}

void loop() {
  int raw = digitalRead(BTN);

  if (raw != lastRaw) {              // bounce or real edge: restart timer
    lastChange = millis();
    lastRaw = raw;
  }

  if ((millis() - lastChange) >= DEBOUNCE_MS) {
    if (raw != stableState) {        // settled to a new level
      stableState = raw;
      if (stableState == LOW) {      // settled press → act ONCE
        ledOn = !ledOn;              // toggle
        digitalWrite(LED, ledOn ? HIGH : LOW);
        presses++;
        Serial.println(presses);
      }
    }
  }
  // ...other non-blocking work can live here; loop() never stalls.
}

The trick is acting only on the settled HIGH→LOW transition, not on every raw edge — that's what collapses a noisy burst into a single toggle. Because nothing blocks, you could debounce ten buttons and blink three LEDs in the same loop(). That composability is the whole reason we abandoned delay().
