通訊：UART、I²C 與 SPI

一顆 MCU 的價值，取決於它能跟哪些東西對話。感測器、顯示器、SD 卡、無線電、其他微控制器——它們全都使用三種匯流排之一。學會這三種，你幾乎就能把晶片接上地球上的任何東西。我們會從每種匯流排的接線一根一根建立起來，做時序計算，最後讓三種匯流排同時運作。

⏱ 約 50 分鐘🧠 匯流排協定⚡ 多裝置🔧 Wire 與 SPI 函式庫

1並列與串列：用時間換取線路

① 直覺 先建立整體圖像

你手上有一個位元組——八個位元——而你需要把它從晶片 A 搬到晶片 B。方法剛好只有兩種。並列（parallel）：拉八條線，把八個位元一次全部送上去，一個時脈滴答就完成。串列（serial）：只拉一條線，把位元一個接一個塞進去，一個位元組要八個滴答。並列每個時脈更快，但吃掉接腳和 PCB 面積；串列每個時脈較慢，但很省接腳、佈線也漂亮。

💡 最根本的取捨

本章每一種匯流排都是串列的：它用時間（每個位元組要更多時脈滴答）去換線路（更少實體接腳）。在一顆只有約 20 支可用接腳的 MCU 上，線路才是稀缺資源——所以串列幾乎在哪裡都勝出。

「但並列比較快啊！」——在 1990 年代、對短纜線而言，確實如此。然後時脈速度一路飆升，並列撞上一堵牆：在高頻下那八條線不再能保持同步（偏移 skew），串擾（crosstalk）也把資料毀掉。這正是為什麼 USB、SATA、PCIe、HDMI 和乙太網路如今全都是串列的。在微控制器上，你必須掌握的三種串列匯流排是 UART、I²C 和 SPI。它們的差別在於用幾條線、是否帶時脈，以及如何挑選要跟哪個裝置對話。

2UART：非同步的主力

① 直覺 沒有時脈線——只要對好速度

UART（Universal Asynchronous Receiver/Transmitter，通用非同步收發器）是最簡單的匯流排，也是你每次呼叫 Serial.print() 時就已經在用的。它是點對點（point-to-point）的：剛好兩個裝置，以兩條資料線相連——TX（傳送）與 RX（接收）——而且交叉接線，讓每一端的 TX 餵給對方的 RX。非同步（asynchronous）這個詞就是整個訣竅所在：沒有共用的時脈線。取而代之，雙方事先講好一個速度——鮑率（baud rate）——然後各自依照自己內部的時脈去取樣線路。

🔑 訊框格式：8N1 約定

因為沒有時脈，接收端需要知道一個位元組何時開始。所以每個位元組都被包進一個訊框（frame）裡。閒置時線路維持 HIGH；一個下降緣就是起始位元（start bit）（1 位元，LOW）。接著是 8 個資料位元（LSB 先送），可選地加上一個同位元（parity bit），再來是 1 個停止位元（stop bit）（HIGH）。常見的設定「8N1」表示 8 個資料位元、No（無）同位、1 個停止位元 → 線路上要 10 個位元才能載運 8 個位元的酬載。

② 數學 鮑率、位元時間與吞吐量

鮑率是每秒的訊號事件數——在這裡就是位元數。線路上一個位元的持續時間，也就是位元時間（bit time），就是它的倒數：

$$ t_{\text{bit}} = \frac{1}{\text{baud}} $$

在無所不在的 $115200$ 鮑率下，$t_{\text{bit}} = 1/115200 \approx 8.68\,\mu\text{s}$；在老派的 $9600$ 鮑率下，每個位元則暴增到 $104.2\,\mu\text{s}$。現在來看關鍵洞見：8N1 每個位元組要花線路上 10 個位元（起始 + 8 + 停止），所以有用的位元組吞吐量（byte throughput）是

$$ \text{throughput} = \frac{\text{baud}}{10} \;\text{bytes/s} \qquad\Rightarrow\qquad \frac{115200}{10} = 11{,}520\ \text{B/s} $$

而送出一個 $N$ 位元組緩衝區所需的時間就只是

$$ t = \frac{10\,N}{\text{baud}} \qquad\Rightarrow\qquad \frac{10 \times 64}{115200} \approx 5.56\,\text{ms（64 位元組）} $$

那個 ÷10 就是訊框格式的額外開銷——你有 20% 的線路時間沒在載運酬載。這是沒有時脈線的代價。注意鮑率和位元組/秒不是同一個數字；把這兩者搞混就是經典的新手錯誤。

③ 程式碼 逐行讀過
uart_echo.ino C++ / Arduino
void setup() {
  // 以 115200 鮑率、8N1（預設訊框）開啟硬體 UART。
  // 在 Uno 上，這個 UART 就是接到 USB 的那一個，所以你 PC 上的
  // 序列監控視窗（Serial Monitor）就是匯流排上的「另一個裝置」。
  Serial.begin(115200);
  Serial.println("Type a number, I'll double it:");
}

void loop() {
  // Serial.available() 回傳已經到達的位元組數，這些位元組存在
  // 核心透過中斷填入的 64 位元組 RX 緩衝區裡。
  if (Serial.available() > 0) {
    // parseInt() 會阻塞，直到收到一個完整的整數，
    // 再把 ASCII 數字轉成 long。
    long n = Serial.parseInt();
    Serial.print("You sent ");
    Serial.print(n);              // print = 經由 TX 送出 ASCII
    Serial.print(", doubled = ");
    Serial.println(n * 2);        // println 會加上 "\r\n"
  }
}
⚠️ 陷阱——鮑率不符與缺少共地

如果兩端對鮑率的設定不一致，接收端會在錯誤的時刻取樣，你就會得到亂碼字元（那種一看就知道的「Ÿ¿Œ」一團亂）。把兩端設成完全一樣。而且 UART 需要共用接地：TX/RX 是相對於 GND的電壓，所以沒有共同參考，這些位元就毫無意義。兩塊沒有共地的板子「在講」UART，是第一名的無聲失敗。

3I²C：兩條線，掛上一整群裝置

① 直覺 一條帶位址的共線派對線

UART 對兩個裝置很棒，但它撐不起一整群。輪到 I²C 上場（Inter-Integrated Circuit，唸作「eye-squared-see」）。它只用兩條線，由匯流排上每一個裝置共用：SDA（serial data，串列資料）與 SCL（serial clock，串列時脈）。和 UART 不同，I²C 確實帶有時脈——這正是它同步（synchronous）的原因——而且有一個裝置，也就是控制器（controller）（歷史上稱「master 主機」），負責驅動該時脈並啟動每一次交易。其他裝置則是目標（target）（「slave 從機」），只有被點名時才回應。

🔑 位址：100 多顆晶片如何共用 2 條線

每個目標都有一個由製造商燒進去的7 位元位址（有時會留幾支接腳讓你微調低位元）。七個位元 → $2^7 = 128$ 個槽位，其中 16 個保留，剩下最多 112 個可用位址——口語上就稱「約 127 個裝置」。控制器廣播一個位址；只有相符的那個目標會回應。這就像一條派對線，你喊一個名字，就有一個人回話。

巧妙之處在於電路設計：兩條線都是開汲極（open-drain）。一個裝置只能把線路拉低到 0；它永遠無法把線路驅動到高。兩顆外接的上拉電阻（pull-up resistor）（通常 4.7 kΩ）會在沒人拉低時，把兩條線輕輕拉到 $V_{cc}$。這表示任意數量的裝置都能共用線路而永遠不會打架（不會有兩個輸出短路），也賦予了 I²C 它的招牌特性：ACK/NACK。每個位元組之後，接收端會把 SDA 拉低一個時脈以表示「收到了」（ACK）；讓它浮回高就表示「沒有」（NACK）。

② 數學 上拉電阻選值與匯流排速度

I²C 可跑 100 kHz（標準模式）或 400 kHz（快速模式）；有些零件可達 1 MHz 以上。上拉電阻值是一個實實在在的工程取捨。線路是一個 RC 電路：匯流排電容 $C_b$（線路 + 每個裝置的接腳電容）透過上拉電阻 $R_p$ 充電。上升時間必須塞得進一個時脈週期。規格把它上限定在

$$ t_r \approx 0.847\,R_p\,C_b \;\le\; \begin{cases} 1000\ \text{ns} & \text{(100 kHz)}\\ 300\ \text{ns} & \text{(400 kHz)} \end{cases} $$

以典型的 $C_b = 100\,\text{pF}$ 來看，要達到 300 ns 的快速模式上限，需要 $R_p \le t_r / (0.847\,C_b) \approx 3.5\,\text{k}\Omega$——這就是為什麼 4.7 kΩ 在 100 kHz 行得通，但在 400 kHz 或長匯流排時要降到約 2.2 kΩ。電阻太大 → 邊緣遲緩 → 資料損毀；電阻太小 → 開汲極電晶體灌不出足夠電流來達到有效的 LOW。中間有一個窗口，而匯流排電容決定了這個窗口。

🏭 真實案例——一條匯流排、三顆晶片、零額外接腳

那張圖是一個真實、可量產的設計：一顆溫度感測器（TMP102，位址 0x48）、一顆即時時鐘（DS3231，位址 0x68）和一片 OLED 顯示器（SSD1306，位址 0x3C），全都掛在同樣兩支接腳上（Uno 上的 A4/A5）。再加第四、第五個裝置，你還是只用兩支接腳。這就是為什麼 I²C 主宰了感測器擴充板——Adafruit 和 SparkFun 出貨了好幾百種。

③ 程式碼 透過 Wire 函式庫讀取一個暫存器
i2c_read_temp.ino C++ / Arduino
#include <Wire.h>

const uint8_t TMP102_ADDR = 0x48;  // 7 位元目標位址
const uint8_t TEMP_REG    = 0x00;  // 存放溫度的暫存器

void setup() {
  Serial.begin(115200);
  Wire.begin();                    // 以控制器身分加入匯流排
}

void loop() {
  // 1) 告訴感測器我們要從「哪一個」暫存器讀取。
  Wire.beginTransmission(TMP102_ADDR);
  Wire.write(TEMP_REG);            // 設定內部暫存器指標
  Wire.endTransmission(false);     // false = 重複起始（repeated-START），保留匯流排

  // 2) 現在向該暫存器要回 2 個位元組。
  Wire.requestFrom(TMP102_ADDR, (uint8_t)2);
  uint8_t msb = Wire.read();       // 最高有效位元組先到
  uint8_t lsb = Wire.read();

  // TMP102 把一個 12 位元的值靠左對齊地塞進 16 位元裡。
  int16_t raw = ((msb << 8) | lsb) >> 4;
  float celsius = raw * 0.0625f;   // 每個 LSB 代表 0.0625 °C

  Serial.print("Temp: "); Serial.print(celsius); Serial.println(" C");
  delay(1000);
}
⚠️ 陷阱——7 位元 vs 8 位元位址

規格書最愛把你搞糊塗。Wire 函式庫要的是 7 位元位址（0x48），它會自己接上讀/寫位元。許多規格書反而印出8 位元位址，也就是已經把那個位元移進去的版本（0x90 寫 / 0x91 讀）。如果你的裝置毫無反應，就檢查你是不是餵給 Wire 一個左移過的位址。拿不定主意時，跑一個 i2c_scanner 草稿碼——它會逐一探測全部 127 個位址，並回報誰有 ACK。

4SPI：四條線，全速前進

① 直覺 純粹的速度、全雙工

當你需要快——把像素串流到顯示器、從 SD 卡讀取數 KB、跟無線電對話——你就會伸手拿 SPI（Serial Peripheral Interface，串列周邊介面）。它花更多線路去買更多速度。四條訊號：SCK（時脈，來自控制器）、MOSI（Controller-Out Peripheral-In，控制器輸出/周邊輸入）、MISO（Controller-In Peripheral-Out，控制器輸入/周邊輸出），以及每個裝置一條的 CS / SS（Chip Select，晶片選擇）。資料同時朝兩個方向流動——這就是全雙工（full duplex）——而且沒有定址協定：你靠把某個晶片的 CS 線拉低（LOW）來選中它。

🔑 移位暫存器心智模型

SPI 實質上就是兩個接成一個環的移位暫存器。在每個 SCK 邊緣，控制器和周邊各自移出一個位元（MOSI / MISO）、移入一個位元。經過 8 個時脈，它們就交換了一整個位元組。所以每一次 SPI 傳輸都是一次交換：要讀，你就必須同時寫（通常是一個無用的虛擬位元組 dummy byte）。這種對稱性正是 SPI 沒有 ACK、也沒有流量控制的原因——它就只是被時脈推動的位元交換，而這恰恰是它快的原因（數十 MHz，相對於 I²C 的 0.4 MHz）。

代價是：每個周邊都要一支專屬的 CS 接腳。三個 SPI 裝置 = SCK + MOSI + MISO 共用 + 3 條 CS = 6 支接腳，相對於 I²C 永遠的 2 支。SPI 用接腳換速度；I²C 用速度換接腳。看你缺哪種資源來選。

② 數學 四種 SPI 模式（CPOL/CPHA）

SPI 唯一的微妙之處是時序極性。兩個位元定義了時脈的行為：CPOL（clock polarity，時脈極性——閒置是 HIGH 還是 LOW？）和 CPHA（clock phase，時脈相位——在第一個還是第二個邊緣取樣？）。它們的四種組合就是四種 SPI 模式：

$$ \text{Mode} = 2\cdot\text{CPOL} + \text{CPHA}, \qquad \text{CPOL},\text{CPHA}\in\{0,1\} $$

傳輸時間簡單到很殘酷——沒有像 UART 那樣的訊框開銷，也沒有像 I²C 那樣的 ACK：$t = 8N / f_{\text{SCK}}$。在 $f_{\text{SCK}} = 8\,\text{MHz}$ 下，一個 512 位元組的 SD 區塊要花 $8 \times 512 / 8{\times}10^6 = 512\,\mu\text{s}$——大約比同樣酬載走 400 kHz I²C 快 11 倍。把模式對到規格書，不然你會在錯誤的邊緣取樣位元，讀到的是純雜訊。

③ 程式碼 設定一次交易
spi_read.ino C++ / Arduino
#include <SPI.h>

const int CS_PIN = 10;             // 此周邊的晶片選擇腳

void setup() {
  pinMode(CS_PIN, OUTPUT);
  digitalWrite(CS_PIN, HIGH);     // HIGH = 未選中（閒置）
  SPI.begin();                    // 設定 SCK、MOSI、MISO 的方向
}

uint8_t readRegister(uint8_t reg) {
  // 一次交易會鎖定匯流排速度、位元順序與模式，這樣兩個設定
  // 不同的 SPI 裝置才能安全共存。
  SPI.beginTransaction(SPISettings(8000000, MSBFIRST, SPI_MODE0));
  digitalWrite(CS_PIN, LOW);      // 選中「這個」裝置
  SPI.transfer(reg | 0x80);       // 許多零件：bit7=1 表示「讀取」
  uint8_t value = SPI.transfer(0x00);  // 送出虛擬位元組，把回覆時脈進來
  digitalWrite(CS_PIN, HIGH);     // 取消選中
  SPI.endTransaction();
  return value;
}

void loop() { /* 視需要呼叫 readRegister(...) */ }
🛠️ 為什麼 SPI.transfer 同時送出又接收

沒有獨立的「讀取」呼叫。SPI.transfer(x) 把 x 從 MOSI 移出，同時回傳周邊從 MISO 移回來的東西——每個方向一個位元組，同樣的 8 個時脈。要從裝置拉資料，你就移出一個用完即丟的 0x00，讓 SCK 持續滴答，回覆就會搭著 MISO 進來。全雙工不是你要去開啟的功能；它就是這條線運作的方式。

#include <Wire.h>      // I2C：溫度感測器
#include <SPI.h>       // SPI：SD 卡（被 SD.h 在底層使用）
#include <SD.h>

const uint8_t TMP102 = 0x48;
const int     SD_CS  = 4;        // SD 擴充板的晶片選擇腳

void setup() {
  Serial.begin(115200);          // UART  → 對 PC 除錯
  Wire.begin();                  // I2C   → 感測器匯流排
  if (!SD.begin(SD_CS)) {        // SPI   → SD 卡（SD 函式庫驅動 SPI）
    Serial.println("SD init failed!");
    while (1);                    // 停住：沒有磁碟就沒有記錄的意義
  }
  Serial.println("Logger ready.");
}

float readTempC() {
  Wire.beginTransmission(TMP102);
  Wire.write(0x00);
  Wire.endTransmission(false);
  Wire.requestFrom(TMP102, (uint8_t)2);
  int16_t raw = ((Wire.read() << 8) | Wire.read()) >> 4;
  return raw * 0.0625f;
}

void loop() {
  float t = readTempC();                       // ← 來自 I2C
  File f = SD.open("log.csv", FILE_WRITE);      // ← 經由 SPI
  if (f) { f.print(millis()); f.print(','); f.println(t); f.close(); }
  Serial.print("Logged "); Serial.println(t);   // ← 經由 UART
  delay(2000);
}
⚠️ 陷阱——匯流排爭用與電壓位準

三種匯流排能共存，純粹是因為它們在不同的接腳上——千萬不要把一條 SPI MISO 接到一條 I²C SDA 上。還有兩個更隱晦的陷阱：(1) 在 Uno 上，接腳 D11–13 是 SPI 匯流排，所以別把它們挪去做別的事；(2) 注意電壓位準——一張 3.3 V 的 SD 卡接在 5 V 的 Uno 上，需要在 MOSI/SCK 上加裝位準轉換器（level shifter），否則你會慢慢把它烤壞。混壓匯流排是業餘專案的無聲殺手。

Q1 觀念檢測
你要在一顆 MCU 上掛八顆慢速溫度感測器，同時盡可能少燒接腳。該用哪種匯流排，而這八顆總共要幾條線？
  • SPI——它最快，所以一律用它（10 條線：3 條共用 + 8 條 CS）
  • I²C——總共 2 條線；每顆感測器在共用匯流排上各有自己的 7 位元位址
  • UART——每顆感測器 2 條線，所以 16 條線
  • SPI——2 條線，就跟 I²C 一樣
對於少接腳上掛許多慢速裝置，答案就是 I²C：八顆感測器全都共用同樣的2 條線（SDA + SCL），各以其 7 位元位址區分。SPI 會需要每個裝置一支獨立的 CS 腳（3 條共用 + 8 = 11 支接腳）；UART 是點對點，所以根本沒辦法把它們匯流。速度不是這裡的限制——接腳數才是。
Q2 算算看
你以 115200 鮑率、8N1 透過 UART 送出一個 64 位元組的封包。這次傳輸花多久時間，以毫秒計？（回想 8N1 每個位元組用 10 個線路位元，所以 $t = \dfrac{10N}{\text{baud}}$。）
ms
$t = \dfrac{10 \times 64}{115200} = \dfrac{640}{115200} \approx 5.56\times10^{-3}\,\text{s} = 5.56\,\text{ms}$。等價地，吞吐量是 $115200/10 = 11{,}520$ B/s，而 $64 / 11520 \approx 5.56$ ms。那個係數 10（而非 8）就是訊框開銷——起始 + 8 個資料 + 停止位元。
Q3 程式挑戰——讀取一個 I²C 暫存器
寫一個函式 uint8_t readReg(uint8_t addr, uint8_t reg)，使用 Wire 函式庫，從一個 I²C 裝置讀取單一個 8 位元暫存器：把裝置的暫存器指標指向 reg，然後請求並回傳一個位元組。這是幾乎每一顆 I²C 晶片通用的「讀取一個暫存器」模式。
提示
這個模式分成兩個階段。寫入階段：beginTransmission(addr)、write(reg)，然後 endTransmission(false)——那個 false 會送出重複起始（repeated START），而不是釋放匯流排。讀取階段：requestFrom(addr, 1)，然後 Wire.read() 回傳該位元組。
顯示一個解答
read_reg.ino C++ / Arduino
#include <Wire.h>

// 從任何 I2C 裝置讀取一個 8 位元暫存器。
uint8_t readReg(uint8_t addr, uint8_t reg) {
  Wire.beginTransmission(addr);   // 對裝置開始一次寫入
  Wire.write(reg);                // 設定它的內部暫存器指標
  Wire.endTransmission(false);    // 重複起始（repeated-START）：保留匯流排

  Wire.requestFrom(addr, (uint8_t)1);   // 要求 1 個位元組
  return Wire.available() ? Wire.read() : 0;
}

void setup() {
  Serial.begin(115200);
  Wire.begin();
  uint8_t whoAmI = readReg(0x68, 0x75); // MPU-6050 WHO_AM_I → 0x68
  Serial.print("WHO_AM_I = 0x");
  Serial.println(whoAmI, HEX);
}

void loop() {}

endTransmission(false) 是關鍵：單純的 true 會送出 STOP、釋放匯流排，而有些裝置會在 STOP 時重設它們的指標。重複起始讓整段對話保持原子性（atomic）。先讀一個 WHO_AM_I 暫存器是經典的健全性檢查——如果它回傳規格書上的魔術數字，就代表你的接線和位址都正確。

你現在能把串列與並列當成一種「線路換時間」的取捨來推理，也已徹底掌握三種匯流排。UART：2 條線、無時脈、講好鮑率、每位元組 10 位元、吞吐量 = 鮑率/10。I²C：2 條共用線加上拉電阻、用 7 位元位址掛約 100 多個裝置、ACK/NACK、100k/400 kHz。SPI：4 條線、全雙工的移位暫存器交換、每裝置一條 CS、MHz 等級快、四種 CPOL/CPHA 模式。你寫了透過 Wire 與 SPI 的暫存器讀取、選了上拉電阻值、算了傳輸時間，並在一台資料記錄器上同時跑了三種匯流排。下一章我們將讓這些匯流排實際動起來，驅動真正的感測器與致動器。
