數位 I/O：GPIO、歐姆定律、按鈕與去彈跳

接腳是晶片與實體世界握手的地方。第 1 章我們讓一顆 LED 閃爍。現在我們要讓接腳真正做事——驅動 LED 而不燒掉它們、傾聽按鈕而不自欺欺人，並馴服金屬接點「啪」一聲閉合時那混亂的電氣現實。這正是韌體與烙鐵交會之處。

⏱ 約 45 分鐘🧠 GPIO 模型⚡ 歐姆定律🔘 按鈕🪲 去彈跳

1一支 GPIO 接腳就是一個可控制的開關

① 直覺 先建立整體圖像

GPIO = 通用輸入/輸出（General-Purpose Input/Output）。Uno 排針上那些編號的接腳，每一支都是一條直通矽晶片的導線，而一小段邏輯電路決定這條導線要做什麼。關鍵在於：GPIO 接腳具有方向，而在任一瞬間，它只可能是以下兩者其中之一：

  • OUTPUT（輸出）——接腳驅動。晶片把導線箝制在約 5 V（HIGH）或 0 V（LOW）。你在說話。
  • INPUT（輸入）——接腳傾聽。晶片放手不管導線，量測其他電路施加在上面的電壓。你在讀取。
💡 一句話模型

GPIO 接腳是一個由軟體控制的開關：作為 OUTPUT 時，它把導線連接到 +5 V 或接地；作為 INPUT 時，它斷開連接並安靜地感測任何傳來的電壓。

當輸出為 HIGH 時，電流從接腳流出進入你的電路——接腳提供（source）電流。當它為 LOW 時，電流從你的電路流入接腳再往下流到地——接腳吸收（sink）電流。同一支接腳，相反的電流方向。你採用哪一種，會改變 LED 的接線方式，後面就會看到。

flowchart TB REG["DDRx 暫存器位元
（方向）"] --> DIR{0 或 1？} DIR -- "1 = OUTPUT" --> DRV["驅動器把導線箝制
到 5V 或 0V"] DIR -- "0 = INPUT" --> SNS["高阻抗緩衝器
讀取導線"] DRV --> PIN(["實體接腳"]) SNS --> PIN
🔑 每支接腳背後的兩個暫存器

在 AVR 上，接腳的方向是 DDRx 暫存器（資料方向暫存器，Data Direction Register）中的一個位元，而它的輸出準位是 PORTx 中的一個位元。讀取輸入接腳則是讀取 PINx 暫存器。pinMode() 與 digitalWrite() 只是友善的包裝，替你翻動那些位元而已。

2電氣模型：歐姆定律與電流上限

① 直覺 電壓推動，電阻阻擋

在把任何東西接到接腳之前，你必須以伏特、安培、歐姆來思考。電壓（$V$，伏特）是電的「壓力」。電流（$I$，安培）是電荷流動的速率。電阻（$R$，歐姆）阻擋這種流動。三者之間的關係，是整個電子學中最重要的一條方程式：

② 數學 推導出電阻值
$$ V = I \cdot R \qquad\Longleftrightarrow\qquad I = \frac{V}{R} \qquad\Longleftrightarrow\qquad R = \frac{V}{I} $$

現在來看新手經典錯誤：把 LED 直接從接腳接到地。LED 是一顆二極體——一旦超過它的順向電壓 $V_f$，它幾乎像短路一樣，於是電流暴衝，直到某個元件燒熔。我們用一顆串聯的限流電阻來馴服它。沿著迴路套用克希荷夫電壓定律：接腳供應 $V_{cc}$，LED 壓降 $V_f$，而電阻必須承擔剩下的部分，也就是 $V_{cc}-V_f$。對電阻套用歐姆定律：

$$ R = \frac{V_{cc} - V_f}{I} $$
∑ 範例演算——計算 LED 電阻值

用一支 5 V 接腳驅動一顆紅色 LED（$V_f \approx 2\,\text{V}$），目標電流 $I = 10\,\text{mA} = 0.010\,\text{A}$：

$$ R = \frac{5\,\text{V} - 2\,\text{V}}{0.010\,\text{A}} = \frac{3\,\text{V}}{0.010\,\text{A}} = 300\ \Omega $$

套件裡沒有 300 Ω？往大取最接近的庫存值（330 Ω）——電阻愈大代表電流愈小，這是安全的方向。220 Ω 會推出 $I = 3/220 \approx 13.6\,\text{mA}$：仍然沒問題，只是更亮一點。

那顆電阻會散逸多少熱？功率等於電壓乘以電流，再結合歐姆定律，可得到你最常引用的幾種形式：

$$ P = V \cdot I = I^2 \cdot R = \frac{V^2}{R} $$

對我們這顆電阻而言：$P = I^2 R = (0.010)^2 \times 300 = 0.03\,\text{W} = 30\,\text{mW}$。一顆標準的 ¼ 瓦（250 mW）電阻對此不屑一顧——將近 10 倍的餘裕。功率計算只有在你驅動馬達或許多 LED 時才會變得有趣，但檢查它的習慣現在就要養成。

⚠️ 陷阱——Uno 接腳的電流上限是玩真的

ATmega328P 的接腳物理上大約可以提供或吸收 40 mA 的絕對最大值——超過它，你就有永久損壞接腳驅動電路的風險。安全的設計目標是每支接腳 ≤ 20 mA，而整顆晶片還有一個合計上限（所有接腳加總約 200 mA，並有各埠（port）的個別上限）。我們的 10 mA LED 安穩地落在界線之下。需要驅動耗電的東西？用電晶體或 MOSFET——絕對不要用裸接腳。

🔬 即時：LED 電阻與功率計算器

拖曳 Vcc、Vf 與目標 I。模擬器會解出 $R=(V_{cc}-V_f)/I$ 與功率 $P=I^2R$，若你的電流超過 20 mA 安全上限，便以紅色警告。

3輸出：提供、吸收與 LED 極性

③ 程式碼 逐行讀

設定一個輸出只需兩個呼叫：在 setup() 裡設定一次方向，然後用 digitalWrite() 驅動準位。以下是接在第 8 腳上的單顆外接 LED，採高態動作（active-high）接線（接腳 → 電阻 → LED → 地），讓它閃爍：

led_output.ino C++ / Arduino
const uint8_t LED_PIN = 8;          // 我們 LED 的陽極側，經一顆 330R 電阻

void setup() {
  pinMode(LED_PIN, OUTPUT);         // D8 的 DDRB 位元 := 1  → 接腳現在「驅動」
}

void loop() {
  digitalWrite(LED_PIN, HIGH);      // 接腳 = 5V → 電流從接腳「提供」流出 → LED 亮
  delay(300);
  digitalWrite(LED_PIN, LOW);       // 接腳 = 0V → LED 上無壓降 → LED 滅
  delay(300);
}

那顆 LED 是高態動作的：HIGH 讓它亮，因為電流從接腳流出（提供），穿過 LED，到地。但你也可以反過來接線。改接成 +5V → 電阻 → LED → 接腳，現在接腳必須拉到 LOW 才能點亮它——電流流入接腳（吸收）往地的方向去。這就是低態動作（active-low）輸出。

接線方式讓 LED 亮的接腳準位電流方向術語
接腳 → R → LED → GNDHIGH（5 V）流出接腳高態動作（提供電流）
+5V → R → LED → 接腳LOW（0 V）流入接腳低態動作（吸收電流）
🛠️ 為什麼有時偏好吸收電流

許多 MCU（以及一般的邏輯家族）能吸收的電流比能提供的略多一點，而且吸收方式讓 LED 維持在固定的 +5 V 軌上，而不是取決於接腳那略低於 5 V 的高準位。對單顆 LED 來說這幾乎無所謂；但對 LED 陣列與七段顯示器就有差。重點是保持一致，並清楚自己接的是哪一種——當「我的 LED 該滅的時候卻亮著」時，軟體裡邏輯接反通常就是元兇。

4輸入與浮接接腳的陷阱

① 直覺 未接的輸入接腳就是一支天線

讀取一支接腳聽起來不難：呼叫 digitalRead()，得到 HIGH 或 LOW。陷阱在於：當沒有任何東西主動驅動接腳時會發生什麼。一支裸 INPUT 接腳具有極高的阻抗——它幾乎不抽取任何電流——所以如果你讓它懸空，它並不會讀到乾淨的 0。它會浮接（float）：它撿起你手部、市電配線、附近馬達的雜散電磁雜訊，在 HIGH 與 LOW 之間隨機跳動。

⚠️ 陷阱——浮接的輸入

按鈕通常在按下時把接腳連到地，放開時則把它連到什麼都沒有。配上單純的 INPUT，「什麼都沒有」就意味著浮接，所以放開時讀到的是垃圾。解法是用一顆上/下拉電阻，溫和地把接腳綁到一個已知電壓。

② 數學 為什麼「拉」需要一顆電阻

上拉（pull-up）電阻把接腳連到 +5 V；下拉（pull-down）則把它連到地。它把閒置的接腳「拉」到一個明確的準位。為什麼不用一條普通導線？因為當按鈕確實把接腳連到相反的電軌時，上/下拉電阻與按鈕會構成一條從 5 V 到地的通路——電阻把那股短路電流限制到無害的程度：

$$ I_{\text{pull}} = \frac{V_{cc}}{R_{\text{pull}}} $$

AVR 的內部上拉大約是 $R_{\text{pull}} \approx 20\text{–}50\,\text{k}\Omega$。取中間值 30 kΩ：當按鈕被按下時它把接腳綁到地，通過上拉的電流為 $I = 5/30000 \approx 0.17\,\text{mA}$——微不足道，卻足以在按鈕開路時維持一個穩固的邏輯準位。那高電阻正是它「弱」的原因：強到足以定義閒置狀態，弱到讓按鈕能輕易壓過它。

🔑 INPUT_PULLUP——免費、內建、正確的預設

與其焊一顆外接電阻，AVR 在每一支 GPIO 裡都內建了上拉。用 pinMode(pin, INPUT_PULLUP) 啟用它。接腳閒置時為 HIGH；接到地的按鈕按下時讀到 LOW。這是在 Arduino 上讀按鈕的標準做法——不需額外零件。

5讀取按鈕：低態動作邏輯

① 直覺 按下 = LOW，這不是 bug

把一顆瞬時按鈕（momentary push-button）接在第 2 腳與地之間，啟用內部上拉，你就得到了 Arduino 的標準按鈕。以下用圖來表示這個電路：

flowchart LR VCC["+5V（內部）"] -- "約 30kΩ 上拉" --> NODE(("接腳 D2")) NODE --> MCU["digitalRead(2)"] NODE -- "按鈕（瞬時）" --> GND["GND"]

追蹤兩種狀態。放開：按鈕開路，所以只有上拉勝出，把 D2 維持在 +5 V → digitalRead 回傳 HIGH。按下：按鈕把 D2 短接到地，壓過微弱的上拉，所以 D2 停在 0 V → digitalRead 回傳 LOW。按下狀態讀到 LOW：這就是低態動作邏輯，而且這是使用上拉時正常、正確的結果。

③ 程式碼 逐行讀
button_read.ino C++ / Arduino
const uint8_t BTN_PIN = 2;          // 按鈕接在 D2 與 GND 之間
const uint8_t LED_PIN = LED_BUILTIN;

void setup() {
  pinMode(BTN_PIN, INPUT_PULLUP);   // 閒置 HIGH；按下拉到 LOW
  pinMode(LED_PIN, OUTPUT);
}

void loop() {
  // 低態動作：LOW 代表按下。反相一下，讓「按下」讀起來為 true。
  bool pressed = (digitalRead(BTN_PIN) == LOW);
  digitalWrite(LED_PIN, pressed ? HIGH : LOW);   // LED 反映按鈕狀態
}
🛠️ 讓意圖一目了然

拿結果與 LOW 比較，並把它命名為 pressed，就把低態動作的反相藏在一個可讀的字眼背後。未來的你（以及下一位工程師）永遠不該需要在腦中重新推導「等等，LOW 是按下還是放開？」——變數名稱就回答了這個問題。

6問題：接點彈跳

① 直覺 金屬拍打金屬會「彈響」

按鈕是兩片有彈性的金屬接點。當你按下時，它們不會乾淨俐落地貼合——它們猛然撞在一起、彈開、再撞、來回顫動數毫秒，才終於安定下來。在電氣上，接腳看到的不是一道乾淨的 HIGH→LOW 邊緣；它看到的是一串爆發的邊緣。典型的機械開關彈跳約 1–10 ms。以下是示波器波形，用時間順序來描述：

HIGH"] --> B["按下！
↯ 彈跳
LOW-HIGH-LOW-HIGH"] B --> C["安定
LOW（1–10ms 後）"] C --> D["放開！
↯ 再次彈跳"] D --> E["放開
HIGH"]

CPU 每秒執行 loop() 數百萬次，遠比那些彈跳快得多。所以單一次的實體按壓，在天真的程式碼看來就像好幾次按壓。如果你在切換一顆 LED，或更糟——在計數按壓次數，那麼一次按壓就會被記成三次、五次、一個隨機數字。這是初學嵌入式時最常見的「我的程式碼鬧鬼了」bug。

🏭 真實案例——說謊的計數器

有一台工業零件計數器，盯著輸送帶上的限位開關，每件物品經過時觸發一次。沒有去彈跳的情況下，接點彈跳讓它在實際只通過 9,800 件時卻計到 12,000 件——22% 的幽靈超計，搞砸了庫存盤點，也導致出貨短缺。輸入調理（conditioning）（韌體裡的去彈跳，有時是硬體的 RC 濾波器）不是錦上添花；它是「可信賴的計數」與「不可信的計數」之間的差別。每一台 PLC、每一個產品級的按鈕輸入，都會去彈跳。

7去彈跳：從阻塞式到非阻塞式

① 直覺 等到「彈響」停止

去彈跳的意思是：忽略那串顫動，回報一次乾淨的轉態。有三種逐步升級的做法。

1. 阻塞式的 delay() 取巧法。看到一道邊緣後，用 delay(20) 睡過彈跳的時間窗，然後再讀一次。簡單，而且有效——但它讓整個程式凍結 20 ms。其他什麼都動不了：其他按鈕不行、感測器不行、閃爍也不行。對單按鈕的玩具還行，在真實產品裡卻是致命的。

debounce_blocking.ino C++ / Arduino
if (digitalRead(BTN_PIN) == LOW) {  // 可能按下了
  delay(20);                        // ⛔ 讓所有東西凍結 20 ms
  if (digitalRead(BTN_PIN) == LOW)  // 還按著？那就是真的
    handlePress();
}
② 數學 挑選時間窗

該等多久？久到能撐過彈跳，短到讓人感覺即時。彈跳最長到 $t_b \approx 10\,\text{ms}$ 時，去彈跳時間窗 $t_d$ 必須滿足：

$$ t_b < t_d \ll t_{\text{human}} \qquad\Rightarrow\qquad 10\,\text{ms} < t_d \ll 100\,\text{ms} $$

人按按鈕的速度不可能快過每秒約 10 次（相隔約 100 ms），所以從 20–50 ms 之間任選一個值，都安穩地落在「比彈跳久」與「比一次真正的二度按壓短」之間。我們會用 25 ms。

③ 程式碼 非阻塞式的 millis() 去彈跳

2. 非阻塞式的 millis() 去彈跳。我們不去睡覺，而是記住最後一次彈跳的時間，只有在輸入維持穩定整個時間窗之後才接受它。millis() 回傳開機以來的毫秒數，而且永不阻塞。我們追蹤最後一次的原始讀值、最後一次穩定的讀值，以及一個時間戳記——這是典型的狀態追蹤：

debounce_millis.ino C++ / Arduino
const uint8_t BTN_PIN = 2;
const unsigned long DEBOUNCE_MS = 25;     // > 最大彈跳，< 人類反應

int  lastRaw     = HIGH;                   // 前一次的*原始*讀值
int  stableState = HIGH;                   // 最後一次*接受*（去彈跳後）的準位
unsigned long lastChange = 0;              // 原始讀值最後一次翻轉的時刻
long pressCount = 0;

void setup() {
  pinMode(BTN_PIN, INPUT_PULLUP);
  Serial.begin(9600);
}

void loop() {
  int raw = digitalRead(BTN_PIN);          // 低態動作：LOW == 按下

  if (raw != lastRaw) {                     // 原始讀值翻轉 → （重新）啟動計時器
    lastChange = millis();                  // 記下彈跳的「時刻」
    lastRaw = raw;
  }

  // 原始準位維持穩定的時間，是否已超過去彈跳時間窗？
  if ((millis() - lastChange) >= DEBOUNCE_MS) {
    if (raw != stableState) {               // 一個全新、已安定的準位
      stableState = raw;
      if (stableState == LOW) {             // 安定在 LOW = 一次真正的按下
        pressCount++;
        Serial.println(pressCount);
      }
    }
  }
  // loop() 永不阻塞——其他工作每一輪都能在這裡執行。
}
💡 計時器真正量的是什麼

每當原始接腳變化時，我們就把 lastChange 重設為「現在」。所以 millis() - lastChange 就是「接腳安靜了多久」。我們只有在那段安靜的時間超過 25 ms 之後才信任這個讀值——也就是說，當彈跳已明確停止之後。只在 HIGH→LOW 的安定邊緣上計數（而非每一道邊緣），正是保證每次按壓只計一次的關鍵。

這與你會在第 5 章遇到的非阻塞式閃爍是同一種模式：保存一個時間戳記、與 millis() 比較、永不 delay()。它是在單執行緒 MCU 上同時做許多事情的骨幹。

🛠️ 別永遠重造輪子——但要徹底理解它一次

正式產品的程式碼常會採用經過驗證的函式庫，例如 Bounce2，它正是包裝了這套狀態機。那很好——前提是你能從零自己寫出來。一個你不理解的函式庫，就是一個你修不了的 bug。

Q1 觀念檢核
你把一顆按鈕接在 D2 與 GND 之間，並使用 pinMode(2, INPUT_PULLUP)。當按鈕被按下時，digitalRead(2) 回傳什麼，又為什麼？
  • HIGH——上拉會在按下時強迫它為高
  • LOW——按鈕把接腳短接到地，壓過了微弱的上拉
  • HIGH——按下時把接腳連到了 +5 V
  • 它會隨機浮接，因為沒有東西驅動這支接腳
配上上拉，接腳閒置時為 HIGH。按下時把它連到地，而由於內部上拉很弱（約 20–50 kΩ），按鈕輕鬆勝出，把接腳拖到 0 V → LOW。這就是低態動作邏輯：按下 = LOW。這不是 bug；這是上拉按鈕被定義好的行為。
Q2 算一算
替一顆順向電壓 $V_f = 2.1\,\text{V}$、由 $V_{cc} = 5\,\text{V}$ 驅動、目標電流 $I = 15\,\text{mA}$ 的 LED 計算限流電阻。使用 $R = (V_{cc}-V_f)/I$。以歐姆為單位給出 $R$。
Ω
$R = \dfrac{5 - 2.1}{0.015} = \dfrac{2.9}{0.015} \approx 193\,\Omega$。實務上你會往大取一顆庫存的 220 Ω（電流略小，是安全的方向）。消耗的功率為 $P = I^2R = (0.015)^2 \times 193 \approx 43\,\text{mW}$——對一顆 ¼ 瓦的元件來說輕而易舉。
Q3 程式挑戰——非阻塞式切換計數器
寫一支草稿碼（sketch），對一顆按鈕（D2，INPUT_PULLUP）使用非阻塞式的 millis() 去彈跳，在每一次乾淨的按壓時切換（toggle）內建 LED，並透過序列埠（Serial）印出累計的按壓次數。任何地方都不准用 delay()。關鍵限制：無論接點彈跳得多糟，每一次實體按壓都剛好切換一次。
提示
重複使用狀態追蹤的模式：儲存 lastRaw、stableState 與 lastChange。每當原始讀值翻轉時就重設 lastChange = millis()。只有在 millis() - lastChange >= 25 且安定後的準位是一個全新的 LOW 時才動作。在那道安定為 LOW 的邊緣上，翻轉一個 ledOn 布林值並用 digitalWrite 寫出它。
顯示一個解答
toggle_counter.ino C++ / Arduino
const uint8_t BTN = 2, LED = LED_BUILTIN;
const unsigned long DEBOUNCE_MS = 25;

int  lastRaw     = HIGH;
int  stableState = HIGH;
unsigned long lastChange = 0;
bool ledOn = false;
long presses = 0;

void setup() {
  pinMode(BTN, INPUT_PULLUP);
  pinMode(LED, OUTPUT);
  Serial.begin(9600);
}

void loop() {
  int raw = digitalRead(BTN);

  if (raw != lastRaw) {              // 彈跳或真實邊緣：重新啟動計時器
    lastChange = millis();
    lastRaw = raw;
  }

  if ((millis() - lastChange) >= DEBOUNCE_MS) {
    if (raw != stableState) {        // 安定到一個新準位
      stableState = raw;
      if (stableState == LOW) {      // 安定的按下 → 只動作一次
        ledOn = !ledOn;              // 切換
        digitalWrite(LED, ledOn ? HIGH : LOW);
        presses++;
        Serial.println(presses);
      }
    }
  }
  // ...其他非阻塞式工作可以放在這裡；loop() 永不卡住。
}

訣竅是只在安定後的 HIGH→LOW 轉態上動作，而非每一道原始邊緣——這正是把一串雜訊爆發塌縮成單一次切換的關鍵。因為沒有任何東西阻塞，你可以在同一個 loop() 裡替十顆按鈕去彈跳、並讓三顆 LED 閃爍。那種可組合性，正是我們捨棄 delay() 的全部理由。
