穩健韌體：狀態機、非阻塞設計與電源管理

一段閃爍範例程式在展示時跑十分鐘沒問題。但一件產品要在牆裡、在田野中、在病患身上運行好幾年。這道鴻溝靠的不是運氣，而是架構。我們把雜亂無章的 loop() 換成刻意設計的狀態機、一個協作式排程器、防溢位的計時、記憶體紀律，以及一份能把 2000 mAh 化為數年運轉時間的電源預算。

⏱ 約 55 分鐘🧠 架構⚡ 低功耗🛡️ 看門狗🔋 電池計算

1從義大利麵式 loop() 到架構

① 直覺 先建立全貌

早期的範例程式靠堆積成長：這裡一個 delay()、那裡一個 if、一個旗標、又一個旗標、一個「就一直輪詢到按鈕放開」的巢狀 while。在工作台上它能跑。然後你加上一段感測器讀取，突然間按鈕變得遲鈍、LED 抖動，而一個你根本不知道存在的看門狗把板子重開機了。程式並沒有變難——它變得糾纏不清。每一個阻塞呼叫都會凍結其他所有東西，每一個散落的旗標都是一個隱含、未紀錄的狀態。

💡 一句話模型

穩健的韌體是一組明確的狀態，由微小的非阻塞步驟在一個你信得過的時鐘上推進。沒有任何東西在等待。沒有任何東西被隱藏。裝置可能處於的每一種「模式」都有名字。

三項紀律帶我們走完剩下的路，而本章會逐一探討：

  • 有限狀態機——為每一個模式命名，於是 loop() 就變成單一的 switch。
  • 協作式排程——用一個 millis() 時鐘、零個 delay()，讓眾多「任務」以不同的間隔執行。
  • 資源紀律——記憶體與電源都是預算。要刻意地花用，並衡量還剩多少。
⚠️ 陷阱——delay() 是昏迷，不是暫停

delay(1000) 並不會「等待」——它讓 CPU 空轉在一個忙等迴圈裡，跑十億次左右的週期，無視按鈕、序列埠位元組，以及其他每一個任務。loop() 裡任何一處的阻塞呼叫，都會決定整個程式的反應速度下限。某種意義上，整章就是一場刪除 delay() 的行動。

2有限狀態機：為每一個模式命名

① 直覺 裝置永遠恰好處於一個模式

一個紅綠燈不是綠、就是黃、或是紅——絕不會同時是兩種，也絕不會是別的什麼。這就是有限狀態機（finite state machine, FSM）：一組有限的狀態，加上當事件發生時在狀態之間移動的規則（轉移）。按鈕、選單、充電序列、通訊協定——幾乎每一種「它先做這個、再做那個，除非……」的行為本身就是一個 FSM，無論你有沒有把它建模出來。明確地將它建模，正是「你能推理的程式碼」與「你只能祈禱別出事的程式碼」之間的差別。

② 數學 剛好足夠的形式化

形式上，一個 FSM 是這個元組 $M = (S,\, \Sigma,\, \delta,\, s_0)$：一組有限的狀態 $S$、一個輸入字母表（事件）$\Sigma$、一個轉移函數 $\delta : S \times \Sigma \to S$，以及一個起始狀態 $s_0 \in S$。你永遠不會把 $\delta$ 寫成一條方程式——你會把它寫成一個 switch。但這套形式化替你買到一項保證：

$$ |S| < \infty \;\Longrightarrow\; \text{每一個 (狀態, 事件) 配對都可窮舉、可測試。} $$

如果你的裝置有 4 個狀態與 3 個不同的事件，最多就有 $|S|\times|\Sigma| = 12$ 個轉移要定義。那是一份有限的檢查清單——正是能避免「但如果按鈕是在校正進行中被按下會怎樣？」這類臭蟲的東西。

③ 程式碼 逐行細讀

標準的實作：為狀態建一個具名的 enum、一個單一的狀態變數，以及一個每次迴圈迭代執行一次的 switch。每個 case 做兩件事——對當前狀態採取行動，並決定是否要轉移。

fsm.ino C++ / Arduino
// 1) 為裝置可能處於的每一個模式命名。編譯器現在會
//    禁止任何不在這份清單上的狀態。
enum class State : uint8_t { Idle, Armed, Running, Fault };

State state = State::Idle;          // 當前狀態 — 唯一的模式變數
const uint8_t BTN = 2;              // 按鈕接到 GND，使用 INPUT_PULLUP

void setup() {
  pinMode(BTN, INPUT_PULLUP);       // 按下 == LOW
  pinMode(LED_BUILTIN, OUTPUT);
  Serial.begin(115200);
}

// 一個極小的邊緣偵測器：在按下的那一刻恰好回傳 true 一次。
bool buttonPressed() {
  static bool prev = HIGH;          // 'static' 讓值在多次呼叫間保留
  bool now = digitalRead(BTN);
  bool edge = (prev == HIGH && now == LOW);  // HIGH→LOW == 按下
  prev = now;
  return edge;
}

void loop() {
  bool press = buttonPressed();     // 每次迭代只取樣事件一次

  switch (state) {                  // 這個 switch 就是轉移函數 δ
    case State::Idle:
      digitalWrite(LED_BUILTIN, LOW);
      if (press) state = State::Armed;
      break;

    case State::Armed:
      digitalWrite(LED_BUILTIN, HIGH);          // 恆亮 = 待命
      if (press) state = State::Running;
      break;

    case State::Running:
      // ... 在這裡做真正的工作，非阻塞 ...
      if (sensorFaulted()) state = State::Fault; // 有條件的轉移
      break;

    case State::Fault:
      if (press) state = State::Idle;            // 操作員清除故障
      break;
  }
}
🔑 為什麼這牢不可破

只有一個狀態變數，而且它只在一個地方改變（switch 內部）。要回答「當 X 發生時裝置會做什麼？」你只要讀恰好一個 case。使用 enum class 意味著打錯字或超出範圍的狀態會是編譯錯誤，而不是凌晨三點的現場故障。整個迴圈是非阻塞的：它在微秒內就跑完離開，所以其他任務仍然能執行。

🏭 真實案例——去彈跳本身就是一個 FSM

那行單行的邊緣偵測器建立在一顆會雜訊的機械按鈕上：接點會彈跳 1–20 ms，觸發數十個假邊緣。產品級的去彈跳是它自己的小 FSM——狀態有穩定、彈跳中、已穩定——由一個 millis() 計時器把關，要求電位穩定維持，比方說 25 ms，才接受它。你會在練習裡建出正是這樣的東西。

3不需要 RTOS 的協作式排程器

① 直覺 多個時鐘，一個迴圈

真實裝置以不同的節奏並行多項工作：以 1 Hz 閃爍一個心跳、以 10 Hz 取樣一個感測器、每 60 s 推送一個無線封包。你不需要為此用上即時作業系統。你需要的是一張工作表格，每項工作記住它上次執行的時刻，由單一的非阻塞迴圈逐一推進。每項工作檢查一個便宜的問題——「我的間隔到了嗎？」——如果到了，就執行並重設它的計時器。這就是協作式多工：任務必須短、必須回傳；沒有人會被搶佔。

③ 程式碼 30 行內的排程器

一項任務就是 {interval, lastRun, fn}。派工器每次迴圈走過這張表，呼叫任何間隔已到的任務。

scheduler.ino C++ / Arduino
struct Task {
  uint32_t interval;     // 每隔 'interval' 毫秒執行一次
  uint32_t lastRun;      // 上次執行時的 millis()
  void (*fn)();          // 指向任務函數的指標
};

// ---- 實際的工作（每項都必須短且非阻塞）----
void heartbeat()  { digitalWrite(LED_BUILTIN, !digitalRead(LED_BUILTIN)); }
void sampleADC()  { /* 讀取一個感測器，存下數值 */ }
void sendPacket() { /* 組裝並傳送一個訊框        */ }

// ---- 排程：interval、lastRun=0、函數 ----
Task tasks[] = {
  { 500,   0, heartbeat  },   // 2 Hz LED 切換
  { 100,   0, sampleADC  },   // 10 Hz 感測器
  { 60000, 0, sendPacket },   // 每分鐘一次
};
const uint8_t NTASKS = sizeof(tasks) / sizeof(tasks[0]);

void setup() { pinMode(LED_BUILTIN, OUTPUT); }

void loop() {
  uint32_t now = millis();                  // 每一輪只取一個時間戳
  for (uint8_t i = 0; i < NTASKS; i++) {
    // 無號減法 — 為何能防溢位請見 §4
    if (now - tasks[i].lastRun >= tasks[i].interval) {
      tasks[i].lastRun = now;               // 在執行「之前」重設
      tasks[i].fn();                        // 執行任務
    }
  }
  // loop() 在微秒內回傳 → 沒有任何東西阻塞其他東西
}

🛠️ 在呼叫 fn() 之前就重設 lastRun

在叫用任務之前就設定 lastRun = now，這樣間隔是從排定的那一刻起算，而不是從任務碰巧結束的時刻起算。這讓節奏即使在某個任務跑得稍久時仍然穩定，並且能防止漂移在數百萬次迭代中一點一滴累積進來。

⚠️ 陷阱——協作式意味著你必須讓出控制權

沒有任何任務會被搶佔。如果 sampleADC() 裡藏了一個 delay(200)，其他每一個任務——包括心跳——都會停滯 200 ms。排程器的反應速度只跟你最慢的任務一樣快。讓每一個都保持簡短；如果某項工作本質上很長，就把它切成它自己的 FSM，每次呼叫只推進一步。

4millis() 溢位回繞：證明那個無號技巧

② 數學 推導它，別只是相信它

millis() 回傳一個 uint32_t——一個自開機以來計數毫秒的 32 位元無號計數器。它一直數到 $2^{32}-1$，然後回繞到 0。那何時會發生？

$$ t_{\text{wrap}} = \frac{2^{32}\ \text{ms}}{1000 \cdot 60 \cdot 60 \cdot 24} \approx \frac{4.295\times10^{9}}{86\,400\,000} \approx 49.7\ \text{天} $$

每一個基於 millis() 的裝置只要持續通電約 50 天，就會碰上這次回繞。天真的防護寫法 if (now > lastRun + interval) 在跨越回繞時會失效：lastRun + interval 溢位，或 now 重設成一個很小的數，比較就出錯了——任務變成永遠觸發或永遠不觸發。修正之道是先做減法，在無號算術之下：

$$ \texttt{if } (\,\underbrace{\texttt{now} - \texttt{lastRun}}_{\text{無號}}\, \geq \texttt{interval}\,) $$

以下是這在即使跨越回繞時也正確的證明。無號 32 位元減法的定義就是以 $2^{32}$ 為模來進行：

$$ (\texttt{now} - \texttt{lastRun}) \bmod 2^{32} \;=\; \text{真正經過的毫秒數},\quad \text{對任何經過時間} < 2^{32}\ \text{成立。} $$

具體地說，假設 lastRun = $2^{32}-10$（就在回繞之前），而 25 ms 之後 now 已經回繞到 $15$。字面上的差值 $15 - (2^{32}-10)$ 是負的——但以 $2^{32}$ 為模計算後，它變成

$$ \big(15 - (2^{32}-10)\big) \bmod 2^{32} \;=\; (25 - 2^{32}) \bmod 2^{32} \;=\; 25\ \text{ms.} $$

恰好就是真正經過的時間。回繞在模算術中自我抵消了，所以無號減法不需要任何特例處理就能回傳正確答案。這就是為什麼 §3 的排程器使用 now - tasks[i].lastRun >= interval 而從不做加法。

🔑 烙印進腦海的規則

對於計時，永遠用兩個無號 millis() 值相減，並把差值與你的間隔比較。絕不比較絕對時間戳、絕不把間隔加到某個時間戳上，並讓兩個運算元保持同一個 uint32_t 型別。做到這些，回繞就是個非事件——你的程式碼會毫無察覺地平順度過那條 49.7 天的界線。

⚠️ 陷阱——一個亂入的 int 就毀了整個證明

這個魔法依賴於無號運算。如果你把 lastRun 存成一個有號的 long 或 int，或混入一個 int 字面值，C++ 的整數提升（integer promotion）就可能把這個減法拖進有號算術，而在那裡溢位是未定義行為。從頭到尾都用 uint32_t（或 unsigned long）。

5記憶體紀律：Flash、堆疊、堆積與 F()

① 直覺 2 KB 的 SRAM 是個吃緊的預算

ATmega328P 只有 2 KB 的 SRAM。那一小片要容納三位互相往彼此方向成長的房客：全域變數/靜態變數（固定的，從底部開始）、堆積（heap，隨著你 malloc／使用 String 向上長），以及堆疊（stack，隨著每次函數呼叫與區域變數向下長）。當堆積與堆疊撞在一起時，晶片不會報錯——它會默默地破壞資料，表現得像鬧鬼一樣。

🔑 字串字面值偷偷吃掉 SRAM

因為 AVR 是哈佛架構（Harvard architecture），像 Serial.println("Booting...") 這樣的常數活在 Flash 裡，但在啟動時會被複製進 SRAM，好讓 CPU 能把它當資料來讀。十幾個除錯字串就能在 setup() 都還沒跑之前，悄悄燒掉你那 2048 位元組裡的數百個。

③ 程式碼 把字串留在 Flash；絕不讓堆積碎片化

兩個工具能修正這點。F() 巨集告訴編譯器把字串字面值留在 Flash 裡，並從那裡逐位串流出來。PROGMEM 對較大的常數表格做同樣的事。而 String 類別——也就是會在堆積上配置記憶體的那個——在長時間運行的韌體中是被禁止的。

memory.ino C++ / Arduino
#include <avr/pgmspace.h>

// 不好：這個字面值會在開機時被複製進寶貴的 SRAM。
// Serial.println("System initialized and ready");

// 好：F() 把它留在 Flash 裡；逐位元組串流出來。
void banner() {
  Serial.println(F("System initialized and ready"));  // 0 位元組 SRAM
}

// 一張完全停在 Flash 裡的常數查找表：
const uint8_t gamma8[] PROGMEM = { 0, 1, 2, 4, 6, 10, 14, 20 };

uint8_t gammaLookup(uint8_t i) {
  return pgm_read_byte(&gamma8[i]);   // 明確地從 Flash 讀取
}

// 在執行期測量空閒 SRAM：堆積頂端與當前堆疊指標之間的間隙。
// 又小、又經典、又可靠。
extern char *__brkval;                 // 堆積頂端（由 malloc 設定）
extern char  __heap_start;             // 堆積區域的起點

int freeRAM() {
  char top;                            // 'top' 的位址 ≈ 堆疊指標
  return (int)&top - (__brkval ? (int)__brkval : (int)&__heap_start);
}

void setup() {
  Serial.begin(115200);
  banner();
  Serial.print(F("Free RAM: "));
  Serial.println(freeRAM());           // 觀察這個數字是否保持「穩定」
}

void loop() { /* freeRAM() 不可隨時間呈現下降趨勢 */ }
⚠️ 陷阱——String 把堆積碎片化到死

Arduino 的 String 類別每次成長或串接時，都會在堆積上配置與釋放。在一個建立 String 的迴圈裡，堆積會被大小不一的洞填滿——碎片化（fragmentation）——直到一個理應放得下的配置卻找不到一塊連續的空隙。裝置運行好幾個小時都正常，然後當機。改用搭配 snprintf() 的固定 char 緩衝區，並看著 freeRAM() 保持水平。一個數小時內穩定不變的空閒 RAM 數字，正是無洩漏韌體的標誌。

6電源與睡眠模式：電池壽命方程式

① 直覺 反正 CPU 幾乎一直都閒著

一個每分鐘取樣一次的感測器節點，99.9% 的時間都無事可做。讓 CPU 全速跑完那段閒置時間，毫無益處地燒掉約 15 mA。睡眠模式讓晶片關掉它沒在用的區塊——先 CPU，再計時器，然後振盪器——把電流降低三到四個數量級。AVR 提供一道階梯，從淺到深：

模式（SLEEP_MODE_*）關掉了什麼喚醒來源典型電流 I（ATmega328P）
IDLECPU 時脈任何中斷約 6 mA
ADCCPU + I/O 時脈ADC 完成等約 1 mA
PWR_SAVE大多數時脈；Timer2 仍存活Timer2、WDT、INT約 1.8 µA + Timer2
PWR_DOWN除 WDT 與外部 INT 外的一切WDT、INT0/1、接腳變化約 6 µA（WDT 開啟）
② 數學 平均電流與電池壽命

裝置交替運作：一段短暫的活躍電流爆發，以 $I_{\text{active}}$ 持續 $t_{\text{active}}$，然後是一段漫長的睡眠，以 $I_{\text{sleep}}$ 持續 $t_{\text{sleep}}$，週期為 $T = t_{\text{active}} + t_{\text{sleep}}$。平均電流是依時間加權的平均值——一個工作週期（duty-cycle）平均：

$$ I_{\text{avg}} = \frac{I_{\text{active}}\,t_{\text{active}} + I_{\text{sleep}}\,t_{\text{sleep}}}{T} $$

而電池壽命由容量（以 mAh 計）除以那個平均消耗（以 mA 計）得出，單位為小時：

$$ t_{\text{life}}\,[\text{h}] = \frac{C\,[\text{mAh}]}{I_{\text{avg}}\,[\text{mA}]} $$

實例演算。取一個節點，醒來 $t_{\text{active}}=0.1\,\text{s}$ 並以 $I_{\text{active}}=20\,\text{mA}$（無線電 + ADC），然後在 PWR_DOWN 下以 $I_{\text{sleep}}=6\,\mu\text{A}=0.006\,\text{mA}$ 睡完一個 $T=60\,\text{s}$ 週期的其餘時間，所以 $t_{\text{sleep}}=59.9\,\text{s}$：

$$ I_{\text{avg}} = \frac{(20)(0.1) + (0.006)(59.9)}{60} = \frac{2.0 + 0.359}{60} \approx 0.0393\ \text{mA} $$

以一顆 $C = 2000\,\text{mAh}$ 的電池組：

$$ t_{\text{life}} = \frac{2000}{0.0393} \approx 50\,900\ \text{h} \approx 2120\ \text{天} \approx \mathbf{5.8\ 年.} $$

注意這個槓桿：活躍爆發在每週期 2.36 mA·s 中貢獻了 2.0，然而因為它如此短暫，平均值塌縮到約 39 µA。睡眠時間才是全局關鍵。同一個節點若讓 CPU 以 15 mA 全速運行，大約只能撐 5.5 天——僅僅一個架構選擇就造成約 380 倍的差距。

🔬 互動：平均電流與電池壽命探索器

拖曳滑桿：縮短 tactive 或拉長週期，看著 Iavg 塌縮，而預估電池壽命從幾天攀升到幾年。

③ 程式碼 睡眠、由看門狗喚醒、重複

這個模式：設定看門狗每約 8 s 觸發一次中斷，落入 PWR_DOWN，並讓那個 WDT 中斷喚醒 CPU。在兩次喚醒之間，晶片只消耗微安培。

sleep_node.ino C++ / Arduino
#include <avr/sleep.h>
#include <avr/wdt.h>
#include <avr/interrupt.h>

volatile bool wokeFromWDT = false;

// 把看門狗當作週期性的喚醒來源（中斷，「不是」重置）。
void setupWDT_8s() {
  cli();                              // 在動暫存器時關閉中斷
  wdt_reset();
  // 定時序列：設定 WDCE+WDE，然後在 4 個週期內設定預除器
  WDTCSR = (1 << WDCE) | (1 << WDE);
  WDTCSR = (1 << WDIE)               // 中斷模式（喚醒，不要重置）
         | (1 << WDP3) | (1 << WDP0); // 預除器 1001b → 約 8 秒逾時
  sei();
}

ISR(WDT_vect) { wokeFromWDT = true; } // 每約 8 秒觸發一次，回到 loop

void powerDownUntilWDT() {
  set_sleep_mode(SLEEP_MODE_PWR_DOWN); // 最深的實用模式
  cli();
  sleep_enable();
  sleep_bod_disable();               // 關掉低電壓偵測器 → 省約 µA
  sei();
  sleep_cpu();                       // ←── CPU 在此停住，約 6 µA
  sleep_disable();                   // WDT 後，執行從這裡恢復
}

void setup() { setupWDT_8s(); pinMode(LED_BUILTIN, OUTPUT); }

void loop() {
  // --- 活躍爆發：簡短，做真正的工作 ---
  digitalWrite(LED_BUILTIN, HIGH);
  readSensorAndMaybeTransmit();      // 約 100 ms 的 20 mA 工作
  digitalWrite(LED_BUILTIN, LOW);

  // --- 睡眠：塌縮到微安培，直到看門狗喚醒我們 ---
  powerDownUntilWDT();
}
🛠️ 榨出最後一點微安培

在 PWR_DOWN 中，主要的耗電是低電壓偵測器（brown-out detector，約 20 µA）與看門狗（約 4 µA）。sleep_bod_disable() 在睡眠這段視窗內關掉前者。也要關閉 ADC（ADCSRA &= ~(1 << ADEN)），並讓未使用的接腳浮接，免得漏電路徑悄悄把電池榨乾。

7看門狗：從卡死中自動復原

① 直覺 韌體的緊急停止裝置

就算是謹慎的韌體也可能卡住：某個感測器函式庫永遠等待一個永遠不會來的 I²C 應答（ack）而原地打轉，或者一次宇宙射線造成的位元翻轉破壞了程式計數器（program counter）。身旁沒有人時，這個裝置就只是死了。看門狗計時器（watchdog timer, WDT）是一個硬體倒數計時，只要它曾數到零，就重置整顆晶片。你的工作是「餵狗」——呼叫 wdt_reset()——比它到期還要快。健康的迴圈會持續餵狗；卡死的迴圈會停止餵狗，WDT 逾時，晶片就重開機進入一個已知良好的狀態。

→ 乾淨地重開機"])
③ 程式碼 重置模式，不是中斷模式

這裡 WDT 被設成重置模式（WDE，而非 WDIE），逾時為 8 s。注意這跟 §6 是同一個周邊，只是設定成不同的工作——復原而非喚醒。

watchdog.ino C++ / Arduino
#include <avr/wdt.h>

void setup() {
  // 我們是經由看門狗重置才走到這裡的嗎？記錄它，然後清除。
  if (MCUSR & (1 << WDRF)) { /* 記錄「WDT 從一次卡死中復原」 */ }
  MCUSR = 0;

  wdt_enable(WDTO_8S);     // 若 8 秒內沒餵狗就重置晶片
}

void loop() {
  wdt_reset();            // 餵狗 — 證明這個迴圈還活著

  doScheduledWork();      // 來自 §2–§3 的你的 FSM + 排程器

  // 若 doScheduledWork() 卡住 > 8 秒，wdt_reset() 永遠不會
  // 被執行到，WDT 觸發，MCU 重開機進入一個乾淨的狀態。
}
⚠️ 陷阱——在卡死處餵狗

絕不要在很長的內層迴圈裡到處撒 wdt_reset()，只為了「讓它開心」。那等於徹底破壞了整個重點：看門狗必須能抓到卡死。只在恰好一個地方餵牠——loop() 的頂端——這樣任何比逾時更久的停滯都保證會觸發它。

🏭 真實案例——撐一整年的鈕扣電池感測器節點

把它們串起來：一個跑在 CR2032（約 225 mAh）或一對 AA 電池（約 2000 mAh）上的土壤濕度節點。它大約每分鐘由看門狗喚醒一次，花約 100 ms 讀感測器並以約 20 mA 發射一個 LoRa 封包，然後落入 PWR_DOWN，降到個位數的微安培。套用 §6 的數學，AA 電池組可撐數年，連鈕扣電池也能超過 12 個月以上——因為這個裝置有 99.9% 的時間在睡覺。那個把它從睡眠中喚醒的同一個 WDT，也會在無線電驅動程式卡住時把它重開機。用狀態機掌管行為、用排程器掌管節奏、用睡眠掌管電源、用看門狗掌管韌性：那就是一件產品，而不是一場展示。

Q1 觀念檢核
兩個設計選擇能讓長時間運行的韌體保持健康。哪一項敘述是正確的？
  • 用 if (now > lastRun + interval) 比較絕對時間戳——把間隔加上去是跨越 millis() 回繞時的安全寫法。
  • 相減無號的 millis() 值（now - lastRun >= interval），讓模 $2^{32}$ 的算術處理那約 49.7 天的回繞，並避免會使堆積碎片化的 String 類別。
  • 優先選用 String 類別而非 char 緩衝區，因為它會自動管理記憶體且永遠不會讓堆積碎片化。
  • 回繞不可能發生，因為 millis() 回傳一個有號 long，會單純地在它的最大值飽和。
無號減法是以 $2^{32}$ 為模計算的，所以即使 now 已經回繞越過 0，now - lastRun 仍回傳真正經過的時間——回繞自我抵消（§4）。把間隔加到時間戳上會溢位並在界線處失效。而 String 會在堆積上配置／釋放，使其碎片化，直到一個本該放得下的配置在數小時後失敗（§5）；搭配 snprintf() 的固定 char 緩衝區能讓空閒 RAM 保持水平。
Q2 算算看
一個節點從 $C = 2000\,\text{mAh}$ 的電池抽取 $I_{\text{avg}} = 0.5\,\text{mA}$ 的平均電流。使用 $t_{\text{life}} = C / I_{\text{avg}}$，電池壽命是幾天？（先算出小時，再除以 24。）
天
$t_{\text{life}} = \dfrac{2000\ \text{mAh}}{0.5\ \text{mA}} = 4000\ \text{h}$，而 $\dfrac{4000}{24} \approx 166$ 天（約 5.5 個月）。透過深度睡眠（§6）把 $I_{\text{avg}}$ 降到 0.04 mA，同一顆電池組就能撐好幾年——平均電流是電池壽命唯一重要的旋鈕。
Q3 程式挑戰——一個非阻塞的去彈跳 FSM
寫一個去彈跳常式，在不使用 delay()的情況下回報一次乾淨的按鈕按下。把它建模成一個小 FSM：追蹤最後一次的原始讀值，以及它上次變化的 millis() 時間；只有在某個穩定的電位已持續穩定 DEBOUNCE_MS（比方說 25 ms）之後才接受它。在一個確認的 HIGH→LOW 邊緣上恰好回傳一次 true。它必須使用無號 millis() 減法（§4），且不可阻塞排程器。
提示
保留三個 static 變數：lastRaw（上次的接腳讀值）、lastChange（一個 uint32_t 時間戳），以及 stable（已接受的、去彈跳後的電位）。每次呼叫：讀取接腳；如果它與 lastRaw 不同，就重啟計時器（lastChange = millis()）。如果 millis() - lastChange >= DEBOUNCE_MS 且原始電位與 stable 不同，就把它確認進 stable，並且只有當新的穩定電位是 LOW 時（一次按下，因為按鈕是 INPUT_PULLUP）才回傳 true。
顯示一個解答
debounce_fsm.ino C++ / Arduino
const uint8_t  BTN = 2;          // 接到 GND，INPUT_PULLUP
const uint32_t DEBOUNCE_MS = 25; // 必須穩定維持這麼久

void setup() {
  pinMode(BTN, INPUT_PULLUP);    // 閒置 HIGH，按下 LOW
  pinMode(LED_BUILTIN, OUTPUT);
}

// 在一次乾淨（去彈跳後）的按下時回傳 true 一次。無 delay()。
bool debouncedPress() {
  static bool     lastRaw    = HIGH;   // 上次的原始讀值
  static bool     stable     = HIGH;   // 已接受的、去彈跳後的電位
  static uint32_t lastChange = 0;      // 原始值上次變化的時刻

  bool raw = digitalRead(BTN);
  if (raw != lastRaw) {                // 彈跳中 → 重啟計時器
    lastRaw    = raw;
    lastChange = millis();
  }

  // 無號減法 → 防溢位（見 §4）
  if ((uint32_t)(millis() - lastChange) >= DEBOUNCE_MS) {
    if (raw != stable) {               // 電位維持夠久：確認
      stable = raw;
      if (stable == LOW) return true;  // HIGH→LOW == 一次真正的按下
    }
  }
  return false;
}

void loop() {
  if (debouncedPress()) {
    // 每次乾淨的按下就切換 LED — 任何地方都不阻塞
    digitalWrite(LED_BUILTIN, !digitalRead(LED_BUILTIN));
  }
  // ... 你的排程器與其他任務每一輪都持續執行 ...
}

這就是 FSM 思維的縮影：stable 是狀態、原始接腳是事件，而 millis() 計時器是轉移的把關條件。因為它從不阻塞且使用無號減法，它能直接嵌入 §3 的排程器，並毫髮無傷地度過那 49.7 天的回繞。把 digitalRead 換成一個會雜訊的感測器，同樣的模式就能對任何抖動的輸入做去彈跳。
