八個章節的碎片——GPIO、ADC、PWM、計時器、Serial、EEPROM、狀態機、電源管理。現在我們把它們熔鑄成一個真正的產品：一台智慧環境監測與警報器。讀到最後，你會擁有一支完整的程式（sketch），它讀取世界、做決策、執行動作、記住資料、進入睡眠——這正是出貨韌體的完整解剖。

1這套系統：我們要打造什麼

① 直覺 先看見整台機器

總整專案（capstone）不是「把所有東西一次塞進來」——而是一個每個部分你都已經理解的小系統，被串接在一起，讓接縫清楚可見。我們的裝置監看一個房間，並在它太熱時提醒你。具體來說，它必須：

  • 從類比腳位上的 TMP36（或熱敏電阻）讀取溫度——第 3 章，ADC；
  • 從 LDR 分壓電路讀取環境光線——第 3 章；
  • 用內建 LED 顯示狀態（慢心跳＝正常，快閃＝警報）——第 1／4 章；
  • 當溫度超過門檻時，發出 PWM 蜂鳴器聲，並加上遲滯（hysteresis）讓它不會抖動鳴叫——第 4 章；
  • 讓一顆按鈕透過有限狀態機循環切換顯示／警報模式——第 8 章；
  • 以非阻塞排程取樣——熱路徑（hot path）中絕不出現 delay()——第 5／8 章；
  • 把警報門檻與執行中的最小／最大值持久化到 EEPROM——第 7 章；
  • 把讀數以 CSV 格式透過 Serial 串流輸出以供記錄——第 6 章；
  • 在兩次取樣之間睡眠，讓一顆鈕扣電池撐上好幾個月——第 8 章，電源管理。
💡 一句話模型

每一個嵌入式產品都是同一個迴圈：感測 → 決策 → 動作 → 記憶 → 等待。總整專案只是讓這五個步驟同時被看見，配上真實的零件與真實的數字。

2需求與模組化架構

① 直覺 把巨石切成模組

寫出難以維護韌體最快的方法，就是用一個巨大的 loop() 包辦一切。解法是模組化：每一項關注點都成為一個職責清楚、介面乾淨的小單元。如此一來，你就能逐一獨立測試、推理、替換它們——這是把業餘 sketch 和正式產品區隔開來、最有價值的一項習慣。

🔑 架構契約

loop() 變成一個輕薄的協調者（orchestrator）：它問排程器「該取樣了嗎？」，呼叫 sensors，把讀數交給 alarm 與 store，讓 fsm 回應按鈕，最後呼叫 power。loop() 本身不含任何業務邏輯——它只負責路由（routing）。每個模組都把自己的暫存器與魔術數字藏在一個動詞後面。

🏭 真實案例——模組為什麼值回票價

明年想把 TMP36 換成熱敏電阻？你只需要動 sensors.readTempC()——警報、FSM、EEPROM 與 Serial 的程式碼根本不知道零件變了。這種解耦正是真實韌體團隊在不必整套重寫的情況下出貨「v2 改用更便宜感測器」的方法。介面就是保險。

接線與連接——讓系統落地的腳位

模組是抽象，但每一個模組最終都會碰到某個腳位——也就是系統真正的邊界。腳位的選擇並非隨意：蜂鳴器需要一個支援 PWM 的腳位，按鈕想要一個支援中斷的腳位（Uno 上的 D2／D3），而類比感測器需要真正的類比腳位（A0–A5）。以下是這個架構對應到的接線清單。

⚠️ 陷阱——ADC 參考電壓

預設情況下，analogReference(DEFAULT) 是以 Vcc 為基準量測（USB 供電的 Uno 上約 5 V，但它會下垂）。為了感測精度，請透過 analogReference(INTERNAL) 使用晶片內部的 1.1 V 帶隙（band-gap）參考電壓——它穩定且能抑制電源雜訊。你的換算數學必須使用你所選的同一個 Vref，否則每一筆讀數都會悄悄出錯。我們會在下一節的感測器模組裡把這件事講明白。

3非阻塞排程器骨架

① 直覺 計時而不停下

在第 1 章我們以 delay() 犯了罪；在第 5 章我們用 millis() 懺悔。一個產品永遠不能阻塞，因為它必須在取樣與閃爍的同時讀取按鈕。這個模式就是一個小小的輔助函式：「自從我上次執行這個任務到現在，period 過去了嗎？」如果是，就執行它並蓋上時間戳。

③ 程式碼 逐行讀過
scheduler.ino C++ / Arduino
// 每 `period` 毫秒最多回傳一次 true，然後重新計時。
// `last` 是呼叫端自行擁有的時間戳（每個任務各一個）。
// 對 unsigned long 做減法可保證溢位安全（overflow-safe）：
// 即使 millis() 在約 49.7 天時回繞（wrap），(now - last) 仍然正確。
bool due(unsigned long &last, unsigned long period) {
  unsigned long now = millis();
  if (now - last >= period) {
    last += period;          // 剛好推進一個時間格——不會漂移（drift）
    return true;
  }
  return false;
}

// --- 用法：每個任務各自帶著自己的 `last` ---
unsigned long tSample = 0, tLog = 0, tBlink = 0;

void loop() {
  if (due(tSample, 2000)) { /* 每 2 秒讀一次感測器     */ }
  if (due(tLog,   10000)) { /* 每 10 秒輸出一列 CSV    */ }
  if (due(tBlink,  500))  { /* 切換心跳 LED          */ }
  // loop() 永不阻塞——三個任務共用 CPU
}
🛠️ 為什麼 last += period 勝過 last = now

如果你寫 last = now，就會丟失花在任務內部的那幾毫秒——數小時後，節奏會逐漸落後漂移。剛好推進 period 能讓排程保持相位鎖定（phase-locked）：一個「2 秒」的取樣會永遠平均維持 2 秒。這跟一個即時系統的 tick 是同一個概念。

4感測器模組與校正數學

② 數學 從計數值到 °C

ADC 是一個 10 位元轉換器：它把 $[0, V_{\text{ref}}]$ 範圍內的輸入電壓對應到 $[0, 1023]$ 範圍內的整數碼。要還原電壓只需要一行代數：

$$ V_{\text{out}} = \frac{\text{ADC}}{1023}\, V_{\text{ref}} $$

TMP36 線性得漂亮：它的輸出每升高 1 °C 上升 $10\,\text{mV}$，並有偏移量使得 $0\,^\circ\text{C}$ 讀作 $500\,\text{mV}$。把這條直線反解就能得到溫度：

$$ T_{^\circ C} = \frac{V_{\text{out}} - 0.5\,\text{V}}{0.01\,\text{V/}^\circ\text{C}} = \frac{V_{\text{out}} - 0.5}{0.01} $$

實作範例：當 $V_{\text{ref}} = 5\,\text{V}$ 且 ADC 讀數為 $154$ 時，我們得到 $V_{\text{out}} = \tfrac{154}{1023}\cdot 5 \approx 0.753\,\text{V}$，因此 $T = \tfrac{0.753 - 0.5}{0.01} \approx 25.3\,^\circ\text{C}$——室溫。注意 TMP36 的 $10\,\text{mV/}^\circ\text{C}$ 意味著在 5 V 下每一個 ADC 碼僅值 $\tfrac{5/1023}{0.01} \approx 0.49\,^\circ\text{C}$ 的解析度——改用 1.1 V 參考電壓，解析度會銳化到 $\approx 0.11\,^\circ\text{C}$。

∑ 替代方案——熱敏電阻（NTC，Beta 模型）

如果你改用 10 kΩ 的 NTC 熱敏電阻，它的電阻值遵循 Beta 方程式。設參考溫度 $T_0$（單位克耳文）下的參考電阻為 $R_0$，材料常數為 $B$（典型值約 3950 K）：

$$ \frac{1}{T} = \frac{1}{T_0} + \frac{1}{B}\ln\!\left(\frac{R}{R_0}\right) $$

從分壓電路讀出 $R$——若 NTC 是上臂、固定電阻 $R_{\text{fix}}$ 接地：$R = R_{\text{fix}}\,\dfrac{V_{\text{ref}} - V_{\text{out}}}{V_{\text{out}}}$。然後用 $T_{^\circ C} = T - 273.15$ 把克耳文換回 °C。

③ 程式碼 兩種轉換，同一個介面
sensors.ino C++ / Arduino
#include <math.h>

const uint8_t PIN_TEMP  = A0;
const uint8_t PIN_LIGHT = A1;

// 我們選用 analogReference(DEFAULT) → Vref = 供電電壓 ≈ 5.0 V。
// 若改呼叫 analogReference(INTERNAL)，這裡請改設 VREF = 1.1。
const float VREF     = 5.0;
const float ADC_FULL = 1023.0;

// 取多次讀數平均以壓低 ADC 雜訊（廉價的低通濾波）。
static uint16_t readAvg(uint8_t pin, uint8_t n = 8) {
  uint32_t acc = 0;
  for (uint8_t i = 0; i < n; i++) acc += analogRead(pin);
  return (uint16_t)(acc / n);
}

// --- TMP36 路徑：10 mV/°C，0 °C 時為 500 mV ---
float readTempC() {
  float v = readAvg(PIN_TEMP) * (VREF / ADC_FULL);  // 計數值 → 伏特
  return (v - 0.5) / 0.01;                           // 伏特 → °C
}

// --- 熱敏電阻路徑（Beta 模型），若你改接一顆 NTC ---
// 取消註解即可使用；回傳同樣的 float 型別，可直接替換。
// float readTempC_NTC() {
//   const float R_FIX = 10000.0, R0 = 10000.0;     // 歐姆
//   const float B = 3950.0, T0 = 298.15;           // K（25 °C）
//   float v = readAvg(PIN_TEMP) * (VREF / ADC_FULL);
//   float r = R_FIX * (VREF - v) / v;              // 分壓 → 電阻
//   float invT = 1.0/T0 + (1.0/B) * log(r / R0);   // Beta 方程式
//   return (1.0 / invT) - 273.15;                  // K → °C
// }

// 把光線當作 0–100 的「亮度」指數（LDR 越亮 → 電壓越高）。
uint8_t readLight() {
  return (uint8_t)(readAvg(PIN_LIGHT) * 100UL / 1023);
}
⚠️ 陷阱——平均不能取代穩定（settling）

AVR 只有一顆 ADC，它被多工（multiplex）切換到所有類比腳位。當你切換通道（A0 → A1）後，取樣保持電容（sample-and-hold capacitor）需要時間透過信號源阻抗充電。像 LDR 分壓這種高阻抗信號源，在第一次讀取時可能給出前一個通道的殘影。切換腳位後丟掉一次空讀 analogRead()，或讓信號源阻抗保持在 < 10 kΩ。

5模式 FSM

① 直覺 一顆按鈕，三種行為

同一顆按鈕在不同時刻必須代表不同意義——這正是有限狀態機（finite-state machine）的用途（第 8 章）。我們的裝置有三種模式；每按一次便前進到下一個並循環回繞。FSM 擁有目前模式與合法的轉換；除它以外，沒有任何東西可以改變模式。

stateDiagram-v2 [*] --> NORMAL NORMAL --> SILENT: 按下 SILENT --> SETPOINT: 按下 SETPOINT --> NORMAL: 按下 note right of NORMAL 心跳 + 警報啟用 end note note right of SILENT 警報靜音，仍持續記錄 end note note right of SETPOINT 按下使門檻 +1°C end note
③ 程式碼 不靠表格的 switch 式 FSM
fsm.ino C++ / Arduino
enum Mode : uint8_t { NORMAL, SILENT, SETPOINT, MODE_COUNT };
enum Event : uint8_t { EV_NONE, EV_PRESS };

volatile Mode mode = NORMAL;     // 「目前模式」的唯一擁有者

// 純轉換函式：給定目前模式與一個事件，回傳下一個模式。
// 沒有副作用 → 極易推理與測試。
Mode fsmNext(Mode m, Event e) {
  if (e != EV_PRESS) return m;          // 只有按下才會推進
  switch (m) {
    case NORMAL:   return SILENT;
    case SILENT:   return SETPOINT;
    case SETPOINT: return NORMAL;
    default:       return NORMAL;
  }
}

// 套用一個事件，並執行各模式的「進入時（on entry）」動作。
void fsmStep(Event e) {
  Mode next = fsmNext(mode, e);
  if (next != mode) {
    mode = next;                        // 提交（commit）這次轉換
    if (mode == SETPOINT) {
      // 進入 SETPOINT 目前還不做事；在此模式下按下會調整門檻
    }
  } else if (e == EV_PRESS && mode == SETPOINT) {
    bumpThreshold();                    // 在 SETPOINT 中，按一下使門檻 +1 °C
  }
}
🔑 把決策與動作分離

fsmNext() 是一個純函式（pure function）——相同輸入給出相同輸出，完全不碰硬體。這讓轉換邏輯可在你的筆電上做單元測試，且不可能被意外破壞。fsmStep() 則是那層薄殼，負責提交新狀態並觸發效果。把 SETPOINT 的加值動作塞進「自迴圈（self-loop）」分支是個刻意的小技巧：在那個模式裡，一次按下不是一次轉換，而是一個狀態內的動作。

⚠️ 陷阱——按鈕彈跳（bounce）

機械按鈕不會產生一個乾淨的邊緣；它會彈跳，在幾毫秒內觸發好幾次轉換。要做去彈跳（debounce）：在 D2 中斷裡記下 millis()，並忽略距上次邊緣 ~30 ms 內的任何邊緣。沒有去彈跳，一次實體按下會跳兩三個模式，使用者會以為韌體鬧鬼。我們會在 §9 把這部分接上。

6警報：PWM、門檻與遲滯

① 直覺 為什麼單一門檻會抖動

天真的做法：「如果 $T > T_{\text{set}}$，警報開；否則關。」但真實的溫度會因 ADC 雜訊而抖動個零點幾度。剛好停在門檻附近時，警報就會每秒開關好幾次——惱人，而且一看就是外行的標誌。解法是遲滯（hysteresis）：兩個門檻，中間夾一個死區（deadband）。

② 數學 兩個門檻，一個死區

選一個設定點 $T_{\text{set}}$ 與死區寬度 $\Delta$。當溫度上升越過上緣時警報開啟，且只有當它跌回下緣以下時才關閉：

$$ T_{\text{hi}} = T_{\text{set}} + \tfrac{\Delta}{2}, \qquad T_{\text{lo}} = T_{\text{set}} - \tfrac{\Delta}{2} $$

於是警報狀態 $a$ 是溫度與它自身前一個值的閂鎖（latched）函式：

$$ a_{n} = \begin{cases} 1 & T \ge T_{\text{hi}} \\ 0 & T \le T_{\text{lo}} \\ a_{n-1} & T_{\text{lo}} < T < T_{\text{hi}} \end{cases} $$

在死區內，警報保持它的上一個狀態——正是那份記憶消滅了抖動。具體而言，當 $T_{\text{set}} = 30\,^\circ\text{C}$ 且 $\Delta = 2\,^\circ\text{C}$ 時：一旦 $T$ 達到 $31\,^\circ\text{C}$ 它就尖叫，且直到 $T$ 跌到 $29\,^\circ\text{C}$ 才肯安靜。幾分之一度的雜訊永遠無法翻動它。

③ 程式碼 遲滯 ＋ 真正的 PWM 音調
alarm.ino C++ / Arduino
const uint8_t PIN_BUZZER = 9;     // Timer1 OC1A——支援 PWM/tone
const float   DEADBAND   = 2.0;   // °C 總寬度（Δ）

float   gThresholdC = 30.0;       // T_set，自 EEPROM 載入
bool    gAlarmOn    = false;      // a_{n-1}——閂鎖狀態

// 若警報（此刻）作用中則回傳 true。純遲滯邏輯。
bool updateAlarm(float tempC) {
  float tHi = gThresholdC + DEADBAND / 2.0;   // T_hi
  float tLo = gThresholdC - DEADBAND / 2.0;   // T_lo

  if (tempC >= tHi)       gAlarmOn = true;    // 向上越界 → 閂鎖為開
  else if (tempC <= tLo)  gAlarmOn = false;   // 向下越界 → 閂鎖為關
  // 否則：在死區內 → gAlarmOn 維持先前值

  return gAlarmOn;
}

// 驅動壓電蜂鳴器。tone() 會設定 Timer1，以給定頻率在 PIN_BUZZER
// 上發出方波，且不阻塞 loop()。
void driveBuzzer(bool on, bool silenced) {
  if (on && !silenced) tone(PIN_BUZZER, 2000);  // 2 kHz——壓電的響亮頻段
  else                 noTone(PIN_BUZZER);       // 釋放該計時器
}
🛠️ 為什麼是 2 kHz，又為什麼用 tone()

壓電蜂鳴器有機械共振，通常落在 2–4 kHz，在那裡同樣的驅動下會大幅變響——挑 2 kHz 等於免費換來音量。tone() 會把 Timer1 設成 CTC 模式，由硬體切換腳位，因此方波在你的 loop() 做其他事的同時持續播放。這就是「把 PWM 當成輸出周邊」的用法，正是第 4 章的概念。呼叫 tone() 同時也會停用腳位 9 與 10 上的 analogWrite()——它們共用 Timer1。

7持久化與睡眠

① 直覺 跨越斷電而記住；取樣之間打盹

還剩兩項非揮發性的工作。第一，持久化：使用者的門檻與執行中的最小／最大值應該在重置後仍存活，所以我們把它們保存在 EEPROM（第 7 章）。第二，電源：在每 2 秒的取樣之間，CPU 無事可做——讓它睡眠能把電流從約 15 mA 降到微安培（第 8 章），把一顆鈕扣電池從幾天變成幾個月。

③ 程式碼 結構進、結構出的 EEPROM
store.ino C++ / Arduino
#include <EEPROM.h>

struct Config {
  uint16_t magic;        // 哨兵值：EEPROM 是否為「我們」初始化過？
  float    thresholdC;   // 使用者設定點
  float    minC, maxC;   // 執行中的極值
};

const int      CFG_ADDR  = 0;
const uint16_t CFG_MAGIC = 0xE5A1;   // 任意的「這是我們的格式」標記

// 載入設定；若是有史以來第一次開機，則寫入合理的預設值。
void loadCfg(Config &c) {
  EEPROM.get(CFG_ADDR, c);           // 整批讀出結構
  if (c.magic != CFG_MAGIC) {        // 空白/外來的 EEPROM → 初始化它
    c.magic      = CFG_MAGIC;
    c.thresholdC = 30.0;
    c.minC       =  999.0;           // 哨兵值，讓第一次取樣必定勝出
    c.maxC       = -999.0;
    EEPROM.put(CFG_ADDR, c);
  }
}

// 只有在改變時才寫入——EEPROM 每個 cell 約可承受 10 萬次寫入，
// 所以我們絕不每個 loop 都寫，只在真正有編輯時寫。
void saveCfg(const Config &c) { EEPROM.put(CFG_ADDR, c); }
⚠️ 陷阱——EEPROM 磨損

一個 AVR EEPROM cell 大約能承受 $10^5$ 次抹除/寫入週期。若你每 2 秒取樣就寫一次最小／最大值，你會在 $\tfrac{10^5 \times 2\,\text{s}}{86400} \approx 2.3$ 天內耗盡一個 cell。只在某個值真正改變時（出現新紀錄，或使用者調動門檻）才呼叫 saveCfg()，而且 EEPROM.put() 本身就會跳過未變動的位元組。

③ 程式碼 用看門狗計時器（watchdog timer）睡眠
power.ino C++ / Arduino
#include <avr/sleep.h>
#include <avr/wdt.h>
#include <avr/power.h>

volatile bool wdtFired = false;

// 看門狗 ISR 只負責喚醒我們；真正的工作回到 loop() 裡做。
ISR(WDT_vect) { wdtFired = true; }

// 以最低功耗模式睡眠約 2 秒，由看門狗計時器喚醒。
void napPwrDown() {
  // 把 WDT 設成每約 2 秒觸發中斷（不是重置）。
  // 2 秒的預分頻 = WDP[3:0] = 0b0111。
  cli();
  wdt_reset();
  WDTCSR |= (1 << WDCE) | (1 << WDE);          // 解鎖以便變更
  WDTCSR  = (1 << WDIE) | (1 << WDP2) | (1 << WDP1) | (1 << WDP0); // 2 秒，中斷模式
  sei();

  set_sleep_mode(SLEEP_MODE_PWR_DOWN);         // 最深的睡眠
  sleep_enable();
  power_adc_disable();                          // 關閉 ADC 省下約 mA
  sleep_mode();                                 // <-- CPU 在此停住
  // ...由 WDT 或按鈕 INT 在此喚醒...
  sleep_disable();
  power_adc_enable();
}
🏭 真實案例——看門狗同時也是你的安全網

我們用看門狗來從睡眠中喚醒，但在生產環境裡，它會以第二種方式名副其實：把它設成重置模式（設定 WDE），並在每個 loop 的某個已知良好的位置呼叫 wdt_reset()。萬一你的韌體當掉了——譬如某個感測器的 I²C 匯流排鎖死——看門狗就會逾時並自動重啟裝置。無人看管的現場硬體就是靠這個。同一個周邊，兩個救命的工作。

8完整整合的 sketch 與除錯

③ 程式碼 全部串在一起

以下這個協調者把全部六個模組綁進一支可出貨的 sketch。先讀 loop()——它純粹是路由，正是 §2 那份契約。上面的各模組函式可直接嵌入；這個檔案展示的是黏合劑、帶去彈跳的按鈕 ISR、Serial CSV 串流，以及統計／持久化邏輯。

env_monitor.ino C++ / Arduino
// ================= 智慧環境監測與警報器 =================
// 整合：ADC 感測器、非阻塞排程器、遲滯警報（PWM tone）、
// 按鈕 FSM、EEPROM 持久化、Serial CSV、WDT 睡眠。
// 腳位：A0 溫度、A1 光線、D2 按鈕、D9 蜂鳴器、D13 狀態 LED。

#include <EEPROM.h>
#include <avr/sleep.h>
#include <avr/wdt.h>

// ---- 腳位 ----
const uint8_t PIN_TEMP = A0, PIN_LIGHT = A1;
const uint8_t PIN_BTN  = 2,  PIN_BUZZER = 9, PIN_LED = LED_BUILTIN;

// ---- 可調參數 ----
const float    VREF = 5.0, ADC_FULL = 1023.0;
const float    DEADBAND = 2.0;
const uint16_t CFG_MAGIC = 0xE5A1;
const int      CFG_ADDR  = 0;

// ---- 持久化設定 ----
struct Config { uint16_t magic; float thresholdC, minC, maxC; } cfg;

// ---- FSM ----
enum Mode : uint8_t { NORMAL, SILENT, SETPOINT };
volatile Mode mode = NORMAL;

// ---- 警報閂鎖 ----
bool gAlarmOn = false;

// ---- 按鈕（來自 ISR 的去彈跳邊緣）----
volatile bool          btnEvent = false;
volatile unsigned long btnLast  = 0;

void onButton() {                       // INT0 下降緣 ISR
  unsigned long now = millis();         // millis() 在 ISR 中可用（以計時器為基礎）
  if (now - btnLast > 30) {             // 30 ms 去彈跳視窗
    btnLast  = now;
    btnEvent = true;                    // 把事件交給 loop()
  }
}

// ---- 排程器 ----
unsigned long tSample = 0, tBlink = 0;
bool due(unsigned long &last, unsigned long period) {
  unsigned long now = millis();
  if (now - last >= period) { last += period; return true; }
  return false;
}

// ---- 感測器 ----
uint16_t readAvg(uint8_t pin, uint8_t n = 8) {
  uint32_t acc = 0;
  for (uint8_t i = 0; i < n; i++) acc += analogRead(pin);
  return acc / n;
}
float readTempC() {
  float v = readAvg(PIN_TEMP) * (VREF / ADC_FULL);
  return (v - 0.5) / 0.01;              // TMP36：10 mV/°C，0 °C 時 0.5 V
}
uint8_t readLight() { return readAvg(PIN_LIGHT) * 100UL / 1023; }

// ---- 持久化 ----
void loadCfg() {
  EEPROM.get(CFG_ADDR, cfg);
  if (cfg.magic != CFG_MAGIC) {
    cfg = { CFG_MAGIC, 30.0, 999.0, -999.0 };
    EEPROM.put(CFG_ADDR, cfg);
  }
}
void saveCfg() { EEPROM.put(CFG_ADDR, cfg); }

// ---- 警報遲滯 ----
bool updateAlarm(float tempC) {
  float tHi = cfg.thresholdC + DEADBAND / 2.0;
  float tLo = cfg.thresholdC - DEADBAND / 2.0;
  if      (tempC >= tHi) gAlarmOn = true;
  else if (tempC <= tLo) gAlarmOn = false;
  return gAlarmOn;                      // 否則：維持先前值（死區）
}

// ---- FSM ----
void fsmStep() {
  switch (mode) {
    case NORMAL:   mode = SILENT;   break;
    case SILENT:   mode = SETPOINT; break;
    case SETPOINT:                       // 在這裡按下會調整設定點...
      // 從 SETPOINT 的第一次按下會回繞到 NORMAL；於下方處理
      mode = NORMAL; break;
  }
}

void setup() {
  Serial.begin(115200);
  pinMode(PIN_LED, OUTPUT);
  pinMode(PIN_BTN, INPUT_PULLUP);
  attachInterrupt(digitalPinToInterrupt(PIN_BTN), onButton, FALLING);
  loadCfg();
  Serial.println(F("# millis,tempC,light,thresh,alarm,mode,min,max"));
}

void loop() {
  // --- 1. 處理按鈕事件（FSM）---
  if (btnEvent) {
    btnEvent = false;
    if (mode == SETPOINT) {              // 在 SETPOINT，按一下使門檻加值
      cfg.thresholdC += 1.0;
      if (cfg.thresholdC > 45.0) cfg.thresholdC = 15.0;  // 範圍回繞
      saveCfg();                          // 持久化使用者的選擇
    } else {
      fsmStep();                          // 否則推進模式
    }
  }

  // --- 2. 取樣 + 決策 + 記憶（每 2 秒）---
  if (due(tSample, 2000)) {
    float t = readTempC();
    uint8_t lx = readLight();
    bool ringing = updateAlarm(t);

    // 只在真正出現新紀錄時更新持久化極值（節省 EEPROM 磨損）
    bool rec = false;
    if (t < cfg.minC) { cfg.minC = t; rec = true; }
    if (t > cfg.maxC) { cfg.maxC = t; rec = true; }
    if (rec) saveCfg();

    // 動作：蜂鳴器，除非在 SILENT 模式被靜音
    bool muted = (mode == SILENT);
    if (ringing && !muted) tone(PIN_BUZZER, 2000);
    else                  noTone(PIN_BUZZER);

    // 透過 Serial 串流一列 CSV（第 6 章）
    Serial.print(millis());   Serial.print(',');
    Serial.print(t, 2);       Serial.print(',');
    Serial.print(lx);         Serial.print(',');
    Serial.print(cfg.thresholdC, 1); Serial.print(',');
    Serial.print(ringing);    Serial.print(',');
    Serial.print((uint8_t)mode); Serial.print(',');
    Serial.print(cfg.minC, 1); Serial.print(',');
    Serial.println(cfg.maxC, 1);
  }

  // --- 3. 心跳：正常時慢、警報時快 ---
  unsigned long beat = gAlarmOn ? 120 : 500;
  if (due(tBlink, beat)) digitalWrite(PIN_LED, !digitalRead(PIN_LED));

  // --- 4.（選用）閒置時在此睡眠以延長電池壽命 ---
  // 若沒有蜂鳴器作用中且 serial 已清空，napPwrDown() 可在取樣間
  // 把我們降到 µA；上面的熱迴圈省略了它，這樣在 USB 供電的展示
  // 工作台上心跳才會保持流暢。
}
💡 注意 loop() 沒有包含什麼

沒有 delay()。沒有戳弄 ADC 暫存器。沒有 EEPROM 位址。沒有遲滯的算術。每一個硬細節都活在一個具名的動詞後面，而 loop() 讀起來幾乎就像 §1 那句話：感測 → 決策 → 動作 → 記憶 → 等待。那份可讀性正是架構回報你的成果。

④ 真實案例 專家如何為整合系統除錯
🏭 真實案例——先隔離，再追蹤

當一個多模組系統行為異常時，絕不要一次除錯全部。先隔離：把警報與睡眠註解掉，確認單靠感測器 + Serial 能產出合理的 CSV。然後一次重新啟用一個模組。再追蹤：你的 Serial CSV 串流就是你的邏輯分析儀——把它導進一張圖，你就能親眼看著溫度曲線越過 $T_{\text{hi}}$、而 alarm 那一欄翻成 1。大多數「不可能發生」的 bug（一個差一格的遲滯邊緣、一個沒能持久化的門檻），在你把日誌畫成圖的那一刻就會變得顯而易見。printf-over-Serial 至今仍是地球上最常被使用的嵌入式除錯器，原因正在於此。

🛠️ 延伸點子（當這太簡單時）

加一個滾動平均（rolling-average）濾波器來平滑溫度。加一個 FAULT 狀態，當感測器讀到不可能的值（開路／短路 → ADC 被釘在 0 或 1023）時閂鎖住。驅動一片 I²C OLED 做真正的顯示。加一顆濕度感測器並算出露點。把資料連同時間戳記錄到 SD 卡。以上每一項都能嵌進某個既有模組而不擾動其他模組——向自己證明這套架構吧。

9⑤ 換你上場

⑤ 練習 由你決定並動手做

三道配得上總整專案的考題。一道架構抉擇、一道遲滯預算、一個你自己加上去的功能。進度會儲存在這台裝置上。

Q1 架構決策
你的隊友提議「為了永不丟資料」，在每一次 2 秒取樣時都把 min／max 存到 EEPROM。門檻很少改變；最小／最大值偶爾改變。最強的反對理由是什麼？
  • EEPROM 寫入太慢，會讓 loop() 阻塞好幾秒
  • EEPROM cell 約能承受 10 萬次寫入；每 2 秒寫一次會在幾天內把它磨壞
  • EEPROM 是揮發性的，所以資料反正會在重置時遺失
  • Config 結構太大，塞不進 1 KB 的 EEPROM
一個 AVR EEPROM cell 額定約 10⁵ 次寫入。以每 2 秒一次計，那是 $\tfrac{10^5\times2}{86400}\approx2.3$ 天就會失效。解法——只在出現真正的新紀錄或使用者編輯時才寫——正是這支 sketch 所做的。（EEPROM 是非揮發性的，而單次 EEPROM.put() 寫約 14 位元組只花幾毫秒，所以其他選項在事實上都是錯的。）
Q2 算給我看——遲滯邊緣
警報使用設定點 $T_{\text{set}} = 30\,^\circ\text{C}$ 與死區 $\Delta = 2\,^\circ\text{C}$，所以它在 $T_{\text{hi}} = T_{\text{set}} + \Delta/2$ 時開啟。一旦在鳴叫，溫度（°C）降到多少時它會關閉？（記得 $T_{\text{lo}} = T_{\text{set}} - \Delta/2$。）
°C
$T_{\text{lo}} = 30 - \tfrac{2}{2} = 29\,^\circ\text{C}$。它在 $T_{\text{hi}}=31\,^\circ\text{C}$ 閂鎖為開，並一直維持到溫度一路跌到 $29\,^\circ\text{C}$。開與關之間的 $2\,^\circ\text{C}$ 間隙就是消滅抖動的死區——任何比它小的雜訊都永遠無法翻動警報。
Q3 程式挑戰——滾動平均平滑化
ADC 雜訊讓原始溫度抖動。加上一個對最近 N = 4 筆取樣的滾動平均（rolling average），讓警報對趨勢反應，而不是單一取樣的尖峰。在 loop() 中用一個平滑後的值取代 readTempC() 的輸出，使用固定的環形緩衝區（circular buffer，不用動態記憶體）。讓每次取樣維持 O(1)。
提示
保留一個靜態陣列 buf[4] 和一個執行中的 sum。每來一筆新取樣：先減掉你即將覆蓋的那一格，寫入新值，把它加進 sum，再把索引對 4 取模前進。平均值就是 sum / 4。完全不需要遍歷緩衝區——那就是 O(1) 的訣竅。
顯示一種解法
smooth.ino C++ / Arduino
// 對最近 4 筆溫度取樣做 O(1) 的滾動平均。
float smoothTempC(float sample) {
  const uint8_t N = 4;
  static float    buf[N] = {0};      // 環形緩衝區（static = 跨呼叫保留）
  static float    sum    = 0;
  static uint8_t  idx    = 0;
  static bool     primed = false;

  sum -= buf[idx];                   // 移除即將被覆蓋的值
  buf[idx] = sample;                 // 存入新取樣
  sum += sample;                     // 把它加進執行中的總和
  idx = (idx + 1) % N;               // 環狀前進
  if (idx == 0) primed = true;       // 緩衝區已繞過一圈

  // 在緩衝區填滿之前，依目前已有的筆數來除
  return primed ? (sum / N) : (sum / (idx == 0 ? N : idx));
}

// 在 loop() 中，把原始讀數送過濾波器：
//   float t = smoothTempC(readTempC());
//   bool ringing = updateAlarm(t);

這些 static 區域變數就是全部的訣竅：它們在多次呼叫之間保留值，又不污染全域命名空間。減掉離開的取樣、加上進來的取樣，使工作量不論 N 多大都保持常數——這是一個貨真價實的移動平均（moving-average）濾波器，與 DSP 和感測器融合（sensor fusion）所用的是同一個基本元件。把 N 調大就能以反應速度換取更平滑的輸出。
