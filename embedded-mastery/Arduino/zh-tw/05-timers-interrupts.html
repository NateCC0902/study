計時器與中斷：不靠 delay() 的即時運作

在第 1 章我們犯了個錯：每一次 delay() 都會凍結整顆 CPU，讓它對外界又盲又聾，直到整段等待結束。專業的韌體絕不會這樣阻塞。本章我們學習三條出路——millis() 慣用法、硬體計時器，以及中斷——再加上那個讓它們挺過 49 天溢位（rollover）的無號數運算技巧。

⏱ 約 50 分鐘🧠 非阻塞⏲ AVR 計時器⚡ 中斷服務常式 (ISR)

1為什麼 delay() 是個陷阱

① 直覺 先建立整體圖像

把你的韌體想像成一位沒有助手的單一工人。delay(500) 告訴這位工人：「完全靜止半秒鐘。」在這 500 毫秒裡，CPU 並不是以有用的方式閒置——它其實是卡在一個緊湊的計數迴圈裡，執行著數百萬條什麼都不做的指令，無法做任何其他事。它無法讀取按鈕，無法服務感測器，無法以不同頻率閃爍第二顆 LED。整台機器都成了這一次等待的人質。

💡 一句話模型

delay() 是阻塞式（blocking）的：它讓你整個程式的時間都停擺。專業韌體則是非阻塞式（non-blocking）的——它從不靜止不動；它不斷地問「時候到了嗎？」，並在空檔處理其他所有事情。

一旦你認得這個症狀，它就再明顯不過。你寫了 Blink，然後想加一顆能切換 LED 速度的按鈕——結果按鈕像沒反應一樣，只有在你剛好趁 loop() 不在 delay() 裡的那短暫瞬間按下去時才會有作用。解法不是換一顆更快的 CPU，而是一種根本不同的控制結構：停止等待，開始輪詢時鐘（polling the clock）。

⚠️ 陷阱——「多撒幾個 delay 就好了」

初學者試著用猜測的延遲值來交錯處理多個任務（這裡 delay(7)、那裡 delay(13)）。這永遠拼不起來：每一次阻塞式等待都會替其他每個任務增加延遲，而且只要你再加一件事，時序就會漂移。阻塞式程式無法擴展到超過一個工作。

2millis() 慣用法：協作式非阻塞

① 直覺 看一眼時鐘，而不是盯著它

與其睡到下一個事件發生，我們記住上一個事件何時發生，然後在每一次經過 loop() 時，去問那個一直在跑的時鐘是否已經過了足夠的時間。millis() 以 unsigned long（一個 32 位元的 uint32_t）回傳開機以來的毫秒數，並在背景裡永遠地往上計數。我們從不等它——只是瞄它一眼。

③ 程式碼 每一行都要讀
blink_nonblocking.ino C++ / Arduino
const uint8_t LED = LED_BUILTIN;
const uint32_t interval = 500;      // 每次切換之間的毫秒數
uint32_t last = 0;                  // 上一次切換的時刻
bool ledOn = false;

void setup() {
  pinMode(LED, OUTPUT);
}

void loop() {
  uint32_t now = millis();          // 瞄一眼時鐘——絕不阻塞
  if (now - last >= interval) {      // 已經過了 500 ms 了嗎？
    last = now;                      // 記住「這個」時刻供下次使用
    ledOn = !ledOn;                  // 翻轉狀態
    digitalWrite(LED, ledOn);
  }
  // ...loop() 裡其他所有事情每一輪都持續執行：
  // 讀按鈕、輪詢感測器、閃第二顆 LED 等等。
}
🔑 把這個模式背起來

在你採取動作的那一刻就把 last = now 存起來——不要有條件地寫成 last += interval，也絕不要再呼叫一次 last = millis()（那會重新讀取時鐘而造成漂移）。每一輪讀一次 now、做一次比較、做一次更新。這個三行骨架就能取代你日後想用的每一個 delay()。

② 數學 為何溢位咬不到你

這就是幾乎讓所有人栽跟頭的部分。millis() 是一個 32 位元的無號計數器。它一路往上數到最大值，然後繞回零（wrap back to zero）：

$$ t_{\text{rollover}} = \frac{2^{32}\ \text{ms}}{1000\cdot 60\cdot 60\cdot 24} \approx 49.71\ \text{days} $$

天真的擔憂是：「當 now 繞回到一個很小的數，而 last 還是個很大的數時，now - last 不就會變成一個巨大的負值，把我的計時器搞壞嗎？」在有號（signed）運算裡，是的——災難。但 now 和 last 都是 uint32_t，而無號減法的定義就是以 $2^{32}$ 取模（modulo）後繞回。硬體計算出的是：

$$ (\,\text{now} - \text{last}\,) \bmod 2^{32} \;=\; \text{true elapsed time} $$

具體來說，假設 $\text{last} = 2^{32}-100 = 4{,}294{,}967{,}196$（繞回前 100 ms），而 250 ms 之後 now 已繞回到 $150$。有號的觀點給出 $150 - 4{,}294{,}967{,}196 = -4{,}294{,}967{,}046$——一派胡言。無號的觀點則給出：

$$ (150 - 4{,}294{,}967{,}196)\bmod 2^{32} = 150 + 100 = 250\ \text{ms} \;\checkmark $$

正好是真實經過的時間。now 的繞回，與減法本身內建的繞回，完美地相互抵消。但這只有在你讓兩個運算元都保持無號、並拿差值去和間隔比較時才成立。絕不要比較 now >= last + interval——那個加法本身就可能溢位，把臭蟲又招回來。

⚠️ 陷阱——錯誤的比較方式

if (millis() >= target)（其中 target = last + interval）是經典的壞寫法：一旦 target 在接近 49 天的關卡溢位，它就變成一個很小的數，於是條件會立刻並永遠地觸發。永遠寫成差值形式：if (now - last >= interval)。它在結構上就是防溢位的。

🔬 即時：無號溢位，以及為何減法能挺過去

把 now 拖過繞回線。看著有號差值瘋狂地變成負值，而無號差值卻一直等於真實經過的時間。

3從一個計時器到一個協作式排程器

① 直覺 多任務，單一迴圈

一旦每個任務都擁有自己的 last 和 interval，它們就能輕鬆共存：loop() 快速旋轉，而每個任務在自己的時鐘說可以時各自獨立觸發。這就是一個協作式排程器（cooperative scheduler）——沒有作業系統，只是一把共享同一個快速迴圈的 if (now - last >= interval) 區塊。

🛠️ 讓 loop() 保持飢餓

協作式排程器只有在 loop() 跑得夠頻繁——理想上每秒數千次——的情況下才能運作。這代表任何地方都不能阻塞：沒有 delay()、沒有忙等待（busy-wait）、沒有列印巨大字串的緩慢 Serial.print。迴圈越快，你的排程解析度就越緊密。一個漏掉的 delay(1000) 就會一次癱瘓所有任務。

4硬體計時器：不佔用 CPU 的精準度

① 直覺 一個會自己跑的計數器

millis() 慣用法很棒，但它的解析度受限於 loop() 旋轉的速度，而一個臃腫的迴圈會讓它抖動（jittery）。對於硬性（hard）時序——以精確的 8 kHz 取樣某個感測器、產生音調、驅動步進馬達——我們把這份工作交給 AVR 內建的專用計時器／計數器（timer/counter）周邊。它們在硬體中計數時鐘脈衝，完全獨立於 CPU 正在做什麼，並在精確的時刻引發中斷。

ATmega328P 有三個這樣的計時器。每一個其實就是一個暫存器 TCNTn，它在每一個脈衝時遞增並繞回：

⚠️ 陷阱——Timer0 已名花有主

重新設定 Timer0 會在你整個草稿碼（sketch）裡破壞 millis()、micros() 和 delay()。要做自訂的週期性中斷時，請伸手去拿 Timer1（16 位元，餘裕最多），別去碰 Timer0。

① 直覺 預除器——一個時鐘分頻器

在 16 MHz 下，一個 8 位元計時器每 $256 / 16{\times}10^6 \approx 16\ \mu\text{s}$ 就溢位一次——對人類尺度的事件來說實在太快了。預除器（prescaler）在系統時鐘抵達計數器之前先把它分頻：當預除器為 256 時，計時器每 256 個系統週期才跳動一次。AVR 提供一份固定的除數選單——1, 8, 64, 256, 1024——由 TCCRnB 暫存器裡的 CSn2:0 時鐘選擇位元來挑選。

① 直覺 CTC 模式——比對符合時清除計時器

取得精確週期的訣竅是 CTC 模式（Clear Timer on Compare Match，比對符合時清除計時器）。你把一個目標值載入比對暫存器 OCRnA。計時器從 $0,1,2,\dots$ 一路數到 OCRnA，引發一個中斷，然後自動歸零並重新開始。沒有軟體重置，沒有漂移——一個矽晶片裡的完美節拍器。

5計時器數學：為任意頻率挑選 OCRnA

② 數學 把它推導出來

在 CTC 模式中，計時器從 0 數到 OCRnA 然後重置——這是每個週期 $(\text{OCRnA}+1)$ 個跳動（這個 $+1$ 是因為它有一個跳動是花在 0 這一格）。每個跳動持續一個經過預除的時鐘週期。所以中斷頻率為：

$$ f_{\text{int}} = \frac{f_{\text{clk}}}{N \,\bigl(1 + \text{OCRnA}\bigr)} $$

其中 $f_{\text{clk}} = 16\ \text{MHz}$，而 $N$ 是預除器（1、8、64、256、1024 之一）。把它反過來解出你必須載入的暫存器值：

$$ \text{OCRnA} = \frac{f_{\text{clk}}}{N\, f_{\text{int}}} - 1 $$

實作範例。我們想要在 Timer1 上得到一個乾淨的 $1\ \text{Hz}$ 中斷——每秒一次跳動——使用預除器 $N = 256$：

$$ \text{OCR1A} = \frac{16\times10^6}{256 \times 1} - 1 = 62500 - 1 = 62499 $$

檢驗一下結果：$62499 \le 65535$，所以它放得進 16 位元的 OCR1A 暫存器。（在 8 位元的 Timer0 或 Timer2 上就放不下了——$62499 \gg 255$——這正是我們選 16 位元 Timer1 的原因。）把它代回正向公式，你會精準地還原出 $16{\times}10^6 / (256 \times 62500) = 1\ \text{Hz}$。挑選預除器是門藝術：選那個能讓所得 OCRnA 仍放得進暫存器的最小 $N$，因為較小的 $N$ 帶來較細的時序解析度。

∑ 解析度檢驗

在 16 MHz 下使用 $N = 256$，每個跳動是 $256/16{\times}10^6 = 16\ \mu\text{s}$。所以 OCR1A 是以 16 µs 為單位量化的——你可以精準打中 1 Hz、2 Hz、4 Hz，但一個古怪的頻率會被湊到最接近的 16 µs。需要更細？降到 $N = 64$（4 µs 為單位）再重新求解——只要新的 OCR1A 仍然放得進 65535。

③ 程式碼 親手設定 Timer1
timer1_ctc.ino C++ / Arduino
const uint8_t LED = LED_BUILTIN;

void setup() {
  pinMode(LED, OUTPUT);

  noInterrupts();           // 設定期間先關閉中斷
  TCCR1A = 0;               // 清除控制暫存器 A（無 PWM，一般埠操作）
  TCCR1B = 0;               // 清除控制暫存器 B
  TCNT1  = 0;               // 把計數器重置為 0

  OCR1A = 62499;            // 比對目標：16e6/(256*1) - 1  → 1 Hz
  TCCR1B |= (1 << WGM12);   // WGM12 = 1 選擇 CTC 模式（頂值 = OCR1A）
  TCCR1B |= (1 << CS12);    // CS12 = 1 選擇預除器 = 256
  TIMSK1 |= (1 << OCIE1A);  // 啟用 Timer1 比對 A 符合中斷
  interrupts();             // 全域重新啟用中斷
}

// 這會以精確的 1 Hz 自動執行——不輪詢，不延遲。
ISR(TIMER1_COMPA_vect) {
  digitalWrite(LED, !digitalRead(LED));   // 翻轉 LED
}

void loop() {
  // 完全空閒。1 Hz 的閃爍在背景發生。
  // 把你真正的工作放在這裡；它永遠不會阻塞計時器。
}
🔑 為什麼是 (1 << BITNAME)

像 WGM12、CS12、OCIE1A 這些常數只是定義在 AVR 標頭檔裡的位元位置（0–7 的數字）。(1 << CS12) 建立一個只設定該位元的遮罩；|= 把它 OR 進暫存器，在不擾動其他位元的情況下把那一位翻成 1。這是去戳單一硬體位元的通用慣用法——你在第 2 章看過它，而且你會永遠用到它。

6硬體中斷：零延遲的反應

① 直覺 硬體拍你的肩膀

計時器中斷在時鐘事件上觸發。外部中斷（external interrupt）則在腳位事件上觸發——一個上升或下降邊緣——而且不論你的程式正在做什麼，它都在幾個時鐘週期內就觸發。CPU 放下一切，跳進你的中斷服務常式（Interrupt Service Routine，ISR），執行它，然後從原本停下的地方精準地繼續。這就是你如何以幾乎零延遲去捕捉一個快速事件——編碼器邊緣、感測器脈衝、按鈕——而不是期盼 loop() 剛好在對的微秒去輪詢。

sequenceDiagram participant W as 外部接線（邊緣） participant H as AVR 硬體 participant M as loop() / 主程式 participant I as ISR M->>M: 正常執行中 W->>H: INT0 腳位上的上升邊緣 H->>M: 完成目前指令 H->>I: 向量跳到 ISR（推入內容） I->>I: 做最低限度的事 I->>M: 返回（彈出內容） M->>M: 從暫停處精準繼續

在 Uno 上，只有腳位 2 和 3 是真正的外部中斷線（INT0、INT1）。你用 Arduino API 來掛上一個 ISR：

encoder_edge.ino C++ / Arduino
const uint8_t ENC_A = 2;          // 編碼器 A 通道 → INT0
volatile int32_t pulses = 0;      // 與 ISR 共用 → 必須是 volatile

void setup() {
  pinMode(ENC_A, INPUT_PULLUP);
  // 在腳位 2 的每個上升（RISING）邊緣呼叫 countPulse()：
  attachInterrupt(digitalPinToInterrupt(ENC_A), countPulse, RISING);
  Serial.begin(115200);
}

// ISR：保持微小。不用 delay()、不用 Serial、不用 malloc。
void countPulse() {
  pulses++;                       // 只記下事件然後離開
}

void loop() {
  // 安全地讀取共用變數：關中斷後快照（snapshot）它。
  noInterrupts();
  int32_t snapshot = pulses;
  interrupts();
  Serial.println(snapshot);       // 緩慢的工作發生在「這裡」，不在 ISR 裡
}
🔑 ISR 的三條誡命

1. 保持微小。ISR 會從其他所有事情那裡奪走 CPU——設一個旗標、加一下計數器，然後馬上離開。
2. 不要 delay()、Serial 或任何會阻塞的東西。millis() 在 ISR 裡甚至不會前進（它的計時器中斷被遮蔽了），而 Serial 依賴現在已被關閉的中斷——死鎖（deadlock）。
3. 每個與 ISR 共用的變數都必須是 volatile。

② 數學 為何 volatile 沒得商量

volatile 告訴編譯器：「這個變數可能在你背後改變——每一次都要從記憶體重新讀取它，絕不要把它快取進暫存器或把存取最佳化掉。」少了它，最佳化器會假設 loop() 是唯一的寫入者。它可能把 pulses 提升（hoist）進一個 CPU 暫存器一次，然後永遠在那份過時的副本上打轉，從不看見 ISR 的更新。你的程式照樣編譯、照樣執行，然後悄無聲息地說謊。

⚠️ 陷阱——撕裂的讀取（torn read）

在 8 位元的 AVR 上，一個 32 位元的 int32_t 是以四個各自獨立的位元組操作來讀取的。如果 ISR 剛好在這些位元組之間觸發，你就會讀到一半的舊值加一半的新值——一次「撕裂的」讀取。這就是為什麼上面的快照用 noInterrupts()/interrupts() 把讀取包起來：它讓這個多位元組讀取變成原子的（atomic）。對單一個 uint8_t 旗標來說不需要這麼做；但對任何寬於 CPU 字長的東西來說，這是強制必要的。

① 直覺 腳位變化中斷 vs 外部中斷

兩個腳位往往不夠用。腳位變化中斷（Pin-change interrupts，PCINT）是替代方案：幾乎每個腳位都能引發一個，但它們較粗糙——一個共用的 ISR 會為整個埠（port）在任何邊緣（上升或下降）時觸發，而你的處理常式必須自己搞清楚是哪個腳位動了、往哪個方向動。能用外部中斷（腳位 2/3）時就用它——它們精確、按腳位區分、且可選擇邊緣。當你需要監看的腳位多到 INT0/INT1 涵蓋不了時，再退而求其次用腳位變化中斷。

7實際案例：確定性取樣與編碼器

④ 實際案例 這些都出貨在產品裡

本章的兩個模式在真實硬體裡無所不在，而且它們恰好對應到你剛認識的兩個中斷來源。

🏭 實際案例——精準的週期性取樣

一個數位音訊或振動感測器必須以穩如磐石的速率讀取，比方說 8 kHz，否則下游的 FFT 會糊掉。從 loop() 用 millis() 去讀它會抖動——有時迴圈正在忙。產品級的答案是一個設定為 8 kHz 的 Timer1 CTC 中斷：在 ISR 裡你讀一個 ADC 樣本進環形緩衝區（ring buffer）然後返回。取樣的瞬間現在是由石英振盪器定義，而不是靠軟體運氣。用預除器 8 設定 $\text{OCR1A} = 16{\times}10^6/(8 \times 8000) - 1 = 249$——正是 §5 的數學。

🏭 實際案例——零延遲的編碼器邊緣

一個馬達的正交編碼器發出脈衝的速度，比任何輪詢迴圈在高速時所能可靠捕捉的還快——漏掉一個邊緣，你的位置估測就會永遠漂移下去。把 A 通道接到一個外部中斷，代表每一個邊緣在它抵達的瞬間就讓計數遞增，不論 loop() 正在做什麼。這就是機器人輪子、CNC 軸與 3D 印表機如何精確知道自己在哪裡。ISR 只做一件事——pulses++——而沉重的控制數學稍後在 loop() 裡執行，去讀取那個 volatile 的快照。

🛠️ 分工

這個反覆出現的架構是：中斷負責捕捉，loop() 負責處理。ISR 在事件發生的精確時刻做不可再簡化的最低限度工作；而緩慢、複雜、能容忍阻塞的工作則回到非阻塞的主迴圈裡進行。掌握這個切分，你就能打造出既即時又有反應的韌體——這正是 delay() 永遠給不了你的東西。

Q1 觀念檢驗
一個 ISR 遞增一個全域的 int32_t pulses;，而 loop() 讀取它。下列其中兩項是錯的。哪一行建議是正確的？
  • 在 ISR 裡呼叫 Serial.println(pulses)，這樣你就能看到每一個脈衝。
  • 把 pulses 留成一個普通的 int32_t——編譯器總是會重新讀取全域變數。
  • 把 pulses 宣告為 volatile，並在 loop() 裡使用它之前，於關閉中斷的狀態下快照它。
  • 在 ISR 裡加一個 delay(1) 來為邊緣去彈跳（debounce）。
ISR 必須保持微小——不要 Serial（它需要被遮蔽掉的中斷）以及不要 delay()（millis() 被凍結，所以會卡住）。一個共用變數必須是 volatile，否則最佳化器可能快取一份過時的副本。而且因為在 8 位元的 AVR 上一個 32 位元值是以四次位元組操作讀取的，你要用 noInterrupts()/interrupts() 把讀取包起來，以避免撕裂的讀取。
Q2 算出來
你把 Timer1 設定成 CTC 模式，$f_{\text{clk}} = 16\ \text{MHz}$、預除器 $N = 256$，想要一個 1 Hz 的中斷。OCR1A 裡要填什麼值？（使用 $\text{OCR1A} = \frac{f_{\text{clk}}}{N\,f_{\text{int}}} - 1$。）
計數
$\text{OCR1A} = \dfrac{16\times10^6}{256 \times 1} - 1 = 62500 - 1 = 62499$。它放得進 16 位元暫存器（最大 65535）——這正是為何這需要 Timer1 而不是 8 位元的 Timer0/Timer2（最大 255）。代回去：$16{\times}10^6/(256\times62500) = 1\ \text{Hz}$。✓
Q3 程式挑戰——兩顆 LED、兩種頻率、不用 delay()
讓腳位 8 上的 LED A 以 2 Hz（每 250 ms 切換一次）閃爍、腳位 9 上的 LED B 以 5 Hz（每 100 ms 切換一次）同時閃爍，只用 millis() 慣用法——任何地方都不用 delay()。這兩個閃爍必須完全獨立運行。
提示
給每顆 LED 自己的 last 時間戳和自己的 interval。每一輪讀一次 now = millis()，然後執行兩個獨立的 if (now - lastX >= intervalX) 區塊。切換的寫法是 state = !state; digitalWrite(pin, state);。
顯示一個解答
two_leds.ino C++ / Arduino
const uint8_t LED_A = 8, LED_B = 9;
const uint32_t intervalA = 250;   // 2 Hz  → 每 250 ms 切換一次
const uint32_t intervalB = 100;   // 5 Hz  → 每 100 ms 切換一次
uint32_t lastA = 0, lastB = 0;
bool stateA = false, stateB = false;

void setup() {
  pinMode(LED_A, OUTPUT);
  pinMode(LED_B, OUTPUT);
}

void loop() {
  uint32_t now = millis();        // 每一輪讀一次時鐘

  if (now - lastA >= intervalA) {  // LED A：獨立計時器
    lastA = now;
    stateA = !stateA;
    digitalWrite(LED_A, stateA);
  }

  if (now - lastB >= intervalB) {  // LED B：獨立計時器
    lastB = now;
    stateB = !stateB;
    digitalWrite(LED_B, stateB);
  }
  // loop() 永不阻塞 → 兩顆 LED 各自保持自己的節奏，
  // 而且這裡還有空間去讀按鈕或感測器。
}

這一課的重點：每個任務擁有自己的 last + interval，而 loop() 保持非阻塞。這就是 §3 那個協作式排程器的迷你版——再加上第三顆 LED、一個按鈕、一個感測器，它們全都能共存，因為沒有任何東西曾經靜止不動。試試用 delay() 來做，看著那兩種頻率相撞。

🎓 本章回顧

你用三種方法逃離了 delay()。你能寫出防溢位的 now - last >= interval 慣用法，並解釋為何無號減法能挺過 49.71 天的繞回。你能把 AVR 計時器設定成 CTC 模式——挑選預除器並求解 $\text{OCRnA} = f_{\text{clk}}/(N f_{\text{int}}) - 1$（Timer1 上 1 Hz 為 62499）——而且你知道 Timer0 是禁區。你能掛上一個外部中斷、讓它的 ISR 保持微小、把共用狀態標為 volatile，並原子地快照寬值。要記住的架構是：中斷負責捕捉，loop() 負責處理。下一章我們讓這些計時器派上用場，用 PWM 產生帶有類比感的輸出。
