類比世界：ADC、感測器與訊號調理

到目前為止，你的接腳都活在二元宇宙裡——HIGH 或 LOW，開或關。但溫度、光線、壓力與電池電壓都是連續的。要讀取它們，你需要這顆晶片中最被低估的周邊：類比數位轉換器（ADC）。我們會從直覺把它建立起來，把它的精度推導到毫伏的層級，並接上真正出貨於真實產品中的感測器。

⏱ 約 45 分鐘🧠 量化⚡ 真實感測器📐 分壓器數學

1類比 vs 數位：世界是一條平滑曲線

① 直覺 先建立圖像

數位訊號恰好只有兩種狀態。類比訊號則可以是某個範圍內的任何值，隨時間平滑變化——麥克風的電壓、溫度感測器的緩慢漂移、旋鈕的位置。大自然不處理 HIGH 與 LOW；它處理的是無限精細的漸層。

然而，你那顆 8 位元的 AVR 只懂數字。它無法把「3.27 伏特」當成一個連續量保存下來——它必須把那個電壓轉換成可以儲存並做運算的整數。執行這項轉譯的周邊就是 ADC（類比數位轉換器），而 analogRead() 就是你通往它的一行門票。

💡 一句話的模型

ADC 是一把量電壓的尺。它在量測範圍上鋪設固定數量、間距均勻的刻度，然後回報你的輸入電壓最接近哪一格刻度——以一個普通整數的形式。

「最接近」這個詞就是整個故事的核心。一把以毫米為刻度的尺無法告訴你某物是 4.37 mm——它會四捨五入到 4 mm。ADC 對電壓做的事一模一樣。這種捨入稱為量化，搞懂它，正是「信任你的感測器資料」與「被它愚弄」之間的分野。

🔑 兩個接腳世界

Uno 上標示為 D0–D13 的接腳是數位（兩種狀態）。標示為 A0–A5 的六支接腳則接到 ADC，每一支都能被讀取為連續電壓。（它們也可以兼作數位腳——但只有 A0–A5 能通到轉換器。）

2解析度：尺上有多少格刻度？

② 數學 推導精度

ADC 的解析度是指它能輸出多少個不同的值，由它的位元數 $N$ 決定。用 $N$ 個位元，你可以編碼出

$$ \text{levels} = 2^{N} $$

ATmega328P 的 ADC 是 10 位元，所以它會產生

$$ 2^{10} = 1024 \text{ levels}, \quad \text{numbered as codes } 0,1,2,\dots,1023 $$

編碼 0 代表「電壓在範圍底端或更低」；編碼 1023 代表「在頂端或更高」。介於兩者之間的一切都會對應到 1022 個內部刻度之一。一格刻度的大小——也就是 ADC 能分辨的最小電壓變化——稱為 LSB（最低有效位元），其值由整個範圍除以步進數而得：

$$ V_{\text{LSB}} = \frac{V_{\text{ref}}}{2^{N}} = \frac{V_{\text{ref}}}{1024} $$

在 Uno 的預設參考電壓 $V_{\text{ref}} = 5\,\text{V}$ 下：

$$ V_{\text{LSB}} = \frac{5\,\text{V}}{1024} \approx 4.883\,\text{mV} $$

這就是你的下限：Uno 無法區分相差小於約 4.9 mV 的兩個電壓。想要更細的解析度？要嘛縮小 $V_{\text{ref}}$（1.1 V 的內部參考可給出約 1.07 mV/LSB，但範圍只有 0–1.1 V），要嘛換到一顆具備 12 位元或 16 位元 ADC 的晶片。

∑ 量化誤差是有界的

因為 ADC 會捨入到最接近的刻度，任何一筆讀數的偏差最多就是半格刻度。最差情況的量化誤差為

$$ \varepsilon_{q} = \pm\tfrac{1}{2}V_{\text{LSB}} = \pm\frac{V_{\text{ref}}}{2048} \approx \pm 2.44\,\text{mV} $$

這個誤差是與生俱來的——即使輸入完美、毫無雜訊，它依然存在。這是用整數去表示一個平滑世界所必須付出的代價。

3從編碼到伏特：analogRead() 與 AREF

② 數學 反轉映射

analogRead(pin) 交給你的是整數編碼，不是電壓。要還原電壓，你得把這把尺反過來算。微妙之處在於：雖然有 1024 個位階，但全幅讀數是從編碼 0 跨到編碼 1023，總共 1023 個步進，因此 Arduino 的慣例是把編碼線性映射到 $[0, V_{\text{ref}}]$：

$$ V = \frac{\text{adc}}{1023}\, V_{\text{ref}} $$

所以在 $V_{\text{ref}} = 5\,\text{V}$ 下，讀數 512 對應的是 $\tfrac{512}{1023}\times 5 \approx 2.503\,\text{V}$——幾乎正好是中刻度，正如你所預期。（你會在後面的練習中再次遇到這個運算。）

③ 程式碼 逐行細讀
read_voltage.ino C++ / Arduino
const uint8_t SENSE_PIN = A0;     // 類比輸入 A0
const float   VREF      = 5.0;    // 參考電壓（Uno 預設）

void setup() {
  Serial.begin(9600);
  // 類比腳做 analogRead 時不需要 pinMode——ADC 的
  // 多工器會處理通道選擇。但讓它們保持未被驅動的狀態。
}

void loop() {
  int   raw     = analogRead(SENSE_PIN);   // 0..1023，耗時約 104 µs
  float volts   = (raw / 1023.0) * VREF;   // 反轉量化器

  Serial.print(raw);
  Serial.print(F("  ->  "));
  Serial.print(volts, 3);                  // 小數點後 3 位
  Serial.println(F(" V"));
  delay(200);
}
⚠️ 陷阱——整數除法吃掉你的讀數

若寫成 raw / 1023（兩者皆整數），C++ 會做整數除法：任何低於 1023 的編碼都會變成 0。你必須以 raw / 1023.0（注意那個 .0）強制使用浮點數，分數電壓才能保留下來。這個沉默的 bug 幾乎每位初學者都會中招一次。

$V_{\text{ref}}$ 在物理上是什麼？它是 ADC 拿來比對的電壓——它的「尺頂」。在 Uno 上，你用 analogReference() 來選擇它：

選項參考電壓LSB使用時機……
DEFAULT5 V（VCC）4.88 mV一般用途；範圍與 5 V 邏輯相符
INTERNAL1.1 V（晶片內建能隙基準）1.07 mV微弱訊號；即使 VCC 下垂仍穩定
EXTERNALAREF 腳上的電壓VAREF/1024你自備精密參考電壓
⚠️ 陷阱——AREF 短路

若你先把外部電壓接到 AREF 腳，然後才呼叫 analogReference(EXTERNAL)，沒問題。但若順序顛倒——在仍連接著預設內部參考的情況下去驅動 AREF——你就會把內部源短路到你的外部源。永遠要在第一次 analogRead() 之前就選好 EXTERNAL。

4它能多快取樣？預除器與奈奎斯特

② 數學 速度從何而來

轉換並非瞬間完成——ADC 使用逐次逼近，本質上是一場二分搜尋，把一個比較電壓對半切十次，逐一鎖定全部十個位元。它由自己的時脈驅動，而該時脈是經由預除器從 16 MHz 系統時脈導出的。AVR 資料手冊規定 ADC 時脈應落在 50 kHz 到 200 kHz 之間，才能維持完整的 10 位元準確度，因此 Arduino 核心選用了 128 的預除值：

$$ f_{\text{ADC}} = \frac{f_{\text{clk}}}{128} = \frac{16\,\text{MHz}}{128} = 125\,\text{kHz} $$

一次正常轉換需要 13 個 ADC 時脈週期。所以每次取樣的時間與由此得到的吞吐量為

$$ t_{\text{conv}} = \frac{13}{f_{\text{ADC}}} = \frac{13}{125\,000} \approx 104\,\mu\text{s}, \qquad f_{s} = \frac{f_{\text{ADC}}}{13} \approx 9.6\,\text{ksps} $$

就是這樣——在預設設定下大約每秒 9600 次取樣，這正是上面程式中 ~104 µs 註解背後的數字。如果你能接受低位元稍微多一些雜訊，可以把預除器降到 16（1 MHz 的 ADC 時脈），達到約 77 ksps——這是音訊級取樣常用的招數。

🔑 一行講完奈奎斯特

要重建頻率為 $f_{\text{sig}}$ 的訊號，你的取樣速度必須快過它的兩倍：

$$ f_{s} > 2\,f_{\text{sig}} $$

在 9.6 ksps 下，Uno 可以忠實捕捉到約 4.8 kHz 以下的訊號。餵給它任何更快的訊號，它就會混疊（alias）——一個 10 kHz 的音調會偽裝成一道緩慢的幽靈波。這就是為什麼音訊設備會在 ADC 之前放一個低通的「抗混疊」濾波器。

5分壓器：你的萬用感測器介面

① 直覺 把電阻變成電壓

大多數廉價感測器並不輸出電壓——它們改變的是自身的電阻：光敏電阻（LDR）在受光時電阻變小，熱敏電阻在變熱時電阻變小。但 ADC 只會讀電壓。連接兩者的橋樑，是整個電子學中最常用的電路：分壓器。

把兩個電阻串聯跨接在你的電源上。兩者之間的中點電位由它們的比值決定。若 $R_1$ 在上方（朝向 Vin），$R_2$ 在下方（朝向接地），則中點電壓為

② 數學 從歐姆定律推導
$$ V_{\text{out}} = V_{\text{in}}\,\frac{R_2}{R_1 + R_2} $$

為什麼？同一股電流 $I = V_{\text{in}}/(R_1+R_2)$ 流過兩個電阻（串聯）。下方電阻上的電壓是 $I\,R_2$；代入 $I$，那個 $R_1+R_2$ 的分母就出現了。檢驗一下極端情況：若 $R_2 \to 0$，輸出為 0 V；若 $R_2 \to \infty$，輸出趨近於完整的 $V_{\text{in}}$。比值不過是把中點上下滑動而已。

flowchart LR SENSOR["電阻式感測器
LDR / 熱敏電阻"] --> DIV["分壓器
固定 R + 感測器"] DIV --> ADC["ADC
10 位元量化器"] ADC --> CODE["analogRead()
編碼 0..1023"] CODE --> CALC["你的程式
伏特 → 物理單位"]

這條鏈——感測器 → 分壓器 → ADC → 編碼 → 物理單位——是你日後幾乎所有類比量測的骨架。徹底掌握它一次，每一個電阻式感測器都會變成同一個問題。

🛠️ 挑選固定電阻

為了得到最佳靈敏度，要把固定電阻選在感測器於你最在意的那個值時的電阻附近。室溫下的 10 kΩ 熱敏電阻與一個 10 kΩ 固定電阻搭配得天衣無縫——分壓器輸出此時正好在你量測的位置擺幅最陡。

6三個真實感測器：LDR、熱敏電阻與 TMP36

③ 程式碼 以及每一個背後的數學

LDR（光線）。光敏電阻的阻值在黑暗中高達數百萬歐姆，在強光下則降到幾百歐姆。把它接成分壓器的上方電阻，下方接一個固定的 10 kΩ 到接地；光線越亮，ADC 讀數越高。不需要花俏的數學——你只要把編碼映射成「較暗/較亮」，或對照已知的勒克斯（lux）做校正即可。

NTC 熱敏電阻（溫度，非線性）。它的電阻隨溫度升高而下降，但是以指數方式，而非線性。簡單的 Beta 模型把它的電阻 $R$ 與以克耳文為單位的溫度連結起來：

$$ \frac{1}{T} = \frac{1}{T_0} + \frac{1}{B}\ln\!\left(\frac{R}{R_0}\right) $$

此處 $R_0$ 是在參考溫度 $T_0$（通常為 25 °C = 298.15 K）下的標稱電阻，而 $B$ 是資料手冊上的「Beta」常數（常見的 10 kΩ 元件約為 3950 K）。你讀取分壓器，解出熱敏電阻的 $R$，然後代入方程式。若要更高的準確度，完整的 Steinhart–Hart 方程式會再加兩個係數，但在一般從室溫到烤箱的範圍內，Beta 模型的誤差在零點幾度以內。

thermistor.ino C++ / Arduino
#include <math.h>

const float VREF   = 5.0;
const float R_FIX  = 10000.0;   // 10 kΩ 固定電阻（接 GND）
const float R0     = 10000.0;   // 熱敏電阻 @ 25 °C 為 10 kΩ
const float T0     = 298.15;    // 25 °C 換算為克耳文
const float BETA   = 3950.0;    // 資料手冊 Beta 常數（K）

void setup() { Serial.begin(9600); }

void loop() {
  int   raw   = analogRead(A0);
  float v     = (raw / 1023.0) * VREF;

  // 熱敏電阻是上方電阻；R_FIX 接 GND 為下方電阻。
  // Vout = VREF * R_FIX / (R_therm + R_FIX)  ->  解出 R_therm：
  float rTherm = R_FIX * (VREF / v - 1.0);

  // Beta 方程式：1/T = 1/T0 + (1/B)·ln(R/R0)
  float invT  = 1.0 / T0 + (1.0 / BETA) * log(rTherm / R0);
  float tempC = (1.0 / invT) - 273.15;   // 克耳文 -> 攝氏

  Serial.print(tempC, 2);
  Serial.println(F(" C"));
  delay(500);
}

TMP36（溫度，線性又簡單）。這顆主動式感測器在內部把難算的部分都做掉了，輸出一個乾淨的電壓：0 °C 時為 500 mV，之後每升高 1 °C 規規矩矩地上升 10 mV。不用分壓器、不用對數——只要把一條直線反過來算：

$$ T_{\circ C} = \frac{V_{\text{out}} - 0.5\,\text{V}}{0.01\,\text{V}/^{\circ}\text{C}} $$
tmp36.ino C++ / Arduino
const float VREF = 5.0;

void setup() { Serial.begin(9600); }

void loop() {
  int   raw   = analogRead(A0);
  float v     = (raw / 1023.0) * VREF;       // 感測器輸出，單位伏特
  float tempC = (v - 0.5) / 0.01;            // 0°C 時 500 mV，每 °C 10 mV

  Serial.print(tempC, 1);
  Serial.println(F(" C"));
  delay(500);
}
⚠️ 陷阱——5 V 參考會晃動

上面每一條公式都假設 $V_{\text{ref}}$ 恰好是 5.000 V。以 USB 供電時，你那條「5 V」電軌在負載下可能下垂到 4.7 V，使每一筆讀數偏掉約 6%。做溫度量測時，請改用 analogReference(INTERNAL)（穩如磐石的 1.1 V 能隙基準），或量測你實際的 VCC 並把它當作 $V_{\text{ref}}$——這正是我們接下來要用的技巧。

7雜訊與平均：用 √N 買位元

② 數學 訊噪比的縮放

原始 ADC 讀數會抖動。電氣雜訊、不乾淨的供電，以及量化下限本身，都會讓對穩定電壓連續取樣的結果跳動個一兩格。最廉價的療法是過取樣（oversampling）：取許多筆讀數再求平均。

如果雜訊是隨機且不相關的，對 $N$ 筆樣本求平均會讓訊號維持不變，卻把雜訊的標準差縮小 $\sqrt{N}$ 倍。所以訊噪比的改善為

$$ \text{SNR}_{\text{avg}} = \sqrt{N}\;\cdot\;\text{SNR}_{\text{single}} $$

對 16 筆樣本求平均，雜訊降為 $\sqrt{16}=4\times$；64 筆樣本則降為 $8\times$。平方根裡藏著一個代價——邊際效益遞減。從 16 筆增加到 64 筆（4 倍的工作量），雜訊削減量也只翻一倍。每多得一個位元的有效解析度，都要付出 $4\times$ 的樣本數，因為 $\sqrt{4}=2$。

averaged_read.ino C++ / Arduino
// 讀取某腳 N 次，並以浮點編碼回傳平均值。
float readAveraged(uint8_t pin, uint8_t n) {
  uint32_t sum = 0;                  // 32 位元：64 × 1023 不會溢位
  for (uint8_t i = 0; i < n; i++) {
    sum += analogRead(pin);
    // 切換 ADC 通道後，加一點微小的安定延遲會有幫助
    delayMicroseconds(50);
  }
  return (float)sum / n;             // 平均後的編碼，0..1023
}

void loop() {
  float code  = readAveraged(A0, 64);     // 64 筆樣本平均
  float volts = (code / 1023.0) * 5.0;
  Serial.println(volts, 4);
  delay(100);
}
🛠️ 硬體勝過軟體

軟體平均有幫助，但要先從源頭殺掉雜訊：在類比腳（或 AREF）與接地之間焊上一個 0.1 µF 去耦電容。它會與感測器的電阻形成一個低通濾波器，在高頻雜波抵達取樣保持電路之前就把它吞掉。一顆便宜的陶瓷電容，往往比 64 筆平均讀數還管用。

8真實案例：監控一顆你無法直接讀取的電池

④ 真實案例 這會出貨於產品中

這是每個電池供電產品都會面臨的問題。一組 2 串（2-cell）的鋰聚合物（LiPo）電池滿電時約在 8.4 V——但 ADC 的上限只到 5 V。把 8.4 V 灌進 A0，你就會把晶片的魔法青煙放出來。解法正是你剛學會的那個分壓器：把電池電壓縮放到 ADC 的安全窗口內，再用軟體把那個數字放大回去。

選 $R_1 = 100\,\text{k}\Omega$（上方，接電池）與 $R_2 = 47\,\text{k}\Omega$（下方，接地）。在滿電的 8.4 V 下，分壓器輸出為

$$ V_{\text{out}} = 8.4 \times \frac{47\text{k}}{100\text{k}+47\text{k}} = 8.4 \times 0.320 \approx 2.69\,\text{V} $$

舒舒服服地落在 5 V 以下。比值 $(R_1+R_2)/R_2 \approx 3.13$ 就是你的軟體乘數：量得 $V_{\text{out}}$，乘回去，就還原出真實的電池組電壓。較大的電阻值（數百 kΩ）能把分壓器的待機電流壓在微安等級——當你想讓電池撐久一點時，這很重要。

battery_monitor.ino C++ / Arduino
const uint8_t BATT_PIN = A0;
const float   VREF     = 5.0;
const float   R1       = 100000.0;   // 上方電阻（接電池 +）
const float   R2       = 47000.0;    // 下方電阻（接 GND）

// 我們必須在軟體中還原的分壓增益：
const float DIVIDER_GAIN = (R1 + R2) / R2;   // ≈ 3.128

void setup() { Serial.begin(9600); }

void loop() {
  // 平均以馴服長感測線上的雜訊
  uint32_t sum = 0;
  for (uint8_t i = 0; i < 16; i++) sum += analogRead(BATT_PIN);
  float code   = sum / 16.0;

  float vAdc   = (code / 1023.0) * VREF;      // 接腳上的電壓
  float vBatt  = vAdc * DIVIDER_GAIN;         // 縮放回電池組電壓

  Serial.print(F("Battery: "));
  Serial.print(vBatt, 2);
  Serial.println(F(" V"));

  if (vBatt < 6.4) Serial.println(F("!! LOW — shut down soon"));
  delay(1000);
}
🏭 真實案例——電位器也是一個分壓器

你儀表板上那個旋鈕？電位器就是一個帶有可動抽頭的分壓器。把外側兩腳接到 5 V 與 GND，中間的滑臂（wiper）接到 A0，轉動它就會讓滑臂電壓從 0 掃到 5 V——編碼 0 到 1023。音量旋鈕、搖桿、油門握把、馬達調速盤，全都是這個完全相同的電路。你剛剛學會的，正是消費性電子產品中最常見的類比輸入。

Q1 觀念檢核
Uno 的 ADC 是 10 位元，配 5 V 參考電壓，得到約 4.88 mV 的 LSB。這個 LSB 值對你的量測來說實際意味著什麼？
  • ADC 每次轉換耗時 4.88 ms。
  • ADC 能分辨的最小電壓變化約為 4.88 mV；比這更細微的差異會被捨入到相同的編碼。
  • 接腳能讀到的最大電壓是 4.88 mV。
  • 每一筆讀數都會隨機地恰好偏掉 4.88 mV。
LSB 是一個編碼步進所代表的電壓：$V_{\text{LSB}} = V_{\text{ref}}/2^{N} = 5/1024 \approx 4.88\,\text{mV}$。相距小於此值的兩個輸入會對應到同一個整數——這就是解析度下限。量化誤差被限制在半個 LSB（±2.44 mV），而轉換時間（約 104 µs）完全是另一回事。
Q2 算一算
在 $V_{\text{ref}} = 5\,\text{V}$ 下，analogRead() 回傳 512。接腳上的電壓是多少伏特？（使用 $V = \dfrac{\text{adc}}{1023}\,V_{\text{ref}}$。）
V
$V = \dfrac{512}{1023}\times 5 = 2.503\,\text{V} \approx 2.50\,\text{V}$。編碼 512 基本上就是中刻度，所以它對應到 5 V 範圍的一半——正是直覺所預測的。（若改除以 1024 而非 1023，會得到 2.500 V；差異是千分之一，遠在容差之內。）
Q3 程式挑戰——N 筆樣本平均
A0 上的光感測器讀數會抖動 ±3 個編碼。請寫一個函式 float smoothRead(uint8_t pin, uint16_t n)，以浮點編碼（0–1023）回傳 n 筆樣本的平均值，然後用它印出一個穩定的電壓。請仔細思考你那個累加和（running sum）的資料型別——如果你用 int 去累加 1000 筆讀數，會發生什麼事？
提示
AVR 上的單一 int 是 16 位元，上限為 32767。光是累加 33 筆全幅讀數（33 × 1023 = 33759）就已經溢位並繞回成垃圾值。請用一個 32 位元累加器（uint32_t），在迴圈中逐一加上每筆 analogRead()，然後以浮點數除以 n，讓小數部分能保留下來。
顯示一種解法
smooth_read.ino C++ / Arduino
// 以分數編碼（0..1023）回傳 n 筆樣本的平均值。
float smoothRead(uint8_t pin, uint16_t n) {
  uint32_t sum = 0;                 // 32 位元，所以大的 n 也不會溢位
  for (uint16_t i = 0; i < n; i++) {
    sum += analogRead(pin);
  }
  return (float)sum / n;            // 除法「之前」先轉型 → 保留小數
}

void setup() { Serial.begin(9600); }

void loop() {
  float code  = smoothRead(A0, 64);          // √64 = 雜訊降為 1/8
  float volts = (code / 1023.0) * 5.0;
  Serial.println(volts, 4);                  // 現在這 4 位小數才有意義
  delay(100);
}

兩個陷阱分別是 32 位元的累加和（16 位元的 int 在約 32 筆全幅讀數後就溢位）以及在除法之前轉型成 float（這樣才不會把平均值截斷）。用 64 筆樣本時，雜訊下降為 $\sqrt{64}=8\times$——此時那 4 位小數才真正有意義，而不是隨機閃爍。
