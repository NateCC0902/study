PWM & Analog Output: Dimming, Servos & Motors

A digital pin can only be fully on or fully off — yet you have dimmed an LED, set a motor's speed, and aimed a servo with one. The trick is time: switch fast enough and the world averages the blur into a smooth value. This is the single most useful "fake" in embedded hardware.

1The trick: blink so fast it looks analog

① Intuition build the picture first

In Chapter 1 you blinked an LED at 1 Hz and saw it flash. Speed that blink up to hundreds of times per second and something strange happens: the flashing vanishes and the LED simply looks dimmer. You didn't change the voltage — it is still snapping between 0 V and 5 V — but your eye can't track the flicker, so it integrates the light over time and reports an average brightness.

That is Pulse-Width Modulation (PWM). The pin is always digital: hard 0 or hard 5 V, never anything in between. We just choose what fraction of each cycle it spends high. Spend 25% of the time high and you get a dim glow; 75% and it's nearly full brightness. The fraction is the only knob.

💡 The one-sentence model

PWM fakes an analog voltage by switching a pin fully on and off very fast; the load — or your eye, or a motor's inertia — averages the square wave into an effective level you can dial from 0% to 100%.

Why bother faking it? Because a true analog output is expensive and lossy. To actually deliver 2.5 V at a useful current, a chip would have to burn the other half of the power as heat in a regulator. PWM throws no power away: the switch is either fully on (almost no voltage across it, little loss) or fully off (no current, no loss). It is the same reason your house dims lights and EVs run motors with switching, not resistors.

⚠️ Pitfall — a PWM pin is not a DAC

An Uno output pin still only ever sits at 0 V or 5 V. If you put a slow voltmeter on a PWM pin you'll read a blurry average; if you put an oscilloscope on it you'll see the raw square wave. There is no analog circuitry behind analogWrite() on the classic AVR — the "analog" is entirely in the timing. (Some boards, like the SAMD-based Zero, do have a real DAC; the Uno does not.)

2The math: duty cycle and average voltage

② Math derive it

One full PWM cycle has a fixed period $T$. Within it the pin is high for a time $t_{\text{on}}$ and low for $t_{\text{off}}$, with $T = t_{\text{on}} + t_{\text{off}}$. The single number that defines the waveform's "level" is the duty cycle — the fraction of the period spent high:

$$ D = \frac{t_{\text{on}}}{T} = \frac{t_{\text{on}}}{t_{\text{on}} + t_{\text{off}}} \in [0, 1] $$

The average voltage is just the time-average of a wave that equals $V_{cc}$ for a fraction $D$ of the period and $0$ for the rest. The average of a two-level signal is the time-weighted mean of its levels:

$$ V_{\text{avg}} = \frac{1}{T}\!\int_0^{T}\! v(t)\,dt = \frac{V_{cc}\cdot t_{\text{on}} + 0\cdot t_{\text{off}}}{T} = D\,V_{cc} $$

Clean and linear: average voltage is duty cycle times supply. At $V_{cc}=5\,\text{V}$, a duty of 40% averages $0.40\times5 = 2.0\,\text{V}$; a duty of 60% averages 3.0 V. That straight line $V_{\text{avg}} = D\,V_{cc}$ is the heart of the whole chapter.

The Arduino API hides $D$ behind an 8-bit integer. analogWrite(pin, val) takes val in $[0, 255]$ and maps it to duty linearly:

$$ D = \frac{\texttt{val}}{255}, \qquad V_{\text{avg}} = \frac{\texttt{val}}{255}\,V_{cc} $$

So analogWrite(pin, 0) is fully off (0 V), 255 is fully on (5 V), and 127 is $127/255 \approx 49.8\%$ — essentially half. To invert the formula and find the value for a target voltage, solve for val:

$$ \texttt{val} = \operatorname{round}\!\left(\frac{V_{\text{target}}}{V_{cc}}\times 255\right) $$

For 3.0 V on a 5 V board: $\texttt{val} = \operatorname{round}(3.0/5.0\times255) = \operatorname{round}(153.0) = 153$. Memorize that round-trip — it's the most common PWM arithmetic you'll do.

∑ Aside — RMS vs. average for a resistive load

A heater or incandescent bulb dissipates power as $P = V_{\text{rms}}^2/R$, and for a PWM wave the RMS voltage is $V_{\text{rms}} = V_{cc}\sqrt{D}$, not $V_{cc}D$. So power scales linearly with duty: $P = D\,V_{cc}^2/R$. That's convenient — 50% duty gives exactly half power into a resistor — but note the resulting RMS voltage (e.g. $5\sqrt{0.5}\approx3.54\,\text{V}$ at 50%) differs from the average voltage (2.5 V). For an LED you care about average light; for a heater you care about average power.

3Where the square wave comes from: a hardware timer

① Intuition not done in software

Here's the part beginners miss: analogWrite() does not sit in a loop toggling the pin. If it did, the CPU could never do anything else. Instead it configures a dedicated piece of silicon — a timer/counter peripheral — to generate the square wave autonomously, in hardware, forever, while your code runs on.

A timer is a counter that increments every clock tick, wrapping back to zero when it overflows. PWM works by comparing that running counter against a compare value you set: while counter < compare the pin is driven high; once counter ≥ compare it flips low. The compare value is your duty cycle. analogWrite(pin, 153) literally writes 153 into a compare register (OCRnx) and the hardware does the rest.

🔑 The counter-vs-threshold model

A free-running counter ramps 0 → 255 → 0 → 255… Each cycle, the pin is HIGH while the counter is below your threshold and LOW above it. Raise the threshold → wider high pulse → higher duty → brighter. The CPU sets the threshold once; the timer redraws the waveform thousands of times a second with zero CPU involvement.

On the ATmega328P there are three timers driving six PWM-capable pins: Timer0 (pins 5, 6), Timer1 (pins 9, 10), and Timer2 (pins 3, 11). The squiggle ~ printed beside those pin numbers on the board marks them. Other pins simply have no compare hardware wired to them, which is why analogWrite() only works on those six.

⚠️ Pitfall — the default frequencies differ

Because the three timers are clocked with different prescalers in the Arduino core, the PWM frequencies aren't uniform. On the Uno, pins 3, 9, 10, 11 run at ~490 Hz, while pins 5 and 6 run at ~980 Hz (Timer0, which also backs millis() and delay()). Don't fight this by reprogramming Timer0 unless you're ready to break timekeeping — we open that hood deliberately in Chapter 5.

We will derive these frequencies from prescalers and counter widths in the next chapter. For now, hold the link: PWM is a timer comparing a counter to a threshold. That sentence is the bridge to Chapter 5.

4Code: dimming an LED (and why linear looks wrong)

③ Code read every line

The simplest possible PWM program: fade an LED up and down. Wire an LED (with its current-limiting resistor from Chapter 3) to a ~ pin — say pin 9 — and sweep the duty from 0 to 255 and back.

fade.ino C++ / Arduino
const int LED = 9;  // must be a PWM (~) pin: 3,5,6,9,10,11 on the Uno

void setup() {
  pinMode(LED, OUTPUT);   // analogWrite still needs an OUTPUT pin
}

void loop() {
  // Fade UP: duty 0 -> 255 (0% -> 100%)
  for (int val = 0; val <= 255; val++) {
    analogWrite(LED, val);   // writes the timer compare register (OCR1A)
    delay(8);                // ~2 s for a full sweep (256 * 8 ms)
  }
  // Fade DOWN: duty 255 -> 0
  for (int val = 255; val >= 0; val--) {
    analogWrite(LED, val);
    delay(8);
  }
}

Upload it and you'll notice something annoying: the LED rushes to "looks full bright" in the first quarter of the sweep, then barely changes for the rest. The fade is mathematically linear but looks badly lopsided. That isn't a bug — it's your eyes.

② Math the eye is logarithmic

Human brightness perception follows roughly a power law (the CIE lightness relationship). Perceived lightness scales like physical intensity raised to about $1/2.2$, which means to get a perceptually linear ramp we must pre-distort the duty with the inverse — a gamma of $\gamma \approx 2.2$:

$$ \texttt{out} = \left(\frac{\texttt{in}}{255}\right)^{\gamma}\!\times 255, \qquad \gamma \approx 2.2 $$

Low inputs get squashed toward zero (so the dim end gets the fine resolution your eye craves), high inputs stay near full. Apply this correction and the fade becomes visually smooth and even:

fade_gamma.ino C++ / Arduino
#include <math.h>
const int LED = 9;

// Map a perceptual 0..255 request to the duty the eye actually needs.
uint8_t gamma8(uint8_t in) {
  float norm = in / 255.0f;                     // 0.0 .. 1.0
  float corrected = powf(norm, 2.2f);           // squash the low end
  return (uint8_t)(corrected * 255.0f + 0.5f);  // +0.5 rounds to nearest
}

void setup() { pinMode(LED, OUTPUT); }

void loop() {
  for (int b = 0; b <= 255; b++) {
    analogWrite(LED, gamma8(b));   // request perceptual b, emit corrected duty
    delay(8);
  }
  for (int b = 255; b >= 0; b--) {
    analogWrite(LED, gamma8(b));
    delay(8);
  }
}
🛠️ Precompute the table

powf() is slow on an 8-bit AVR with no FPU. In production you'd compute the 256-entry curve once and store it in Flash as const uint8_t gamma_lut[256] PROGMEM, then index it — turning a floating-point exponentiation into a single array read. The Adafruit NeoPixel library ships exactly such a table for this reason.

5Driving a motor: why the pin can't, and the MOSFET that can

④ Real case this ships in products

Now the most important safety lesson in this chapter. You cannot connect a DC motor directly to an Arduino pin. An ATmega328P pin sources at most ~20 mA safely (40 mA absolute max, and the whole chip has a combined budget). A small toy motor pulls hundreds of milliamps running, and several amps at stall. Wire one to a pin and you don't get a slow motor — you get a dead microcontroller.

⚠️ Pitfall — the pin is a signal, not a power source

Treat every output pin as a control wire, not a power wire. Its job is to tell a bigger switch what to do, while the motor's current flows from the supply through that switch — never through the chip. Anything past ~20 mA (motors, relays, solenoids, filament bulbs, long LED strips) needs an external switching device.

That switch is a MOSFET — a transistor used as an electronically controlled relay with no moving parts. A logic-level N-channel MOSFET (e.g. an IRLZ44N, or a 2N7000 for tiny loads) turns fully on when its gate is driven high by the pin, connecting the motor's negative terminal to ground. Feed the gate your PWM signal and the motor switches on and off hundreds of times per second; its rotational inertia averages the pulses into a steady speed — the same averaging trick as the LED, now mechanical.

flowchart LR PIN["Arduino pin 9
PWM out"] -->|gate| Q["N-ch MOSFET
IRLZ44N"] VPLUS(["+12 V supply"]) --> M["DC motor"] M -->|drain| Q Q -->|source| GND[("GND")] M -. flyback .-> D["Diode 1N4007
cathode to +12 V"] D -.-> VPLUS VPLUS --- GND

Notice the diode across the motor — the flyback (freewheeling) diode. A motor is an inductor, and the defining law of an inductor is $v_L = L\,\dfrac{di}{dt}$. When the MOSFET switches off, current that was flowing is forced to stop almost instantly, so $di/dt$ is enormous and negative — the motor generates a large positive voltage spike (hundreds of volts) trying to keep its current flowing. Without a path, that spike punches through the MOSFET and destroys it.

🔑 The flyback diode is mandatory, not optional

Wire the diode reverse-biased across the motor: cathode (the striped end) to the +supply, anode to the drain. In normal operation it's off and invisible. At switch-off it flips on and gives the collapsing inductor current a safe loop to die in, clamping the spike to ~0.7 V. Skip it and you'll burn out MOSFETs at every PWM edge — and PWM produces thousands of edges per second.

🏭 Real case — your laptop's cooling fan

That 4-pin PWM fan header on every motherboard is exactly this circuit, productized. Pin 4 carries a ~25 kHz PWM signal (chosen above human hearing so the switching doesn't whine); the fan's own driver IC reads the duty and sets speed, with the flyback handled internally. When your CPU heats up, firmware raises the duty — the literal $D\,V_{cc}$ you just learned — and the fan spins faster. Same math, shipping in a billion machines.

6Servos: PWM where the width, not the duty, is the message

① Intuition a different flavor of PWM

A hobby servo (an SG90, an MG996R) is a little geared motor with a position-control circuit inside. You don't tell it a speed; you tell it an angle, and it drives itself there and holds. The command is also a pulse train — but here the information is encoded in the absolute pulse width, and the period is fixed by convention.

The standard is a 50 Hz frame: a pulse every $T = 20\,\text{ms}$. Within each frame the high-pulse width selects the angle:

Pulse widthAngleDuty within 20 ms
1.0 ms0°5%
1.5 ms90° (center)7.5%
2.0 ms180°10%

So the servo only ever sees a duty between 5% and 10% — the duty isn't the message, the pulse width is. A linear map from angle $\theta$ (degrees) to pulse width $w$ (microseconds) is:

$$ w(\theta) = 1000 + \frac{\theta}{180}\,(2000 - 1000) = 1000 + \frac{1000\,\theta}{180}\ \ [\mu s] $$

You could bit-bang this by hand, but generating a precise 1–2 ms pulse every 20 ms while doing other work is exactly what a timer is for. The Servo library commandeers Timer1 to do it cleanly:

sweep.ino C++ / Arduino
#include <Servo.h>

Servo arm;            // a Servo object manages one output pin via Timer1

void setup() {
  arm.attach(9);      // signal wire on pin 9 (3-wire: power, ground, signal)
  // arm.attach(9, 1000, 2000);  // optional: set your servo's true min/max us
}

void loop() {
  for (int deg = 0; deg <= 180; deg++) {
    arm.write(deg);   // library converts deg -> pulse width and runs the timer
    delay(15);        // give the servo time to physically move
  }
  for (int deg = 180; deg >= 0; deg--) {
    arm.write(deg);
    delay(15);
  }
}
⚠️ Pitfall — power the servo from the rail, not the pin

The servo's signal wire is a control wire (fine for a pin), but its motor draws hundreds of mA from the power wires. Feed those from a proper 5 V supply and tie its ground to the Arduino's ground (a common ground is mandatory, or the pulse has no reference). Powering a servo off the Uno's 5 V pin will brown out the board and reset it mid-sweep. Also note: the Servo library uses Timer1, so analogWrite() on pins 9 and 10 stops working while a servo is attached.

7Making it truly analog: the RC low-pass filter

② Math recovering a real DC level

Sometimes you need an actual smooth voltage, not a square wave — to bias a transistor, feed an op-amp, or drive a circuit that can't tolerate switching. You can convert PWM into a real analog DC level by physically averaging it with a resistor–capacitor low-pass filter: the resistor and capacitor form a bucket that fills and drains slowly, smoothing the pulses into their mean.

A first-order RC low-pass passes DC freely and attenuates everything above its cutoff frequency, where the output has dropped to $1/\sqrt{2}$ (−3 dB) of the input:

$$ f_c = \frac{1}{2\pi R C} $$

The design rule is simple: put the cutoff well below the PWM frequency so the switching ripple is crushed, but high enough that your signal can still change reasonably fast. For 490 Hz PWM, aim a decade lower, around 50 Hz. With $R = 10\,\text{k}\Omega$:

$$ C = \frac{1}{2\pi R f_c} = \frac{1}{2\pi (10^4)(50)} \approx 3.2\times10^{-7}\,\text{F} = 0.32\,\mu\text{F} $$

A standard 0.33 µF cap is perfect. The result on your meter: a steady $D\,V_{cc}$ with only a few millivolts of leftover ripple. You've built a crude 8-bit DAC from one resistor and one capacitor.

🛠️ The ripple-vs-speed tradeoff

Lower $f_c$ (bigger R or C) = smoother output but sluggish response — the voltage takes many milliseconds to settle after you change the duty (one time constant is $\tau = RC$). Higher $f_c$ = snappier but with more visible ripple. There's no free lunch: a passive RC trades ripple against response time. If you need both, that's when you reach for a real DAC chip (e.g. an MCP4725 over I²C — coming in a later chapter).

Q1 Conceptual check
You raise the duty cycle of a PWM signal driving an LED from 25% to 75%. What physically changes at the pin?
  • The pin's high level rises from ~1.25 V to ~3.75 V
  • The pin still snaps between 0 V and 5 V, but spends a larger fraction of each period high
  • The PWM frequency increases threefold
  • A DAC inside the chip outputs a smooth 3.75 V
PWM never changes the levels — they're always 0 V and $V_{cc}$. Duty changes the width of the high pulse, so the average rises to $0.75\times5 = 3.75\,\text{V}$ even though the instantaneous voltage is still pure digital. The eye, not the pin, does the averaging.
Q2 Compute it
An Uno ($V_{cc}=5\,\text{V}$) outputs PWM at a 40% duty cycle. What is the average voltage, in volts? (Use $V_{\text{avg}} = D\,V_{cc}$.)
V
$V_{\text{avg}} = D\,V_{cc} = 0.40 \times 5\,\text{V} = 2.0\,\text{V}$. As a sanity check, the analogWrite value for 40% is $\operatorname{round}(0.40\times255)=102$, and indeed $102/255\times5 \approx 2.0\,\text{V}$. To instead command 3.0 V you'd use $\operatorname{round}(3/5\times255)=153$.
Q3 Coding challenge — a breathing LED
Make an LED "breathe": brightness should rise and fall smoothly like slow breaths, on a roughly 4-second cycle. Use a sine wave for the smooth motion, and apply gamma correction so the dim end looks as gradual as the bright end. Pin 9, no delay()-free constraints — keep it simple with millis()-style timing or a plain loop.
Hint
A sine mapped to 0..1 gives smooth oscillation: (sin(phase) + 1) / 2. Multiply by 255 to get a perceptual brightness request, then push it through your gamma8() from §4 before analogWrite. Advance phase a little each loop; a full breath is $2\pi$ radians of phase.
Show one solution
breathe.ino C++ / Arduino
#include <math.h>
const int LED = 9;

uint8_t gamma8(uint8_t in) {                 // perceptual -> duty (from §4)
  float n = in / 255.0f;
  return (uint8_t)(powf(n, 2.2f) * 255.0f + 0.5f);
}

void setup() { pinMode(LED, OUTPUT); }

void loop() {
  // phase sweeps 0 -> 2*pi; full breath ~4 s at this step/delay
  for (float phase = 0; phase < 2 * PI; phase += 0.05f) {
    float s = (sinf(phase) + 1.0f) / 2.0f;   // 0.0 .. 1.0, smooth
    uint8_t want = (uint8_t)(s * 255.0f);    // perceptual brightness
    analogWrite(LED, gamma8(want));          // gamma-corrected duty
    delay(16);                               // ~125 steps * 16 ms ~ 2 s up+down
  }
}

The two lessons stack: the sine gives smooth physical motion, and gamma makes that motion look smooth to a nonlinear eye. Drop the gamma8() call and you'll see the breath spend too long looking "bright" and snap through the dim end — the same lopsidedness from §4, now obvious in motion.

🎓 Chapter recap

You can now explain PWM as fast on/off switching averaged by the load, derive $V_{\text{avg}} = D\,V_{cc}$ and the analogWrite mapping $D = \texttt{val}/255$, and recall the Uno's ~490 Hz / ~980 Hz pin split. You know PWM is generated by a timer comparing a counter to a threshold (the seed of Chapter 5), why dimming needs gamma correction, why a motor demands a MOSFET plus a mandatory flyback diode, how a 50 Hz / 1–2 ms frame aims a servo, and how an RC filter with $f_c = 1/(2\pi RC)$ recovers a true analog voltage.
