Communication: UART, I²C & SPI

An MCU is only as useful as the things it can talk to. Sensors, displays, SD cards, radios, other microcontrollers — all of them speak one of three buses. Learn these three and you can wire your chip to almost anything on Earth. We build each from its wires up, do the timing math, and end by running all three at once.

⏱ ~50 min🧠 Bus protocols⚡ Multi-device🔧 Wire & SPI libs

1Parallel vs serial: trading wires for time

① Intuition build the picture first

You have a byte — eight bits — and you need to move it from chip A to chip B. There are exactly two ways. Parallel: run eight wires, slap all eight bits on at once, done in a single clock tick. Serial: run one wire and shove the bits down it one after another, eight ticks for one byte. Parallel is faster per clock but eats pins and PCB area; serial is slower per clock but sips pins and routes beautifully.

💡 The fundamental trade

Every bus in this chapter is serial: it trades time (more clock ticks per byte) for wires (fewer physical pins). On an MCU with ~20 usable pins, wires are the scarce resource — so serial wins almost everywhere.

"But parallel is faster!" — it was, in the 1990s, for short cables. Then clock speeds climbed and parallel hit a wall: at high frequency the eight wires no longer stay in step (skew) and crosstalk wrecks the data. That is why USB, SATA, PCIe, HDMI, and Ethernet are all serial today. On a microcontroller the three serial buses you must know are UART, I²C, and SPI. They differ in how many wires they use, whether they carry a clock, and how they pick which device to talk to.

2UART: the asynchronous workhorse

① Intuition no clock wire — just agree on speed

UART (Universal Asynchronous Receiver/Transmitter) is the simplest bus and the one you already use every time you call Serial.print(). It is point-to-point: exactly two devices, connected by two data wires — TX (transmit) and RX (receive) — crossed over so each one's TX feeds the other's RX. The word asynchronous is the whole trick: there is no shared clock wire. Instead both sides agree in advance on a speed — the baud rate — and each samples the line on its own internal clock.

🔑 Framing: the 8N1 contract

Because there is no clock, the receiver needs to know when a byte begins. So every byte is wrapped in a frame. The idle line sits HIGH; a falling edge is the start bit (1 bit, LOW). Then come 8 data bits (LSB first), optionally a parity bit, then 1 stop bit (HIGH). The common config "8N1" means 8 data bits, No parity, 1 stop bit → 10 bits on the wire to carry 8 bits of payload.

② Math baud, bit time & throughput

The baud rate is the number of signalling events — here, bits — per second. The duration of one bit on the wire, the bit time, is the reciprocal:

$$ t_{\text{bit}} = \frac{1}{\text{baud}} $$

At the ubiquitous $115200$ baud that is $t_{\text{bit}} = 1/115200 \approx 8.68\,\mu\text{s}$; at the old-school $9600$ baud it balloons to $104.2\,\mu\text{s}$ per bit. Now the key insight: 8N1 spends 10 wire-bits per byte (start + 8 + stop), so the useful byte throughput is

$$ \text{throughput} = \frac{\text{baud}}{10} \;\text{bytes/s} \qquad\Rightarrow\qquad \frac{115200}{10} = 11{,}520\ \text{B/s} $$

And the time to send a buffer of $N$ bytes is simply

$$ t = \frac{10\,N}{\text{baud}} \qquad\Rightarrow\qquad \frac{10 \times 64}{115200} \approx 5.56\,\text{ms for 64 bytes} $$

That ÷10 is the framing overhead — 20% of your wire time carries no payload. It is the price of having no clock wire. Notice baud and bytes/s are not the same number; mixing them up is the classic rookie error.

③ Code read every line
uart_echo.ino C++ / Arduino
void setup() {
  // Open the hardware UART at 115200 baud, 8N1 (the default frame).
  // On the Uno this is the same UART wired to USB, so the Serial
  // Monitor on your PC is the "other device" on the bus.
  Serial.begin(115200);
  Serial.println("Type a number, I'll double it:");
}

void loop() {
  // Serial.available() returns how many bytes have arrived in the
  // 64-byte RX buffer the core fills via an interrupt.
  if (Serial.available() > 0) {
    // parseInt() blocks until a full integer has been received,
    // then converts the ASCII digits to a long.
    long n = Serial.parseInt();
    Serial.print("You sent ");
    Serial.print(n);              // print = send ASCII over TX
    Serial.print(", doubled = ");
    Serial.println(n * 2);        // println adds "\r\n"
  }
}
⚠️ Pitfall — baud mismatch & missing GND

If the two ends disagree on baud, the receiver samples at the wrong moments and you get garbage characters (the tell-tale "Ÿ¿Œ" salad). Set both ends identically. And UART needs a shared ground: TX/RX are voltages relative to GND, so without a common reference the bits are meaningless. Two un-grounded boards "talking" UART is the #1 silent failure.

3I²C: two wires, a whole crowd of devices

① Intuition a shared party line with addresses

UART is great for two devices, but it can't host a crowd. Enter I²C (Inter-Integrated Circuit, "eye-squared-see"). It uses just two wires shared by every device on the bus: SDA (serial data) and SCL (serial clock). Unlike UART, I²C does carry a clock — that is what makes it synchronous — and one device, the controller (historically "master"), drives that clock and starts every transaction. The other devices are targets ("slaves") that respond only when called by name.

🔑 Addresses: how 100+ chips share 2 wires

Every target has a 7-bit address baked in by its manufacturer (sometimes a few pins let you tweak the low bits). Seven bits → $2^7 = 128$ slots, of which 16 are reserved, leaving up to 112 usable addresses — call it "~127 devices" colloquially. The controller broadcasts an address; only the target that matches answers. It's a party line where you shout a name and one person replies.

The clever part is the electrical design: both lines are open-drain. A device can only pull a line down to 0; it can never drive it high. Two external pull-up resistors (typically 4.7 kΩ) gently pull both lines up to $V_{cc}$ when nobody is pulling down. This means any number of devices can share the wire without ever fighting (no two outputs shorting), and it gives I²C its signature feature: ACK/NACK. After each byte the receiver pulls SDA low for one clock to say "got it" (ACK); letting it float high means "no" (NACK).

② Math pull-up sizing & bus speed

I²C runs at 100 kHz (Standard mode) or 400 kHz (Fast mode); some parts go to 1 MHz+. The pull-up value is a real engineering trade-off. The line is an RC circuit: bus capacitance $C_b$ (wires + every device's pin capacitance) charges through the pull-up $R_p$. The rise time must fit inside a clock period. The spec caps it at

$$ t_r \approx 0.847\,R_p\,C_b \;\le\; \begin{cases} 1000\ \text{ns} & \text{(100 kHz)}\\ 300\ \text{ns} & \text{(400 kHz)} \end{cases} $$

With a typical $C_b = 100\,\text{pF}$, hitting the 300 ns Fast-mode limit needs $R_p \le t_r / (0.847\,C_b) \approx 3.5\,\text{k}\Omega$ — which is why 4.7 kΩ works at 100 kHz but you drop to ~2.2 kΩ for 400 kHz or long buses. Too big a resistor → sluggish edges → corrupt data; too small → the open-drain transistor can't sink enough current to reach a valid LOW. There's a window, and bus capacitance sets it.

🏭 Real case — one bus, three chips, zero extra pins

That diagram is a real, shippable design: a temperature sensor (TMP102 at 0x48), a real-time clock (DS3231 at 0x68), and an OLED display (SSD1306 at 0x3C) all hang off the same two pins (A4/A5 on an Uno). Add a fourth or fifth device and you still use only two pins. That is why I²C dominates sensor breakout boards — Adafruit and SparkFun ship hundreds of them.

③ Code read a register over the Wire library
i2c_read_temp.ino C++ / Arduino
#include <Wire.h>

const uint8_t TMP102_ADDR = 0x48;  // 7-bit target address
const uint8_t TEMP_REG    = 0x00;  // register holding temperature

void setup() {
  Serial.begin(115200);
  Wire.begin();                    // join the bus as controller
}

void loop() {
  // 1) Tell the sensor WHICH register we want to read from.
  Wire.beginTransmission(TMP102_ADDR);
  Wire.write(TEMP_REG);            // set the internal register pointer
  Wire.endTransmission(false);     // false = repeated-START, keep bus

  // 2) Now request 2 bytes back from that register.
  Wire.requestFrom(TMP102_ADDR, (uint8_t)2);
  uint8_t msb = Wire.read();       // most-significant byte first
  uint8_t lsb = Wire.read();

  // TMP102 packs a 12-bit value, left-justified in 16 bits.
  int16_t raw = ((msb << 8) | lsb) >> 4;
  float celsius = raw * 0.0625f;   // 0.0625 °C per LSB

  Serial.print("Temp: "); Serial.print(celsius); Serial.println(" C");
  delay(1000);
}
⚠️ Pitfall — 7-bit vs 8-bit addresses

Datasheets love to confuse you. The Wire library wants the 7-bit address (0x48) and appends the read/write bit itself. Many datasheets instead print the 8-bit address with that bit already shifted in (0x90 write / 0x91 read). If your device is silent, check whether you fed Wire a left-shifted address. When in doubt, run an i2c_scanner sketch — it pings all 127 addresses and reports who ACKs.

4SPI: four wires, full speed ahead

① Intuition raw speed, full duplex

When you need fast — streaming pixels to a display, reading kilobytes off an SD card, talking to a radio — you reach for SPI (Serial Peripheral Interface). It spends more wires to buy more speed. Four signals: SCK (clock, from the controller), MOSI (Controller-Out Peripheral-In), MISO (Controller-In Peripheral-Out), and one CS / SS (Chip Select) per device. Data flows both directions simultaneously — that's full duplex — and there is no addressing protocol: you select a chip by pulling its CS line LOW.

🔑 The shift-register mental model

SPI is literally two shift registers connected in a ring. On each SCK edge, the controller and peripheral each shift one bit out (MOSI / MISO) and one bit in. After 8 clocks they have swapped a full byte. So every SPI transfer is an exchange: to read, you must also write (often a dummy byte). That symmetry is why SPI has no ACK and no flow control — it's just clocked bit-swapping, which is exactly why it's fast (tens of MHz, vs I²C's 0.4 MHz).

The cost: one dedicated CS pin per peripheral. Three SPI devices = SCK + MOSI + MISO shared + 3 CS = 6 pins, vs I²C's eternal 2. SPI trades pins for speed; I²C trades speed for pins. Pick based on which resource you're short on.

② Math the four SPI modes (CPOL/CPHA)

SPI's one subtlety is timing polarity. Two bits define the clock's behaviour: CPOL (clock polarity — is idle HIGH or LOW?) and CPHA (clock phase — sample on the first or second edge?). Their four combinations are the four SPI modes:

$$ \text{Mode} = 2\cdot\text{CPOL} + \text{CPHA}, \qquad \text{CPOL},\text{CPHA}\in\{0,1\} $$

The transfer time is brutally simple — no framing overhead like UART, no ACK like I²C: $t = 8N / f_{\text{SCK}}$. At $f_{\text{SCK}} = 8\,\text{MHz}$, a 512-byte SD block takes $8 \times 512 / 8{\times}10^6 = 512\,\mu\text{s}$ — about 11× faster than the same payload over 400 kHz I²C. Match the mode to the datasheet or you'll sample bits at the wrong edge and read pure noise.

③ Code configure a transaction
spi_read.ino C++ / Arduino
#include <SPI.h>

const int CS_PIN = 10;             // chip-select for this peripheral

void setup() {
  pinMode(CS_PIN, OUTPUT);
  digitalWrite(CS_PIN, HIGH);     // HIGH = deselected (idle)
  SPI.begin();                    // sets SCK, MOSI, MISO directions
}

uint8_t readRegister(uint8_t reg) {
  // A transaction locks the bus speed, bit order and mode so that
  // two SPI devices with different settings can coexist safely.
  SPI.beginTransaction(SPISettings(8000000, MSBFIRST, SPI_MODE0));
  digitalWrite(CS_PIN, LOW);      // select THIS device
  SPI.transfer(reg | 0x80);       // many parts: bit7=1 means "read"
  uint8_t value = SPI.transfer(0x00);  // send dummy, clock the reply in
  digitalWrite(CS_PIN, HIGH);     // deselect
  SPI.endTransaction();
  return value;
}

void loop() { /* call readRegister(...) as needed */ }
🛠️ Why SPI.transfer both sends and receives

There is no separate "read" call. SPI.transfer(x) shifts x out on MOSI and returns whatever the peripheral shifted back on MISO — one byte each direction, same 8 clocks. To pull data from a device you clock out a throwaway 0x00 so the SCK keeps ticking and the reply rides in on MISO. Full duplex isn't a feature you opt into; it's just how the wire works.

🔑 A one-line decision rule

Lots of slow sensors and pins are scarce? I²C. One fast device — display, SD, radio? SPI. Talking to a PC, a GPS, or another board over a cable? UART. Real products use all three simultaneously, each for what it's best at.

6Real case: all three buses, one datalogger

④ Real case this ships in products

Here is a genuine board you could build this weekend: an environmental datalogger. A temperature sensor on I²C, an SD card on SPI, and debug output on UART — three buses running at the same time, each playing to its strength. The MCU is the hub; nothing conflicts because each bus owns its own pins.

datalogger.ino C++ / Arduino
#include <Wire.h>      // I2C: the temperature sensor
#include <SPI.h>       // SPI: the SD card (used by SD.h under the hood)
#include <SD.h>

const uint8_t TMP102 = 0x48;
const int     SD_CS  = 4;        // SD breakout chip-select pin

void setup() {
  Serial.begin(115200);          // UART  → debug to the PC
  Wire.begin();                  // I2C   → sensor bus
  if (!SD.begin(SD_CS)) {        // SPI   → SD card (SD lib drives SPI)
    Serial.println("SD init failed!");
    while (1);                    // halt: no point logging with no disk
  }
  Serial.println("Logger ready.");
}

float readTempC() {
  Wire.beginTransmission(TMP102);
  Wire.write(0x00);
  Wire.endTransmission(false);
  Wire.requestFrom(TMP102, (uint8_t)2);
  int16_t raw = ((Wire.read() << 8) | Wire.read()) >> 4;
  return raw * 0.0625f;
}

void loop() {
  float t = readTempC();                       // ← from I2C
  File f = SD.open("log.csv", FILE_WRITE);      // ← over SPI
  if (f) { f.print(millis()); f.print(','); f.println(t); f.close(); }
  Serial.print("Logged "); Serial.println(t);   // ← over UART
  delay(2000);
}
⚠️ Pitfall — bus contention & voltage levels

Three buses coexist only because they're on different pins — never wire an SPI MISO onto an I²C SDA. Two subtler traps: (1) on an Uno, pins D11–13 are the SPI bus, so don't reuse them for something else; (2) watch voltage levels — a 3.3 V SD card on a 5 V Uno needs a level shifter on MOSI/SCK or you'll slowly cook it. Mixed-voltage buses are the silent killers of hobby projects.

Q1 Conceptual check
You need to hang eight slow temperature sensors off one MCU while burning as few pins as possible. Which bus, and how many wires does it take for all eight?
  • SPI — it's the fastest, so always use it (10 wires: 3 shared + 8 CS)
  • I²C — 2 wires total; each sensor has its own 7-bit address on the shared bus
  • UART — 2 wires per sensor, so 16 wires
  • SPI — 2 wires, just like I²C
For many slow devices on few pins, I²C is the answer: all eight sensors share the same 2 wires (SDA + SCL), each distinguished by its 7-bit address. SPI would need a separate CS pin per device (3 shared + 8 = 11 pins); UART is point-to-point so it can't bus them at all. Speed isn't the constraint here — pin count is.
Q2 Compute it
You send a 64-byte packet over UART at 115200 baud, 8N1. How long does the transmission take, in milliseconds? (Recall 8N1 uses 10 wire-bits per byte, so $t = \dfrac{10N}{\text{baud}}$.)
ms
$t = \dfrac{10 \times 64}{115200} = \dfrac{640}{115200} \approx 5.56\times10^{-3}\,\text{s} = 5.56\,\text{ms}$. Equivalently the throughput is $115200/10 = 11{,}520$ B/s, and $64 / 11520 \approx 5.56$ ms. The factor of 10 (not 8) is the framing overhead — start + 8 data + stop bits.
Q3 Coding challenge — read one I²C register
Write a function uint8_t readReg(uint8_t addr, uint8_t reg) that, using the Wire library, reads a single 8-bit register from an I²C device: point the device's register pointer at reg, then request and return one byte. This is the universal "read a register" pattern for nearly every I²C chip.
Hint
The pattern is two phases. Write phase: beginTransmission(addr), write(reg), then endTransmission(false) — the false sends a repeated START instead of releasing the bus. Read phase: requestFrom(addr, 1), then Wire.read() returns the byte.
Show one solution
read_reg.ino C++ / Arduino
#include <Wire.h>

// Read one 8-bit register from any I2C device.
uint8_t readReg(uint8_t addr, uint8_t reg) {
  Wire.beginTransmission(addr);   // start a write to the device
  Wire.write(reg);                // set its internal register pointer
  Wire.endTransmission(false);    // repeated-START: keep the bus

  Wire.requestFrom(addr, (uint8_t)1);   // ask for 1 byte
  return Wire.available() ? Wire.read() : 0;
}

void setup() {
  Serial.begin(115200);
  Wire.begin();
  uint8_t whoAmI = readReg(0x68, 0x75); // MPU-6050 WHO_AM_I → 0x68
  Serial.print("WHO_AM_I = 0x");
  Serial.println(whoAmI, HEX);
}

void loop() {}

The endTransmission(false) is the crux: a plain true would send a STOP, freeing the bus, and some devices reset their pointer on STOP. The repeated-START keeps the conversation atomic. Reading a WHO_AM_I register first is the classic sanity check — if it returns the datasheet's magic number, your wiring and address are correct.

🎓 Chapter recap

You can now reason about serial vs parallel as a wires-for-time trade, and you know the three buses cold. UART: 2 wires, no clock, agree on baud, 10 bits/byte, throughput = baud/10. I²C: 2 shared wires with pull-ups, 7-bit addresses for ~100+ devices, ACK/NACK, 100k/400 kHz. SPI: 4 wires, full-duplex shift-register exchange, one CS per device, MHz-fast, four CPOL/CPHA modes. You wrote register reads over Wire and SPI, sized pull-ups, computed transmission times, and ran all three buses on one datalogger.
