The Analog World: ADC, Sensors & Signal Conditioning

So far your pins have lived in a binary universe — HIGH or LOW, on or off. But temperature, light, pressure, and battery voltage are continuous. To read them you need the chip's most under-appreciated peripheral: the analog-to-digital converter. We'll build it from intuition, derive its precision to the millivolt, and wire up real sensors that ship in real products.

1Analog vs digital: the world is a smooth curve

① Intuition build the picture first

A digital signal has exactly two states. An analog signal can be any value in a range, varying smoothly through time — the voltage from a microphone, the slow drift of a temperature sensor, the position of a knob. Nature does not deal in HIGH and LOW; it deals in infinitely fine gradations.

Your 8-bit AVR, however, only understands numbers. It cannot hold "3.27 volts" as a continuous quantity — it must convert that voltage into an integer it can store and do math on. The peripheral that performs this translation is the ADC (analog-to-digital converter), and analogRead() is your one-line door to it.

💡 The one-sentence model

An ADC is a ruler for voltage. It lays a fixed number of evenly spaced tick marks across its measurement range and reports which tick your input voltage is closest to — as a plain integer.

That word "closest" is the whole story. A ruler with millimetre ticks cannot tell you something is 4.37 mm — it rounds to 4 mm. An ADC does the same to voltage. This rounding is called quantization, and understanding it is the difference between trusting your sensor data and being fooled by it.

🔑 Two pin worlds

The Uno's pins labelled D0–D13 are digital (two states). The six pins labelled A0–A5 route to the ADC and can each be read as a continuous voltage. (They double as digital pins too — but only A0–A5 reach the converter.)

2Resolution: how many ticks on the ruler?

② Math derive the precision

An ADC's resolution is how many distinct values it can output, set by its bit count $N$. With $N$ bits you can encode

$$ \text{levels} = 2^{N} $$

The ATmega328P's ADC is 10-bit, so it produces

$$ 2^{10} = 1024 \text{ levels}, \quad \text{numbered as codes } 0,1,2,\dots,1023 $$

Code 0 means "voltage at or below the bottom of the range"; code 1023 means "at or above the top." Everything in between maps to one of the 1022 interior ticks. The size of one tick — the smallest voltage change the ADC can resolve — is the LSB (least-significant bit), found by dividing the full range by the number of steps:

$$ V_{\text{LSB}} = \frac{V_{\text{ref}}}{2^{N}} = \frac{V_{\text{ref}}}{1024} $$

With the Uno's default reference of $V_{\text{ref}} = 5\,\text{V}$:

$$ V_{\text{LSB}} = \frac{5\,\text{V}}{1024} \approx 4.883\,\text{mV} $$

That is your floor: the Uno cannot distinguish two voltages closer than ~4.9 mV apart. Want finer resolution? Either shrink $V_{\text{ref}}$ (a 1.1 V internal reference gives ~1.07 mV/LSB but only over 0–1.1 V) or move to a chip with a 12- or 16-bit ADC.

∑ Quantization error is bounded

Because the ADC rounds to the nearest tick, any single reading is off by at most half a tick. The worst-case quantization error is

$$ \varepsilon_{q} = \pm\tfrac{1}{2}V_{\text{LSB}} = \pm\frac{V_{\text{ref}}}{2048} \approx \pm 2.44\,\text{mV} $$

This error is inherent — it exists even with a perfect, noise-free input. It is the price of representing a smooth world with integers.

3From code to volts: analogRead() and AREF

② Math invert the mapping

analogRead(pin) hands you the integer code, not a voltage. To recover the voltage you invert the ruler. The subtle bit: although there are 1024 levels, the full-scale reading spans 1023 steps from code 0 to code 1023, so the Arduino convention maps the code linearly onto $[0, V_{\text{ref}}]$ with

$$ V = \frac{\text{adc}}{1023}\, V_{\text{ref}} $$

So a reading of 512 at $V_{\text{ref}} = 5\,\text{V}$ is $\tfrac{512}{1023}\times 5 \approx 2.503\,\text{V}$ — almost exactly mid-scale, as you'd expect. (You'll meet this exact computation again in the exercises.)

③ Code read every line
read_voltage.ino C++ / Arduino
const uint8_t SENSE_PIN = A0;     // analog input A0
const float   VREF      = 5.0;    // reference voltage (Uno default)

void setup() {
  Serial.begin(9600);
  // Analog pins do NOT need pinMode for analogRead — the ADC
  // multiplexer handles selection. But leave them un-driven.
}

void loop() {
  int   raw     = analogRead(SENSE_PIN);   // 0..1023, takes ~104 µs
  float volts   = (raw / 1023.0) * VREF;   // invert the quantizer

  Serial.print(raw);
  Serial.print(F("  ->  "));
  Serial.print(volts, 3);                  // 3 decimal places
  Serial.println(F(" V"));
  delay(200);
}
⚠️ Pitfall — integer division eats your reading

Write raw / 1023 (both integers) and C++ does integer division: any code below 1023 becomes 0. You must force floating point with raw / 1023.0 (note the .0) so the fractional voltage survives. This silent bug bites nearly every beginner exactly once.

What is $V_{\text{ref}}$ physically? It's the voltage the ADC compares against — its "top of ruler." On the Uno you choose it with analogReference():

⚠️ Pitfall — AREF short-circuit

If you wire an external voltage to the AREF pin and then call analogReference(EXTERNAL), you're fine. Do it in the other order — leaving the default internal reference connected while driving AREF — and you short the internal source to your external one. Always select EXTERNAL before the first analogRead().

4How fast can it sample? Prescaler & Nyquist

② Math where the speed comes from

A conversion is not instant — the ADC uses successive approximation, essentially a binary search that halves a comparison voltage ten times to pin down all ten bits. It runs from its own clock, derived from the 16 MHz system clock through a prescaler. The AVR datasheet specifies the ADC clock should sit between 50 kHz and 200 kHz for full 10-bit accuracy, so the Arduino core picks a prescaler of 128:

$$ f_{\text{ADC}} = \frac{f_{\text{clk}}}{128} = \frac{16\,\text{MHz}}{128} = 125\,\text{kHz} $$

One normal conversion takes 13 ADC clock cycles. So the time per sample and the resulting throughput are

$$ t_{\text{conv}} = \frac{13}{f_{\text{ADC}}} = \frac{13}{125\,000} \approx 104\,\mu\text{s}, \qquad f_{s} = \frac{f_{\text{ADC}}}{13} \approx 9.6\,\text{ksps} $$

That's it — roughly 9600 samples per second at the default settings, the number behind the ~104 µs comment in the code above. You can drop the prescaler to 16 (1 MHz ADC clock) for ~77 ksps if you accept slightly noisier low bits — a common trick for audio-rate sampling.

🔑 Nyquist in one line

To reconstruct a signal of frequency $f_{\text{sig}}$ you must sample faster than twice it:

$$ f_{s} > 2\,f_{\text{sig}} $$

At 9.6 ksps the Uno can faithfully capture signals up to ~4.8 kHz. Feed it anything faster and it aliases — a 10 kHz tone masquerades as a slow phantom wave. That's why audio gear puts a low-pass "anti-aliasing" filter before the ADC.

5The voltage divider: your universal sensor interface

① Intuition turn resistance into voltage

Most cheap sensors don't output a voltage — they change their resistance: an LDR gets less resistive in light, a thermistor less resistive when warm. But the ADC only reads voltage. The bridge between the two is the most-used circuit in all of electronics: the voltage divider.

Put two resistors in series across your supply. The midpoint between them sits at a voltage set by their ratio. If $R_1$ is on top (toward Vin) and $R_2$ on the bottom (toward ground), the midpoint voltage is

② Math derive it from Ohm's law
$$ V_{\text{out}} = V_{\text{in}}\,\frac{R_2}{R_1 + R_2} $$

Why? The same current $I = V_{\text{in}}/(R_1+R_2)$ flows through both resistors (series). The voltage across the bottom resistor is $I\,R_2$; substitute $I$ and the $R_1+R_2$ denominator appears. Sanity-check the extremes: if $R_2 \to 0$ the output is 0 V; if $R_2 \to \infty$ the output approaches the full $V_{\text{in}}$. The ratio simply slides the midpoint up and down.

That chain — sensor → divider → ADC → code → physical units — is the skeleton of nearly every analog measurement you'll ever build. Master it once and every resistive sensor becomes the same problem.

🛠️ Picking the fixed resistor

For best sensitivity, choose the fixed resistor near the sensor's resistance at the value you care most about. A 10 kΩ thermistor at room temperature pairs beautifully with a 10 kΩ fixed resistor — the divider output then swings most steeply right where you're measuring.

6Three real sensors: LDR, thermistor & TMP36

③ Code and the math behind each

The LDR (light). A light-dependent resistor drops from megohms in the dark to a few hundred ohms in bright light. Wire it as the top resistor of a divider with a fixed 10 kΩ to ground; brighter light raises the ADC reading. No fancy math — you just map the code to "darker/lighter" or calibrate against known lux.

The NTC thermistor (temperature, nonlinear). Its resistance falls as it heats, but exponentially, not linearly. The simple Beta model relates its resistance $R$ to temperature in kelvin:

$$ \frac{1}{T} = \frac{1}{T_0} + \frac{1}{B}\ln\!\left(\frac{R}{R_0}\right) $$

Here $R_0$ is the rated resistance at reference temperature $T_0$ (usually 25 °C = 298.15 K), and $B$ is the datasheet "Beta" constant (~3950 K for common 10 kΩ parts). You read the divider, solve for the thermistor's $R$, then plug into the equation. For higher accuracy the full Steinhart–Hart equation adds two more coefficients, but Beta is within a fraction of a degree over a normal room-to-oven range.

thermistor.ino C++ / Arduino
#include <math.h>

const float VREF   = 5.0;
const float R_FIX  = 10000.0;   // 10 kΩ fixed resistor (to GND)
const float R0     = 10000.0;   // thermistor 10 kΩ @ 25 °C
const float T0     = 298.15;    // 25 °C in kelvin
const float BETA   = 3950.0;    // datasheet Beta constant (K)

void setup() { Serial.begin(9600); }

void loop() {
  int   raw   = analogRead(A0);
  float v     = (raw / 1023.0) * VREF;

  // Thermistor is the TOP resistor; R_FIX to GND is the bottom.
  // Vout = VREF * R_FIX / (R_therm + R_FIX)  ->  solve for R_therm:
  float rTherm = R_FIX * (VREF / v - 1.0);

  // Beta equation: 1/T = 1/T0 + (1/B)·ln(R/R0)
  float invT  = 1.0 / T0 + (1.0 / BETA) * log(rTherm / R0);
  float tempC = (1.0 / invT) - 273.15;   // kelvin -> Celsius

  Serial.print(tempC, 2);
  Serial.println(F(" C"));
  delay(500);
}

The TMP36 (temperature, linear and easy). This active sensor does the hard work internally and outputs a clean voltage: 500 mV at 0 °C, rising a tidy 10 mV per °C. No divider, no logarithms — just invert a line:

$$ T_{\circ C} = \frac{V_{\text{out}} - 0.5\,\text{V}}{0.01\,\text{V}/^{\circ}\text{C}} $$
tmp36.ino C++ / Arduino
const float VREF = 5.0;

void setup() { Serial.begin(9600); }

void loop() {
  int   raw   = analogRead(A0);
  float v     = (raw / 1023.0) * VREF;       // sensor output, volts
  float tempC = (v - 0.5) / 0.01;            // 500 mV @ 0°C, 10 mV/°C

  Serial.print(tempC, 1);
  Serial.println(F(" C"));
  delay(500);
}
⚠️ Pitfall — the 5 V reference wobbles

Every formula above assumes $V_{\text{ref}}$ is exactly 5.000 V. Powered over USB, your "5 V" rail can sag to 4.7 V under load, skewing every reading by ~6%. For temperature work, switch to analogReference(INTERNAL) (the rock-stable 1.1 V bandgap) or measure your actual VCC and use it as $V_{\text{ref}}$ — a trick we use next.

7Noise & averaging: buying bits with √N

② Math signal-to-noise scaling

Raw ADC readings jitter. Electrical noise, a noisy supply, and the quantization floor itself make consecutive samples of a steady voltage dance by a count or two. The cheapest cure is oversampling: take many readings and average them.

If the noise is random and uncorrelated, averaging $N$ samples leaves the signal unchanged but shrinks the noise's standard deviation by $\sqrt{N}$. So the signal-to-noise ratio improves as

$$ \text{SNR}_{\text{avg}} = \sqrt{N}\;\cdot\;\text{SNR}_{\text{single}} $$

Averaging 16 samples cuts noise by $\sqrt{16}=4\times$; 64 samples by $8\times$. There's a catch baked into the square root — diminishing returns. Going from 16 to 64 samples (4× the work) only doubles your noise reduction. Each extra bit of effective resolution costs $4\times$ the samples, since $\sqrt{4}=2$.

averaged_read.ino C++ / Arduino
// Read a pin N times and return the mean as a float code.
float readAveraged(uint8_t pin, uint8_t n) {
  uint32_t sum = 0;                  // 32-bit: 64 × 1023 won't overflow
  for (uint8_t i = 0; i < n; i++) {
    sum += analogRead(pin);
    // a tiny settling delay helps after switching ADC channels
    delayMicroseconds(50);
  }
  return (float)sum / n;             // averaged code, 0..1023
}

void loop() {
  float code  = readAveraged(A0, 64);     // 64-sample mean
  float volts = (code / 1023.0) * 5.0;
  Serial.println(volts, 4);
  delay(100);
}
🛠️ Hardware beats software

Software averaging helps, but kill noise at the source first: solder a 0.1 µF decoupling capacitor from the analog pin (or AREF) to ground. It forms a low-pass filter with the sensor's resistance, swallowing high-frequency hash before it ever reaches the sample-and-hold. One cheap ceramic cap often does more than 64 averaged reads.

8Real case: monitoring a battery you can't read directly

④ Real case this ships in products

Here's a problem every battery-powered product faces. A 2-cell LiPo pack sits around 8.4 V fully charged — but the ADC tops out at 5 V. Feed 8.4 V into A0 and you'll let the magic smoke out of the chip. The fix is exactly the divider you just learned: scale the battery voltage down into the ADC's safe window, then scale the number back up in software.

Choose $R_1 = 100\,\text{k}\Omega$ (top, to the battery) and $R_2 = 47\,\text{k}\Omega$ (bottom, to ground). At a full 8.4 V the divider output is

$$ V_{\text{out}} = 8.4 \times \frac{47\text{k}}{100\text{k}+47\text{k}} = 8.4 \times 0.320 \approx 2.69\,\text{V} $$

Comfortably under 5 V. The ratio $(R_1+R_2)/R_2 \approx 3.13$ is your software multiplier: measure $V_{\text{out}}$, multiply back up, recover the true pack voltage. Large resistor values (hundreds of kΩ) keep the divider's idle current in microamps — important when you're trying to make a battery last.

battery_monitor.ino C++ / Arduino
const uint8_t BATT_PIN = A0;
const float   VREF     = 5.0;
const float   R1       = 100000.0;   // top resistor (to battery +)
const float   R2       = 47000.0;    // bottom resistor (to GND)

// Divider gain we must undo in software:
const float DIVIDER_GAIN = (R1 + R2) / R2;   // ≈ 3.128

void setup() { Serial.begin(9600); }

void loop() {
  // average to tame noise on a long sense wire
  uint32_t sum = 0;
  for (uint8_t i = 0; i < 16; i++) sum += analogRead(BATT_PIN);
  float code   = sum / 16.0;

  float vAdc   = (code / 1023.0) * VREF;      // voltage at the pin
  float vBatt  = vAdc * DIVIDER_GAIN;         // scale back to pack voltage

  Serial.print(F("Battery: "));
  Serial.print(vBatt, 2);
  Serial.println(F(" V"));

  if (vBatt < 6.4) Serial.println(F("!! LOW — shut down soon"));
  delay(1000);
}
🏭 Real case — the potentiometer is a divider too

That knob on your dashboard? A potentiometer is a voltage divider with a moving tap. Wire the outer pins to 5 V and GND, the wiper to A0, and turning it sweeps the wiper voltage from 0 to 5 V — codes 0 to 1023. Volume knobs, joysticks, throttle grips, and motor-speed dials are all this exact circuit. You just learned the most common analog input in consumer electronics.

Q1 Conceptual check
The Uno's ADC is 10-bit with a 5 V reference, giving an LSB of ~4.88 mV. What does that LSB value actually mean for your measurements?
  • The ADC takes 4.88 ms per conversion.
  • The smallest voltage change the ADC can distinguish is ~4.88 mV; finer differences round to the same code.
  • The maximum voltage the pin can read is 4.88 mV.
  • Every reading is randomly wrong by exactly 4.88 mV.
The LSB is the voltage represented by one code step: $V_{\text{LSB}} = V_{\text{ref}}/2^{N} = 5/1024 \approx 4.88\,\text{mV}$. Two inputs closer than that map to the same integer — that's the resolution floor. Quantization error is bounded at half an LSB (±2.44 mV), and conversion time (~104 µs) is a separate property entirely.
Q2 Compute it
analogRead() returns 512 with $V_{\text{ref}} = 5\,\text{V}$. What voltage is on the pin, in volts? (Use $V = \dfrac{\text{adc}}{1023}\,V_{\text{ref}}$.)
V
$V = \dfrac{512}{1023}\times 5 = 2.503\,\text{V} \approx 2.50\,\text{V}$. Code 512 is essentially mid-scale, so it maps to half of the 5 V range — exactly what intuition predicts. (Dividing by 1024 instead of 1023 gives 2.500 V; the difference is one part in a thousand, well inside tolerance.)
Q3 Coding challenge — N-sample averaging
A light sensor on A0 jitters by ±3 codes. Write a function float smoothRead(uint8_t pin, uint16_t n) that returns the mean of n samples as a float code (0–1023), then use it to print a stable voltage. Think carefully about the data type of your running sum — what happens if you accumulate 1000 readings in an int?
Hint
A single int on the AVR is 16-bit and maxes out at 32767. Summing just 33 full-scale readings (33 × 1023 = 33759) already overflows it and wraps to garbage. Use a 32-bit accumulator (uint32_t), add each analogRead() in a loop, then divide by n as a float so the fractional part survives.
Show one solution
smooth_read.ino C++ / Arduino
// Return the mean of n samples as a fractional code (0..1023).
float smoothRead(uint8_t pin, uint16_t n) {
  uint32_t sum = 0;                 // 32-bit so big n can't overflow
  for (uint16_t i = 0; i < n; i++) {
    sum += analogRead(pin);
  }
  return (float)sum / n;            // cast BEFORE dividing → keeps fraction
}

void setup() { Serial.begin(9600); }

void loop() {
  float code  = smoothRead(A0, 64);          // √64 = 8× less noise
  float volts = (code / 1023.0) * 5.0;
  Serial.println(volts, 4);                  // 4 decimals now meaningful
  delay(100);
}

The two traps are the 32-bit sum (a 16-bit int overflows after ~32 full-scale reads) and casting to float before the division (so you don't truncate the average). With 64 samples the noise drops by $\sqrt{64}=8\times$ — now those four decimal places actually mean something instead of flickering randomly.

🎓 Chapter recap

You can now explain why the analog world must be quantized, compute an ADC's resolution and LSB from its bit count ($V_{\text{LSB}} = V_{\text{ref}}/2^{N}$), convert a raw code back to volts, and reason about sampling rate and Nyquist. You can wire a resistive sensor through a voltage divider ($V_{\text{out}} = V_{\text{in}}\,R_2/(R_1+R_2)$), linearise a TMP36 or apply the thermistor Beta equation, and beat noise down with $\sqrt{N}$ averaging and a decoupling cap. The battery monitor and potentiometer prove these aren't toys — they're the analog front-end of real products. Next we make pins produce smoothly varying outputs with PWM.
