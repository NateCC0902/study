Robust Firmware: State Machines, Non-Blocking Design & Power

A blink sketch runs for ten minutes in a demo. A product runs for years in a wall, in a field, in a patient. That gap is not luck — it is architecture. We trade the spaghetti loop() for deliberate state machines, a cooperative scheduler, rollover-proof timing, memory discipline, and a power budget that turns 2000 mAh into years of uptime.

1From a spaghetti loop() to architecture

① Intuition build the picture first

Early sketches grow by accretion: a delay() here, an if there, a flag, another flag, a nested while that "just polls until the button is released." It works on the bench. Then you add a sensor read, and suddenly the button feels mushy, the LED stutters, and a watchdog you didn't know existed reboots the board. The code didn't get harder — it got tangled. Every blocking call freezes everything else, and every loose flag is an implicit, undocumented state.

💡 The one-sentence model

Robust firmware is a set of explicit states advanced by small non-blocking steps, on a clock you trust. Nothing waits. Nothing hides. Every "mode" the device can be in has a name.

Three disciplines carry us the rest of the way, and this chapter takes them one at a time:

  • Finite state machines — name every mode, and the loop becomes a single switch.
  • Cooperative scheduling — run many "tasks" on different intervals with one millis() clock and zero delay().
  • Resource discipline — memory and power are budgets. Spend them on purpose, measure what's left.
⚠️ Pitfall — delay() is a coma, not a pause

delay(1000) doesn't "wait" — it spins the CPU in a busy loop for a billion-ish cycles, ignoring buttons, serial bytes, and every other task. One blocking call anywhere in loop() sets the responsiveness floor for the whole program. The entire chapter is, in a sense, the campaign to delete delay().

2Finite state machines: name every mode

① Intuition a device is always in exactly one mode

A traffic light is Green, Yellow, or Red — never two at once, never something else. That is a finite state machine (FSM): a finite set of states, plus rules (transitions) that move between them when events occur. Buttons, menus, charging sequences, communication protocols — almost every "it does this, then that, unless…" behavior is an FSM, whether you model it or not. Modeling it explicitly is the difference between code you can reason about and code you can only pray over.

② Math just enough formalism

Formally an FSM is the tuple $M = (S,\, \Sigma,\, \delta,\, s_0)$: a finite set of states $S$, an input alphabet (events) $\Sigma$, a transition function $\delta : S \times \Sigma \to S$, and a start state $s_0 \in S$. You never write $\delta$ as an equation — you write it as a switch. But the formalism buys you a guarantee:

$$ |S| < \infty \;\Longrightarrow\; \text{every (state, event) pair is enumerable and testable.} $$

If your device has 4 states and 3 distinct events, there are at most $|S|\times|\Sigma| = 12$ transitions to define. That's a finite checklist — exactly the kind of thing that prevents the "but what happens if the button is pressed during calibration?" class of bug.

③ Code read every line

The canonical implementation: a named enum for the states, a single state variable, and a switch that runs once per loop iteration. Each case does two things — act on the current state, and decide whether to transition.

fsm.ino C++ / Arduino
// 1) Name every mode the device can be in. The compiler now
//    forbids a state that isn't on this list.
enum class State : uint8_t { Idle, Armed, Running, Fault };

State state = State::Idle;          // current state — the ONLY mode variable
const uint8_t BTN = 2;              // button to GND, INPUT_PULLUP

void setup() {
  pinMode(BTN, INPUT_PULLUP);       // pressed == LOW
  pinMode(LED_BUILTIN, OUTPUT);
  Serial.begin(115200);
}

// A tiny edge detector: true exactly once, on the press.
bool buttonPressed() {
  static bool prev = HIGH;          // 'static' keeps value across calls
  bool now = digitalRead(BTN);
  bool edge = (prev == HIGH && now == LOW);  // HIGH→LOW == press
  prev = now;
  return edge;
}

void loop() {
  bool press = buttonPressed();     // sample events ONCE per iteration

  switch (state) {                  // this switch IS the transition fn δ
    case State::Idle:
      digitalWrite(LED_BUILTIN, LOW);
      if (press) state = State::Armed;
      break;

    case State::Armed:
      digitalWrite(LED_BUILTIN, HIGH);          // solid = armed
      if (press) state = State::Running;
      break;

    case State::Running:
      // ... do the real work here, non-blocking ...
      if (sensorFaulted()) state = State::Fault; // guarded transition
      break;

    case State::Fault:
      if (press) state = State::Idle;            // operator clears fault
      break;
  }
}
🔑 Why this is bulletproof

There is one state variable and one place it changes (inside the switch). To answer "what does the device do when X happens?" you read exactly one case. Using enum class means a typo'd or out-of-range state is a compile error, not a 3 a.m. field failure. The whole loop is non-blocking: it falls through in microseconds, so other tasks still run.

🏭 Real case — debounce is an FSM

That one-line edge detector lies on a noisy mechanical button: contacts bounce for 1–20 ms, firing dozens of false edges. Production debounce is its own little FSM — states Stable, Bouncing, Settled — gated by a millis() timer that requires the level to hold steady for, say, 25 ms before accepting it. You'll build exactly that in the exercises.

3A cooperative scheduler without an RTOS

① Intuition many clocks, one loop

Real devices juggle jobs on different rhythms: blink a heartbeat at 1 Hz, sample a sensor at 10 Hz, push a radio packet every 60 s. You don't need a real-time operating system for this. You need a table of jobs, each remembering when it last ran, stepped by a single non-blocking loop. Each job checks one cheap question — "has my interval elapsed?" — and if so, runs and resets its timer. This is cooperative multitasking: tasks must be short and must return; nobody is preempted.

③ Code the scheduler in 30 lines

A task is just {interval, lastRun, fn}. The dispatcher walks the table every loop and calls any task whose interval has elapsed.

scheduler.ino C++ / Arduino
struct Task {
  uint32_t interval;     // run every 'interval' ms
  uint32_t lastRun;      // millis() of last execution
  void (*fn)();          // pointer to the task function
};

// ---- the actual jobs (each must be short & non-blocking) ----
void heartbeat()  { digitalWrite(LED_BUILTIN, !digitalRead(LED_BUILTIN)); }
void sampleADC()  { /* read a sensor, stash the value */ }
void sendPacket() { /* assemble & transmit a frame    */ }

// ---- the schedule: interval, lastRun=0, function ----
Task tasks[] = {
  { 500,   0, heartbeat  },   // 2 Hz LED toggle
  { 100,   0, sampleADC  },   // 10 Hz sensor
  { 60000, 0, sendPacket },   // once a minute
};
const uint8_t NTASKS = sizeof(tasks) / sizeof(tasks[0]);

void setup() { pinMode(LED_BUILTIN, OUTPUT); }

void loop() {
  uint32_t now = millis();                  // ONE timestamp per pass
  for (uint8_t i = 0; i < NTASKS; i++) {
    // unsigned subtraction — see §4 for why this is rollover-safe
    if (now - tasks[i].lastRun >= tasks[i].interval) {
      tasks[i].lastRun = now;               // reset BEFORE running
      tasks[i].fn();                        // run the task
    }
  }
  // loop() returns in microseconds → nothing blocks anything else
}

🛠️ Reset lastRun before calling fn()

Set lastRun = now before invoking the task, so the interval is measured from the scheduled tick, not from when the task happens to finish. This keeps the cadence stable even if a task runs a little long, and it prevents drift from creeping in over millions of iterations.

⚠️ Pitfall — cooperative means you must yield

No task is preempted. If sampleADC() contains a delay(200), every other task — including the heartbeat — stalls for 200 ms. The scheduler is only as responsive as your slowest task. Keep each one short; if a job is inherently long, chop it into its own FSM that advances one step per call.

4millis() rollover: proving the unsigned trick

② Math derive it, don't trust it

millis() returns a uint32_t — a 32-bit unsigned counter of milliseconds since boot. It counts up to $2^{32}-1$ and then wraps back to 0. When does that happen?

$$ t_{\text{wrap}} = \frac{2^{32}\ \text{ms}}{1000 \cdot 60 \cdot 60 \cdot 24} \approx \frac{4.295\times10^{9}}{86\,400\,000} \approx 49.7\ \text{days} $$

Every millis()-based device that stays powered for ~50 days hits this wrap. The naive guard if (now > lastRun + interval) breaks across the wrap: lastRun + interval overflows or now resets to a small number, and the comparison goes wrong — a task that fires forever or never. The fix is to subtract first, in unsigned arithmetic:

$$ \texttt{if } (\,\underbrace{\texttt{now} - \texttt{lastRun}}_{\text{unsigned}}\, \geq \texttt{interval}\,) $$

Here's the proof that this is correct even across the wrap. Unsigned 32-bit subtraction is defined to be performed modulo $2^{32}$:

$$ (\texttt{now} - \texttt{lastRun}) \bmod 2^{32} \;=\; \text{true elapsed ms},\quad \text{for any elapsed} < 2^{32}. $$

Concretely, suppose lastRun = $2^{32}-10$ (just before the wrap) and 25 ms later now has wrapped to $15$. The literal difference $15 - (2^{32}-10)$ is negative — but computed modulo $2^{32}$ it becomes

$$ \big(15 - (2^{32}-10)\big) \bmod 2^{32} \;=\; (25 - 2^{32}) \bmod 2^{32} \;=\; 25\ \text{ms.} $$

Exactly the true elapsed time. The wrap cancels itself in modular arithmetic, so the unsigned subtraction returns the right answer with no special-casing. This is why the scheduler in §3 uses now - tasks[i].lastRun >= interval and never adds.

🔑 The rule, burned in

For timing, always subtract two unsigned millis() values and compare the difference to your interval. Never compare absolute timestamps, never add the interval to a timestamp, and keep both operands the same uint32_t type. Do that and rollover is a non-event — your code sails through the 49.7-day boundary without noticing.

⚠️ Pitfall — a stray int ruins the proof

The magic depends on unsigned math. If you store lastRun as a signed long or int, or mix in an int literal, C++ integer promotion can drag the subtraction into signed arithmetic where overflow is undefined behavior. Use uint32_t (or unsigned long) end to end.

5Memory discipline: Flash, stack, heap & F()

① Intuition 2 KB of SRAM is a tight budget

The ATmega328P has just 2 KB of SRAM. That sliver holds three tenants that grow toward each other: globals/statics (fixed, from the bottom), the heap (grows up as you malloc/use String), and the stack (grows down with every function call and local). When heap and stack collide, the chip doesn't error — it silently corrupts data and behaves like it's haunted.

🔑 String literals secretly eat SRAM

Because the AVR is a Harvard architecture, a constant like Serial.println("Booting...") lives in Flash but is copied into SRAM at startup so the CPU can read it as data. A dozen debug strings can quietly burn hundreds of your 2048 bytes before setup() even runs.

③ Code keep strings in Flash; never fragment the heap

Two tools fix this. The F() macro tells the compiler to leave a string literal in Flash and stream it from there. PROGMEM does the same for larger constant tables. And the String class — the heap-allocating one — is banned in long-running firmware.

memory.ino C++ / Arduino
#include <avr/pgmspace.h>

// BAD: this literal is copied into precious SRAM at boot.
// Serial.println("System initialized and ready");

// GOOD: F() keeps it in Flash; it is streamed byte-by-byte.
void banner() {
  Serial.println(F("System initialized and ready"));  // 0 bytes SRAM
}

// A constant lookup table parked entirely in Flash:
const uint8_t gamma8[] PROGMEM = { 0, 1, 2, 4, 6, 10, 14, 20 };

uint8_t gammaLookup(uint8_t i) {
  return pgm_read_byte(&gamma8[i]);   // explicit Flash read
}

// Measure free SRAM at runtime: the gap between the heap top
// and the current stack pointer. Tiny, classic, reliable.
extern char *__brkval;                 // top of heap (set by malloc)
extern char  __heap_start;             // start of heap region

int freeRAM() {
  char top;                            // address of 'top' ≈ stack ptr
  return (int)&top - (__brkval ? (int)__brkval : (int)&__heap_start);
}

void setup() {
  Serial.begin(115200);
  banner();
  Serial.print(F("Free RAM: "));
  Serial.println(freeRAM());           // watch this number stay STABLE
}

void loop() { /* freeRAM() must not trend downward over time */ }
⚠️ Pitfall — String fragments the heap to death

Arduino's String class allocates and frees on the heap every time it grows or concatenates. In a loop that builds Strings, the heap fills with mismatched holes — fragmentation — until an allocation that should fit can't find a contiguous gap. The device runs fine for hours, then crashes. Use fixed char buffers with snprintf() instead, and watch freeRAM() hold flat. A stable free-RAM number over hours is the signature of leak-free firmware.

6Power & sleep modes: the battery-life equation

① Intuition the CPU is almost always idle anyway

A sensor node that samples once a minute spends 99.9% of its life with nothing to do. Running the CPU full-tilt through that idle time burns ~15 mA for no benefit. Sleep modes let the chip switch off blocks it isn't using — CPU, then timers, then the oscillator — dropping current by three or four orders of magnitude. The AVR offers a ladder, from light to deep:

② Math average current & battery life

The device alternates: a short active burst of current $I_{\text{active}}$ for $t_{\text{active}}$, then a long sleep at $I_{\text{sleep}}$ for $t_{\text{sleep}}$, with period $T = t_{\text{active}} + t_{\text{sleep}}$. The average current is the time-weighted mean — a duty-cycle average:

$$ I_{\text{avg}} = \frac{I_{\text{active}}\,t_{\text{active}} + I_{\text{sleep}}\,t_{\text{sleep}}}{T} $$

And battery life follows from capacity (in mAh) divided by that average draw (in mA), giving hours:

$$ t_{\text{life}}\,[\text{h}] = \frac{C\,[\text{mAh}]}{I_{\text{avg}}\,[\text{mA}]} $$

Worked example. Take a node that wakes for $t_{\text{active}}=0.1\,\text{s}$ at $I_{\text{active}}=20\,\text{mA}$ (radio + ADC), then sleeps in PWR_DOWN at $I_{\text{sleep}}=6\,\mu\text{A}=0.006\,\text{mA}$ for the rest of a $T=60\,\text{s}$ cycle, so $t_{\text{sleep}}=59.9\,\text{s}$:

$$ I_{\text{avg}} = \frac{(20)(0.1) + (0.006)(59.9)}{60} = \frac{2.0 + 0.359}{60} \approx 0.0393\ \text{mA} $$

From a $C = 2000\,\text{mAh}$ pack:

$$ t_{\text{life}} = \frac{2000}{0.0393} \approx 50\,900\ \text{h} \approx 2120\ \text{days} \approx \mathbf{5.8\ years.} $$

Notice the lever: the active burst contributes 2.0 of the 2.36 mA·s per cycle, yet because it's so brief the average collapses to ~39 µA. Sleep time is the whole game. The same node running the CPU full-tilt at 15 mA would last about 5.5 days — a ~380× difference from one architectural choice.

③ Code sleep, wake on the watchdog, repeat

The pattern: configure the watchdog to fire an interrupt every ~8 s, drop into PWR_DOWN, and let that WDT interrupt wake the CPU. Between wakes the chip draws microamps.

sleep_node.ino C++ / Arduino
#include <avr/sleep.h>
#include <avr/wdt.h>
#include <avr/interrupt.h>

volatile bool wokeFromWDT = false;

// Watchdog as a periodic wake source (interrupt, NOT reset).
void setupWDT_8s() {
  cli();                              // disable interrupts while we poke regs
  wdt_reset();
  // timed sequence: set WDCE+WDE, then program prescaler within 4 cycles
  WDTCSR = (1 << WDCE) | (1 << WDE);
  WDTCSR = (1 << WDIE)               // INTERRUPT mode (wake, don't reset)
         | (1 << WDP3) | (1 << WDP0); // prescaler 1001b → ~8 s timeout
  sei();
}

ISR(WDT_vect) { wokeFromWDT = true; } // fires every ~8 s, returns to loop

void powerDownUntilWDT() {
  set_sleep_mode(SLEEP_MODE_PWR_DOWN); // deepest practical mode
  cli();
  sleep_enable();
  sleep_bod_disable();               // kill brown-out detector → save ~µA
  sei();
  sleep_cpu();                       // ←── CPU halts here at ~6 µA
  sleep_disable();                   // execution resumes HERE after WDT
}

void setup() { setupWDT_8s(); pinMode(LED_BUILTIN, OUTPUT); }

void loop() {
  // --- ACTIVE burst: brief, do the real work ---
  digitalWrite(LED_BUILTIN, HIGH);
  readSensorAndMaybeTransmit();      // ~100 ms of 20 mA work
  digitalWrite(LED_BUILTIN, LOW);

  // --- SLEEP: collapse to microamps until the watchdog wakes us ---
  powerDownUntilWDT();
}
🛠️ Squeeze the last microamps

In PWR_DOWN the dominant draws are the brown-out detector (~20 µA) and the watchdog (~4 µA). sleep_bod_disable() kills the former for the sleep window. Also disable the ADC (ADCSRA &= ~(1 << ADEN)) and float unused pins so leakage paths don't quietly drain the cell.

7The watchdog: auto-recovery from hangs

① Intuition a dead-man's switch for firmware

Even careful firmware can wedge: a sensor library spins forever waiting for an I²C ack that never comes, or a cosmic-ray bit-flip corrupts the program counter. With no human nearby, the device is just dead. The watchdog timer (WDT) is a hardware countdown that, if it ever reaches zero, resets the whole chip. Your job is to "pet" it — call wdt_reset() — faster than it expires. A healthy loop keeps petting; a hung loop stops, the WDT times out, and the chip reboots into a known-good state.

flowchart LR A["loop() runs"] --> B["wdt_reset()
pet the dog"] B --> A A -. hangs .-> C["no pet"] C --> D["WDT counts to 0"] D --> E(["Hardware RESET
→ reboot clean"])
③ Code reset mode, not interrupt mode

Here the WDT is armed in reset mode (WDE, not WDIE) with an 8 s timeout. Note this is the same peripheral as §6, configured for a different job — recovery instead of waking.

watchdog.ino C++ / Arduino
#include <avr/wdt.h>

void setup() {
  // Did we get here via a watchdog reset? Log it, then clear.
  if (MCUSR & (1 << WDRF)) { /* record "WDT recovered a hang" */ }
  MCUSR = 0;

  wdt_enable(WDTO_8S);     // reset the chip if not petted within 8 s
}

void loop() {
  wdt_reset();            // pet the dog — proof the loop is alive

  doScheduledWork();      // your FSM + scheduler from §2–§3

  // If doScheduledWork() ever hangs > 8 s, wdt_reset() is never
  // reached, the WDT fires, and the MCU reboots into a clean state.
}
⚠️ Pitfall — petting inside the hang

Never sprinkle wdt_reset() inside long inner loops just to "keep it happy." That defeats the entire point: the watchdog must be able to catch a hang. Pet it in exactly one place — the top of loop() — so that any stall longer than the timeout is guaranteed to trip it.

🏭 Real case — a coin-cell sensor node that lasts a year

Tie it together: a soil-moisture node on a CR2032 (~225 mAh) or a pair of AAs (~2000 mAh). It wakes on the watchdog roughly every minute, spends ~100 ms reading the sensor and firing one LoRa packet at ~20 mA, then drops to PWR_DOWN at single-digit microamps. With the §6 math, the AA pack yields years and even the coin cell clears 12+ months — because the device is asleep 99.9% of the time. The same WDT that wakes it from sleep also reboots it if a radio driver ever wedges. State machine for behavior, scheduler for cadence, sleep for power, watchdog for resilience: that's a product, not a demo.

Q1 Conceptual check
Two design choices keep long-running firmware healthy. Which statement is correct?
  • Compare absolute timestamps with if (now > lastRun + interval) — adding the interval is the safe form across the millis() wrap.
  • Subtract unsigned millis() values (now - lastRun >= interval) so modulo-$2^{32}$ math handles the ~49.7-day wrap, and avoid the heap-fragmenting String class.
  • Prefer the String class over char buffers because it manages memory automatically and never fragments the heap.
  • Rollover is impossible because millis() returns a signed long that simply saturates at its maximum.
Unsigned subtraction is computed modulo $2^{32}$, so now - lastRun returns the true elapsed time even when now has wrapped past 0 — the wrap cancels itself (§4). Adding the interval to a timestamp overflows and breaks at the boundary. And String allocates/frees on the heap, fragmenting it until a fit-able allocation fails hours later (§5); fixed char buffers with snprintf() keep free RAM flat.

Q3 Coding challenge — a non-blocking debounce FSM
Write a debounce routine that reports a clean button press with no delay(). Model it as a tiny FSM: track the last raw reading and the millis() time it last changed; only accept a new stable level once it has held steady for DEBOUNCE_MS (say 25 ms). Return true exactly once, on a confirmed HIGH→LOW edge. It must use unsigned millis() subtraction (§4) and must not block the scheduler.
Hint
Keep three static variables: lastRaw (last pin reading), lastChange (a uint32_t timestamp), and stable (the accepted, debounced level). Each call: read the pin; if it differs from lastRaw, restart the timer (lastChange = millis()). If millis() - lastChange >= DEBOUNCE_MS and the raw level differs from stable, commit it to stable and return true only when the new stable level is LOW (a press, since the button is INPUT_PULLUP).
Show one solution
debounce_fsm.ino C++ / Arduino
const uint8_t  BTN = 2;          // wired to GND, INPUT_PULLUP
const uint32_t DEBOUNCE_MS = 25; // must hold steady this long

void setup() {
  pinMode(BTN, INPUT_PULLUP);    // idle HIGH, pressed LOW
  pinMode(LED_BUILTIN, OUTPUT);
}

// Returns true ONCE on a clean (debounced) press. No delay().
bool debouncedPress() {
  static bool     lastRaw    = HIGH;   // previous raw reading
  static bool     stable     = HIGH;   // accepted, debounced level
  static uint32_t lastChange = 0;      // when raw last changed

  bool raw = digitalRead(BTN);
  if (raw != lastRaw) {                // bouncing → restart timer
    lastRaw    = raw;
    lastChange = millis();
  }

  // unsigned subtraction → rollover-safe (see §4)
  if ((uint32_t)(millis() - lastChange) >= DEBOUNCE_MS) {
    if (raw != stable) {               // level held long enough: commit
      stable = raw;
      if (stable == LOW) return true;  // HIGH→LOW == a real press
    }
  }
  return false;
}

void loop() {
  if (debouncedPress()) {
    // toggle the LED on each clean press — no blocking anywhere
    digitalWrite(LED_BUILTIN, !digitalRead(LED_BUILTIN));
  }
  // ... your scheduler & other tasks keep running every pass ...
}

This is the FSM mindset in miniature: stable is the state, the raw pin is the event, and the millis() timer is the guard on the transition. Because it never blocks and uses unsigned subtraction, it slots straight into the §3 scheduler and survives the 49.7-day wrap untouched. Swap digitalRead for a noisy sensor and the same pattern debounces any jittery input.

🎓 Chapter recap

You can now architect firmware that survives the field: model behavior as an explicit finite state machine (one enum, one switch, every mode named); run many jobs from a cooperative scheduler table of {interval, lastRun, fn} with zero delay(); prove that unsigned millis() subtraction sails through the ~49.7-day rollover; defend your 2 KB of SRAM with F(), PROGMEM, and a String-free heap while measuring freeRAM(); budget power with $I_{\text{avg}} = (I_a t_a + I_s t_s)/T$ and $t_{\text{life}} = C/I_{\text{avg}}$ to turn 2000 mAh into years; and bolt on a watchdog so a hung loop reboots itself. State machine for behavior, scheduler for cadence, sleep for power, watchdog for resilience — that is the anatomy of firmware that runs for years.
