Master Capstone: A Complete Embedded System

Eight chapters of fragments — GPIO, ADC, PWM, timers, Serial, EEPROM, state machines, power. Now we fuse them into one real product: a Smart Environmental Monitor & Alarm. By the end you will have a single sketch that reads the world, decides, acts, remembers, and sleeps — the full anatomy of shipped firmware.

1The system: what we are building

① Intuition see the whole machine first

A capstone is not "everything at once" — it is a small system whose every part you already understand, wired together so the seams show. Our device watches a room and warns you when it gets too hot. Concretely it must:

  • read temperature from a TMP36 (or thermistor) on an analog pin — Ch 3, ADC;
  • read ambient light from an LDR voltage divider — Ch 3;
  • show status on the built-in LED (slow heartbeat = OK, fast = alarm) — Ch 1/4;
  • sound a PWM buzzer when temperature crosses a threshold, with hysteresis so it doesn't chatter — Ch 4;
  • let a button cycle display/alarm modes through a finite-state machine — Ch 8;
  • sample on a non-blocking schedule — never a delay() in the hot path — Ch 5/8;
  • persist the alarm threshold and running min/max to EEPROM — Ch 7;
  • stream readings over Serial as CSV for logging — Ch 6;
  • sleep between samples to stretch a coin cell to months — Ch 8, power.
💡 The one-sentence model

Every embedded product is the same loop: sense → decide → act → remember → wait. The capstone just makes all five visible at once, with real parts and real numbers.

Here is the data flowing through the device. Sensors feed the MCU; the MCU drives outputs and persistence; the button feeds back into the decision logic.

2Requirements & modular architecture

① Intuition cut the monolith into modules

The fastest way to write unmaintainable firmware is one giant loop() that does everything. The fix is modularity: each concern becomes a small unit with a clear job and a clean interface. You can then test, reason about, and replace each in isolation — the single most valuable habit separating a hobby sketch from a product.

🔑 The architectural contract

loop() becomes a thin orchestrator: it asks the scheduler "is a sample due?", calls sensors, hands the reading to alarm and store, lets fsm react to the button, then calls power. No business logic lives in loop() itself — it only routes. Each module hides its registers and magic numbers behind a verb.

🏭 Real case — why modules earn their keep

Swap the TMP36 for a thermistor next year? You touch only sensors.readTempC() — the alarm, FSM, EEPROM, and Serial code never know the part changed. That decoupling is exactly how a real firmware team ships a "v2 with a cheaper sensor" without a full rewrite. Interfaces are insurance.

Wiring & connections — the pins that ground it

Modules are abstractions, but every one of them eventually touches a pin — the real system boundary. Pin choice is not arbitrary: the buzzer needs a PWM-capable pin, the button wants an interrupt-capable pin (D2/D3 on the Uno), and the analog sensors need actual analog pins (A0–A5). Here is the bill of connections this architecture maps onto.

⚠️ Pitfall — the ADC reference voltage

By default analogReference(DEFAULT) measures against Vcc (≈5 V on a USB-powered Uno, but it sags). For sensor accuracy use the chip's internal 1.1 V band-gap reference via analogReference(INTERNAL) — it's stable and rejects supply noise. Your conversion math must use the same Vref you selected, or every reading is silently wrong. We make this explicit in the sensor module next.

3The non-blocking scheduler skeleton

① Intuition time without stopping

In Chapter 1 we sinned with delay(); in Chapter 5 we repented with millis(). A product can never block, because it must read a button while it samples and blinks. The pattern is a tiny helper: "has period elapsed since I last ran this task?" If yes, run it and stamp the time.

③ Code read every line
scheduler.ino C++ / Arduino
// Returns true at most once per `period` ms, then re-arms.
// `last` is a caller-owned timestamp (one per task).
// Using subtraction of unsigned longs makes this overflow-safe:
// even when millis() wraps at ~49.7 days, (now - last) stays correct.
bool due(unsigned long &last, unsigned long period) {
  unsigned long now = millis();
  if (now - last >= period) {
    last += period;          // advance by exactly one slot — no drift
    return true;
  }
  return false;
}

// --- usage: each task carries its own `last` ---
unsigned long tSample = 0, tLog = 0, tBlink = 0;

void loop() {
  if (due(tSample, 2000)) { /* read sensors every 2 s   */ }
  if (due(tLog,   10000)) { /* emit a CSV row every 10 s */ }
  if (due(tBlink,  500))  { /* toggle heartbeat LED      */ }
  // loop() never blocks — all three tasks share the CPU
}
🛠️ Why last += period beats last = now

If you write last = now, you lose the few milliseconds spent inside the task — over hours the cadence drifts late. Advancing by exactly period keeps the schedule phase-locked: a "2-second" sample stays 2 seconds on average forever. This is the same idea as a real-time tick.

4The sensor module & calibration math

② Math from counts to °C

The ADC is a 10-bit converter: it maps an input voltage in $[0, V_{\text{ref}}]$ onto an integer code in $[0, 1023]$. Recovering the voltage is one line of algebra:

$$ V_{\text{out}} = \frac{\text{ADC}}{1023}\, V_{\text{ref}} $$

The TMP36 is beautifully linear: its output rises $10\,\text{mV}$ per °C, offset so that $0\,^\circ\text{C}$ reads $500\,\text{mV}$. Invert that line to get temperature:

$$ T_{^\circ C} = \frac{V_{\text{out}} - 0.5\,\text{V}}{0.01\,\text{V/}^\circ\text{C}} = \frac{V_{\text{out}} - 0.5}{0.01} $$

Worked example: with $V_{\text{ref}} = 5\,\text{V}$ and an ADC reading of $154$, we get $V_{\text{out}} = \tfrac{154}{1023}\cdot 5 \approx 0.753\,\text{V}$, so $T = \tfrac{0.753 - 0.5}{0.01} \approx 25.3\,^\circ\text{C}$ — room temperature. Note the TMP36's $10\,\text{mV/}^\circ\text{C}$ means each ADC code at 5 V is worth only $\tfrac{5/1023}{0.01} \approx 0.49\,^\circ\text{C}$ of resolution — switch to the 1.1 V reference and resolution sharpens to $\approx 0.11\,^\circ\text{C}$.

∑ Alternative — a thermistor (NTC, Beta model)

If you use a 10 kΩ NTC thermistor instead, its resistance follows the Beta equation. With reference resistance $R_0$ at reference temperature $T_0$ (in kelvin) and material constant $B$ (typ. ~3950 K):

$$ \frac{1}{T} = \frac{1}{T_0} + \frac{1}{B}\ln\!\left(\frac{R}{R_0}\right) $$

Read $R$ from the divider — if the NTC is the top leg with fixed $R_{\text{fix}}$ to ground: $R = R_{\text{fix}}\,\dfrac{V_{\text{ref}} - V_{\text{out}}}{V_{\text{out}}}$. Then convert kelvin back to °C with $T_{^\circ C} = T - 273.15$.

③ Code both conversions, one interface
sensors.ino C++ / Arduino
#include <math.h>

const uint8_t PIN_TEMP  = A0;
const uint8_t PIN_LIGHT = A1;

// We selected analogReference(DEFAULT) → Vref = supply ≈ 5.0 V.
// If you call analogReference(INTERNAL), set VREF = 1.1 here instead.
const float VREF     = 5.0;
const float ADC_FULL = 1023.0;

// Average several reads to knock down ADC noise (cheap low-pass).
static uint16_t readAvg(uint8_t pin, uint8_t n = 8) {
  uint32_t acc = 0;
  for (uint8_t i = 0; i < n; i++) acc += analogRead(pin);
  return (uint16_t)(acc / n);
}

// --- TMP36 path: 10 mV/°C, 500 mV at 0 °C ---
float readTempC() {
  float v = readAvg(PIN_TEMP) * (VREF / ADC_FULL);  // counts → volts
  return (v - 0.5) / 0.01;                           // volts → °C
}

// --- Thermistor path (Beta model), if you wire an NTC instead ---
// Uncomment to use; same float return type, drop-in replacement.
// float readTempC_NTC() {
//   const float R_FIX = 10000.0, R0 = 10000.0;     // ohms
//   const float B = 3950.0, T0 = 298.15;           // K (25 °C)
//   float v = readAvg(PIN_TEMP) * (VREF / ADC_FULL);
//   float r = R_FIX * (VREF - v) / v;              // divider → resistance
//   float invT = 1.0/T0 + (1.0/B) * log(r / R0);   // Beta equation
//   return (1.0 / invT) - 273.15;                  // K → °C
// }

// Light as a 0–100 "brightness" index (LDR brighter → higher voltage).
uint8_t readLight() {
  return (uint8_t)(readAvg(PIN_LIGHT) * 100UL / 1023);
}
⚠️ Pitfall — averaging is not a substitute for settling

The AVR's single ADC is multiplexed across all analog pins. After you switch channels (A0 → A1), the sample-and-hold capacitor needs time to charge through the source impedance. A high-impedance source like an LDR divider can give a ghost of the previous channel on the first read. Throw away one dummy analogRead() after switching pins, or keep source impedance < 10 kΩ.

5The mode FSM

① Intuition one button, three behaviors

A single button must mean different things at different times — that is precisely what a finite-state machine is for (Chapter 8). Our device has three modes; each press advances to the next and wraps around. The FSM owns the current mode and the legal transitions; nothing else may change the mode.

stateDiagram-v2 [*] --> NORMAL NORMAL --> SILENT: press SILENT --> SETPOINT: press SETPOINT --> NORMAL: press note right of NORMAL heartbeat + alarm armed end note note right of SILENT alarm muted, still logs end note note right of SETPOINT press nudges threshold +1°C end note
③ Code the FSM as a table-free switch
fsm.ino C++ / Arduino
enum Mode : uint8_t { NORMAL, SILENT, SETPOINT, MODE_COUNT };
enum Event : uint8_t { EV_NONE, EV_PRESS };

volatile Mode mode = NORMAL;     // the single owner of "current mode"

// Pure transition function: given the current mode and an event,
// return the next mode. No side effects → trivial to reason about & test.
Mode fsmNext(Mode m, Event e) {
  if (e != EV_PRESS) return m;          // only a press moves us
  switch (m) {
    case NORMAL:   return SILENT;
    case SILENT:   return SETPOINT;
    case SETPOINT: return NORMAL;
    default:       return NORMAL;
  }
}

// Apply an event and run any per-mode "on entry" action.
void fsmStep(Event e) {
  Mode next = fsmNext(mode, e);
  if (next != mode) {
    mode = next;                        // commit the transition
    if (mode == SETPOINT) {
      // entering SETPOINT does nothing yet; presses here adjust threshold
    }
  } else if (e == EV_PRESS && mode == SETPOINT) {
    bumpThreshold();                    // in SETPOINT, a press nudges +1 °C
  }
}
🔑 Separate the decision from the action

fsmNext() is a pure function — same inputs, same output, no hardware touched. That makes the transition logic unit-testable on your laptop and impossible to corrupt by accident. fsmStep() is the thin shell that commits the new state and fires effects. Pushing the SETPOINT bump into the "self-loop" branch is a deliberate trick: in that mode a press isn't a transition, it's an in-state action.

⚠️ Pitfall — button bounce

A mechanical button doesn't make one clean edge; it bounces, firing several transitions in a few milliseconds. Debounce it: in the D2 interrupt, record millis() and ignore any edge within ~30 ms of the last. Without debounce, one physical press jumps two or three modes and the user thinks the firmware is haunted. We wire this in §9.

6The alarm: PWM, threshold & hysteresis

① Intuition why a single threshold chatters

Naively: "if $T > T_{\text{set}}$, alarm on; else off." But real temperature wobbles by tenths of a degree from ADC noise. Sit right at the threshold and the alarm flickers on/off many times a second — maddening and a sign of an amateur. The fix is hysteresis: two thresholds with a deadband between them.

② Math two thresholds, one deadband

Pick a setpoint $T_{\text{set}}$ and a deadband width $\Delta$. The alarm turns on when temperature rises past the upper edge and turns off only when it falls back below the lower edge:

$$ T_{\text{hi}} = T_{\text{set}} + \tfrac{\Delta}{2}, \qquad T_{\text{lo}} = T_{\text{set}} - \tfrac{\Delta}{2} $$

The alarm state $a$ is then a latched function of temperature and its own previous value:

$$ a_{n} = \begin{cases} 1 & T \ge T_{\text{hi}} \\ 0 & T \le T_{\text{lo}} \\ a_{n-1} & T_{\text{lo}} < T < T_{\text{hi}} \end{cases} $$

Inside the deadband the alarm holds its last state — that memory is what kills the chatter. Concretely, with $T_{\text{set}} = 30\,^\circ\text{C}$ and $\Delta = 2\,^\circ\text{C}$: it screams once $T$ reaches $31\,^\circ\text{C}$ and won't fall silent until $T$ drops to $29\,^\circ\text{C}$. Noise of a few tenths can never flip it.

③ Code hysteresis + a real PWM tone
alarm.ino C++ / Arduino
const uint8_t PIN_BUZZER = 9;     // Timer1 OC1A — PWM/tone capable
const float   DEADBAND   = 2.0;   // °C total width (Δ)

float   gThresholdC = 30.0;       // T_set, loaded from EEPROM
bool    gAlarmOn    = false;      // a_{n-1} — the latched state

// Returns true if the alarm is (now) active. Pure hysteresis logic.
bool updateAlarm(float tempC) {
  float tHi = gThresholdC + DEADBAND / 2.0;   // T_hi
  float tLo = gThresholdC - DEADBAND / 2.0;   // T_lo

  if (tempC >= tHi)       gAlarmOn = true;    // cross up  → latch on
  else if (tempC <= tLo)  gAlarmOn = false;   // cross down → latch off
  // else: inside deadband → gAlarmOn keeps its previous value

  return gAlarmOn;
}

// Drive the piezo. tone() configures Timer1 to emit a square wave
// at the given frequency on PIN_BUZZER without blocking loop().
void driveBuzzer(bool on, bool silenced) {
  if (on && !silenced) tone(PIN_BUZZER, 2000);  // 2 kHz — piezo's loud band
  else                 noTone(PIN_BUZZER);       // releases the timer
}
🛠️ Why 2 kHz, and why tone()

Piezo buzzers have a mechanical resonance, typically 2–4 kHz, where they are dramatically louder for the same drive — picking 2 kHz buys you free volume. tone() sets up Timer1 in CTC mode to toggle the pin in hardware, so the square wave keeps playing while your loop() does other work. That's PWM-as-an-output-peripheral, exactly the Chapter 4 idea. Calling tone() also disables analogWrite() on pins 9 and 10 — they share Timer1.

7Persistence & sleep

① Intuition remember across power loss; doze between samples

Two non-volatile jobs remain. First, persistence: the user's threshold and the running min/max should survive a reset, so we keep them in EEPROM (Chapter 7). Second, power: between 2-second samples the CPU has nothing to do — putting it to sleep drops current from ~15 mA to microamps (Chapter 8), turning a coin cell from days into months.

③ Code struct-in, struct-out EEPROM
store.ino C++ / Arduino
#include <EEPROM.h>

struct Config {
  uint16_t magic;        // sentinel: is EEPROM initialised for *us*?
  float    thresholdC;   // user setpoint
  float    minC, maxC;   // running extremes
};

const int      CFG_ADDR  = 0;
const uint16_t CFG_MAGIC = 0xE5A1;   // arbitrary "this is our format" tag

// Load config, or write sane defaults the first time we ever boot.
void loadCfg(Config &c) {
  EEPROM.get(CFG_ADDR, c);           // bulk read the struct
  if (c.magic != CFG_MAGIC) {        // blank/foreign EEPROM → seed it
    c.magic      = CFG_MAGIC;
    c.thresholdC = 30.0;
    c.minC       =  999.0;           // sentinels so first sample wins
    c.maxC       = -999.0;
    EEPROM.put(CFG_ADDR, c);
  }
}

// Persist only when changed — EEPROM endures ~100k writes per cell,
// so we never write every loop, only on real edits.
void saveCfg(const Config &c) { EEPROM.put(CFG_ADDR, c); }
⚠️ Pitfall — EEPROM wear

An AVR EEPROM cell is rated for roughly $10^5$ erase/write cycles. Write min/max every 2-second sample and you'd exhaust a cell in $\tfrac{10^5 \times 2\,\text{s}}{86400} \approx 2.3$ days. Only call saveCfg() when a value actually changes (a new record, or the user bumps the threshold), and EEPROM.put() already skips bytes that are unchanged.

③ Code sleep with the watchdog timer
power.ino C++ / Arduino
#include <avr/sleep.h>
#include <avr/wdt.h>
#include <avr/power.h>

volatile bool wdtFired = false;

// The watchdog ISR just wakes us; the real work happens back in loop().
ISR(WDT_vect) { wdtFired = true; }

// Sleep ~2 s in the lowest-power mode, woken by the watchdog timer.
void napPwrDown() {
  // Configure WDT for interrupt (not reset) every ~2 s.
  // 2 s prescaler = WDP[3:0] = 0b0111.
  cli();
  wdt_reset();
  WDTCSR |= (1 << WDCE) | (1 << WDE);          // unlock change
  WDTCSR  = (1 << WDIE) | (1 << WDP2) | (1 << WDP1) | (1 << WDP0); // 2 s, INT mode
  sei();

  set_sleep_mode(SLEEP_MODE_PWR_DOWN);         // deepest sleep
  sleep_enable();
  power_adc_disable();                          // ADC off saves ~mA
  sleep_mode();                                 // <-- CPU halts here
  // ...wakes here on WDT or button INT...
  sleep_disable();
  power_adc_enable();
}
🏭 Real case — the watchdog is also your safety net

We use the watchdog to wake from sleep, but in production it earns its name a second way: configure it in reset mode (WDE set) and call wdt_reset() from a known-good point each loop. If your firmware ever hangs — a sensor I²C bus locks up, say — the watchdog times out and reboots the device automatically. Unattended hardware in the field relies on this. Same peripheral, two life-saving jobs.

8The full integrated sketch & debugging

③ Code everything, wired together

Here is the orchestrator that ties all six modules into one shipping sketch. Read loop() first — it is pure routing, exactly the contract from §2. The module functions above slot straight in; this file shows the glue, the button ISR with debounce, the Serial CSV stream, and the stats/persistence logic.

env_monitor.ino C++ / Arduino
// ================= Smart Environmental Monitor & Alarm =================
// Integrates: ADC sensors, non-blocking scheduler, hysteresis alarm
// (PWM tone), button FSM, EEPROM persistence, Serial CSV, WDT sleep.
// Pins: A0 temp, A1 light, D2 button, D9 buzzer, D13 status LED.

#include <EEPROM.h>
#include <avr/sleep.h>
#include <avr/wdt.h>

// ---- pins ----
const uint8_t PIN_TEMP = A0, PIN_LIGHT = A1;
const uint8_t PIN_BTN  = 2,  PIN_BUZZER = 9, PIN_LED = LED_BUILTIN;

// ---- tunables ----
const float    VREF = 5.0, ADC_FULL = 1023.0;
const float    DEADBAND = 2.0;
const uint16_t CFG_MAGIC = 0xE5A1;
const int      CFG_ADDR  = 0;

// ---- persisted config ----
struct Config { uint16_t magic; float thresholdC, minC, maxC; } cfg;

// ---- FSM ----
enum Mode : uint8_t { NORMAL, SILENT, SETPOINT };
volatile Mode mode = NORMAL;

// ---- alarm latch ----
bool gAlarmOn = false;

// ---- button (debounced edge from ISR) ----
volatile bool          btnEvent = false;
volatile unsigned long btnLast  = 0;

void onButton() {                       // INT0 falling-edge ISR
  unsigned long now = millis();         // millis() runs in ISRs (timer-based)
  if (now - btnLast > 30) {             // 30 ms debounce window
    btnLast  = now;
    btnEvent = true;                    // hand the event to loop()
  }
}

// ---- scheduler ----
unsigned long tSample = 0, tBlink = 0;
bool due(unsigned long &last, unsigned long period) {
  unsigned long now = millis();
  if (now - last >= period) { last += period; return true; }
  return false;
}

// ---- sensors ----
uint16_t readAvg(uint8_t pin, uint8_t n = 8) {
  uint32_t acc = 0;
  for (uint8_t i = 0; i < n; i++) acc += analogRead(pin);
  return acc / n;
}
float readTempC() {
  float v = readAvg(PIN_TEMP) * (VREF / ADC_FULL);
  return (v - 0.5) / 0.01;              // TMP36: 10 mV/°C, 0.5 V @ 0 °C
}
uint8_t readLight() { return readAvg(PIN_LIGHT) * 100UL / 1023; }

// ---- persistence ----
void loadCfg() {
  EEPROM.get(CFG_ADDR, cfg);
  if (cfg.magic != CFG_MAGIC) {
    cfg = { CFG_MAGIC, 30.0, 999.0, -999.0 };
    EEPROM.put(CFG_ADDR, cfg);
  }
}
void saveCfg() { EEPROM.put(CFG_ADDR, cfg); }

// ---- alarm hysteresis ----
bool updateAlarm(float tempC) {
  float tHi = cfg.thresholdC + DEADBAND / 2.0;
  float tLo = cfg.thresholdC - DEADBAND / 2.0;
  if      (tempC >= tHi) gAlarmOn = true;
  else if (tempC <= tLo) gAlarmOn = false;
  return gAlarmOn;                      // else: hold previous (deadband)
}

// ---- FSM ----
void fsmStep() {
  switch (mode) {
    case NORMAL:   mode = SILENT;   break;
    case SILENT:   mode = SETPOINT; break;
    case SETPOINT:                       // press here adjusts setpoint...
      // first press from SETPOINT wraps to NORMAL; handled below
      mode = NORMAL; break;
  }
}

void setup() {
  Serial.begin(115200);
  pinMode(PIN_LED, OUTPUT);
  pinMode(PIN_BTN, INPUT_PULLUP);
  attachInterrupt(digitalPinToInterrupt(PIN_BTN), onButton, FALLING);
  loadCfg();
  Serial.println(F("# millis,tempC,light,thresh,alarm,mode,min,max"));
}

void loop() {
  // --- 1. handle button events (FSM) ---
  if (btnEvent) {
    btnEvent = false;
    if (mode == SETPOINT) {              // in SETPOINT a press bumps threshold
      cfg.thresholdC += 1.0;
      if (cfg.thresholdC > 45.0) cfg.thresholdC = 15.0;  // wrap range
      saveCfg();                          // persist the user's choice
    } else {
      fsmStep();                          // otherwise advance mode
    }
  }

  // --- 2. sample + decide + remember (every 2 s) ---
  if (due(tSample, 2000)) {
    float t = readTempC();
    uint8_t lx = readLight();
    bool ringing = updateAlarm(t);

    // update persisted extremes only on a real record (saves EEPROM wear)
    bool rec = false;
    if (t < cfg.minC) { cfg.minC = t; rec = true; }
    if (t > cfg.maxC) { cfg.maxC = t; rec = true; }
    if (rec) saveCfg();

    // act: buzzer, unless muted in SILENT mode
    bool muted = (mode == SILENT);
    if (ringing && !muted) tone(PIN_BUZZER, 2000);
    else                  noTone(PIN_BUZZER);

    // stream a CSV row over Serial (Ch 6)
    Serial.print(millis());   Serial.print(',');
    Serial.print(t, 2);       Serial.print(',');
    Serial.print(lx);         Serial.print(',');
    Serial.print(cfg.thresholdC, 1); Serial.print(',');
    Serial.print(ringing);    Serial.print(',');
    Serial.print((uint8_t)mode); Serial.print(',');
    Serial.print(cfg.minC, 1); Serial.print(',');
    Serial.println(cfg.maxC, 1);
  }

  // --- 3. heartbeat: slow when OK, fast when alarming ---
  unsigned long beat = gAlarmOn ? 120 : 500;
  if (due(tBlink, beat)) digitalWrite(PIN_LED, !digitalRead(PIN_LED));

  // --- 4. (optional) sleep here when idle for battery life ---
  // If no buzzer is active and serial has flushed, napPwrDown() would
  // drop us to µA between samples; omitted from the hot loop above so
  // the heartbeat stays smooth on a USB-powered demo bench.
}
💡 Notice what loop() does not contain

No delay(). No ADC register pokes. No EEPROM addresses. No hysteresis arithmetic. Every hard detail lives behind a named verb, and loop() reads almost like the §1 sentence: sense → decide → act → remember → wait. That readability is the architecture paying off.

④ Real case how pros debug an integrated system
🏭 Real case — isolate, then trace

When a multi-module system misbehaves, never debug all of it at once. Isolate: comment out the alarm and sleep, confirm sensors + Serial alone produce sane CSV. Then re-enable one module at a time. Trace: your Serial CSV stream is your logic analyzer — pipe it into a plot and you can literally watch the temperature curve cross $T_{\text{hi}}$ and the alarm column flip to 1. Most "impossible" bugs (a hysteresis edge off by one, a threshold that didn't persist) become obvious the instant you graph the log. Printf-over-Serial remains the most-used embedded debugger on Earth for exactly this reason.

🛠️ Extension ideas (for when this is too easy)

Add a rolling-average filter to smooth temperature. Add a FAULT state that latches if the sensor reads an impossible value (open/short → ADC pinned at 0 or 1023). Drive an I²C OLED for a real display. Add a humidity sensor and a dew-point calc. Log to an SD card with timestamps. Every one of these slots into an existing module without disturbing the others — prove the architecture to yourself.

Q1 Architecture decision
Your teammate proposes saving min/max to EEPROM on every 2-second sample "so we never lose data." The threshold rarely changes; min/max change occasionally. What's the strongest objection?
  • EEPROM writes are too slow and would block loop() for seconds
  • EEPROM cells endure ~100k writes; writing every 2 s wears one out in a few days
  • EEPROM is volatile, so the data would be lost on reset anyway
  • The Config struct is too large to fit in 1 KB of EEPROM
An AVR EEPROM cell is rated ~10⁵ writes. At one write per 2 s that's $\tfrac{10^5\times2}{86400}\approx2.3$ days to failure. The fix — write only on a real record or user edit — is exactly what the sketch does. (EEPROM is non-volatile, and a single EEPROM.put() of ~14 bytes takes only a few ms, so the other options are wrong on the facts.)
Q2 Compute it — hysteresis edge
The alarm uses setpoint $T_{\text{set}} = 30\,^\circ\text{C}$ and deadband $\Delta = 2\,^\circ\text{C}$, so it turns on at $T_{\text{hi}} = T_{\text{set}} + \Delta/2$. Once ringing, at what temperature (°C) does it turn off? (Recall $T_{\text{lo}} = T_{\text{set}} - \Delta/2$.)
°C
$T_{\text{lo}} = 30 - \tfrac{2}{2} = 29\,^\circ\text{C}$. It latches on at $T_{\text{hi}}=31\,^\circ\text{C}$ and stays on until temperature falls all the way to $29\,^\circ\text{C}$. The $2\,^\circ\text{C}$ gap between on and off is the deadband that kills chatter — any noise smaller than that can never flip the alarm.
Q3 Coding challenge — rolling-average smoothing
ADC noise makes the raw temperature jitter. Add a rolling average over the last N = 4 samples so the alarm reacts to the trend, not single-sample spikes. Replace readTempC()'s output in loop() with a smoothed value, using a fixed circular buffer (no dynamic memory). Keep it O(1) per sample.
Hint
Keep a static array buf[4] and a running sum. Each new sample: subtract the slot you're about to overwrite, write the new value, add it to sum, advance the index modulo 4. The average is sum / 4. No loops over the buffer needed — that's the O(1) trick.
Show one solution
smooth.ino C++ / Arduino
// O(1) rolling average over the last 4 temperature samples.
float smoothTempC(float sample) {
  const uint8_t N = 4;
  static float    buf[N] = {0};      // circular buffer (static = persists)
  static float    sum    = 0;
  static uint8_t  idx    = 0;
  static bool     primed = false;

  sum -= buf[idx];                   // remove the value we'll overwrite
  buf[idx] = sample;                 // store the new sample
  sum += sample;                     // add it to the running sum
  idx = (idx + 1) % N;               // advance circularly
  if (idx == 0) primed = true;       // buffer has wrapped once

  // before the buffer fills, divide by however many we have
  return primed ? (sum / N) : (sum / (idx == 0 ? N : idx));
}

// in loop(), feed the raw read through the filter:
//   float t = smoothTempC(readTempC());
//   bool ringing = updateAlarm(t);

The static locals are the whole trick: they keep their values between calls without polluting global namespace. Subtracting the outgoing sample and adding the incoming one keeps the work constant regardless of N — a real moving-average filter, the same primitive used in DSP and sensor fusion. Bump N to trade responsiveness for smoothness.
