Timers & Interrupts: Real-Time Without delay()

In Chapter 1 we sinned: every delay() freezes the entire CPU, blind and deaf to the world, for the whole wait. Real firmware never blocks. Here we learn the three escapes — the millis() idiom, hardware timers, and interrupts — and the unsigned-arithmetic trick that makes them survive the 49-day rollover.

⏱ ~50 min🧠 Non-blocking⏲ AVR timers⚡ ISRs

1Why delay() is a trap

① Intuition build the picture first

Picture your firmware as a single worker with no assistant. delay(500) tells that worker: "stand perfectly still for half a second." During those 500 ms the CPU is not idle in a useful sense — it is spinning in a tight counting loop, executing millions of do-nothing instructions, unable to do anything else. It cannot read a button. It cannot service a sensor. It cannot blink a second LED at a different rate. The whole machine is hostage to one wait.

💡 The one-sentence model

delay() is blocking: it stops time for your entire program. Professional firmware is non-blocking — it never stands still; it constantly checks "is it time yet?" and gets on with everything else in between.

The symptom is unmistakable once you know it. You write Blink, then try to add a button that toggles the LED's speed — and the button feels dead, responding only if you happen to press it during the brief instant loop() isn't inside a delay(). The fix is not a faster CPU. It is a fundamentally different control structure: stop waiting, start polling the clock.

⚠️ Pitfall — "just sprinkle in more delays"

Beginners try to interleave tasks by guessing delay values (delay(7) here, delay(13) there). It never composes: every blocking wait adds latency to every other task, and the timing drifts the moment you add one more thing. Blocking code does not scale past one job.

2The millis() idiom: cooperative non-blocking

① Intuition check the clock, don't watch it

Instead of sleeping until the next event, we remember when the last event happened and, each pass through loop(), ask the running clock whether enough time has elapsed. millis() returns the number of milliseconds since boot as an unsigned long (a 32-bit uint32_t), ticking up forever in the background. We never wait on it — we just glance at it.

③ Code read every line
blink_nonblocking.ino C++ / Arduino
const uint8_t LED = LED_BUILTIN;
const uint32_t interval = 500;      // ms between toggles
uint32_t last = 0;                  // when we last toggled
bool ledOn = false;

void setup() {
  pinMode(LED, OUTPUT);
}

void loop() {
  uint32_t now = millis();          // glance at the clock — never block
  if (now - last >= interval) {      // has 500 ms elapsed?
    last = now;                      // remember THIS moment for next time
    ledOn = !ledOn;                  // flip state
    digitalWrite(LED, ledOn);
  }
  // ...everything else in loop() keeps running every pass:
  // read buttons, poll sensors, blink a second LED, etc.
}
🔑 The pattern, memorized

Store last = now the instant you act — not last += interval conditionally, and never last = millis() a second time (that re-reads the clock and drifts). One read of now per pass, one comparison, one update. This three-line skeleton replaces every delay() you will ever want.

② Math why the rollover doesn't bite

Here is the part that trips up nearly everyone. millis() is a 32-bit unsigned counter. It counts up to its maximum and then wraps back to zero:

$$ t_{\text{rollover}} = \frac{2^{32}\ \text{ms}}{1000\cdot 60\cdot 60\cdot 24} \approx 49.71\ \text{days} $$

The naïve worry: "when now wraps to a tiny number but last is still huge, won't now - last be a giant negative value and break my timer?" In signed math, yes — disaster. But now and last are both uint32_t, and unsigned subtraction is defined to wrap modulo $2^{32}$. The hardware computes:

$$ (\,\text{now} - \text{last}\,) \bmod 2^{32} \;=\; \text{true elapsed time} $$

Concretely, suppose $\text{last} = 2^{32}-100 = 4{,}294{,}967{,}196$ (100 ms before the wrap) and 250 ms later now has wrapped to $150$. The signed view gives $150 - 4{,}294{,}967{,}196 = -4{,}294{,}967{,}046$ — nonsense. The unsigned view gives:

$$ (150 - 4{,}294{,}967{,}196)\bmod 2^{32} = 150 + 100 = 250\ \text{ms} \;\checkmark $$

Exactly the true elapsed time. The wrap of now and the wrap built into the subtraction cancel perfectly. This is only true if you keep both operands unsigned and compare the difference against the interval. Never compare now >= last + interval — that addition can itself overflow and reintroduce the bug.

⚠️ Pitfall — the wrong comparison

if (millis() >= target) where target = last + interval is the classic broken form: once target overflows near the 49-day mark it becomes small, and the condition fires instantly forever. Always write the difference form: if (now - last >= interval). It is rollover-proof by construction.

3From one timer to a cooperative scheduler

① Intuition many tasks, one loop

Once each task owns its own last and interval, they coexist trivially: loop() spins fast, and each task fires independently when its clock says so. This is a cooperative scheduler — no operating system, just a handful of if (now - last >= interval) blocks sharing one fast loop.

🛠️ Keep loop() hungry

A cooperative scheduler only works if loop() runs often — ideally thousands of times per second. That means no blocking anywhere: no delay(), no busy-wait, no slow Serial.print of huge strings. The faster the loop, the tighter your scheduling resolution. One stray delay(1000) stalls every task at once.

4Hardware timers: precision without the CPU

① Intuition a counter that runs itself

The millis() idiom is great, but its resolution is limited by how fast loop() turns, and a fat loop can make it jittery. For hard timing — sampling a sensor at exactly 8 kHz, generating a tone, driving a stepper — we hand the job to dedicated timer/counter peripherals built into the AVR. They count clock ticks in hardware, completely independent of what the CPU is doing, and raise an interrupt at a precise moment.

The ATmega328P has three of them. Each is just a register, TCNTn, that increments on every tick and rolls over:

⚠️ Pitfall — Timer0 is spoken for

Reconfiguring Timer0 breaks millis(), micros(), and delay() across your whole sketch. For custom periodic interrupts, reach for Timer1 (16-bit, the most headroom) and leave Timer0 alone.

① Intuition the prescaler — a clock divider

At 16 MHz, an 8-bit timer overflows every $256 / 16{\times}10^6 \approx 16\ \mu\text{s}$ — far too fast for human-scale events. The prescaler divides the system clock before it reaches the counter: with a prescaler of 256, the timer only ticks once every 256 system cycles. The AVR offers a fixed menu of divisors — 1, 8, 64, 256, 1024 — selected by the CSn2:0 clock-select bits in the TCCRnB register.

① Intuition CTC mode — clear timer on compare match

The trick for a precise period is CTC mode (Clear Timer on Compare Match). You load a target into the compare register OCRnA. The timer counts $0,1,2,\dots$ up to OCRnA, fires an interrupt, then auto-resets to zero and starts again. No software reset, no drift — a perfect metronome in silicon.

5Timer math: picking OCRnA for any frequency

② Math derive it

In CTC mode the timer counts from 0 up to OCRnA and resets — that is $(\text{OCRnA}+1)$ ticks per cycle (the $+1$ is because it spends one tick at 0). Each tick lasts the prescaled clock period. So the interrupt frequency is:

$$ f_{\text{int}} = \frac{f_{\text{clk}}}{N \,\bigl(1 + \text{OCRnA}\bigr)} $$

where $f_{\text{clk}} = 16\ \text{MHz}$ and $N$ is the prescaler (one of 1, 8, 64, 256, 1024). Invert it to solve for the register value you must load:

$$ \text{OCRnA} = \frac{f_{\text{clk}}}{N\, f_{\text{int}}} - 1 $$

Worked example. We want a clean $1\ \text{Hz}$ interrupt — one tick per second — on Timer1, with prescaler $N = 256$:

$$ \text{OCR1A} = \frac{16\times10^6}{256 \times 1} - 1 = 62500 - 1 = 62499 $$

Sanity-check the result: $62499 \le 65535$, so it fits in the 16-bit OCR1A register. (On 8-bit Timer0 or Timer2 it would not — $62499 \gg 255$ — which is precisely why we chose the 16-bit Timer1.) Plug it back into the forward formula and you recover exactly $16{\times}10^6 / (256 \times 62500) = 1\ \text{Hz}$. Choosing the prescaler is the art: pick the smallest $N$ whose resulting OCRnA still fits the register, because smaller $N$ gives finer timing resolution.

∑ Resolution check

With $N = 256$ at 16 MHz, each tick is $256/16{\times}10^6 = 16\ \mu\text{s}$. So OCR1A is quantized in 16 µs steps — you can hit 1 Hz, 2 Hz, 4 Hz exactly, but an oddball frequency rounds to the nearest 16 µs. Need finer? Drop to $N = 64$ (4 µs steps) and re-solve — as long as the new OCR1A still fits 65535.

③ Code configure Timer1 by hand
timer1_ctc.ino C++ / Arduino
const uint8_t LED = LED_BUILTIN;

void setup() {
  pinMode(LED, OUTPUT);

  noInterrupts();           // disable interrupts while we configure
  TCCR1A = 0;               // clear control reg A (no PWM, normal port op)
  TCCR1B = 0;               // clear control reg B
  TCNT1  = 0;               // reset the counter to 0

  OCR1A = 62499;            // compare target: 16e6/(256*1) - 1  → 1 Hz
  TCCR1B |= (1 << WGM12);   // WGM12 = 1 selects CTC mode (top = OCR1A)
  TCCR1B |= (1 << CS12);    // CS12 = 1 selects prescaler = 256
  TIMSK1 |= (1 << OCIE1A);  // enable Timer1 Compare-A Match interrupt
  interrupts();             // re-enable interrupts globally
}

// This runs AUTOMATICALLY at exactly 1 Hz — no polling, no delay.
ISR(TIMER1_COMPA_vect) {
  digitalWrite(LED, !digitalRead(LED));   // toggle the LED
}

void loop() {
  // Completely free. The 1 Hz blink happens in the background.
  // Put your real work here; it never blocks the timer.
}
🔑 Why (1 << BITNAME)

Constants like WGM12, CS12, OCIE1A are just bit positions (numbers 0–7) defined in the AVR headers. (1 << CS12) builds a mask with that one bit set; |= ORs it into the register, flipping that bit on without disturbing the others. This is the universal idiom for poking a single hardware bit — you saw it in Chapter 2 and you will use it forever.

6Hardware interrupts: zero-latency reactions

① Intuition a hardware tap on the shoulder

A timer interrupt fires on a clock event. An external interrupt fires on a pin event — a rising or falling edge — and it does so within a few clock cycles, no matter what your code is doing. The CPU drops everything, jumps to your Interrupt Service Routine (ISR), runs it, then resumes exactly where it left off. This is how you catch a fast event — an encoder edge, a sensor pulse, a button — with essentially zero latency, instead of hoping loop() happens to poll at the right microsecond.

sequenceDiagram participant W as External wire (edge) participant H as AVR hardware participant M as loop() / main code participant I as ISR M->>M: running normally W->>H: rising edge on INT0 pin H->>M: finish current instruction H->>I: vector to ISR (push context) I->>I: do the bare minimum I->>M: return (pop context) M->>M: resume exactly where paused

On the Uno, only pins 2 and 3 are true external-interrupt lines (INT0, INT1). You attach an ISR with the Arduino API:

encoder_edge.ino C++ / Arduino
const uint8_t ENC_A = 2;          // encoder channel A → INT0
volatile int32_t pulses = 0;      // shared with ISR → MUST be volatile

void setup() {
  pinMode(ENC_A, INPUT_PULLUP);
  // Call countPulse() on every RISING edge of pin 2:
  attachInterrupt(digitalPinToInterrupt(ENC_A), countPulse, RISING);
  Serial.begin(115200);
}

// ISR: keep it TINY. No delay(), no Serial, no malloc.
void countPulse() {
  pulses++;                       // just record the event and leave
}

void loop() {
  // Read the shared variable safely: snapshot it with interrupts off.
  noInterrupts();
  int32_t snapshot = pulses;
  interrupts();
  Serial.println(snapshot);       // slow work happens HERE, not in the ISR
}
🔑 The three ISR commandments

1. Keep it tiny. An ISR steals the CPU from everything else — set a flag, bump a counter, and get out.
2. No delay(), Serial, or anything blocking. millis() won't even advance inside an ISR (its timer interrupt is masked), and Serial relies on interrupts that are now disabled — deadlock.
3. Every variable shared with the ISR must be volatile.

② Math why volatile is non-negotiable

volatile tells the compiler: "this variable can change behind your back — re-read it from memory every single time, and never cache it in a register or optimize the access away." Without it, the optimizer assumes loop() is the only writer. It may hoist pulses into a CPU register once and loop forever on the stale copy, never seeing the ISR's updates. Your code compiles, runs, and silently lies.

⚠️ Pitfall — the torn read

On an 8-bit AVR, a 32-bit int32_t is read in four separate byte operations. If the ISR fires between those bytes, you read half the old value and half the new — a "torn" read. That is why the snapshot above brackets the read with noInterrupts()/interrupts(): it makes the multi-byte read atomic. For a single uint8_t flag this isn't needed; for anything wider than the CPU word, it is mandatory.

① Intuition pin-change vs external interrupts

Two pins is rarely enough. Pin-change interrupts (PCINT) are the alternative: almost every pin can raise one, but they are coarser — a single shared ISR fires for a whole port on any edge (rising or falling), and your handler must figure out which pin moved and in which direction. Use external interrupts (pins 2/3) when you can — they are precise, per-pin, and edge-selectable. Fall back to pin-change interrupts when you need to watch more pins than INT0/INT1 can cover.

7Real case: deterministic sampling & the encoder

④ Real case this ships in products

Two patterns from this chapter are everywhere in real hardware, and they map exactly onto the two interrupt sources you just met.

🏭 Real case — precise periodic sampling

A digital audio or vibration sensor must be read at a rock-steady rate, say 8 kHz, or the downstream FFT smears. Reading it from loop() with millis() jitters — sometimes the loop is busy. The production answer is a Timer1 CTC interrupt set to 8 kHz: in the ISR you read one ADC sample into a ring buffer and return. The sampling instant is now defined by a crystal, not by software luck. Set $\text{OCR1A} = 16{\times}10^6/(8 \times 8000) - 1 = 249$ with prescaler 8 — exactly the math from §5.

🏭 Real case — zero-latency encoder edges

A motor's quadrature encoder emits pulses faster than any polling loop can reliably catch at speed — miss one edge and your position estimate drifts forever. Wiring channel A to an external interrupt means every single edge increments the count the instant it arrives, regardless of what loop() is doing. This is how robot wheels, CNC axes, and 3-D printers know exactly where they are. The ISR does one thing — pulses++ — and the heavy control math runs later in loop(), reading the volatile snapshot.

🛠️ Division of labour

The recurring architecture: interrupts capture, loop() processes. ISRs do the irreducible minimum at the exact moment an event happens; the slow, complex, blocking-tolerant work happens back in the non-blocking main loop. Master that split and you can build firmware that is both real-time and responsive — the thing delay() could never give you.

Q3 Coding challenge — two LEDs, two rates, no delay()
Blink LED A on pin 8 at 2 Hz (toggle every 250 ms) and LED B on pin 9 at 5 Hz (toggle every 100 ms) simultaneously, using only the millis() idiom — no delay() anywhere. The two blinks must run completely independently.
Hint
Give each LED its own last timestamp and its own interval. Read now = millis() once per pass, then run two independent if (now - lastX >= intervalX) blocks. The toggle is state = !state; digitalWrite(pin, state);.
Show one solution
two_leds.ino C++ / Arduino
const uint8_t LED_A = 8, LED_B = 9;
const uint32_t intervalA = 250;   // 2 Hz  → toggle every 250 ms
const uint32_t intervalB = 100;   // 5 Hz  → toggle every 100 ms
uint32_t lastA = 0, lastB = 0;
bool stateA = false, stateB = false;

void setup() {
  pinMode(LED_A, OUTPUT);
  pinMode(LED_B, OUTPUT);
}

void loop() {
  uint32_t now = millis();        // ONE clock read per pass

  if (now - lastA >= intervalA) {  // LED A: independent timer
    lastA = now;
    stateA = !stateA;
    digitalWrite(LED_A, stateA);
  }

  if (now - lastB >= intervalB) {  // LED B: independent timer
    lastB = now;
    stateB = !stateB;
    digitalWrite(LED_B, stateB);
  }
  // loop() never blocks → both LEDs keep their own rhythm,
  // and there's still room here to read a button or sensor.
}

The lesson: each task owns its last + interval, and loop() stays non-blocking. This is the cooperative scheduler from §3 in miniature — add a third LED, a button, a sensor, and they all coexist because nothing ever stands still. Try it with delay() and watch the two rates collide.
