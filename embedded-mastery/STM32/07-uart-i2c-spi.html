Serial Buses: USART, I²C, SPI & Baud Math

On the Arduino you had one hardware UART, one I²C, one SPI — and they blocked the CPU. The STM32 hands you a fleet of bus controllers, each able to fire interrupts or pump bytes through DMA while the core does real work. We rebuild the three classic buses from their registers up, derive the baud and prescaler math exactly, and wire a console, an EEPROM, and an IMU onto one board.

1Three buses, one board — a fast recap

① Intuition build the picture first

You met all three buses on the Arduino in Arduino Chapter 06. The protocols are identical on STM32 — same wires, same framing, same idea — so we recap fast and then go where the 8-bit AVR could not follow: multiple instances, interrupt-driven transfers, and DMA.

Every serial bus solves the same problem: move bytes between chips using as few wires as possible. They differ in how many wires, who drives the clock, and how a receiver knows where a byte begins.

💡 The one-line distinction

UART is asynchronous — no clock wire, so both ends must already agree on the bit rate. I²C and SPI are synchronous — a clock wire travels with the data, so timing is self-describing. Async trades a wire for the burden of getting the baud rate exactly right; that trade is why §2 exists.

The acronym jump matters too: the AVR had a UART. Most STM32 peripherals are USARTs — the extra "S" is Synchronous, meaning they can also emit a clock and run in SPI-like or smartcard modes. You almost always use them in plain asynchronous UART mode, but the registers carry the USART name.

2USART baud rate: the math that actually ships

① Intuition sampling in the dark

With no clock wire, the receiver must guess where each bit sits. The trick: run an internal clock 16× faster than the expected bit rate (oversampling by 16), wait for the falling edge of the start bit, then sample each data bit near its centre — around the 8th of 16 ticks. Get the divisor slightly wrong and the sample point drifts; drift far enough across a frame and you read garbage.

② Math derive USARTDIV

The peripheral clock $f_{CK}$ (the bus clock feeding the USART — PCLK1 or PCLK2) is divided by 16 and by a fractional divisor $\text{USARTDIV}$ to produce the baud rate. With oversampling by 16:

$$ \text{baud} = \frac{f_{CK}}{16 \cdot \text{USARTDIV}} \quad\Longrightarrow\quad \text{USARTDIV} = \frac{f_{CK}}{16 \cdot \text{baud}} $$

$\text{USARTDIV}$ is a fixed-point number: an integer mantissa plus a 4-bit fraction in sixteenths, packed into the baud-rate register BRR as [mantissa : fraction]. Let's program a real part. Take an STM32F4 with the USART on PCLK2 = 84 MHz, targeting 115200 baud:

$$ \text{USARTDIV} = \frac{84\times10^{6}}{16 \cdot 115200} = \frac{84\times10^{6}}{1\,843\,200} \approx 45.5729 $$

Split it: the mantissa is $45$; the fraction is $0.5729 \times 16 \approx 9.17 \to 9$ (rounded to the nearest sixteenth). So we program

$$ \text{BRR} = (45 \ll 4) \,|\, 9 = 0x2D9, \qquad \text{USARTDIV}_{\text{prog}} = 45 + \tfrac{9}{16} = 45.5625 $$

Now check what baud that actually produces, and the resulting error:

$$ \text{baud}_{\text{actual}} = \frac{84\times10^{6}}{16 \cdot 45.5625} = 115226\ \text{baud}, \qquad \varepsilon = \frac{115226 - 115200}{115200} \approx +0.02\% $$
🔑 The 2% rule

Both ends accumulate error across a 10-bit frame (start + 8 data + stop). The combined transmitter+receiver error budget is roughly ±2–3% before the sample point drifts off the bit. Our +0.02% is luxuriously safe. This is why integer-friendly clocks matter: at an awkward $f_{CK}$ some baud rates simply cannot be hit accurately — a recurring reason to pick crystal and PLL values deliberately back in Chapter 02.

③ Code read every line

HAL hides BRR behind Init.BaudRate, computing USARTDIV for you from the live clock tree. But knowing the math lets you sanity-check it and debug a console that comes up as mojibake.

usart_init.c C / STM32 HAL
// USART2 on STM32F4, clocked from PCLK1; here APB2/USART1 example on PCLK2 = 84 MHz.
UART_HandleTypeDef huart1;

void uart1_init(void) {
  __HAL_RCC_USART1_CLK_ENABLE();          // gate the USART clock on
  huart1.Instance        = USART1;        // PCLK2 = 84 MHz on this part
  huart1.Init.BaudRate   = 115200;        // HAL solves USARTDIV -> BRR = 0x2D9
  huart1.Init.WordLength = UART_WORDLENGTH_8B;
  huart1.Init.StopBits   = UART_STOPBITS_1;
  huart1.Init.Parity     = UART_PARITY_NONE;     // 8N1 — the universal default
  huart1.Init.Mode       = UART_MODE_TX_RX;
  huart1.Init.OverSampling = UART_OVERSAMPLING_16; // baud = f_CK / (16 * USARTDIV)
  HAL_UART_Init(&huart1);                 // writes BRR, CR1, CR2, CR3
}

// Blocking send — the CPU spins until every byte has left the shift register.
void uart1_puts(const char *s) {
  HAL_UART_Transmit(&huart1, (uint8_t*)s, strlen(s), HAL_MAX_DELAY);
}
⚠️ Pitfall — wrong PCLK, wrong baud

If you change the clock tree (a new PLL config, a different APB prescaler) but the USART still thinks it runs at the old $f_{CK}$, every baud rate shifts and the console fills with garbage. HAL reads the clock at HAL_UART_Init time, so re-init after any clock change. HAL_RCC_GetPCLK2Freq() tells you the real number it used.

3Beyond blocking: UART by interrupt and by DMA

① Intuition stop babysitting bytes

HAL_UART_Transmit(..., HAL_MAX_DELAY) is honest but wasteful — at 115200 baud each byte takes about $\frac{10\ \text{bits}}{115200} \approx 87\,\mu s$, and the CPU spins the whole time. Sending a 1 KB log line blocks for ~89 ms. On a real-time system that is a catastrophe. The STM32 gives you two escapes.

Interrupt mode — the USART raises an interrupt each time its transmit register empties; an ISR feeds the next byte. The CPU only visits once per byte. DMA mode — a dedicated Direct Memory Access engine streams the whole buffer to the USART data register with zero CPU involvement; the core is interrupted exactly once, at completion.

③ Code non-blocking transmit
uart_dma.c C / STM32 HAL
// Assume CubeMX wired DMA2_Stream7 (USART1_TX) and the DMA IRQ handlers.
uint8_t log_buf[256];

void log_line_dma(const char *msg) {
  int n = snprintf((char*)log_buf, sizeof log_buf, "%lu: %s\r\n",
                   HAL_GetTick(), msg);
  // Returns IMMEDIATELY; DMA pushes the bytes while loop() keeps running.
  HAL_UART_Transmit_DMA(&huart1, log_buf, n);
}

// HAL calls this from the DMA transfer-complete ISR when the buffer is drained.
void HAL_UART_TxCpltCallback(UART_HandleTypeDef *huart) {
  if (huart->Instance == USART1) {
    tx_busy = 0;                 // safe to queue the next line now
  }
}

// Receive: arm a DMA into a ring buffer once, let it fill in the background.
uint8_t rx_buf[64];
void rx_start(void) {
  HAL_UART_Receive_DMA(&huart1, rx_buf, sizeof rx_buf);  // non-blocking RX
}
⚠️ Pitfall — touching a buffer mid-DMA

The DMA engine reads log_buf directly from SRAM. If you overwrite it before TxCpltCallback fires, you corrupt the in-flight transfer — you'll see torn or interleaved output. Gate it with a tx_busy flag (above) or a real queue. Same rule for RX: don't parse rx_buf faster than DMA fills it.

🛠️ Pick the right mode

Blocking: one-off setup prints, where simplicity wins. Interrupt: short, bursty traffic and you're out of DMA streams. DMA: high-throughput logging, audio, or anything where the CPU must stay free. A 1 KB log line that blocked for 89 ms costs the CPU effectively 0 ms over DMA.

4I²C on STM32: addressing, timing, and register-mapped devices

① Intuition a two-wire party line

I²C puts every device on the same two open-drain wires — SDA (data) and SCL (clock) — each pulled high by a resistor. The controller starts a transaction by sending a 7-bit address plus a read/write bit; only the matching device answers. That's how an EEPROM, an RTC, and three sensors coexist on two pins.

② Math address space & clock tiers

A 7-bit address gives $2^{7} = 128$ slots, but reserved ranges trim that to 112 usable device addresses on one bus. The shifted byte that actually goes on the wire is the address in the top 7 bits with the R/W̄ bit at the bottom:

$$ \text{byte} = (\text{addr}_{7} \ll 1)\,|\,\overline{\text{R/W}}, \qquad 0x50 \to 0xA0\ \text{(write)},\ 0xA1\ \text{(read)} $$

I²C runs in fixed clock tiers: 100 kHz (Standard), 400 kHz (Fast), and 1 MHz (Fast-mode Plus). On modern STM32 (F0/F3/F7/L4/G4…) you don't set a simple divisor — you load a computed TIMINGR register that encodes the SCL high/low periods and the data setup/hold times in cycles of the I²C kernel clock. CubeMX (or the ST timing tool) generates this constant for your clock; treat it as derived, not magic.

③ Code the killer convenience: Mem_Read/Write

Most I²C chips are register-mapped: to read a value you first write the register address, send a repeated-start, then read. HAL bundles that whole dance into HAL_I2C_Mem_Read/Write — you give it the device address and the internal register, and it handles the repeated start.

eeprom_i2c.c C / STM32 HAL
#define EE_ADDR  (0x50 << 1)   // 7-bit 0x50, HAL wants it left-shifted
#define EE_MEMSZ I2C_MEMADD_SIZE_16BIT  // 24LC256 uses a 16-bit word address

// Write one page (<=64 B) to a 24LC256 EEPROM at internal address `at`.
HAL_StatusTypeDef ee_write(uint16_t at, uint8_t *data, uint16_t len) {
  return HAL_I2C_Mem_Write(&hi2c1, EE_ADDR, at, EE_MEMSZ,
                           data, len, HAL_MAX_DELAY);
}

// Read `len` bytes back. Mem_Read does write-addr + repeated-start + read.
HAL_StatusTypeDef ee_read(uint16_t at, uint8_t *data, uint16_t len) {
  return HAL_I2C_Mem_Read(&hi2c1, EE_ADDR, at, EE_MEMSZ,
                          data, len, HAL_MAX_DELAY);
}

// EEPROMs need a few ms to commit a page — poll until it ACKs again.
void ee_wait_ready(void) {
  while (HAL_I2C_IsDeviceReady(&hi2c1, EE_ADDR, 3, 10) != HAL_OK) { /* spin */ }
}
⚠️ Pitfall — the shift trap & missing pull-ups

HAL's I²C API expects the address already left-shifted (the R/W̄ bit is set internally). Pass a raw 0x50 and you'll address the wrong device or get HAL_ERROR. Second classic: I²C lines are open-drain — with no external pull-ups (typically 4.7 kΩ) the bus never reaches a logic high and every transfer times out. The bus looks dead, not broken.

5SPI on STM32: prescaler, modes, and full duplex

① Intuition four wires, raw speed

SPI trades I²C's wire-thrift for speed. Four signals — SCK (clock), MOSI (controller→peripheral), MISO (peripheral→controller), CS (chip-select, one per device) — and no addressing overhead. Data shifts out on MOSI and in on MISO on the same clock edges: full duplex, every clock carries a bit each way.

② Math the baud prescaler

SPI has no fractional divisor — just a 3-bit prescaler BR in CR1 that divides the bus clock by a power of two:

$$ f_{SPI} = \frac{f_{PCLK}}{2^{\,(\text{BR}+1)}}, \qquad \text{BR} \in \{0,1,\dots,7\} $$

So BR selects a /2, /4, /8, … /256 divider. On an STM32F4 with the SPI on PCLK2 = 84 MHz, picking BR = 2 gives

$$ f_{SPI} = \frac{84\times10^{6}}{2^{(2+1)}} = \frac{84\times10^{6}}{8} = 10.5\ \text{MHz} $$

Want it faster? BR = 1 → /4 → 21 MHz; BR = 0 → /2 → 42 MHz. The ceiling is whatever the slowest device on the wire can tolerate — check its datasheet's max SCK.

The second half of SPI is mode: two bits, CPOL (clock idle polarity) and CPHA (sample on first or second edge), pick which of four timing conventions you use. Both ends must agree exactly or you read shifted, garbled bits.

③ Code full-duplex transfer
spi_imu.c C / STM32 HAL
// SPI1 on PCLK2 = 84 MHz. Prescaler /8 -> 10.5 MHz; Mode 3 for an ICM-20948.
SPI_HandleTypeDef hspi1;

void spi1_init(void) {
  __HAL_RCC_SPI1_CLK_ENABLE();
  hspi1.Instance               = SPI1;
  hspi1.Init.Mode              = SPI_MODE_MASTER;
  hspi1.Init.Direction         = SPI_DIRECTION_2LINES;     // full duplex
  hspi1.Init.DataSize          = SPI_DATASIZE_8BIT;
  hspi1.Init.CLKPolarity       = SPI_POLARITY_HIGH;        // CPOL = 1
  hspi1.Init.CLKPhase          = SPI_PHASE_2EDGE;          // CPHA = 1  -> Mode 3
  hspi1.Init.NSS               = SPI_NSS_SOFT;             // drive CS in software
  hspi1.Init.BaudRatePrescaler = SPI_BAUDRATEPRESCALER_8;  // BR=2 -> /8 -> 10.5 MHz
  hspi1.Init.FirstBit          = SPI_FIRSTBIT_MSB;
  HAL_SPI_Init(&hspi1);
}

// Read an IMU register: send (reg | 0x80) to request a read, clock a dummy
// byte to shift the answer back in. CS is active-low, driven by hand.
uint8_t imu_read(uint8_t reg) {
  uint8_t tx[2] = { (uint8_t)(reg | 0x80), 0x00 };
  uint8_t rx[2] = { 0 };
  HAL_GPIO_WritePin(CS_PORT, CS_PIN, GPIO_PIN_RESET);     // CS low: select
  HAL_SPI_TransmitReceive(&hspi1, tx, rx, 2, HAL_MAX_DELAY); // full duplex
  HAL_GPIO_WritePin(CS_PORT, CS_PIN, GPIO_PIN_SET);       // CS high: deselect
  return rx[1];                                            // byte 0 was the cmd echo
}
⚠️ Pitfall — the wrong mode reads plausible garbage

If CPOL/CPHA don't match the device, you don't get an error — you get numbers, just shifted by one clock edge, so a WHO_AM_I that should read 0xEA comes back as 0xD5 or similar. Always verify the chip's identity register first; a correct WHO_AM_I proves your mode, prescaler, and wiring are all right before you trust any data.

6Choosing a bus: the engineering trade-off

④ Real case how products decide

There is no "best" bus — each wins a different corner of the design space. Internalize this table and bus selection becomes a reflex.

🔑 Decision heuristic

Talking to a human/PC/modem over distance? UART. Wiring up many slow chips on two pins? I²C. Need raw throughput from a flash, display, or IMU? SPI. And whenever the transfer is large or periodic, reach for DMA regardless of bus — that's the STM32's superpower over the AVR.

7Real case: one board, three buses, all at once

④ Real case this ships in products

Picture a flight controller or a robot's sensor board. The STM32 isn't choosing one bus — it's running all three simultaneously, each matched to its job:

  • IMU on SPI at 10.5 MHz (BR=2, /8). Attitude data must arrive fast and often — the SPI's full-duplex throughput keeps a 1 kHz control loop fed.
  • EEPROM on I²C at 400 kHz. Calibration constants and config are tiny and read rarely; two wires shared with an RTC and a barometer is the frugal choice.
  • Debug console on USART at 115200 baud, logged via DMA. Telemetry streams out continuously without ever stealing a cycle from the control loop.

🏭 Real case — why the console runs on DMA

On a flight controller the control loop is sacred: jitter is the enemy. Blocking printf over UART would stall that loop for milliseconds while bytes drain. Routing the console through UART TX DMA means the CPU writes a buffer pointer and walks away — the DMA engine clocks out kilobytes of telemetry while the 1 kHz loop ticks undisturbed. This single design choice is the difference between a stable autopilot and one that twitches every time you log a line.

🛠️ Bus arbitration in software

Each peripheral instance is independent, but they can fight over DMA streams and interrupt priorities. Give the time-critical bus (here, the IMU SPI and its DMA) a higher NVIC priority than the console UART. The buses run in parallel hardware; your job is only to rank whose interrupt wins when two fire at once.

Q1 Conceptual check
A USART running in standard async mode uses oversampling by 16. What is the receiver doing with those 16 internal clock ticks per bit, and which of the three buses is inherently full duplex?
  • It sends each bit 16 times for redundancy; I²C is full duplex.
  • It samples near the centre of each bit (≈ tick 8 of 16) to tolerate baud error; SPI is full duplex.
  • It clocks the data 16× faster onto the wire; UART is full duplex but I²C is not.
  • Oversampling sets the parity bit; all three buses are equally full duplex.
Oversampling-by-16 lets the receiver lock onto the start-bit edge and then sample each bit near its midpoint (around the 8th of 16 ticks), giving slack against the ±2% baud mismatch. SPI is full duplex — MOSI and MISO carry a bit each way on every clock. I²C is half duplex (one shared SDA line), and while UART has separate TX/RX wires, the question targets the synchronous buses: SPI is the full-duplex one among I²C/SPI.
Q2 Compute it
A USART is clocked from $f_{CK} = 84\,\text{MHz}$ and you want 115200 baud with oversampling by 16. Compute the exact USARTDIV (the fractional divisor before rounding). Recall $\text{USARTDIV} = \dfrac{f_{CK}}{16\cdot\text{baud}}$.
USARTDIV
$\text{USARTDIV} = \dfrac{84\times10^{6}}{16 \cdot 115200} = \dfrac{84\times10^{6}}{1\,843\,200} \approx 45.57$. You program mantissa $45$ and fraction $\lfloor 0.5729\times16 \rceil = 9$, giving $\text{BRR}=0x2D9$ and an actual baud of $115226$ — about $+0.02\%$ error, well inside the ±2% budget. (Sanity check the SPI side too: with $f_{PCLK}=84\,\text{MHz}$ and $\text{BR}=2$, $f_{SPI}=84/2^{3}=10.5\,\text{MHz}$.)
Q3 Coding challenge — non-blocking telemetry
Write a function send_telemetry() that formats a line like "t=12345 roll=1.20\r\n" into a buffer and ships it over USART1 via DMA, returning immediately. It must refuse to start a new transfer while a previous DMA send is still in flight (no buffer corruption). Use HAL.
Hint
Keep a volatile uint8_t tx_busy flag. Set it before calling HAL_UART_Transmit_DMA; clear it in HAL_UART_TxCpltCallback. If tx_busy is already set when send_telemetry() is called, just drop the line (or return a "busy" status) instead of overwriting the buffer the DMA is still reading.
Show one solution
telemetry.c C / STM32 HAL
static uint8_t tx_buf[64];
static volatile uint8_t tx_busy = 0;

// Returns 1 if the line was queued, 0 if a DMA send is still in flight.
int send_telemetry(uint32_t t_ms, float roll) {
  if (tx_busy) return 0;                 // previous DMA still reading tx_buf
  int n = snprintf((char*)tx_buf, sizeof tx_buf,
                   "t=%lu roll=%.2f\r\n", (unsigned long)t_ms, roll);
  if (n <= 0) return 0;
  tx_busy = 1;                           // claim the buffer BEFORE arming DMA
  if (HAL_UART_Transmit_DMA(&huart1, tx_buf, (uint16_t)n) != HAL_OK) {
    tx_busy = 0;                         // arming failed -> release the buffer
    return 0;
  }
  return 1;                              // DMA streams it; CPU is free now
}

// HAL fires this from the DMA transfer-complete ISR.
void HAL_UART_TxCpltCallback(UART_HandleTypeDef *huart) {
  if (huart->Instance == USART1) tx_busy = 0;  // buffer free to reuse
}

The tx_busy flag is the whole lesson: DMA reads tx_buf from SRAM asynchronously, so you must own the buffer for the full transfer. Claiming it before arming and releasing it only in the completion callback makes the handoff race-free. A production system would replace the "drop the line" policy with a ring buffer so nothing is lost.
