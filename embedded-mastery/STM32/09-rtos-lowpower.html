FreeRTOS & Low Power: Tasks, Scheduling & Sleep Modes

Two things separate a hobby project from a shipped product: it does many independent jobs at once without dropping a beat, and it runs for months on a coin cell. Concurrency and power. We'll earn both — first by replacing the superloop with a preemptive RTOS, then by putting the chip to sleep so deeply it sips microamps between heartbeats.

⏱ ~55 min🧠 Concurrency⚡ FreeRTOS🔋 Low power

1When the superloop breaks

① Intuition build the picture first

Every chapter so far has lived inside one infinite while(1) — the superloop. Read a sensor, update a display, blink an LED, check a button, repeat. For a handful of jobs with loose timing, it's perfect: simple, deterministic, no hidden machinery.

It breaks the moment your jobs have different, tight, independent timing requirements. Imagine three jobs on one chip:

JobMust runTakes
Read IMU over SPIevery 1 ms~0.2 ms
Stream telemetry over UARTevery 50 ms~8 ms (blocks!)
Service a display refreshevery 100 ms~12 ms (blocks!)

In a superloop, the moment you start that 8 ms UART transmit, the IMU read is stuck waiting — you'll miss eight of your 1 ms deadlines. You can hand-roll a state machine to interleave everything, but it grows into an unmaintainable tangle the instant requirements change. The honest fix is to stop pretending one loop can serve every master.

💡 The one-sentence model

An RTOS gives you many independent "loops" (tasks) that appear to run at once. A scheduler rapidly switches the single CPU between them, and a preemptive scheduler can yank the CPU away from a low-priority task the instant a high-priority one is ready.

"Real-time" doesn't mean "fast" — it means predictable. A real-time system guarantees that a deadline is met every single time, not on average. FreeRTOS, the most widely deployed RTOS on Cortex-M, gives you the primitives to build that guarantee; the math in §5 tells you whether your guarantee actually holds.

2Tasks, priorities & the preemptive scheduler

① Intuition what a task actually is

A task in FreeRTOS is just a C function that never returns — it loops forever, exactly like a tiny superloop of its own. Each task gets its own stack and its own copy of the CPU registers, saved and restored on every switch. That saved state is the context; swapping it is a context switch.

🔑 A task is a function that loops forever

It must never return and never run to the closing brace. Either loop on for(;;) forever, or delete yourself with vTaskDelete(NULL) before falling off the end. A task that returns crashes the scheduler.

Tasks have priorities (integer; higher number = more urgent, 0 = lowest). The scheduler's rule on a Cortex-M port is brutally simple: always run the highest-priority task that is ready to run. If two share the top priority, it time-slices between them round-robin on each tick.

The scheduler is driven by a periodic timer interrupt — the tick, usually 1 kHz (every 1 ms), generated from SysTick (Chapter 02). On each tick the scheduler re-evaluates who should run. Between ticks, a task that blocks (waits for data, sleeps, takes a held mutex) yields immediately — you don't have to wait for the next tick to give up the CPU.

① Intuition task states

At any instant every task is in exactly one of four states. Master this diagram and FreeRTOS stops being mysterious:

🔑 The idle task is mandatory

FreeRTOS auto-creates one task at priority 0: the idle task. It runs only when every other task is Blocked or Suspended. Its job is to free the memory of deleted tasks and — crucially for us — to be the hook where we put the CPU to sleep (§7). If the idle task never runs, the chip never saves power.

3Creating tasks: xTaskCreate, delays & stacks

③ Code read every line

You hand the scheduler a task with xTaskCreate: the function pointer, a debug name, a stack size (in words, not bytes!), an optional parameter, a priority, and a slot for the returned handle. Then vTaskStartScheduler() never returns — control passes to the scheduler forever.

main.c C / FreeRTOS
#include "FreeRTOS.h"
#include "task.h"

// A task is a function that loops forever. pvParameters is the
// optional argument you passed to xTaskCreate (here: unused).
void vBlinkTask(void *pvParameters)
{
    const TickType_t period = pdMS_TO_TICKS(500);  // 500 ms in ticks
    TickType_t wake = xTaskGetTickCount();         // snapshot "now"
    for (;;) {                                     // NEVER return
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);     // toggle the LED
        // Sleep until exactly wake+period. Fixed rate, no drift.
        vTaskDelayUntil(&wake, period);
    }
}

int main(void)
{
    HAL_Init();
    SystemClock_Config();                          // see Chapter 02
    MX_GPIO_Init();

    // stack is in WORDS: 128 words = 512 bytes on 32-bit Cortex-M.
    xTaskCreate(vBlinkTask,        // function
                "blink",           // name (debug only)
                128,               // stack depth in WORDS
                NULL,              // parameter
                2,                 // priority (0 = lowest)
                NULL);             // handle (we don't need it)

    vTaskStartScheduler();         // hands over the CPU — never returns
    for (;;) { }                   // unreachable; here for safety
}
⚠️ Pitfall — vTaskDelay drifts, vTaskDelayUntil doesn't

vTaskDelay(N) means "block for N ticks from now" — so the period silently grows by however long your work took. If the loop body runs 2 ms and you vTaskDelay(pdMS_TO_TICKS(500)), your true period is 502 ms and it drifts. vTaskDelayUntil(&wake, N) blocks until an absolute deadline, so the rate is rock-steady regardless of work time. Use it for any periodic sampling.

⚠️ Pitfall — stack overflow is silent corruption

Each task's stack is fixed at creation. A deep call chain, a big local array, or a printf (huge stack appetite) can run the stack past its end and quietly trample the next task's memory — a heisenbug that crashes somewhere else. Defenses: enable configCHECK_FOR_STACK_OVERFLOW = 2 (writes a known pattern and checks it), call uxTaskGetStackHighWaterMark() to see the closest you've come to the limit, and size stacks from measurement, not guesswork.

4Talking between tasks: queues, semaphores & mutexes

① Intuition three tools, three jobs

Tasks are isolated by design, so you need safe ways to pass data and coordinate. FreeRTOS gives you three primitives — and choosing the right one is a classic interview question because people confuse them constantly.

PrimitiveCarries data?Use it for
QueueYes — copies itemsPassing values producer→consumer (a thread-safe FIFO)
Semaphore (binary/counting)No — just a countSignaling: "an event happened" — especially ISR→task
MutexNo — ownership tokenMutual exclusion: guarding one shared resource (e.g. an I²C bus)
🔑 Semaphore vs mutex — the real difference

A binary semaphore is a flag any task or ISR can "give"; whoever's waiting gets woken. It has no owner — perfect for ISR-to-task signaling. A mutex is a semaphore with an owner: only the task that took it may give it back, and it adds priority inheritance. Use a semaphore to signal, a mutex to protect.

③ Code ISR signals a task via a semaphore

The canonical pattern: an interrupt fires, but you must not do heavy work in an ISR. Instead the ISR Gives a semaphore and a high-priority task, blocked waiting on it, wakes up to do the real work. This is "deferred interrupt processing".

sensor.c C / FreeRTOS
SemaphoreHandle_t xDataReady;   // created with xSemaphoreCreateBinary()

// --- Interrupt: keep it tiny. Just signal and leave. ---
void EXTI0_IRQHandler(void)
{
    BaseType_t hpw = pdFALSE;   // "higher-prio task woken?"
    __HAL_GPIO_EXTI_CLEAR_IT(GPIO_PIN_0);
    // FromISR variant is mandatory inside an interrupt:
    xSemaphoreGiveFromISR(xDataReady, &hpw);
    // If we woke a task more urgent than the interrupted one,
    // switch to it the instant we return from the ISR:
    portYIELD_FROM_ISR(hpw);
}

// --- Task: does the heavy lifting outside interrupt context ---
void vSensorTask(void *pv)
{
    for (;;) {
        // Block (0 % CPU) until the ISR gives the semaphore.
        // portMAX_DELAY = wait forever.
        if (xSemaphoreTake(xDataReady, portMAX_DELAY) == pdTRUE) {
            read_and_process_sample();   // safe: full task context
        }
    }
}
① Intuition priority inversion

Mutexes solve a subtle, famous bug. Suppose a low-priority task L takes a mutex on the I²C bus. A high-priority task H then wants the same bus and blocks waiting for L to release it. Fine so far — but now a medium-priority task M (which doesn't need the bus) becomes ready. M preempts L, because M outranks L. So M runs, L can't finish, L can't release the mutex, and H — the highest priority task in the system — is stuck behind M. The priorities are effectively inverted.

🏭 Real case — Mars Pathfinder, 1997

This isn't a textbook hypothetical. NASA's Mars Pathfinder kept resetting itself on the Martian surface because of exactly this priority inversion on a shared data bus protected by a mutex. The watchdog saw the high-priority task starve and rebooted the spacecraft. JPL diagnosed it remotely and uploaded a patch that enabled priority inheritance — the same feature one flag turns on in FreeRTOS today.

🔑 Priority inheritance

A FreeRTOS mutex fixes inversion automatically: while H waits on the mutex L holds, L is temporarily promoted to H's priority. Now M can't preempt L, so L finishes fast, releases the mutex, drops back to its own priority, and H runs. A plain semaphore has no owner, so it cannot do this — another reason to reach for a mutex when guarding a resource.

5Scheduling math: will it actually meet its deadlines?

② Math derive it

Creating tasks is easy; proving they all hit their deadlines is the real engineering. Model each periodic task $i$ by its worst-case execution time $C_i$ (how long one run takes) and its period $T_i$ (how often it must run). The fraction of the CPU it consumes is $C_i / T_i$, so total CPU utilization is the sum:

$$ U = \sum_{i=1}^{n} \frac{C_i}{T_i} $$

If $U > 1$ you've asked for more than 100% of the CPU — no scheduler can save you; it's mathematically impossible. But $U \le 1$ is necessary, not sufficient. For rate-monotonic scheduling (RMS — assign higher priority to the shorter-period task, the provably optimal fixed-priority rule), Liu & Layland proved a clean sufficient bound:

$$ U \le n\left(2^{1/n} - 1\right) $$

That bound shrinks as you add tasks. For one task it's $1.0$; for two, $2(\sqrt{2}-1)\approx 0.828$; and as $n\to\infty$ it converges to a famous limit:

$$ \lim_{n\to\infty} n\left(2^{1/n}-1\right) = \ln 2 \approx 0.693 $$
🔑 The 69% rule of thumb

If your total utilization stays under ~69%, rate-monotonic scheduling guarantees every deadline is met, no matter how many tasks. Above the bound it might still work (the bound is conservative — you'd run an exact response-time analysis), but under it you're safe by proof.

② Math worked example

Two tasks: an IMU read taking $C_1 = 2\,\text{ms}$ every $T_1 = 10\,\text{ms}$, and a control loop taking $C_2 = 1\,\text{ms}$ every $T_2 = 5\,\text{ms}$. Utilization:

$$ U = \frac{2}{10} + \frac{1}{5} = 0.2 + 0.2 = 0.4 $$

The two-task RMS bound is $2(2^{1/2}-1) \approx 0.828$. Since $0.4 \le 0.828$, this set is schedulable — every deadline provably met, with comfortable headroom for a third task or an interrupt storm. (And note RMS would give the 5 ms control loop the higher priority, because its period is shorter.)

6Low power: Run, Sleep, Stop & Standby

① Intuition trade wake-up speed for current

A battery product spends most of its life doing nothing. The art is making "nothing" cost almost no current. STM32 chips offer a ladder of low-power modes; each one shuts down more of the chip, saving more current but retaining less state and taking longer to wake. The numbers below are typical for an STM32L4 — the exact figures live in the datasheet, but the shape is universal.

ModeCPUSRAM / regs retainedWake sourcesTypical current
Runexecutingall—~5–10 mA
Sleephalted (clock off)allany interrupt~1–2 mA
Stopoff (clocks off)all SRAM + registers keptEXTI line, RTC, IWDG~1–10 µA
Standbyoff (core powered down)only backup regs + RTCWKUP pin, RTC, reset~0.1–1 µA
🔑 What survives the sleep

Sleep just gates the CPU clock — fastest wake, least savings, everything intact. Stop kills the clocks but keeps SRAM and registers powered, so you resume right where you left off; this is the workhorse for sensor nodes. Standby powers the core domain off — lowest current, but RAM is gone and waking is a near-full reset: you re-run main() and recover state only from the tiny battery-backed domain (RTC + a few backup registers).

② Math battery life from average current

Battery life is just charge divided by the rate you drain it. With capacity $Q$ (in mAh) and average current $I_{\text{avg}}$ (in mA):

$$ t_{\text{life}} = \frac{Q}{I_{\text{avg}}} $$

The whole game is duty cycling: be awake for a tiny fraction $d$ of the time at the high run current $I_{\text{run}}$, and in Stop mode the rest at the tiny $I_{\text{stop}}$. The time-weighted average is:

$$ I_{\text{avg}} = d\,I_{\text{run}} + (1-d)\,I_{\text{stop}} $$

Concrete: wake for $0.2\,\text{s}$ every $10\,\text{s}$ (so $d = 0.02$) at $I_{\text{run}} = 50\,\text{mA}$, idling in Stop at $I_{\text{stop}} = 10\,\mu\text{A} = 0.01\,\text{mA}$:

$$ I_{\text{avg}} = 0.02(50) + 0.98(0.01) \approx 1.0\,\text{mA} \;\Rightarrow\; t_{\text{life}} = \frac{2000\,\text{mAh}}{1.0\,\text{mA}} \approx 2000\,\text{h} \approx 83\,\text{days} $$

The same hardware without sleep — held in Run at 50 mA — would die in $2000/50 = 40$ hours, under two days. Stop mode bought us a 50× longer life from one design decision. That ratio is why this section exists.

7Tickless idle: making FreeRTOS sleep

① Intuition the tick is the enemy of sleep

Here's the tension: that 1 kHz tick wakes the CPU every millisecond just to ask "anyone ready?" If all tasks are blocked for the next 500 ms, you've pointlessly woken 500 times, and the chip never got deep enough to save real power. The fix is tickless idle.

🔑 What tickless idle does

When the idle task runs and FreeRTOS sees that every task is blocked for, say, the next 500 ms, it stops the tick, programs a low-power timer (the RTC or a wake timer) to fire in 500 ms, and drops the chip into Stop mode. On wake it reads how long it actually slept and corrects the tick count forward in one jump, so the kernel's sense of time stays accurate. You set configUSE_TICKLESS_IDLE = 1 to turn it on.

③ Code the idle hook that sleeps

The simplest entry point is the idle hook — a function FreeRTOS calls from the idle task on every spin. For a Sleep-mode design you can drop the core to sleep right here with the Cortex-M WFI ("wait for interrupt") instruction; for deeper Stop-mode sleeps you'd implement the full portSUPPRESS_TICKS_AND_SLEEP macro, but the idle hook shows the idea cleanly:

freertos_hooks.c C / FreeRTOS
// Called by the idle task whenever NOTHING else is ready to run.
// Requires configUSE_IDLE_HOOK == 1 in FreeRTOSConfig.h
void vApplicationIdleHook(void)
{
    // __WFI() halts the core until the next interrupt fires.
    // The tick interrupt (or any peripheral IRQ) wakes us, the
    // scheduler re-checks the ready list, and life goes on.
    // Current drops from ~mA (spinning) to a fraction of that.
    __WFI();   // "Wait For Interrupt" — Cortex-M sleep instruction
}

// For real deep sleep, enable tickless idle instead:
//   #define configUSE_TICKLESS_IDLE 1
// then FreeRTOS calls portSUPPRESS_TICKS_AND_SLEEP(xExpectedIdleTime),
// which you implement to enter STOP mode and reprogram the RTC.
⚠️ Pitfall — a busy task starves the idle task

The chip only sleeps when the idle task runs, which only happens when every other task is Blocked. One task that polls in a tight for(;;) loop without ever blocking (no vTaskDelay, no queue wait) pins the CPU in Run mode forever — your "low-power" node now draws full current. Always block on a delay or an event; never busy-wait.

8Real case: a battery sensor node

④ Real case this ships in products

Let's assemble everything into a real product: a wireless soil-moisture node that must run a year on a 2000 mAh cell. It has two jobs — sample the sensor and send a packet — and must be asleep the rest of the time. The architecture is a textbook RTOS + low-power design:

flowchart TB RTC([RTC wake
every 10 s]) --> SAMP["Sample task
read ADC, ~5 ms"] SAMP -->|xQueueSend| Q[["Queue
(moisture reading)"]] Q -->|xQueueReceive| COMMS["Comms task
TX radio packet, ~150 ms"] COMMS --> IDLE["Idle task
all blocked"] IDLE -->|tickless idle| STOP([Enter STOP mode
~10 µA]) STOP -.RTC fires.-> RTC

The sample task wakes on the RTC tick, reads the ADC, and pushes the reading into a queue. The comms task blocks on that queue; it only runs when a reading arrives, transmits it, then blocks again. Once both are blocked, the idle task runs and tickless idle drops the chip into Stop until the next RTC wake. Here are the two tasks and the queue wiring them together:

node.c C / FreeRTOS
QueueHandle_t xMoistureQ;   // created in main, holds uint16_t readings

// Producer: woken by the RTC ~every 10 s, sample and enqueue.
void vSampleTask(void *pv)
{
    const TickType_t period = pdMS_TO_TICKS(10000);   // 10 s
    TickType_t wake = xTaskGetTickCount();
    for (;;) {
        uint16_t moisture = adc_read(ADC_CHANNEL_3);  // ~5 ms of work
        // Non-blocking send; drop the sample if the queue is full.
        xQueueSend(xMoistureQ, &moisture, 0);
        vTaskDelayUntil(&wake, period);               // fixed 10 s rate
    }
}

// Consumer: sleeps until a reading exists, then transmits it.
void vCommsTask(void *pv)
{
    uint16_t reading;
    for (;;) {
        // Block (0 % CPU) until the producer sends. portMAX_DELAY
        // => the task is fully asleep, letting the chip reach STOP.
        if (xQueueReceive(xMoistureQ, &reading, portMAX_DELAY) == pdTRUE) {
            radio_transmit(&reading, sizeof reading);  // ~150 ms
        }
    }
}

int main(void)
{
    HAL_Init();  SystemClock_Config();  MX_ADC_Init();  MX_RTC_Init();
    xMoistureQ = xQueueCreate(4, sizeof(uint16_t));     // depth 4
    xTaskCreate(vSampleTask, "samp",  192, NULL, 3, NULL);
    xTaskCreate(vCommsTask,  "comms", 256, NULL, 2, NULL);
    vTaskStartScheduler();                              // never returns
    for (;;) { }
}
🏭 Real case — why the queue, not a shared global

You could let both tasks touch one global uint16_t — and you'd be one badly-timed context switch away from a torn read or a lost sample. The queue makes the handoff atomic and blocking: the consumer sleeps at zero CPU until data exists, which is exactly what lets the idle task run and the chip reach Stop. The queue isn't just tidy; it's what makes the low-power design possible. This pattern — producer, queue, consumer, deep sleep — is the skeleton of essentially every commercial battery sensor on the market.

Q1 Conceptual check
Two tasks share a single I²C bus, and you need to stop them from issuing overlapping transactions and avoid the priority-inversion bug from Mars Pathfinder. Which primitive do you reach for, and why?
  • A binary semaphore — it has the lowest overhead
  • A mutex — it has an owner and adds priority inheritance
  • A queue — it copies the bus data safely between tasks
  • A counting semaphore initialised to the number of tasks
A mutex is the right tool to protect a shared resource: only its owner can release it, and FreeRTOS mutexes implement priority inheritance, which is exactly the fix JPL applied to Pathfinder. A binary semaphore guards mutual exclusion too but has no owner and no inheritance, so it can't prevent the inversion. A queue is for passing data, not locking.
Q2 Compute it
A system runs two periodic tasks: an IMU read of $C_1 = 2\,\text{ms}$ every $T_1 = 10\,\text{ms}$, and a control loop of $C_2 = 1\,\text{ms}$ every $T_2 = 5\,\text{ms}$. Compute the total CPU utilization $U = \sum C_i / T_i$ (a unitless fraction between 0 and 1).
(fraction)
$U = \dfrac{2}{10} + \dfrac{1}{5} = 0.2 + 0.2 = 0.4$. The two-task rate-monotonic bound is $2(2^{1/2}-1)\approx 0.828$, and $0.4 \le 0.828$, so this set is schedulable with comfortable headroom. You're using 40% of the CPU; 60% is free for more tasks or interrupts.
Q3 Coding challenge — producer, queue, consumer
Write two FreeRTOS tasks linked by a queue. A producer reads a button (read_button() returns a uint8_t) every 100 ms at a fixed rate and sends the value into a queue. A consumer blocks on the queue and, whenever it receives a value, calls process(value). Don't forget to create the queue before starting the scheduler — and make sure the consumer truly blocks (so the chip can sleep) rather than polling.
Hint
Create the queue with xQueueCreate(depth, sizeof(uint8_t)) in main before vTaskStartScheduler(). In the producer use vTaskDelayUntil for the fixed 100 ms rate and xQueueSend(q, &val, 0). In the consumer use xQueueReceive(q, &val, portMAX_DELAY) — portMAX_DELAY means "block forever until data arrives", which keeps the task at 0% CPU between events.
Show one solution
button_node.c C / FreeRTOS
QueueHandle_t xBtnQ;

// Producer: sample the button at a fixed 100 ms rate.
void vProducer(void *pv)
{
    const TickType_t period = pdMS_TO_TICKS(100);
    TickType_t wake = xTaskGetTickCount();
    for (;;) {
        uint8_t val = read_button();        // 0 or 1
        xQueueSend(xBtnQ, &val, 0);          // non-blocking enqueue
        vTaskDelayUntil(&wake, period);      // exact 100 ms, no drift
    }
}

// Consumer: fully asleep until a value arrives.
void vConsumer(void *pv)
{
    uint8_t val;
    for (;;) {
        if (xQueueReceive(xBtnQ, &val, portMAX_DELAY) == pdTRUE) {
            process(val);                    // handle the event
        }
    }
}

int main(void)
{
    HAL_Init();  SystemClock_Config();  MX_GPIO_Init();
    xBtnQ = xQueueCreate(8, sizeof(uint8_t));   // depth 8
    xTaskCreate(vProducer, "prod", 128, NULL, 2, NULL);
    xTaskCreate(vConsumer, "cons", 192, NULL, 2, NULL);
    vTaskStartScheduler();                       // never returns
    for (;;) { }
}

The lesson: the consumer's portMAX_DELAY is what makes this power-friendly — it sits Blocked at zero CPU, so once the producer is also delayed, the idle task runs and tickless idle can drop the chip into Stop. Swap the queue for a binary semaphore and you'd signal "a press happened" without carrying which button — and swap vTaskDelayUntil back to vTaskDelay and you'd reintroduce the timing drift from §3.

🎓 Chapter recap

You can now explain why the superloop fails for many independent timing-sensitive jobs, and replace it with FreeRTOS tasks — functions that loop forever, scheduled preemptively by priority off the SysTick tick, with the mandatory idle task as your sleep hook. You create tasks with xTaskCreate, hold a fixed rate with vTaskDelayUntil (never the drifting vTaskDelay), and size stacks from uxTaskGetStackHighWaterMark. You pick the right IPC primitive — queue for data, semaphore for ISR-to-task signaling, mutex for mutual exclusion with priority inheritance (the Pathfinder fix). You can prove schedulability with $U = \sum C_i/T_i$ against the RMS bound $n(2^{1/n}-1)\to\ln 2\approx 0.693$. And you slash average current with Run/Sleep/Stop/Standby modes, computing battery life as $Q/I_{\text{avg}}$ and using tickless idle to reach Stop between events — turning a two-day device into a multi-month one. That's the leap from hobby to product.
