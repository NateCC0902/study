GPIO at the Register Level: MODER, BSRR & HAL

On Arduino you wrote digitalWrite(13, HIGH) and a pin went high. That one line hides a dozen hardware bits. On STM32 we're going to lift the hood completely: configure a pin by writing the exact registers the silicon reads, learn why one of them — BSRR — is the difference between correct firmware and a heisenbug that bites only when an interrupt fires. Then we'll do the same blink with HAL and know precisely what HAL did for us.

⏱ ~45 min🧠 Register model⚡ Bare-metal GPIO🔧 HAL vs CMSIS

1Ports, pins & the register file

① Intuition build the picture first

An STM32's pins aren't loose wires you address one at a time. They're organized into ports of 16 pins each: GPIOA, GPIOB, … through GPIOK on the largest parts. The pin we'll obsess over all chapter — the green user LED on a Nucleo-F446 board — is PA5: port A, pin 5.

💡 The one-sentence model

Each GPIO port is a small block of memory-mapped registers; configuring or driving a pin is nothing more than writing the right bits into the right register at the port's base address.

Because ports come in groups of 16, the hardware packs the control for all 16 pins into a handful of 32-bit registers. Two bits per pin × 16 pins = 32 bits — exactly one register. That arithmetic is not a coincidence; it's why MODER, OSPEEDR, and PUPDR are all "2 bits per pin" registers, and why OTYPER and IDR, which need only one bit per pin, leave their upper half unused.

In code, a port is a C struct overlaid on its base address. CMSIS (the vendor header set) gives you GPIOA as a pointer to that struct, so GPIOA->MODER dereferences straight to the silicon. Here is the register file you'll meet, in declaration order:

⚠️ Pitfall — the port is born unclocked

As we hammered in Chapter 02, every peripheral wakes up with its clock off. Read GPIOA->MODER before enabling the GPIOA clock and you'll get zeros and your writes silently vanish. The very first GPIO line in any STM32 program is always an RCC clock-enable.

2MODER: the math of two bits per pin

② Math derive the bit positions

MODER is the register that decides what a pin is. Each pin gets a 2-bit field, and the four values are:

ValueModeMeaning
00InputHi-Z digital input (read via IDR)
01OutputGeneral-purpose output (drive via ODR/BSRR)
10AlternateA peripheral owns the pin (timer, UART, SPI…)
11AnalogADC/DAC, or lowest-power "disconnect"

The two bits for pin $n$ sit at positions $2n$ (low) and $2n+1$ (high). That's the whole rule:

$$ \text{MODER field for pin } n \;=\; \text{bits } [\,2n+1 : 2n\,] $$

For our LED on PA5, substitute $n=5$:

$$ 2n = 10, \qquad 2n+1 = 11 \;\;\Rightarrow\;\; \text{PA5 uses MODER bits } [\,11:10\,] $$

To make PA5 an output we need the value 01 in that field. As a raw 32-bit word, that is $1$ shifted left into bit 10:

$$ \texttt{0b01} \ll 10 \;=\; 1 \times 2^{10} \;=\; 1024 \;=\; \texttt{0x00000400} $$
∑ The read-modify-write idiom

You can't just write 1024 to MODER — that would also clobber the other 15 pins' fields to 00. You must clear the field, then set it: first MODER &= ~(0b11 << 10) zeros both bits of PA5, then MODER |= (0b01 << 10) writes the new value. Clear-then-set is the universal pattern for any multi-bit field.

Generalize the shift and you have the formula every register macro is built on. For a 2-bit field, pin $n$'s shift is $2n$; for the 4-bit AFR nibble it's $4n$ (modulo the low/high split). Memorize the family:

$$ \text{shift}_{\,k\text{-bit field}}(n) \;=\; k \cdot n, \qquad \text{mask} \;=\; \big(2^{k}-1\big) \ll k n $$

3BSRR: why atomic writes matter

① Intuition the race you can't see

Once a pin is an output, there are two ways to change its level. The obvious one is ODR — the output data register, one bit per pin. To set PA5 you'd write GPIOA->ODR |= (1 << 5). That looks innocent. It is a trap.

|= is not one machine operation. The CPU must read the whole 32-bit ODR into a register, OR in your bit, then write the result back — three steps. If an interrupt fires between the read and the write, and that ISR also touches a pin on the same port, your write-back uses a stale copy of ODR and silently undoes the ISR's change.

⚠️ Pitfall — the lost-update race

Main code runs ODR |= (1<<5). It reads ODR, gets interrupted, the ISR does ODR |= (1<<3) on a different pin and returns. Main resumes and writes back its stale value — pin 3 just got cleared. Nothing errors. The bug appears once an hour, in the field, never on your bench. This is the single most common bare-metal GPIO bug.

② Math the two-halves register

STM32 fixes this with BSRR — Bit Set/Reset Register. It is 32 bits split into two halves, and crucially it is write-only with no read, so there's nothing to read-modify-write:

$$ \underbrace{\text{BR}_{15}\ldots\text{BR}_{0}}_{\text{bits }[31:16]\;\text{RESET}} \;\;\Big|\;\; \underbrace{\text{BS}_{15}\ldots\text{BS}_{0}}_{\text{bits }[15:0]\;\text{SET}} $$

Writing a 1 to bit $n$ (low half) sets pin $n$. Writing a 1 to bit $n+16$ (high half) resets it. Writing 0 anywhere does nothing. So for PA5:

$$ \text{set PA5: } \texttt{BSRR} = 1 \ll 5 = \texttt{0x0020}; \qquad \text{reset PA5: } \texttt{BSRR} = 1 \ll (5{+}16) = \texttt{0x0020\,0000} $$
🔑 Why this is correct by construction

A write to BSRR is a single store instruction the hardware applies atomically — it touches only the bits you wrote 1 to and leaves every other pin alone. There is no read step, so there is no window for an interrupt to corrupt. Always drive outputs with BSRR, never ODR |=. HAL's HAL_GPIO_WritePin uses BSRR internally for exactly this reason.

You can even set and reset different pins in one write: BSRR = (1<<5) | (1<<(3+16)) sets PA5 and clears PA3 simultaneously — one atomic store, two pins. If you set and reset the same pin in one write, reset wins: the high half has priority.

🛠️ Where do the addresses come from?

On STM32F446, GPIOA's base is 0x40020000 and MODER is at offset 0x00, IDR at 0x10, ODR at 0x14, BSRR at 0x18. You never hard-code these — the CMSIS header stm32f4xx.h declares the GPIO_TypeDef struct so the compiler computes them. GPIOA->BSRR is *(volatile uint32_t*)(0x40020000 + 0x18).

5Blink PA5 from registers, fully annotated

③ Code read every line

Now we assemble everything: enable the clock, configure PA5 as a push-pull output with no pull, and toggle it forever using BSRR. This is pure CMSIS — no HAL, no Arduino — and it is exactly what the silicon executes.

blink_reg.c C / CMSIS
#include "stm32f4xx.h"   // CMSIS: GPIOA, RCC, register bit masks

#define LED_PIN  5       // PA5 = green user LED on Nucleo-F446

void delay(volatile uint32_t n) { while (n--) __NOP(); }

int main(void) {
    // 1) CLOCK: GPIOA lives on the AHB1 bus. Enable its clock first,
    //    or every write below is swallowed by an unclocked peripheral.
    RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN;

    // 2) MODE: set MODER[11:10] = 0b01 (output) for PA5.
    //    Clear the 2-bit field first, then OR in the new value.
    GPIOA->MODER &= ~(0x3u << (2 * LED_PIN));   // clear bits 11:10
    GPIOA->MODER |=  (0x1u << (2 * LED_PIN));   // 01 = output

    // 3) OUTPUT TYPE: 0 = push-pull (drives both high AND low).
    GPIOA->OTYPER &= ~(0x1u << LED_PIN);

    // 4) PULL: 00 = no pull-up/pull-down (an LED needs neither).
    GPIOA->PUPDR &= ~(0x3u << (2 * LED_PIN));

    while (1) {
        // SET PA5 — low half of BSRR. One atomic store, no read.
        GPIOA->BSRR = (1u << LED_PIN);          // == 0x00000020
        delay(800000);
        // RESET PA5 — high half of BSRR, bit (pin + 16).
        GPIOA->BSRR = (1u << (LED_PIN + 16));   // == 0x00200000
        delay(800000);
    }
}
💡 What each block really does

(1) flips one bit in RCC->AHB1ENR to route a clock to GPIOA. (2) clear-then-set on the 2-bit MODER field makes the pin a driven output. (3) & (4) choose push-pull and no internal resistor. The loop never calls a function to "write" the pin — it just stores a constant into BSRR. That store is the digitalWrite, and it's atomic.

⚠️ Pitfall — the missing volatile

delay()'s counter is volatile on purpose. Without it, an optimizing compiler sees a loop that computes nothing observable and deletes it entirely — your LED appears to never blink. The CMSIS register structs are already declared volatile, which is why GPIOA->BSRR = … is never optimized away.

6The same blink in HAL — and what HAL hid

③ Code same result, different altitude

STM32's HAL (Hardware Abstraction Layer) wraps those registers in named structs and functions. Here is the identical blink. Watch how each HAL line maps to a register write you now understand.

blink_hal.c C / STM32 HAL
#include "stm32f4xx_hal.h"

int main(void) {
    HAL_Init();
    // __HAL_RCC_GPIOA_CLK_ENABLE() expands to the SAME line as before:
    //   RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN;  (+ a read-back for sync)
    __HAL_RCC_GPIOA_CLK_ENABLE();

    GPIO_InitTypeDef cfg = {0};
    cfg.Pin   = GPIO_PIN_5;              // bit mask 1<<5, NOT the number 5
    cfg.Mode  = GPIO_MODE_OUTPUT_PP;     // -> MODER=01, OTYPER=0 (push-pull)
    cfg.Pull  = GPIO_NOPULL;             // -> PUPDR=00
    cfg.Speed = GPIO_SPEED_FREQ_LOW;     // -> OSPEEDR=00
    HAL_GPIO_Init(GPIOA, &cfg);          // does the clear-then-set for us

    while (1) {
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);  // reads ODR, writes BSRR
        HAL_Delay(500);                          // 500 ms via SysTick
    }
}
🔑 HAL is a thin glove, not a black box

HAL_GPIO_Init does precisely the clear-then-set on MODER/OTYPER/OSPEEDR/PUPDR you wrote by hand. HAL_GPIO_WritePin writes BSRR (set in the low half, reset in the high half). HAL_GPIO_TogglePin reads the current ODR bit and writes the opposite half of BSRR. Knowing the registers means you can read HAL's source, debug it, and bypass it where the cycles matter.

⚠️ Pitfall — GPIO_PIN_5 is a mask, not an index

HAL's GPIO_PIN_5 is (1 << 5) = 0x20, a bitmask — not the integer 5. Pass the number 5 where HAL wants a mask and you'll configure pin 2 and pin 0 (because 5 = 0b101). The bare-metal side uses the index 5 with explicit shifts; HAL uses the pre-shifted mask. Don't mix them.

7Alternate function & reading a button

① Intuition handing a pin to a peripheral

So far PA5 has been a plain output we drive from software. But the same pin can instead be wired to a peripheral: a timer's PWM channel, a UART's TX line, an SPI clock. That's what MODER = 10 (Alternate) means — and AFRL/AFRH select which of the 16 possible alternate functions (AF0–AF15) the pin routes to.

Each pin gets a 4-bit nibble. Pins 0–7 live in AFRL (the low register), pins 8–15 in AFRH. PA5's alternate-function 1 (AF1) is TIM2_CH1 — the timer channel we'll use for PWM in a later chapter. To route PA5 to a peripheral you set MODER=10 and write the AF nibble:

$$ \text{AFR nibble shift for pin } n<8 \;=\; 4n \quad\Rightarrow\quad \text{PA5} \to \text{AFRL bits }[23:20] $$
af_pwm_pin.c C / CMSIS
// Route PA5 to TIM2_CH1 (alternate function AF1) for hardware PWM.
GPIOA->MODER  &= ~(0x3u << (2 * 5));   // clear MODER[11:10]
GPIOA->MODER  |=  (0x2u << (2 * 5));   // 10 = ALTERNATE function

GPIOA->AFR[0] &= ~(0xFu << (4 * 5));   // AFR[0] is AFRL; clear nibble[23:20]
GPIOA->AFR[0] |=  (0x1u << (4 * 5));   // AF1 = TIM2_CH1
// From here the TIMER, not the CPU, owns the pin's voltage.
③ Code now read an input

Reading is the mirror image. Configure the pin as input (MODER=00), enable a pull-up so the line idles high, then read bit $n$ of IDR. The on-board user button B1 on a Nucleo is wired to PC13 and pulls to ground when pressed, so a pull-up plus "pressed = 0" is the natural reading.

button.c C / CMSIS
RCC->AHB1ENR |= RCC_AHB1ENR_GPIOCEN;      // clock port C

GPIOC->MODER &= ~(0x3u << (2 * 13));       // 00 = input (PC13)
GPIOC->PUPDR &= ~(0x3u << (2 * 13));       // clear pull field
GPIOC->PUPDR |=  (0x1u << (2 * 13));       // 01 = pull-UP, line idles high

// IDR bit 13 reads the LIVE pin voltage. Button grounds it when pressed.
if ((GPIOC->IDR & (1u << 13)) == 0) {
    // pressed: pin pulled to 0 V against the pull-up
}
🛠️ Why a pull resistor is mandatory

A bare input pin with nothing driving it is floating — its voltage drifts on noise and IDR returns random 1s and 0s. PUPDR's internal pull-up (~40 kΩ) ties the line to a known level so an open switch reads a clean 1 and a press reads 0. No pull, no reliable read.

8Real case: how fast can a pin toggle?

④ Real case this ships in products

"Toggle a pin" sounds trivial until you need to bit-bang a protocol — drive a clock, a NeoPixel strip, or a soft-SPI line by hand because no spare peripheral is free. Then the maximum toggle frequency becomes a hard limit on what you can talk to.

On an STM32F446 the CPU runs at up to $f_{\text{clk}} = 180\,\text{MHz}$, and GPIO sits on the AHB1 bus at the full HCLK. A BSRR write is a single-cycle store from the CPU's view. In a tight assembly loop that alternates set/reset, the practical ceiling is roughly HCLK divided by the cycles per toggle:

$$ f_{\text{toggle}} \;\approx\; \frac{f_{\text{clk}}}{\text{cycles per edge}} \;=\; \frac{180\,\text{MHz}}{\sim 2} \;\approx\; 90\,\text{MHz (best case, asm loop)} $$

A realistic C while(1){ BSRR=set; BSRR=reset; } loop, with loop overhead, lands around $10\text{–}45\,\text{MHz}$ depending on flash wait-states and the instruction cache. Now compare the Arduino Uno's digitalWrite(): it does pin-to-port lookups, interrupt-disabling, and a read-modify-write each call — about 50 CPU cycles at 16 MHz, so under ~150 kHz. Direct port writes on the AVR (PORTB ^= 1<<5) reach ~2–4 MHz. The register-level STM32 is two to three orders of magnitude faster.

🏭 Real case — bit-banging a WS2812 LED strip

WS2812 "NeoPixel" LEDs have no clock line; each bit is encoded purely by how long the data pin stays high — a 1 is ~0.8 µs high, a 0 is ~0.4 µs high, with ±150 ns tolerance. Hitting that timing means counting CPU cycles between BSRR writes. On a 16 MHz AVR it's painfully tight and must be hand-written in assembly; on a 180 MHz STM32 with single-cycle BSRR you have cycles to spare — which is exactly why fast LED-strip drivers moved to ARM. The toggle-speed math above is the budget that decides whether your protocol is even possible in software.

Q1 Conceptual check
Why is GPIOA->BSRR = (1<<5) preferred over GPIOA->ODR |= (1<<5) to set a pin?
  • BSRR is faster because it skips the clock-enable step
  • BSRR is a single atomic write, so an interrupt can't corrupt a read-modify-write
  • ODR can only be written once per power cycle
  • BSRR drives more current onto the pin
ODR |= is a read-modify-write: read ODR, OR the bit, write back. An interrupt landing between the read and the write can change another pin on the port, and the stale write-back undoes it. BSRR is write-only and applies as one atomic store touching only the bits you set — no read, no race.
Q2 Compute it
In MODER, pin $n$'s 2-bit field occupies bits $[\,2n{+}1 : 2n\,]$. To make PA5 an output you write the value 01 into that field, i.e. 0b01 << X. What is the shift amount X (the low bit position of the field)?
bit
$2n = 2 \times 5 = 10$, so PA5's field is bits $[11:10]$ and the shift is 10. Thus 0b01 << 10 = 1024 = 0x400, which is the value you OR into MODER (after clearing bits 11:10) to make PA5 an output.
Q3 Coding challenge — register-level blink on PA5
From scratch, write the bare-metal setup that makes PA5 a push-pull output and then toggles it forever without ever reading ODR. You may keep a software state flag and pick the BSRR half from it. No HAL allowed.
Hint
Enable the GPIOA clock (RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN), clear-then-set MODER[11:10]=01, then in the loop write BSRR = (1<<5) to go high and BSRR = (1<<21) — that's 1<<(5+16) — to go low. Flip a local state each pass so you never need to read the pin back.
Show one solution
blink_pa5.c C / CMSIS
#include "stm32f4xx.h"
#define PIN 5

static void delay(volatile uint32_t n){ while(n--) __NOP(); }

int main(void) {
    RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN;        // 1) clock GPIOA

    GPIOA->MODER &= ~(0x3u << (2 * PIN));        // 2) clear MODER[11:10]
    GPIOA->MODER |=  (0x1u << (2 * PIN));        //    01 = output
    GPIOA->OTYPER &= ~(0x1u << PIN);            //    push-pull
    GPIOA->PUPDR  &= ~(0x3u << (2 * PIN));      //    no pull

    uint32_t state = 0;
    while (1) {
        // pick BSRR half from state: high half (reset) when state, else set
        GPIOA->BSRR = state ? (1u << (PIN + 16))    // reset: bit 21
                            : (1u <<  PIN);         // set:   bit 5
        state ^= 1u;
        delay(800000);
    }
}

The lesson: every level change is a single atomic BSRR store, chosen by a software flag — never a read of the hardware. Swap the delay() for a SysTick timestamp (Chapter 02) and you have a non-blocking heartbeat. Route the same pin to TIM2_CH1 via AFR instead, and the timer blinks it for free while the CPU sleeps.

🎓 Chapter recap

You can now configure any STM32 pin straight from CMSIS: enable its port clock in RCC->AHB1ENR, set the 2-bit MODER field at bits $[2n{+}1:2n]$, choose push-pull/open-drain in OTYPER and a pull in PUPDR. You drive outputs atomically through BSRR — set in the low half, reset in the high half — and you know exactly why ODR |= is a lurking race. You can read inputs through IDR with a pull-up, hand a pin to a peripheral with MODER=10 plus the AFR nibble, and you've seen that the same blink in HAL is just these register writes wearing names.
