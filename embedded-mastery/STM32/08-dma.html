DMA Deep Dive: Channels, Circular Mode & Double Buffering

DMA is the moment STM32 stops feeling like a faster Arduino and starts feeling like a different class of machine. A second engine moves bytes between peripherals and memory while the CPU computes — or sleeps. Master it and you unlock gapless audio, megabaud logging, and LED strips that never flicker. We build the model from the bus up, do the throughput math, then wire a real circular-DMA ADC capture.

⏱ ~50 min🧠 Bus arbitration⚡ Circular & double buffer🔧 ADC · UART · SPI

1What is DMA, really?

① Intuition build the picture first

Without DMA, every byte that moves between a peripheral and memory passes through the CPU. The ADC finishes a conversion, raises a flag, the CPU wakes, reads ADC1->DR into a register, writes that register into a RAM array, increments a pointer, checks a counter, loops. For one 16-bit sample that is a dozen instructions. At 1 mega-sample per second the CPU does nothing but shuttle bytes — it has no cycles left to actually compute.

💡 The one-sentence model

DMA (Direct Memory Access) is a second data-mover — a tiny dedicated engine on the bus that copies bytes between a peripheral and memory without the CPU touching a single one. You program it once with "from here, to there, this many, this wide" and it runs in the background while the core computes or sleeps.

The DMA controller is a bus master, exactly like the CPU. It arbitrates for the AHB/APB buses, performs the read and the write, increments addresses, decrements a counter, and raises an interrupt only when there is something worth telling you — typically "buffer half full" or "buffer full," not "one byte arrived." That collapse from per-byte CPU work to per-buffer CPU work is the entire point.

🔑 Key distinction

Polling and interrupts both cost CPU time per data item. DMA costs CPU time per transfer block. The peripheral's data-ready signal goes straight to the DMA engine as a hardware request — the core is never in the loop. That is why DMA throughput is bounded by the bus, not by your interrupt latency.

2Controller architecture: streams, channels & request mapping

① Intuition how the silicon is wired

The hardware differs by family, and the vocabulary trips up everyone once. On the STM32F4 (Cortex-M4) there are two DMA controllers, DMA1 and DMA2. Each controller has 8 streams. Each stream can be connected to one of 8 channels, and the channel selects which peripheral drives that stream's request line. So a stream is the physical transfer engine; a channel is a multiplexer selecting its trigger source.

🔑 The naming trap (F4 vs F1)

On F4: a stream is the engine, and you pick a channel (0–7) to route the peripheral request to it. On the older STM32F1 (Cortex-M3) there are no streams — DMA1 has 7 channels, DMA2 has 5, and each channel is the engine with a fixed peripheral mapping. Same idea, different words. When a tutorial says "DMA channel," check which family it means.

The request mapping is fixed in silicon and printed in the reference manual as a table. You do not get to wire any peripheral to any stream — for example, on the F4, ADC1 is DMA2, Stream 0, Channel 0 (and also Stream 4, Channel 0). SPI1_TX is DMA2, Stream 3, Channel 3. Pick the wrong stream/channel pair and the peripheral's request simply never reaches the engine and nothing moves. This is the single most common "my DMA does nothing" bug.

⚠️ Pitfall — stream collisions

Two peripherals that share the same stream cannot run DMA simultaneously, even if they use different channels — there is only one engine per stream. If you need ADC1 and SPI1_TX both on DMA2, choose their alternate stream mappings so they land on different streams. Always cross-check the request table before committing a pin/peripheral plan.

3Anatomy of a transfer: the six parameters

① Intuition what you actually configure

Every DMA transfer is defined by a small set of parameters. Get these right and the engine does exactly what you mean; get one wrong and you corrupt RAM or read garbage. Here is the whole vocabulary:

ParameterRegister fieldMeaning
DirectionDMA_SxCR.DIRPeripheral→Memory, Memory→Peripheral, or Memory→Memory
Peripheral addressDMA_SxPARSource/dest peripheral data register, e.g. &ADC1->DR
Memory addressDMA_SxM0ARYour RAM buffer base pointer
Data widthPSIZE / MSIZEByte (8), half-word (16), or word (32) per side
IncrementPINC / MINCStep the address after each item — usually fixed peripheral, incrementing memory
CountDMA_SxNDTRNumber of data items to move (not bytes)
⚠️ Pitfall — NDTR counts items, not bytes

NDTR is the number of transfers, sized by the source width. If PSIZE = half-word and you want to capture 1024 ADC samples, set NDTR = 1024, not 2048. The engine moves NDTR items and stops (normal mode) or reloads (circular). Off-by-a-factor-of-two here is a classic buffer overrun.

🛠️ Increment is the magic

The peripheral data register lives at one fixed address — its DR — so PINC = 0. Your RAM buffer is an array, so MINC = 1 and the engine walks buf[0], buf[1], buf[2] … automatically. That single asymmetry — fixed peripheral, incrementing memory — is what turns a stream of DR reads into a populated array with zero CPU effort.

4Normal vs circular mode

① Intuition one-shot vs forever

In normal mode, the engine moves NDTR items, raises the transfer-complete interrupt, then stops. To go again you must re-arm it: clear flags, reload NDTR, re-enable the stream. Fine for a one-shot block read; useless for a continuous stream because the gap while you re-arm means dropped samples.

In circular mode (set DMA_SxCR.CIRC = 1), when the counter hits zero the hardware automatically reloads NDTR and the memory pointer back to the start, and keeps going. The buffer becomes a ring: the ADC writes buf[0..N-1], wraps to buf[0], forever, with no CPU intervention. This is how an ADC or an audio codec streams indefinitely.

🔑 When to reach for circular mode

Use circular mode whenever data arrives continuously and indefinitely at a fixed rate — ADC sampling, I²S/audio, periodic DAC waveform output, continuous UART reception. Use normal mode for finite, on-demand transfers — read 256 bytes from a flash chip, send one telemetry packet. The rule: continuous stream → circular; bounded job → normal.

5Double buffering & the half-transfer trick

① Intuition fill one, process the other

Circular mode solves "never stop," but it creates a new hazard: if the engine is writing into the same buffer you are reading, you get a race — half-fresh, half-stale data. The fix is to split the work in time so the producer (DMA) and consumer (CPU) never touch the same region at once.

The cheapest version uses the half-transfer (HT) and transfer-complete (TC) interrupts of a single circular buffer. The HT interrupt fires when the engine crosses the midpoint, meaning the first half is full and safe to process while the engine fills the second half. The TC interrupt fires at wrap, meaning the second half is full and the engine is now refilling the first. You always process the half the engine is not writing.

💡 Ping-pong, the picture

Think of two buckets under a tap that the tap fills alternately. While bucket A fills, you empty bucket B; the instant A is full the tap swings to B and you empty A. Nobody ever reaches into the bucket currently under the tap. Gapless — the tap never has to stop. STM32 even offers true double-buffer mode (DMA_SxCR.DBM = 1) with two distinct addresses, M0AR and M1AR, that the engine swaps automatically and reports via the CT (current target) bit.

⚠️ Pitfall — too slow to keep up

Double buffering only works if you finish processing one half before the engine finishes filling the other. If your handler runs longer than the half-buffer fill time, the engine laps you and overwrites unprocessed data — a silent overrun. The fix is a bigger buffer (more time per half) or a faster handler. We compute that fill-time budget in the next section.

6Throughput math: bytes/s & buffer-fill time

② Math where the numbers come from

Sizing a DMA buffer is arithmetic, and getting it right is the difference between gapless streaming and mysterious clicks. The raw data rate a stream produces is the product of three things — how often, how many parallel channels, and how wide each sample is:

$$ R = f_s \cdot N_{ch} \cdot W \quad\left[\tfrac{\text{bytes}}{\text{s}}\right] $$

where $f_s$ is the sample rate, $N_{ch}$ the channel count, and $W$ the bytes per sample. Take CD-quality stereo audio: $f_s = 48\,\text{kHz}$, $N_{ch} = 2$ (left + right), $W = 2$ bytes (16-bit):

$$ R = 48{,}000 \times 2 \times 2 = 192{,}000 \;\tfrac{\text{B}}{\text{s}} \approx 187.5\,\text{KiB/s} $$

That is the byte stream the DMA engine must absorb forever. Now the buffer-fill time. If your circular buffer holds $B$ bytes, the time to fill it completely is simply size over rate:

$$ t_{\text{fill}} = \frac{B}{R}, \qquad t_{\text{half}} = \frac{B}{2R} $$

For a 1024-sample stereo buffer, $B = 1024 \times 2 \times 2 = 4096$ bytes, so

$$ t_{\text{fill}} = \frac{4096}{192{,}000} \approx 21.3\,\text{ms}, \qquad t_{\text{half}} \approx 10.7\,\text{ms} $$

That $t_{\text{half}} \approx 10.7\,\text{ms}$ is your real-time budget: every HT/TC interrupt, your handler has ~10.7 ms to process 512 samples before the engine laps it. Miss the budget and you overrun. Bigger buffer → more slack but more latency; it is always this trade. One more reframing — for a single mono channel, the time between individual samples is just the period $T_s = 1/f_s$, and the engine must complete one DR→RAM transfer (a handful of bus cycles) well inside it.

7Bus arbitration & the FIFO

① Intuition sharing the bus

The DMA engine and the CPU are both bus masters competing for the same AHB matrix. Only one master drives a given bus slave per cycle, so a bus arbiter grants access. On Cortex-M3/M4 STM32s the matrix is multi-layer, so DMA-to-SRAM and CPU-to-Flash can often proceed in parallel — but when they target the same slave (both want SRAM), the arbiter interleaves them and the loser stalls a cycle. DMA usually wins by priority, so heavy DMA traffic can mildly slow the CPU. This is the price of "free" transfers — it is cheap, not literally free.

The F4 streams add a FIFO: a small (4×32-bit) buffer between the peripheral side and the memory side. It earns its keep two ways. First, burst & packing: it lets the engine collect several narrow peripheral reads and write them to memory in one wide burst, cutting the number of bus transactions. Second, width conversion: with the FIFO on you can read bytes from a peripheral and pack them into 32-bit memory words. With the FIFO disabled (direct mode), PSIZE and MSIZE must match and every item is an immediate single transfer.

🏭 Real case — why the FIFO matters for SD cards

High-bandwidth peripherals like the SDIO/SDMMC interface to an SD card require FIFO mode with bursts. The SD clock does not pause for bus contention, so the DMA must use burst transfers to keep up with the controller's hardware FIFO; direct mode simply cannot sustain the rate and you get underrun errors. The FIFO threshold (1/4, 1/2, 3/4, full) tunes how full the FIFO gets before it bursts — a real knob you turn when chasing throughput.

8Real cases: ADC, UART, WS2812 & mem-to-mem

③ Code read every line

Here is the headline use: a free-running ADC streaming straight into a circular buffer. The CPU configures it once and is then completely uninvolved in moving samples. This is register-level for clarity; the HAL wraps the same fields.

adc_dma_circular.c C / STM32F4 (register level)
// ADC1 -> SRAM, circular, half-word samples.
// F4 mapping: ADC1 = DMA2, Stream 0, Channel 0.
#define NSAMP 1024
volatile uint16_t adc_buf[NSAMP];   // 16-bit ADC results

void adc_dma_init(void) {
    RCC->AHB1ENR |= RCC_AHB1ENR_DMA2EN;   // clock the DMA2 controller

    DMA2_Stream0->CR &= ~DMA_SxCR_EN;     // disable before configuring
    while (DMA2_Stream0->CR & DMA_SxCR_EN) { }  // wait until actually off

    DMA2_Stream0->PAR  = (uint32_t)&ADC1->DR;     // source: ADC data reg (fixed)
    DMA2_Stream0->M0AR = (uint32_t)adc_buf;       // dest: RAM buffer base
    DMA2_Stream0->NDTR = NSAMP;                   // ITEMS (not bytes!)

    DMA2_Stream0->CR =
          (0u << DMA_SxCR_CHSEL_Pos)   // Channel 0 selects ADC1 request
        | (1u << DMA_SxCR_PSIZE_Pos)   // peripheral size = half-word (16-bit)
        | (1u << DMA_SxCR_MSIZE_Pos)   // memory size     = half-word
        | DMA_SxCR_MINC               // increment memory address
        | DMA_SxCR_CIRC               // CIRCULAR: auto-reload forever
        | DMA_SxCR_HTIE               // half-transfer interrupt enable
        | DMA_SxCR_TCIE;              // transfer-complete interrupt enable
        // DIR = 00 (peripheral-to-memory), PINC = 0 (peripheral fixed)

    NVIC_EnableIRQ(DMA2_Stream0_IRQn);
    DMA2_Stream0->CR |= DMA_SxCR_EN;   // arm the engine

    ADC1->CR2 |= ADC_CR2_DMA | ADC_CR2_DDS;  // ADC issues DMA requests, keep requesting
    ADC1->CR2 |= ADC_CR2_CONT | ADC_CR2_ADON;  // continuous + ON
    ADC1->CR2 |= ADC_CR2_SWSTART;              // kick off conversions
}
💡 What just happened

After adc_dma_init() returns, the ADC converts on its own, raises a DMA request after each result, and the engine drops the 16-bit value into the next slot of adc_buf[] — wrapping forever because CIRC is set. The CPU's only remaining job is the IRQ handler below, called twice per buffer. Note DDS (DMA Disable Selection): it keeps the ADC issuing requests after the last transfer so circular mode never stalls.

dma2_stream0_irq.c C / STM32F4
void DMA2_Stream0_IRQHandler(void) {
    if (DMA2->LISR & DMA_LISR_HTIF0) {        // half-transfer: first half ready
        DMA2->LIFCR = DMA_LIFCR_CHTIF0;       // clear the flag (write-1-to-clear)
        process_block(&adc_buf[0], NSAMP / 2);   // safe: engine fills 2nd half
    }
    if (DMA2->LISR & DMA_LISR_TCIF0) {        // transfer-complete: second half ready
        DMA2->LIFCR = DMA_LIFCR_CTCIF0;       // clear the flag
        process_block(&adc_buf[NSAMP / 2], NSAMP / 2);  // safe: engine refills 1st
    }
}
⚠️ Pitfall — forgetting to clear flags

DMA status flags are write-1-to-clear via the dedicated LIFCR/HIFCR registers — you cannot clear them by writing the status register. Forget the clear and your handler re-fires instantly, livelocking the CPU. Also note F4 spreads 8 streams across two pairs: streams 0–3 use the Low registers (LISR/LIFCR), streams 4–7 the High ones (HISR/HIFCR).

🏭 Real case — UART log streaming, WS2812 & mem-to-mem

UART TX (memory→peripheral): point DMA at your log string, set DIR = memory-to-peripheral, MINC = 1, PINC = 0, NDTR = length, enable USART->CR3.DMAT. The whole buffer streams out at line rate while main() moves on — no blocking printf.
WS2812 LED strip (memory→peripheral via SPI/timer): these "NeoPixel" strips need a precise ~800 kHz one-wire waveform. Encode each color bit as an SPI byte (or a timer-PWM duty), then DMA the whole encoded frame to SPI->DR. DMA's exact timing is what makes the strip glitch-free — bit-banging from the CPU jitters and the strip flickers.
Memory-to-memory: set DIR = mem-to-mem (DMA2 only on F4), both PINC and MINC = 1, and it becomes a hardware memcpy that runs while the CPU does other work. Note: circular and direct mode are not available for mem-to-mem.

Q1 Conceptual check
You are streaming an ADC continuously into a single circular buffer and processing the data on the CPU. What core problem do the half-transfer and transfer-complete interrupts (double buffering) actually solve?
  • They make the DMA engine run faster by skipping the bus arbiter
  • They let you process one half of the buffer while the engine safely fills the other, avoiding a read/write race on the same data
  • They convert peripheral byte data into 32-bit memory words
  • They stop the ADC so the CPU can catch up between blocks
Double buffering is about separating producer and consumer in time. The HT interrupt says "first half is full, second is filling — process the first now"; the TC interrupt says the reverse. You always touch the half the engine is not writing, so there is never a race and the stream stays gapless. Speed and width-conversion are unrelated jobs (the FIFO/bursts handle those).
Q2 Compute it
You capture 48 kHz stereo (2 channels) audio at 16 bits (2 bytes) per sample. What raw byte rate, in bytes per second, must your DMA stream sustain? (Use $R = f_s \cdot N_{ch} \cdot W$.)

Q3 Coding challenge — circular ADC with half/full callbacks (HAL)
Using the STM32 HAL, configure ADC1 to sample continuously into a circular DMA buffer of 256 half-words, and implement the half-complete and full-complete callbacks so each processes the correct half. Assume hadc1 and its linked DMA handle are already initialized by CubeMX with circular mode. Think about which half each callback owns.
Hint
Start the stream once with HAL_ADC_Start_DMA(&hadc1, (uint32_t*)buf, 256) — the count is in items. Then override the two weak callbacks: HAL_ADC_ConvHalfCpltCallback fires at the midpoint (first 128 samples ready) and HAL_ADC_ConvCpltCallback fires at wrap (last 128 ready). Each should process the half the engine just finished — and, crucially, return fast enough to beat the fill time.
Show one solution
adc_circular_hal.c C / STM32 HAL
#define NSAMP 256
static volatile uint16_t buf[NSAMP];   // circular DMA target

void app_start(void) {
    // Start ADC -> DMA in circular mode. Count is in ITEMS, not bytes.
    // CubeMX set the DMA handle to Circular + HT/TC interrupts enabled.
    HAL_ADC_Start_DMA(&hadc1, (uint32_t *)buf, NSAMP);
}

// Fires when the engine crosses the midpoint: buf[0 .. 127] is full,
// and the engine is now writing buf[128 .. 255]. Process the FIRST half.
void HAL_ADC_ConvHalfCpltCallback(ADC_HandleTypeDef *hadc) {
    if (hadc->Instance == ADC1)
        process_block((uint16_t *)&buf[0], NSAMP / 2);
}

// Fires at wrap: buf[128 .. 255] is full, and the engine has wrapped
// back to refill buf[0 .. 127]. Process the SECOND half.
void HAL_ADC_ConvCpltCallback(ADC_HandleTypeDef *hadc) {
    if (hadc->Instance == ADC1)
        process_block((uint16_t *)&buf[NSAMP / 2], NSAMP / 2);
}

// Your DSP / copy-out. MUST finish before the engine fills the other half.
static void process_block(uint16_t *p, uint32_t n) {
    for (uint32_t i = 0; i < n; i++) {
        // e.g. accumulate, filter, or copy out to a transmit buffer
        (void)p[i];
    }
}

The lesson is the ownership split: half-callback owns indices [0, N/2), full-callback owns [N/2, N), and the engine is always writing the other half — so neither callback ever races the DMA. Keep process_block well inside the per-half fill budget (§6) or you overrun. For absolute safety on M7/cache parts, also invalidate the D-cache for the half you read.
