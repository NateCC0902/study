Timer Mastery: PWM, Input Capture & Encoders

The STM32 timer is the crown-jewel peripheral. Underneath it is just a counter — but wrap that counter in a prescaler, an auto-reload, and a handful of compare channels and you can generate motor drive, dim LEDs without burning CPU, measure a pulse to the nanosecond, and decode a quadrature encoder in hardware. We build all of it from one register: CNT.

1A timer is just a counter that counts the clock

① Intuition build the picture first

Forget "timer" for a moment. A general-purpose timer like TIM2 or TIM3 on an STM32 is a hardware counter — a register called CNT that ticks up by one every clock edge it's fed. That's the whole idea. Everything else (PWM, period measurement, encoder decoding) is built by deciding how fast it ticks and what happens at certain counts.

💡 The one-sentence model

A timer is a counter (CNT) fed by a divided clock, that counts up to a programmable ceiling (ARR), wraps back to 0, and can fire actions when it passes programmable comparison points (CCRx).

Three knobs shape its behaviour, and you'll touch them in every single timer you ever configure:

  • PSC — the prescaler. It divides the incoming timer clock so CNT ticks more slowly. Divide-by is $(\text{PSC}+1)$.
  • ARR — the auto-reload register. The counter counts 0, 1, 2, … ARR then resets to 0 and raises an update event. So the period is $(\text{ARR}+1)$ ticks.
  • CCRx — the capture/compare register for channel x. In output mode it's the threshold that sets PWM duty; in input mode it latches the count when an edge arrives.
🔑 Why "+1" everywhere

The counter spends one tick on each value from 0 through ARR inclusive. That's $\text{ARR}+1$ distinct values — hence period $=\text{ARR}+1$, not ARR. The same off-by-one logic makes the prescaler divide by $\text{PSC}+1$. Hardware counts from zero; humans forget the zero. Burn this in.

2The counting math: PSC, ARR & the update frequency

② Math derive it

The timer's input clock $f_{\text{tim}}$ is divided by the prescaler to give the rate at which CNT increments:

$$ f_{\text{CNT}} = \frac{f_{\text{tim}}}{\text{PSC}+1} $$

The counter then takes $\text{ARR}+1$ of those ticks to overflow once. Each overflow is the update event — the heartbeat of the timer. Its frequency is the input clock divided by both factors:

$$ f_{\text{update}} = \frac{f_{\text{tim}}}{(\text{PSC}+1)(\text{ARR}+1)} $$

This single equation is the foundation of the entire chapter. PWM frequency, sample-rate interrupts, software clocks — they are all just $f_{\text{update}}$ with different ARR/CCR meanings layered on top.

⚠️ Pitfall — the APB timer-clock ×2 rule

$f_{\text{tim}}$ is not always the APB bus clock you set in the clock tree. On STM32F4/F7/H7, when an APB prescaler is greater than 1, the timer clock on that bus is doubled. On a 168 MHz F407 with APB1 at 42 MHz (÷4), the APB1 timer clock is $2\times42 = 84\,\text{MHz}$ — exactly what feeds TIM2–7. We met this rule in Chapter 2; here it decides every number you compute. Always read $f_{\text{tim}}$ off the CubeMX clock-tree's "to TIMx" node, not the bus label.

② Math solving for a target

In practice you work the equation backwards: you know $f_{\text{tim}}$ and a target $f_{\text{update}}$, and you must choose PSC and ARR. Rearranging:

$$ (\text{PSC}+1)(\text{ARR}+1) = \frac{f_{\text{tim}}}{f_{\text{update}}} $$

Worked example. A 84 MHz timer; we want a 1 kHz update (a clean 1 ms tick). The product we need is

$$ (\text{PSC}+1)(\text{ARR}+1) = \frac{84\times10^6}{1\times10^3} = 84\,000 $$

Now factor 84 000 into two integers, each $\le 65536$ (these are 16-bit registers). A tidy split is $84 \times 1000$, giving PSC = 83 and ARR = 999. Check it: $84 \times 1000 = 84\,000$, and $84\times10^6 / 84\,000 = 1000\,\text{Hz}$. Done.

🛠️ Choosing the split wisely

Many (PSC, ARR) pairs hit the same $f_{\text{update}}$, but they're not equal. For PWM you want ARR large (PSC small or zero) so duty resolution is fine — duty steps are $1/(\text{ARR}+1)$. For a slow software clock you want PSC large so a 16-bit ARR can still reach a long period. Pick the factor that puts resolution where you need it.

3Inside the block: how CNT, ARR & CCR interact

① Intuition the datapath

Picture the counter as a rising sawtooth: CNT ramps from 0 to ARR, snaps back to 0, repeats. ARR is the ceiling that sets the period. Each compare register CCRx is a horizontal line through that sawtooth — the moment CNT crosses it, the channel's output flips. That crossing is what turns a counter into a PWM generator.

The bus clock is divided by the prescaler to feed CNT. CNT ramps to ARR and wraps (the update event). Along the way a comparator watches CNT against each CCRx; the crossing drives the output pin. ARR sets the period; CCR sets the duty. Two registers, the whole story.

4PWM: turning the counter into an analog-ish output

① Intuition average, not toggle

In PWM mode 1 the channel output is high while $\text{CNT} < \text{CCRx}$ and low afterward, until the counter wraps. So in every period the pin is high for CCRx ticks out of ARR+1 total. Drive a motor or an LED with this and the load responds to the average voltage — a fast square wave at 25% duty looks like 25% of the supply.

② Math duty & frequency

Two formulas govern PWM, and they fall straight out of §2. The duty cycle is the high-time fraction of the period:

$$ D = \frac{\text{CCR}}{\text{ARR}+1} $$

and the PWM frequency is just the update frequency (one PWM cycle per counter wrap):

$$ f_{\text{PWM}} = \frac{f_{\text{tim}}}{(\text{PSC}+1)(\text{ARR}+1)} $$

Worked example — a 20 kHz motor PWM. 20 kHz is the sweet spot for motor drive: above audible hiss, below switching-loss pain. With $f_{\text{tim}} = 84\,\text{MHz}$ and PSC = 0 (no prescale, for maximum duty resolution):

$$ \text{ARR}+1 = \frac{f_{\text{tim}}}{f_{\text{PWM}}} = \frac{84\times10^6}{20\times10^3} = 4200 \;\Rightarrow\; \text{ARR}=4199 $$

That gives 4200 duty steps — about 0.024% resolution, plenty. For 25% duty:

$$ \text{CCR} = D\,(\text{ARR}+1) = 0.25 \times 4200 = 1050 $$

So PSC=0, ARR=4199, CCR=1050 gives a 20 kHz, 25%-duty wave. Want 50%? Write CCR=2100. The beauty: once the timer is configured, changing brightness or motor speed is a single register write with zero CPU overhead afterwards.

③ Code read every line

Here's the STM32 HAL version — TIM3 channel 1, 20 kHz, starting at 25% duty. Assume the clock tree gave TIM3 an 84 MHz timer clock.

pwm.c C / STM32 HAL
TIM_HandleTypeDef htim3;

void PWM_Init(void)
{
  __HAL_RCC_TIM3_CLK_ENABLE();                 // power the timer

  htim3.Instance           = TIM3;
  htim3.Init.Prescaler     = 0;                // PSC = 0  -> divide by 1
  htim3.Init.CounterMode   = TIM_COUNTERMODE_UP;
  htim3.Init.Period        = 4199;             // ARR = 4199 -> 84MHz/4200 = 20 kHz
  htim3.Init.ClockDivision = TIM_CLOCKDIVISION_DIV1;
  HAL_TIM_PWM_Init(&htim3);

  TIM_OC_InitTypeDef sConfig = {0};
  sConfig.OCMode     = TIM_OCMODE_PWM1;        // high while CNT < CCR
  sConfig.Pulse      = 1050;                   // CCR = 1050 -> 25% duty
  sConfig.OCPolarity = TIM_OCPOLARITY_HIGH;
  HAL_TIM_PWM_ConfigChannel(&htim3, &sConfig, TIM_CHANNEL_1);

  HAL_TIM_PWM_Start(&htim3, TIM_CHANNEL_1);    // start driving the pin
}

// Later, change duty on the fly with ZERO recompute:
void PWM_SetDutyPercent(uint8_t pct)           // 0..100
{
  uint32_t arr = __HAL_TIM_GET_AUTORELOAD(&htim3);   // 4199
  uint32_t ccr = ((arr + 1U) * pct) / 100U;          // CCR = D*(ARR+1)
  __HAL_TIM_SET_COMPARE(&htim3, TIM_CHANNEL_1, ccr);
}
💡 What each field maps to

Init.Prescaler is PSC, Init.Period is ARR, and sConfig.Pulse is CCR. The HAL names hide the registers, but the math is identical. __HAL_TIM_SET_COMPARE() is just a write to TIM3->CCR1 — instantaneous, glitch-free, and updated cleanly at the next reload because the registers are preloaded/buffered.

If you prefer the metal, the same configuration in bare registers makes the mapping unmistakable:

pwm_regs.c C / STM32 (registers)
RCC->APB1ENR  |= RCC_APB1ENR_TIM3EN;          // clock the timer
TIM3->PSC      = 0;                            // prescaler  = PSC

TIM3->CCR1     = 1050;                         // compare    = CCR (25% duty)
TIM3->CCMR1   |= (6 << TIM_CCMR1_OC1M_Pos);    // OC1M = 110 -> PWM mode 1

TIM3->EGR     |= TIM_EGR_UG;                   // force an update to load shadows
TIM3->CR1     |= TIM_CR1_CEN;                  // CEN = 1 -> counter runs
🏭 Real case — servo, motor & LED on one peripheral

The exact same timer, retuned, drives three classic products. A hobby servo wants a 50 Hz frame with a 1–2 ms high pulse: set PSC=83, ARR=19999 (1 MHz tick → 20 ms period), then CCR=1500 for centre. A brushed-motor H-bridge wants 20 kHz (the example above) so it never whines. LED dimming wants ≥1 kHz (flicker-free to the eye) with a large ARR for smooth fades. One IP block, three industries — that's why mastering it pays off.

5Input capture: measuring the outside world

① Intuition latch the count on an edge

PWM is the timer talking. Input capture is the timer listening. You route an external signal to a channel pin and tell the hardware: "on each rising edge, snapshot the current CNT into CCRx." The CPU isn't involved in the timing — the silicon latches the count the instant the edge arrives, so you measure period, pulse width, or frequency with near-zero jitter.

② Math ticks to seconds to Hz

Subtract two consecutive captures to get the period in timer ticks. Each tick lasts $1/f_{\text{CNT}}$ seconds, so the measured frequency of the input signal is

$$ f_{\text{signal}} = \frac{f_{\text{CNT}}}{\Delta\text{CNT}} = \frac{f_{\text{tim}}/(\text{PSC}+1)}{\text{CCR}_n - \text{CCR}_{n-1}} $$

Mind the wrap: if $\text{CCR}_n < \text{CCR}_{n-1}$ the counter overflowed between edges, so add $\text{ARR}+1$ before subtracting. Worked example: with a 1 MHz tick (1 µs each) and a captured delta of 2000 ticks, the input is $f = 10^6 / 2000 = 500\,\text{Hz}$ — a 2 ms period, read straight off the hardware.

③ Code capture in the ISR
capture.c C / STM32 HAL
// TIM2 CH1 in input-capture, rising edge. Tick = 1 us (PSC = 83 on 84 MHz).
volatile uint32_t g_period_ticks = 0;
static   uint32_t last = 0;

void HAL_TIM_IC_CaptureCallback(TIM_HandleTypeDef *htim)
{
  if (htim->Channel == HAL_TIM_ACTIVE_CHANNEL_1) {
    uint32_t now = HAL_TIM_ReadCapturedValue(htim, TIM_CHANNEL_1);
    uint32_t arr = __HAL_TIM_GET_AUTORELOAD(htim);
    // wrap-safe delta: add one full period if the counter rolled over
    g_period_ticks = (now >= last) ? (now - last)
                                   : (now + (arr + 1U) - last);
    last = now;
  }
}

// f_signal = 1e6 / g_period_ticks  (Hz), since one tick is 1 us here.
⚠️ Pitfall — missed overflows on slow signals

The wrap-fix above only handles a single overflow between edges. If the input is slower than one full counter period, the counter wraps more than once and a naive delta is wrong. For low frequencies either enlarge the prescaler (slower tick, longer reach) or count update-event interrupts and fold them in. Always know your measurable range: $f_{\min} = f_{\text{CNT}}/(\text{ARR}+1)$.

6Encoder mode: free quadrature decoding in hardware

① Intuition the counter becomes a position

A quadrature encoder outputs two square waves, A and B, 90° out of phase. The phase order tells direction: A-leads-B is forward, B-leads-A is reverse. In encoder mode the timer wires both channels into the counter so that CNT increments on forward edges and decrements on reverse edges — automatically. After configuration, CNT simply is the wheel position, updated in hardware with no interrupts and no missed counts even at high RPM.

② Math counts, RPM & the ×4

In the common ×4 mode the timer counts every edge of both channels, so one encoder with N lines per revolution yields $4N$ counts per revolution. Read the count delta over a known interval $\Delta t$ to get speed:

$$ \text{RPM} = \frac{\Delta\text{CNT}}{4N} \cdot \frac{60}{\Delta t} $$

Worked example: a 500-line encoder (so 2000 counts/rev in ×4) advances 200 counts in a 10 ms window. Then $\text{RPM} = \frac{200}{2000}\cdot\frac{60}{0.01} = 0.1 \cdot 6000 = 600\,\text{RPM}$. The counter did all the edge-tracking; your loop just reads CNT and subtracts.

③ Code read position with one register
encoder.c C / STM32 HAL
TIM_HandleTypeDef htim4;

void Encoder_Init(void)                         // TIM4 CH1=A, CH2=B
{
  TIM_Encoder_InitTypeDef enc = {0};
  htim4.Instance        = TIM4;
  htim4.Init.Prescaler  = 0;                    // count every edge, no divide
  htim4.Init.Period     = 0xFFFF;               // 16-bit wrap (free-running)
  htim4.Init.CounterMode = TIM_COUNTERMODE_UP;
  enc.EncoderMode = TIM_ENCODERMODE_TI12;       // x4: count edges of A and B
  enc.IC1Polarity = TIM_ICPOLARITY_RISING;
  enc.IC2Polarity = TIM_ICPOLARITY_RISING;
  HAL_TIM_Encoder_Init(&htim4, &enc);
  HAL_TIM_Encoder_Start(&htim4, TIM_CHANNEL_ALL);
}

// In your control loop, position is literally the counter:
int16_t Encoder_Read(void)
{
  return (int16_t)__HAL_TIM_GET_COUNTER(&htim4); // signed -> handles reverse
}
🏭 Real case — closed-loop motor control

This is the backbone of every servo drive, robot joint, and CNC axis. One timer in PWM mode commands the motor; a second timer in encoder mode reads its position — both in hardware, both interrupt-free. A periodic control loop (often a third timer firing an ISR at, say, 1 kHz) reads the encoder delta, computes a PID correction, and writes a new CCR. Three timers, a complete motion-control system, and the CPU is mostly idle.

Q1 Conceptual check
A timer in PWM mode 1 has ARR = 999 and CCR = 250. What duty cycle does the output have, and which register would you change to make it 75%?
  • 2.5% duty; change ARR to 750
  • 25% duty; write CCR = 750
  • 40% duty; write CCR = 400
  • 25% duty; change the prescaler PSC
Duty is $D = \text{CCR}/(\text{ARR}+1) = 250/1000 = 25\%$. To reach 75% you change the duty, not the period, so write $\text{CCR} = 0.75 \times 1000 = 750$. Changing ARR or PSC would alter the frequency, not the duty.
Q2 Compute it
A timer is clocked at 16 MHz with PSC = 15 and ARR = 999. What is the update frequency $f_{\text{update}}$ in Hz? (Recall $f_{\text{update}} = \dfrac{f_{\text{tim}}}{(\text{PSC}+1)(\text{ARR}+1)}$.)
Hz
$(\text{PSC}+1)(\text{ARR}+1) = 16 \times 1000 = 16\,000$, so $f = \dfrac{16\times10^6}{16\,000} = 1000\,\text{Hz}$ — a clean 1 ms tick. This is the classic "give me a 1 kHz control-loop interrupt on a 16 MHz timer" recipe.
Q3 Coding challenge — drive a hobby servo
Configure TIM3 CH1 on an 84 MHz timer to produce a standard 50 Hz servo frame (20 ms period) with a 1.5 ms centre pulse. Choose PSC/ARR for a convenient 1 µs tick, then find the CCR for centre. Write the HAL init.
Hint
Aim for a 1 µs tick: $f_{\text{CNT}} = 84\,\text{MHz}/(\text{PSC}+1) = 1\,\text{MHz} \Rightarrow \text{PSC} = 83$. Then a 20 ms period is 20 000 ticks, so $\text{ARR} = 19999$. A 1.5 ms pulse is 1500 ticks, so $\text{CCR} = 1500$. (1 ms → 1000, 2 ms → 2000: the full servo range.)
Show one solution
servo.c C / STM32 HAL
// 84 MHz timer -> PSC=83 gives a 1 us tick (1 MHz).
// 20 ms frame -> ARR=19999. 1.5 ms centre -> CCR=1500.
TIM_HandleTypeDef htim3;

void Servo_Init(void)
{
  __HAL_RCC_TIM3_CLK_ENABLE();

  htim3.Instance         = TIM3;
  htim3.Init.Prescaler   = 83;                 // 84MHz/84 = 1 MHz  (1 us/tick)
  htim3.Init.CounterMode = TIM_COUNTERMODE_UP;
  htim3.Init.Period      = 19999;              // 20000 ticks = 20 ms = 50 Hz
  HAL_TIM_PWM_Init(&htim3);

  TIM_OC_InitTypeDef oc = {0};
  oc.OCMode     = TIM_OCMODE_PWM1;
  oc.Pulse      = 1500;                         // 1.5 ms -> centre position
  oc.OCPolarity = TIM_OCPOLARITY_HIGH;
  HAL_TIM_PWM_ConfigChannel(&htim3, &oc, TIM_CHANNEL_1);
  HAL_TIM_PWM_Start(&htim3, TIM_CHANNEL_1);
}

// Aim the servo: angle 0..180 deg -> pulse 1000..2000 us.
void Servo_SetAngle(uint8_t deg)               // 0..180
{
  uint32_t us = 1000U + ((uint32_t)deg * 1000U) / 180U;  // 1000..2000
  __HAL_TIM_SET_COMPARE(&htim3, TIM_CHANNEL_1, us);       // CCR in 1-us ticks
}

The 1 µs tick is the trick: it makes CCR read in microseconds directly, so a 1500 µs pulse is literally CCR=1500. Choosing your prescaler to give a human-friendly tick is a hallmark of clean timer code.

🎓 Chapter recap

A timer is a counter: CNT ticks at $f_{\text{tim}}/(\text{PSC}+1)$, wraps every $\text{ARR}+1$ ticks, and the update frequency is $f_{\text{tim}}/((\text{PSC}+1)(\text{ARR}+1))$ — remember the APB ×2 timer-clock rule. In PWM, ARR sets the frequency and CCR sets duty ($D=\text{CCR}/(\text{ARR}+1)$); 84 MHz → 20 kHz is ARR=4199, and 25% duty is CCR=1050. Input capture latches CNT on an edge to measure period and frequency wrap-safely. Encoder mode turns CNT into a hardware position counter for closed-loop motor control. One peripheral, three superpowers — and the CPU barely lifts a finger. Next we'll feed these timers into the ADC and DMA for sampling that runs entirely in hardware.
