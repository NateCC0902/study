The Clock Tree: HSI/HSE/PLL, RCC & SysTick

This is the single most common thing beginners get wrong on STM32. The chip boots slow, every peripheral is born asleep, and nothing you write works until you understand where the megahertz come from. We'll build the clock tree from one oscillator up — then derive 168 MHz with real numbers and a real bug story.

⏱ ~45 min🧠 Clock tree⚡ RCC registers🔧 PLL math

1Why a chip needs a clock at all

① Intuition build the picture first

An MCU is a giant grid of synchronous logic. Almost every flip-flop inside it updates on the edge of a clock — a square wave that beats the rhythm for the whole chip. No clock, no instructions executed, no UART bytes shifted, no timer counting. The clock is the heartbeat, and on STM32 there isn't just one: there's a whole tree of derived clocks, each feeding a different region of silicon.

💡 The one-sentence model

On STM32, every peripheral runs on a clock that is derived, step by step, from a single chosen source — and that peripheral does nothing, reads as zero, and consumes no power until you explicitly switch its clock on.

Here's the part that ambushes everyone coming from Arduino: an Arduino Uno wakes up at its full 16 MHz and every pin is ready. An STM32 does not. After reset an STM32F4 runs at a leisurely 16 MHz off its internal oscillator, the PLL is off, the GPIO ports are unclocked, and if you poke a peripheral register it stares back at you with zeros. The chip hands you a slow, mostly-dark machine and trusts you to bring it up. That bring-up is what this chapter is about.

🔑 Two jobs you must do at boot

(1) Choose & raise the system clock — pick a source, run it through the PLL, switch the CPU onto it. (2) Enable each peripheral's clock via the RCC before you touch it. Skip job 2 and the peripheral is electrically present but functionally dead.

2The four oscillators: HSI, HSE, LSI, LSE

① Intuition where the raw beats come from

Before anything is multiplied or divided, the megahertz have to be generated. STM32 gives you four raw clock sources, split along two axes: internal vs. external (precision/cost) and high-speed vs. low-speed (run the CPU vs. keep time while asleep).

SourceMeansTypical freqAccuracyUsed for
HSIHigh-Speed Internal16 MHz (F4); 8 MHz (F1)±1–2% (RC, drifts with temp)Boot default, quick start, fallback
HSEHigh-Speed External4–26 MHz crystal (often 8)±20–50 ppm (crystal)PLL input when timing must be exact
LSILow-Speed Internal~32 kHz RCCoarse (±5–10%)Independent watchdog (IWDG), low-power RTC
LSELow-Speed External32.768 kHz crystal±20 ppmAccurate RTC / calendar time-of-day
🔑 Internal = convenient, External = precise

HSI is an on-chip RC oscillator — free, instant, zero external parts, but it's an analog R·C network so its frequency wanders with temperature and voltage. HSE is a quartz crystal you solder on; quartz is mechanically resonant and astonishingly stable. The number 32.768 kHz for LSE isn't arbitrary — it's $2^{15}$, so a 15-bit counter overflows exactly once per second. Perfect for a clock that ticks seconds.

④ Real case why USB forces a crystal
🏭 Real case — USB will not tolerate HSI

Full-speed USB requires its 48 MHz clock to be within about ±0.25%. HSI's ±1–2% RC drift blows straight through that budget, so the host drops the device or it enumerates intermittently. This is why nearly every STM32 board with a USB port has an 8 MHz (or 25 MHz) crystal sitting next to the chip: you need HSE to clock USB reliably. The RC oscillator is fine for blinking an LED; it is not fine for a protocol with a tight frequency tolerance.

3The PLL: turning 8 MHz into 168 MHz

① Intuition a frequency multiplier

Your crystal runs at 8 MHz. Your CPU wants to run at 168 MHz. The bridge between them is the PLL (Phase-Locked Loop): a feedback circuit containing a voltage-controlled oscillator (VCO) that the chip tunes until its divided-down output exactly matches the reference. The net effect is a clean, programmable frequency multiplier — feed in 8 MHz, get out almost any frequency you like.

② Math derive the F4 formula

On the STM32F4 the PLL has three integer knobs in series: a pre-divider $M$, a multiplier $N$, and a post-divider $P$ for the system clock. The signal flows in that order:

$$ f_{\text{VCO,in}} = \frac{f_{\text{in}}}{M}, \qquad f_{\text{VCO}} = f_{\text{VCO,in}} \cdot N, \qquad f_{\text{SYSCLK}} = \frac{f_{\text{VCO}}}{P} $$

Stage one, $M$, drops the input into the VCO's required input window of 1–2 MHz (the datasheet recommends landing on exactly 2 MHz, but 1 MHz is legal and used by ST's own examples). Stage two, $N$, multiplies up to the VCO output, which must stay inside 100–432 MHz. Stage three, $P \in \{2,4,6,8\}$, divides back down to the 168 MHz the core can sustain. Let's plug in the canonical F4 values $f_{\text{in}}=8\,\text{MHz (HSE)},\,M=8,\,N=336,\,P=2$:

$$ f_{\text{VCO,in}} = \frac{8\,\text{MHz}}{8} = 1\,\text{MHz}, \quad f_{\text{VCO}} = 1\,\text{MHz}\times 336 = 336\,\text{MHz}, \quad f_{\text{SYSCLK}} = \frac{336\,\text{MHz}}{2} = 168\,\text{MHz} $$

Both rails check out: VCO input 1 MHz is in [1, 2] and VCO output 336 MHz is in [100, 432]. There's a fourth knob, $Q$, that taps the same 336 MHz VCO to make the USB / SDIO / RNG clock. We need 48 MHz, so:

$$ f_{\text{USB}} = \frac{f_{\text{VCO}}}{Q} = \frac{336\,\text{MHz}}{7} = 48\,\text{MHz} \;\checkmark $$

That single shared VCO is exactly why ST chose $N=336$ rather than, say, 168 with $P=1$: 336 is divisible by 7, so the same loop yields both a clean 168 MHz core and a spec-perfect 48 MHz USB. The clock tree is a constraint-satisfaction puzzle, and these are the magic numbers that solve it.

💡 The F1 version is simpler — one multiplier

An older STM32F1 (Blue Pill, "Bluepill" = STM32F103) has no $M/N/P$ split. It feeds $\text{HSE}=8\,\text{MHz}$ straight into a single multiplier $\times 9$: $\;f_{\text{SYSCLK}} = 8\,\text{MHz}\times 9 = 72\,\text{MHz}$, the F1's maximum. Same idea (multiply the crystal up), fewer dials.

4Bus prescalers: HCLK, PCLK1, PCLK2 & the timer ×2 rule

① Intuition not everything can run at 168 MHz

The core can take 168 MHz, but the peripheral buses cannot all keep up — and there's no point burning power clocking a slow UART at full tilt. So after SYSCLK, the clock fans out through a cascade of prescalers (programmable divide-by-power-of-two) that produce the bus clocks:

ClockBusSourceF4 maxFeeds
HCLKAHBSYSCLK ÷ AHB_PSC168 MHzCore, DMA, memory, GPIO
PCLK1APB1 (low-speed)HCLK ÷ APB1_PSC42 MHzUSART2/3, I²C, SPI2/3, TIM2–7
PCLK2APB2 (high-speed)HCLK ÷ APB2_PSC84 MHzUSART1/6, SPI1, ADC, TIM1/8/9–11

For the 168 MHz config you set AHB ÷1 (HCLK = 168), APB1 ÷4 (PCLK1 = 42), APB2 ÷2 (PCLK2 = 84) — the maximum legal value on each bus.

② Math the rule that trips everyone

Now the subtle one. Timers hang off APB1/APB2, but ST's silicon plays a trick: if an APB prescaler is greater than 1, the timer clock on that bus is doubled. Formally, the clock feeding a timer's counter is

$$ f_{\text{TIMx}} = \begin{cases} f_{\text{PCLK}} & \text{if } \text{APB\_PSC} = 1 \\[4pt] 2 \cdot f_{\text{PCLK}} & \text{if } \text{APB\_PSC} > 1 \end{cases} $$

So with APB1 ÷4: $f_{\text{PCLK1}} = 42\,\text{MHz}$, but a TIM2–7 counter actually clocks at $2\times 42 = 84\,\text{MHz}$. With APB2 ÷2: PCLK2 = 84 MHz, but TIM1/8 see $2\times 84 = 168\,\text{MHz}$. Forget this factor of two and every PWM frequency and every delay_us you compute from $f_{\text{PCLK}}$ comes out exactly half-right — the classic "my servo runs at double speed" bug. We'll lean on this formula hard in the timer chapter.

5Flash wait states: why fast cores must wait

① Intuition the memory can't keep up

There's a physical catch to running at 168 MHz. The CPU fetches every instruction from on-chip Flash, and Flash is slow — its sense amplifiers need on the order of 25–30 ns to return a word. At 168 MHz a clock cycle is only $1/168\,\text{MHz} \approx 5.95\,\text{ns}$, far shorter than Flash can answer. If the core demanded data every cycle, it would read garbage.

The fix is wait states: the Flash controller stalls the CPU for a few cycles per fetch so the memory has time to respond. More speed ⇒ shorter cycles ⇒ more wait states. On the F4 (at 3.3 V) the threshold is one wait state per ~30 MHz band, and 168 MHz lands at 5 wait states (FLASH_ACR_LATENCY_5WS).

⚠️ Pitfall — order of operations bricks the boot

You must raise the Flash latency before you switch SYSCLK up to 168 MHz. Switch the clock first while latency is still 0, and the very next instruction fetch reads corrupt Flash — the core executes nonsense and hangs (or HardFaults) before your main() ever runs. Set latency high, then speed up. When you slow back down, drop latency after.

🛠️ The ART accelerator hides the cost

5 wait states sounds brutal — a fetch every 6th cycle? In practice ST's ART Accelerator (a prefetch + instruction cache in front of Flash) keeps the pipeline fed, so linear code still runs at near zero-wait-state throughput. The wait states are real; the cache just usually pays them off your behalf. Enable FLASH_ACR_PRFTEN | ICEN | DCEN and forget about it.

6RCC: the bug everyone hits once

① Intuition peripherals are gated off by default

Here is the rite-of-passage STM32 bug. You write perfect GPIO code to blink an LED on port D. You flash it. Nothing. No errors, no warnings, the debugger shows your code running — and the pin does nothing. Reads of GPIOD->MODER come back as zero no matter what you write.

The reason: to save power, the RCC (Reset and Clock Control) unit keeps every peripheral's clock gated off at reset. An unclocked peripheral is frozen — its registers won't even latch the values you write. Before you configure any peripheral, you must set its enable bit in the matching RCC register.

🔑 The rule, burned in

The enable register is named after the bus the peripheral lives on: RCC->AHB1ENR for GPIO & DMA, RCC->APB1ENR for TIM2–7 / USART2 / I²C, RCC->APB2ENR for USART1 / SPI1 / ADC. Set the bit, then touch the peripheral. Never the other way round.

③ Code read every line
gpio_clock.c C / STM32 (CMSIS)
#include "stm32f4xx.h"   // CMSIS device header: defines RCC, GPIOD, bit masks

void led_init(void) {
    // STEP 1 — turn ON the clock to GPIO port D. WITHOUT this line,
    // every write below is silently dropped and MODER reads back 0.
    RCC->AHB1ENR |= RCC_AHB1ENR_GPIODEN;   // bit 3 of AHB1ENR

    // A read-back here forces the enable to take effect before we
    // touch the peripheral (guards against a 1-cycle clock-domain gap).
    (void)RCC->AHB1ENR;

    // STEP 2 — now the peripheral is alive, configure PD12 as output.
    // MODER is 2 bits/pin: 01 = general-purpose output. Pin 12 → bits 25:24.
    GPIOD->MODER &= ~(0x3u << (12 * 2));    // clear the 2-bit field first
    GPIOD->MODER |=  (0x1u << (12 * 2));    // 01 = output mode

    // STEP 3 — drive it. BSRR is the atomic set/reset register:
    // writing bit 12 sets PD12 high with no read-modify-write race.
    GPIOD->BSRR = (1u << 12);               // PD12 = HIGH → LED on
}
⚠️ Pitfall — the symptom is total silence

Forgetting the RCC->AHB1ENR line produces no error of any kind. The compiler is happy, the linker is happy, the debugger single-steps cleanly through your GPIO writes — and the peripheral registers simply refuse to change. If a peripheral register reads back exactly what it had at reset no matter what you store to it, your first suspect is always a missing RCC clock-enable bit.

7SysTick: a 1 ms heartbeat from SYSCLK

① Intuition the timer that's always there

Every Cortex-M core ships with a tiny 24-bit down-counter called SysTick, baked into the CPU itself (so it's identical across all STM32). You load a reload value; SysTick counts down from it to zero at the clock rate, fires an interrupt at zero, reloads, and repeats. It's the canonical way to make a periodic "tick" — the 1 ms beat behind HAL_Delay() and most RTOS schedulers.

② Math computing the reload value

SysTick (by default) counts at the full processor clock $f_{\text{SYSCLK}}$. To get an interrupt every $T$ seconds you need it to count $f_{\text{SYSCLK}}\cdot T$ ticks. Because the counter spends one extra cycle passing through zero before reloading, the value you write is one less than the tick count:

$$ \text{RELOAD} = f_{\text{SYSCLK}} \cdot T - 1 $$

For a 1 ms tick, $T = 10^{-3}\,\text{s}$, so $\text{RELOAD} = f_{\text{SYSCLK}}/1000 - 1$. On the F1 at 72 MHz and the F4 at 168 MHz:

$$ \text{RELOAD}_{72} = \frac{72\times10^{6}}{1000} - 1 = 71{,}999, \qquad \text{RELOAD}_{168} = \frac{168\times10^{6}}{1000} - 1 = 167{,}999 $$

Both fit in SysTick's 24-bit register (max $2^{24}-1 = 16{,}777{,}215$), so a 1 ms tick is comfortable at any STM32 speed. The CMSIS one-liner SysTick_Config(SystemCoreClock / 1000) does this exact subtraction for you and also enables the interrupt.

③ Code by register and by helper
systick.c C / STM32 (CMSIS)
#include "stm32f4xx.h"

volatile uint32_t g_ticks = 0;   // 1 ms counter, written by the ISR

// The CMSIS one-liner: load reload, clear count, pick clock source,
// enable counter + interrupt. SystemCoreClock holds your real SYSCLK.
void systick_init_1ms(void) {
    SysTick_Config(SystemCoreClock / 1000u);   // e.g. 168000000/1000 = 168000
}                                              //   → RELOAD = 168000 - 1 = 167999

// The handler the NVIC calls every time SysTick hits zero (every 1 ms).
void SysTick_Handler(void) {
    g_ticks++;
}

// A non-blocking millisecond delay built on the tick.
void delay_ms(uint32_t ms) {
    uint32_t start = g_ticks;
    while ((g_ticks - start) < ms) {
        __WFI();   // sleep until the next interrupt — saves power vs. busy-wait
    }
}
🏭 Real case — the tick that runs the whole product

That g_ticks counter is the spine of real firmware. FreeRTOS uses SysTick as its scheduler tick to time-slice tasks; bare-metal code uses it to schedule "do this every 10 ms" jobs without blocking; comms stacks use it for timeouts and retransmits. One 24-bit counter, derived all the way down from your 8 MHz crystal through the PLL, becomes the sense of time for the entire device. Get the clock tree wrong and every timeout in the product is wrong by the same factor.

8Putting it together: the bring-up sequence

④ Real case this is literally SystemInit()

Every piece now slots into one ordered ritual that runs before main(). The order is not optional — each step depends on the last, and getting it wrong (especially Flash latency) hangs the chip.

🔑 Why each "wait for ready" matters

Oscillators and the PLL take real time to stabilize (HSE startup is milliseconds; PLL lock is microseconds). The HSERDY / PLLRDY flags are the hardware telling you "I'm stable now." Switch onto a source before its ready flag is set and you clock the core off a wobbling, half-locked signal — intermittent, maddening, heisenbug-grade failures. Always poll the ready bit.

In real projects you almost never hand-write this — STM32CubeMX generates SystemClock_Config() from a graphical clock-tree diagram, and CMSIS ships a SystemInit(). But now you can read that generated code, spot when the numbers are wrong, and fix the day your USB won't enumerate because someone left the source on HSI.

Q1 Conceptual check
You write code to configure GPIOD->MODER, but every read of that register returns 0 and the LED never lights. The compiler and debugger show no errors. What is the most likely cause?
  • The crystal (HSE) failed to start, so SYSCLK is 0
  • You forgot RCC->AHB1ENR |= RCC_AHB1ENR_GPIODEN — port D is unclocked, so its registers won't latch
  • PD12 needs an external pull-up resistor to read back as written
  • Flash wait states are too low for the current clock speed
This is the classic STM32 bug. At reset the RCC gates every peripheral clock off to save power; an unclocked peripheral is frozen and its registers ignore writes (reading back the reset value, usually 0). Enable the clock in the matching RCC xENR register before touching the peripheral. A dead HSE would hang the whole chip, not just port D; pull-ups don't affect output writes; wrong wait states corrupt code fetches, not GPIO reads.

Q3 Coding challenge — bring SYSCLK to 168 MHz
Write the register-level sequence that switches an STM32F4 from its reset HSI to 168 MHz off the 8 MHz HSE via the PLL (M=8, N=336, P=2, Q=7), then starts a 1 ms SysTick. Get the order right: which step must come before the clock switch, and why? Set AHB÷1, APB1÷4, APB2÷2.
Hint
Order is everything: (1) enable HSE and poll RCC_CR_HSERDY. (2) raise Flash latency to 5 WS before going fast, or the first 168 MHz fetch reads garbage. (3) set the bus prescalers in RCC->CFGR. (4) program RCC->PLLCFGR with M/N/P/Q and HSE as source. (5) enable PLL, poll PLLRDY. (6) select PLL as SYSCLK and poll SWS. (7) SysTick_Config(168000000/1000). Note P=2 encodes as 0b00 in the PLLCFGR P field.
Show one solution
clock_168.c C / STM32F4 (CMSIS)
#include "stm32f4xx.h"

void clock_config_168mhz(void) {
    // (1) Start HSE (8 MHz crystal) and wait until it is stable.
    RCC->CR |= RCC_CR_HSEON;
    while (!(RCC->CR & RCC_CR_HSERDY)) { }

    // (2) Flash: 5 wait states + prefetch & caches, BEFORE speeding up.
    FLASH->ACR = FLASH_ACR_LATENCY_5WS | FLASH_ACR_PRFTEN
               | FLASH_ACR_ICEN        | FLASH_ACR_DCEN;

    // (3) Bus prescalers: AHB /1 (HCLK=168), APB1 /4 (42), APB2 /2 (84).
    RCC->CFGR = RCC_CFGR_HPRE_DIV1 | RCC_CFGR_PPRE1_DIV4
              | RCC_CFGR_PPRE2_DIV2;

    // (4) PLL: M=8, N=336, P=2 (field=0), Q=7, source = HSE.
    RCC->PLLCFGR = (8u   << RCC_PLLCFGR_PLLM_Pos)   // /M = 8  → 1 MHz
                 | (336u << RCC_PLLCFGR_PLLN_Pos)   // xN = 336 → 336 MHz VCO
                 | (0u   << RCC_PLLCFGR_PLLP_Pos)   // /P: 00 = ÷2 → 168 MHz
                 | (7u   << RCC_PLLCFGR_PLLQ_Pos)   // /Q = 7  → 48 MHz USB
                 | RCC_PLLCFGR_PLLSRC_HSE;          // PLL input = HSE

    // (5) Turn the PLL on and wait for lock.
    RCC->CR |= RCC_CR_PLLON;
    while (!(RCC->CR & RCC_CR_PLLRDY)) { }

    // (6) Switch SYSCLK to the PLL and confirm the switch happened.
    RCC->CFGR |= RCC_CFGR_SW_PLL;
    while ((RCC->CFGR & RCC_CFGR_SWS) != RCC_CFGR_SWS_PLL) { }

    SystemCoreClock = 168000000u;   // tell CMSIS the new rate

    // (7) 1 ms SysTick: RELOAD = 168000000/1000 - 1 = 167999.
    SysTick_Config(SystemCoreClock / 1000u);
}

The load-bearing detail is step 2 before step 6: raise Flash latency while still slow, then speed up. Reverse them and the first instruction fetched at 168 MHz comes back corrupt and the core HardFaults before main(). This is essentially what CubeMX's SystemClock_Config() generates — you can now read and audit it.
