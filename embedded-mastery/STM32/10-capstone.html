Master Capstone: Real-Time Motor Control with PID

Everything in this track was a piece of one machine. Now we assemble it: a quadrature encoder feeding a timer, a fixed-rate PID running in an interrupt, PWM driving an H-bridge, and telemetry streaming out over UART+DMA — a closed-loop DC-motor speed controller that holds a setpoint while the world pushes back. This is the grand finale, where clocks, timers, interrupts, and DMA stop being topics and become a system.

1The whole machine, one loop

① Intuition build the picture first

A motor is stubborn. Tell it "spin at 3000 RPM" with a fixed voltage and it will obey only until something changes — a heavier load, a warmer winding, a lower battery — and then it drifts. Open-loop control (pick a duty cycle and hope) cannot fight back. The fix is the oldest idea in engineering: measure the output, compare it to what you wanted, and correct the difference, continuously. That is a closed loop, and the controller in the middle is a PID.

💡 The one-sentence model

A feedback controller spins a wheel of measure → compare → correct thousands of times a second, nudging the actuator just enough to drive the error — the gap between setpoint and reality — toward zero.

3Measuring speed: from edges to RPM

① Intuition counting teeth

A quadrature encoder bolted to the motor shaft puts out two square waves, A and B, 90° out of phase. The phase order tells direction; the edge count tells distance. In encoder mode (Ch 5) the timer's hardware does the decoding for free: every A/B edge increments or decrements TIMx->CNT. The CPU never sees an interrupt per edge — it just reads a register that already holds the position. To get speed, we sample that counter once per control period and look at how much it moved.

② Math derive RPM honestly

Let $\text{CPR}$ be the encoder's counts per revolution. A datasheet usually quotes lines (PPR); in 4× quadrature decoding the timer counts every edge of both channels, so

$$ \text{CPR} = 4 \times \text{PPR} $$

Say the shaft turns through $\Delta\text{counts}$ in one control period $T_s$ (seconds), and a gearbox of ratio $G$ sits between motor and output (encoder on the motor side). Revolutions of the output shaft in that window are $\Delta\text{counts}/(\text{CPR}\cdot G)$, and there are $60/T_s$ windows per minute, so:

$$ \text{RPM} = \frac{\Delta\text{counts}}{\text{CPR}\cdot G}\cdot\frac{60}{T_s} $$

Worked example: a 500-line encoder ($\text{PPR}=500 \Rightarrow \text{CPR}=2000$), no gearbox ($G=1$), $T_s = 1\,\text{ms}$. If $\Delta\text{counts}=100$ in one tick:

$$ \text{RPM} = \frac{100}{2000\cdot 1}\cdot\frac{60}{0.001} = 0.05 \times 60000 = 3000\ \text{RPM} $$
⚠️ Pitfall — counter wraparound

The delta must wrap correctly. On a 16-bit counter, going from 65530 to 4 is +10, not −65526. Compute delta = (int16_t)(now - prev) and let two's-complement subtraction do the wrap for you, or use a 32-bit timer (TIM2/TIM5) and a wider cast. Get this wrong and your "speed" glitches violently once per overflow.

③ Code read every line
encoder.c C / STM32 (CMSIS)
// TIM2 is preconfigured in encoder mode (TI1+TI2, 4x).
// CPR = 4 * PPR. Ts is the control period in seconds.
#define ENC_CPR   2000.0f   // 500-line encoder, 4x quadrature
#define ENC_GEAR  1.0f
#define CTRL_TS   0.001f    // 1 kHz

static uint32_t enc_prev = 0;

// Call exactly once per control period.
float encoder_read_rpm(void) {
    uint32_t now = TIM2->CNT;                 // hardware-decoded position
    // signed delta with correct 32-bit wrap
    int32_t  d   = (int32_t)(now - enc_prev);
    enc_prev = now;
    // RPM = (d / (CPR*GEAR)) * (60 / Ts)
    return ((float)d / (ENC_CPR * ENC_GEAR)) * (60.0f / CTRL_TS);
}

One subtraction, one cast, one multiply — that is the entire speed sensor. All the heavy lifting (debouncing edges, decoding direction) is done by the timer peripheral while the CPU sleeps. That is the whole point of using the silicon instead of polling pins.

4The PID controller

① Intuition three voices arguing

PID is three corrections summed. Proportional reacts to the error now — bigger gap, harder push, but pure P always leaves a residual offset. Integral remembers past error, accumulating until even a tiny steady gap is stamped out — it kills offset but can overshoot. Derivative anticipates the future by reacting to how fast the error is changing — it damps, like a shock absorber. Tune their volumes and you get a controller that arrives fast, settles flat, and doesn't ring.

② Math continuous, then discrete

With error $e(t) = \text{setpoint} - \text{measured}$, the classic continuous control law is:

$$ u(t) = K_p\,e(t) + K_i\!\int_0^{t}\! e(\tau)\,d\tau + K_d\,\frac{de(t)}{dt} $$

A microcontroller can't integrate continuously — it runs in discrete ticks of width $T_s$. Replace the integral with a running sum (rectangular rule) and the derivative with a backward difference. At step $k$:

$$ u_k = K_p\,e_k \;+\; K_i\,T_s\!\sum_{j=0}^{k} e_j \;+\; K_d\,\frac{e_k - e_{k-1}}{T_s} $$

Look closely at where $T_s$ appears: it multiplies the integral and divides the derivative. The same $K_i$ and $K_d$ mean completely different things if $T_s$ wanders. That single fact is why §6 exists — the loop must tick at a constant rate or your tuning is a lie.

∑ Anti-windup — the integral's dark side

When the motor saturates (duty pinned at 100%) the output can't grow, but $\sum e_j$ keeps piling up. When the setpoint finally eases, that bloated integral takes ages to unwind, causing a huge overshoot. The cure is clamping: stop accumulating whenever the output is saturated, and bound the integral to the range that can actually produce a legal output:

$$ I_{\min} \le K_i\,T_s\!\sum e_j \le I_{\max}, \qquad I_{\max}=u_{\max} $$

③ Code the update function
pid.c C / STM32
typedef struct {
    float kp, ki, kd;     // gains
    float ts;             // sample period [s]
    float integ;          // running ACCUMULATED integral term (already * ki*ts)
    float prev_e;         // e_{k-1}
    float out_min, out_max;
} pid_t;

float pid_update(pid_t *p, float setpoint, float measured) {
    float e = setpoint - measured;             // e_k

    float P = p->kp * e;                        // proportional
    p->integ += p->ki * p->ts * e;              // accumulate Ki*Ts*Σe
    // clamp integral (anti-windup) BEFORE using it
    if (p->integ > p->out_max) p->integ = p->out_max;
    if (p->integ < p->out_min) p->integ = p->out_min;
    float D = p->kd * (e - p->prev_e) / p->ts; // backward difference
    p->prev_e = e;

    float u = P + p->integ + D;
    if (u > p->out_max) u = p->out_max;        // clamp output
    if (u < p->out_min) u = p->out_min;
    return u;                                   // signed: sign = direction
}

Note u is signed. Its magnitude is "how hard"; its sign is "which way". That sign is exactly what the H-bridge needs next.

5Turning u into voltage: PWM & the H-bridge

① Intuition magnitude and direction

The PID hands us a signed number. A motor needs two things to act on it: how much average voltage (set by PWM duty cycle) and which polarity (set by an H-bridge's switch pattern). We split u into |u| → duty and sign(u) → direction pins.

② Math duty from PWM registers

From Ch 5: a timer counts to $\text{ARR}$ and the compare register $\text{CCR}$ sets the duty. The PWM frequency and duty are:

$$ f_{\text{pwm}} = \frac{f_{\text{tim}}}{(\text{PSC}+1)(\text{ARR}+1)}, \qquad D = \frac{\text{CCR}}{\text{ARR}+1} $$

On TIM1 clocked at $f_{\text{tim}}=84\,\text{MHz}$ (APB2 timer clock), choosing $\text{PSC}=0$ and $\text{ARR}=4199$ gives $f_{\text{pwm}} = 84{\times}10^6/4200 = 20\,\text{kHz}$ — above audible range, so the motor doesn't whine, and fine enough that $\text{CCR}\in[0,4200]$ gives ~0.024% duty resolution.

🏭 Real case — why 20 kHz, not 1 kHz

Drone ESCs, e-bike controllers, and CNC spindle drivers nearly all run their motor PWM at 16–30 kHz for one reason: human hearing tops out around 20 kHz. Push the switching above that and the magnetostriction "whine" disappears into silence. Below it, every quadcopter on the bench screams. The control loop still runs at 1 kHz; only the PWM carrier is fast. Keep the two rates separate in your head.

③ Code split sign and magnitude
motor.c C / STM32
#define PWM_ARR   4199u            // 20 kHz @ 84 MHz, PSC=0

// u is the signed PID output, here scaled so |u| in [0, 1].
void motor_set(float u) {
    // direction: drive the H-bridge enable pins
    if (u >= 0.0f) {
        GPIOA->BSRR = (1u << 8);          // IN1 = 1  (forward)
        GPIOA->BSRR = (1u << (9 + 16));   // IN2 = 0  (reset bit 9)
    } else {
        GPIOA->BSRR = (1u << (8 + 16));   // IN1 = 0
        GPIOA->BSRR = (1u << 9);          // IN2 = 1  (reverse)
        u = -u;                          // take magnitude
    }
    if (u > 1.0f) u = 1.0f;
    // magnitude -> CCR.  D = CCR/(ARR+1)  =>  CCR = D*(ARR+1)
    TIM1->CCR1 = (uint32_t)(u * (PWM_ARR + 1));
}
⚠️ Pitfall — shoot-through

An H-bridge has two transistors per leg. If the top and bottom of one leg are ever both on, you short the supply through them — a "shoot-through" that can vaporize the bridge in microseconds. Real gate drivers insert dead-time (a few hundred ns where both are off) between switching. TIM1 generates this in hardware via the BDTR register's DTG field. Never drive a half-bridge from naked GPIO without dead-time.

6The fixed-rate control loop

① Intuition the metronome

The whole loop lives inside one timer interrupt that fires every $T_s$. Not "roughly every millisecond when the main loop gets around to it" — exactly every millisecond, jitter-free, because a hardware timer (TIM6) generates the interrupt independent of whatever else the CPU is doing. Inside the ISR we run the three steps and leave. Everything else (telemetry formatting, command parsing) happens in the slow main loop.

🔑 Why fixed $T_s$ is non-negotiable

Recall the discrete law: the integral term is $K_i T_s \sum e$ and the derivative is $K_d (e_k - e_{k-1})/T_s$. Both scale directly with $T_s$. If the period jitters — say the ISR is sometimes delayed by a busy printf — then the effective $K_i$ and $K_d$ jitter with it. A controller tuned for 1 ms behaves differently at 1.3 ms: the derivative kick weakens, the integral over-accumulates, and a stable loop can start to ring or even go unstable. Fixed $T_s$ is what makes the math you tuned actually be the math that runs.

② Math generating the 1 kHz tick

TIM6 on APB1 sees a 84 MHz timer clock on this part. For a 1 kHz update we want $(\text{PSC}+1)(\text{ARR}+1) = f_{\text{tim}}/f_s$:

$$ (\text{PSC}+1)(\text{ARR}+1) = \frac{84\times10^6}{1000} = 84000 $$

Pick $\text{PSC}=83$ (divide by 84 → 1 MHz tick) and $\text{ARR}=999$ (count 1000 ticks → 1 ms). Clean, exact, and re-derivable in your head at 3 a.m.

7The full integrated firmware

③ Code read every line

Here is the control ISR — the beating heart. It is deliberately tiny: three driver calls and a flag. Determinism comes from doing less, not more, inside the interrupt.

control.c C / STM32 (CMSIS)
volatile float g_setpoint_rpm = 0.0f;   // commanded by host
volatile float g_measured_rpm = 0.0f;   // last measurement (for telemetry)
volatile uint8_t g_tlm_ready  = 0;      // ISR -> main handshake

static pid_t pid = {
    .kp = 0.0008f, .ki = 0.012f, .kd = 0.00003f,
    .ts = CTRL_TS, .integ = 0.0f, .prev_e = 0.0f,
    .out_min = -1.0f, .out_max = 1.0f
};

// TIM6 fires this at EXACTLY 1 kHz. Keep it short and branch-light.
void TIM6_DAC_IRQHandler(void) {
    if (TIM6->SR & TIM_SR_UIF) {        // confirm it was the update event
        TIM6->SR &= ~TIM_SR_UIF;        // clear the interrupt flag (W0C)

        float rpm = encoder_read_rpm();           // 1. measure
        float u   = pid_update(&pid,
                               g_setpoint_rpm, rpm); // 2. compute
        motor_set(u);                              // 3. actuate

        g_measured_rpm = rpm;          // publish for telemetry
        g_tlm_ready    = 1;            // wake the main loop's sender
    }
}

The main() brings up clocks and peripherals, then loops forever doing only the slow, non-critical work: throttling telemetry to ~50 Hz and shipping it out via UART+DMA so the CPU never blocks on the slow serial line (Ch 7/8).

main.c C / STM32
static char tlm[48];

int main(void) {
    clock_config_84mhz();      // Ch 2: PLL -> 84 MHz timer clock
    encoder_init_tim2();       // Ch 5: TIM2 encoder mode, 4x
    pwm_init_tim1_20khz();     // Ch 5: TIM1 PWM + dead-time (BDTR)
    uart_dma_init();           // Ch 7/8: USART2 TX via DMA1
    ctrl_timer_init_1khz();    // TIM6, PSC=83, ARR=999, enable IRQ

    g_setpoint_rpm = 1500.0f;  // demo: hold 1500 RPM

    uint32_t div = 0;
    for (;;) {
        if (g_tlm_ready) {
            g_tlm_ready = 0;
            // decimate 1 kHz -> ~50 Hz so we don't flood the UART
            if (++div >= 20) {
                div = 0;
                int n = snprintf(tlm, sizeof tlm, "%ld,%ld\n",
                                 (long)g_setpoint_rpm,
                                 (long)g_measured_rpm);
                uart_dma_send((uint8_t *)tlm, n);  // non-blocking
            }
        }
        // CPU is otherwise free: sleep until next event
        __WFI();
    }
}
🏭 Real case — why DMA, not printf

At 115200 baud, pushing ~14 bytes of telemetry byte-by-byte with a blocking HAL_UART_Transmit stalls the CPU for over a millisecond — long enough to miss the next control tick. Hand the buffer to DMA and the transfer happens in the background while the core sleeps in __WFI(). This is exactly the pattern in production motor drives, flight controllers, and lab instruments: the fast deterministic loop is sacred, and slow I/O is always offloaded. Break that rule and your loop jitter (and thus your tuning) falls apart.

8Tuning & debugging

④ Real case making it actually behave

Gains don't come from a formula — they come from a method. A Ziegler–Nichols-style procedure gives you a sane starting point:

StepWhat you doWhat you watch
1. P onlySet $K_i=K_d=0$; raise $K_p$ until the speed oscillates steadilyOnset of sustained oscillation → that's the ultimate gain $K_u$
2. MeasureRecord the oscillation period $T_u$Period between peaks on the telemetry plot
3. ApplyClassic ZN: $K_p=0.6K_u$, $K_i=1.2K_u/T_u$, $K_d=0.075K_uT_u$Step response: rise, overshoot, settle
4. TrimNudge by feel: more $K_d$ to damp ringing, less $K_i$ to cut overshootOvershoot < 10%, settle < 200 ms

The single most useful diagnostic is the step response: command a sudden jump in setpoint and plot measured RPM versus the commanded value over time. Its shape is your tuning, read at a glance:

🛠️ Watching the loop from outside

Three windows into a running controller, cheapest first: (1) the UART+DMA telemetry stream → plot it live in a serial plotter. (2) SWD/ITM — toggle a GPIO at the top and bottom of the ISR and scope it; the pulse width is your ISR execution time, and its spacing proves $T_s$ is jitter-free. (3) the debugger's live-watch on g_measured_rpm and pid.integ — a creeping integral that never settles is a windup smell.

⚠️ Common failure modes

Wrong feedback sign — encoder A/B swapped, so error correction adds energy: the motor slams to full speed. Fix: negate the count delta or swap channels.
ISR overrun — the control ISR takes longer than $T_s$; the next tick is missed and $T_s$ silently doubles. Fix: shrink the ISR, move work to main.
Integral windup — slow recovery and big overshoot after saturation. Fix: the clamp from §4.
Derivative noise — a noisy encoder makes $D$ scream; filter the measurement or lower $K_d$.

9Where to go next: cascade & current limits

④ Real case how the pros build it

A single speed loop is the foundation; serious drives stack loops. The standard architecture is a cascade: an outer position/speed PID whose output is the setpoint of an inner current (torque) PID running even faster. The inner loop limits current directly — and current limiting is torque limiting is protecting your motor and bridge from burning up.

flowchart LR SP([Speed SP]) --> S["Speed PID
(1 kHz)"] S -- "current SP" --> C["Current PID
(10 kHz)"] C -- "duty" --> P["PWM + bridge"] P --> M(["Motor"]) M -- "shaft speed" --> S M -- "phase current
(ADC shunt)" --> C

The inner loop reads motor current from a shunt resistor via the ADC, often synchronized to the PWM (sample at the bottom of the count, when current is steady). With that in place you get hard, fast current limiting — and you are one step from the field-oriented control (FOC) that drives every modern brushless motor, from drones to EVs. The pieces you assembled here are exactly the pieces FOC is built from.

🔑 The pattern generalizes

Measure with a timer/ADC, compute in a fixed-rate ISR, actuate with PWM, observe over DMA telemetry. Swap the plant — heater, drone attitude, battery charger, laser power — and the skeleton is identical. You didn't just learn motor control; you learned the shape of every real-time control system.

Q1 Conceptual check
Why must the PID control loop run at a fixed sample period $T_s$, rather than just "as often as the main loop gets to it"?
  • Because the encoder hardware only updates its counter on a fixed schedule
  • Because the discrete integral ($K_iT_s\sum e$) and derivative ($K_d\,\Delta e/T_s$) terms scale with $T_s$, so a jittering period changes the effective gains and can destabilize a tuned loop
  • Because PWM cannot be generated unless the loop period is constant
  • Because UART+DMA telemetry requires a constant data rate
The integral multiplies by $T_s$ and the derivative divides by it. If $T_s$ drifts (e.g. a blocking printf delays the ISR), the effective $K_i$ and $K_d$ drift too — the controller you carefully tuned is no longer the controller that runs, and a stable loop can begin to ring or go unstable. The encoder, PWM, and telemetry all tolerate jitter; the control math does not.
Q2 Compute it
A 500-line encoder (so $\text{CPR}=4\times500=2000$) is on the motor shaft with no gearbox ($G=1$). The control loop runs at 1 kHz ($T_s=0.001\,\text{s}$). In one control period the counter advances by $\Delta\text{counts}=100$. What is the speed in RPM? Use $\text{RPM} = \dfrac{\Delta\text{counts}}{\text{CPR}\cdot G}\cdot\dfrac{60}{T_s}$.
RPM
$\text{RPM} = \dfrac{100}{2000\cdot 1}\cdot\dfrac{60}{0.001} = 0.05 \times 60000 = 3000\ \text{RPM}$. The fraction $100/2000 = 0.05$ is revolutions in this 1 ms window; multiplying by $60/T_s = 60000$ windows-per-minute gives RPM. Halve $\Delta\text{counts}$ and you halve the speed; double $T_s$ and the same delta reads half the RPM.
Q3 Coding challenge — the discrete PID update
Write a C function pid_update(pid_t *p, float setpoint, float measured) that implements the discrete law $u_k = K_p e_k + K_i T_s\sum e + K_d (e_k - e_{k-1})/T_s$, returns a signed output clamped to [out_min, out_max], and includes integral anti-windup clamping. Assume the struct holds kp, ki, kd, ts, integ, prev_e, out_min, out_max.
Hint
Compute e = setpoint - measured first. Accumulate the integral as integ += ki*ts*e and clamp integ to the output bounds before using it (that's the anti-windup). The derivative is kd*(e - prev_e)/ts — remember to update prev_e afterward. Sum the three terms, clamp the total, return it. The sign of the return value is the motor direction.
Show one solution
pid.c C / STM32
float pid_update(pid_t *p, float setpoint, float measured) {
    float e = setpoint - measured;             // e_k

    float P = p->kp * e;                        // proportional

    p->integ += p->ki * p->ts * e;              // Ki*Ts*Σe
    if (p->integ > p->out_max) p->integ = p->out_max;  // anti-windup
    if (p->integ < p->out_min) p->integ = p->out_min;

    float D = p->kd * (e - p->prev_e) / p->ts; // backward difference
    p->prev_e = e;                             // store for next tick

    float u = P + p->integ + D;
    if (u > p->out_max) u = p->out_max;        // output clamp
    if (u < p->out_min) u = p->out_min;
    return u;   // |u| -> PWM duty, sign(u) -> H-bridge direction
}

Two clamps, not one: clamp the integral so it can't wind up past what the actuator can deliver, and clamp the total output to the legal duty range. Storing prev_e is what makes the derivative a true difference across ticks — and it's exactly why the loop must run at fixed $T_s$.
