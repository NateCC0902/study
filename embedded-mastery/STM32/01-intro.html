From AVR to ARM: Cortex-M, Memory Map & Toolchains

You know how an 8-bit AVR blinks an LED. Now we step up to a 32-bit ARM Cortex-M core running an order of magnitude faster, with peripherals that do work while the CPU sleeps. Same physics, far bigger machine. We rebuild your mental model around a 32-bit memory map and the toolchains that program it — then blink an LED two ways to see exactly what the comfort of a library costs you.

1Why leave the comfort of 8-bit AVR?

① Intuition build the picture first

The ATmega328P served you well: 8-bit, 16 MHz, 2 KB of SRAM, a fistful of peripherals. It is a wonderful teacher precisely because it is small enough to hold entirely in your head. But the moment a project wants USB, an LCD, audio, motor control, or a TCP/IP stack, the 328P runs out of clock cycles and RAM long before it runs out of pins. You need a bigger machine — and the industry's answer is the ARM Cortex-M family, packaged by ST Microelectronics as STM32.

💡 The one-sentence model

An STM32 is the same idea as your Arduino's MCU — one chip with CPU, memory, and peripherals — but built on a 32-bit ARM core running 4–10× faster, with 10–100× more memory and peripherals smart enough to move data without the CPU.

Concretely, compare the Uno to two STM32 boards you can buy for a few dollars: the STM32F103C8T6 ("Blue Pill", a Cortex-M3) and the STM32F411CEU6 ("Black Pill", a Cortex-M4 with floating-point unit). The numbers tell the story.

🔑 The real upgrade is DMA

More megahertz is nice, but the game-changer is DMA (Direct Memory Access): a dedicated engine that copies bytes between a peripheral and memory autonomously. On an AVR, reading 1000 ADC samples means the CPU babysits every single one. On STM32, you arm a DMA channel and the CPU is free to do other work — or sleep to save power — while the samples stream into a buffer behind its back. That is how a 100 MHz chip out-performs its clock rate.

2The Cortex-M family & the STM32 line-up

① Intuition one core architecture, many sizes

"Cortex-M" is not one chip — it is a licensable CPU core design from ARM. ST buys the core, then wraps its own memory and peripherals around it to make an STM32. Knowing which Cortex-M sits inside tells you the chip's instruction set, whether it has a floating-point unit, and roughly how much it costs.

🛠️ Reading a part number

In STM32F411CEU6: F4 = line, 11 = sub-family, C = 48 pins, E = 512 KB Flash, U = QFN package, 6 = −40…85 °C temperature grade. You can decode any STM32 part number at a glance once you know the fields — and the datasheet's first page lists them all.

For this whole track we will keep two chips as running examples: the STM32F103 (cheap, M3, the canonical first STM32) and the STM32F4 family (M4 + FPU, fast, our workhorse). Everything you learn transfers across the line — the register names and memory map are remarkably consistent.

3The 32-bit memory map

① Intuition one flat 4 GB address space

This is the single most important idea in the jump from AVR to ARM. The AVR used a Harvard architecture: program and data lived in separate address spaces, which is why PROGMEM tricks existed. Cortex-M flips to a single, flat, 32-bit address space shared by everything — Flash, RAM, peripherals, and the core's own control registers all sit at different addresses in one map.

A 32-bit pointer can name $2^{32}$ distinct byte addresses, which is exactly 4 GiB of address space, even though the chip only populates a few hundred kilobytes of it:

$$ 2^{32} \text{ bytes} = 4{,}294{,}967{,}296 \text{ bytes} = 4\,\text{GiB} = \texttt{0x0000\,0000} \;\text{…}\; \texttt{0xFFFF\,FFFF} $$

ARM defines a standard layout for that 4 GiB so every Cortex-M chip feels familiar. The four regions you must memorize:

RegionBase addressWhat lives here
Flash (code)0x0800 0000Your program, the vector table, constants
SRAM (data)0x2000 0000Variables, stack, heap — volatile
Peripherals0x4000 0000GPIO, timers, UART, ADC… register blocks
Core (private)0xE000 0000NVIC, SysTick, SCB — the Cortex-M itself
🔑 Everything is an address

On Cortex-M there is no special "I/O instruction" like the AVR's OUT. A peripheral register is just memory at some address up in the 0x4000 0000 region. To turn an LED on you literally write a 32-bit value to a numeric address. The whole machine is reachable through one pointer type.

② Math base + offset = register address

Each peripheral occupies a contiguous block starting at a base address. Within that block, each register sits at a fixed offset defined by the reference manual. The address of any register is simply:

$$ \text{addr}_{\text{reg}} = \text{base}_{\text{periph}} + \text{offset}_{\text{reg}} $$

Work a concrete one. On the STM32F4, GPIO port A hangs off the AHB1 bus with base address 0x4002 0000. The output data register (ODR) — the one whose bits map directly to pin levels — sits at offset 0x14 inside the port. So:

$$ \texttt{GPIOA->ODR} = \underbrace{\texttt{0x4002\,0000}}_{\text{base}} + \underbrace{\texttt{0x14}}_{\text{offset}} = \texttt{0x4002\,0014} $$

In hex that addition is painless: the low nibble 0x0 + 0x14 = 0x14, everything above is unchanged. The CMSIS struct GPIOA->ODR that you will write in code resolves, at compile time, to exactly this constant. The compiler is doing base-plus-offset arithmetic so you do not have to remember 0x40020014 — but now you know where it comes from.

⚠️ Pitfall — Flash is at 0x0800 0000, not 0x0

The very bottom of the map (0x0000 0000) is an alias region whose contents depend on the boot pins — it usually mirrors Flash so the CPU can fetch its first instruction there. But the real, fixed address of your code Flash is 0x0800 0000. Always link and reason about Flash at 0x0800 0000; treat the 0x0 alias as a boot-time convenience only.

4Registers are memory: volatile & CMSIS

③ Code read every line

If a register is just memory at a known address, you can reach it with a plain pointer. Here is GPIOA's ODR the raw way, then the way you will actually write it:

raw_register.c C (bare metal)
#include <stdint.h>

// The address we derived: GPIOA base 0x40020000 + ODR offset 0x14.
// 'volatile' is MANDATORY: it tells the compiler this memory can change
// on its own (hardware!), so it must NOT cache the value in a CPU register
// or optimize the access away.
#define GPIOA_ODR  (*(volatile uint32_t *)0x40020014u)

void set_pin5_high(void) {
    GPIOA_ODR |= (1u << 5);   // set bit 5 → drive PA5 high
}

Writing magic numbers everywhere is error-prone, so ARM and ST ship CMSIS (Cortex Microcontroller Software Interface Standard): a set of header files that describe every peripheral as a C struct whose members line up exactly with the register offsets. You include one device header and the addresses are done for you:

cmsis_register.c C (CMSIS)
#include "stm32f4xx.h"   // ST's CMSIS device header

// GPIO_TypeDef is a struct laid out to match the hardware:
//   typedef struct {
//     volatile uint32_t MODER;   // offset 0x00
//     volatile uint32_t OTYPER;  // offset 0x04
//     volatile uint32_t OSPEEDR; // offset 0x08
//     volatile uint32_t PUPDR;   // offset 0x0C
//     volatile uint32_t IDR;     // offset 0x10
//     volatile uint32_t ODR;     // offset 0x14  ← here
//     volatile uint32_t BSRR;    // offset 0x18
//     ...
//   } GPIO_TypeDef;
// And: #define GPIOA ((GPIO_TypeDef *) 0x40020000UL)

void set_pin5_high(void) {
    GPIOA->ODR |= (1u << 5);   // identical to the raw version, but readable
}
🔑 Why volatile is non-negotiable

A normal variable lives in RAM and only your code changes it, so the compiler may keep it in a CPU register and skip redundant reads. A hardware register can change between two of your instructions (a button was pressed; a byte arrived) and writing it has side effects (a pin moves). volatile forces the compiler to emit a real load/store for every access, in program order. Forget it and your loop polling USART->SR can hang forever because the compiler "knows" the value never changes. CMSIS marks every register volatile for you — but you must understand why.

5Toolchains & the HAL / LL / CMSIS choice

① Intuition three levels of abstraction

On Arduino there was one way to write code: the Arduino core. STM32 gives you a spectrum, from "click a GUI and it generates code" down to "poke registers yourself". The standard tooling is STM32CubeIDE (a free Eclipse-based IDE) with STM32CubeMX, a graphical configurator that lets you click pins and clocks and then emits a project. The interesting choice is which abstraction layer your code sits on:

LayerWhat it isEaseControlCode sizeSpeed
HAL (Hardware Abstraction Layer)Portable, high-level functions (HAL_GPIO_TogglePin)HighestLowestLargestSlowest
LL (Low-Layer)Thin inline wrappers, ~1:1 with registersMediumHighSmallFast
CMSIS / bare registerYou write GPIOA->BSRR = … directlyLowestHighestSmallestFastest
🛠️ What each layer trades

HAL buys you portability and speed of development — the same call works across the whole STM32 line, with built-in error handling and timeouts — at the cost of larger, slower code and a stack of function calls between you and the silicon. LL keeps you readable but compiles down to nearly the same instructions as hand-written register access. Bare CMSIS gives ultimate control and the tightest code, but you own every datasheet detail and there is no safety net. Most products start in HAL and drop to LL/registers only on the hot paths that need it.

🏭 Real case — HAL bloat in a bootloader

A bootloader that must fit in the first 16 KB of Flash often cannot afford HAL: pulling in HAL_GPIO, HAL_RCC, and the SysTick machinery can blow the budget before you have done anything useful. Teams routinely write bootloaders in pure CMSIS register code for exactly this reason, then switch to HAL in the main application where the Flash budget is generous. Choosing the layer per-component, not per-project, is a mark of an experienced firmware engineer.

6How an STM32 boots

① Intuition before main() runs

On the Uno a bootloader copied your program in and jumped to it. On STM32 there is no Arduino bootloader by default — the chip boots directly from Flash using a small, ARM-defined ritual. Understanding it demystifies every "my global is zero but I set it to 5" bug you will ever hit.

At 0x0800 0000, the very first words of Flash are not code — they are the vector table. The first 32-bit word is the initial stack pointer; the second word is the reset vector, the address of the first instruction to run. The CPU loads these two values on power-up before executing anything:

vector_table (conceptual) C (CMSIS)
// First entries of the vector table, placed at 0x08000000 by the linker.
// The CPU reads word[0] into SP, then jumps to word[1] (Reset_Handler).
__attribute__((section(".isr_vector")))
const uint32_t vector_table[] = {
    (uint32_t)&_estack,          // [0] 0x08000000: initial Main Stack Pointer
    (uint32_t)Reset_Handler,     // [1] 0x08000004: reset vector → first code
    (uint32_t)NMI_Handler,       // [2] 0x08000008: non-maskable interrupt
    (uint32_t)HardFault_Handler, // [3] 0x0800000C: hard fault
    // ... SysTick, then one entry per peripheral IRQ ...
};

The reset vector points at Reset_Handler, a startup routine (usually startup_stm32f4xx.s) that prepares the C runtime before your main() can safely run. It does four jobs in order:

flowchart TB P([Power / Reset]) --> SP["Load SP ← word[0]
at 0x08000000"] SP --> RV["Jump to Reset_Handler
(word[1] @ 0x08000004)"] RV --> DATA["Copy .data
Flash → SRAM
(initialized globals)"] DATA --> BSS["Zero .bss
in SRAM
(uninit globals)"] BSS --> SI["Call SystemInit()
set up clocks / FPU"] SI --> MAIN["Call main()"] MAIN --> LOOP(["your while(1) loop"])
🔑 Why globals start at the right value

An initialized global like int speed = 5; stores its value (5) in Flash but lives at runtime in SRAM. The startup code's .data copy moves those bytes Flash→SRAM. An uninitialized global like int count; must read as 0 per the C standard — that is the .bss zeroing step. If you ever skip or break startup (common when hand-rolling a bare-metal project), globals contain garbage and you will chase a ghost. This is the firmware version of "it works on my machine".

⚠️ Pitfall — SystemInit runs before main

SystemInit() typically configures the PLL so the chip actually runs at 72/84/100 MHz instead of its slow internal default. If your timing is mysteriously 8× too slow, your clock tree was never set up — check that SystemInit() ran and that SystemCoreClock matches reality.

7Blink two ways: HAL vs. raw registers

③ Code the same blink, two abstraction levels

Time to make the contrast concrete. Both programs blink the user LED on an STM32F4 Nucleo/Black Pill (PA5). First the HAL version — short, portable, what CubeMX generates:

blink_hal.c C (HAL)
#include "stm32f4xx_hal.h"

int main(void) {
    HAL_Init();                       // sets up SysTick, flash latency, etc.
    __HAL_RCC_GPIOA_CLK_ENABLE();     // turn on the clock to GPIO port A

    GPIO_InitTypeDef io = {0};
    io.Pin   = GPIO_PIN_5;            // PA5
    io.Mode  = GPIO_MODE_OUTPUT_PP;   // push-pull output
    io.Speed = GPIO_SPEED_FREQ_LOW;
    HAL_GPIO_Init(GPIOA, &io);

    while (1) {
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5); // flip the pin
        HAL_Delay(500);                        // 500 ms, driven by SysTick
    }
}

Now the same blink in raw register code. No library beyond the CMSIS struct definitions — every line maps to a datasheet field. Read the comments; this is the whole point of the chapter:

blink_registers.c C (CMSIS / bare metal)
#include "stm32f4xx.h"   // just the register-struct definitions

int main(void) {
    // 1) Peripherals are clock-gated OFF at reset to save power.
    //    Enable the clock to GPIOA via RCC->AHB1ENR bit 0 (GPIOAEN).
    RCC->AHB1ENR |= (1u << 0);                 // RCC_AHB1ENR_GPIOAEN

    // 2) MODER has 2 bits per pin. Set PA5 to '01' = general-purpose output.
    //    Clear the 2-bit field first, then write 01.
    GPIOA->MODER &= ~(3u << (5 * 2));          // clear bits 11:10
    GPIOA->MODER |=  (1u << (5 * 2));          // set 01 → output

    while (1) {
        // 3) BSRR is the atomic set/reset register. Writing a 1 to a LOW
        //    bit SETS the pin; writing 1 to a HIGH bit (bit + 16) RESETS it.
        //    No read-modify-write, so no interrupt can corrupt it.
        GPIOA->BSRR = (1u << 5);               // PA5 = HIGH
        for (volatile uint32_t d = 0; d < 800000; d++) { }   // crude delay

        GPIOA->BSRR = (1u << (5 + 16));        // PA5 = LOW (reset half)
        for (volatile uint32_t d = 0; d < 800000; d++) { }
    }
}
💡 What the contrast teaches

The HAL version reads like prose and works on any STM32; the register version reads like the datasheet and is a few instructions long. Three details only the raw version exposes: peripherals are clock-gated off until you enable them in RCC (a top cause of "my GPIO does nothing" — the AVR never made you do this); MODER uses two bits per pin, so you mask then set; and BSRR lets you atomically set or reset a pin without a read-modify-write that an interrupt could corrupt. HAL does all three for you — now you know what it is hiding.

⚠️ Pitfall — forgetting the RCC clock enable

The single most common beginner bug on STM32: you configure MODER and write BSRR perfectly, and nothing happens — because you never set the GPIOAEN bit in RCC->AHB1ENR. A clock-gated peripheral ignores all writes silently. Enable the clock first, always.

8Real case: why industry standardized on STM32

④ Real case this ships in millions of products

Walk through the teardown of almost any modern gadget — a drone flight controller, a 3-D printer board, an e-bike controller, an industrial sensor — and you will keep finding STM32 chips. That dominance is not an accident; it is four forces stacking up.

ForceWhy it locks in adoption
PeripheralsA huge, consistent catalog: timers, ADC/DAC, USB, CAN, SPI/I²C/UART, DMA — pick a part with exactly what you need and the code ports across the line.
PerformanceCortex-M scales from cents-cheap M0 to 480 MHz M7 dual-core H7, all binary-compatible at the source level. Start small, scale up without a rewrite.
EcosystemFree CubeIDE/CubeMX, CMSIS, mountains of community code, and the same ARM toolchain (arm-none-eabi-gcc) the whole industry uses. Hiring and tooling are solved problems.
Price & supplyMultiple sources, broad availability, and dollar-class boards make prototyping and production both cheap.
🏭 Real case — the flight controller in your drone

Open-source flight stacks like Betaflight and ArduPilot target STM32 F4/F7/H7 almost exclusively. The reason is concrete: a quadcopter needs to read a gyro over SPI, drive four motor ESCs with hardware timers, talk to a radio receiver over UART, and run a tight PID loop — simultaneously, at hundreds of hertz. The M4's FPU makes the PID math fast, DMA streams the sensor data, and the timer peripherals generate motor signals with zero CPU jitter. That exact combination of peripherals, performance, and free tooling is why an entire hobby-and-industry sits on STM32. You are learning the platform that flies the drone.

Q1 Conceptual check
You access GPIOA->ODR on an STM32F4 to read pin states. In which region of the 32-bit memory map does that register address live, and what does choosing HAL over raw registers mainly trade away?
  • SRAM region (0x2000 0000); HAL trades away portability.
  • Peripheral region (0x4000 0000); HAL trades away code size and speed.
  • Flash region (0x0800 0000); HAL trades away nothing at all.
  • Core/private region (0xE000 0000); HAL trades away portability.
GPIO register blocks sit in the peripheral region at 0x4000 0000 (GPIOA's base is 0x40020000). HAL buys ease and portability; what it costs is code size and speed — extra function-call layers and larger binaries versus a one-line register write. Flash holds code, SRAM holds variables, and 0xE000 0000 is the core's own NVIC/SysTick.
Q2 Compute it
On the STM32F4, GPIO port A has base address 0x40020000 and the output data register ODR sits at offset 0x14. Using $\text{addr}_{\text{reg}} = \text{base} + \text{offset}$, what is the absolute address of GPIOA->ODR? Enter it in hex (e.g. 0x40020014) or as a decimal integer.
addr
$\texttt{0x40020000} + \texttt{0x14} = \texttt{0x40020014}$ (decimal $1073872916$). In hex only the low byte changes: $\texttt{0x00} + \texttt{0x14} = \texttt{0x14}$, everything above untouched. This is the exact constant the CMSIS struct GPIOA->ODR compiles down to.
Q3 Coding challenge — register-level blink on a Blue Pill
The STM32F103 "Blue Pill" has its on-board LED on PC13 (and it is active-low — driving the pin LOW lights it). The F1 family uses different GPIO registers than the F4: a CRH (configuration register high) for pins 8–15, an APB2ENR clock-enable for GPIOC, and a BSRR/BRR pair. Write a bare-register blink for PC13. Think: which clock bus, which config field, how to toggle.
Hint
On F1, GPIOC is on the APB2 bus: set the IOPCEN bit (bit 4) in RCC->APB2ENR. PC13 is configured by GPIOC->CRH: each pin uses a 4-bit field, and PC13's field starts at bit (13 − 8) × 4 = 20. Write 0b0010 there for "output, 2 MHz, push-pull". Use BSRR to set the pin (LED off, since active-low) and the upper half of BSRR — i.e. BRR or bit+16 — to reset it (LED on).
Show one solution
blink_pc13.c C (CMSIS / bare metal)
#include "stm32f1xx.h"

int main(void) {
    // 1) GPIOC is on APB2. Enable its clock (IOPCEN = bit 4).
    RCC->APB2ENR |= (1u << 4);                 // RCC_APB2ENR_IOPCEN

    // 2) PC13 lives in CRH (pins 8..15). Its 4-bit field starts at
    //    bit (13-8)*4 = 20. Clear it, then write 0010 =
    //    MODE=10 (output 2 MHz), CNF=00 (push-pull).
    GPIOC->CRH &= ~(0xFu << 20);               // clear PC13's 4-bit field
    GPIOC->CRH |=  (0x2u << 20);               // 0010: output, push-pull

    while (1) {
        // PC13 is ACTIVE-LOW: reset = LED on, set = LED off.
        GPIOC->BSRR = (1u << (13 + 16));       // reset PC13 → LED ON
        for (volatile uint32_t d = 0; d < 300000; d++) { }

        GPIOC->BSRR = (1u << 13);              // set PC13   → LED OFF
        for (volatile uint32_t d = 0; d < 300000; d++) { }
    }
}

Two lessons land here. First, the register map differs by family: F1 uses CRH/CRL with 4-bit fields and APB2 clocks, while F4 uses MODER with 2-bit fields and AHB1 — yet the workflow (enable clock → configure mode → toggle via BSRR) is identical. Second, active-low hardware inverts your logic: resetting the pin lights the LED. This is why HAL exists — but also why reading the reference manual is a skill you now have.
