計時器精通：PWM、輸入捕捉與編碼器

STM32 的計時器是最具代表性的核心周邊。它底層其實只是一個計數器——但只要在這個計數器外面包上一個預除器（prescaler）、一個自動重載暫存器，再加上幾個比較通道，你就能產生馬達驅動訊號、不耗用 CPU 就把 LED 調暗、以奈秒等級量測脈衝，還能在硬體中解碼正交編碼器（quadrature encoder）。這一切，我們都從一個暫存器建立起來：CNT。

⏱ 約 50 分鐘🧠 計數與比較⚡ PWM／捕捉／編碼器📐 時脈運算

1計時器只是一個對時脈計數的計數器

① 直覺 先建立整體圖像

暫時忘掉「計時器」這個詞。STM32 上像 TIM2 或 TIM3 這樣的通用型計時器，本質上是一個硬體計數器——一個叫做 CNT 的暫存器，每接收到一個時脈邊緣，它就往上加一。整個概念就是這樣。其餘的一切（PWM、週期量測、編碼器解碼）都是透過決定它跳得多快，以及在某些計數值時發生什麼事所建立出來的。

💡 一句話模型

計時器是一個由分頻後時脈所驅動的計數器（CNT），它向上數到一個可程式化的上限（ARR），接著繞回 0，並且能在經過可程式化的比較點（CCRx）時觸發動作。

有三個旋鈕決定了它的行為，而你設定任何一個計時器時都會碰到它們：

  • PSC——預除器（prescaler）。它把進入計時器的時脈分頻，讓 CNT 跳得更慢。分頻倍數為 $(\text{PSC}+1)$。
  • ARR——自動重載暫存器（auto-reload register）。計數器數 0, 1, 2, … ARR，然後歸零並發出一個更新事件（update event）。所以週期是 $(\text{ARR}+1)$ 個計數刻（tick）。
  • CCRx——通道 x 的捕捉／比較暫存器（capture/compare register）。在輸出模式下，它是決定 PWM 工作週期（duty）的門檻值；在輸入模式下，當邊緣到來時它會鎖存（latch）當下的計數值。
🔑 為什麼到處都是「+1」

計數器在 0 到 ARR 之間（含兩端）的每一個值上各停留一個計數刻。這共有 $\text{ARR}+1$ 個不同的值——因此週期 $=\text{ARR}+1$，而不是 ARR。同樣的差一（off-by-one）邏輯讓預除器的分頻倍數變成 $\text{PSC}+1$。硬體從零開始數；人類卻常常忘了那個零。把這點刻進腦海裡。

2計數運算：PSC、ARR 與更新頻率

② 數學 推導出來

計時器的輸入時脈 $f_{\text{tim}}$ 經過預除器分頻後，得到 CNT 遞增的速率：

$$ f_{\text{CNT}} = \frac{f_{\text{tim}}}{\text{PSC}+1} $$

接著計數器需要 $\text{ARR}+1$ 個這樣的計數刻才會溢位（overflow）一次。每一次溢位就是一次更新事件——計時器的心跳。它的頻率是輸入時脈被兩個因子同時分頻後的結果：

$$ f_{\text{update}} = \frac{f_{\text{tim}}}{(\text{PSC}+1)(\text{ARR}+1)} $$

這一條方程式是整章的基礎。PWM 頻率、取樣率中斷、軟體時鐘——它們全都只是套上不同 ARR／CCR 意義的 $f_{\text{update}}$。

⚠️ 陷阱——APB 計時器時脈 ×2 規則

$f_{\text{tim}}$ 並不總是等於你在時脈樹（clock tree）中設定的 APB 匯流排時脈。在 STM32F4/F7/H7 上，當某個 APB 預除器大於 1 時，該匯流排上的計時器時脈會加倍。在一顆 168 MHz 的 F407 上，APB1 為 42 MHz（÷4），那麼 APB1 的計時器時脈就是 $2\times42 = 84\,\text{MHz}$——這正是餵給 TIM2–7 的頻率。我們在第 2 章遇過這條規則；而在這裡，它決定了你算出來的每一個數字。永遠要從 CubeMX 時脈樹的「to TIMx」節點讀取 $f_{\text{tim}}$，而不是匯流排的標籤。

② 數學 反推目標值

實務上你會反向運用這條方程式：你已知 $f_{\text{tim}}$ 和一個目標 $f_{\text{update}}$，然後必須選出 PSC 和 ARR。重新整理：

$$ (\text{PSC}+1)(\text{ARR}+1) = \frac{f_{\text{tim}}}{f_{\text{update}}} $$

實例演練。一個 84 MHz 的計時器；我們想要 1 kHz 的更新（一個漂亮的 1 ms 計時刻）。我們需要的乘積是

$$ (\text{PSC}+1)(\text{ARR}+1) = \frac{84\times10^6}{1\times10^3} = 84\,000 $$

現在把 84 000 分解成兩個整數，每個都 $\le 65536$（因為這些是 16 位元暫存器）。一個俐落的拆法是 $84 \times 1000$，得到 PSC = 83 和 ARR = 999。驗算一下：$84 \times 1000 = 84\,000$，且 $84\times10^6 / 84\,000 = 1000\,\text{Hz}$。搞定。

🛠️ 聰明地選擇拆法

許多 (PSC, ARR) 組合都能命中相同的 $f_{\text{update}}$，但它們並不等價。對於 PWM，你會想要 ARR 大（PSC 小或為零），這樣工作週期的解析度才細——每一階工作週期的大小是 $1/(\text{ARR}+1)$。對於慢速的軟體時鐘，你會想要 PSC 大，這樣 16 位元的 ARR 仍能達到很長的週期。選那個能把解析度擺在你需要之處的因子。

3方塊內部：CNT、ARR 與 CCR 如何互動

① 直覺 資料路徑（datapath）

把計數器想像成一道上升的鋸齒波：CNT 從 0 爬升到 ARR，瞬間跳回 0，如此重複。ARR 是設定週期的上限。每一個比較暫存器 CCRx 都是穿過這道鋸齒波的一條水平線——當 CNT 跨越它的那一刻，該通道的輸出就翻轉。正是這個跨越，把一個計數器變成了 PWM 產生器。

匯流排時脈被預除器分頻後餵給 CNT。CNT 爬升到 ARR 再繞回（即更新事件）。一路上，有一個比較器拿 CNT 跟每一個 CCRx 比較；跨越的那一刻就驅動輸出腳位。ARR 設定週期；CCR 設定工作週期。兩個暫存器，就是整個故事。

4PWM：把計數器變成近似類比的輸出

① 直覺 取平均，而非單純切換

在 PWM 模式 1 下，通道輸出在 $\text{CNT} < \text{CCRx}$ 時為高電位，之後則為低電位，直到計數器繞回為止。所以在每個週期裡，腳位在總共 ARR+1 個計數刻中有 CCRx 個刻為高電位。用它去驅動馬達或 LED，負載會對平均電壓做出反應——一個 25% 工作週期的快速方波，看起來就像供應電壓的 25%。

② 數學 工作週期與頻率

有兩條公式主宰 PWM，而它們都直接從 §2 推導出來。工作週期（duty cycle）是高電位時間佔週期的比例：

$$ D = \frac{\text{CCR}}{\text{ARR}+1} $$

而 PWM 頻率就是更新頻率（計數器每繞回一次就是一個 PWM 週期）：

$$ f_{\text{PWM}} = \frac{f_{\text{tim}}}{(\text{PSC}+1)(\text{ARR}+1)} $$

實例演練——20 kHz 的馬達 PWM。20 kHz 是馬達驅動的甜蜜點：高於可聽見的嘶嘶聲，又低於切換損耗（switching-loss）造成困擾的頻率。當 $f_{\text{tim}} = 84\,\text{MHz}$ 且 PSC = 0（不分頻，以取得最大的工作週期解析度）：

$$ \text{ARR}+1 = \frac{f_{\text{tim}}}{f_{\text{PWM}}} = \frac{84\times10^6}{20\times10^3} = 4200 \;\Rightarrow\; \text{ARR}=4199 $$

這給了 4200 階工作週期——約 0.024% 的解析度，綽綽有餘。若要 25% 工作週期：

$$ \text{CCR} = D\,(\text{ARR}+1) = 0.25 \times 4200 = 1050 $$

所以 PSC=0, ARR=4199, CCR=1050 就產生一個 20 kHz、25% 工作週期的波形。想要 50%？寫入 CCR=2100。妙處在於：一旦計時器設定完成，要改變亮度或馬達轉速，就只是一次暫存器寫入，之後完全沒有 CPU 開銷。

③ 程式碼 逐行閱讀

以下是 STM32 HAL 的版本——TIM3 通道 1，20 kHz，起始為 25% 工作週期。假設時脈樹給了 TIM3 一個 84 MHz 的計時器時脈。

pwm.c C / STM32 HAL
TIM_HandleTypeDef htim3;

void PWM_Init(void)
{
  __HAL_RCC_TIM3_CLK_ENABLE();                 // 為計時器供電（開啟時脈）

  htim3.Instance           = TIM3;
  htim3.Init.Prescaler     = 0;                // PSC = 0  -> 分頻倍數為 1
  htim3.Init.CounterMode   = TIM_COUNTERMODE_UP;
  htim3.Init.Period        = 4199;             // ARR = 4199 -> 84MHz/4200 = 20 kHz
  htim3.Init.ClockDivision = TIM_CLOCKDIVISION_DIV1;
  HAL_TIM_PWM_Init(&htim3);

  TIM_OC_InitTypeDef sConfig = {0};
  sConfig.OCMode     = TIM_OCMODE_PWM1;        // CNT < CCR 時為高電位
  sConfig.Pulse      = 1050;                   // CCR = 1050 -> 25% 工作週期
  sConfig.OCPolarity = TIM_OCPOLARITY_HIGH;
  HAL_TIM_PWM_ConfigChannel(&htim3, &sConfig, TIM_CHANNEL_1);

  HAL_TIM_PWM_Start(&htim3, TIM_CHANNEL_1);    // 開始驅動腳位
}

// 之後，零重算就能即時改變工作週期：
void PWM_SetDutyPercent(uint8_t pct)           // 0..100
{
  uint32_t arr = __HAL_TIM_GET_AUTORELOAD(&htim3);   // 4199
  uint32_t ccr = ((arr + 1U) * pct) / 100U;          // CCR = D*(ARR+1)
  __HAL_TIM_SET_COMPARE(&htim3, TIM_CHANNEL_1, ccr);
}
💡 每個欄位對應到什麼

Init.Prescaler 就是 PSC，Init.Period 就是 ARR，sConfig.Pulse 就是 CCR。HAL 的命名把暫存器藏了起來，但數學是完全一樣的。__HAL_TIM_SET_COMPARE() 只不過是對 TIM3->CCR1 的一次寫入——瞬間完成、無毛刺（glitch-free），而且因為這些暫存器有預載／緩衝（preload/buffer），會在下一次重載時乾淨地更新。

如果你偏好直接操作硬體，同樣的設定改用裸暫存器寫法，能讓對應關係一目了然：

pwm_regs.c C / STM32（暫存器）
RCC->APB1ENR  |= RCC_APB1ENR_TIM3EN;          // 開啟計時器時脈
TIM3->PSC      = 0;                            // 預除器    = PSC
TIM3->ARR      = 4199;                         // 自動重載  = ARR（週期-1）
TIM3->CCR1     = 1050;                         // 比較值    = CCR（25% 工作週期）
TIM3->CCMR1   |= (6 << TIM_CCMR1_OC1M_Pos);    // OC1M = 110 -> PWM 模式 1
TIM3->CCMR1   |= TIM_CCMR1_OC1PE;              // 預載 CCR1（無毛刺更新）
TIM3->CCER    |= TIM_CCER_CC1E;                // 啟用通道 1 輸出
TIM3->CR1     |= TIM_CR1_ARPE;                 // ARR 也啟用預載
TIM3->EGR     |= TIM_EGR_UG;                   // 強制一次更新以載入影子暫存器
TIM3->CR1     |= TIM_CR1_CEN;                  // CEN = 1 -> 計數器開始運轉
🏭 真實案例——同一個周邊上的伺服機、馬達與 LED

同樣一個計時器，只要重新調校，就能驅動三種經典產品。一個玩具伺服機（hobby servo）需要 50 Hz 的訊框，搭配 1–2 ms 的高電位脈衝：設定 PSC=83, ARR=19999（1 MHz 計時刻 → 20 ms 週期），再用 CCR=1500 對應中位。一個有刷馬達 H 橋（H-bridge）需要 20 kHz（即上面的例子），這樣它才不會發出嘯叫。LED 調光需要 ≥1 kHz（讓眼睛察覺不到閃爍），並搭配較大的 ARR 以達成平滑的漸變。一個 IP 方塊，三個產業——這就是精通它為什麼划算的原因。

5輸入捕捉：量測外部世界

① 直覺 在邊緣到來時鎖存計數值

PWM 是計時器在說話。輸入捕捉（input capture）則是計時器在聆聽。你把一個外部訊號接到某個通道腳位，並告訴硬體：「每當有上升邊緣時，就把當下的 CNT 拍快照存進 CCRx。」CPU 完全不參與計時——當邊緣到來的那一瞬間，矽晶片就鎖存了計數值，所以你能以近乎零抖動（jitter）地量測週期、脈衝寬度或頻率。

② 數學 從計數刻到秒再到 Hz

把連續兩次捕捉值相減，就得到以計時刻為單位的週期。每個計時刻歷時 $1/f_{\text{CNT}}$ 秒，因此所量測到的輸入訊號頻率為

$$ f_{\text{signal}} = \frac{f_{\text{CNT}}}{\Delta\text{CNT}} = \frac{f_{\text{tim}}/(\text{PSC}+1)}{\text{CCR}_n - \text{CCR}_{n-1}} $$

注意繞回問題：如果 $\text{CCR}_n < \text{CCR}_{n-1}$，表示計數器在兩個邊緣之間溢位了，所以在相減之前要先加上 $\text{ARR}+1$。實例演練：當計時刻為 1 MHz（每刻 1 µs）、捕捉到的差值為 2000 刻時，輸入頻率為 $f = 10^6 / 2000 = 500\,\text{Hz}$——對應 2 ms 週期，直接從硬體讀出。

③ 程式碼 在 ISR 中捕捉
capture.c C / STM32 HAL
// TIM2 CH1 設為輸入捕捉，上升邊緣。計時刻 = 1 us（84 MHz 上 PSC = 83）。
volatile uint32_t g_period_ticks = 0;
static   uint32_t last = 0;

void HAL_TIM_IC_CaptureCallback(TIM_HandleTypeDef *htim)
{
  if (htim->Channel == HAL_TIM_ACTIVE_CHANNEL_1) {
    uint32_t now = HAL_TIM_ReadCapturedValue(htim, TIM_CHANNEL_1);
    uint32_t arr = __HAL_TIM_GET_AUTORELOAD(htim);
    // 防繞回的差值：若計數器翻轉過，就加上一整個完整週期
    g_period_ticks = (now >= last) ? (now - last)
                                   : (now + (arr + 1U) - last);
    last = now;
  }
}

// f_signal = 1e6 / g_period_ticks（Hz），因為這裡每個計時刻為 1 us。
⚠️ 陷阱——慢速訊號上漏掉的溢位

上面的繞回修正只處理兩個邊緣之間的單一次溢位。如果輸入訊號比一整個計數器週期還要慢，計數器就會繞回不只一次，天真的差值計算就會出錯。對於低頻訊號，要嘛把預除器加大（計時刻更慢、量測範圍更長），要嘛去計數更新事件中斷的次數並把它折算進來。永遠要清楚你的可量測範圍：$f_{\min} = f_{\text{CNT}}/(\text{ARR}+1)$。

6編碼器模式：硬體中免費的正交解碼

① 直覺 計數器變成了位置

一個正交編碼器（quadrature encoder）輸出兩道方波，A 與 B，相位相差 90°。相位的先後順序告訴你方向：A 領先 B 是正轉，B 領先 A 是反轉。在編碼器模式下，計時器把兩個通道接進計數器，使得 CNT 在正轉邊緣遞增、在反轉邊緣遞減——全自動完成。設定完成後，CNT 本身就是輪子的位置，由硬體更新，不需中斷，即使在高轉速（RPM）下也不會漏掉任何一個計數。

② 數學 計數值、轉速與那個 ×4

在常見的 ×4 模式下，計時器對兩個通道的每一個邊緣都計數，所以一個每轉 N 條線（lines per revolution）的編碼器，每轉會產生 $4N$ 個計數。在一個已知的時間區間 $\Delta t$ 內讀取計數差值，就能算出轉速：

$$ \text{RPM} = \frac{\Delta\text{CNT}}{4N} \cdot \frac{60}{\Delta t} $$

實例演練：一個 500 線的編碼器（所以在 ×4 模式下每轉 2000 計數）在一個 10 ms 的時窗內前進了 200 個計數。那麼 $\text{RPM} = \frac{200}{2000}\cdot\frac{60}{0.01} = 0.1 \cdot 6000 = 600\,\text{RPM}$。所有的邊緣追蹤都是計數器做的；你的迴圈只要讀 CNT 再相減即可。

③ 程式碼 用一個暫存器讀取位置
encoder.c C / STM32 HAL
TIM_HandleTypeDef htim4;

void Encoder_Init(void)                         // TIM4 CH1=A, CH2=B
{
  TIM_Encoder_InitTypeDef enc = {0};
  htim4.Instance        = TIM4;
  htim4.Init.Prescaler  = 0;                    // 對每個邊緣計數，不分頻
  htim4.Init.Period     = 0xFFFF;               // 16 位元繞回（自由運轉）
  htim4.Init.CounterMode = TIM_COUNTERMODE_UP;
  enc.EncoderMode = TIM_ENCODERMODE_TI12;       // x4：對 A 與 B 的邊緣都計數
  enc.IC1Polarity = TIM_ICPOLARITY_RISING;
  enc.IC2Polarity = TIM_ICPOLARITY_RISING;
  HAL_TIM_Encoder_Init(&htim4, &enc);
  HAL_TIM_Encoder_Start(&htim4, TIM_CHANNEL_ALL);
}

// 在你的控制迴圈裡，位置就是計數器本身：
int16_t Encoder_Read(void)
{
  return (int16_t)__HAL_TIM_GET_COUNTER(&htim4); // 有號型別 -> 可處理反轉
}
🏭 真實案例——閉迴路馬達控制

這是每一台伺服驅動器、機器人關節與 CNC 軸的骨幹。一個計時器以 PWM 模式命令馬達；第二個計時器以編碼器模式讀取它的位置——兩者都在硬體中、都不需中斷。一個週期性的控制迴圈（通常是第三個計時器，例如以 1 kHz 觸發 ISR）讀取編碼器差值、計算 PID 修正量，然後寫入一個新的 CCR。三個計時器，一套完整的運動控制系統，而 CPU 大半時間都閒著。

🔬 即時模擬：由 ARR 與 CCR 決定的 PWM（84 MHz 計時器時脈，PSC = 0）

拖動 ARR 與 CCR：觀察波形，以及即時的 fPWM = 84 MHz/(ARR+1) 與工作週期 = CCR/(ARR+1)。CCR 會被夾限（clamp）到 ARR+1（即 100% 工作週期）。

  • 40% 工作週期；寫入 CCR = 400
  • 25% 工作週期；改變預除器 PSC
工作週期為 $D = \text{CCR}/(\text{ARR}+1) = 250/1000 = 25\%$。要達到 75%，你改的是工作週期而非週期，所以寫入 $\text{CCR} = 0.75 \times 1000 = 750$。改變 ARR 或 PSC 會改變的是頻率，而不是工作週期。
Q2 算算看
一個計時器以 16 MHz 計時，PSC = 15 且 ARR = 999。更新頻率 $f_{\text{update}}$ 是多少 Hz？（回想 $f_{\text{update}} = \dfrac{f_{\text{tim}}}{(\text{PSC}+1)(\text{ARR}+1)}$。）
Hz
$(\text{PSC}+1)(\text{ARR}+1) = 16 \times 1000 = 16\,000$，所以 $f = \dfrac{16\times10^6}{16\,000} = 1000\,\text{Hz}$——一個漂亮的 1 ms 計時刻。這就是經典的「在 16 MHz 計時器上給我一個 1 kHz 控制迴圈中斷」的配方。
Q3 程式設計挑戰——驅動一個玩具伺服機
在一個 84 MHz 的計時器上設定 TIM3 CH1，產生一個標準的 50 Hz 伺服機訊框（20 ms 週期），搭配 1.5 ms 的中位脈衝。為了方便的 1 µs 計時刻，選出 PSC／ARR，接著算出對應中位的 CCR。寫出 HAL 的初始化。
提示
目標是 1 µs 的計時刻：$f_{\text{CNT}} = 84\,\text{MHz}/(\text{PSC}+1) = 1\,\text{MHz} \Rightarrow \text{PSC} = 83$。接著 20 ms 的週期等於 20 000 個計時刻，所以 $\text{ARR} = 19999$。1.5 ms 的脈衝是 1500 個計時刻，所以 $\text{CCR} = 1500$。（1 ms → 1000，2 ms → 2000：即伺服機的完整行程範圍。）
顯示一種解答
servo.c C / STM32 HAL
// 84 MHz 計時器 -> PSC=83 給出 1 us 的計時刻（1 MHz）。
// 20 ms 訊框 -> ARR=19999。1.5 ms 中位 -> CCR=1500。
TIM_HandleTypeDef htim3;

void Servo_Init(void)
{
  __HAL_RCC_TIM3_CLK_ENABLE();

  htim3.Instance         = TIM3;
  htim3.Init.Prescaler   = 83;                 // 84MHz/84 = 1 MHz  (1 us/計時刻)
  htim3.Init.CounterMode = TIM_COUNTERMODE_UP;
  htim3.Init.Period      = 19999;              // 20000 計時刻 = 20 ms = 50 Hz
  HAL_TIM_PWM_Init(&htim3);

  TIM_OC_InitTypeDef oc = {0};
  oc.OCMode     = TIM_OCMODE_PWM1;
  oc.Pulse      = 1500;                         // 1.5 ms -> 中位
  oc.OCPolarity = TIM_OCPOLARITY_HIGH;
  HAL_TIM_PWM_ConfigChannel(&htim3, &oc, TIM_CHANNEL_1);
  HAL_TIM_PWM_Start(&htim3, TIM_CHANNEL_1);
}

// 瞄準伺服機：角度 0..180 度 -> 脈衝 1000..2000 us。
void Servo_SetAngle(uint8_t deg)               // 0..180
{
  uint32_t us = 1000U + ((uint32_t)deg * 1000U) / 180U;  // 1000..2000
  __HAL_TIM_SET_COMPARE(&htim3, TIM_CHANNEL_1, us);       // CCR 以 1-us 計時刻為單位
}

1 µs 計時刻就是訣竅所在：它讓 CCR 直接以微秒為單位讀取，所以 1500 µs 的脈衝就字面上是 CCR=1500。選擇預除器以得到一個對人類友善的計時刻，正是乾淨計時器程式碼的標誌。

🎓 本章回顧

計時器就是一個計數器：CNT 以 $f_{\text{tim}}/(\text{PSC}+1)$ 的速率計數，每 $\text{ARR}+1$ 個計時刻繞回一次，而更新頻率是 $f_{\text{tim}}/((\text{PSC}+1)(\text{ARR}+1))$——記得 APB ×2 計時器時脈規則。在 PWM 中，ARR 設定頻率、CCR 設定工作週期（$D=\text{CCR}/(\text{ARR}+1)$）；84 MHz → 20 kHz 對應 ARR=4199，而 25% 工作週期是 CCR=1050。輸入捕捉在邊緣到來時鎖存 CNT，以防繞回的方式量測週期與頻率。編碼器模式把 CNT 變成一個硬體位置計數器，用於閉迴路馬達控制。一個周邊，三項超能力——而 CPU 幾乎不費吹灰之力。下一章我們會把這些計時器接到 ADC 與 DMA，做出完全在硬體中運行的取樣。
