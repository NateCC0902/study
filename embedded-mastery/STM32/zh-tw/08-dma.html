DMA 深入剖析：通道、循環模式與雙緩衝

DMA 是 STM32 不再只是「跑得比較快的 Arduino」、而開始像是另一個等級機器的關鍵時刻。一具獨立的引擎在週邊與記憶體之間搬運位元組，而 CPU 此時正在運算——或在睡眠。精通它，你就能解鎖無縫銜接的音訊、megabaud 等級的紀錄輸出，以及永不閃爍的 LED 燈條。我們會從匯流排往上建立模型、做吞吐量的數學運算，然後實際接出一個循環 DMA 的 ADC 擷取。

⏱ 約 50 分鐘🧠 匯流排仲裁⚡ 循環與雙緩衝🔧 ADC · UART · SPI

1DMA 究竟是什麼？

① 直覺 先建立整體畫面

沒有 DMA 時，週邊與記憶體之間搬移的每一個位元組都要經過 CPU。ADC 完成一次轉換、拉起一個旗標、CPU 醒來、把 ADC1->DR 讀進暫存器、再把暫存器寫進 RAM 陣列、把指標遞增、檢查計數器、迴圈。光是一個 16 位元樣本就是十幾條指令。在每秒一百萬次取樣（1 mega-sample）的速率下，CPU 除了搬運位元組之外什麼都不做——根本沒有剩餘的週期可以真正拿來運算。

💡 一句話模型

DMA（Direct Memory Access，直接記憶體存取）是第二個資料搬運者——它是匯流排上一具小巧的專用引擎，在週邊與記憶體之間複製位元組，而且完全不需要 CPU 碰到任何一個位元組。你只需要程式化設定一次「從這裡、到那裡、搬多少個、每個多寬」，它就會在背景執行，而核心則去運算或睡眠。

DMA 控制器是一個匯流排主控（bus master），就跟 CPU 一模一樣。它向 AHB/APB 匯流排仲裁取得使用權、執行讀取與寫入、遞增位址、遞減計數器，而且只在真正值得通知你的時候才拉起中斷——通常是「緩衝區半滿」或「緩衝區滿」，而不是「一個位元組到了」。這種從每位元組的 CPU 工作量壓縮成每緩衝區的工作量，正是 DMA 的全部重點所在。

🔑 關鍵區別

輪詢（polling）與中斷（interrupt）兩者的成本都是每筆資料項目耗費的 CPU 時間。DMA 的成本則是每個傳輸區塊耗費的 CPU 時間。週邊的「資料就緒」訊號會直接以硬體請求送進 DMA 引擎——核心從頭到尾都不在這個迴圈裡。這就是為什麼 DMA 的吞吐量受限於匯流排，而不是受限於你的中斷延遲。

2控制器架構：串流（stream）、通道（channel）與請求對應

① 直覺 矽晶片是怎麼接線的

硬體會因系列而異，而這套術語每個人總會被絆倒一次。在 STM32F4（Cortex-M4）上有兩個 DMA 控制器，DMA1 與 DMA2。每個控制器有 8 個串流（stream）。每個串流可以連接到 8 個通道（channel）其中之一，而通道則選擇哪一個週邊來驅動該串流的請求線。所以串流是實體的傳輸引擎；通道則是一個多工器，用來選擇它的觸發來源。

🔑 命名陷阱（F4 vs F1）

在 F4 上：串流是引擎，而你挑選一個通道（0–7）把週邊請求繞接到它上面。在較舊的 STM32F1（Cortex-M3）上則沒有串流——DMA1 有 7 個通道、DMA2 有 5 個，而且每個通道本身就是引擎，且週邊對應是固定的。同樣的概念，不同的用詞。當教學文件說「DMA channel」時，要先確認它指的是哪個系列。

請求對應在矽晶片中是固定的，並以表格形式印在參考手冊裡。你不能把任意週邊接到任意串流——舉例來說，在 F4 上，ADC1 是 DMA2、Stream 0、Channel 0（也可以用 Stream 4、Channel 0）。SPI1_TX 是 DMA2、Stream 3、Channel 3。挑錯了串流／通道組合，週邊的請求就根本到不了引擎，於是什麼都不會動。這就是最常見的「我的 DMA 完全沒反應」這個 bug。

⚠️ 陷阱——串流衝突

兩個共用同一串流的週邊不能同時跑 DMA，就算它們用的是不同的通道也不行——因為每個串流只有一具引擎。如果你需要 ADC1 和 SPI1_TX 都跑在 DMA2 上，就選用它們的替代串流對應，讓它們落在不同的串流上。在敲定接腳／週邊規劃之前，務必先對照請求表交叉檢查。

3一次傳輸的解剖：六個參數

① 直覺 你實際上要設定什麼

每一次 DMA 傳輸都由一小組參數定義。把這些設對，引擎就會完全照你的意思做；設錯一個，你就會破壞 RAM 或讀到垃圾資料。這裡是整套詞彙：

參數暫存器欄位意義

週邊位址DMA_SxPAR來源／目的週邊資料暫存器，例如 &ADC1->DR
記憶體位址DMA_SxM0AR你的 RAM 緩衝區基底指標

計數（Count）DMA_SxNDTR要搬移的資料項目數量（不是位元組數）
⚠️ 陷阱——NDTR 算的是項目，不是位元組

NDTR 是傳輸的數量，以來源寬度為單位。如果 PSIZE = 半字、而你想擷取 1024 個 ADC 樣本，請設 NDTR = 1024，不是 2048。引擎會搬移 NDTR 個項目然後停止（normal 模式）或重新載入（circular 模式）。這裡差了兩倍是經典的緩衝區溢位錯誤。

🛠️ 遞增才是魔法所在

週邊資料暫存器位於單一固定位址——也就是它的 DR——所以 PINC = 0。你的 RAM 緩衝區是一個陣列，所以 MINC = 1，引擎就會自動走訪 buf[0]、buf[1]、buf[2] …。這個單一的不對稱性——週邊固定、記憶體遞增——正是把一連串 DR 讀取轉化成一個被填滿的陣列、而且 CPU 完全不費力的關鍵。

4Normal 模式 vs Circular 模式

① 直覺 一次性 vs 永遠跑

在 normal 模式下，引擎搬移 NDTR 個項目、拉起「傳輸完成（transfer-complete）」中斷，然後停止。要再跑一次，你必須重新武裝它：清除旗標、重新載入 NDTR、再次啟用串流。這對一次性的區塊讀取沒問題；但對連續串流就毫無用處，因為你重新武裝的這段空檔意味著樣本會被丟掉。

在 circular 模式（設定 DMA_SxCR.CIRC = 1）下，當計數器歸零時，硬體會自動重新載入 NDTR 與記憶體指標回到起點，並繼續跑下去。緩衝區變成一個環（ring）：ADC 寫入 buf[0..N-1]、繞回 buf[0]，永遠如此，完全不需要 CPU 介入。這就是 ADC 或音訊編解碼器能無限期串流的方式。

🔑 何時該動用 circular 模式

只要資料是以固定速率連續且無限期地抵達，就用 circular 模式——ADC 取樣、I²S／音訊、週期性 DAC 波形輸出、連續 UART 接收。對於有限、按需的傳輸則用 normal 模式——從快閃晶片讀取 256 個位元組、送出一個遙測封包。準則就是：連續串流→circular；有界限的工作→normal。

5雙緩衝與半傳輸技巧

① 直覺 填一個、處理另一個

circular 模式解決了「永不停止」，但它製造出一個新的隱患：如果引擎正在寫入的緩衝區，正是你正在讀取的那一個，你就會遇到競爭（race）——一半是新資料、一半是舊資料。修正的方法是在時間上把工作切開，讓生產者（DMA）與消費者（CPU）永遠不會同時碰到同一塊區域。

最省成本的版本利用單一 circular 緩衝區的半傳輸（HT，half-transfer）與傳輸完成（TC，transfer-complete）中斷。當引擎越過中點時 HT 中斷觸發，意思是前半部已滿、可以安全處理，而引擎正在填後半部。當繞回時 TC 中斷觸發，意思是後半部已滿，引擎現在正在重新填前半部。你永遠處理引擎沒在寫入的那一半。

💡 乒乓（ping-pong）的畫面

想像水龍頭下面有兩個水桶，水龍頭輪流去裝。當 A 桶在裝水時，你去倒 B 桶；A 一裝滿的瞬間，水龍頭就擺向 B，而你去倒 A。沒有人會去伸手進當下正在水龍頭底下的那個桶。無縫銜接（gapless）——水龍頭永遠不必停。STM32 甚至提供真正的雙緩衝模式（DMA_SxCR.DBM = 1），有兩個不同的位址 M0AR 與 M1AR，引擎會自動切換它們，並透過 CT（current target，當前目標）位元回報。

⚠️ 陷阱——跟不上速度

雙緩衝只有在你趕在引擎填滿另一半之前處理完一半時才有效。如果你的處理常式跑得比半緩衝填滿的時間還久，引擎就會超前繞過你、覆寫掉尚未處理的資料——這是一種無聲的溢位（overrun）。修正方法是用更大的緩衝區（每一半有更多時間）或更快的處理常式。我們會在下一節計算這個填充時間的預算。

6吞吐量數學：bytes/s 與緩衝區填滿時間

② 數學 數字是怎麼來的

調整 DMA 緩衝區大小是純算術，而算對與否決定了你得到的是無縫串流、還是莫名其妙的喀嗒聲。一個串流產生的原始資料速率是三個東西的乘積——多頻繁、多少並行通道、以及每個樣本多寬：

$$ R = f_s \cdot N_{ch} \cdot W \quad\left[\tfrac{\text{bytes}}{\text{s}}\right] $$

其中 $f_s$ 是取樣率、$N_{ch}$ 是通道數、$W$ 是每個樣本的位元組數。以 CD 品質的立體聲音訊為例：$f_s = 48\,\text{kHz}$、$N_{ch} = 2$（左 + 右）、$W = 2$ 位元組（16 位元）：

$$ R = 48{,}000 \times 2 \times 2 = 192{,}000 \;\tfrac{\text{B}}{\text{s}} \approx 187.5\,\text{KiB/s} $$

那就是 DMA 引擎必須永遠吸收的位元組串流。現在來看緩衝區填滿時間。如果你的 circular 緩衝區容納 $B$ 位元組，把它完全填滿的時間就只是大小除以速率：

$$ t_{\text{fill}} = \frac{B}{R}, \qquad t_{\text{half}} = \frac{B}{2R} $$

對於一個 1024 樣本的立體聲緩衝區，$B = 1024 \times 2 \times 2 = 4096$ 位元組，所以

$$ t_{\text{fill}} = \frac{4096}{192{,}000} \approx 21.3\,\text{ms}, \qquad t_{\text{half}} \approx 10.7\,\text{ms} $$

那個 $t_{\text{half}} \approx 10.7\,\text{ms}$ 就是你的即時預算：每一次 HT/TC 中斷，你的處理常式有約 10.7 ms 來處理 512 個樣本，在引擎超前繞過它之前。錯過這個預算，你就會溢位。緩衝區越大→餘裕越多，但延遲也越大；永遠是這個取捨。再換一個說法——對於單一單聲道通道，每個樣本之間的時間就只是週期 $T_s = 1/f_s$，而引擎必須在這個時間內好好地（well inside）完成一次 DR→RAM 傳輸（少數幾個匯流排週期）。

🔬 即時模擬：DMA 緩衝區填滿預算

拖動取樣率、通道數、寬度與緩衝區大小：觀看位元組速率 R、完整填滿時間，以及你每一半的處理預算即時重新計算。把速率往上推或把緩衝區往下調，看著預算朝零縮小。

7匯流排仲裁與 FIFO

① 直覺 共用匯流排

DMA 引擎與 CPU 兩者都是匯流排主控，競爭同一個 AHB 矩陣。每個週期只有一個主控能驅動某個特定的匯流排從屬（slave），所以一個匯流排仲裁器（bus arbiter）來授予存取權。在 Cortex-M3/M4 的 STM32 上，這個矩陣是多層的（multi-layer），所以 DMA-到-SRAM 與 CPU-到-Flash 通常可以並行進行——但當它們瞄準同一個從屬（兩者都想要 SRAM）時，仲裁器會交錯處理它們，輸的那一方會停一個週期。DMA 通常會以優先權勝出，所以繁重的 DMA 流量會輕微拖慢 CPU。這就是「免費」傳輸的代價——它很便宜，但不是字面上的免費。

F4 的串流加上了一個 FIFO：一個位於週邊側與記憶體側之間的小型（4×32 位元）緩衝區。它的價值體現在兩個方面。第一，突發與打包（burst & packing）：它讓引擎可以收集數個窄的週邊讀取，再以一次寬的突發寫入記憶體，減少匯流排交易的數量。第二，寬度轉換：開啟 FIFO 後，你可以從週邊讀取位元組，再把它們打包成 32 位元的記憶體字（word）。FIFO 停用時（direct mode，直接模式），PSIZE 與 MSIZE 必須相符，而且每一項都是立即的單次傳輸。

🏭 真實案例——FIFO 為何對 SD 卡很重要

像 SDIO/SDMMC 介接 SD 卡這類高頻寬週邊，要求使用帶突發的 FIFO 模式。SD 時脈不會為了匯流排競爭而暫停，所以 DMA 必須使用突發傳輸才能跟上控制器的硬體 FIFO；直接模式根本無法維持這個速率，於是你會得到欠載（underrun）錯誤。FIFO 閾值（1/4、1/2、3/4、滿）可以調整 FIFO 在突發之前要裝多滿——這是你在追求吞吐量時會去轉的一個真正旋鈕。

8真實案例：ADC、UART、WS2812 與記憶體對記憶體

③ 程式碼 每一行都要讀

這裡是最招牌的用法：一個自由運行的 ADC 直接串流進一個 circular 緩衝區。CPU 設定它一次，之後就完全不參與樣本的搬移。為了清楚起見這是暫存器層級的寫法；HAL 包裝的也是同樣這些欄位。

adc_dma_circular.c C / STM32F4（暫存器層級）
// ADC1 -> SRAM，circular 模式，半字樣本。
// F4 對應：ADC1 = DMA2, Stream 0, Channel 0。
#define NSAMP 1024
volatile uint16_t adc_buf[NSAMP];   // 16 位元 ADC 結果

void adc_dma_init(void) {
    RCC->AHB1ENR |= RCC_AHB1ENR_DMA2EN;   // 開啟 DMA2 控制器時脈

    DMA2_Stream0->CR &= ~DMA_SxCR_EN;     // 設定前先停用
    while (DMA2_Stream0->CR & DMA_SxCR_EN) { }  // 等到真正關閉

    DMA2_Stream0->PAR  = (uint32_t)&ADC1->DR;     // 來源：ADC 資料暫存器（固定）
    DMA2_Stream0->M0AR = (uint32_t)adc_buf;       // 目的：RAM 緩衝區基底
    DMA2_Stream0->NDTR = NSAMP;                   // 項目數（不是位元組數！）

    DMA2_Stream0->CR =
          (0u << DMA_SxCR_CHSEL_Pos)   // Channel 0 選擇 ADC1 請求
        | (1u << DMA_SxCR_PSIZE_Pos)   // 週邊大小 = 半字（16 位元）
        | (1u << DMA_SxCR_MSIZE_Pos)   // 記憶體大小 = 半字
        | DMA_SxCR_MINC               // 遞增記憶體位址
        | DMA_SxCR_CIRC               // CIRCULAR：永遠自動重載
        | DMA_SxCR_HTIE               // 啟用半傳輸中斷
        | DMA_SxCR_TCIE;              // 啟用傳輸完成中斷
        // DIR = 00（週邊到記憶體），PINC = 0（週邊固定）

    NVIC_EnableIRQ(DMA2_Stream0_IRQn);
    DMA2_Stream0->CR |= DMA_SxCR_EN;   // 武裝引擎

    ADC1->CR2 |= ADC_CR2_DMA | ADC_CR2_DDS;  // ADC 發出 DMA 請求，持續請求
    ADC1->CR2 |= ADC_CR2_CONT | ADC_CR2_ADON;  // 連續 + 開啟
    ADC1->CR2 |= ADC_CR2_SWSTART;              // 啟動轉換
}
💡 剛剛發生了什麼

在 adc_dma_init() 回傳之後，ADC 自行轉換、在每個結果之後拉起一個 DMA 請求，而引擎就把那個 16 位元值放進 adc_buf[] 的下一個槽位——因為設了 CIRC，所以會永遠繞回。CPU 唯一剩下的工作就是下面這個 IRQ 處理常式，每個緩衝區會被呼叫兩次。注意 DDS（DMA Disable Selection，DMA 停用選擇）：它讓 ADC 在最後一次傳輸之後仍持續發出請求，這樣 circular 模式才不會卡住。

dma2_stream0_irq.c C / STM32F4
void DMA2_Stream0_IRQHandler(void) {
    if (DMA2->LISR & DMA_LISR_HTIF0) {        // 半傳輸：前半部就緒
        DMA2->LIFCR = DMA_LIFCR_CHTIF0;       // 清除旗標（寫 1 清除）
        process_block(&adc_buf[0], NSAMP / 2);   // 安全：引擎正在填後半部
    }
    if (DMA2->LISR & DMA_LISR_TCIF0) {        // 傳輸完成：後半部就緒
        DMA2->LIFCR = DMA_LIFCR_CTCIF0;       // 清除旗標
        process_block(&adc_buf[NSAMP / 2], NSAMP / 2);  // 安全：引擎正在重填前半部
    }
}
⚠️ 陷阱——忘記清除旗標

DMA 狀態旗標是透過專用的 LIFCR/HIFCR 暫存器以「寫 1 清除（write-1-to-clear）」方式清除的——你不能藉由寫入狀態暫存器來清除它們。忘了清除，你的處理常式就會立刻重新觸發，把 CPU 卡進活鎖（livelock）。另外注意 F4 把 8 個串流分散在兩對暫存器中：串流 0–3 使用 Low 暫存器（LISR/LIFCR），串流 4–7 使用 High 的那組（HISR/HIFCR）。

🏭 真實案例——UART 紀錄串流、WS2812 與記憶體對記憶體

UART TX（記憶體→週邊）：把 DMA 指向你的紀錄字串、設 DIR = 記憶體到週邊、MINC = 1、PINC = 0、NDTR = 長度、啟用 USART->CR3.DMAT。整個緩衝區會以線路速率串流出去，而 main() 繼續往前跑——不會有阻塞的 printf。
WS2812 LED 燈條（記憶體→週邊，透過 SPI／計時器）：這些「NeoPixel」燈條需要精確的約 800 kHz 單線波形。把每個顏色位元編碼成一個 SPI 位元組（或一個計時器 PWM 工作週期），然後用 DMA 把整個編碼好的影格送到 SPI->DR。DMA 精準的時序正是讓燈條無瑕疵（glitch-free）的原因——若從 CPU 進行軟體位元操作（bit-banging）會產生抖動，燈條就會閃爍。
記憶體對記憶體：設 DIR = 記憶體到記憶體（F4 上僅 DMA2 可用）、PINC 與 MINC 都 = 1，它就變成一個在 CPU 做其他工作時執行的硬體 memcpy。注意：circular 模式與 direct 模式對記憶體對記憶體不可用。

Q1 概念檢核
你正把 ADC 連續串流進一個單一的 circular 緩衝區，並在 CPU 上處理資料。半傳輸（half-transfer）與傳輸完成（transfer-complete）中斷（雙緩衝）實際上解決了什麼核心問題？
  • 它們藉由跳過匯流排仲裁器讓 DMA 引擎跑得更快
  • 它們讓你能處理緩衝區的其中一半，而引擎安全地填另一半，避免對同一份資料發生讀／寫競爭
  • 它們把週邊的位元組資料轉換成 32 位元的記憶體字
  • 它們停下 ADC，好讓 CPU 在區塊之間趕上進度
雙緩衝講的是在時間上分開生產者與消費者。HT 中斷說「前半部滿了、後半部正在填——現在處理前半部」；TC 中斷說的則相反。你永遠碰引擎沒在寫入的那一半，所以從不發生競爭，串流也保持無縫。速度與寬度轉換是無關的工作（那些由 FIFO／突發處理）。
Q2 算算看
你以 48 kHz 擷取立體聲（2 通道）音訊，每個樣本 16 位元（2 位元組）。你的 DMA 串流必須維持的原始位元組速率是多少，以每秒位元組數計？（使用 $R = f_s \cdot N_{ch} \cdot W$。）
B/s
$R = 48{,}000 \times 2 \times 2 = 192{,}000\ \text{B/s}$（≈ 187.5 KiB/s）。那麼一個 4096 位元組的緩衝區會在 $4096/192000 \approx 21.3\,\text{ms}$ 內填滿，給你約 10.7 ms 的每一半處理預算。把取樣率或寬度加倍，位元組速率——以及對你處理常式的壓力——也會跟著加倍。
Q3 程式挑戰——帶半／全完成回呼的 circular ADC（HAL）
使用 STM32 HAL，設定 ADC1 連續取樣進一個 256 個半字的 circular DMA 緩衝區，並實作半完成（half-complete）與全完成（full-complete）回呼，讓每一個都處理正確的那一半。假設 hadc1 及其連結的 DMA 控制代碼（handle）已由 CubeMX 以 circular 模式初始化好。思考一下每個回呼擁有哪一半。
提示
用 HAL_ADC_Start_DMA(&hadc1, (uint32_t*)buf, 256) 啟動串流一次——這個計數是以項目為單位。然後覆寫那兩個 weak 回呼：HAL_ADC_ConvHalfCpltCallback 在中點觸發（前 128 個樣本就緒），而 HAL_ADC_ConvCpltCallback 在繞回時觸發（後 128 個就緒）。每一個都應該處理引擎剛剛填完的那一半——而且至關重要的是，要回得夠快以贏過填滿時間。
顯示一個解答
adc_circular_hal.c C / STM32 HAL
#define NSAMP 256
static volatile uint16_t buf[NSAMP];   // circular DMA 目標

void app_start(void) {
    // 以 circular 模式啟動 ADC -> DMA。計數以項目（ITEMS）為單位，不是位元組。
    // CubeMX 已把 DMA 控制代碼設為 Circular + 啟用 HT/TC 中斷。
    HAL_ADC_Start_DMA(&hadc1, (uint32_t *)buf, NSAMP);
}

// 當引擎越過中點時觸發：buf[0 .. 127] 已滿，
// 而引擎現在正在寫 buf[128 .. 255]。處理「前半部」。
void HAL_ADC_ConvHalfCpltCallback(ADC_HandleTypeDef *hadc) {
    if (hadc->Instance == ADC1)
        process_block((uint16_t *)&buf[0], NSAMP / 2);
}

// 在繞回時觸發：buf[128 .. 255] 已滿，而引擎已繞回
// 重新填 buf[0 .. 127]。處理「後半部」。
void HAL_ADC_ConvCpltCallback(ADC_HandleTypeDef *hadc) {
    if (hadc->Instance == ADC1)
        process_block((uint16_t *)&buf[NSAMP / 2], NSAMP / 2);
}

// 你的 DSP／拷出。必須在引擎填滿另一半之前完成。
static void process_block(uint16_t *p, uint32_t n) {
    for (uint32_t i = 0; i < n; i++) {
        // 例如：累加、濾波，或拷出到傳輸緩衝區
        (void)p[i];
    }
}

這一課的重點是所有權分割：半完成回呼擁有索引 [0, N/2)、全完成回呼擁有 [N/2, N)，而引擎永遠在寫另一半——所以兩個回呼都不會與 DMA 競爭。讓 process_block 牢牢落在每一半的填滿預算內（§6），否則你會溢位。在 M7／帶快取的零件上為了絕對安全，也要對你讀取的那一半使無效化（invalidate）D-cache。
