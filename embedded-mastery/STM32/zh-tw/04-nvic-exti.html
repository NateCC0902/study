把中斷做對：NVIC、EXTI 與優先權

輪詢（polling）會讓 CPU 每秒問「有東西了嗎？」上百萬次，白白燒掉運算資源。中斷（interrupt）則讓硬體只在真正發生事情時，才拍一下 CPU 的肩膀。這件事做錯，你的韌體就會卡死、無限重複觸發，或是餓死某個關鍵任務；做對了，你就能打造出在微安級電流下休眠、又能在微秒內甦醒的系統。

⏱ 約 45 分鐘🧠 NVIC 與向量表⚡ GPIO 上的 EXTI🔢 優先權數學

1輪詢 vs. 中斷：是誰在發問？

① 直覺 先把畫面建立起來

想像你正在等一個包裹。輪詢就是每隔十秒走到大門去看包裹到了沒——既累人又浪費，而且兩次查看之間還是可能漏掉。中斷則像門鈴：你想做什麼就做什麼，包裹一落地的瞬間，鈴聲就把你的注意力拉到門口。處理完之後，你立刻回到原本在做的事。

在程式裡，輪詢一顆按鈕看起來像這樣——一個緊湊迴圈不斷讀取腳位，幾乎所有週期都耗在「還沒、還沒」上：

polling.c C / STM32 HAL
while (1) {
  // CPU 永遠在這裡空轉，每秒讀取腳位上百萬次。
  // 其中 99.999% 的讀取結果都是「還沒被按下」。
  if (HAL_GPIO_ReadPin(GPIOA, GPIO_PIN_0) == GPIO_PIN_SET) {
    do_something();
  }
}
💡 一句話的心智模型

中斷就是一段硬體：它暫停你的主程式、執行一小段處理常式（handler），然後從原本停下的地方精確地接續執行——而你的 main() 迴圈完全不知道這件事發生過。

在 Cortex-M 上，協調這一切的晶片區塊就是 NVIC——Nested Vectored Interrupt Controller（巢狀向量中斷控制器）。它是 CPU 核心內部的一個專用區塊（不是你外接上去的周邊），它唯一的工作就是監看數十個中斷來源、判斷哪一個最緊急，並把控制權交給正確的處理常式。稱為「巢狀（Nested）」是因為更緊急的中斷可以打斷正在執行的較不緊急中斷；稱為「向量（Vectored）」是因為每個來源都直接跳到自己的處理常式位址，不需要任何軟體分派（dispatch）。

🔑 為什麼這很重要

中斷決定了一顆 CPU 是忙碌還是有用。輪詢會把你的核心釘死在 100% 負載，並禁止真正的休眠。中斷讓核心可以閒置（甚至斷電省電），並以有界、可預測的延遲做出回應。每一個反應靈敏、低功耗的嵌入式產品，都是建立在它之上的。

2向量表與例外模型

① 直覺 一本處理常式的電話簿

當中斷觸發時，CPU 需要知道該往哪裡跳。答案就放在向量表（vector table）裡：一個函式指標的陣列，停放在 Flash 的最起始處（在 STM32F4 上是位址 0x0800_0000）。每個槽位存放著某個例外或中斷對應之處理常式的位址。NVIC 只要用中斷編號去索引這張表，把它載入程式計數器（program counter）就好——不需要 switch 敘述，也沒有額外開銷。

偏移量向量它是什麼
0x00初始 SP重置時載入的堆疊頂端值
0x04Reset上電後的第一道指令
0x08NMI不可遮罩中斷（Non-maskable interrupt）
0x0CHardFault「出了大問題」的萬用攔截器
…SysTick1 kHz 的 RTOS / HAL_Delay 滴答（tick）
0x40+IRQ0…IRQn周邊中斷：EXTI、計時器、UART、DMA…

具有負例外編號的例外（Reset、NMI、HardFault、SysTick）是核心的「系統例外（system exceptions）」。所有具非負 IRQ 編號的項目——EXTI0_IRQn、TIM2_IRQn、USART2_IRQn——都是由 NVIC 管理的外部中斷。你在 startup_stm32f4xx.s 裡看到的處理常式名稱（例如 EXTI0_IRQHandler）都是弱（weak）符號：只要在你的 C 檔裡定義一個同名函式，連結器就會自動把你的版本接進向量表。

⚠️ 陷阱——名稱必須完全一致

如果你把處理常式拼成 EXTI0_IRQ_Handler（多了一個底線）而不是 EXTI0_IRQHandler，連結器不會抱怨——它只會繼續沿用預設的弱處理常式，而那通常是一個無限的 while(1) 迴圈。你的中斷觸發後跳進那個陷阱，韌體就無聲無息地凍結了。請直接從啟動檔（startup file）裡複製名稱。

3EXTI：把一個腳位邊緣變成中斷

① 直覺 從腳位到 NVIC 的接線

一個 GPIO 腳位無法直接和 NVIC 對話。介於兩者之間的是 EXTI——EXTernal Interrupt/event controller（外部中斷／事件控制器）。EXTI 有一組獨立的線路（line）；第 0 條到第 15 條是 GPIO 線路（更高的線路服務內部來源，如 RTC 鬧鐘與 USB 喚醒）。要讓按鈕上的一個下降邊緣（falling edge）產生中斷，你必須依序設定四件事：

  1. SYSCFG——把某個埠（port）繞接到線路上。SYSCFG->EXTICR[] 決定哪個埠（A、B、C…）驅動某一條 EXTI 線路編號。
  2. 邊緣選擇——為該線路設定上升觸發暫存器 EXTI->RTSR 和／或下降觸發暫存器 EXTI->FTSR。
  3. 解除遮罩（Unmask）——在中斷遮罩暫存器 EXTI->IMR 中設定該線路的位元，邊緣才會真正以中斷的形式傳播出去（若維持遮罩狀態，邊緣會被偵測到但被忽略）。
  4. 在 NVIC 中啟用——為擁有該線路的 IRQ 呼叫 NVIC_EnableIRQ()。
⚠️ 陷阱——線路共用的坑

EXTI 線路 N 由所有埠中編號為 N 的每個腳位共用。PA0、PB0、PC0… 全都對映到 EXTI0。SYSCFG->EXTICR 一次只允許一個埠擁有線路 0——你無法同時在 PA0 和 PB0 上各設一個中斷。更糟的是，線路 5–9 共用單一個 IRQ（EXTI9_5_IRQn），線路 10–15 又共用另一個（EXTI15_10_IRQn），因此單一個處理常式必須去檢查待處理（pending）暫存器，才能知道究竟是哪個腳位觸發。

以下是針對 PA0 上一顆按鈕、以下降邊緣觸發的暫存器層級設定。這正是 HAL 在背後所做的事：

exti_setup.c C / STM32F4 CMSIS
// 假設 GPIOA 與 SYSCFG 的時脈已啟用、PA0 已設為輸入。
// 1) 把 EXTI 線路 0 繞接到埠 A（EXTICR[0] 的位元 [3:0] = 0b0000 = 埠 A）
SYSCFG->EXTICR[0] &= ~SYSCFG_EXTICR1_EXTI0;     // 清除 -> 選擇埠 A

// 2) 僅在「下降」邊緣觸發（按鈕把腳位拉到 GND）
EXTI->FTSR |=  (1U << 0);   // 為線路 0 啟用下降邊緣
EXTI->RTSR &= ~(1U << 0);   // 確保上升邊緣是關閉的

// 3) 解除線路 0 的遮罩，讓邊緣成為真正的中斷請求
EXTI->IMR  |=  (1U << 0);

// 4) 在 NVIC 中啟用 EXTI0 並給它一個優先權（見 §4）
NVIC_SetPriority(EXTI0_IRQn, 5);
NVIC_EnableIRQ(EXTI0_IRQn);

4優先權：搶佔、次優先權與位元數學

② 數學 推導出層級數量

並非所有中斷都同等重要。馬達過電流故障必須被允許去打斷一個慵懶的 UART 位元組。NVIC 把每個中斷的緊急程度，編碼成每個來源各一個 8 位元的優先權欄位——但特定晶片只會實作最高的那幾個位元。STM32F4 實作了 4 個位元：

$$ \text{levels} = 2^{N_{\text{prio}}} = 2^{4} = 16 $$

所以你會得到 0 到 15 的優先權值。那條關鍵又違反直覺的規則：

🔑 數字越小 = 優先權越高

優先權 0 是最緊急的；優先權 15 是最不緊急的。每個人都會在這裡栽一次跟頭。把它想成「名次」——第 0 名（rank 0）獲勝。Reset、NMI 與 HardFault 位於固定的負優先權，因此它們的位階高於任何你能設定的中斷。

那 4 個位元會被優先權分組（priority grouping）設定切成兩個欄位：一個搶佔（群組）優先權和一個次優先權（sub-priority）。這個切分在開機時用 NVIC_SetPriorityGrouping(g) 設定一次。分組編號 $g$ 決定了低位元中有幾個會變成次優先權。在 4 個實作位元的情況下，各欄位的數量為：

$$ N_{\text{sub}} = \max\!\bigl(0,\; (g+1) - (8 - N_{\text{prio}})\bigr), \qquad N_{\text{pre}} = N_{\text{prio}} - N_{\text{sub}} $$

這條公式看起來很龜毛，是因為欄位在 8 位元暫存器中是靠左對齊（left-aligned）儲存的；STM32F4（$N_{\text{prio}}=4$）的實際結果其實就是一張你可以背起來的表：

分組常數搶佔位元數次優先權位元數搶佔層級數次優先權層級數
NVIC_PRIORITYGROUP_004116
NVIC_PRIORITYGROUP_22244
NVIC_PRIORITYGROUP_440161

這兩個欄位各司其職：

  • 搶佔優先權（Preemption priority）決定巢狀（nesting）。一個搶佔優先權較高（數字較小）的中斷可以打斷目前正在執行、搶佔優先權較低的處理常式。這就是 NVIC 裡「巢狀」的意思。
  • 次優先權（Sub-priority）只用來打破平手：如果兩個搶佔層級相同的中斷同時待處理，次優先權較小的會先執行——但它不會去搶佔一個搶佔層級相等、且已在執行中的處理常式。
💡 具體範例

假設 IRQ A 的搶佔優先權是 1，IRQ B 的搶佔優先權是 3。當 B 的處理常式正在執行時 A 觸發了——因為 $1 < 3$，A 會搶佔 B、執行到完成，然後 B 才接續執行。若改成兩者搶佔優先權都是 2 但次優先權不同，則正在執行的那個會繼續執行；另一個得等到輪到它。

HAL 把編碼與設定的步驟包進單一個呼叫，讓你永遠不必親手位移（shift）位元：

priority.c C / STM32 HAL
// 為整個系統選擇 4 個搶佔層級 + 4 個次優先權層級。
HAL_NVIC_SetPriorityGrouping(NVIC_PRIORITYGROUP_2);

// EXTI0：搶佔優先權 = 1（相當緊急），次優先權 = 0
HAL_NVIC_SetPriority(EXTI0_IRQn, /*preempt*/ 1, /*sub*/ 0);
HAL_NVIC_EnableIRQ(EXTI0_IRQn);

// 一個比較吵、較不緊急的 UART：搶佔優先權 = 3，次優先權 = 0
HAL_NVIC_SetPriority(USART2_IRQn, /*preempt*/ 3, /*sub*/ 0);
HAL_NVIC_EnableIRQ(USART2_IRQn);
// 現在按鈕邊緣會打斷正在進行的 UART ISR，但反過來不行。
⚠️ 陷阱——分組只設定一次

HAL 的 HAL_Init() 預設已經呼叫了 HAL_NVIC_SetPriorityGrouping(NVIC_PRIORITYGROUP_4)——全部 4 個位元都變成搶佔優先權，沒有任何次優先權。如果你稍後重新分組，你先前設定過的每一個優先權數字的意義都會無聲地改變。在啟動時挑定一種分組，然後就別再動它。要混用 FreeRTOS 嗎？核心要求使用 NVIC_PRIORITYGROUP_4——別跟它對著幹。

🔬 即時：A 會搶佔 B 嗎？

設定每個中斷的搶佔層級（0 = 最緊急）。時間軸顯示 B 正在執行、A 在處理常式進行到一半時觸發，以及 A 是否被允許巢狀疊在 B 之上。

5完整流程——以及你「務必」要清除的那個位元

① 直覺 追蹤單一個邊緣從頭到尾

讓我們追蹤一次按鈕按壓，從銅線一路到處理常式。初學者會跳過、然後付出慘痛代價的那一步，正是最後一步：清除待處理位元（pending bit）。

flowchart TD A["PA0 上的邊緣"] --> B["EXTI 偵測到邊緣
設定待處理位元 EXTI→PR.0"] B --> C["NVIC 看到請求
檢查優先權"] C --> D["CPU 推入情境（context）
向量跳到 EXTI0_IRQHandler"] D --> E["你的 ISR 執行
做該做的工作"] E --> F{"清除 EXTI→PR.0？"} F -- "否" --> B F -- "是（寫入 1）" --> G["返回 → 接續主程式"]
🔑 待處理位元是「rc_w1」——靠寫入 1 來清除

當邊緣觸發時，EXTI 會在 EXTI->PR 裡鎖存（latch）一個待處理（pending）旗標。正是這個旗標告訴 NVIC 來呼叫你的。如果你的處理常式沒有清除它就返回，NVIC 會看到請求仍然處於宣告（asserted）狀態，並立刻重新進入你的 ISR——永無止境。你要靠對該位元寫入一個 1 來清除它（寫入 0 不會有任何作用）。這是一個刻意設計得很奇怪的「寫 1 清除（write-1-to-clear）」暫存器，這樣兩個處理常式才不會不小心清掉彼此的旗標。

③ 程式碼 逐行讀過

裸機（bare-metal）ISR——注意那個寫 1 清除，以及「這真的是我的線路嗎？」的守衛（guard）：

isr_baremetal.c C / STM32F4 CMSIS
void EXTI0_IRQHandler(void) {
  // 守衛：確認待處理的「這一條」線路就是它。
  if (EXTI->PR & (1U << 0)) {
    EXTI->PR = (1U << 0);          // 寫 1 清除。要最先或盡早做這件事。

    GPIOC->ODR ^= (1U << 13);      // 切換 PC13 上的 LED——快速、不可分割（atomic）。
  }
  // ISR 要保持「短」：別在這裡用 printf、HAL_Delay 或沉重的迴圈。
}

如果你偏好 HAL，這些底層接線會被藏進兩個函式裡。HAL 的 IRQ 處理常式會讀取 EXTI->PR、替你清除旗標，然後呼叫一個你可以覆寫的弱（weak）回呼（callback）：

isr_hal.c C / STM32 HAL
// 1) 真正的向量表進入點。讓 HAL 去解析並清除待處理位元。
void EXTI0_IRQHandler(void) {
  HAL_GPIO_EXTI_IRQHandler(GPIO_PIN_0);   // 內部清除 EXTI→PR.0
}

// 2) 接著 HAL 會呼叫「這個」弱回呼。用你的邏輯覆寫它。
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin) {
  if (GPIO_Pin == GPIO_PIN_0) {
    HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13);   // 你的工作寫在這裡
  }
}
⚠️ 陷阱——讓 ISR 保持簡短

ISR 執行時你的主程式是暫停的，而且（依優先權而定）其他中斷會被擋住。只做最低限度的事：清除旗標、設定一個 volatile 旗標或推入佇列（queue），然後馬上離開。在 ISR 裡呼叫 HAL_Delay() 是一個經典的卡死：它等待 SysTick 中斷，而 SysTick 的優先權可能低於你的處理常式，因此永遠無法觸發。結果就是看起來什麼事都沒發生的永久凍結。

6實戰案例：微秒級擷取與從睡眠中喚醒

④ 實戰案例 這些都會出貨到產品裡

有兩種量產模式完全仰賴 EXTI + NVIC。

低延遲事件擷取。光學編碼器、霍爾感測器，或通訊用的 PHY，每秒可能產生數千次邊緣，並帶有嚴格的時序要求。輪詢會漏掉兩次讀取之間的邊緣；而一個 EXTI 中斷則能在邊緣發生的瞬間，於 ISR 內部讀取一個自由運行（free-running）的計時器，為每個邊緣打上時間戳記。在 STM32F4 上，從邊緣到 ISR 第一道指令的延遲，是確定性（deterministic）的硬體推入約 12 個 CPU 週期，再加上你的優先權等待——是奈秒等級，而非毫秒。

🏭 實戰案例——用一個腳位從睡眠中喚醒

這是 EXTI 安靜的超能力。把一顆 STM32 置入 Stop 模式，CPU 時脈就會停下——核心的耗電降到個位數微安，對於要休眠數天的電池感測器來說再完美不過。但 EXTI 區塊在 Stop 模式下仍然存活。把一顆喚醒按鈕（或某個感測器的中斷線路）設定為 EXTI 來源、呼叫 HAL_PWR_EnterSTOPMode(...)，晶片就會待在近乎零功耗的睡眠裡，直到一個邊緣到來。那個邊緣觸發 EXTI、喚醒核心、執行你的處理常式，然後再回到深層睡眠。一個門鈴感測器、一支電視遙控器、一台血糖機——全都靠著正是這個機制，用一顆鈕扣電池運作好幾年。

🛠️ 去彈跳（debounce）該放在 ISR 之外

機械式按鈕會彈跳（bounce）：一次按壓可能在短短幾毫秒內產生數十個邊緣，把你的 ISR 觸發十幾次。別在處理常式裡用 delay() 來去彈跳——那會把一切都擋住。正確做法是：在第一個邊緣時記錄一個時間戳記，並在接下來約 20 ms 內忽略後續邊緣（在 ISR 裡做，或是停用該線路、再用一個計時器把它重新武裝）。軟體去彈跳本身是一整個主題；你只要先知道 ISR 是個不適合等待的地方就好。

7延遲數學：CPU 反應有多快？

② 數學 把週期換算成時間

「中斷很快」是一種感覺；工程需要一個數字。在 Cortex-M4 上，硬體需要固定的 12 個週期來堆疊情境並向量跳到你的處理常式（假設是零等待狀態的 Flash）。在 STM32F4 典型的 $f_{\text{clk}} = 168\,\text{MHz}$ 下，週期時間為

$$ T_{\text{cycle}} = \frac{1}{f_{\text{clk}}} = \frac{1}{168\times10^{6}\,\text{Hz}} \approx 5.95\,\text{ns} $$

因此最佳情況下的中斷進入延遲——從邊緣被接受到你 ISR 的第一道指令——為

$$ t_{\text{latency}} = 12 \cdot T_{\text{cycle}} = 12 \times 5.95\,\text{ns} \approx 71\,\text{ns} $$

這是下限。真實的最壞情況延遲還會加上兩件你能掌控的事：任何搶佔優先權更高或相等且已在執行的 ISR（你必須等它跑完——把你的優先權設計成讓緊急線路勝出），以及任何你用 __disable_irq() 停用中斷的區域。把那些臨界區段（critical sections）控制在寥寥幾道指令內，系統就能保持靈敏。如此一來，最大可接受的事件速率便直接得出：如果你的 ISR 加上開銷耗時 $t_{\text{isr}}$，你最多能持續承受

$$ f_{\max} = \frac{1}{t_{\text{isr}}} $$

每秒這麼多個邊緣，超過這個速率，ISR 就會開始堆積、而你就跟不上了。一個 1 µs 的 ISR 把你的上限封在接近 1 MHz 的事件——對按鈕來說綽綽有餘，對高速編碼器則相當吃緊，而那正是你該從 EXTI 升級到硬體計時器擷取（timer-capture）周邊的時機。

Q1 觀念檢核
你的 EXTI0_IRQHandler 在按下按鈕時跑了一次，然後 CPU 看起來就鎖死了——處理常式不斷地無止境重新進入。你還注意到，你設為優先權 2 的某個中斷，從來不會去搶佔一個優先權 9 的中斷。哪兩個敘述同時正確？
  • 你忘了 NVIC_EnableIRQ，而且數字較大代表優先權較高。
  • 你忘了清除 EXTI->PR 中的待處理位元，而且數字較小代表優先權較高（所以 2 應該搶佔 9）。
  • 向量表壞掉了，而且優先權 9 的位階高於優先權 2。
  • EXTI 會自動清除它的旗標，而且相等的次優先權永遠不會搶佔。
無止境重新進入是「沒有清除 EXTI->PR」的招牌徵兆（要對該線路的位元寫入一個 1）。而 Cortex-M 的優先權是「第 0 名（rank 0）獲勝」：數字較小 = 優先權較高，所以優先權 2 的中斷應該搶佔優先權 9 的。如果沒有，那你的優先權分組八成把那些位元放進了次優先權，而次優先權只負責打破平手、永遠不會搶佔。
Q2 算出來
STM32F4 在 NVIC 中實作了 4 個優先權位元。這給了你多少個不同的中斷優先權層級？（回想 $\text{levels} = 2^{N_{\text{prio}}}$。）
層級
$2^{4} = 16$ 個層級，從 0（最緊急）編號到 15（最不緊急）。只實作 3 個位元的晶片會給出 $2^3 = 8$ 個層級；實作 8 個位元的高階零件則給出 $2^8 = 256$ 個。重點在於實作的位元數——而非完整的 8 位元欄位。
Q3 程式挑戰——按鈕 EXTI 切換一顆 LED
使用 HAL，把 PA0 上的板載使用者按鈕設定為下降邊緣的 EXTI 來源，並在每次按壓時切換 PC13 上的 LED。請寫出 GPIO/NVIC 初始化以及中斷處理常式 + 回呼。記住：處理常式必須清除待處理旗標（或交給 HAL 去做），LED 切換放在回呼裡，而且 ISR 要保持簡短。
提示
先啟用 GPIOA、GPIOC 和 SYSCFG 的時脈。用 GPIO_MODE_IT_FALLING 設定 PA0（HAL 的「下降邊緣中斷」模式會替你完成 SYSCFG/EXTI 的接線）。接著為 EXTI0_IRQn 呼叫 HAL_NVIC_SetPriority + HAL_NVIC_EnableIRQ。向量進入點呼叫 HAL_GPIO_EXTI_IRQHandler(GPIO_PIN_0)；你的切換動作放在 HAL_GPIO_EXTI_Callback 裡。
顯示一種解答
button_exti.c C / STM32 HAL
void EXTI_Button_Init(void) {
  __HAL_RCC_GPIOA_CLK_ENABLE();   // 按鈕所在的埠
  __HAL_RCC_GPIOC_CLK_ENABLE();   // LED 所在的埠
  __HAL_RCC_SYSCFG_CLK_ENABLE();  // EXTI 埠繞接所必需

  // PC13 上的 LED：推挽（push-pull）輸出
  GPIO_InitTypeDef led = {0};
  led.Pin   = GPIO_PIN_13;
  led.Mode  = GPIO_MODE_OUTPUT_PP;
  led.Pull  = GPIO_NOPULL;
  led.Speed = GPIO_SPEED_FREQ_LOW;
  HAL_GPIO_Init(GPIOC, &led);

  // PA0 上的按鈕：在「下降」邊緣中斷，並帶上拉（pull-up）
  GPIO_InitTypeDef btn = {0};
  btn.Pin  = GPIO_PIN_0;
  btn.Mode = GPIO_MODE_IT_FALLING;   // 接好 SYSCFG + EXTI FTSR + IMR
  btn.Pull = GPIO_PULLUP;
  HAL_GPIO_Init(GPIOA, &btn);

  // NVIC：啟用 EXTI0，給個適中的優先權
  HAL_NVIC_SetPriority(EXTI0_IRQn, /*preempt*/ 2, /*sub*/ 0);
  HAL_NVIC_EnableIRQ(EXTI0_IRQn);
}

// 向量表進入點——讓 HAL 替我們解析並清除 EXTI→PR。
void EXTI0_IRQHandler(void) {
  HAL_GPIO_EXTI_IRQHandler(GPIO_PIN_0);
}

// HAL 在清除旗標後呼叫這個。保持簡短。
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin) {
  if (GPIO_Pin == GPIO_PIN_0) {
    HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13);
  }
}

注意這裡的分工：GPIO_MODE_IT_FALLING 處理所有 SYSCFG/EXTI 暫存器寫入，NVIC 的呼叫是明確寫出的，而實際工作就是回呼裡的一次切換。在量產中你還會加上軟體去彈跳，這樣單一次按壓才不會被記成好幾次——但中斷的骨架就正是這樣。
