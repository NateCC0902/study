從 AVR 到 ARM：Cortex-M、記憶體映射與工具鏈

你已經知道一顆 8 位元 AVR 如何讓 LED 閃爍。現在我們要升級到一顆 32 位元的 ARM Cortex-M 核心，它的執行速度快上一個數量級，而且它的周邊（peripheral）能在 CPU 休眠時繼續工作。物理原理相同，但這是一台大得多的機器。我們會圍繞著一張 32 位元記憶體映射（memory map），以及對它燒錄程式的工具鏈，重新建立你的心智模型——然後用兩種方式讓 LED 閃爍，親眼看看「函式庫帶來的舒適」到底讓你付出了什麼代價。

⏱ 約 45 分鐘🧠 心智模型🗺️ 記憶體映射🔧 工具鏈⚡ 暫存器層級的閃爍

1為什麼要離開 8 位元 AVR 的舒適圈？

① 直覺 先把整體圖像建立起來

ATmega328P 一直為你服務得很好：8 位元、16 MHz、2 KB 的 SRAM，外加一把周邊。它之所以是個絕佳的教學對象，正是因為它夠小，小到你能整顆裝進腦袋裡。但只要一個專案想要 USB、LCD、音訊、馬達控制或一套 TCP/IP 協定堆疊，328P 的時脈週期與 RAM 早在腳位用完之前就會耗盡。你需要一台更大的機器——而業界的答案就是 ARM Cortex-M 系列，由意法半導體（ST Microelectronics）封裝為 STM32。

💡 一句話模型

一顆 STM32 與你 Arduino 上的 MCU 是同一個概念——單一晶片內含 CPU、記憶體與周邊——但它建構在一顆 32 位元 ARM 核心上，執行速度快上 4–10 倍，記憶體與周邊多上 10–100 倍，而且這些周邊聰明到能在 不用 CPU 的情況下搬移資料。

具體來說，把 Uno 拿來和兩塊你只要花幾美元就能買到的 STM32 開發板比較：STM32F103C8T6（「Blue Pill」，一顆 Cortex-M3）與 STM32F411CEU6（「Black Pill」，一顆帶浮點運算單元的 Cortex-M4）。數字會說明一切。

🔑 真正的升級是 DMA

更多的兆赫（MHz）固然不錯，但真正改變遊戲規則的是 DMA（直接記憶體存取，Direct Memory Access）：一個專屬引擎，能 自主地 在周邊與記憶體之間複製位元組。在 AVR 上，讀取 1000 個 ADC 取樣意味著 CPU 必須一個一個盯著。在 STM32 上，你只要啟動一個 DMA 通道，CPU 就能空出來做其他工作——或是進入休眠以省電——而取樣會在它背後源源不絕地流進緩衝區。這正是一顆 100 MHz 晶片能勝過其時脈數字的原因。

2Cortex-M 家族與 STM32 產品線

① 直覺 一套核心架構，多種尺寸

「Cortex-M」並不是某一顆晶片——它是 ARM 提供的一個 可授權的 CPU 核心設計。ST 買下這個核心，然後在它周圍包上自家的記憶體與周邊，做成一顆 STM32。知道裡面坐著哪一顆 Cortex-M，就能告訴你這顆晶片的指令集、是否帶有浮點運算單元，以及它大概多少錢。

核心特點FPU？典型角色
Cortex-M0 / M0+最小、最便宜、功耗最低無微型低成本 MCU（STM32F0、L0）
Cortex-M3完整 Thumb-2，經典的主力無STM32F1（Blue Pill）、F2
Cortex-M4M3 + DSP 指令 + 選配 FPU單精度STM32F4、L4——控制與訊號處理
Cortex-M7超純量、快取、非常快單 + 雙精度STM32F7、H7——重度運算

ST 把自家晶片分成多條 產品線（line），每條產品線建構在所選的核心上，並針對某個用途調校。一旦看出規律，命名其實很有邏輯：

🛠️ 解讀料號

在 STM32F411CEU6 中：F4 = 產品線，11 = 子家族，C = 48 腳，E = 512 KB Flash，U = QFN 封裝，6 = −40…85 °C 溫度等級。一旦你知道這些欄位的含義，就能一眼解碼任何一個 STM32 料號——而且資料手冊的第一頁就會把它們全部列出來。

整個進度中，我們會固定以兩顆晶片作為貫穿全程的範例：STM32F103（便宜、M3、最具代表性的第一顆 STM32）與 STM32F4 家族（M4 + FPU、快速、我們的主力）。你學到的一切都能在整條產品線間移植——暫存器名稱與記憶體映射保持得相當一致。

332 位元記憶體映射

① 直覺 一個平坦的 4 GB 位址空間

這是從 AVR 跳到 ARM 時最重要的一個觀念。AVR 使用 哈佛（Harvard）架構：程式與資料活在 分開 的位址空間，這也是為什麼會有 PROGMEM 這類技巧存在。Cortex-M 翻轉為 單一、平坦的 32 位元位址空間，由所有東西共享——Flash、RAM、周邊，以及核心自己的控制暫存器，全都坐落在 同一 張映射的不同位址上。

一個 32 位元指標可以命名 $2^{32}$ 個不同的位元組位址，這恰好是 4 GiB 的位址空間，即使這顆晶片實際上只填入其中幾百 KB：

$$ 2^{32} \text{ bytes} = 4{,}294{,}967{,}296 \text{ bytes} = 4\,\text{GiB} = \texttt{0x0000\,0000} \;\text{…}\; \texttt{0xFFFF\,FFFF} $$

ARM 為這 4 GiB 定義了一個標準佈局，因此每一顆 Cortex-M 晶片用起來都很熟悉。你必須牢記的四個區域：

區域基底位址這裡住著什麼
Flash（程式碼）0x0800 0000你的程式、向量表、常數
SRAM（資料）0x2000 0000變數、堆疊（stack）、堆積（heap）——揮發性
周邊0x4000 0000GPIO、計時器、UART、ADC… 的暫存器區塊
核心（私有）0xE000 0000NVIC、SysTick、SCB——Cortex-M 本身
🔑 一切都是位址

在 Cortex-M 上，沒有像 AVR 的 OUT 那種特殊的「I/O 指令」。一個周邊暫存器 就只是 位於 0x4000 0000 區域某個位址上的記憶體。要點亮一顆 LED，你就真的是把一個 32 位元的值寫到某個數字位址。整台機器都能透過同一種指標型別觸及。

② 數學 基底 + 偏移 = 暫存器位址

每個周邊都佔據一段連續的區塊，從一個 基底位址（base address）開始。在那個區塊內，每個暫存器坐落在參考手冊定義的固定 偏移（offset）上。任何暫存器的位址就只是：

$$ \text{addr}_{\text{reg}} = \text{base}_{\text{periph}} + \text{offset}_{\text{reg}} $$

來算一個具體的例子。在 STM32F4 上，GPIO 埠 A 掛在 AHB1 匯流排上，基底位址為 0x4002 0000。輸出資料暫存器（ODR）——那個位元直接對應到腳位電位的暫存器——坐落在埠內偏移 0x14 處。所以：

$$ \texttt{GPIOA->ODR} = \underbrace{\texttt{0x4002\,0000}}_{\text{base}} + \underbrace{\texttt{0x14}}_{\text{offset}} = \texttt{0x4002\,0014} $$

用十六進位做這個加法毫不費力：低位的 nibble 0x0 + 0x14 = 0x14，上面的部分原封不動。你在程式裡寫的 CMSIS 結構 GPIOA->ODR，在編譯期就會被解析成恰好這個常數。編譯器替你做了「基底加偏移」的算術，讓你不必記得 0x40020014——但現在你知道它是從哪裡來的了。

⚠️ 陷阱——Flash 在 0x0800 0000，不在 0x0

映射最底端（0x0000 0000）是一個 別名（alias）區域，其內容取決於開機腳位（boot pins）——它通常會鏡射 Flash，好讓 CPU 能在那裡擷取它的第一條指令。但你程式碼 Flash 的 真正、固定 位址是 0x0800 0000。連結（link）與推理 Flash 時，永遠以 0x0800 0000 為準；把 0x0 別名只當成開機時的方便手段就好。

4暫存器就是記憶體：volatile 與 CMSIS

③ 程式碼 每一行都要讀

如果一個暫存器就只是位於已知位址的記憶體，那你用一個普通的指標就能觸及它。以下先是用最原始的方式存取 GPIOA 的 ODR，接著是你實際上會採用的寫法：

raw_register.c C（裸機，bare metal）
#include <stdint.h>

// 我們推導出來的位址：GPIOA 基底 0x40020000 + ODR 偏移 0x14。
// 'volatile' 是必要的：它告訴編譯器這塊記憶體會自行改變
// （硬體！），所以絕對不可以把這個值快取在 CPU 暫存器裡，
// 也不可以把存取最佳化掉。
#define GPIOA_ODR  (*(volatile uint32_t *)0x40020014u)

void set_pin5_high(void) {
    GPIOA_ODR |= (1u << 5);   // 設定位元 5 → 把 PA5 拉高
}

到處寫魔術數字（magic number）很容易出錯，所以 ARM 與 ST 提供了 CMSIS（Cortex 微控制器軟體介面標準，Cortex Microcontroller Software Interface Standard）：一組標頭檔，把每個周邊描述成一個 C struct，其成員 恰好 對齊暫存器的偏移。你只要 include 一個裝置標頭檔，位址的事就替你搞定了：

cmsis_register.c C（CMSIS）
#include "stm32f4xx.h"   // ST 的 CMSIS 裝置標頭檔

// GPIO_TypeDef 是一個佈局與硬體相符的 struct：
//   typedef struct {
//     volatile uint32_t MODER;   // 偏移 0x00
//     volatile uint32_t OTYPER;  // 偏移 0x04
//     volatile uint32_t OSPEEDR; // 偏移 0x08
//     volatile uint32_t PUPDR;   // 偏移 0x0C
//     volatile uint32_t IDR;     // 偏移 0x10
//     volatile uint32_t ODR;     // 偏移 0x14  ← 就在這裡
//     volatile uint32_t BSRR;    // 偏移 0x18
//     ...
//   } GPIO_TypeDef;
// 並且：#define GPIOA ((GPIO_TypeDef *) 0x40020000UL)

void set_pin5_high(void) {
    GPIOA->ODR |= (1u << 5);   // 與原始版本完全相同，但更易讀
}
🔑 為什麼 volatile 不容妥協

一個普通變數住在 RAM 裡，而且只有你的程式碼會改它，所以編譯器可能把它留在 CPU 暫存器中，並略過多餘的讀取。但一個 硬體 暫存器可能在你的兩條指令之間就改變了（有人按了按鈕；某個位元組到達了），而且寫它會有 副作用（某個腳位動了）。volatile 強迫編譯器為 每一次 存取都產生真正的載入／儲存（load/store），並依程式順序執行。忘了它，你那個輪詢 USART->SR 的迴圈就可能永遠卡死，因為編譯器「以為」這個值永遠不會變。CMSIS 已經替你把每個暫存器標上 volatile——但你必須理解為什麼。

5工具鏈與 HAL / LL / CMSIS 的抉擇

① 直覺 三種抽象層級

在 Arduino 上，寫程式只有一種方式：Arduino core。STM32 給你的是一個 光譜，從「點一點 GUI 就替你生成程式碼」一路往下到「自己戳暫存器」。標準工具是 STM32CubeIDE（一個免費、以 Eclipse 為基礎的 IDE），搭配 STM32CubeMX，一個圖形化設定器，讓你用點選的方式配置腳位與時脈，然後生出一個專案。有意思的抉擇在於：你的程式碼要坐在哪一個 抽象層 上：

抽象層它是什麼易用度掌控度程式碼大小速度
HAL（硬體抽象層，Hardware Abstraction Layer）可移植的高階函式（HAL_GPIO_TogglePin）最高最低最大最慢
LL（低階層，Low-Layer）輕薄的 inline 包裝，與暫存器約 1:1 對應中等高小快
CMSIS／裸暫存器你直接寫 GPIOA->BSRR = …最低最高最小最快
🛠️ 每一層各自取捨了什麼

HAL 替你換來可移植性與開發速度——同一個呼叫能跨越整條 STM32 產品線運作，內建錯誤處理與逾時機制——代價是更大、更慢的程式碼，以及在你與矽晶片之間隔著一疊函式呼叫。LL 讓你保持可讀性，但編譯下來幾乎與手寫的暫存器存取是同一批指令。裸 CMSIS 給你最終極的掌控與最精簡的程式碼，但你得自己扛下每一個資料手冊細節，而且沒有任何安全網。多數產品從 HAL 起步，只在真正需要的熱路徑（hot path）上才降到 LL／暫存器。

🏭 真實案例——開機載入程式裡的 HAL 肥大

一個必須塞進 Flash 前 16 KB 的開機載入程式（bootloader），往往 負擔不起 HAL：把 HAL_GPIO、HAL_RCC 與 SysTick 那套機制全拉進來，可能在你做出任何有用的事情之前就先把預算炸掉。團隊們經常正是基於這個理由，用純 CMSIS 暫存器程式碼來寫開機載入程式，然後在 Flash 預算寬裕的主應用程式裡再切換回 HAL。按 元件（component） 而非按 專案 來選擇抽象層，是一位資深韌體工程師的標誌。

6STM32 如何開機

① 直覺 在 main() 執行之前

在 Uno 上，是一個開機載入程式把你的程式複製進去，然後跳到它。在 STM32 上，預設並沒有 Arduino 那種開機載入程式——晶片是依照一套小巧的、由 ARM 定義的儀式 直接 從 Flash 開機。理解它，就能破解你這輩子會遇到的每一個「我的全域變數明明設成 5，怎麼會是 0」的 bug。

在 0x0800 0000，Flash 最前面那幾個字（word）並不是程式碼——它們是 向量表（vector table）。第一個 32 位元的字是 初始堆疊指標（initial stack pointer）；第二個字是 重置向量（reset vector），也就是要執行的第一條指令的位址。CPU 在執行任何東西之前，會在上電時載入這兩個值：

vector_table（概念示意） C（CMSIS）
// 向量表的最前面幾個項目，由連結器（linker）放在 0x08000000。
// CPU 把 word[0] 讀進 SP，然後跳到 word[1]（Reset_Handler）。
__attribute__((section(".isr_vector")))
const uint32_t vector_table[] = {
    (uint32_t)&_estack,          // [0] 0x08000000：初始主堆疊指標（MSP）
    (uint32_t)Reset_Handler,     // [1] 0x08000004：重置向量 → 第一段程式碼
    (uint32_t)NMI_Handler,       // [2] 0x08000008：不可遮罩中斷（NMI）
    (uint32_t)HardFault_Handler, // [3] 0x0800000C：硬錯誤（hard fault）
    // ... SysTick，接著每個周邊 IRQ 各一個項目 ...
};

重置向量指向 Reset_Handler，這是一段啟動程式（通常是 startup_stm32f4xx.s），它會在你的 main() 能安全執行 之前，先把 C 執行期環境（runtime）準備好。它依序做四件事：

flowchart TB P([上電 / 重置]) --> SP["載入 SP ← word[0]
位於 0x08000000"] SP --> RV["跳到 Reset_Handler
（word[1] @ 0x08000004）"] RV --> DATA["複製 .data
Flash → SRAM
（已初始化的全域變數）"] DATA --> BSS["將 .bss 清零
於 SRAM 中
（未初始化的全域變數）"] BSS --> SI["呼叫 SystemInit()
設定時脈 / FPU"] SI --> MAIN["呼叫 main()"] MAIN --> LOOP(["你的 while(1) 迴圈"])
🔑 為什麼全域變數會以正確的值起始

像 int speed = 5; 這樣一個已初始化的全域變數，它的 值（5）儲存在 Flash 中，但在執行期住在 SRAM 裡。啟動程式的 .data 複製 把那些位元組從 Flash 搬到 SRAM。像 int count; 這樣一個未初始化的全域變數，依 C 標準必須讀作 0——那就是 .bss 清零 這一步。如果你曾跳過或弄壞啟動流程（自己手刻一個裸機專案時很常見），全域變數就會裝著垃圾值，而你會去追一個幽靈。這就是韌體版的「在我的機器上明明可以跑」。

⚠️ 陷阱——SystemInit 在 main 之前執行

SystemInit() 通常會設定 PLL，好讓晶片真正以 72/84/100 MHz 運行，而非它緩慢的內部預設值。如果你的時序莫名其妙地慢了 8 倍，那就是你的時脈樹（clock tree）從來沒被設定起來——檢查一下 SystemInit() 是否有執行，以及 SystemCoreClock 是否與現實相符。

7兩種方式閃爍：HAL 對上裸暫存器

③ 程式碼 同一個閃爍，兩種抽象層級

是時候把這個對比變得具體了。兩支程式都讓 STM32F4 Nucleo／Black Pill 上的使用者 LED（PA5）閃爍。先看 HAL 版本——短小、可移植，也正是 CubeMX 生成的東西：

blink_hal.c C（HAL）
#include "stm32f4xx_hal.h"

int main(void) {
    HAL_Init();                       // 設定 SysTick、flash 延遲等等
    __HAL_RCC_GPIOA_CLK_ENABLE();     // 開啟通往 GPIO 埠 A 的時脈

    GPIO_InitTypeDef io = {0};
    io.Pin   = GPIO_PIN_5;            // PA5
    io.Mode  = GPIO_MODE_OUTPUT_PP;   // 推挽（push-pull）輸出
    io.Speed = GPIO_SPEED_FREQ_LOW;
    HAL_GPIO_Init(GPIOA, &io);

    while (1) {
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5); // 翻轉腳位
        HAL_Delay(500);                        // 500 毫秒，由 SysTick 驅動
    }
}

現在用 裸暫存器 程式碼做同一個閃爍。除了 CMSIS 的結構定義外不用任何函式庫——每一行都對應到一個資料手冊欄位。讀讀那些註解；這正是本章的重點所在：

blink_registers.c C（CMSIS／裸機）
#include "stm32f4xx.h"   // 只用暫存器結構定義

int main(void) {
    // 1) 周邊在重置時被時脈閘控（clock-gated）為關閉以省電。
    //    透過 RCC->AHB1ENR 的位元 0（GPIOAEN）啟用通往 GPIOA 的時脈。
    RCC->AHB1ENR |= (1u << 0);                 // RCC_AHB1ENR_GPIOAEN

    // 2) MODER 每個腳位佔 2 個位元。把 PA5 設成 '01' = 通用輸出。
    //    先清除這 2 位元的欄位，再寫入 01。
    GPIOA->MODER &= ~(3u << (5 * 2));          // 清除位元 11:10
    GPIOA->MODER |=  (1u << (5 * 2));          // 設成 01 → 輸出

    while (1) {
        // 3) BSRR 是原子的設定/重置暫存器。對 LOW 半部的某位元寫 1
        //    會「設定」該腳位；對 HIGH 半部（位元 + 16）寫 1 則「重置」它。
        //    沒有讀-改-寫，因此沒有中斷能破壞它。
        GPIOA->BSRR = (1u << 5);               // PA5 = HIGH
        for (volatile uint32_t d = 0; d < 800000; d++) { }   // 粗略的延遲

        GPIOA->BSRR = (1u << (5 + 16));        // PA5 = LOW（重置半部）
        for (volatile uint32_t d = 0; d < 800000; d++) { }
    }
}
💡 這個對比教會我們什麼

HAL 版本讀起來像散文，而且能在任何 STM32 上運作；暫存器版本讀起來像資料手冊，而且短短幾條指令就完事。有三個只有原始版本才會曝露出來的細節：周邊在你於 RCC 中啟用它們之前，都被 時脈閘控為關閉（這是「我的 GPIO 毫無動靜」的頭號原因——AVR 從來不會逼你做這件事）；MODER 每個腳位用 兩個位元，所以你要先遮罩再設定；而 BSRR 讓你能 原子地（atomically） 設定或重置一個腳位，不需要可能被中斷破壞的讀-改-寫。HAL 替你做了這三件事——現在你知道它藏起了什麼。

⚠️ 陷阱——忘了啟用 RCC 時脈

STM32 上最常見的初學者 bug：你把 MODER 設定得完美無缺、也把 BSRR 寫得漂漂亮亮，結果什麼都沒發生——因為你從來沒在 RCC->AHB1ENR 裡設定那個 GPIOAEN 位元。一個被時脈閘控的周邊會默默無視所有寫入。永遠要 先 啟用時脈。

Q1 觀念檢核
你在一顆 STM32F4 上存取 GPIOA->ODR 來讀取腳位狀態。這個暫存器位址位於 32 位元記憶體映射的哪個區域，以及選擇 HAL 而非裸暫存器，主要犧牲掉的是什麼？
  • SRAM 區域（0x2000 0000）；HAL 犧牲掉可移植性。
  • 周邊區域（0x4000 0000）；HAL 犧牲掉程式碼大小與速度。
  • Flash 區域（0x0800 0000）；HAL 什麼都沒犧牲。
  • 核心／私有區域（0xE000 0000）；HAL 犧牲掉可移植性。
GPIO 暫存器區塊坐落在 0x4000 0000 的 周邊區域（GPIOA 的基底是 0x40020000）。HAL 替你換來易用性與可移植性；它付出的代價是 程式碼大小與速度——比起一行就搞定的暫存器寫入，它多了好幾層函式呼叫與更大的二進位檔。Flash 裝程式碼，SRAM 裝變數，而 0xE000 0000 是核心自己的 NVIC/SysTick。
Q2 算出來
在 STM32F4 上，GPIO 埠 A 的基底位址為 0x40020000，輸出資料暫存器 ODR 坐落在偏移 0x14 處。使用 $\text{addr}_{\text{reg}} = \text{base} + \text{offset}$，GPIOA->ODR 的絕對位址是多少？以十六進位輸入（例如 0x40020014）或以十進位整數輸入。
addr
$\texttt{0x40020000} + \texttt{0x14} = \texttt{0x40020014}$（十進位 $1073872916$）。在十六進位中只有最低位元組改變：$\texttt{0x00} + \texttt{0x14} = \texttt{0x14}$，上面的部分原封不動。這正是 CMSIS 結構 GPIOA->ODR 編譯下來的確切常數。
Q3 程式挑戰——在 Blue Pill 上做暫存器層級的閃爍
STM32F103「Blue Pill」的板載 LED 接在 PC13 上（而且它是 低電位有效（active-low）——把腳位拉 LOW 才會點亮它）。F1 家族使用的 GPIO 暫存器與 F4 不同：腳位 8–15 用 CRH（高位設定暫存器，configuration register high），GPIOC 的時脈啟用用 APB2ENR，還有一對 BSRR／BRR。請為 PC13 寫一支裸暫存器的閃爍程式。想一想：哪條時脈匯流排、哪個設定欄位、如何切換。
提示
在 F1 上，GPIOC 在 APB2 匯流排上：在 RCC->APB2ENR 中設定 IOPCEN 位元（位元 4）。PC13 由 GPIOC->CRH 設定：每個腳位用一個 4 位元的欄位，而 PC13 的欄位起始於位元 (13 − 8) × 4 = 20。在那裡寫入 0b0010 代表「輸出、2 MHz、推挽」。用 BSRR 來設定腳位（LED 熄滅，因為是低電位有效），用 BSRR 的上半部——也就是 BRR 或 位元+16——來重置它（LED 點亮）。
顯示一種解答
blink_pc13.c C（CMSIS／裸機）
#include "stm32f1xx.h"

int main(void) {
    // 1) GPIOC 在 APB2 上。啟用它的時脈（IOPCEN = 位元 4）。
    RCC->APB2ENR |= (1u << 4);                 // RCC_APB2ENR_IOPCEN

    // 2) PC13 位於 CRH（腳位 8..15）。它的 4 位元欄位起始於
    //    位元 (13-8)*4 = 20。先清除它，再寫入 0010 =
    //    MODE=10（輸出 2 MHz）、CNF=00（推挽）。
    GPIOC->CRH &= ~(0xFu << 20);               // 清除 PC13 的 4 位元欄位
    GPIOC->CRH |=  (0x2u << 20);               // 0010：輸出、推挽

    while (1) {
        // PC13 是低電位有效：重置 = LED 亮，設定 = LED 滅。
        GPIOC->BSRR = (1u << (13 + 16));       // 重置 PC13 → LED 亮
        for (volatile uint32_t d = 0; d < 300000; d++) { }

        GPIOC->BSRR = (1u << 13);              // 設定 PC13   → LED 滅
        for (volatile uint32_t d = 0; d < 300000; d++) { }
    }
}

這裡有兩個體會。第一，暫存器映射因家族而異：F1 用 CRH/CRL 搭配 4 位元欄位與 APB2 時脈，而 F4 用 MODER 搭配 2 位元欄位與 AHB1——然而其工作流程（啟用時脈 → 設定模式 → 透過 BSRR 切換）卻完全相同。第二，低電位有效 的硬體會把你的邏輯反過來：重置腳位反而點亮 LED。這正是 HAL 存在的理由——但也正是為什麼「讀參考手冊」是一項你現在已經具備的技能。
