串列匯流排：USART、I²C、SPI 與鮑率數學

在 Arduino 上你只有一個硬體 UART、一個 I²C、一個 SPI——而且它們都會阻塞 CPU。STM32 則交給你一整支匯流排控制器艦隊，每一個都能在核心進行真正工作的同時觸發中斷或透過 DMA 泵送位元組。我們將從暫存器層往上重建這三條經典匯流排，精確推導鮑率與分頻器的數學，並在同一塊板子上接上主控台、一顆 EEPROM 與一顆 IMU。

⏱ 約 50 分鐘🧠 匯流排協定⚡ DMA 與中斷🔢 鮑率數學

1三條匯流排，一塊板——快速回顧

① 直覺 先把畫面建立起來

你在 Arduino 第 06 章已經見過這三條匯流排。它們的協定在 STM32 上完全相同——同樣的線、同樣的訊框、同樣的概念——所以我們快速回顧，然後前往 8 位元 AVR 無法跟隨之處：多個實例、中斷驅動的傳輸，以及 DMA。

每一條串列匯流排都在解決同一個問題：用盡可能少的線在晶片之間搬移位元組。它們的差別在於需要幾條線、由誰驅動時脈，以及接收端如何知道一個位元組從哪裡開始。

匯流排線時脈拓撲典型角色
USART/UARTTX、RX（+GND）無——約定好的鮑率點對點主控台、GPS、無線數據機
I²CSDA、SCL共用，由控制器驅動多點掛載、定址EEPROM、RTC、慢速感測器
SPISCK、MOSI、MISO、CS由控制器驅動星形（每個裝置一條 CS）Flash、顯示器、高速 IMU
💡 一句話區分

UART 是非同步的——沒有時脈線，所以兩端必須事先就位元速率達成共識。I²C 與 SPI 是同步的——時脈線與資料同行，因此時序是自我描述的。非同步用一條線換取「把鮑率調到精準」的負擔；正是這個取捨，使得 §2 存在。

名稱上的縮寫跳躍也很重要：AVR 有的是 UART。大多數 STM32 周邊則是 USART——多出來的那個「S」是 Synchronous（同步），意思是它們也能輸出時脈，並以類似 SPI 或智慧卡的模式運作。你幾乎總是以單純的非同步 UART 模式使用它們，但暫存器仍冠上 USART 之名。

2USART 鮑率：真正能上線的數學

① 直覺 在黑暗中取樣

沒有時脈線，接收端就必須猜每個位元落在哪裡。訣竅是：跑一個比預期位元速率快 16 倍的內部時脈（16 倍過取樣），等待起始位元的下降邊緣，然後在每個資料位元的中央附近取樣——大約是 16 個時脈刻度中的第 8 個。把除數算得稍微偏一點，取樣點就會漂移；在一個訊框內漂移得夠遠，你讀到的就是亂碼。

② 數學 推導 USARTDIV

周邊時脈 $f_{CK}$（餵給 USART 的匯流排時脈——PCLK1 或 PCLK2）會被除以 16，再除以一個小數分頻器 $\text{USARTDIV}$，以產生鮑率。在 16 倍過取樣下：

$$ \text{baud} = \frac{f_{CK}}{16 \cdot \text{USARTDIV}} \quad\Longrightarrow\quad \text{USARTDIV} = \frac{f_{CK}}{16 \cdot \text{baud}} $$

$\text{USARTDIV}$ 是一個定點數：一個整數尾數（mantissa）加上一個以十六分之一為單位的 4 位元小數（fraction），以 [尾數 : 小數] 的形式打包進鮑率暫存器 BRR。我們來實際程式化一段。取一顆 STM32F4，USART 掛在 PCLK2 = 84 MHz 上，目標為 115200 鮑：

$$ \text{USARTDIV} = \frac{84\times10^{6}}{16 \cdot 115200} = \frac{84\times10^{6}}{1\,843\,200} \approx 45.5729 $$

把它拆開：尾數是 $45$；小數是 $0.5729 \times 16 \approx 9.17 \to 9$（四捨五入到最接近的十六分之一）。於是我們程式化

$$ \text{BRR} = (45 \ll 4) \,|\, 9 = 0x2D9, \qquad \text{USARTDIV}_{\text{prog}} = 45 + \tfrac{9}{16} = 45.5625 $$

現在來檢查它實際產生的鮑率，以及由此產生的誤差：

$$ \text{baud}_{\text{actual}} = \frac{84\times10^{6}}{16 \cdot 45.5625} = 115226\ \text{baud}, \qquad \varepsilon = \frac{115226 - 115200}{115200} \approx +0.02\% $$
🔑 2% 法則

兩端會在一個 10 位元訊框（起始 + 8 資料 + 停止）內累積誤差。傳送端與接收端合計的誤差預算大約是 ±2–3%，超過取樣點就會漂出位元之外。我們的 +0.02% 安全得奢侈。這正是為什麼整數友善的時脈很重要：在某個尷尬的 $f_{CK}$ 下，有些鮑率根本無法被準確命中——這也是一再出現的理由，提醒我們要在 第 02 章就刻意挑選晶振與 PLL 數值。

③ 程式碼 逐行細讀

HAL 把 BRR 藏在 Init.BaudRate 之後，會根據即時的時脈樹替你計算 USARTDIV。但懂得這套數學，能讓你對它做合理性檢查，並在主控台冒出亂碼時除錯。

usart_init.c C / STM32 HAL
// STM32F4 上的 USART2，由 PCLK1 供時脈；此處示範 APB2/USART1，PCLK2 = 84 MHz。
UART_HandleTypeDef huart1;

void uart1_init(void) {
  __HAL_RCC_USART1_CLK_ENABLE();          // 開啟 USART 時脈閘
  huart1.Instance        = USART1;        // 這顆零件上 PCLK2 = 84 MHz
  huart1.Init.BaudRate   = 115200;        // HAL 解出 USARTDIV -> BRR = 0x2D9
  huart1.Init.WordLength = UART_WORDLENGTH_8B;
  huart1.Init.StopBits   = UART_STOPBITS_1;
  huart1.Init.Parity     = UART_PARITY_NONE;     // 8N1——萬用預設值
  huart1.Init.Mode       = UART_MODE_TX_RX;
  huart1.Init.OverSampling = UART_OVERSAMPLING_16; // baud = f_CK / (16 * USARTDIV)
  HAL_UART_Init(&huart1);                 // 寫入 BRR、CR1、CR2、CR3
}

// 阻塞式傳送——CPU 會空轉，直到每個位元組都離開移位暫存器。
void uart1_puts(const char *s) {
  HAL_UART_Transmit(&huart1, (uint8_t*)s, strlen(s), HAL_MAX_DELAY);
}
⚠️ 陷阱——PCLK 錯了，鮑率就錯了

如果你變更了時脈樹（新的 PLL 設定、不同的 APB 分頻器），但 USART 仍以為自己跑在舊的 $f_{CK}$ 上，那麼每一個鮑率都會偏移，主控台就會塞滿亂碼。HAL 是在 HAL_UART_Init 當下讀取時脈的，所以任何時脈變更後都要重新初始化。HAL_RCC_GetPCLK2Freq() 會告訴你它實際採用的真實數值。

🔬 即時：USARTDIV 與鮑率誤差探索器

拖曳 fCK 與目標鮑率：觀察精確的 USARTDIV、實際程式化進 BRR 的四捨五入後尾數／小數，以及由此產生、對照 ±2% 預算的鮑率誤差。

3超越阻塞：以中斷與 DMA 驅動 UART

① 直覺 別再保母式照看位元組

HAL_UART_Transmit(..., HAL_MAX_DELAY) 雖然誠實，卻浪費——在 115200 鮑下，每個位元組約需 $\frac{10\ \text{bits}}{115200} \approx 87\,\mu s$，而 CPU 全程空轉。送出一行 1 KB 的日誌會阻塞約 89 ms。在即時系統上這是一場災難。STM32 給了你兩條逃生路。

中斷模式——每當 USART 的傳送暫存器清空時，它就會引發一次中斷；一個 ISR 餵入下一個位元組。CPU 每個位元組只造訪一次。DMA 模式——一個專屬的直接記憶體存取（Direct Memory Access）引擎，在零 CPU 介入的情況下，把整個緩衝區串流到 USART 資料暫存器；核心只在傳輸完成時被中斷恰好一次。

③ 程式碼 非阻塞傳送
uart_dma.c C / STM32 HAL
// 假設 CubeMX 已接好 DMA2_Stream7（USART1_TX）以及 DMA IRQ 處理常式。
uint8_t log_buf[256];

void log_line_dma(const char *msg) {
  int n = snprintf((char*)log_buf, sizeof log_buf, "%lu: %s\r\n",
                   HAL_GetTick(), msg);
  // 立即回傳；DMA 在 loop() 繼續運行的同時推送這些位元組。
  HAL_UART_Transmit_DMA(&huart1, log_buf, n);
}

// 緩衝區被抽乾時，HAL 會從 DMA 傳輸完成的 ISR 呼叫此函式。
void HAL_UART_TxCpltCallback(UART_HandleTypeDef *huart) {
  if (huart->Instance == USART1) {
    tx_busy = 0;                 // 現在可以安全地排入下一行
  }
}

// 接收：一次性把 DMA 武裝指向環形緩衝區，讓它在背景中填滿。
uint8_t rx_buf[64];
void rx_start(void) {
  HAL_UART_Receive_DMA(&huart1, rx_buf, sizeof rx_buf);  // 非阻塞 RX
}
⚠️ 陷阱——DMA 進行中觸碰緩衝區

DMA 引擎直接從 SRAM 讀取 log_buf。如果你在 TxCpltCallback 觸發之前就覆寫它，就會破壞傳輸中的資料——你會看到撕裂或交錯的輸出。用一個 tx_busy 旗標（如上）或一個真正的佇列來把關。RX 也是同一條規則：不要以比 DMA 填充更快的速度去解析 rx_buf。

🛠️ 選對模式

阻塞：一次性的設定列印，簡單為上。中斷：短促、突發的流量，且你的 DMA 串流已用完。DMA：高吞吐量的日誌、音訊，或任何必須讓 CPU 保持空閒的場合。一行 1 KB 的日誌若用阻塞會佔住 89 ms，改用 DMA 對 CPU 來說實際成本約為 0 ms。

4STM32 上的 I²C：定址、時序與暫存器映射裝置

① 直覺 兩線式的共用線路

I²C 把每個裝置都接到同樣的兩條開汲極線上——SDA（資料）與 SCL（時脈）——各由一顆電阻上拉至高電位。控制器透過送出一個 7 位元位址加上一個讀／寫位元來開始一次交易；只有位址相符的裝置會回應。這就是一顆 EEPROM、一顆 RTC 與三顆感測器如何在兩個腳位上共存的方式。

② 數學 位址空間與時脈階層

一個 7 位元位址給出 $2^{7} = 128$ 個槽位，但保留範圍把它修剪到一條匯流排上 112 個可用的裝置位址。實際送上線的移位後位元組，是位址置於高 7 位元、R/W̄ 位元置於最低位：

$$ \text{byte} = (\text{addr}_{7} \ll 1)\,|\,\overline{\text{R/W}}, \qquad 0x50 \to 0xA0\ \text{（寫入）},\ 0xA1\ \text{（讀取）} $$

I²C 跑在固定的時脈階層：100 kHz（Standard 標準模式）、400 kHz（Fast 快速模式）與 1 MHz（Fast-mode Plus）。在現代 STM32（F0/F3/F7/L4/G4…）上你不是設一個簡單的除數——而是載入一個計算好的 TIMINGR 暫存器，它以 I²C 核心時脈的週期數編碼了 SCL 高／低期間以及資料的建立／保持時間。CubeMX（或 ST 的時序工具）會為你的時脈產生這個常數；把它當成推導出來的、而非魔法。

③ 程式碼 殺手級便利：Mem_Read/Write

大多數 I²C 晶片都是暫存器映射的：要讀一個值，你得先寫入暫存器位址，送出一個重複起始（repeated-start），然後才讀取。HAL 把這整套舞步打包進 HAL_I2C_Mem_Read/Write——你給它裝置位址與內部暫存器，它就替你處理重複起始。

eeprom_i2c.c C / STM32 HAL
#define EE_ADDR  (0x50 << 1)   // 7 位元 0x50，HAL 要的是左移後的值
#define EE_MEMSZ I2C_MEMADD_SIZE_16BIT  // 24LC256 使用 16 位元字組位址

// 將一頁（<=64 B）寫入 24LC256 EEPROM 的內部位址 `at`。
HAL_StatusTypeDef ee_write(uint16_t at, uint8_t *data, uint16_t len) {
  return HAL_I2C_Mem_Write(&hi2c1, EE_ADDR, at, EE_MEMSZ,
                           data, len, HAL_MAX_DELAY);
}

// 讀回 `len` 個位元組。Mem_Read 會做：寫位址 + 重複起始 + 讀取。
HAL_StatusTypeDef ee_read(uint16_t at, uint8_t *data, uint16_t len) {
  return HAL_I2C_Mem_Read(&hi2c1, EE_ADDR, at, EE_MEMSZ,
                          data, len, HAL_MAX_DELAY);
}

// EEPROM 需要幾毫秒來提交一頁——輪詢直到它再次 ACK。
void ee_wait_ready(void) {
  while (HAL_I2C_IsDeviceReady(&hi2c1, EE_ADDR, 3, 10) != HAL_OK) { /* 空轉 */ }
}
⚠️ 陷阱——移位陷阱與缺少上拉電阻

HAL 的 I²C API 預期位址是已經左移過的（R/W̄ 位元由內部設定）。傳入原始的 0x50，你就會定址到錯的裝置或拿到 HAL_ERROR。第二個經典問題：I²C 線是開汲極的——沒有外部上拉電阻（通常 4.7 kΩ）時匯流排永遠到不了邏輯高電位，每一次傳輸都會逾時。匯流排看起來像死了，而不是壞了。

5STM32 上的 SPI：分頻器、模式與全雙工

① 直覺 四條線，純粹的速度

SPI 以 I²C 的省線換取速度。四條訊號——SCK（時脈）、MOSI（控制器→周邊）、MISO（周邊→控制器）、CS（晶片選擇，每個裝置一條）——而且沒有定址的額外開銷。在同樣的時脈邊緣上，資料於 MOSI 移出且於 MISO 移入：全雙工，每個時脈各向承載一個位元。

② 數學 鮑率分頻器

SPI 沒有小數分頻器——只有 CR1 裡一個 3 位元的分頻器 BR，它把匯流排時脈除以 2 的冪：

$$ f_{SPI} = \frac{f_{PCLK}}{2^{\,(\text{BR}+1)}}, \qquad \text{BR} \in \{0,1,\dots,7\} $$

所以 BR 選出 /2、/4、/8、… /256 的分頻。在一顆 STM32F4 上，SPI 掛在 PCLK2 = 84 MHz，選 BR = 2 得到

$$ f_{SPI} = \frac{84\times10^{6}}{2^{(2+1)}} = \frac{84\times10^{6}}{8} = 10.5\ \text{MHz} $$

想要更快？BR = 1 → /4 → 21 MHz；BR = 0 → /2 → 42 MHz。上限是線上最慢的那個裝置所能承受的速率——查它資料手冊上的最大 SCK。

SPI 的後半段是模式：兩個位元，CPOL（時脈閒置極性）與 CPHA（在第一個還是第二個邊緣取樣），挑出你要使用的四種時序慣例之一。兩端必須完全一致，否則你讀到的就是位移、扭曲的位元。

③ 程式碼 全雙工傳輸
spi_imu.c C / STM32 HAL
// SPI1 掛在 PCLK2 = 84 MHz。分頻 /8 -> 10.5 MHz；ICM-20948 用 Mode 3。
SPI_HandleTypeDef hspi1;

void spi1_init(void) {
  __HAL_RCC_SPI1_CLK_ENABLE();
  hspi1.Instance               = SPI1;
  hspi1.Init.Mode              = SPI_MODE_MASTER;
  hspi1.Init.Direction         = SPI_DIRECTION_2LINES;     // 全雙工
  hspi1.Init.DataSize          = SPI_DATASIZE_8BIT;
  hspi1.Init.CLKPolarity       = SPI_POLARITY_HIGH;        // CPOL = 1
  hspi1.Init.CLKPhase          = SPI_PHASE_2EDGE;          // CPHA = 1  -> Mode 3
  hspi1.Init.NSS               = SPI_NSS_SOFT;             // 以軟體驅動 CS
  hspi1.Init.BaudRatePrescaler = SPI_BAUDRATEPRESCALER_8;  // BR=2 -> /8 -> 10.5 MHz
  hspi1.Init.FirstBit          = SPI_FIRSTBIT_MSB;
  HAL_SPI_Init(&hspi1);
}

// 讀取一個 IMU 暫存器：送出 (reg | 0x80) 以請求讀取，再時脈出一個
// 虛擬位元組以把答案移回來。CS 為低有效，由手動驅動。
uint8_t imu_read(uint8_t reg) {
  uint8_t tx[2] = { (uint8_t)(reg | 0x80), 0x00 };
  uint8_t rx[2] = { 0 };
  HAL_GPIO_WritePin(CS_PORT, CS_PIN, GPIO_PIN_RESET);     // CS 拉低：選取
  HAL_SPI_TransmitReceive(&hspi1, tx, rx, 2, HAL_MAX_DELAY); // 全雙工
  HAL_GPIO_WritePin(CS_PORT, CS_PIN, GPIO_PIN_SET);       // CS 拉高：取消選取
  return rx[1];                                            // byte 0 是命令回音
}
⚠️ 陷阱——錯的模式會讀出看似合理的亂碼

如果 CPOL/CPHA 與裝置不相符，你不會拿到錯誤——你會拿到數字，只是位移了一個時脈邊緣，所以一個本該讀到 0xEA 的 WHO_AM_I 會回來變成 0xD5 之類。永遠先驗證晶片的身分暫存器；一個正確的 WHO_AM_I 證明你的模式、分頻器與接線在你信任任何資料之前都已正確無誤。

🔑 決策法則

要與人／PC／數據機跨距離溝通？UART。要在兩個腳位上接很多慢速晶片？I²C。需要從 flash、顯示器或 IMU 取得純粹的吞吐量？SPI。而且，只要傳輸量大或是週期性的，無論用哪條匯流排都伸手去拿 DMA——那正是 STM32 對 AVR 的超能力。

7真實案例：一塊板，三條匯流排，同時全開

④ 真實案例 這會出貨到產品裡

想像一個飛控板或一塊機器人的感測器板。STM32 不是在挑一條匯流排——而是同時跑全部三條，每一條都對應到它的工作：

  • IMU 接 SPI，10.5 MHz（BR=2，/8）。姿態資料必須又快又頻繁地到達——SPI 的全雙工吞吐量讓一個 1 kHz 的控制迴圈持續有料可吃。
  • EEPROM 接 I²C，400 kHz。校正常數與設定都很小、也很少讀取；用兩條線與一顆 RTC 和一顆氣壓計共用，是節儉之選。
  • 除錯主控台接 USART，115200 鮑，透過 DMA 記錄日誌。遙測資料持續串流出去，卻從不從控制迴圈偷走一個時脈週期。

🏭 真實案例——為什麼主控台跑在 DMA 上

在飛控板上，控制迴圈是神聖的：抖動是敵人。透過 UART 的阻塞式 printf 會在位元組排空時讓那個迴圈停滯數毫秒。把主控台繞經 UART TX DMA，意味著 CPU 寫一個緩衝區指標就走人——DMA 引擎時脈出數 KB 的遙測資料，而 1 kHz 的迴圈不受打擾地滴答前進。這個單一的設計選擇，就是一個穩定的自動駕駛儀，與一個每次你記錄一行就抽動的自動駕駛儀之間的差別。

🛠️ 軟體中的匯流排仲裁

每個周邊實例都是獨立的，但它們可能會搶奪 DMA 串流與中斷優先序。給時間關鍵的匯流排（此處是 IMU 的 SPI 與其 DMA）一個比主控台 UART 更高的 NVIC 優先序。匯流排在平行的硬體中運行；你的工作只是排定當兩個中斷同時觸發時誰勝出。

Q1 觀念檢查
一個以標準非同步模式運行的 USART 使用 16 倍過取樣。接收端在每個位元的那 16 個內部時脈刻度裡在做什麼？而三條匯流排中，哪一條本質上是全雙工的？
  • 它把每個位元送 16 次以增加冗餘；I²C 是全雙工。
  • 它在每個位元的中央附近（≈ 16 個刻度中的第 8 個）取樣以容忍鮑率誤差；SPI 是全雙工。
  • 它把資料以快 16 倍的速度時脈到線上；UART 是全雙工，但 I²C 不是。
  • 過取樣設定奇偶校驗位元；三條匯流排同等全雙工。
16 倍過取樣讓接收端鎖定起始位元的邊緣，然後在每個位元的中點附近（約 16 個刻度中的第 8 個）取樣，對 ±2% 的鮑率失配留出餘裕。SPI 是全雙工——MOSI 與 MISO 在每個時脈各向承載一個位元。I²C 是半雙工（單一共用的 SDA 線），而雖然 UART 有分開的 TX/RX 線，本題鎖定的是同步匯流排：在 I²C/SPI 之中，SPI 才是全雙工的那一個。
Q2 算給我看
一個 USART 由 $f_{CK} = 84\,\text{MHz}$ 供時脈，你想要 115200 鮑且使用 16 倍過取樣。計算精確的 USARTDIV（四捨五入前的小數分頻器）。回想 $\text{USARTDIV} = \dfrac{f_{CK}}{16\cdot\text{baud}}$。
USARTDIV
$\text{USARTDIV} = \dfrac{84\times10^{6}}{16 \cdot 115200} = \dfrac{84\times10^{6}}{1\,843\,200} \approx 45.57$。你程式化尾數 $45$ 與小數 $\lfloor 0.5729\times16 \rceil = 9$，得到 $\text{BRR}=0x2D9$ 與實際鮑率 $115226$——約 $+0.02\%$ 誤差，遠在 ±2% 預算之內。（順便檢查 SPI 那一側：在 $f_{PCLK}=84\,\text{MHz}$ 且 $\text{BR}=2$ 時，$f_{SPI}=84/2^{3}=10.5\,\text{MHz}$。）
Q3 程式挑戰——非阻塞遙測
寫一個函式 send_telemetry()，把一行像 "t=12345 roll=1.20\r\n" 的內容格式化進一個緩衝區，並透過 USART1 經 DMA 送出，且立即回傳。當前一次 DMA 傳送仍在進行中時，它必須拒絕開始一次新傳輸（不可有緩衝區毀損）。使用 HAL。
提示
保留一個 volatile uint8_t tx_busy 旗標。在呼叫 HAL_UART_Transmit_DMA 之前設定它；在 HAL_UART_TxCpltCallback 中清除它。如果 send_telemetry() 被呼叫時 tx_busy 已被設定，就直接丟棄這一行（或回傳一個「忙碌」狀態），而不要覆寫 DMA 仍在讀取的緩衝區。
顯示一個解答
telemetry.c C / STM32 HAL
static uint8_t tx_buf[64];
static volatile uint8_t tx_busy = 0;

// 若該行已排入佇列回傳 1，若 DMA 傳送仍在進行中回傳 0。
int send_telemetry(uint32_t t_ms, float roll) {
  if (tx_busy) return 0;                 // 前一次 DMA 仍在讀取 tx_buf
  int n = snprintf((char*)tx_buf, sizeof tx_buf,
                   "t=%lu roll=%.2f\r\n", (unsigned long)t_ms, roll);
  if (n <= 0) return 0;
  tx_busy = 1;                           // 在武裝 DMA「之前」先佔住緩衝區
  if (HAL_UART_Transmit_DMA(&huart1, tx_buf, (uint16_t)n) != HAL_OK) {
    tx_busy = 0;                         // 武裝失敗 -> 釋放緩衝區
    return 0;
  }
  return 1;                              // DMA 串流它；CPU 現在自由了
}

// HAL 從 DMA 傳輸完成的 ISR 觸發此函式。
void HAL_UART_TxCpltCallback(UART_HandleTypeDef *huart) {
  if (huart->Instance == USART1) tx_busy = 0;  // 緩衝區可重新使用
}

tx_busy 旗標就是整堂課的精華：DMA 非同步地從 SRAM 讀取 tx_buf，所以你必須在整段傳輸期間擁有這個緩衝區。在武裝之前佔住它、並且只在完成回呼中釋放它，使這個交接無競爭。一個正式上線的系統會用一個環形緩衝區取代「丟棄該行」的策略，這樣就不會丟失任何資料。

🎓 本章回顧

你現在能從 $f_{CK}$ 與目標鮑率推導 USARTDIV、程式化 BRR，並對 ±2% 的誤差預算進行推理；用中斷或 DMA 把 UART 流量移離 CPU；用 HAL_I2C_Mem_Read/Write 與暫存器映射的 I²C 裝置溝通，並遵守位址移位與上拉電阻的規則；用 $f_{SPI}=f_{PCLK}/2^{(\text{BR}+1)}$ 設定 SPI 的大小並匹配 CPOL/CPHA；以及依速度、線數、定址與距離在三條匯流排之間做選擇。接下來我們會在期限壓力下，讓這些匯流排去驅動真正的感測器與致動器。
