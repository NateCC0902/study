暫存器層級的 GPIO：MODER、BSRR 與 HAL

在 Arduino 上，你寫下 digitalWrite(13, HIGH)，一支腳位就拉成高電位。那一行程式碼藏了十幾個硬體位元。在 STM32 上，我們要把引擎蓋完全掀開：透過寫入晶片真正會讀取的確切暫存器來設定一支腳位，並了解為什麼其中一個暫存器 ── BSRR ── 正是「正確韌體」與「只在中斷觸發時才咬你一口的海森堡蟲（heisenbug）」之間的分水嶺。接著我們會用 HAL 做出同樣的閃爍，並精準掌握 HAL 究竟替我們做了什麼。

⏱ 約 45 分鐘🧠 暫存器模型⚡ 裸機 GPIO🔧 HAL vs CMSIS

1埠、腳位與暫存器檔案

① 直覺 先建立整體圖像

STM32 的腳位不是讓你一支一支單獨定址的散亂電線。它們被組織成每組 16 支腳位的埠（port）：GPIOA、GPIOB……一直到最大型號上的 GPIOK。我們整章都會緊盯的那支腳位 ── Nucleo-F446 開發板上的綠色使用者 LED ── 是 PA5：A 埠、第 5 號腳位。

💡 一句話模型

每個 GPIO 埠都是一小塊記憶體映射（memory-mapped）的暫存器；設定或驅動一支腳位，無非就是把正確的位元寫進該埠基底位址上正確的暫存器裡。

因為埠以 16 支為一組，硬體把全部 16 支腳位的控制塞進少數幾個 32 位元暫存器。每支腳位 2 位元 × 16 支腳位 = 32 位元 ── 剛好就是一個暫存器。這個算術不是巧合；這就是為什麼 MODER、OSPEEDR 和 PUPDR 全都是「每支腳位 2 位元」的暫存器，而 OTYPER 和 IDR 因為每支腳位只需 1 位元，所以上半部留著沒用。

在程式碼裡，一個埠就是一個疊在它基底位址上的 C struct。CMSIS（供應商提供的標頭檔集合）把 GPIOA 給你當成指向該結構的指標，於是 GPIOA->MODER 會直接解參考到晶片上。以下是你即將遇到的暫存器檔案，依宣告順序列出：

暫存器位元 / 腳位作用第 n 支腳位使用
MODER2方向：輸入 / 輸出 / 替代 / 類比位元 [2n+1 : 2n]
OTYPER1輸出型式：推挽（push-pull）/ 開漏（open-drain）位元 [n]
OSPEEDR2轉換率（slew-rate）/ 驅動速度位元 [2n+1 : 2n]
PUPDR2內部上下拉：無 / 上拉 / 下拉位元 [2n+1 : 2n]
IDR1輸入資料（讀取當下的腳位）位元 [n]（唯讀）
ODR1輸出資料（被鎖存的驅動值）位元 [n]
BSRR2對 ODR 位元進行原子式的設定 / 重設設定：位元 [n] · 重設：位元 [n+16]
AFRL/AFRH4哪個周邊擁有這支腳位（AF0–AF15）每支腳位一個 4 位元半位元組（nibble）
⚠️ 陷阱 ── 埠出生時沒有時脈

正如我們在第 02 章反覆強調的，每個周邊喚醒時時脈都是關閉的。在啟用 GPIOA 時脈之前就去讀 GPIOA->MODER，你會得到一堆 0，而你寫進去的值會悄無聲息地消失。任何 STM32 程式裡，GPIO 的第一行永遠是 RCC 的時脈啟用。

2MODER：每支腳位兩位元的數學

② 數學 推導位元位置

MODER 是決定一支腳位是什麼的暫存器。每支腳位有一個 2 位元欄位，四個值分別是：

值模式意義
00輸入高阻抗（Hi-Z）數位輸入（透過 IDR 讀取）
01輸出通用輸出（透過 ODR/BSRR 驅動）
10替代（Alternate）某個周邊擁有這支腳位（計時器、UART、SPI……）
11類比（Analog）ADC/DAC，或最低功耗的「斷開」

第 $n$ 支腳位的兩個位元位於位置 $2n$（低位）與 $2n+1$（高位）。整條規則就是這樣：

$$ \text{MODER field for pin } n \;=\; \text{bits } [\,2n+1 : 2n\,] $$

對於我們在 PA5 上的 LED，代入 $n=5$：

$$ 2n = 10, \qquad 2n+1 = 11 \;\;\Rightarrow\;\; \text{PA5 uses MODER bits } [\,11:10\,] $$

要把 PA5 設為輸出，我們需要在該欄位填入值 01。以原始的 32 位元字組來看，那就是把 $1$ 左移進位元 10：

$$ \texttt{0b01} \ll 10 \;=\; 1 \times 2^{10} \;=\; 1024 \;=\; \texttt{0x00000400} $$
∑ 讀取-修改-寫回（read-modify-write）慣用法

你不能直接把 1024 寫入 MODER ── 那會把其餘 15 支腳位的欄位也清成 00。你必須先清除欄位，再設定它：先 MODER &= ~(0b11 << 10) 把 PA5 的兩個位元都歸零，再 MODER |= (0b01 << 10) 寫入新值。先清除再設定，是任何多位元欄位的通用模式。

把這個位移一般化，你就得到了每一個暫存器巨集所建立的公式。對於 2 位元欄位，第 $n$ 支腳位的位移是 $2n$；對於 4 位元的 AFR 半位元組則是 $4n$（再考慮低/高的拆分）。把這一家子背起來：

$$ \text{shift}_{\,k\text{-bit field}}(n) \;=\; k \cdot n, \qquad \text{mask} \;=\; \big(2^{k}-1\big) \ll k n $$

3BSRR：為什麼原子式寫入很重要

① 直覺 你看不見的競爭條件

一旦腳位成為輸出，就有兩種方式可以改變它的電位。最顯而易見的是 ODR ── 輸出資料暫存器，每支腳位 1 位元。要設定 PA5，你會寫 GPIOA->ODR |= (1 << 5)。看起來人畜無害。它其實是個陷阱。

|= 不是單一的機器操作。CPU 必須把整個 32 位元的 ODR 讀進一個暫存器，OR 上你的位元，再把結果寫回去 ── 三個步驟。如果在這個讀取與寫入之間有中斷觸發，而該 ISR 又剛好碰到同一個埠上的某支腳位，你的寫回會用一份過時的 ODR 副本，悄無聲息地把 ISR 的變更覆蓋掉。

⚠️ 陷阱 ── 更新遺失（lost-update）競爭

主程式執行 ODR |= (1<<5)。它讀取 ODR，被中斷打斷，ISR 對另一支腳位執行 ODR |= (1<<3) 後返回。主程式恢復執行並把自己那份過時的值寫回 ── 第 3 號腳位剛剛就被清除了。沒有任何錯誤訊息。這個臭蟲一小時才出現一次，在現場發生，永遠不會在你的工作台上重現。這是最常見的裸機 GPIO 臭蟲。

② 數學 分成兩半的暫存器

STM32 用 BSRR ── 位元設定/重設暫存器（Bit Set/Reset Register）── 來解決這個問題。它是 32 位元、拆成兩半，而最關鍵的是它唯寫、無法讀取，所以根本沒有東西可以讀取-修改-寫回：

$$ \underbrace{\text{BR}_{15}\ldots\text{BR}_{0}}_{\text{bits }[31:16]\;\text{RESET}} \;\;\Big|\;\; \underbrace{\text{BS}_{15}\ldots\text{BS}_{0}}_{\text{bits }[15:0]\;\text{SET}} $$

對位元 $n$（低半部）寫入 1 會設定第 $n$ 支腳位。對位元 $n+16$（高半部）寫入 1 會重設它。在任何地方寫入 0 都不會有任何作用。所以對 PA5 而言：

$$ \text{set PA5: } \texttt{BSRR} = 1 \ll 5 = \texttt{0x0020}; \qquad \text{reset PA5: } \texttt{BSRR} = 1 \ll (5{+}16) = \texttt{0x0020\,0000} $$
🔑 為什麼這在構造上就是正確的

對 BSRR 的一次寫入是一條單一儲存指令（store），硬體會以原子方式套用它 ── 它只會碰到你寫入 1 的那些位元，其他每支腳位都不動。沒有讀取步驟，所以也沒有讓中斷得以破壞的時間窗。請一律用 BSRR 驅動輸出，絕不要用 ODR |=。 HAL 的 HAL_GPIO_WritePin 內部用的正是 BSRR，原因就在這裡。

你甚至可以在一次寫入中對不同腳位同時設定與重設：BSRR = (1<<5) | (1<<(3+16)) 會同時設定 PA5 並清除 PA3 ── 一次原子式儲存，兩支腳位。如果你在同一次寫入中對同一支腳位同時設定與重設，重設勝出：高半部具有優先權。

🛠️ 位址是從哪來的？

在 STM32F446 上，GPIOA 的基底是 0x40020000，MODER 在偏移 0x00、IDR 在 0x10、ODR 在 0x14、BSRR 在 0x18。你永遠不會硬寫死這些值 ── CMSIS 標頭檔 stm32f4xx.h 宣告了 GPIO_TypeDef 結構，讓編譯器去計算它們。GPIOA->BSRR 就是 *(volatile uint32_t*)(0x40020000 + 0x18)。

5從暫存器讓 PA5 閃爍，逐行詳註

③ 程式碼 逐行閱讀

現在我們把一切組合起來：啟用時脈、把 PA5 設定成無上下拉的推挽輸出，並用 BSRR 永遠切換它。這是純 CMSIS ── 沒有 HAL、沒有 Arduino ── 而且正是晶片實際執行的內容。

blink_reg.c C / CMSIS
#include "stm32f4xx.h"   // CMSIS：GPIOA、RCC、暫存器位元遮罩

#define LED_PIN  5       // PA5 = Nucleo-F446 上的綠色使用者 LED

void delay(volatile uint32_t n) { while (n--) __NOP(); }

int main(void) {
    // 1) 時脈：GPIOA 掛在 AHB1 匯流排上。要先啟用它的時脈，
    //    否則底下每一個寫入都會被未供時脈的周邊吞掉。
    RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN;

    // 2) 模式：把 PA5 的 MODER[11:10] 設為 0b01（輸出）。
    //    先清除這個 2 位元欄位，再 OR 上新值。
    GPIOA->MODER &= ~(0x3u << (2 * LED_PIN));   // 清除位元 11:10
    GPIOA->MODER |=  (0x1u << (2 * LED_PIN));   // 01 = 輸出

    // 3) 輸出型式：0 = 推挽（同時能驅動高電位「與」低電位）。
    GPIOA->OTYPER &= ~(0x1u << LED_PIN);

    // 4) 上下拉：00 = 無上拉/下拉（LED 兩者皆不需要）。
    GPIOA->PUPDR &= ~(0x3u << (2 * LED_PIN));

    while (1) {
        // 設定 PA5 ── BSRR 的低半部。一次原子式儲存，無讀取。
        GPIOA->BSRR = (1u << LED_PIN);          // == 0x00000020
        delay(800000);
        // 重設 PA5 ── BSRR 的高半部，位元（pin + 16）。
        GPIOA->BSRR = (1u << (LED_PIN + 16));   // == 0x00200000
        delay(800000);
    }
}
💡 每個區塊真正做了什麼

(1) 翻動 RCC->AHB1ENR 裡的一個位元，把時脈接到 GPIOA。(2) 對 2 位元的 MODER 欄位先清除再設定，讓腳位成為受驅動的輸出。(3) 與 (4) 選擇推挽以及不接內部電阻。迴圈從不呼叫任何函式來「寫入」腳位 ── 它只是把一個常數存入 BSRR。那一次儲存就是 digitalWrite，而且是原子式的。

⚠️ 陷阱 ── 缺少 volatile

delay() 的計數器刻意宣告為 volatile。少了它，會做最佳化的編譯器看見一個算不出任何可觀察結果的迴圈，就會把它整個刪掉 ── 你的 LED 看起來永遠不閃。CMSIS 的暫存器結構本身就已宣告為 volatile，這正是為什麼 GPIOA->BSRR = … 永遠不會被最佳化掉。

6用 HAL 做同一個閃爍 ── 以及 HAL 藏起了什麼

③ 程式碼 結果相同，視角不同

STM32 的 HAL（硬體抽象層，Hardware Abstraction Layer）把那些暫存器包進具名的結構與函式裡。以下是完全相同的閃爍。注意每一行 HAL 如何對應到你現在已經理解的暫存器寫入。

blink_hal.c C / STM32 HAL
#include "stm32f4xx_hal.h"

int main(void) {
    HAL_Init();
    // __HAL_RCC_GPIOA_CLK_ENABLE() 展開後就是先前那「同一行」：
    //   RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN;  （外加一次回讀以同步）
    __HAL_RCC_GPIOA_CLK_ENABLE();

    GPIO_InitTypeDef cfg = {0};
    cfg.Pin   = GPIO_PIN_5;              // 位元遮罩 1<<5，而非數字 5
    cfg.Mode  = GPIO_MODE_OUTPUT_PP;     // -> MODER=01、OTYPER=0（推挽）
    cfg.Pull  = GPIO_NOPULL;             // -> PUPDR=00
    cfg.Speed = GPIO_SPEED_FREQ_LOW;     // -> OSPEEDR=00
    HAL_GPIO_Init(GPIOA, &cfg);          // 替我們做「先清除再設定」

    while (1) {
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);  // 讀 ODR、寫 BSRR
        HAL_Delay(500);                          // 經由 SysTick 延遲 500 ms
    }
}
🔑 HAL 是一層薄手套，不是黑盒子

HAL_GPIO_Init 對 MODER/OTYPER/OSPEEDR/PUPDR 所做的，正是你親手寫的那個「先清除再設定」。HAL_GPIO_WritePin 寫入 BSRR（設定在低半部、重設在高半部）。HAL_GPIO_TogglePin 讀取目前的 ODR 位元，再寫入 BSRR 相反的那一半。懂了這些暫存器，你就能讀懂 HAL 的原始碼、為它除錯，並在計較週期數的地方繞過它。

⚠️ 陷阱 ── GPIO_PIN_5 是遮罩，不是索引

HAL 的 GPIO_PIN_5 是 (1 << 5) = 0x20，是一個位元遮罩 ── 不是整數 5。在 HAL 要遮罩的地方傳入數字 5，你會去設定到第 2 號腳位和第 0 號腳位（因為 5 = 0b101）。裸機那一側用的是索引 5 搭配明確的位移；HAL 用的是事先位移好的遮罩。別把兩者搞混。

7替代功能與讀取按鈕

① 直覺 把腳位交給周邊

到目前為止，PA5 都只是一支由軟體驅動的普通輸出。但同一支腳位也可以改接到某個周邊：計時器的 PWM 通道、UART 的 TX 線、SPI 時脈。這就是 MODER = 10（替代）的意思 ── 而 AFRL/AFRH 則選擇腳位要路由到 16 種可能替代功能（AF0–AF15）中的哪一個。

每支腳位有一個 4 位元的半位元組。0–7 號腳位位於 AFRL（低位暫存器），8–15 號腳位位於 AFRH。PA5 的替代功能 1（AF1）是 TIM2_CH1 ── 我們在後面章節做 PWM 時會用到的計時器通道。要把 PA5 路由到某個周邊，你要把 MODER=10 並寫入 AF 半位元組：

$$ \text{AFR nibble shift for pin } n<8 \;=\; 4n \quad\Rightarrow\quad \text{PA5} \to \text{AFRL bits }[23:20] $$
af_pwm_pin.c C / CMSIS
// 把 PA5 路由到 TIM2_CH1（替代功能 AF1）以使用硬體 PWM。
GPIOA->MODER  &= ~(0x3u << (2 * 5));   // 清除 MODER[11:10]
GPIOA->MODER  |=  (0x2u << (2 * 5));   // 10 = 替代（ALTERNATE）功能

GPIOA->AFR[0] &= ~(0xFu << (4 * 5));   // AFR[0] 即 AFRL；清除半位元組[23:20]
GPIOA->AFR[0] |=  (0x1u << (4 * 5));   // AF1 = TIM2_CH1
// 從這裡開始，是計時器（而非 CPU）擁有這支腳位的電壓。
③ 程式碼 現在來讀一個輸入

讀取是反過來的鏡像。把腳位設定為輸入（MODER=00），啟用上拉讓線路閒置時為高電位，然後讀取 IDR 的第 $n$ 號位元。Nucleo 板上的使用者按鈕 B1 接到 PC13，按下時把線路拉到接地，所以「上拉 + 按下 = 0」是最自然的讀法。

button.c C / CMSIS
RCC->AHB1ENR |= RCC_AHB1ENR_GPIOCEN;      // 為 C 埠供應時脈

GPIOC->MODER &= ~(0x3u << (2 * 13));       // 00 = 輸入（PC13）
GPIOC->PUPDR &= ~(0x3u << (2 * 13));       // 清除上下拉欄位
GPIOC->PUPDR |=  (0x1u << (2 * 13));       // 01 = 上拉，線路閒置時為高電位

// IDR 位元 13 讀取「當下」的腳位電壓。按鈕按下時把它接地。
if ((GPIOC->IDR & (1u << 13)) == 0) {
    // 已按下：腳位被對抗上拉而拉到 0 V
}
🛠️ 為什麼上下拉電阻是必要的

一支沒有任何東西在驅動的裸輸入腳位是浮接（floating）的 ── 它的電壓會隨雜訊飄移，IDR 會回傳隨機的 1 與 0。PUPDR 的內部上拉（約 40 kΩ）把線路綁到一個已知電位，讓斷開的開關讀到乾淨的 1、按下時讀到 0。沒有上下拉，就沒有可靠的讀取。

8實際案例：一支腳位能切換多快？

④ 實際案例 這會出貨進產品

「切換一支腳位」聽起來微不足道，直到你需要去位元敲擊（bit-bang）一個協定 ── 親手驅動一條時脈線、一條 NeoPixel 燈條或一條軟體 SPI 線，因為沒有空閒的周邊可用。這時最大切換頻率就成了你能跟什麼東西溝通的硬性上限。

在 STM32F446 上，CPU 最高跑到 $f_{\text{clk}} = 180\,\text{MHz}$，而 GPIO 掛在 AHB1 匯流排上以完整的 HCLK 運作。從 CPU 的角度看，一次 BSRR 寫入是單週期的儲存。在一個交替設定/重設的緊湊組合語言迴圈裡，實務上的天花板大約是 HCLK 除以每次切換所需的週期數：

$$ f_{\text{toggle}} \;\approx\; \frac{f_{\text{clk}}}{\text{cycles per edge}} \;=\; \frac{180\,\text{MHz}}{\sim 2} \;\approx\; 90\,\text{MHz (best case, asm loop)} $$

一個實際的 C 語言 while(1){ BSRR=set; BSRR=reset; } 迴圈，把迴圈額外負擔算進去，會落在大約 $10\text{–}45\,\text{MHz}$，視快閃記憶體的等待狀態（wait-state）與指令快取而定。現在拿來跟 Arduino Uno 的 digitalWrite() 比一比：它每次呼叫都要做腳位對埠的查表、關閉中斷，以及一次讀取-修改-寫回 ── 在 16 MHz 下約 50 個 CPU 週期，所以不到 約 150 kHz。在 AVR 上做直接埠寫入（PORTB ^= 1<<5）可達約 2–4 MHz。暫存器層級的 STM32 快了兩到三個數量級。

🏭 實際案例 ── 位元敲擊一條 WS2812 LED 燈條

WS2812「NeoPixel」LED 沒有時脈線；每個位元純粹靠資料腳位維持高電位的時間長短來編碼 ── 1 是約 0.8 µs 高電位，0 是約 0.4 µs 高電位，容差 ±150 ns。要打中那個時序，意味著要在 BSRR 寫入之間數 CPU 週期。在 16 MHz 的 AVR 上這緊到令人痛苦，而且必須手寫組合語言；在 180 MHz、有單週期 BSRR 的 STM32 上，你有用不完的週期 ── 這正是為什麼高速 LED 燈條驅動程式都轉向了 ARM。上面那段切換速度的數學，就是決定你的協定在軟體裡到底可不可行的預算。

Q1 觀念檢核
為什麼設定一支腳位時，GPIOA->BSRR = (1<<5) 比 GPIOA->ODR |= (1<<5) 更好？
  • BSRR 比較快，因為它跳過了時脈啟用步驟
  • BSRR 是單一原子式寫入，所以中斷無法破壞讀取-修改-寫回
  • ODR 每個電源週期只能寫入一次
  • BSRR 會把更多電流驅動到腳位上
ODR |= 是一個讀取-修改-寫回：讀 ODR、OR 上位元、寫回。在讀取與寫入之間落下的中斷可能改變該埠上的另一支腳位，而過時的寫回又把它覆蓋掉。BSRR 是唯寫的，並以單一原子式儲存套用，只碰到你設定的那些位元 ── 沒有讀取、沒有競爭。
Q2 算一算
在 MODER 裡，第 $n$ 支腳位的 2 位元欄位佔據位元 $[\,2n{+}1 : 2n\,]$。要把 PA5 設為輸出，你要在該欄位寫入值 01，也就是 0b01 << X。位移量 X（欄位的低位元位置）是多少？
位元
$2n = 2 \times 5 = 10$，所以 PA5 的欄位是位元 $[11:10]$，位移為 10。因此 0b01 << 10 = 1024 = 0x400，這就是你（在清除位元 11:10 之後）OR 進 MODER 以把 PA5 設為輸出的值。
Q3 程式挑戰 ── 暫存器層級的 PA5 閃爍
從零開始，寫出把 PA5 設為推挽輸出、然後永遠切換它的裸機設定，過程中絕不讀取 ODR。你可以保留一個軟體 state 旗標，並由它挑選要用 BSRR 的哪一半。不准用 HAL。
提示
啟用 GPIOA 時脈（RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN），先清除再設定 MODER[11:10]=01，然後在迴圈裡寫 BSRR = (1<<5) 拉高、寫 BSRR = (1<<21) ── 那就是 1<<(5+16) ── 拉低。每一輪翻轉一個本地的 state，這樣你就永遠不需要回讀腳位。
顯示一種解法
blink_pa5.c C / CMSIS
#include "stm32f4xx.h"
#define PIN 5

static void delay(volatile uint32_t n){ while(n--) __NOP(); }

int main(void) {
    RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN;        // 1) 為 GPIOA 供應時脈

    GPIOA->MODER &= ~(0x3u << (2 * PIN));        // 2) 清除 MODER[11:10]
    GPIOA->MODER |=  (0x1u << (2 * PIN));        //    01 = 輸出
    GPIOA->OTYPER &= ~(0x1u << PIN);            //    推挽
    GPIOA->PUPDR  &= ~(0x3u << (2 * PIN));      //    無上下拉

    uint32_t state = 0;
    while (1) {
        // 由 state 挑選 BSRR 的哪一半：state 為真時用高半部（重設），否則設定
        GPIOA->BSRR = state ? (1u << (PIN + 16))    // 重設：位元 21
                            : (1u <<  PIN);         // 設定：位元 5
        state ^= 1u;
        delay(800000);
    }
}

這堂課的重點：每一次電位改變都是單一的原子式 BSRR 儲存，由一個軟體旗標來挑選 ── 而不是去讀取硬體。把 delay() 換成 SysTick 時間戳（第 02 章），你就有了一個非阻塞的心跳。改用 AFR 把同一支腳位路由到 TIM2_CH1，計時器就會免費替你閃它，而 CPU 還能去睡覺。

🎓 本章回顧

你現在能直接用 CMSIS 設定任何一支 STM32 腳位了：在 RCC->AHB1ENR 啟用它的埠時脈、把位元 $[2n{+}1:2n]$ 處 2 位元的 MODER 欄位設好、在 OTYPER 選推挽/開漏、在 PUPDR 選上下拉。你透過 BSRR 原子式地驅動輸出 ── 設定在低半部、重設在高半部 ── 而且你確切知道為什麼 ODR |= 是個潛伏的競爭。你能用上拉透過 IDR 讀取輸入、用 MODER=10 加上 AFR 半位元組把腳位交給周邊，而且你已經看見同一個閃爍在 HAL 裡不過就是這些換上了名字的暫存器寫入。下一章，我們讓周邊自己替我們做切換 ── 計時器與 PWM。
