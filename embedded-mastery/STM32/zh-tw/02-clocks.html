時脈樹：HSI/HSE/PLL、RCC 與 SysTick

這是初學者在 STM32 上最常搞錯的一件事。晶片開機時跑得很慢，每個周邊出生時都是沉睡的，在你搞懂這些 MHz 從哪裡來之前，你寫的任何東西都不會動。我們將從一顆振盪器開始，一層一層建出整棵時脈樹 — 然後用真實數字與一個真實的除錯故事，推導出 168 MHz。

⏱ 約 45 分鐘🧠 時脈樹⚡ RCC 暫存器🔧 PLL 數學

1晶片為什麼需要時脈

① 直覺 先把整體畫面建立起來

MCU 是一張由同步邏輯構成的巨大網格。它內部幾乎每一個正反器（flip-flop）都在時脈的邊緣更新 — 時脈就是一道為整顆晶片打拍子的方波。沒有時脈，就沒有指令執行、沒有 UART 位元組被移出、沒有計時器在計數。時脈就是心跳，而在 STM32 上時脈不只一個：那是一整棵由衍生時脈構成的樹，每一個都餵給矽片上不同的區域。

💡 一句話模型

在 STM32 上，每個周邊都跑在一個由單一選定來源逐步衍生而來的時脈上 — 而在你明確地把它的時脈打開之前，那個周邊什麼都不做、讀回來都是零、也不消耗任何電力。

這裡有一個讓每個從 Arduino 過來的人都措手不及的地方：Arduino Uno 一醒來就是全速 16 MHz，每根腳位都已就緒。STM32 不是這樣。重置之後，STM32F4 用它的內部振盪器悠哉地跑在 16 MHz，PLL 是關的，GPIO 埠是未供時脈的，而你若去戳某個周邊暫存器，它只會用一堆零回瞪你。晶片交給你的是一台緩慢、大部分還是黑暗的機器，並信任你會把它喚醒。這個喚醒過程就是本章的主題。

🔑 開機時你必須做的兩件工作

(1) 選擇並拉高系統時脈 — 挑一個來源，把它送過 PLL，再把 CPU 切換到它上面。(2) 在動每個周邊之前，透過 RCC 開啟它的時脈。略過第 2 件，周邊在電氣上存在，但功能上是死的。

2四顆振盪器：HSI、HSE、LSI、LSE

① 直覺 原始拍子從哪裡來

在任何相乘或相除之前，這些 MHz 必須先被產生出來。STM32 給你四個原始時脈來源，沿兩個軸向分類：內部 vs. 外部（精度／成本）以及高速 vs. 低速（用來跑 CPU vs. 在睡眠時保持計時）。

來源意義典型頻率準確度用途
HSI高速內部（High-Speed Internal）16 MHz（F4）；8 MHz（F1）±1–2%（RC，隨溫度漂移）開機預設、快速啟動、後備
HSE高速外部（High-Speed External）4–26 MHz 石英晶體（常見 8）±20–50 ppm（晶體）當計時必須精準時作為 PLL 輸入
LSI低速內部（Low-Speed Internal）約 32 kHz RC粗略（±5–10%）獨立看門狗（IWDG）、低功耗 RTC

🔑 內部＝方便，外部＝精準

HSI 是片上的 RC 振盪器 — 免費、即時、零外部零件，但它是一個類比的 R·C 網路，所以頻率會隨溫度與電壓游移。HSE 是你焊上去的石英晶體；石英以機械方式共振，穩定得驚人。LSE 的 32.768 kHz 這個數字並非隨意 — 它是 $2^{15}$，所以一個 15 位元的計數器剛好每秒溢位一次。對於一個以秒為拍的時鐘來說，這正合適。

④ 真實案例 為什麼 USB 逼你用晶體
🏭 真實案例 — USB 不會容忍 HSI

全速 USB 要求它的 48 MHz 時脈落在約 ±0.25% 之內。HSI 的 ±1–2% RC 漂移直接衝破這個預算，所以主機會丟掉這個裝置，或者它只能間歇性地列舉成功。這就是為什麼幾乎每一塊帶 USB 埠的 STM32 板子都會在晶片旁邊放一顆 8 MHz（或 25 MHz）晶體：你需要 HSE 才能可靠地為 USB 供應時脈。RC 振盪器拿來閃個 LED 沒問題；但對一個有著緊湊頻率容差的協定來說，它不夠格。

3PLL：把 8 MHz 變成 168 MHz

① 直覺 一個頻率倍頻器

你的晶體跑在 8 MHz。你的 CPU 想跑在 168 MHz。連接兩者的橋梁就是 PLL（鎖相迴路，Phase-Locked Loop）：一個包含壓控振盪器（VCO）的回授電路，晶片會去調整它，直到它分頻後的輸出剛好對上參考訊號。整體效果就是一個乾淨、可程式化的頻率倍頻器 — 餵進 8 MHz，吐出幾乎任何你想要的頻率。

② 數學 推導 F4 的公式

在 STM32F4 上，PLL 串聯了三個整數旋鈕：一個前置除頻器 $M$、一個倍頻器 $N$，以及一個給系統時脈用的後置除頻器 $P$。訊號就按這個順序流動：

$$ f_{\text{VCO,in}} = \frac{f_{\text{in}}}{M}, \qquad f_{\text{VCO}} = f_{\text{VCO,in}} \cdot N, \qquad f_{\text{SYSCLK}} = \frac{f_{\text{VCO}}}{P} $$

第一階 $M$，把輸入降進 VCO 要求的輸入視窗 1–2 MHz（資料手冊建議剛好落在 2 MHz，但 1 MHz 是合法的，ST 自己的範例也這麼用）。第二階 $N$，倍頻到 VCO 輸出，這個輸出必須維持在 100–432 MHz 之內。第三階 $P \in \{2,4,6,8\}$，再除回核心可以承受的 168 MHz。我們把標準的 F4 數值代進去 $f_{\text{in}}=8\,\text{MHz（HSE）},\,M=8,\,N=336,\,P=2$：

$$ f_{\text{VCO,in}} = \frac{8\,\text{MHz}}{8} = 1\,\text{MHz}, \quad f_{\text{VCO}} = 1\,\text{MHz}\times 336 = 336\,\text{MHz}, \quad f_{\text{SYSCLK}} = \frac{336\,\text{MHz}}{2} = 168\,\text{MHz} $$

兩條限制都過關：VCO 輸入 1 MHz 在 [1, 2] 內，而 VCO 輸出 336 MHz 在 [100, 432] 內。還有第四個旋鈕 $Q$，它從同一個 336 MHz 的 VCO 取出來，做出 USB／SDIO／RNG 的時脈。我們需要 48 MHz，所以：

$$ f_{\text{USB}} = \frac{f_{\text{VCO}}}{Q} = \frac{336\,\text{MHz}}{7} = 48\,\text{MHz} \;\checkmark $$

那唯一共用的 VCO 正是為什麼 ST 選 $N=336$ 而不是比方說 168 配 $P=1$：336 可以被 7 整除，所以同一個迴路同時產出乾淨的 168 MHz 核心以及規格完美的 48 MHz USB。時脈樹是一道約束滿足（constraint-satisfaction）的謎題，而這些就是解開它的魔術數字。

💡 F1 的版本更簡單 — 只有一個倍頻器

較舊的 STM32F1（Blue Pill，「Bluepill」＝STM32F103）沒有 $M/N/P$ 的拆分。它把 $\text{HSE}=8\,\text{MHz}$ 直接餵進單一個倍頻器 $\times 9$：$\;f_{\text{SYSCLK}} = 8\,\text{MHz}\times 9 = 72\,\text{MHz}$，正是 F1 的上限。一樣的概念（把晶體倍頻上去），只是旋鈕較少。

4匯流排除頻器：HCLK、PCLK1、PCLK2 與計時器 ×2 規則

① 直覺 不是所有東西都能跑 168 MHz

核心可以吃 168 MHz，但周邊匯流排沒辦法全部跟上 — 而且把一個慢速 UART 用全速供時脈、白白燒電力，根本沒有意義。所以在 SYSCLK 之後，時脈會經過一連串除頻器（可程式化的「除以 2 的次方」）扇出，產生各個匯流排時脈：

時脈匯流排來源F4 上限供給
HCLKAHBSYSCLK ÷ AHB_PSC168 MHz核心、DMA、記憶體、GPIO
PCLK1APB1（低速）HCLK ÷ APB1_PSC42 MHzUSART2/3、I²C、SPI2/3、TIM2–7
PCLK2APB2（高速）HCLK ÷ APB2_PSC84 MHzUSART1/6、SPI1、ADC、TIM1/8/9–11

對於 168 MHz 的設定，你會設成 AHB ÷1（HCLK = 168）、APB1 ÷4（PCLK1 = 42）、APB2 ÷2（PCLK2 = 84）— 每條匯流排上的最大合法值。

② 數學 那條絆倒每個人的規則

現在來看微妙的那一條。計時器掛在 APB1/APB2 上，但 ST 的矽片玩了一個把戲：如果某個 APB 除頻器大於 1，那條匯流排上的計時器時脈會被加倍。正式地說，餵給計時器計數器的時脈是

$$ f_{\text{TIMx}} = \begin{cases} f_{\text{PCLK}} & \text{若 } \text{APB\_PSC} = 1 \\[4pt] 2 \cdot f_{\text{PCLK}} & \text{若 } \text{APB\_PSC} > 1 \end{cases} $$

所以在 APB1 ÷4 的情況下：$f_{\text{PCLK1}} = 42\,\text{MHz}$，但 TIM2–7 的計數器其實是用 $2\times 42 = 84\,\text{MHz}$ 在計時。在 APB2 ÷2 的情況下：PCLK2 = 84 MHz，但 TIM1/8 看到的是 $2\times 84 = 168\,\text{MHz}$。忘掉這個 2 倍因子，你從 $f_{\text{PCLK}}$ 算出來的每一個 PWM 頻率、每一個 delay_us 都會剛好對一半 — 就是那個經典的「我的伺服機跑兩倍速」的 bug。我們在計時器那一章會大量倚賴這條公式。

5Flash 等待狀態：為什麼快速核心必須等待

① 直覺 記憶體跟不上

跑在 168 MHz 有一個物理上的陷阱。CPU 從片上 Flash 取出每一條指令，而 Flash 很慢 — 它的感測放大器（sense amplifier）大約需要 25–30 ns 才能回傳一個字（word）。在 168 MHz 下，一個時脈週期只有 $1/168\,\text{MHz} \approx 5.95\,\text{ns}$，遠比 Flash 能回應的時間還短。如果核心每個週期都要資料，它讀到的會是垃圾。

解法是等待狀態（wait states）：Flash 控制器每次取指令時讓 CPU 停頓幾個週期，好讓記憶體有時間回應。速度越高 ⇒ 週期越短 ⇒ 需要越多等待狀態。在 F4（3.3 V）上，門檻大約是每約 30 MHz 一個等待狀態的區間，而 168 MHz 落在 5 個等待狀態（FLASH_ACR_LATENCY_5WS）。

⚠️ 陷阱 — 操作順序會弄死開機

你必須先拉高 Flash 延遲，然後才把 SYSCLK 升到 168 MHz。若你在延遲還是 0 的時候先切換時脈，下一條指令的取指就會讀到損毀的 Flash — 核心在你的 main() 跑起來之前就執行了垃圾並當機（或 HardFault）。先把延遲設高，然後再加速。當你要降速回來時，則要在降速之後才降低延遲。

🛠️ ART 加速器把成本藏起來了

5 個等待狀態聽起來很殘忍 — 每 6 個週期才取一次指令？實務上 ST 的 ART 加速器（一個放在 Flash 前面的預取＋指令快取）會持續餵飽管線，所以線性的程式碼仍然以接近零等待狀態的吞吐量在跑。等待狀態是真實存在的；只是快取通常會替你把它們攤平。啟用 FLASH_ACR_PRFTEN | ICEN | DCEN，然後就別再操心了。

6RCC：每個人都會中一次的 bug

① 直覺 周邊預設是被關閘的

這就是那個成年禮等級的 STM32 bug。你寫了完美的 GPIO 程式碼，要在 D 埠上閃一顆 LED。你燒進去。沒反應。沒有錯誤、沒有警告，除錯器顯示你的程式碼在跑 — 而腳位什麼都不做。無論你寫什麼，去讀 GPIOD->MODER 回來都是零。

原因是：為了省電，RCC（重置與時脈控制，Reset and Clock Control）單元在重置時讓每個周邊的時脈都保持關閘狀態。一個未供時脈的周邊是凍結的 — 它的暫存器甚至不會鎖存（latch）你寫進去的值。在你設定任何周邊之前，你必須在對應的 RCC 暫存器裡設定它的啟用位元。

🔑 牢牢記住這條規則

啟用暫存器是以周邊所在的匯流排來命名：GPIO 與 DMA 用 RCC->AHB1ENR，TIM2–7 / USART2 / I²C 用 RCC->APB1ENR，USART1 / SPI1 / ADC 用 RCC->APB2ENR。先設好那個位元，然後才動周邊。絕對不要反過來。

③ 程式碼 逐行細讀
gpio_clock.c C / STM32 (CMSIS)
#include "stm32f4xx.h"   // CMSIS 裝置標頭檔：定義 RCC、GPIOD、位元遮罩

void led_init(void) {
    // 步驟 1 — 打開 GPIO D 埠的時脈。沒有這一行，
    // 下面每一次寫入都會被悄悄丟棄，而 MODER 讀回來是 0。
    RCC->AHB1ENR |= RCC_AHB1ENR_GPIODEN;   // AHB1ENR 的第 3 位元

    // 在這裡讀回一次，強制啟用在我們動周邊之前生效
    //（防範 1 個週期的時脈域間隙）。
    (void)RCC->AHB1ENR;

    // 步驟 2 — 現在周邊活了，把 PD12 設定為輸出。
    // MODER 每根腳位 2 位元：01 = 通用輸出。第 12 腳 → 第 25:24 位元。
    GPIOD->MODER &= ~(0x3u << (12 * 2));    // 先清掉這 2 位元欄位
    GPIOD->MODER |=  (0x1u << (12 * 2));    // 01 = 輸出模式

    // 步驟 3 — 驅動它。BSRR 是不可分割的 set/reset 暫存器：
    // 寫第 12 位元會把 PD12 拉高，沒有讀-修改-寫的競態。
    GPIOD->BSRR = (1u << 12);               // PD12 = HIGH → LED 亮
}
⚠️ 陷阱 — 症狀是徹底的沉默

忘掉 RCC->AHB1ENR 那一行不會產生任何錯誤。編譯器很開心、連結器很開心、除錯器乾淨地單步走過你的 GPIO 寫入 — 而周邊暫存器就是拒絕改變。如果某個周邊暫存器無論你存什麼進去，讀回來都剛好是它在重置時的值，你的第一嫌疑犯永遠是漏掉的 RCC 時脈啟用位元。

7SysTick：從 SYSCLK 產生 1 ms 的心跳

① 直覺 那個永遠都在的計時器

每一個 Cortex-M 核心都內建一個微小的 24 位元向下計數器，叫做 SysTick，它烘焙在 CPU 本身裡（所以在所有 STM32 上都一模一樣）。你載入一個重載值（reload value）；SysTick 以時脈速率從它向下數到零，數到零時觸發一個中斷，重載，然後重複。這是製造週期性「tick」的標準做法 — 也就是 HAL_Delay() 與大多數 RTOS 排程器背後那個 1 ms 的拍子。

② 數學 計算重載值

SysTick（在預設下）以完整的處理器時脈 $f_{\text{SYSCLK}}$ 計數。要每隔 $T$ 秒得到一個中斷，你需要它數 $f_{\text{SYSCLK}}\cdot T$ 個 tick。因為計數器在重載前要多花一個週期通過零，所以你寫進去的值會比 tick 數少一：

$$ \text{RELOAD} = f_{\text{SYSCLK}} \cdot T - 1 $$

對於 1 ms 的 tick，$T = 10^{-3}\,\text{s}$，所以 $\text{RELOAD} = f_{\text{SYSCLK}}/1000 - 1$。在 72 MHz 的 F1 與 168 MHz 的 F4 上：

$$ \text{RELOAD}_{72} = \frac{72\times10^{6}}{1000} - 1 = 71{,}999, \qquad \text{RELOAD}_{168} = \frac{168\times10^{6}}{1000} - 1 = 167{,}999 $$

兩者都裝得進 SysTick 的 24 位元暫存器（上限 $2^{24}-1 = 16{,}777{,}215$），所以無論 STM32 跑多快，1 ms 的 tick 都綽綽有餘。CMSIS 的一行式 SysTick_Config(SystemCoreClock / 1000) 會替你做這個減一，也順便啟用中斷。

③ 程式碼 用暫存器也用輔助函式
systick.c C / STM32 (CMSIS)
#include "stm32f4xx.h"

volatile uint32_t g_ticks = 0;   // 1 ms 計數器，由 ISR 寫入

// CMSIS 的一行式：載入 reload、清掉計數值、選時脈來源、
// 啟用計數器＋中斷。SystemCoreClock 存著你真正的 SYSCLK。
void systick_init_1ms(void) {
    SysTick_Config(SystemCoreClock / 1000u);   // 例如 168000000/1000 = 168000
}                                              //   → RELOAD = 168000 - 1 = 167999

// NVIC 在每次 SysTick 數到零時（每 1 ms）呼叫的處理常式。
void SysTick_Handler(void) {
    g_ticks++;
}

// 建立在這個 tick 上的非阻塞毫秒延遲。
void delay_ms(uint32_t ms) {
    uint32_t start = g_ticks;
    while ((g_ticks - start) < ms) {
        __WFI();   // 睡到下一個中斷 — 比忙等（busy-wait）省電
    }
}
🏭 真實案例 — 那個撐起整個產品的 tick

那個 g_ticks 計數器是真實韌體的脊椎。FreeRTOS 用 SysTick 當它的排程器 tick 來對任務做時間分片；裸機（bare-metal）程式用它在不阻塞的情況下排程「每 10 ms 做一次這件事」的工作；通訊堆疊用它來做逾時與重送。一個 24 位元的計數器，一路從你的 8 MHz 晶體經過 PLL 衍生下來，成了整個裝置對時間的感知。時脈樹搞錯，產品裡的每一個逾時都會以同一個倍率錯掉。

8全部組起來：喚醒序列

🔑 為什麼每個「等待就緒」都重要

振盪器與 PLL 都需要真實的時間去穩定（HSE 啟動是毫秒級；PLL 鎖定是微秒級）。HSERDY / PLLRDY 旗標就是硬體在告訴你「我現在穩了」。在某個來源的就緒旗標還沒設起來之前就切換到它，你就是讓核心跑在一個搖晃、半鎖定的訊號上 — 那是間歇性、令人抓狂、海森堡（heisenbug）等級的失敗。永遠要輪詢就緒位元。

在真實專案裡你幾乎不會手寫這段 — STM32CubeMX 會從一張圖形化的時脈樹圖產生 SystemClock_Config()，而 CMSIS 也附了一個 SystemInit()。但現在你可以讀懂那些產生出來的程式碼、看出數字什麼時候錯了，並在某天 USB 因為有人把來源留在 HSI 而無法列舉時把它修好。

⑤ 練習 你來寫、你來決定

Q1 觀念檢查
你寫程式去設定 GPIOD->MODER，但每次讀那個暫存器都回傳 0，LED 永遠不亮。編譯器與除錯器都沒顯示任何錯誤。最有可能的原因是什麼？
  • 晶體（HSE）啟動失敗，所以 SYSCLK 是 0
  • 你忘了 RCC->AHB1ENR |= RCC_AHB1ENR_GPIODEN — D 埠未供時脈，所以它的暫存器不會鎖存
  • PD12 需要一個外部上拉電阻才能讀回你寫進去的值
  • 對目前的時脈速度而言，Flash 等待狀態太低了
這就是那個經典的 STM32 bug。在重置時 RCC 把每個周邊時脈都關閘以省電；一個未供時脈的周邊是凍結的，它的暫存器會忽略寫入（讀回重置值，通常是 0）。在動周邊之前，先在對應的 RCC xENR 暫存器啟用時脈。一個壞掉的 HSE 會弄死整顆晶片，而不只是 D 埠；上拉電阻不影響輸出寫入；錯誤的等待狀態損毀的是程式取指，不是 GPIO 讀取。
Q2 算出來
一顆 STM32F1 的 SYSCLK 跑在 72 MHz。SysTick 以完整的處理器時脈計數。什麼樣的 RELOAD 值能剛好每 1 ms 觸發一次中斷？（回想一下 $\text{RELOAD} = f_{\text{SYSCLK}}\cdot T - 1$。）
ticks
$\text{RELOAD} = \dfrac{72\times10^{6}}{1000} - 1 = 72{,}000 - 1 = 71{,}999$。減一是為了計數器要通過零。在 168 MHz 下，同樣的 1 ms tick 需要 $168{,}000 - 1 = 167{,}999$。兩者都輕鬆裝進 SysTick 的 24 位元（$\le 16{,}777{,}215$）重載暫存器裡。
Q3 程式挑戰 — 把 SYSCLK 拉到 168 MHz
寫出暫存器層級的序列，把一顆 STM32F4 從它重置時的 HSI 透過 PLL 切換到 用 8 MHz HSE 跑 168 MHz（M=8, N=336, P=2, Q=7），然後啟動一個 1 ms 的 SysTick。把順序弄對：哪一步必須在切換時脈之前、為什麼？設定 AHB÷1、APB1÷4、APB2÷2。
提示
順序就是一切：(1) 啟用 HSE 並輪詢 RCC_CR_HSERDY。(2) 在加速之前把 Flash 延遲拉到 5 WS，否則第一次 168 MHz 的取指會讀到垃圾。(3) 在 RCC->CFGR 裡設好匯流排除頻器。(4) 用 M/N/P/Q 與 HSE 作為來源把 RCC->PLLCFGR 寫好。(5) 啟用 PLL、輪詢 PLLRDY。(6) 選 PLL 作為 SYSCLK 並輪詢 SWS。(7) SysTick_Config(168000000/1000)。注意 P=2 在 PLLCFGR 的 P 欄位裡編碼為 0b00。
顯示一種解答
clock_168.c C / STM32F4 (CMSIS)
#include "stm32f4xx.h"

void clock_config_168mhz(void) {
    // (1) 啟動 HSE（8 MHz 晶體）並等到它穩定。
    RCC->CR |= RCC_CR_HSEON;
    while (!(RCC->CR & RCC_CR_HSERDY)) { }

    // (2) Flash：5 個等待狀態＋預取與快取，在加速之前先設好。
    FLASH->ACR = FLASH_ACR_LATENCY_5WS | FLASH_ACR_PRFTEN
               | FLASH_ACR_ICEN        | FLASH_ACR_DCEN;

    // (3) 匯流排除頻器：AHB /1（HCLK=168）、APB1 /4（42）、APB2 /2（84）。
    RCC->CFGR = RCC_CFGR_HPRE_DIV1 | RCC_CFGR_PPRE1_DIV4
              | RCC_CFGR_PPRE2_DIV2;

    // (4) PLL：M=8、N=336、P=2（欄位=0）、Q=7、來源 = HSE。
    RCC->PLLCFGR = (8u   << RCC_PLLCFGR_PLLM_Pos)   // /M = 8  → 1 MHz
                 | (336u << RCC_PLLCFGR_PLLN_Pos)   // xN = 336 → 336 MHz VCO
                 | (0u   << RCC_PLLCFGR_PLLP_Pos)   // /P: 00 = ÷2 → 168 MHz
                 | (7u   << RCC_PLLCFGR_PLLQ_Pos)   // /Q = 7  → 48 MHz USB
                 | RCC_PLLCFGR_PLLSRC_HSE;          // PLL 輸入 = HSE

    // (5) 打開 PLL 並等待鎖定。
    RCC->CR |= RCC_CR_PLLON;
    while (!(RCC->CR & RCC_CR_PLLRDY)) { }

    // (6) 把 SYSCLK 切到 PLL 並確認切換完成。
    RCC->CFGR |= RCC_CFGR_SW_PLL;
    while ((RCC->CFGR & RCC_CFGR_SWS) != RCC_CFGR_SWS_PLL) { }

    SystemCoreClock = 168000000u;   // 告訴 CMSIS 新的速率

    // (7) 1 ms SysTick：RELOAD = 168000000/1000 - 1 = 167999。
    SysTick_Config(SystemCoreClock / 1000u);
}

最關鍵的細節是步驟 2 要在步驟 6 之前：在還慢的時候先拉高 Flash 延遲，然後才加速。把它們倒過來，第一條在 168 MHz 取出的指令就會回傳損毀內容，核心在 main() 之前就 HardFault。這基本上就是 CubeMX 的 SystemClock_Config() 產出的東西 — 你現在可以讀懂並稽核它了。

🎓 本章回顧

你現在能解釋為什麼 STM32 在 HSI 上開機很慢、在計時必須精準時為什麼選 HSE（以及為什麼 USB 非用它不可），並能從頭到尾推導出一個系統時脈：$8\to{\div}M\to{\times}N\to{\div}P = 168\,\text{MHz}$，再用 $\div Q$ 給出 48 MHz 的 USB。你懂 HCLK/PCLK1/PCLK2 與計時器 ×2 規則、為什麼快速核心需要 Flash 等待狀態（以及要先設好它們）、那個會讓周邊沉默的 RCC 時脈啟用 bug，以及如何把 SYSCLK 變成一個 1 ms 的 SysTick 拍子（$\text{RELOAD}=f/1000-1$）。下一章我們要來花這份時脈預算：在暫存器層級操作 GPIO，又快又不可分割。
