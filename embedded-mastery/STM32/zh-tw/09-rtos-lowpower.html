FreeRTOS 與低功耗：任務、排程與睡眠模式

有兩件事把業餘專案和量產產品區分開來：它能同時處理許多獨立的工作而不漏掉任何一拍，而且靠一顆鈕扣電池就能運作好幾個月。並行性與功耗。這兩樣我們都要拿下——先用搶佔式 RTOS 取代超級迴圈（superloop），再讓晶片深度睡眠，在每次心跳之間只啜飲幾微安的電流。

⏱ 約 55 分鐘🧠 並行性⚡ FreeRTOS🔋 低功耗

1當超級迴圈崩潰時

① 直覺 先建立全貌

到目前為止每一章都活在一個無窮的 while(1) 裡——也就是超級迴圈（superloop）。讀感測器、更新顯示器、閃 LED、檢查按鈕，然後重複。對於少數幾個時序要求寬鬆的工作來說，它很完美：簡單、確定性、沒有隱藏的機制。

但只要你的工作有了各自不同、緊湊且獨立的時序需求，它就會崩潰。想像同一顆晶片上有三個工作：

工作必須執行耗時
透過 SPI 讀取 IMU每 1 ms 一次約 0.2 ms
透過 UART 串流遙測資料每 50 ms 一次約 8 ms（會阻塞！）
處理一次顯示器刷新每 100 ms 一次約 12 ms（會阻塞！）

在超級迴圈中，當你一啟動那個 8 ms 的 UART 傳輸，IMU 的讀取就卡住等待——你會錯過八次 1 ms 的截止時間。你可以手工打造一個狀態機把所有事情交錯處理，但只要需求一變，它就會長成無法維護的一團亂麻。誠實的解法是別再假裝一個迴圈能伺候所有主人。

💡 一句話模型

RTOS 給你許多獨立的「迴圈」（任務，task），它們看起來是同時執行的。排程器（scheduler）在這唯一的 CPU 上快速地切換它們，而搶佔式（preemptive）排程器能在高優先權任務一準備就緒的瞬間，把 CPU 從低優先權任務手中奪走。

「即時（real-time）」不代表「快」——它代表可預測。一個即時系統保證每一次都能達到截止時間，而不是平均能達到。FreeRTOS 是 Cortex-M 上部署最廣泛的 RTOS，它給你建立那份保證所需的原語（primitive）；而 §5 的數學會告訴你，你的保證是否真的成立。

2任務、優先權與搶佔式排程器

① 直覺 任務到底是什麼

FreeRTOS 中的任務（task）其實就是一個永不返回的 C 函式——它永遠迴圈下去，活脫脫就是它自己的一個小型超級迴圈。每個任務都有自己的堆疊（stack）和自己的一份 CPU 暫存器，在每次切換時被儲存與還原。那份被儲存的狀態就是上下文（context）；交換它的動作就是上下文切換（context switch）。

🔑 任務就是一個永遠迴圈的函式

它必須永不 return，也永不執行到結尾的大括號。要嘛用 for(;;) 永遠迴圈，要嘛在掉出函式尾端之前用 vTaskDelete(NULL) 把自己刪除。一個會返回的任務會讓排程器當機。

任務有優先權（priority）（整數；數字越大越緊急，0 = 最低）。在 Cortex-M 移植版上排程器的規則極其簡單：永遠執行已就緒任務中優先權最高的那一個。如果有兩個任務共用最高優先權，它就在每個節拍（tick）上以輪詢（round-robin）方式在它們之間做時間切片。

排程器由一個週期性的計時器中斷驅動——也就是節拍（tick），通常為 1 kHz（每 1 ms 一次），由 SysTick（第 02 章）產生。每個節拍上排程器重新評估該由誰執行。在兩次節拍之間，一個阻塞（block）的任務（等待資料、睡眠、取得已被持有的互斥鎖）會立即讓出 CPU——你不必等到下一個節拍才放棄 CPU。

① 直覺 任務狀態

在任何一個瞬間，每個任務都恰好處於四種狀態之一。掌握這張圖，FreeRTOS 就不再神秘：

🔑 閒置任務是必備的

FreeRTOS 會自動建立一個優先權為 0 的任務：閒置任務（idle task）。只有當所有其他任務都處於阻塞或暫停狀態時，它才會執行。它的職責是釋放被刪除任務的記憶體，而且——對我們至關重要的是——它是我們讓 CPU 進入睡眠的掛鉤點（§7）。如果閒置任務永遠不執行，晶片就永遠不省電。

3建立任務：xTaskCreate、延遲與堆疊

③ 程式碼 逐行讀

你用 xTaskCreate 把一個任務交給排程器：函式指標、一個除錯用的名稱、一個堆疊大小（單位是字（word），不是位元組！）、一個選用的參數、一個優先權，以及一個用來接收回傳控制代碼（handle）的槽位。接著 vTaskStartScheduler() 永不返回——控制權永遠交給了排程器。

main.c C / FreeRTOS
#include "FreeRTOS.h"
#include "task.h"

// 任務就是一個永遠迴圈的函式。pvParameters 是你傳給
// xTaskCreate 的選用參數（這裡未使用）。
void vBlinkTask(void *pvParameters)
{
    const TickType_t period = pdMS_TO_TICKS(500);  // 500 ms 換算成 ticks
    TickType_t wake = xTaskGetTickCount();         // 快照「現在」時間
    for (;;) {                                     // 永不 return
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);     // 切換 LED
        // 睡到剛好 wake+period。固定速率，不漂移。
        vTaskDelayUntil(&wake, period);
    }
}

int main(void)
{
    HAL_Init();
    SystemClock_Config();                          // 見第 02 章
    MX_GPIO_Init();

    // 堆疊單位是「字（WORD）」：128 字 = 32 位元 Cortex-M 上的 512 位元組。
    xTaskCreate(vBlinkTask,        // 函式
                "blink",           // 名稱（僅供除錯）
                128,               // 堆疊深度，單位為字（WORD）
                NULL,              // 參數
                2,                 // 優先權（0 = 最低）
                NULL);             // 控制代碼（我們不需要）

    vTaskStartScheduler();         // 交出 CPU——永不返回
    for (;;) { }                   // 不可達；放在這裡保險
}
⚠️ 陷阱——vTaskDelay 會漂移，vTaskDelayUntil 不會

vTaskDelay(N) 的意思是「從現在起阻塞 N 個節拍」——所以週期會悄悄地隨著你的工作所耗的時間增長。如果迴圈本體執行了 2 ms，而你用 vTaskDelay(pdMS_TO_TICKS(500))，你的真實週期就是 502 ms，而且會漂移。vTaskDelayUntil(&wake, N) 則阻塞到一個絕對的截止時間，所以無論工作耗時多久，速率都穩如磐石。任何週期性取樣都該用它。

⚠️ 陷阱——堆疊溢位是無聲的記憶體破壞

每個任務的堆疊在建立時就固定了。一條很深的呼叫鏈、一個很大的區域陣列，或一個 printf（吃堆疊的胃口超大）都可能讓堆疊衝過邊界，悄悄踐踏下一個任務的記憶體——這是一種會在別處當機的海森堡蟲（heisenbug）。防禦手段有：啟用 configCHECK_FOR_STACK_OVERFLOW = 2（寫入一個已知圖樣並檢查它）、呼叫 uxTaskGetStackHighWaterMark() 看你最接近極限的程度，並且根據量測而非猜測來決定堆疊大小。

4任務間溝通：佇列、號誌與互斥鎖

① 直覺 三種工具，三種職責

任務在設計上是相互隔離的，所以你需要安全的方式來傳遞資料與協調。FreeRTOS 給你三種原語——而選對哪一個是經典的面試題，因為人們老是把它們搞混。

原語會攜帶資料嗎？用於
佇列（Queue）會——複製項目由生產者→消費者傳遞值（執行緒安全的 FIFO）
號誌（Semaphore）（二元 / 計數）不會——只是個計數訊號通知：「某事件發生了」——尤其是 ISR→任務
互斥鎖（Mutex）不會——是所有權令牌互斥（mutual exclusion）：守護單一共享資源（例如 I²C 匯流排）
🔑 號誌 vs 互斥鎖——真正的差別

二元號誌（binary semaphore）是一個任何任務或 ISR 都能「給出（give）」的旗標；正在等待的那一方會被喚醒。它沒有擁有者——非常適合 ISR 對任務的訊號通知。互斥鎖（mutex）則是一個帶擁有者的號誌：只有取得它的任務才可以把它還回去，而且它加上了優先權繼承（priority inheritance）。用號誌來通知，用互斥鎖來保護。

③ 程式碼 ISR 透過號誌通知任務

典範模式：一個中斷觸發了，但你絕不該在 ISR 裡做繁重的工作。取而代之，ISR Give（給出）一個號誌，而一個在它上面阻塞等待的高優先權任務便會醒來去做真正的工作。這就是「延遲中斷處理（deferred interrupt processing）」。

sensor.c C / FreeRTOS
SemaphoreHandle_t xDataReady;   // 用 xSemaphoreCreateBinary() 建立

// --- 中斷：保持精簡。只通知然後離開。---
void EXTI0_IRQHandler(void)
{
    BaseType_t hpw = pdFALSE;   // 「是否喚醒了更高優先權的任務？」
    __HAL_GPIO_EXTI_CLEAR_IT(GPIO_PIN_0);
    // 在中斷內部一定要用 FromISR 變體：
    xSemaphoreGiveFromISR(xDataReady, &hpw);
    // 如果我們喚醒的任務比被中斷的那個更緊急，
    // 就在從 ISR 返回的瞬間切換過去：
    portYIELD_FROM_ISR(hpw);
}

// --- 任務：在中斷情境之外做繁重的工作 ---
void vSensorTask(void *pv)
{
    for (;;) {
        // 阻塞（0 % CPU）直到 ISR 給出號誌。
        // portMAX_DELAY = 永遠等待。
        if (xSemaphoreTake(xDataReady, portMAX_DELAY) == pdTRUE) {
            read_and_process_sample();   // 安全：完整的任務情境
        }
    }
}
① 直覺 優先權反轉

互斥鎖解決了一個微妙又著名的臭蟲。假設一個低優先權任務 L 取得了 I²C 匯流排上的互斥鎖。接著一個高優先權任務 H 想用同一條匯流排，於是阻塞等待 L 釋放它。到此為止都沒問題——但現在一個中優先權任務 M（它根本不需要那條匯流排）變成就緒了。M 搶佔了 L，因為 M 的位階高於 L。於是 M 執行，L 沒法完成，L 沒法釋放互斥鎖，而 H——系統中優先權最高的任務——被卡在 M 後面。優先權實際上被反轉了。

🏭 真實案例——火星探路者號，1997 年

這不是教科書上的假想。NASA 的火星探路者號（Mars Pathfinder）在火星表面不斷重置自己，原因正是這個發生在以互斥鎖保護的共享資料匯流排上的優先權反轉。看門狗（watchdog）發現高優先權任務挨餓，於是重啟了太空船。JPL 遠端診斷出問題，並上傳了一個啟用優先權繼承的修補程式——和今天在 FreeRTOS 裡只要一個旗標就能開啟的功能一模一樣。

🔑 優先權繼承

FreeRTOS 的互斥鎖會自動修正反轉：當 H 在等待 L 持有的互斥鎖時，L 會被暫時提升到 H 的優先權。如此一來 M 就無法搶佔 L，於是 L 很快完成、釋放互斥鎖、降回它自己的優先權，H 便得以執行。單純的號誌沒有擁有者，所以它無法做到這一點——這也是守護資源時要伸手去拿互斥鎖的另一個理由。

5排程數學：它真的能趕上截止時間嗎？

② 數學 推導它

建立任務很容易；證明它們全都趕得上截止時間才是真正的工程。把每個週期性任務 $i$ 用它的最差情況執行時間 $C_i$（跑一次要多久）和它的週期 $T_i$（多久必須跑一次）來建模。它消耗的 CPU 比例是 $C_i / T_i$，所以總CPU 利用率（utilization）就是這些的總和：

$$ U = \sum_{i=1}^{n} \frac{C_i}{T_i} $$

如果 $U > 1$，你要求的就超過了 100% 的 CPU——沒有任何排程器救得了你；那在數學上是不可能的。但 $U \le 1$ 是必要條件，不是充分條件。對於速率單調排程（rate-monotonic scheduling，RMS——把較高優先權指派給週期較短的任務，這是可證明為最佳的固定優先權規則），Liu & Layland 證明了一個漂亮的充分界限：

$$ U \le n\left(2^{1/n} - 1\right) $$

這個界限會隨著你加入更多任務而縮小。一個任務時是 $1.0$；兩個任務時是 $2(\sqrt{2}-1)\approx 0.828$；而當 $n\to\infty$ 時，它收斂到一個著名的極限：

$$ \lim_{n\to\infty} n\left(2^{1/n}-1\right) = \ln 2 \approx 0.693 $$
🔑 69% 的經驗法則

如果你的總利用率維持在約 69%以下，速率單調排程就保證每一個截止時間都能達到，無論有多少個任務。超過這個界限它可能還是行得通（這個界限是保守的——你可以做精確的回應時間分析），但在界限以下，你是有證明在身的安全。

② 數學 實例演練

兩個任務：一個 IMU 讀取，每 $T_1 = 10\,\text{ms}$ 耗時 $C_1 = 2\,\text{ms}$；以及一個控制迴圈，每 $T_2 = 5\,\text{ms}$ 耗時 $C_2 = 1\,\text{ms}$。利用率：

$$ U = \frac{2}{10} + \frac{1}{5} = 0.2 + 0.2 = 0.4 $$

雙任務的 RMS 界限是 $2(2^{1/2}-1) \approx 0.828$。由於 $0.4 \le 0.828$，這組任務是可排程的（schedulable）——每個截止時間都可證明能達到，還有充裕的餘裕容納第三個任務或一場中斷風暴。（並且注意，RMS 會把較高的優先權給那個 5 ms 的控制迴圈，因為它的週期較短。）

6低功耗：執行、睡眠、停止與待命

① 直覺 用喚醒速度換電流

一個電池產品的生命大部分時間都在無所事事。藝術就在於讓「無所事事」幾乎不花任何電流。STM32 晶片提供了一道低功耗模式的階梯；每往下一階就關掉晶片更多的部分，省下更多電流，但保留的狀態更少、喚醒所需的時間更長。下面的數字對 STM32L4 而言是典型值——確切數字在資料手冊裡，但其形態是普遍適用的。

模式CPU保留的 SRAM / 暫存器喚醒來源典型電流
執行（Run）執行中全部—約 5–10 mA
睡眠（Sleep）停住（時脈關閉）全部任何中斷約 1–2 mA
停止（Stop）關閉（時脈關閉）保留所有 SRAM + 暫存器EXTI 線、RTC、IWDG約 1–10 µA
待命（Standby）關閉（核心斷電）僅備援暫存器 + RTCWKUP 腳位、RTC、重置約 0.1–1 µA
🔑 睡眠後什麼會存活

睡眠（Sleep）只是把 CPU 時脈閘掉——喚醒最快、省得最少、一切完好。停止（Stop）關掉時脈，但讓 SRAM 和暫存器持續供電，所以你能從離開的地方原地恢復；這是感測器節點的主力模式。待命（Standby）把核心電域關掉——電流最低，但 RAM 沒了，喚醒近乎一次完整的重置（reset）：你會重新跑 main()，而且只能從那塊極小的電池備援電域（RTC + 幾個備援暫存器）恢復狀態。

② 數學 從平均電流算電池壽命

電池壽命就是電荷除以你耗用它的速率。設容量為 $Q$（單位 mAh），平均電流為 $I_{\text{avg}}$（單位 mA）：

$$ t_{\text{life}} = \frac{Q}{I_{\text{avg}}} $$

整場遊戲就是工作週期化（duty cycling）：只在極小的時間比例 $d$ 內以高執行電流 $I_{\text{run}}$ 清醒，其餘時間都在停止模式下以極小的 $I_{\text{stop}}$ 待著。時間加權的平均值是：

$$ I_{\text{avg}} = d\,I_{\text{run}} + (1-d)\,I_{\text{stop}} $$

具體來說：每 $10\,\text{s}$ 清醒 $0.2\,\text{s}$（所以 $d = 0.02$），執行電流 $I_{\text{run}} = 50\,\text{mA}$，在停止模式下閒置電流 $I_{\text{stop}} = 10\,\mu\text{A} = 0.01\,\text{mA}$：

$$ I_{\text{avg}} = 0.02(50) + 0.98(0.01) \approx 1.0\,\text{mA} \;\Rightarrow\; t_{\text{life}} = \frac{2000\,\text{mAh}}{1.0\,\text{mA}} \approx 2000\,\text{h} \approx 83\,\text{天} $$

同樣的硬體若不睡眠——維持在執行模式吃 50 mA——會在 $2000/50 = 40$ 小時內耗盡，不到兩天。停止模式靠一個設計決策就為我們換來了50 倍長的壽命。這個比率正是本節存在的理由。

7無節拍閒置：讓 FreeRTOS 睡覺

① 直覺 節拍是睡眠的敵人

這裡有個矛盾：那個 1 kHz 的節拍每一毫秒就喚醒一次 CPU，只為了問「有誰準備好了嗎？」如果所有任務在接下來的 500 ms 都阻塞著，你就毫無意義地喚醒了 500 次，晶片從來沒能深到足以省下真正的電。解法是無節拍閒置（tickless idle）。

🔑 無節拍閒置做了什麼

當閒置任務執行，而 FreeRTOS 看到每一個任務在接下來的（譬如）500 ms 都阻塞著時，它就停掉節拍，設定一個低功耗計時器（RTC 或一個喚醒計時器）在 500 ms 後觸發，並把晶片降入停止模式。喚醒時它讀出實際睡了多久，並一次跳躍地把節拍計數向前修正，讓核心對時間的感知保持準確。你設定 configUSE_TICKLESS_IDLE = 1 來開啟它。

③ 程式碼 會睡眠的閒置掛鉤

最簡單的進入點是閒置掛鉤（idle hook）——一個 FreeRTOS 在閒置任務每次轉一圈時都會呼叫的函式。對於睡眠模式的設計，你可以在這裡直接用 Cortex-M 的 WFI（「等待中斷」，wait for interrupt）指令讓核心睡著；對於更深的停止模式睡眠，你會去實作完整的 portSUPPRESS_TICKS_AND_SLEEP 巨集，但閒置掛鉤把這個想法乾淨俐落地展示了出來：

freertos_hooks.c C / FreeRTOS
// 每當沒有任何其他東西準備好執行時，閒置任務就呼叫它。
// 需要 FreeRTOSConfig.h 中 configUSE_IDLE_HOOK == 1
void vApplicationIdleHook(void)
{
    // __WFI() 讓核心停住，直到下一個中斷觸發。
    // 節拍中斷（或任何周邊 IRQ）會喚醒我們，排程器
    // 重新檢查就緒清單，然後生活繼續。
    // 電流從約 mA（空轉）降到它的一小部分。
    __WFI();   // 「等待中斷」——Cortex-M 的睡眠指令
}

// 想要真正的深度睡眠，就改為啟用無節拍閒置：
//   #define configUSE_TICKLESS_IDLE 1
// 然後 FreeRTOS 會呼叫 portSUPPRESS_TICKS_AND_SLEEP(xExpectedIdleTime)，
// 你要去實作它來進入 STOP 模式並重新設定 RTC。
⚠️ 陷阱——一個忙碌的任務會讓閒置任務挨餓

晶片只有在閒置任務執行時才會睡眠，而這只有在每一個其他任務都阻塞時才會發生。一個在緊湊的 for(;;) 迴圈裡輪詢、卻從不阻塞（沒有 vTaskDelay、沒有佇列等待）的任務，會永遠把 CPU 釘在執行模式——你那個「低功耗」節點現在抽的是滿載電流。永遠要阻塞在一個延遲或一個事件上；絕不要忙等（busy-wait）。

8真實案例：一個電池感測器節點

④ 真實案例 這會出貨到產品裡

讓我們把一切組裝成一個真實產品：一個無線土壤濕度節點，它必須靠一顆 2000 mAh 的電池運作一年。它有兩個工作——取樣感測器、送出封包——而其餘時間都必須在睡眠。這個架構是教科書級的 RTOS + 低功耗設計：

每 10 秒]) --> SAMP["取樣任務

取樣任務在 RTC 節拍上醒來，讀取 ADC，並把讀值推進一個佇列。通訊任務則阻塞在那個佇列上；只有當一筆讀值到達時它才執行，傳送出去，然後再次阻塞。一旦兩者都阻塞，閒置任務就執行，無節拍閒置便把晶片降入停止模式，直到下一次 RTC 喚醒。以下是這兩個任務，以及把它們串接起來的佇列：

node.c C / FreeRTOS
QueueHandle_t xMoistureQ;   // 在 main 中建立，存放 uint16_t 讀值

// 生產者：約每 10 秒被 RTC 喚醒，取樣並入列。
void vSampleTask(void *pv)
{
    const TickType_t period = pdMS_TO_TICKS(10000);   // 10 秒
    TickType_t wake = xTaskGetTickCount();
    for (;;) {
        uint16_t moisture = adc_read(ADC_CHANNEL_3);  // 約 5 ms 的工作
        // 非阻塞送出；若佇列已滿就丟棄這筆樣本。
        xQueueSend(xMoistureQ, &moisture, 0);
        vTaskDelayUntil(&wake, period);               // 固定 10 秒速率
    }
}

// 消費者：睡到有一筆讀值存在，然後把它傳送出去。
void vCommsTask(void *pv)
{
    uint16_t reading;
    for (;;) {
        // 阻塞（0 % CPU）直到生產者送出。portMAX_DELAY
        // => 任務完全睡著，讓晶片得以抵達 STOP。
        if (xQueueReceive(xMoistureQ, &reading, portMAX_DELAY) == pdTRUE) {
            radio_transmit(&reading, sizeof reading);  // 約 150 ms
        }
    }
}

int main(void)
{
    HAL_Init();  SystemClock_Config();  MX_ADC_Init();  MX_RTC_Init();
    xMoistureQ = xQueueCreate(4, sizeof(uint16_t));     // 深度 4
    xTaskCreate(vSampleTask, "samp",  192, NULL, 3, NULL);
    xTaskCreate(vCommsTask,  "comms", 256, NULL, 2, NULL);
    vTaskStartScheduler();                              // 永不返回
    for (;;) { }
}
🏭 真實案例——為什麼用佇列，而不是共享全域變數

你大可讓兩個任務都去碰同一個全域 uint16_t——然後你就離一次時機不巧的上下文切換只有一步之遙，等著遇上撕裂讀取（torn read）或丟失樣本。佇列讓這次交接原子且阻塞：消費者以零 CPU 睡著，直到資料存在，這正是讓閒置任務得以執行、晶片得以抵達停止模式的關鍵。佇列不只是整潔；它是讓低功耗設計成為可能的東西。這個模式——生產者、佇列、消費者、深度睡眠——是市面上幾乎每一款商用電池感測器的骨架。

9⑤ 換你上場

⑤ 練習 由你撰寫與決定

三個檢查點。點一個答案、輸入一個數字，或展開一個程式挑戰。你的進度會儲存在這台裝置上，並顯示在首頁。

Q1 觀念檢查
兩個任務共用一條 I²C 匯流排，你需要阻止它們發出重疊的交易，並且避開火星探路者號那個優先權反轉的臭蟲。你會伸手去拿哪一個原語，又為什麼？
  • 二元號誌——它的額外開銷最低
  • 互斥鎖——它有擁有者，並加上了優先權繼承
  • 佇列——它能在任務之間安全地複製匯流排資料
  • 一個初始化為任務數量的計數號誌
互斥鎖是保護共享資源的正確工具：只有它的擁有者能釋放它，而且 FreeRTOS 的互斥鎖實作了優先權繼承，這正是 JPL 對探路者號施加的修正。二元號誌雖然也能做互斥，卻沒有擁有者、沒有繼承，所以它無法防止反轉。佇列是用來傳遞資料的，不是上鎖。
Q2 算一算
一個系統執行兩個週期性任務：一個 IMU 讀取，每 $T_1 = 10\,\text{ms}$ 耗時 $C_1 = 2\,\text{ms}$；以及一個控制迴圈，每 $T_2 = 5\,\text{ms}$ 耗時 $C_2 = 1\,\text{ms}$。計算總 CPU 利用率 $U = \sum C_i / T_i$（一個介於 0 與 1 之間的無因次分數）。
（分數）
$U = \dfrac{2}{10} + \dfrac{1}{5} = 0.2 + 0.2 = 0.4$。雙任務的速率單調界限是 $2(2^{1/2}-1)\approx 0.828$，而 $0.4 \le 0.828$，所以這組任務是可排程的，還有充裕的餘裕。你用掉了 40% 的 CPU；剩下 60% 留給更多任務或中斷。
Q3 程式挑戰——生產者、佇列、消費者
寫兩個由佇列串接的 FreeRTOS 任務。一個生產者以固定速率每 100 ms 讀取一個按鈕（read_button() 回傳一個 uint8_t），並把該值送進佇列。一個消費者阻塞在佇列上，每當它收到一個值，就呼叫 process(value)。別忘了在啟動排程器之前建立佇列——並確保消費者是真正地阻塞（這樣晶片才能睡眠），而不是輪詢。
提示
在 main 中、於 vTaskStartScheduler() 之前用 xQueueCreate(depth, sizeof(uint8_t)) 建立佇列。在生產者裡用 vTaskDelayUntil 維持固定的 100 ms 速率，並用 xQueueSend(q, &val, 0)。在消費者裡用 xQueueReceive(q, &val, portMAX_DELAY)——portMAX_DELAY 的意思是「永遠阻塞直到資料到達」，這讓任務在事件之間維持 0% CPU。
顯示一個解答
button_node.c C / FreeRTOS
QueueHandle_t xBtnQ;

// 生產者：以固定的 100 ms 速率取樣按鈕。
void vProducer(void *pv)
{
    const TickType_t period = pdMS_TO_TICKS(100);
    TickType_t wake = xTaskGetTickCount();
    for (;;) {
        uint8_t val = read_button();        // 0 或 1
        xQueueSend(xBtnQ, &val, 0);          // 非阻塞入列
        vTaskDelayUntil(&wake, period);      // 精確 100 ms，不漂移
    }
}

// 消費者：完全睡著，直到有一個值到達。
void vConsumer(void *pv)
{
    uint8_t val;
    for (;;) {
        if (xQueueReceive(xBtnQ, &val, portMAX_DELAY) == pdTRUE) {
            process(val);                    // 處理該事件
        }
    }
}

int main(void)
{
    HAL_Init();  SystemClock_Config();  MX_GPIO_Init();
    xBtnQ = xQueueCreate(8, sizeof(uint8_t));   // 深度 8
    xTaskCreate(vProducer, "prod", 128, NULL, 2, NULL);
    xTaskCreate(vConsumer, "cons", 192, NULL, 2, NULL);
    vTaskStartScheduler();                       // 永不返回
    for (;;) { }
}

課題重點：消費者的 portMAX_DELAY 正是讓這套設計友善於功耗的關鍵——它以零 CPU 坐在阻塞狀態，所以一旦生產者也在延遲中，閒置任務就會執行，無節拍閒置便能把晶片降入停止模式。把佇列換成二元號誌，你就能通知「發生了一次按壓」卻不攜帶是哪個按鈕——而把 vTaskDelayUntil 換回 vTaskDelay，你就會重新引入 §3 的時序漂移。

🎓 本章回顧

你現在能解釋為什麼超級迴圈對許多獨立且時序敏感的工作會失靈，並用 FreeRTOS 任務來取代它——那些永遠迴圈的函式，依優先權由 SysTick 節拍驅動、被搶佔式地排程，並以必備的閒置任務作為你的睡眠掛鉤。你用 xTaskCreate 建立任務，用 vTaskDelayUntil（絕不用會漂移的 vTaskDelay）維持固定速率，並從 uxTaskGetStackHighWaterMark 決定堆疊大小。你會挑選正確的 IPC 原語——佇列用於資料、號誌用於 ISR 對任務的訊號通知、互斥鎖用於帶優先權繼承的互斥（探路者號的修正）。你能用 $U = \sum C_i/T_i$ 對上 RMS 界限 $n(2^{1/n}-1)\to\ln 2\approx 0.693$ 來證明可排程性。而且你會用執行 / 睡眠 / 停止 / 待命模式來大幅削減平均電流，以 $Q/I_{\text{avg}}$ 計算電池壽命，並用無節拍閒置在事件之間抵達停止模式——把一個撐兩天的裝置變成撐好幾個月的裝置。這就是從業餘躍升到產品的那一跳。
