大師級總整合專題：使用 PID 的即時馬達控制

這個學習軌的所有內容，其實都是一台機器的零件。現在我們要把它們組裝起來：一個正交編碼器（quadrature encoder）餵資料給計時器、一個在中斷中以固定速率執行的 PID、用 PWM 驅動 H 橋（H-bridge），以及透過 UART+DMA 串流送出的遙測資料（telemetry）—— 一個閉迴路直流馬達轉速控制器，在外界不斷干擾下仍能守住設定值（setpoint）。這是全套課程的壓軸大戲，在這裡時脈、計時器、中斷與 DMA 不再只是各自獨立的主題，而是融合成一個系統。

① 直覺 先把整幅圖建立起來

馬達很固執。用固定電壓叫它「轉到 3000 RPM」，它只會乖乖照辦到某個條件改變為止 —— 負載變重、繞線變熱、電池電壓下降 —— 接著它就會漂移。開迴路（open-loop）控制（選一個工作週期然後祈禱它聽話）無法回擊。解法是工程學裡最古老的點子：量測輸出、跟你想要的值比較、修正兩者的差，並且持續不斷地做。那就是閉迴路（closed loop），而位於中央的控制器就是 PID。

💡 一句話模型

回授控制器一秒鐘轉動量測 → 比較 → 修正這個輪子好幾千次，每次只把致動器（actuator）推動剛剛好的量，使誤差 —— 設定值與現實之間的落差 —— 趨近於零。

2需求與模組地圖

① 直覺 先有規格再碰矽晶

在動任何一個暫存器之前，先把「好」的定義釘死。轉速控制器有可量測的需求，而它們會驅動後面所有的選擇 —— 時脈速率、取樣週期，甚至是哪個計時器分配到哪裡。

需求目標為何重要
控制速率 $f_s = 1/T_s$1 kHz（Ts = 1 ms）相對於馬達的機械時間常數要夠快；固定下來才能有穩定的積分／微分
穩態誤差≈ 0 RPM這正是積分項的全部工作
安定時間（settling time）對步階輸入 < 200 ms感覺「俐落」而不拖泥帶水
過衝（overshoot）< 10 %保護機構；限制電流尖峰
遙測速率50–100 Hz足以畫出步階響應圖；透過 DMA 成本低廉

這份韌體由四個彼此協作的模組構成。要讓它們保持解耦 —— PID 不該知道自己正在驅動一個 H 橋，而編碼器讀取器也不該知道 PID 的存在。乾淨的接縫，正是讓你能換馬達或重新調校而不必重寫一切的關鍵。

🛠️ STM32F4 上的計時器分配

我們使用 Nucleo 等級的零件（例如 STM32F411，Cortex-M4 @ 100 MHz）。TIM2（32 位元，掛在 APB1 上）= 編碼器計數器。TIM1（進階型，掛在 APB2 上）= 給 H 橋用的互補式 PWM。TIM6（基本型）= 1 kHz 的控制迴路節拍。三個計時器、三項工作 —— 互不爭用。

3量測轉速：從邊緣到 RPM

① 直覺 數齒

一個鎖在馬達軸上的正交編碼器會輸出兩個方波 A 與 B，相位差 90°。相位的先後順序告訴你方向；邊緣計數告訴你距離。在編碼器模式（第 5 章）下，計時器的硬體會免費替你完成解碼：每一個 A/B 邊緣都會讓 TIMx->CNT 遞增或遞減。CPU 從不需要為每個邊緣處理一次中斷 —— 它只要讀一個早已存好位置的暫存器即可。要得到轉速，我們每個控制週期取樣那個計數器一次，看看它移動了多少。

② 數學 誠實地推導 RPM

令 $\text{CPR}$ 為編碼器的每轉計數（counts per revolution）。資料手冊通常標的是線數（PPR，每轉脈衝數）；在 4 倍正交解碼下，計時器會計入兩個通道的每一個邊緣，因此

$$ \text{CPR} = 4 \times \text{PPR} $$

假設軸在一個控制週期 $T_s$（秒）內轉過 $\Delta\text{counts}$，而馬達與輸出之間夾著一個減速比為 $G$ 的齒輪箱（編碼器裝在馬達側）。在那段時間窗內，輸出軸的轉數是 $\Delta\text{counts}/(\text{CPR}\cdot G)$，而一分鐘有 $60/T_s$ 個這樣的時間窗，所以：

$$ \text{RPM} = \frac{\Delta\text{counts}}{\text{CPR}\cdot G}\cdot\frac{60}{T_s} $$

實例計算：一個 500 線的編碼器（$\text{PPR}=500 \Rightarrow \text{CPR}=2000$）、無齒輪箱（$G=1$）、$T_s = 1\,\text{ms}$。若一個節拍內 $\Delta\text{counts}=100$：

$$ \text{RPM} = \frac{100}{2000\cdot 1}\cdot\frac{60}{0.001} = 0.05 \times 60000 = 3000\ \text{RPM} $$
⚠️ 陷阱 —— 計數器回繞（wraparound）

差值的回繞必須正確。在一個 16 位元計數器上，從 65530 變到 4 是 +10，而不是 −65526。請計算 delta = (int16_t)(now - prev)，讓二補數減法替你完成回繞，或者使用 32 位元計時器（TIM2/TIM5）並搭配更寬的型別轉換。把這件事弄錯，你的「轉速」每溢位一次就會劇烈跳動一次。

③ 程式碼 每一行都讀過
encoder.c C / STM32 (CMSIS)
// TIM2 已預先設定為編碼器模式（TI1+TI2，4 倍）。
// CPR = 4 * PPR。Ts 是控制週期，單位為秒。
#define ENC_CPR   2000.0f   // 500 線編碼器，4 倍正交
#define ENC_GEAR  1.0f
#define CTRL_TS   0.001f    // 1 kHz

static uint32_t enc_prev = 0;

// 每個控制週期恰好呼叫一次。
float encoder_read_rpm(void) {
    uint32_t now = TIM2->CNT;                 // 硬體解碼後的位置
    // 帶正負號的差值，並做正確的 32 位元回繞
    int32_t  d   = (int32_t)(now - enc_prev);
    enc_prev = now;
    // RPM = (d / (CPR*GEAR)) * (60 / Ts)
    return ((float)d / (ENC_CPR * ENC_GEAR)) * (60.0f / CTRL_TS);
}

一次減法、一次型別轉換、一次乘法 —— 整個轉速感測器就這樣。所有的粗重活（去抖動邊緣、解碼方向）都由計時器周邊在 CPU 睡覺時完成。這正是用矽晶硬體取代輪詢接腳的全部意義所在。

4PID 控制器

① 直覺 三個聲音在爭論

PID 是三項修正量的加總。比例（P）對當下的誤差做反應 —— 落差越大、推得越用力，但純 P 永遠會留下一個殘餘偏差。積分（I）記得過去的誤差，持續累加直到連一丁點穩態落差都被消滅 —— 它能消除偏差但可能造成過衝。微分（D）藉由對誤差變化的快慢做反應來預測未來 —— 它能阻尼振盪，就像避震器一樣。調好它們各自的音量，你就會得到一個快速到達、平穩安定、且不會振鈴（ring）的控制器。

② 數學 先連續，再離散

以誤差 $e(t) = \text{設定值} - \text{量測值}$ 表示，經典的連續控制律是：

$$ u(t) = K_p\,e(t) + K_i\!\int_0^{t}\! e(\tau)\,d\tau + K_d\,\frac{de(t)}{dt} $$

微控制器無法連續積分 —— 它以寬度為 $T_s$ 的離散節拍運行。把積分換成累加和（矩形法則），把微分換成後向差分。在第 $k$ 步：

$$ u_k = K_p\,e_k \;+\; K_i\,T_s\!\sum_{j=0}^{k} e_j \;+\; K_d\,\frac{e_k - e_{k-1}}{T_s} $$

仔細看 $T_s$ 出現的位置：它乘進積分項，又除進微分項。如果 $T_s$ 飄移，同樣的 $K_i$ 與 $K_d$ 就代表完全不同的意義。光是這一個事實，就是 §6 之所以存在的原因 —— 迴圈必須以固定速率打節拍，否則你的調校就是一場謊言。

∑ 抗積分飽和（Anti-windup）—— 積分項的黑暗面

當馬達飽和（工作週期釘死在 100%）時，輸出已經無法再變大，但 $\sum e_j$ 仍持續堆積。等到設定值終於放鬆下來，那個臃腫的積分要花上好久才能消化，造成巨大的過衝。解藥是箝位（clamping）：只要輸出飽和就停止累加，並把積分限制在實際能產生合法輸出的範圍內：

$$ I_{\min} \le K_i\,T_s\!\sum e_j \le I_{\max}, \qquad I_{\max}=u_{\max} $$

③ 程式碼 更新函式
pid.c C / STM32
typedef struct {
    float kp, ki, kd;     // 增益
    float ts;             // 取樣週期 [s]
    float integ;          // 持續累加的積分項（已乘上 ki*ts）
    float prev_e;         // e_{k-1}
    float out_min, out_max;
} pid_t;

float pid_update(pid_t *p, float setpoint, float measured) {
    float e = setpoint - measured;             // e_k

    float P = p->kp * e;                        // 比例
    p->integ += p->ki * p->ts * e;              // 累加 Ki*Ts*Σe
    // 在使用之前先箝位積分（抗積分飽和）
    if (p->integ > p->out_max) p->integ = p->out_max;
    if (p->integ < p->out_min) p->integ = p->out_min;
    float D = p->kd * (e - p->prev_e) / p->ts; // 後向差分
    p->prev_e = e;

    float u = P + p->integ + D;
    if (u > p->out_max) u = p->out_max;        // 箝位輸出
    if (u < p->out_min) u = p->out_min;
    return u;                                   // 帶正負號：正負號 = 方向
}

請注意 u 是帶正負號的。它的大小代表「用多大力」；它的正負號代表「往哪個方向」。那個正負號，正是 H 橋接下來所需要的。

5把 u 變成電壓：PWM 與 H 橋

① 直覺 大小與方向

PID 交給我們一個帶正負號的數。馬達需要兩樣東西才能依此行動：多少平均電壓（由 PWM 工作週期設定）以及哪個極性（由 H 橋的開關模式設定）。我們把 u 拆成 |u| → 工作週期，以及 sign(u) → 方向接腳。

② 數學 從 PWM 暫存器算工作週期

承第 5 章：計時器計數到 $\text{ARR}$，而比較暫存器 $\text{CCR}$ 設定工作週期。PWM 頻率與工作週期為：

$$ f_{\text{pwm}} = \frac{f_{\text{tim}}}{(\text{PSC}+1)(\text{ARR}+1)}, \qquad D = \frac{\text{CCR}}{\text{ARR}+1} $$

在時脈為 $f_{\text{tim}}=84\,\text{MHz}$（APB2 計時器時脈）的 TIM1 上，選擇 $\text{PSC}=0$ 與 $\text{ARR}=4199$ 可得 $f_{\text{pwm}} = 84{\times}10^6/4200 = 20\,\text{kHz}$ —— 高於可聽範圍，所以馬達不會發出嘯叫，而且夠精細，使得 $\text{CCR}\in[0,4200]$ 能提供約 0.024% 的工作週期解析度。

🏭 實際案例 —— 為什麼是 20 kHz 而不是 1 kHz

無人機電調（ESC）、電動自行車控制器、以及 CNC 主軸驅動器，幾乎全部都把馬達 PWM 跑在 16–30 kHz，只為一個理由：人類聽覺的上限大約是 20 kHz。把切換頻率推到那之上，磁致伸縮（magnetostriction）的「嘯叫」就會消失在寂靜中。低於那之下，工作台上每一架四旋翼都在尖叫。控制迴路仍然跑在 1 kHz；只有 PWM 載波是快的。在腦中要把這兩個速率分開。

③ 程式碼 拆分正負號與大小
motor.c C / STM32
#define PWM_ARR   4199u            // 20 kHz @ 84 MHz，PSC=0

// u 是帶正負號的 PID 輸出，此處已縮放至 |u| 落在 [0, 1]。
void motor_set(float u) {
    // 方向：驅動 H 橋的致能接腳
    if (u >= 0.0f) {
        GPIOA->BSRR = (1u << 8);          // IN1 = 1  (正轉)
        GPIOA->BSRR = (1u << (9 + 16));   // IN2 = 0  (重置 bit 9)
    } else {
        GPIOA->BSRR = (1u << (8 + 16));   // IN1 = 0
        GPIOA->BSRR = (1u << 9);          // IN2 = 1  (反轉)
        u = -u;                          // 取絕對值
    }
    if (u > 1.0f) u = 1.0f;
    // 大小 -> CCR。  D = CCR/(ARR+1)  =>  CCR = D*(ARR+1)
    TIM1->CCR1 = (uint32_t)(u * (PWM_ARR + 1));
}
⚠️ 陷阱 —— 直通短路（shoot-through）

H 橋的每一臂有兩個電晶體。如果某一臂的上半與下半曾經同時導通，你就會透過它們把電源短路 —— 那是一種「直通短路」，能在數微秒內把整座橋汽化。真正的閘極驅動器會在切換之間插入死區時間（dead-time）（數百奈秒，期間兩者皆關閉）。TIM1 透過 BDTR 暫存器的 DTG 欄位，以硬體產生這段死區時間。永遠不要用沒有死區時間的裸 GPIO 去驅動半橋。

6固定速率的控制迴路

① 直覺 節拍器

整個迴圈都活在一個每隔 $T_s$ 觸發一次的計時器中斷裡。不是「大概每毫秒、等主迴圈有空時」—— 而是恰好每毫秒、毫無抖動（jitter），因為硬體計時器（TIM6）產生中斷的方式與 CPU 此刻在做什麼無關。在 ISR 內部我們執行那三個步驟然後離開。其他所有事情（遙測格式化、命令解析）都發生在較慢的主迴圈裡。

🔑 為何固定的 $T_s$ 沒有妥協餘地

回想離散控制律：積分項是 $K_i T_s \sum e$，而微分項是 $K_d (e_k - e_{k-1})/T_s$。兩者都直接隨 $T_s$ 縮放。如果週期抖動 —— 比方說 ISR 偶爾被一個忙碌的 printf 延遲 —— 那麼有效的 $K_i$ 與 $K_d$ 也會跟著抖動。一個為 1 ms 調好的控制器，在 1.3 ms 時行為會不同：微分的踢力減弱、積分過度累加，一個原本穩定的迴圈可能開始振鈴、甚至變得不穩定。固定的 $T_s$ 才能讓你調好的那套數學，真正就是實際運行的那套數學。

② 數學 產生 1 kHz 的節拍

在這個零件上，掛在 APB1 的 TIM6 看到的是 84 MHz 的計時器時脈。要做到 1 kHz 的更新，我們要 $(\text{PSC}+1)(\text{ARR}+1) = f_{\text{tim}}/f_s$：

$$ (\text{PSC}+1)(\text{ARR}+1) = \frac{84\times10^6}{1000} = 84000 $$

選 $\text{PSC}=83$（除以 84 → 1 MHz 的節拍）與 $\text{ARR}=999$（數 1000 個節拍 → 1 ms）。乾淨、精確，凌晨 3 點也能在腦中重新推導出來。

7完整整合的韌體

③ 程式碼 每一行都讀過

下面是控制 ISR —— 跳動的心臟。它刻意做得極小：三個驅動程式呼叫加上一個旗標。決定性來自在中斷內部做得更少，而不是更多。

control.c C / STM32 (CMSIS)
volatile float g_setpoint_rpm = 0.0f;   // 由主機下達命令
volatile float g_measured_rpm = 0.0f;   // 最新量測值（供遙測用）
volatile uint8_t g_tlm_ready  = 0;      // ISR -> 主迴圈的交握

static pid_t pid = {
    .kp = 0.0008f, .ki = 0.012f, .kd = 0.00003f,
    .ts = CTRL_TS, .integ = 0.0f, .prev_e = 0.0f,
    .out_min = -1.0f, .out_max = 1.0f
};

// TIM6 以「恰好」1 kHz 觸發此函式。讓它保持短小、分支少。
void TIM6_DAC_IRQHandler(void) {
    if (TIM6->SR & TIM_SR_UIF) {        // 確認這是更新事件
        TIM6->SR &= ~TIM_SR_UIF;        // 清除中斷旗標（W0C）

        float rpm = encoder_read_rpm();           // 1. 量測
        float u   = pid_update(&pid,
                               g_setpoint_rpm, rpm); // 2. 計算
        motor_set(u);                              // 3. 致動

        g_measured_rpm = rpm;          // 發布給遙測使用
        g_tlm_ready    = 1;            // 喚醒主迴圈的傳送者
    }
}

main() 啟動時脈與周邊，然後永遠循環，只做那些緩慢、非關鍵的工作：把遙測節流到約 50 Hz，並透過 UART+DMA 送出去，讓 CPU 永遠不會卡在那條緩慢的序列線上（第 7/8 章）。

main.c C / STM32
static char tlm[48];

int main(void) {
    clock_config_84mhz();      // 第 2 章：PLL -> 84 MHz 計時器時脈
    encoder_init_tim2();       // 第 5 章：TIM2 編碼器模式，4 倍
    pwm_init_tim1_20khz();     // 第 5 章：TIM1 PWM + 死區時間（BDTR）
    uart_dma_init();           // 第 7/8 章：USART2 TX 透過 DMA1
    ctrl_timer_init_1khz();    // TIM6，PSC=83，ARR=999，啟用 IRQ

    g_setpoint_rpm = 1500.0f;  // 示範：守住 1500 RPM

    uint32_t div = 0;
    for (;;) {
        if (g_tlm_ready) {
            g_tlm_ready = 0;
            // 抽取 1 kHz -> ~50 Hz，以免淹沒 UART
            if (++div >= 20) {
                div = 0;
                int n = snprintf(tlm, sizeof tlm, "%ld,%ld\n",
                                 (long)g_setpoint_rpm,
                                 (long)g_measured_rpm);
                uart_dma_send((uint8_t *)tlm, n);  // 非阻塞
            }
        }
        // CPU 此外都是閒置的：睡到下一個事件再說
        __WFI();
    }
}
🏭 實際案例 —— 為什麼用 DMA 而不是 printf

在 115200 鮑率（baud）下，用阻塞式的 HAL_UART_Transmit 一個位元組一個位元組地推送約 14 位元組的遙測，會讓 CPU 停擺超過一毫秒 —— 長到足以錯過下一個控制節拍。把緩衝區交給 DMA，傳輸就會在背景進行，同時核心在 __WFI() 中沉睡。這正是量產馬達驅動、飛行控制器與實驗室儀器裡的固定模式：快速且具決定性的迴圈是神聖不可侵犯的，而緩慢的 I/O 永遠要卸載出去。打破這條規則，你的迴圈抖動（從而你的調校）就會土崩瓦解。

8調校與除錯

④ 實際案例 讓它真的乖乖聽話

增益不是從公式裡來的 —— 它是從方法裡來的。一套 Ziegler–Nichols 風格的程序，能給你一個合理的起點：

步驟你做什麼你觀察什麼
1. 只用 P設 $K_i=K_d=0$；逐步提高 $K_p$ 直到轉速穩定地振盪持續振盪的起始點 → 那就是臨界增益 $K_u$
2. 量測記錄振盪週期 $T_u$遙測圖上波峰與波峰之間的週期
3. 套用經典 ZN：$K_p=0.6K_u$，$K_i=1.2K_u/T_u$，$K_d=0.075K_uT_u$步階響應：上升、過衝、安定
4. 微調憑感覺撥動：多加 $K_d$ 來阻尼振鈴，減少 $K_i$ 來削減過衝過衝 < 10%，安定 < 200 ms

最有用的單一診斷工具就是步階響應：對設定值下達一個突然的跳變，然後把量測 RPM 隨時間相對於命令值畫出來。它的形狀就是你的調校，一眼即可讀懂：

🛠️ 從外部觀察迴圈

觀察一個運行中控制器的三扇窗，由便宜到貴：(1) UART+DMA 遙測串流 → 在序列繪圖器（serial plotter）中即時畫出來。(2) SWD/ITM —— 在 ISR 的開頭與結尾切換一個 GPIO 並用示波器量測；脈衝寬度就是你的 ISR 執行時間，而它的間距則證明了 $T_s$ 毫無抖動。(3) 偵錯器對 g_measured_rpm 與 pid.integ 的即時監看（live-watch）—— 一個不斷爬升、永遠無法安定的積分，就是飽和（windup）的氣味。

⚠️ 常見的失效模式

回授正負號搞錯 —— 編碼器 A/B 接反，導致誤差修正反而注入能量：馬達瞬間衝到全速。解法：把計數差值取負，或交換通道。
ISR 超時（overrun） —— 控制 ISR 花的時間超過 $T_s$；下一個節拍被錯過，而 $T_s$ 悄悄地倍增。解法：縮小 ISR，把工作移到主迴圈。
積分飽和（windup） —— 飽和之後恢復緩慢且過衝很大。解法：§4 的那個箝位。
微分雜訊 —— 一個有雜訊的編碼器讓 $D$ 尖叫；請對量測值濾波，或降低 $K_d$。

9接下來往哪走：串級控制與電流限制

④ 實際案例 專業人士怎麼蓋

單一速度迴圈是地基；認真的驅動器會疊起多重迴圈。標準架構是串級（cascade）：一個外層的位置／速度 PID，其輸出就是一個跑得更快的內層電流（轉矩）PID 的設定值。內層迴圈直接限制電流 —— 而限制電流就是限制轉矩就是保護你的馬達與橋免於燒毀。

flowchart LR SP([速度設定值]) --> S["速度 PID
(1 kHz)"] S -- "電流設定值" --> C["電流 PID
(10 kHz)"] C -- "工作週期" --> P["PWM + 橋"] P --> M(["馬達"]) M -- "軸轉速" --> S M -- "相電流
(ADC 分流電阻)" --> C

內層迴圈透過 ADC 從一個分流電阻（shunt resistor）讀取馬達電流，通常與 PWM 同步（在計數的底部取樣，此時電流穩定）。有了它，你就得到了硬性、快速的電流限制 —— 而你距離驅動每一台現代無刷馬達（從無人機到電動車）的磁場導向控制（FOC, field-oriented control）就只差一步了。你在這裡組裝起來的零件，正是 FOC 賴以建構的那些零件。

🔑 這個模式可以一般化

用計時器／ADC 量測、在固定速率 ISR 中計算、用 PWM 致動、透過 DMA 遙測觀察。換掉被控對象（plant）—— 加熱器、無人機姿態、電池充電器、雷射功率 —— 而這副骨架完全相同。你學到的不只是馬達控制；你學到的是每一個即時控制系統的形狀。

10⑤ 換你上場

⑤ 練習 你來寫、你來決定

三道檢核：一道概念、一道計算、一段程式碼。你的進度會儲存在這台裝置上，並顯示在首頁。

Q1 概念檢核
為什麼 PID 控制迴路必須以固定取樣週期 $T_s$ 運行，而不能只是「等主迴圈有空就跑」？
  • 因為編碼器硬體只在固定排程下更新它的計數器
  • 因為離散的積分項（$K_iT_s\sum e$）與微分項（$K_d\,\Delta e/T_s$）會隨 $T_s$ 縮放，所以抖動的週期會改變有效增益，可能讓一個調好的迴圈失穩
  • 因為除非迴圈週期固定，否則無法產生 PWM
  • 因為 UART+DMA 遙測需要固定的資料速率
積分項乘上 $T_s$，而微分項除以它。如果 $T_s$ 漂移（例如一個阻塞的 printf 延遲了 ISR），那麼有效的 $K_i$ 與 $K_d$ 也跟著漂移 —— 你精心調好的那個控制器，不再是實際運行的那個控制器，而一個穩定的迴圈可能開始振鈴或變得不穩定。編碼器、PWM 與遙測都能容忍抖動；唯獨控制數學不行。
Q2 算算看
一個 500 線的編碼器（所以 $\text{CPR}=4\times500=2000$）裝在馬達軸上，無齒輪箱（$G=1$）。控制迴路跑在 1 kHz（$T_s=0.001\,\text{s}$）。在一個控制週期內計數器前進了 $\Delta\text{counts}=100$。轉速是多少 RPM？請用 $\text{RPM} = \dfrac{\Delta\text{counts}}{\text{CPR}\cdot G}\cdot\dfrac{60}{T_s}$。
RPM
$\text{RPM} = \dfrac{100}{2000\cdot 1}\cdot\dfrac{60}{0.001} = 0.05 \times 60000 = 3000\ \text{RPM}$。分數 $100/2000 = 0.05$ 是這個 1 ms 時間窗內的轉數；乘上每分鐘 $60/T_s = 60000$ 個時間窗，就得到 RPM。把 $\Delta\text{counts}$ 減半，轉速就減半；把 $T_s$ 加倍，同樣的差值讀出的 RPM 就只剩一半。
Q3 程式挑戰 —— 離散 PID 更新
寫一個 C 函式 pid_update(pid_t *p, float setpoint, float measured)，實作離散控制律 $u_k = K_p e_k + K_i T_s\sum e + K_d (e_k - e_{k-1})/T_s$，回傳一個箝位在 [out_min, out_max] 的帶正負號輸出，並包含積分的抗飽和箝位。假設此結構持有 kp, ki, kd, ts, integ, prev_e, out_min, out_max。
提示
先計算 e = setpoint - measured。把積分以 integ += ki*ts*e 累加，並在使用它之前把 integ 箝位到輸出邊界（那就是抗積分飽和）。微分是 kd*(e - prev_e)/ts —— 記得之後要更新 prev_e。把三項加總、箝位總和、回傳。回傳值的正負號就是馬達方向。
顯示一種解法
pid.c C / STM32
float pid_update(pid_t *p, float setpoint, float measured) {
    float e = setpoint - measured;             // e_k

    float P = p->kp * e;                        // 比例

    p->integ += p->ki * p->ts * e;              // Ki*Ts*Σe
    if (p->integ > p->out_max) p->integ = p->out_max;  // 抗積分飽和
    if (p->integ < p->out_min) p->integ = p->out_min;

    float D = p->kd * (e - p->prev_e) / p->ts; // 後向差分
    p->prev_e = e;                             // 為下一個節拍儲存

    float u = P + p->integ + D;
    if (u > p->out_max) u = p->out_max;        // 輸出箝位
    if (u < p->out_min) u = p->out_min;
    return u;   // |u| -> PWM 工作週期，sign(u) -> H 橋方向
}

是兩個箝位，不是一個：箝位積分，讓它無法累加到超過致動器所能輸出的程度；以及箝位總輸出到合法的工作週期範圍。儲存 prev_e 正是讓微分成為跨節拍真正差分的關鍵 —— 而這也正是迴圈必須以固定 $T_s$ 運行的原因。

🎓 本章回顧

你把整個 STM32 學習軌組裝成了一台閉迴路機器。你能在處理正確回繞的前提下，從編碼器計數推導出 RPM；能寫出離散 PID 控制律，並解釋為何固定的 $T_s$ 是承重結構；能把一個帶正負號的控制輸出，在 20 kHz 下拆分成 PWM 大小與 H 橋方向；能在一個具決定性的 1 kHz 計時器 ISR 中運行整個迴圈，同時把遙測卸載給 UART+DMA；並能從步階響應來調校與除錯它。同一副骨架 —— 量測、在固定速率 ISR 中計算、致動、透過 DMA 觀察 —— 同樣驅動著加熱器、無人機與電動車馬達。那就是即時控制的形狀，而現在它屬於你了。
