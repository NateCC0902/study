Interrupts Done Right: NVIC, EXTI & Priorities

Polling burns the CPU asking "anything yet?" a million times a second. An interrupt lets the hardware tap the CPU on the shoulder only when something actually happened. Get this wrong and your firmware hangs, re-fires forever, or starves a critical task. Get it right and you build systems that sleep at microamps and wake in microseconds.

1Polling vs. interrupts: who asks the question?

① Intuition build the picture first

Imagine you're expecting a package. Polling is walking to the front door every ten seconds to check if it arrived — exhausting, wasteful, and you still might miss it between checks. An interrupt is a doorbell: you do whatever you like, and the instant the package lands the bell yanks your attention to the door. When it's handled, you go right back to what you were doing.

In code, polling a button looks like this — a tight loop reading the pin, spending nearly all its cycles on "nope, not yet":

polling.c C / STM32 HAL
while (1) {
  // CPU spins here forever, reading the pin millions of times/second.
  // 99.999% of those reads return "still not pressed."
  if (HAL_GPIO_ReadPin(GPIOA, GPIO_PIN_0) == GPIO_PIN_SET) {
    do_something();
  }
}
💡 The one-sentence model

An interrupt is hardware that pauses your main code, runs a tiny handler, and resumes exactly where it left off — all without your main() loop knowing it happened.

On a Cortex-M, the chip that orchestrates all of this is the NVIC — the Nested Vectored Interrupt Controller. It is a dedicated block inside the CPU core (not a peripheral you bolt on) whose entire job is to watch dozens of interrupt sources, decide which is most urgent, and hand control to the right handler. "Nested" because a more urgent interrupt can interrupt a less urgent one already running. "Vectored" because each source jumps directly to its own handler address with no software dispatch.

🔑 Why this matters

Interrupts are the difference between a CPU that's busy and one that's useful. Polling pins your core at 100% load and forbids real sleep. Interrupts let the core idle (or power down) and respond with bounded, predictable latency. Every responsive, low-power embedded product is built on them.

2The vector table & exception model

① Intuition a phone book of handlers

When an interrupt fires, the CPU needs to know where to jump. The answer lives in the vector table: an array of function pointers parked at the very start of Flash (address 0x0800_0000 on an STM32F4). Each slot holds the address of the handler for one exception or interrupt. The NVIC simply indexes into this table by the interrupt number and loads the program counter — no switch statement, no overhead.

OffsetVectorWhat it is
0x00Initial SPTop-of-stack value loaded on reset
0x04ResetFirst instruction after power-up
0x08NMINon-maskable interrupt
0x0CHardFaultThe "something went badly wrong" catch-all

0x40+IRQ0…IRQnPeripheral interrupts: EXTI, timers, UART, DMA…

Exceptions with a negative exception number (Reset, NMI, HardFault, SysTick) are core "system exceptions." Everything with a non-negative IRQ number — EXTI0_IRQn, TIM2_IRQn, USART2_IRQn — is an external interrupt managed by the NVIC. The handler names you see in startup_stm32f4xx.s (e.g. EXTI0_IRQHandler) are weak symbols: define a function with the exact same name in your C file and the linker wires your version into the table automatically.

⚠️ Pitfall — the name must be exact

If you spell the handler EXTI0_IRQ_Handler (with an extra underscore) instead of EXTI0_IRQHandler, the linker won't complain — it just keeps the default weak handler, which is usually an infinite while(1) loop. Your interrupt fires, jumps into that trap, and your firmware silently freezes. Copy the name straight from the startup file.

3EXTI: turning a pin edge into an interrupt

① Intuition the wiring from pin to NVIC

A GPIO pin can't talk to the NVIC directly. Between them sits the EXTI — the EXTernal Interrupt/event controller. EXTI has a set of independent lines; line 0 through line 15 are the GPIO lines (higher lines serve internal sources like the RTC alarm and USB wakeup). To make a falling edge on a button generate an interrupt, you must configure four things in order:

  1. SYSCFG — route a port to the line. SYSCFG->EXTICR[] picks which port (A, B, C…) drives a given EXTI line number.
  2. Edge select — set the rising-trigger register EXTI->RTSR and/or the falling-trigger register EXTI->FTSR for that line.
  3. Unmask — set the line's bit in the interrupt-mask register EXTI->IMR so the edge actually propagates as an interrupt (leave it masked and the edge is detected but ignored).
  4. Enable in NVIC — call NVIC_EnableIRQ() for the IRQ that owns that line.
⚠️ Pitfall — the line-sharing gotcha

EXTI line N is shared by every pin numbered N across all ports. PA0, PB0, PC0… all map to EXTI0. SYSCFG->EXTICR lets only one port own line 0 at a time — you cannot have an interrupt on both PA0 and PB0 simultaneously. Worse, lines 5–9 share a single IRQ (EXTI9_5_IRQn) and lines 10–15 share another (EXTI15_10_IRQn), so one handler must inspect the pending register to learn which pin fired.

Here is the register-level setup for a button on PA0, falling-edge triggered. This is what HAL does under the hood:

exti_setup.c C / STM32F4 CMSIS
// Assumes GPIOA & SYSCFG clocks already enabled, PA0 set to input.
// 1) Route EXTI line 0 to port A (bits [3:0] of EXTICR[0] = 0b0000 = port A)
SYSCFG->EXTICR[0] &= ~SYSCFG_EXTICR1_EXTI0;     // clear -> selects port A

// 2) Trigger on the FALLING edge only (button pulls the pin to GND)
EXTI->FTSR |=  (1U << 0);   // falling-edge enable for line 0
EXTI->RTSR &= ~(1U << 0);   // make sure rising is OFF

// 3) Unmask line 0 so the edge becomes a real interrupt request
EXTI->IMR  |=  (1U << 0);

// 4) Enable EXTI0 in the NVIC and give it a priority (see §4)
NVIC_SetPriority(EXTI0_IRQn, 5);
NVIC_EnableIRQ(EXTI0_IRQn);

4Priorities: preemption, sub-priority & the bit math

② Math derive the level count

Not all interrupts are equal. A motor over-current fault must be allowed to interrupt a lazy UART byte. The NVIC encodes each interrupt's urgency as an 8-bit priority field per source — but a given chip only implements the top few bits. The STM32F4 implements 4 bits:

$$ \text{levels} = 2^{N_{\text{prio}}} = 2^{4} = 16 $$

So you get priority values 0 through 15. The crucial, counter-intuitive rule:

🔑 Lower number = higher priority

Priority 0 is the most urgent; priority 15 is the least. This trips up everyone once. Think of it as "first place" — rank 0 wins. Reset, NMI, and HardFault sit at fixed negative priorities, so they outrank any interrupt you can configure.

Those 4 bits are split into two fields by the priority grouping setting: a preemption (group) priority and a sub-priority. The split is set once at boot with NVIC_SetPriorityGrouping(g). The grouping number $g$ determines how many of the low bits become sub-priority. With 4 implemented bits, the count of each field is:

$$ N_{\text{sub}} = \max\!\bigl(0,\; (g+1) - (8 - N_{\text{prio}})\bigr), \qquad N_{\text{pre}} = N_{\text{prio}} - N_{\text{sub}} $$

That formula looks fussy because the field is stored left-aligned in the 8-bit register; the practical results for the STM32F4 ($N_{\text{prio}}=4$) are just a table you can memorise:

Grouping constantPreempt bitsSub bitsPreempt levelsSub levels
NVIC_PRIORITYGROUP_004116
NVIC_PRIORITYGROUP_22244
NVIC_PRIORITYGROUP_440161

The two fields do different jobs:

  • Preemption priority decides nesting. A higher-preemption interrupt (lower number) can interrupt a currently-running lower-preemption handler. This is what "nested" in NVIC means.
  • Sub-priority only breaks ties: if two interrupts of the same preemption level are pending at once, the lower sub-priority runs first — but it will not preempt an already-running handler of equal preemption.
💡 Concrete example

Say IRQ A has preemption 1 and IRQ B has preemption 3. While B's handler is running, A fires — because $1 < 3$, A preempts B, runs to completion, then B resumes. If instead both had preemption 2 but different sub-priorities, whichever was running keeps running; the other waits its turn.

HAL wraps the encode-and-set step in one call so you never shift bits by hand:

priority.c C / STM32 HAL
// Choose 4 preemption levels + 4 sub-priority levels for the whole system.
HAL_NVIC_SetPriorityGrouping(NVIC_PRIORITYGROUP_2);

// EXTI0: preemption = 1 (quite urgent), sub-priority = 0
HAL_NVIC_SetPriority(EXTI0_IRQn, /*preempt*/ 1, /*sub*/ 0);
HAL_NVIC_EnableIRQ(EXTI0_IRQn);

// A chattier, less urgent UART: preemption = 3, sub = 0
HAL_NVIC_SetPriority(USART2_IRQn, /*preempt*/ 3, /*sub*/ 0);
HAL_NVIC_EnableIRQ(USART2_IRQn);
// Now a button edge will interrupt an in-progress UART ISR, but not vice-versa.
⚠️ Pitfall — set the grouping exactly once

HAL's HAL_Init() already calls HAL_NVIC_SetPriorityGrouping(NVIC_PRIORITYGROUP_4) by default — all 4 bits become preemption, zero sub-priority. If you re-group later, the meaning of every priority number you already set silently changes. Pick a grouping at startup and leave it. Mixing FreeRTOS in? The kernel requires NVIC_PRIORITYGROUP_4 — don't fight it.

5The full flow — and the bit you MUST clear

① Intuition follow one edge end-to-end

Let's trace a single button press from copper to handler. The one step beginners skip — and pay for dearly — is the last one: clearing the pending bit.

flowchart TD A["Edge on PA0"] --> B["EXTI detects edge
sets pending bit EXTI→PR.0"] B --> C["NVIC sees request
checks priority"] C --> D["CPU pushes context
vectors to EXTI0_IRQHandler"] D --> E["Your ISR runs
do the work"] E --> F{"Clear EXTI→PR.0?"} F -- "no" --> B F -- "yes (write 1)" --> G["return → resume main"]
🔑 The pending bit is "rc_w1" — clear by writing a 1

When the edge fires, EXTI latches a pending flag in EXTI->PR. This flag is what told the NVIC to call you. If your handler returns without clearing it, the NVIC sees the request is still asserted and immediately re-enters your ISR — forever. You clear it by writing a 1 to that bit (writing 0 does nothing). It's a deliberately odd "write-1-to-clear" register so two handlers can't accidentally clear each other's flags.

③ Code read every line

The bare-metal ISR — note the write-1-to-clear and the "is it really my line?" guard:

isr_baremetal.c C / STM32F4 CMSIS
void EXTI0_IRQHandler(void) {
  // Guard: confirm THIS line is the one that's pending.
  if (EXTI->PR & (1U << 0)) {
    EXTI->PR = (1U << 0);          // write-1-to-clear. DO THIS FIRST or early.

    GPIOC->ODR ^= (1U << 13);      // toggle the LED on PC13 — fast, atomic.
  }
  // Keep ISRs SHORT: no printf, no HAL_Delay, no heavy loops in here.
}

If you prefer HAL, the plumbing is hidden inside two functions. HAL's IRQ handler reads EXTI->PR, clears the flag for you, then calls a weak callback you override:

isr_hal.c C / STM32 HAL
// 1) The real vector-table entry. Let HAL parse & clear the pending bit.
void EXTI0_IRQHandler(void) {
  HAL_GPIO_EXTI_IRQHandler(GPIO_PIN_0);   // clears EXTI→PR.0 internally
}

// 2) HAL then calls THIS weak callback. Override it with your logic.
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin) {
  if (GPIO_Pin == GPIO_PIN_0) {
    HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13);   // your work goes here
  }
}
⚠️ Pitfall — keep ISRs short

An ISR runs with your main code paused and (depending on priority) other interrupts blocked. Do the minimum: clear the flag, set a volatile flag or push to a queue, get out. Calling HAL_Delay() inside an ISR is a classic hang — it waits on the SysTick interrupt, which may have lower priority than your handler and can therefore never fire. The result is a permanent freeze that looks like nothing happened.

6Real case: microsecond capture & wake-from-sleep

④ Real case this ships in products

Two production patterns lean entirely on EXTI + NVIC.

Low-latency event capture. An optical encoder, a hall sensor, or a comms PHY can produce edges thousands of times per second with hard timing requirements. Polling would miss edges between reads; an EXTI interrupt timestamps each edge the instant it happens by reading a free-running timer inside the ISR. The latency from edge to first ISR instruction on an STM32F4 is a deterministic ~12 CPU cycles of hardware push plus your priority wait — nanoseconds, not milliseconds.

🏭 Real case — wake-from-sleep on a pin

This is EXTI's quiet superpower. Put an STM32 into Stop mode and the CPU clock halts — the core draws single-digit microamps, perfect for a battery sensor that sleeps for days. But the EXTI block stays alive in Stop mode. Configure a wake button (or a sensor's interrupt line) as an EXTI source, call HAL_PWR_EnterSTOPMode(...), and the chip sits in near-zero-power sleep until an edge arrives. That edge fires the EXTI, which wakes the core, runs your handler, and returns to deep sleep. A doorbell sensor, a TV remote, a glucose monitor — all live for years on a coin cell because of exactly this mechanism.

🛠️ Debounce belongs out of the ISR

Mechanical buttons bounce: one press can generate dozens of edges over a few milliseconds, firing your ISR a dozen times. Don't delay() inside the handler to debounce — that blocks everything. Instead, on the first edge, record a timestamp and ignore further edges for ~20 ms (in the ISR, or by disabling the line and re-arming it from a timer). Software debounce is a whole topic; just know the ISR is the wrong place to wait.

7Latency math: how fast does the CPU respond?

② Math turning cycles into time

"Interrupts are fast" is a feeling; engineering needs a number. On a Cortex-M4 the hardware takes a fixed 12 cycles to stack the context and vector to your handler (assuming zero-wait-state Flash). At the STM32F4's typical $f_{\text{clk}} = 168\,\text{MHz}$, the cycle time is

$$ T_{\text{cycle}} = \frac{1}{f_{\text{clk}}} = \frac{1}{168\times10^{6}\,\text{Hz}} \approx 5.95\,\text{ns} $$

so the best-case interrupt entry latency — edge accepted to first instruction of your ISR — is

$$ t_{\text{latency}} = 12 \cdot T_{\text{cycle}} = 12 \times 5.95\,\text{ns} \approx 71\,\text{ns} $$

That's the floor. Real worst-case latency adds two things you control: any higher-or-equal preemption ISR already running (which you must wait out — design your priorities so the urgent line wins), and any region where you've disabled interrupts with __disable_irq(). Keep those critical sections to a few instructions and the system stays responsive. The maximum acceptable event rate then follows directly: if your ISR plus overhead takes $t_{\text{isr}}$, you can sustain at most

$$ f_{\max} = \frac{1}{t_{\text{isr}}} $$

edges per second before ISRs start stacking up and you fall behind. A 1 µs ISR caps you near 1 MHz of events — comfortable for a button, tight for a high-speed encoder, which is exactly when you graduate from EXTI to a hardware timer-capture peripheral.

Q1 Conceptual check
Your EXTI0_IRQHandler runs once on a button press, then the CPU appears to lock up — the handler keeps re-entering endlessly. You also notice that an interrupt you set to priority 2 never preempts one at priority 9. Which two statements are both correct?
  • You forgot NVIC_EnableIRQ, and a higher number means higher priority.
  • You forgot to clear the pending bit in EXTI->PR, and a lower number means higher priority (so 2 should preempt 9).
  • The vector table is corrupt, and priority 9 outranks priority 2.
  • EXTI auto-clears its flag, and equal sub-priorities never preempt.
The endless re-entry is the signature of not clearing EXTI->PR (write a 1 to the line's bit). And Cortex-M priority is "rank 0 wins": lower number = higher priority, so a priority-2 interrupt should preempt a priority-9 one. If it isn't, your priority grouping probably put those bits into sub-priority, which only breaks ties and never preempts.
Q2 Compute it
The STM32F4 implements 4 priority bits in the NVIC. How many distinct interrupt priority levels does that give you? (Recall $\text{levels} = 2^{N_{\text{prio}}}$.)
levels
$2^{4} = 16$ levels, numbered 0 (most urgent) through 15 (least). A chip implementing only 3 bits would give $2^3 = 8$ levels; a high-end part with 8 bits gives $2^8 = 256$. The number of implemented bits — not the full 8-bit field — is what counts.
Q3 Coding challenge — button EXTI toggles an LED
Using HAL, configure the on-board user button on PA0 as a falling-edge EXTI source, and toggle the LED on PC13 on each press. Write both the GPIO/NVIC init and the interrupt handler + callback. Remember: the handler must clear the pending flag (or let HAL do it), the LED toggle goes in the callback, and ISRs stay short.
Hint
Enable the GPIOA, GPIOC, and SYSCFG clocks first. Configure PA0 with GPIO_MODE_IT_FALLING (HAL's "interrupt on falling edge" mode does the SYSCFG/EXTI wiring for you). Then HAL_NVIC_SetPriority + HAL_NVIC_EnableIRQ for EXTI0_IRQn. The vector entry calls HAL_GPIO_EXTI_IRQHandler(GPIO_PIN_0); your toggle lives in HAL_GPIO_EXTI_Callback.
Show one solution
button_exti.c C / STM32 HAL
void EXTI_Button_Init(void) {
  __HAL_RCC_GPIOA_CLK_ENABLE();   // button port
  __HAL_RCC_GPIOC_CLK_ENABLE();   // LED port
  __HAL_RCC_SYSCFG_CLK_ENABLE();  // needed for EXTI port routing

  // LED on PC13: push-pull output
  GPIO_InitTypeDef led = {0};
  led.Pin   = GPIO_PIN_13;
  led.Mode  = GPIO_MODE_OUTPUT_PP;
  led.Pull  = GPIO_NOPULL;
  led.Speed = GPIO_SPEED_FREQ_LOW;
  HAL_GPIO_Init(GPIOC, &led);

  // Button on PA0: interrupt on FALLING edge, with pull-up
  GPIO_InitTypeDef btn = {0};
  btn.Pin  = GPIO_PIN_0;
  btn.Mode = GPIO_MODE_IT_FALLING;   // wires SYSCFG + EXTI FTSR + IMR
  btn.Pull = GPIO_PULLUP;
  HAL_GPIO_Init(GPIOA, &btn);

  // NVIC: enable EXTI0, modest priority
  HAL_NVIC_SetPriority(EXTI0_IRQn, /*preempt*/ 2, /*sub*/ 0);
  HAL_NVIC_EnableIRQ(EXTI0_IRQn);
}

// Vector-table entry — HAL parses & clears EXTI→PR for us.
void EXTI0_IRQHandler(void) {
  HAL_GPIO_EXTI_IRQHandler(GPIO_PIN_0);
}

// HAL calls this after clearing the flag. Keep it short.
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin) {
  if (GPIO_Pin == GPIO_PIN_0) {
    HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13);
  }
}

Notice the division of labour: GPIO_MODE_IT_FALLING handles all the SYSCFG/EXTI register writes, the NVIC calls are explicit, and the actual work is one toggle in the callback. In production you'd add software debounce so a single press doesn't register as several — but the interrupt skeleton is exactly this.

🎓 Chapter recap

You can now explain why interrupts beat polling, how the NVIC dispatches through the vector table, and the four-step EXTI path (SYSCFG route → edge select → IMR unmask → NVIC enable) — including the line-sharing gotcha where PA0 and PB0 both land on EXTI0. You know that the STM32F4's 4 priority bits give $2^4=16$ levels where lower number = higher priority, and that preemption priority enables nesting while sub-priority only breaks ties. Above all, you'll never forget to write-1-to-clear EXTI->PR — the bit that, left alone, re-fires your ISR forever. Next we'll use this to capture timer events with hardware precision.
